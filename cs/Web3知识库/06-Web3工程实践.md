### 1.4 测试合约

```javascript
// test/Token.test.js
const { expect } = require("chai");
const { ethers } = require("hardhat");

describe("MyToken", function () {
  let token, owner, addr1;
  
  beforeEach(async function () {
    [owner, addr1] = await ethers.getSigners();
    const Token = await ethers.getContractFactory("MyToken");
    token = await Token.deploy(ethers.parseEther("1000000"));
  });
  
  describe("Deployment", function () {
    it("Should set the right owner", async function () {
      expect(await token.owner()).to.equal(owner.address);
    });
    
    it("Should assign total supply to owner", async function () {
      const balance = await token.balanceOf(owner.address);
      expect(balance).to.equal(ethers.parseEther("1000000"));
    });
  });
  
  describe("Transfers", function () {
    it("Should transfer tokens", async function () {
      await token.transfer(addr1.address, ethers.parseEther("100"));
      expect(await token.balanceOf(addr1.address))
        .to.equal(ethers.parseEther("100"));
    });
    
    it("Should fail if sender doesn't have enough tokens", async function () {
      await expect(
        token.connect(addr1).transfer(owner.address, 1)
      ).to.be.revertedWithCustomError(token, "ERC20InsufficientBalance");
    });
  });
});
```

}

## 5. The Graph 链上数据索引

### 5.1 为什么需要 The Graph

```
链上数据查询问题
─────────────────────────────────────────────────────────────

直接查询 RPC:
- 只能按区块/交易查询
- 无法聚合、排序、过滤
- 性能差，成本高

The Graph 方案:
                    ┌─────────────────┐
   Events ─────────►│    Indexer      │
                    │  (监听+索引)    │
                    └────────┬────────┘
                             │
                             ▼
                    ┌─────────────────┐
                    │    PostgreSQL   │
                    └────────┬────────┘
                             │
                             ▼
                    ┌─────────────────┐
   GraphQL ────────►│   Query Node    │────► 结果
                    └─────────────────┘
```

### 5.2 Subgraph 定义

### 5.3 Schema 定义

```graphql
# schema.graphql
type Account @entity {
  id: Bytes!
  balance: BigInt!
  transfersFrom: [Transfer!]! @derivedFrom(field: "from")
  transfersTo: [Transfer!]! @derivedFrom(field: "to")
}

type Transfer @entity(immutable: true) {
  id: Bytes!
  from: Account!
  to: Account!
  value: BigInt!
  blockNumber: BigInt!
  timestamp: BigInt!
  transactionHash: Bytes!
}
```

### 5.4 Mapping 处理

```typescript
// src/mapping.ts
import { Transfer as TransferEvent } from "../generated/Token/Token";
import { Transfer, Account } from "../generated/schema";
import { BigInt, Bytes } from "@graphprotocol/graph-ts";

export function handleTransfer(event: TransferEvent): void {
  // 创建 Transfer 实体
  let transfer = new Transfer(
    event.transaction.hash.concatI32(event.logIndex.toI32())
  );
  
  transfer.from = event.params.from;
  transfer.to = event.params.to;
  transfer.value = event.params.value;
  transfer.blockNumber = event.block.number;
  transfer.timestamp = event.block.timestamp;
  transfer.transactionHash = event.transaction.hash;
  transfer.save();
  
  // 更新发送方账户
  let fromAccount = Account.load(event.params.from);
  if (!fromAccount) {
    fromAccount = new Account(event.params.from);
    fromAccount.balance = BigInt.zero();
  }
  fromAccount.balance = fromAccount.balance.minus(event.params.value);
  fromAccount.save();
  
  // 更新接收方账户
  let toAccount = Account.load(event.params.to);
  if (!toAccount) {
    toAccount = new Account(event.params.to);
    toAccount.balance = BigInt.zero();
  }
  toAccount.balance = toAccount.balance.plus(event.params.value);
  toAccount.save();
}
```

## 6. IPFS 去中心化存储

### 6.1 IPFS 基础

```
IPFS 内容寻址
─────────────────────────────────────────────────────────────

传统 URL (位置寻址):
https://example.com/image.png
└─ 内容可能被篡改

IPFS CID (内容寻址):
ipfs://QmXnnyufdzAWL5CqZ2RnSNgPbvCc1ALT73s6epPrRnZ1Xy
└─ CID = Hash(内容)，内容不可篡改

数据结构:
┌─────────────────────────────────────────┐
│  Merkle DAG                             │
│                                         │
│       Root CID                          │
│          │                              │
│    ┌─────┼─────┐                       │
│    ▼     ▼     ▼                       │
│  Block  Block  Block                    │
│    │                                    │
│    ▼                                    │
│  Chunk  Chunk  Chunk                    │
│  (256KB max)                            │
└─────────────────────────────────────────┘
```

// 部署前检查脚本
async function preDeployCheck() {
  // 检查网络
  const network = await ethers.provider.getNetwork();
  console.log(`Network: ${network.name} (${network.chainId})`);
  
  // 检查部署者余额
  const [deployer] = await ethers.getSigners();
  const balance = await ethers.provider.getBalance(deployer.address);
  console.log(`Deployer balance: ${ethers.formatEther(balance)} ETH`);
  
  // 估算 Gas
  const Token = await ethers.getContractFactory("Token");
  const deployTx = await Token.getDeployTransaction(initialSupply);
  const gasEstimate = await ethers.provider.estimateGas(deployTx);
  const gasPrice = (await ethers.provider.getFeeData()).gasPrice;
  const cost = gasEstimate * gasPrice;
  console.log(`Estimated cost: ${ethers.formatEther(cost)} ETH`);
  
  // 确认
  if (balance < cost * 2n) {
    throw new Error("Insufficient balance");
  }
}

### 8.3 升级部署

```javascript
const { upgrades } = require("hardhat");

// 首次部署
async function deployProxy() {
  const TokenV1 = await ethers.getContractFactory("TokenV1");
  const proxy = await upgrades.deployProxy(
    TokenV1,
    [ethers.parseEther("1000000")],
    { kind: "uups" }
  );
  await proxy.waitForDeployment();
  console.log("Proxy deployed:", await proxy.getAddress());
}

// 升级
async function upgradeProxy(proxyAddress) {
  const TokenV2 = await ethers.getContractFactory("TokenV2");
  
  // 验证升级安全性
  await upgrades.validateUpgrade(proxyAddress, TokenV2);
  
  // 执行升级
  const upgraded = await upgrades.upgradeProxy(proxyAddress, TokenV2);
  console.log("Upgraded to V2");
  
  // 验证
  console.log("Version:", await upgraded.version());
}
```

## 面试题自查

**Q1: Hardhat 和 Foundry 的主要区别是什么？各自适用什么场景？**

Hardhat 使用 JavaScript/TypeScript，生态丰富，学习曲线平缓，适合 Web 开发者和需要复杂脚本的项目。Foundry 使用 Solidity 编写测试，编译和测试速度极快，内置 Fuzzing，适合追求效率和深度测试的项目。

**Q2: ethers.js v6 相比 v5 有哪些主要变化？**

v6 的主要变化：(1) BigNumber 改为原生 BigInt；(2) `parseEther/formatEther` 移到 `ethers` 顶层；(3) Provider 改名为 `JsonRpcProvider`；(4) Contract 方法调用方式变化；(5) 整体更轻量，Tree-shaking 友好。

**Q3: viem 相比 ethers.js 有什么优势？**

viem 的优势：(1) TypeScript 优先，类型推导完善；(2) 更小的包体积（Tree-shaking）；(3) 函数式 API，无副作用；(4) 更好的错误处理；(5) 原生支持 wagmi，适合 React。

**Q4: 如何安全地在前端管理用户钱包连接？**

最佳实践：(1) 使用 WalletConnect/Web3Modal 等标准方案；(2) 监听 `accountsChanged` 和 `chainChanged` 事件；(3) 断开时清理状态；(4) 切换网络时提示用户；(5) 不在前端存储私钥；(6) 显示交易确认状态。

**Q5: The Graph 的 Subgraph 工作原理是什么？**

Subgraph 定义了如何索引链上数据：(1) `subgraph.yaml` 指定数据源和处理器；(2) `schema.graphql` 定义数据模型；(3) Mapping（AssemblyScript）处理事件并存入数据库；(4) 通过 GraphQL 查询索引后的数据。

**Q6: IPFS 的 CID 是如何生成的？为什么说内容不可篡改？**

CID = Hash(内容)，使用内容哈希作为地址。如果内容变化，哈希就变化，CID 也变化。获取 CID 对应的内容时会验证哈希，篡改内容无法通过验证。

**Q7: 合约 Fuzz Testing 的目的是什么？Foundry 如何实现？**

Fuzz Testing 用随机输入测试合约边界条件，发现手写测试难以覆盖的漏洞。Foundry 在测试函数中添加参数，运行器自动生成随机输入，使用 `bound()` 限制范围，`vm.assume()` 过滤无效输入。

**Q8: 列举几种常见的合约安全漏洞及其防护方法。**

(1) 重入攻击：Checks-Effects-Interactions 模式 + ReentrancyGuard；(2) 整数溢出：Solidity 0.8+ 默认检查；(3) 闪电贷攻击：使用 TWAP 预言机；(4) 权限绕过：OpenZeppelin AccessControl；(5) 前端运行：commit-reveal 模式。

**Q9: UUPS 和 Transparent Proxy 升级模式的区别是什么？**

UUPS：升级逻辑在实现合约中，Gas 更省，但实现合约必须包含升级函数。Transparent Proxy：升级逻辑在 Proxy 中，admin 调用走 Proxy 逻辑，用户调用走实现逻辑，避免函数选择器冲突但 Gas 稍高。

**Q10: 如何确保合约升级不会破坏存储布局？**

(1) 使用 OpenZeppelin Upgrades 插件自动检查；(2) 新版本只能在末尾添加状态变量；(3) 不能改变已有变量的顺序或类型；(4) 使用 Gap 预留存储槽；(5) 继承顺序保持一致。

**Q11: 什么是 wagmi？它解决了什么问题？**

wagmi 是 React Hooks 库，封装了 Web3 常用操作（连接钱包、读写合约、监听事件等）。解决了：(1) 状态管理；(2) 缓存和重新验证；(3) 类型安全；(4) 多链支持；(5) 与主流钱包集成。

**Q12: 如何处理前端与合约交互时的 Gas 估算和错误？**

Gas 估算：使用 `estimateGas()` 获取预估值，乘以安全系数（1.1-1.2）。错误处理：捕获交易 revert 原因，解析 custom error，显示友好提示。设置合理的 Gas Limit 上限，避免意外消耗过多 Gas。

**Q13: Slither、Mythril、Echidna 分别用于什么类型的安全分析？**

Slither：静态分析，快速检测常见漏洞模式。Mythril：符号执行，深度分析执行路径。Echidna：Fuzzing，用大量随机输入测试不变量。三者互补，应结合使用。

**Q14: 如何设计一个安全的多签钱包合约？**

关键设计：(1) 门槛设置（如 3/5）；(2) 交易提案和确认分离；(3) Nonce 防重放；(4) 时间锁可选；(5) 签名者增删需多签；(6) 紧急暂停机制；(7) 事件记录所有操作。

**Q15: Web3 项目的典型技术栈是什么？**

合约层：Solidity + Hardhat/Foundry + OpenZeppelin。前端：React/Next.js + wagmi + viem。数据：The Graph（索引）+ IPFS（存储）。钱包：WalletConnect + Web3Modal。监控：Tenderly/Forta。
