## 开始前先看

### 这篇解决什么问题

以太坊学习里最容易被误判为“语法问题”的部分，其实是运行约束。这里先建立下面几件事：

- 合约运行在什么环境里
- 为什么状态写入这么贵
- 为什么合约之间的调用方式会影响安全
- 为什么升级模式、权限控制、Gas 优化在链上是基础题

如果这些约束没有先建立，后面的代码示例就容易变成“会抄，不会判断”。

以太坊不是“把后端代码搬上链”，而是在一个公开、确定性、按资源收费的虚拟机里执行程序。

这意味着你写的每一行状态修改代码，都同时受到：

- 执行成本约束
- 安全约束
- 可升级性约束
- 公共可见性约束

## 1. 先建立以太坊心智模型

### 1.0 以太坊可以先理解成什么

可以先把以太坊理解成三层：

```text
用户与钱包层
  ↓
交易与账户层
  ↓
EVM 与合约执行层
```

用户通过钱包签名交易，交易改变账户状态，合约代码在 EVM 里执行并修改 storage。

### 1.0.1 为什么以太坊和比特币差别这么大

比特币更像“只为价值转移设计的账本系统”，而以太坊进一步把“可编程状态机”放进了链上。

于是以太坊不仅能转账，还能：

- 发代币
- 写交易所
- 写借贷协议
- 写 NFT、DAO、链上治理系统

但代价是：

- 执行环境更复杂
- 攻击面更大
- Gas 成本和状态管理更关键

### 1.0.2 学这篇时先带着 4 个问题

1. 合约代码运行时，到底在读写什么？
2. 为什么 `storage` 比 `memory` 贵这么多？
3. 为什么 `delegatecall` 既强大又危险？
4. 为什么合约升级在链上不是可有可无，而是架构级问题？

带着这 4 个问题往下读，后面的 EVM、Gas、代理、安全章节会更容易串起来。

### 1.1 先建立账户、状态、Gas 的关系

以太坊入门阶段常见的问题是：词都认识，但不知道它们怎么连起来。先看最小关系图：

```text
用户钱包
  ↓ 用私钥控制
EOA 账户
  ↓ 发起交易
交易
  ↓ 进入 EVM 执行
合约代码
  ↓ 读写
memory / storage / logs
  ↓ 消耗
gas
  ↓ 导致
状态变化 + 手续费支出
```

这张图里最重要的是两个认识：

1. 以太坊不是“调用接口”，而是“提交一笔会修改全局状态的交易”。
2. 每次状态修改都不是免费的，它会消耗 Gas。

这两个点站稳之后，后面的 EVM、Storage、Opcode、代理模式才有上下文。

### 1.2 为什么智能合约不是普通后端代码

如果你有后端开发背景，最容易犯的错是把 Solidity 合约理解成“部署在链上的服务逻辑”。

这不准确。它和普通后端代码至少有 5 个根本差别：

| 维度 | 普通后端 | 智能合约 |
|------|----------|----------|
| 执行环境 | 自己控制的服务器 | 全网验证的公开虚拟机 |
| 状态修改成本 | 通常近似忽略 | 每次写状态都要付 Gas |
| 可见性 | 内部系统可隐藏 | 代码和状态通常公开可检查 |
| 升级方式 | 可直接发版替换 | 升级复杂，常需代理模式 |
| Bug 后果 | 可热修、可回滚 | 可能直接造成资产损失 |

所以智能合约开发的核心不是“功能写出来”，而是同时约束：

- 安全性
- Gas 成本
- 权限边界
- 升级策略
- 与其他合约交互的风险

如果 Web3 文档只讲 Solidity 语法，阅读体验会越来越割裂，因为真正难的是执行模型，不是语法。

## 2. EVM 执行模型

### 2.1 EVM 概述

**EVM（Ethereum Virtual Machine）**：以太坊虚拟机，执行智能合约字节码。

```
EVM 特点：
- 栈式虚拟机（Stack-based）
- 256 位字长（适合密码学运算）
- 确定性执行（相同输入必定相同输出）
- 沙盒隔离（合约间通过消息调用交互）
```

### 2.2 EVM 架构

```
EVM 执行环境：

┌─────────────────────────────────────────────────────────────┐
│                         EVM                                 │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐ │
│  │    Stack    │  │   Memory    │  │      Storage        │ │
│  │  (1024深度) │  │  (线性字节) │  │  (持久化键值对)     │ │
│  │   256位元素 │  │   按字节寻址 │  │   256位→256位       │ │
│  └─────────────┘  └─────────────┘  └─────────────────────┘ │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                Program Counter (PC)                  │   │
│  │                当前执行的字节码位置                   │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                    Gas Counter                       │   │
│  │                剩余 Gas / 已消耗 Gas                  │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘

存储成本：
- Stack: 免费（操作本身有Gas成本）
- Memory: 便宜，线性增长收费
- Storage: 昂贵，20000 gas/新slot，5000 gas/修改
```

### 2.3 常用操作码

| 操作码 | 名称 | Gas | 说明 |
|--------|------|-----|------|
| ADD | 加法 | 3 | 栈顶两元素相加 |
| MUL | 乘法 | 5 | 栈顶两元素相乘 |
| MLOAD | 内存读 | 3 | 读取32字节 |
| MSTORE | 内存写 | 3 | 写入32字节 |
| SLOAD | 存储读 | 2100 | 冷读取/100热读取 |
| SSTORE | 存储写 | 20000 | 0→非0/2900修改 |
| CALL | 调用 | 100+ | 调用其他合约 |
| CREATE | 创建 | 32000 | 部署新合约 |

### 2.4 调用类型

```
合约间调用方式：

1. CALL：标准调用
   - 切换上下文到被调用合约
   - msg.sender = 调用者
   - 可以发送 ETH

2. DELEGATECALL：委托调用
   - 在调用者上下文执行被调用合约代码
   - msg.sender = 原始调用者
   - 存储操作影响调用者
   - 用于代理模式

3. STATICCALL：静态调用
   - 只读调用，不允许状态修改
   - 用于 view/pure 函数

┌─────────────┐    CALL     ┌─────────────┐
│  Contract A │ ──────────► │  Contract B │
│  storage A  │             │  storage B  │
└─────────────┘             └─────────────┘
    执行 B 代码，修改 B 存储

┌─────────────┐ DELEGATECALL┌─────────────┐
│  Contract A │ ──────────► │  Contract B │
│  storage A  │             │   (代码)    │
└─────────────┘             └─────────────┘
    执行 B 代码，修改 A 存储
```

## 3. 以太坊状态树

### 3.1 Modified Merkle Patricia Trie (MPT)

```
以太坊状态存储结构：

World State Trie（世界状态树）
├── Key: 账户地址 (Keccak256 Hash)
└── Value: 账户状态 (nonce, balance, storageRoot, codeHash)

Account Storage Trie（账户存储树）
├── Key: 存储槽位置
└── Value: 存储值

Transaction Trie（交易树）
├── Key: 交易索引
└── Value: 交易数据

Receipt Trie（收据树）
├── Key: 交易索引
└── Value: 执行结果、日志
```

### 3.2 状态根与区块头

```
区块头包含三个重要的根哈希：

Block Header
├── stateRoot      → World State Trie 的根
├── transactionsRoot → Transaction Trie 的根
└── receiptsRoot   → Receipt Trie 的根

作用：
- 轻节点可通过 Merkle Proof 验证状态
- 每个区块是完整的状态快照
```

## 4. 交易与 Gas

### 4.1 交易结构

```
以太坊交易（EIP-1559后）：

┌─────────────────────────────────────────────────────────────┐
│  Transaction                                                │
├─────────────────────────────────────────────────────────────┤
│  type           : 2 (EIP-1559)                             │
│  chainId        : 1 (Mainnet)                              │
│  nonce          : 发送者交易序号                            │
│  maxPriorityFeePerGas : 矿工小费上限                       │
│  maxFeePerGas   : 总费用上限（base + priority）            │
│  gasLimit       : Gas 上限                                 │
│  to             : 接收地址（合约创建时为null）              │
│  value          : 转账 ETH 金额（wei）                     │
│  data           : 调用数据 / 合约字节码                    │
│  accessList     : 预访问的地址和存储槽                     │
│  v, r, s        : ECDSA 签名                               │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 EIP-1559 费用机制

```
EIP-1559 Gas 费用：

实际费用 = gasUsed × (baseFee + priorityFee)

其中：
- baseFee：协议计算，根据区块使用率动态调整，会被销毁
- priorityFee：用户设置的小费，给验证者

baseFee 调整：
- 区块 Gas 使用 > 目标（15M）：baseFee 上涨（最多 12.5%）
- 区块 Gas 使用 < 目标（15M）：baseFee 下降（最多 12.5%）

用户设置：
- maxFeePerGas：愿意支付的最高单价
- maxPriorityFeePerGas：愿意给验证者的小费
- 实际 priorityFee = min(maxPriorityFee, maxFee - baseFee)
```

```solidity
    }
```

### 6.1 存储布局

```solidity
contract StorageLayout {

```

### 6.2 Gas 优化技巧

```solidity
contract GasOptimization {
    // 1. 变量打包
    // Bad: 3 个存储槽
    uint256 a;
    uint8 b;
    uint256 c;
    
    // Good: 2 个存储槽
    uint256 a;
    uint256 c;
    uint8 b;
    
    // 2. 使用 calldata 而非 memory
    function process(uint256[] calldata data) external {
        // calldata 更便宜，但只读
    }
    
    // 3. 缓存存储变量
    uint256 public counter;
    
    function badIncrement(uint256 times) external {
        for (uint i = 0; i < times; i++) {
            counter++;  // 每次循环都读写存储！
        }
    }
    
    function goodIncrement(uint256 times) external {
        uint256 _counter = counter;  // 缓存到内存
        for (uint i = 0; i < times; i++) {
            _counter++;
        }
        counter = _counter;  // 只写一次存储
    }
    
    // 4. 短路求值
    function check(bool a, bool b) external view returns (bool) {
        return a && b;  // 如果 a 为 false，不会计算 b
    }
    
    // 5. 使用 unchecked（确认安全时）
    function sum(uint256[] calldata arr) external pure returns (uint256) {
        uint256 total;
        for (uint i = 0; i < arr.length;) {
            unchecked {
                total += arr[i];
                i++;
            }
        }
        return total;
    }
}
```

## 7. 合约设计模式

### 7.1 代理模式（可升级合约）

```solidity
// 代理合约（不变）
contract Proxy {
    address public implementation;
    address public admin;
    
    constructor(address _impl) {
        implementation = _impl;
        admin = msg.sender;
    }
    
    function upgrade(address newImpl) external {
        require(msg.sender == admin);
        implementation = newImpl;
    }
    
    fallback() external payable {
        address impl = implementation;
        assembly {
            calldatacopy(0, 0, calldatasize())
            let result := delegatecall(gas(), impl, 0, calldatasize(), 0, 0)
            returndatacopy(0, 0, returndatasize())
            switch result
            case 0 { revert(0, returndatasize()) }
            default { return(0, returndatasize()) }
        }
    }
}

// 实现合约 V1
contract ImplementationV1 {
    uint256 public value;
    
    function setValue(uint256 _value) external {
        value = _value;
    }
}

// 实现合约 V2（升级后）
contract ImplementationV2 {
    uint256 public value;
    
    function setValue(uint256 _value) external {
        value = _value * 2;  // 新逻辑
    }
    
    function newFunction() external pure returns (string memory) {
        return "V2";
    }
}
```

## 8. 智能合约安全

### 8.1 重入攻击

```solidity
// 漏洞合约
contract Vulnerable {
    mapping(address => uint256) public balances;
    
    function withdraw() external {
        uint256 amount = balances[msg.sender];
        (bool success, ) = msg.sender.call{value: amount}("");
        require(success);
        balances[msg.sender] = 0;  // 在外部调用之后！
    }
}

// 攻击合约
contract Attacker {
    Vulnerable target;
    
    function attack() external payable {
        target.withdraw();
    }
    
    receive() external payable {
        if (address(target).balance > 0) {
            target.withdraw();  // 重复调用
        }
    }
}

// 修复方案：检查-生效-交互模式
contract Fixed {
    mapping(address => uint256) public balances;
    
    function withdraw() external {
        uint256 amount = balances[msg.sender];
        balances[msg.sender] = 0;  // 先修改状态
        (bool success, ) = msg.sender.call{value: amount}("");
        require(success);
    }
}
```

### 8.2 其他常见漏洞

| 漏洞类型 | 说明 | 防护 |
|---------|------|------|
| 整数溢出 | Solidity 0.8 前无检查 | 使用 0.8+ 或 SafeMath |
| 闪电贷攻击 | 单交易内操纵价格 | 使用 TWAP 预言机 |
| 前端运行 | MEV 机器人抢跑 | Commit-Reveal、私有内存池 |
| 拒绝服务 | 循环遍历大数组 | 限制数组大小、分页 |
| 权限控制 | 敏感函数无权限检查 | OpenZeppelin AccessControl |

## 9. ERC 标准

### 9.1 ERC-20（同质化代币）

```solidity
interface IERC20 {
    function totalSupply() external view returns (uint256);
    function balanceOf(address account) external view returns (uint256);
    function transfer(address to, uint256 amount) external returns (bool);
```

## 10. 面试题自查

**Q1: EVM 的三种数据存储位置有什么区别？**

Stack：免费但深度限制1024，存储临时运算数据；Memory：按需扩展的线性字节数组，交易结束后清除；Storage：持久化的256位键值存储，最昂贵（20000 gas 写新槽）。

**Q2: CALL 和 DELEGATECALL 的区别是什么？**

CALL 在被调用合约上下文执行，修改被调用合约存储；DELEGATECALL 在调用者上下文执行被调用合约代码，修改调用者存储，msg.sender 保持不变。代理模式基于 DELEGATECALL。

**Q3: 什么是重入攻击？如何防护？**

攻击者合约在外部调用（如 call 发送 ETH）的回调中重复调用目标函数，在状态更新前多次提取资金。防护：检查-生效-交互模式（先更新状态再外部调用）、使用 ReentrancyGuard。

**Q4: Solidity 存储变量如何打包？**

每个存储槽 32 字节，小于 32 字节的变量会顺序打包到同一槽。优化建议：将相同大小的小变量放在一起声明，减少存储槽使用。

**Q5: EIP-1559 的 baseFee 和 priorityFee 是什么？**

baseFee 是协议计算的基础费用，根据区块使用率动态调整，会被销毁；priorityFee 是用户给验证者的小费。实际费用 = gasUsed × (baseFee + priorityFee)。

**Q6: 什么是闪电贷？有什么风险？**

单笔交易内借款并归还的无抵押贷款。风险：攻击者可用大量资金在单交易内操纵价格预言机，套利攻击 DeFi 协议。防护：使用 TWAP 时间加权平均价格。

**Q7: ERC-20 的 approve + transferFrom 模式有什么问题？**

两步操作用户体验差；批准后如果改变批准额度可能被前端运行攻击（先使用旧额度再使用新额度）。改进：ERC-2612 permit（签名批准）。

**Q8: 代理合约升级有什么存储冲突风险？**

新实现合约的存储布局必须兼容旧版本，不能改变已有变量顺序或删除变量，只能在末尾添加。否则存储槽会错位导致数据损坏。

**Q9: view 和 pure 函数的区别？**

view 函数可以读取状态但不能修改；pure 函数不能读取也不能修改任何状态，只能使用参数进行计算。两者都不消耗 gas（外部调用时）。

**Q10: 什么是以太坊的 MPT（Merkle Patricia Trie）？**

Modified Merkle Patricia Trie 是以太坊状态存储结构，结合了 Merkle Tree（验证完整性）和 Patricia Trie（键值存储）。状态根存储在区块头，支持轻节点验证。

**Q11: ERC-721 和 ERC-1155 的主要区别？**

ERC-721 每个代币唯一，一个合约管理一类 NFT；ERC-1155 支持多种代币（FT+NFT）在同一合约，批量转账更省 gas。

**Q12: 如何防止整数溢出？**

Solidity 0.8+ 默认有溢出检查；0.8 之前使用 SafeMath 库。需要节省 gas 且确认安全时可用 unchecked 块禁用检查。

**Q13: 什么是 MEV？合约如何缓解前端运行？**

MEV（Maximal Extractable Value）是验证者/搜索者通过重排交易获取的价值。缓解：Commit-Reveal 模式、使用 Flashbots 私有内存池、设置滑点保护。

**Q14: Gas 优化有哪些常用技巧？**

变量打包、用 calldata 替代 memory、缓存存储变量到内存、短路求值、unchecked 数学运算、用 custom error 替代 require 字符串。

**Q15: 智能合约的权限控制有哪些方案？**

简单方案：Ownable（单一 owner）；复杂方案：AccessControl（角色权限）、多签（Gnosis Safe）、时间锁（Timelock）延迟执行敏感操作。
