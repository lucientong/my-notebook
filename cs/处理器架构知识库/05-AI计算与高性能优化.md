## 1. 矩阵运算与 GPU

### 1.1 深度学习计算本质

神经网络核心运算是矩阵乘法（GEMM）：

```
Y = X × W + b

X: [Batch, InputDim]    输入
W: [InputDim, OutputDim] 权重
Y: [Batch, OutputDim]    输出

计算量: 2 × B × I × O FLOPs
```

**为什么 GPU 适合深度学习**：

| 特性 | CPU | GPU |
|------|-----|-----|
| 核心数 | 8-64 | 数千个 |
| 设计目标 | 低延迟 | 高吞吐 |
| 内存带宽 | ~100 GB/s | ~2 TB/s (HBM) |
| 适合任务 | 串行复杂逻辑 | 大规模并行计算 |

### 1.2 Roofline 模型

```
性能分析
─────────────────────────────────────────────────────────────

           性能 (TFLOPS)
               │
峰值算力 ──────┼────────────────────────────
               │              ╱
               │            ╱
               │          ╱   计算密集区
               │        ╱
               │      ╱
               │    ╱
               │  ╱   带宽受限区
               │╱
               └──────────────────────────── 算术强度
                        脊点

A100 参数:
- FP16 峰值: 312 TFLOPS
- 带宽: 2 TB/s
- 脊点: 312/2 = 156 FLOPs/Byte
```

## 2. 混合精度训练

### 2.1 浮点格式对比

| 格式 | 位宽 | 指数位 | 尾数位 | 范围 | 精度 |
|------|------|--------|--------|------|------|
| FP32 | 32 | 8 | 23 | ±3.4×10³⁸ | 7位 |
| FP16 | 16 | 5 | 10 | ±6.5×10⁴ | 3位 |
| BF16 | 16 | 8 | 7 | ±3.4×10³⁸ | 2位 |
| TF32 | 19 | 8 | 10 | ±3.4×10³⁸ | 3位 |
| FP8 (E4M3) | 8 | 4 | 3 | ±240 | 1位 |

### 2.2 混合精度流程

```
┌─────────────────┐
│  FP32 主权重    │◄───────────────────────┐
└────────┬────────┘                        │
         │ 转换                            │
         ▼                                 │
┌─────────────────┐                        │
│  FP16 权重副本  │                        │
└────────┬────────┘                        │
         │                                 │
         ▼                                 │
┌─────────────────┐    ┌───────────────┐   │
│  FP16 前向传播  │───►│ Loss Scaling  │   │
└─────────────────┘    └───────┬───────┘   │
                               │           │
                               ▼           │
                       ┌───────────────┐   │
                       │ FP16 反向传播 │   │
                       └───────┬───────┘   │
                               │           │
                               ▼           │
                       ┌───────────────┐   │
                       │ FP32 梯度累加 │───┘
                       │ + 权重更新    │
                       └───────────────┘
```

### 2.3 PyTorch AMP 使用

```python
from torch.cuda.amp import autocast, GradScaler

scaler = GradScaler()

for data, target in dataloader:
    optimizer.zero_grad()
    
    # 自动混合精度
    with autocast():
        output = model(data)
        loss = criterion(output, target)
    
    # 缩放 loss 并反向传播
    scaler.scale(loss).backward()
    
    # 梯度裁剪（需要先 unscale）
    scaler.unscale_(optimizer)
    torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
    
    # 更新权重
    scaler.step(optimizer)
    scaler.update()
```

Tensor Core 单次操作 (Ampere)

## 4. 数据并行与 AllReduce

### 4.1 数据并行原理

```
数据并行 (DP)
─────────────────────────────────────────────────────────────

     完整数据集
         │
    ┌────┼────┐
    │    │    │
    ▼    ▼    ▼
 Batch0 Batch1 Batch2    各 GPU 分配不同 mini-batch
    │    │    │
    ▼    ▼    ▼
 ┌─────┬─────┬─────┐
 │GPU0 │GPU1 │GPU2 │    每个 GPU 持有完整模型副本
 │完整 │完整 │完整 │
 │模型 │模型 │模型 │
 └──┬──┴──┬──┴──┬──┘
    │     │     │
    └─────┼─────┘
          │
    AllReduce 同步梯度
          │
    ┌─────┼─────┐
    │     │     │
    ▼     ▼     ▼
  权重更新（所有 GPU 保持一致）
```

### 4.2 Ring-AllReduce

```
Ring-AllReduce 算法 (4 GPUs)
─────────────────────────────────────────────────────────────

GPU 排列成环: 0 → 1 → 2 → 3 → 0

Step 1: Reduce-Scatter (3轮)
每轮每个 GPU 发送 1/4 数据给下一个

轮1: GPU0 发 chunk0 给 GPU1
     GPU1 发 chunk1 给 GPU2
     ...
     
轮2: 继续传递并累加
轮3: 完成累加

结果: 每个 GPU 持有完整结果的 1/4

Step 2: All-Gather (3轮)
将完整结果分发给所有 GPU

通信量: 2 × (N-1)/N × DataSize ≈ 2 × DataSize
```

### 4.3 NCCL 通信

```python
import torch.distributed as dist

# 初始化进程组
dist.init_process_group(backend='nccl')

# AllReduce 操作
tensor = torch.randn(1024, 1024).cuda()
dist.all_reduce(tensor, op=dist.ReduceOp.SUM)
tensor /= dist.get_world_size()  # 求平均
```

## 5. 模型并行与张量并行

### 5.1 模型并行类型

```
模型并行分类
─────────────────────────────────────────────────────────────

1. 朴素模型并行 (Layer-wise)
   GPU0: Layer 1-4
   GPU1: Layer 5-8
   问题: GPU 利用率低（顺序执行）

2. 张量并行 (Tensor Parallel)
   单层内部切分：
   GPU0: W[:, 0:N/2]
   GPU1: W[:, N/2:N]
   优点: 层内并行，延迟低

3. 专家并行 (Expert Parallel)
   MoE 模型：不同专家放不同 GPU
```

### 5.2 张量并行实现

```
MLP 张量并行切分
─────────────────────────────────────────────────────────────

原始 MLP:
Y = GeLU(X × W1) × W2

张量并行:
        X
        │
   ┌────┴────┐
   │         │
   ▼         ▼
W1_col0   W1_col1      列切分
   │         │
   ▼         ▼
 GeLU      GeLU
   │         │
   ▼         ▼
W2_row0   W2_row1      行切分
   │         │
   └────┬────┘
        │
   AllReduce
        │
        ▼
        Y
```

**代码实现**：

```python
class ColumnParallelLinear(nn.Module):
    """列并行: 权重按列切分"""
    def __init__(self, in_features, out_features, world_size, rank):
        super().__init__()
        self.out_features_per_gpu = out_features // world_size
        self.weight = nn.Parameter(
            torch.randn(self.out_features_per_gpu, in_features)
        )
    
    def forward(self, x):
        # 每个 GPU 计算部分输出
        return F.linear(x, self.weight)

class RowParallelLinear(nn.Module):
    """行并行: 权重按行切分"""
    def __init__(self, in_features, out_features, world_size, rank):
        super().__init__()
        self.in_features_per_gpu = in_features // world_size
        self.weight = nn.Parameter(
            torch.randn(out_features, self.in_features_per_gpu)
        )
    
    def forward(self, x):
        output = F.linear(x, self.weight)
        # AllReduce 聚合结果
        dist.all_reduce(output)
        return output
```

## 6. 流水线并行与 3D 并行

### 6.1 流水线并行

```
流水线并行 (4 stages, 4 micro-batches)
─────────────────────────────────────────────────────────────

时间 →
      t0   t1   t2   t3   t4   t5   t6   t7
GPU0 │ F0 │ F1 │ F2 │ F3 │ B3 │ B2 │ B1 │ B0 │
GPU1 │    │ F0 │ F1 │ F2 │ F3 │ B3 │ B2 │ B1 │
GPU2 │    │    │ F0 │ F1 │ F2 │ F3 │ B3 │ B2 │
GPU3 │    │    │    │ F0 │ F1 │ F2 │ F3 │ B3 │

F = Forward, B = Backward
气泡 (idle): 开头和结尾的空闲时间
气泡比例: (P-1) / (M + P - 1)
P = 流水线 stages, M = micro-batches
```

### 6.2 1F1B 调度

```
1F1B: One Forward One Backward
─────────────────────────────────────────────────────────────

     t0   t1   t2   t3   t4   t5   t6   t7   t8
GPU0 │ F0 │ F1 │ F2 │ F3 │B3F4│B2F5│B1F6│B0F7│ B7 │
GPU1 │    │ F0 │ F1 │ F2 │B2F3│B1F4│B0F5│ B4 │ B5 │
...

优点:
1. 减少 activation 内存占用
2. 更稳定的内存使用模式
```

### 6.3 3D 并行

```
3D 并行 = 数据并行 × 张量并行 × 流水线并行
─────────────────────────────────────────────────────────────

                    数据并行 (DP=2)
                   ╱              ╲
              Replica 0        Replica 1
                 │                 │
        ┌────────┴────────┐       ...
        │                 │
   流水线并行 (PP=4)     流水线并行
        │
 ┌──────┼──────┬──────┐
 │      │      │      │
Stage0 Stage1 Stage2 Stage3
 │      │      │      │
 └──────┴──────┴──────┘
        │
   张量并行 (TP=2)
   每个 Stage 内部横向切分

总 GPU 数 = DP × PP × TP = 2 × 4 × 2 = 16
```

## 7. ZeRO 内存优化

### 7.1 训练内存分析

```
模型训练内存组成 (10B 参数, FP16+Adam)
─────────────────────────────────────────────────────────────

参数 (FP16):        10B × 2 bytes = 20 GB
梯度 (FP16):        10B × 2 bytes = 20 GB
优化器状态 (Adam):
  - FP32 参数副本:  10B × 4 bytes = 40 GB
  - FP32 动量:      10B × 4 bytes = 40 GB
  - FP32 方差:      10B × 4 bytes = 40 GB
────────────────────────────────────────────
总计:               160 GB (单卡无法容纳)
```

### 7.2 ZeRO 三阶段

```
ZeRO 优化
─────────────────────────────────────────────────────────────

ZeRO-1: 优化器状态切分
        每个 GPU 只保存 1/N 的优化器状态
        内存: 4Ψ + 12Ψ/N (Ψ=参数量, N=GPU数)

ZeRO-2: + 梯度切分
        每个 GPU 只保存 1/N 的梯度
        内存: 2Ψ + (2+12)Ψ/N

ZeRO-3: + 参数切分
        每个 GPU 只保存 1/N 的参数
        内存: 16Ψ/N

示例 (10B 参数, 8 GPU):
┌─────────┬───────────┬───────────┬───────────┐
│         │ 基线      │ ZeRO-2    │ ZeRO-3    │
├─────────┼───────────┼───────────┼───────────┤
│ 单卡内存│ 160 GB    │ 37.5 GB   │ 20 GB     │
└─────────┴───────────┴───────────┴───────────┘
```

### 7.3 DeepSpeed ZeRO 使用

```python

```
TensorRT 图优化
─────────────────────────────────────────────────────────────

## 9. 量化原理与实践

### 9.1 量化基础

```
量化映射
─────────────────────────────────────────────────────────────

对称量化:
x_q = round(x / scale)
x = x_q × scale

scale = max(|x|) / 127  (INT8)

非对称量化:
x_q = round(x / scale) + zero_point
x = (x_q - zero_point) × scale

scale = (max - min) / 255
zero_point = round(-min / scale)
```

### 9.2 量化方式对比

| 方式 | 精度 | 速度 | 适用场景 |
|------|------|------|---------|
| FP16 | 高 | 2x | 通用加速 |
| INT8 | 中 | 2-4x | CNN/Transformer |
| INT4 | 低 | 4-8x | 大模型推理 |
| GPTQ | 中 | 4x | LLM 权重量化 |
| AWQ | 中高 | 4x | LLM 激活感知量化 |

### 9.3 GPTQ 量化

## 10. 大模型推理优化

### 10.1 KV Cache

```
KV Cache 原理
─────────────────────────────────────────────────────────────

自回归生成:
Token 1 → Token 2 → Token 3 → ...

不使用 Cache:
生成 Token N 时重新计算所有 K,V
计算量: O(N²)

使用 KV Cache:
缓存之前的 K,V，只计算新 Token
计算量: O(N)

内存占用:
KV Cache = 2 × num_layers × seq_len × hidden_dim × batch × dtype
```

### 10.2 PagedAttention (vLLM)

```
PagedAttention 内存管理
─────────────────────────────────────────────────────────────

传统方式:
每个请求预分配最大长度的 KV Cache
大量内存浪费

PagedAttention:
┌─────────────────────────────────────────────────────┐
│  物理内存 (KV Blocks)                               │
│  ┌─────┬─────┬─────┬─────┬─────┬─────┐            │
│  │ B0  │ B1  │ B2  │ B3  │ B4  │ B5  │            │
│  └─────┴─────┴─────┴─────┴─────┴─────┘            │
│      ▲     ▲     ▲     ▲                          │
│      │     │     │     │                          │
│  ┌───┴─────┴─────┴─────┴───┐                      │
│  │    Block Table           │                      │
│  │  Req1: [B0, B1, B3]     │                      │
│  │  Req2: [B2, B4]         │                      │
│  └─────────────────────────┘                      │
│                                                    │
│  - 按需分配 Block                                  │
│  - 非连续存储                                      │
│  - 接近 100% 内存利用率                           │
└─────────────────────────────────────────────────────┘
```

### 10.3 推测解码

```
Speculative Decoding
─────────────────────────────────────────────────────────────

传统自回归: 每次生成 1 个 token，顺序执行

推测解码:
1. 小模型(Draft)快速生成 K 个候选 token
2. 大模型(Target)并行验证 K 个 token
3. 接受正确的前缀，拒绝后重新生成

     Draft Model          Target Model
         │                     │
    生成 [t1,t2,t3,t4]        │
         │                     │
         └──────────────────► 验证
                               │
                          接受 [t1,t2,t3]
                          拒绝 t4
                               │
                          返回正确 token

加速比: 2-3x (取决于 Draft 准确率)
```

### 10.4 FlashAttention

```
FlashAttention 原理
─────────────────────────────────────────────────────────────

标准 Attention:
S = Q × K^T         O(N²) 内存
P = softmax(S)      O(N²) 内存  
O = P × V           O(N²) 内存
总内存: 3 × N² × d

FlashAttention:
分块计算，避免存储完整中间结果
┌─────────────────────────────────────────────────────┐
│  for each Q_block:                                  │
│      for each K_block, V_block:                    │
│          计算部分 S, P, O                          │
│          增量更新 softmax 统计量                   │
│          累加到输出                                │
└─────────────────────────────────────────────────────┘

总内存: O(N) 
IO 复杂度: O(N²d / M)  (M = SRAM 大小)
```

## 面试题自查

**Q1: 为什么深度学习训练适合在 GPU 上运行？**

深度学习核心是矩阵乘法（GEMM），具有高度并行性。GPU 有数千个核心，高内存带宽（~2 TB/s），适合大规模并行计算。相比 CPU，GPU 在矩阵运算上快 10-100 倍。

**Q2: 解释混合精度训练的原理和 Loss Scaling 的作用。**

混合精度使用 FP16 进行前向/反向传播（计算快、省内存），FP32 存储主权重和优化器状态（保证精度）。Loss Scaling 将 loss 放大后再反向传播，避免 FP16 梯度下溢为 0，更新时再缩放回来。

**Q3: Tensor Core 与普通 CUDA Core 有什么区别？**

CUDA Core 是标量/向量计算单元，Tensor Core 是矩阵计算单元。Tensor Core 单条指令完成 16×16×16 矩阵乘累加，比 CUDA Core 快一个数量级。A100 的 FP16 Tensor Core 峰值 312 TFLOPS，而 CUDA Core FP16 峰值 78 TFLOPS。

**Q4: 数据并行如何保证各 GPU 权重一致？**

通过 AllReduce 同步梯度：每个 GPU 计算本地梯度后，使用 Ring-AllReduce 算法求和/平均，确保所有 GPU 获得相同的聚合梯度，然后各自更新权重（更新逻辑相同，权重保持一致）。

**Q5: Ring-AllReduce 的通信量是多少？为什么比 Parameter Server 更高效？**

Ring-AllReduce 通信量为 2×(N-1)/N×DataSize ≈ 2×DataSize，与 GPU 数量无关。Parameter Server 通信量为 2×N×DataSize，随 GPU 数量线性增长。Ring-AllReduce 还避免了中心瓶颈。

**Q6: 张量并行和流水线并行的区别是什么？**

张量并行在层内切分权重，多 GPU 并行计算同一层，延迟低但通信频繁；流水线并行将不同层放不同 GPU，减少通信但有气泡（空闲时间）。大模型通常结合使用（3D 并行）。

**Q7: ZeRO-3 如何实现参数分片？前向传播时如何获取完整参数？**

ZeRO-3 将参数按 GPU 数量均分，每个 GPU 只存 1/N。前向传播时，通过 AllGather 从其他 GPU 收集需要的参数，计算完成后释放；反向传播同理。用通信换内存。

**Q8: TensorRT 的主要优化技术有哪些？**

(1) 层融合：Conv+BN+ReLU 融合为单个 Kernel；(2) 常量折叠：编译时计算常量；(3) 精度转换：FP32→FP16/INT8；(4) Kernel 自动调优：选择最优实现；(5) 动态形状优化。

**Q9: INT8 量化如何计算 scale 和 zero_point？**

对称量化：scale = max(|x|) / 127；非对称量化：scale = (max-min) / 255，zero_point = round(-min / scale)。校准数据用于确定 min/max 范围。

**Q10: 什么是 KV Cache？为什么大模型推理必须使用它？**

KV Cache 缓存 Transformer 自回归生成过程中已计算的 Key 和 Value。不使用 Cache，每生成一个 token 需重新计算所有 K/V，计算量 O(N²)；使用 Cache 只需计算新 token 的 K/V，计算量 O(N)。

**Q11: PagedAttention 解决了什么问题？**

传统 KV Cache 为每个请求预分配最大序列长度内存，造成大量浪费。PagedAttention 借鉴 OS 虚拟内存，将 KV Cache 分成小块按需分配，支持非连续存储，内存利用率接近 100%。

**Q12: FlashAttention 如何减少内存占用？**

标准 Attention 需存储 N²×d 的中间矩阵（Q×K^T 和 softmax）。FlashAttention 分块计算，利用增量 softmax 算法，只需 O(N) 内存存储累积统计量，不需要物化完整的 N² 矩阵。

**Q13: 推测解码（Speculative Decoding）的原理是什么？**

用小模型（Draft）快速生成多个候选 token，大模型（Target）并行验证。若 Draft 准确率高，可一次验证多个 token，将串行生成变为半并行，加速 2-3x。验证失败时拒绝后续 token 并重新生成。

**Q14: 描述 DGX H100 的内部互联架构。**

DGX H100 有 8 个 H100 GPU，通过 4 个 NVSwitch 芯片实现全连接互联，每 GPU 双向带宽 900 GB/s（4th Gen NVLink）。节点间通过 8 个 400Gbps InfiniBand 连接。

**Q15: 如何使用 Nsight 工具分析 GPU 程序的性能瓶颈？**

Nsight Systems 做系统级分析：看 CPU/GPU 重叠、Kernel 并发、数据传输。Nsight Compute 做 Kernel 级分析：看 Occupancy、Memory Throughput、Compute Throughput。通过 Roofline 模型判断是计算密集还是内存密集。

**Q16: 什么是 3D 并行？如何确定 DP/TP/PP 的配置？**

3D 并行 = 数据并行(DP) × 张量并行(TP) × 流水线并行(PP)。配置原则：TP 在节点内（NVLink 高带宽），PP 跨节点，DP 最外层。TP 受限于通信，通常 ≤ 8；PP 增加气泡；剩余 GPU 用 DP。

**Q17: BF16 相比 FP16 有什么优势？**

BF16 有 8 位指数（与 FP32 相同），动态范围大（±3.4×10³⁸），训练时不需要 Loss Scaling。FP16 只有 5 位指数，范围小（±6.5×10⁴），容易溢出。BF16 精度稍低（7 位尾数 vs 10 位）但训练更稳定。

**Q18: GPTQ 和 AWQ 量化的区别是什么？**

GPTQ 是逐层量化，通过最小化量化误差（Hessian 信息）优化权重。AWQ（Activation-aware）观察激活值分布，保护对输出影响大的权重通道，用更高精度或不量化这些通道。AWQ 通常精度更高。

**Q19: 如何分析模型是计算密集还是内存密集？**

计算算术强度 = FLOPs / Bytes。与 GPU 脊点（峰值 TFLOPS / 内存带宽）比较：大于脊点是计算密集，小于脊点是内存密集。或用 Nsight 看 SM 利用率和内存带宽利用率的对比。

**Q20: MIG（Multi-Instance GPU）是什么？适用于什么场景？**

MIG 将一块 GPU（A100/H100）分成最多 7 个独立实例，每个实例有独立的内存和计算资源，相互隔离。适用于多租户云环境、推理服务、小模型训练，提高 GPU 利用率。
