## 1. Embedding 技术

### 1.1 Embedding 的本质

```
┌──────── Embedding 的作用 ────────┐
│                                  │
│  高维稀疏 ────→ 低维稠密          │
│                                  │
│  One-hot (10^7 维)               │
│  [0,0,...,1,...,0,0]             │
│         ↓ Embedding Layer        │
│  Dense (128 维)                  │
│  [0.12, -0.34, 0.56, ...]       │
│                                  │
│  核心价值：                       │
│  1. 降维：从百万维到百维          │
│  2. 泛化：相似实体向量接近        │
│  3. 交叉：向量内积 = 特征交叉    │
└──────────────────────────────────┘
```

### 1.2 ID Embedding 与特征交叉

```python
import torch
import torch.nn as nn

class EmbeddingLayer(nn.Module):
    """推荐模型的 Embedding 层"""
    def __init__(self, feature_configs):
        super().__init__()
        self.embeddings = nn.ModuleDict()
        for name, config in feature_configs.items():
            self.embeddings[name] = nn.Embedding(
                num_embeddings=config['vocab_size'],
                embedding_dim=config['embed_dim'],
                padding_idx=0
            )
            # 初始化策略
            nn.init.xavier_uniform_(self.embeddings[name].weight)

    def forward(self, features: dict) -> dict:
        """
        features: {name: tensor of shape (batch_size,)}
        returns: {name: tensor of shape (batch_size, embed_dim)}
        """
        outputs = {}
        for name, tensor in features.items():
            if name in self.embeddings:
                outputs[name] = self.embeddings[name](tensor)
        return outputs


# 特征配置示例
feature_configs = {
    'user_id':     {'vocab_size': 10_000_000, 'embed_dim': 64},
    'item_id':     {'vocab_size': 5_000_000,  'embed_dim': 64},
    'category_id': {'vocab_size': 10_000,     'embed_dim': 16},
    'city_id':     {'vocab_size': 1_000,      'embed_dim': 8},
}
```

### 1.3 预训练 Embedding

| 方法 | 原理 | 适用 |
|------|------|------|
| **Word2Vec (Item2Vec)** | Skip-gram 在行为序列上学习物品 Embedding | 物品 Embedding |
| **Node2Vec** | 图上随机游走 + Skip-gram | 图结构 Embedding |
| **BERT4Rec** | MLM 在行为序列上预训练 | 行为序列 Embedding |
| **多模态预训练** | CLIP 等对齐图文 Embedding | 内容 Embedding |

```python
# Item2Vec: 将用户行为序列视为"句子"，物品视为"词"
# 用户行为：[item_A, item_B, item_C, item_D]
# 训练 Skip-gram：给定 item_B，预测上下文 item_A, item_C

from gensim.models import Word2Vec

# 用户行为序列
user_sequences = [
    ['item_1', 'item_5', 'item_3', 'item_8'],
    ['item_2', 'item_5', 'item_7', 'item_1'],
    # ... 百万用户的行为序列

model = Word2Vec(
    sentences=user_sequences,
    vector_size=64,
    window=5,
    min_count=5,
    sg=1,  # Skip-gram
    workers=16
)

# 获取物品向量
item_vector = model.wv['item_5']
similar_items = model.wv.most_similar('item_5', topn=10)
```

### 1.4 Embedding 工程实践

| 挑战 | 解决方案 |
|------|---------|
| **Embedding 表过大**（百亿参数） | 参数服务器分布式存储 |
| **低频 ID 过拟合** | 频次阈值 + 哈希桶 + 正则化 |
| **新 ID 无 Embedding** | 哈希 Embedding + 内容 Embedding 兜底 |
| **多模型共享 Embedding** | 统一 Embedding 服务，多模型共用 |
| **训练效率** | 只更新本 batch 涉及的 Embedding 行 |

```
Embedding 大小估算：
- 1亿 user_id × 64 维 × 4 字节 = 25.6 GB
- 5000万 item_id × 64 维 × 4 字节 = 12.8 GB
- 总 Embedding 参数：~40 GB
→ 单机放不下，需要参数服务器
```

## 2. 经典推荐模型详解

### 2.1 FM 与 FFM

#### FM (Factorization Machine)

```
FM 核心：自动学习二阶特征交叉

ŷ = w₀ + Σᵢ wᵢxᵢ + Σᵢ Σⱼ>ᵢ <vᵢ, vⱼ> xᵢxⱼ

其中 <vᵢ, vⱼ> = Σₖ vᵢₖ·vⱼₖ 是隐向量内积

时间复杂度优化（O(n²k) → O(nk)）：
Σᵢ Σⱼ>ᵢ <vᵢ, vⱼ> xᵢxⱼ = ½ Σₖ [(Σᵢ vᵢₖxᵢ)² - Σᵢ (vᵢₖxᵢ)²]
```

```python
class FM(nn.Module):
    """Factorization Machine"""
    def __init__(self, num_features, embed_dim):
        super().__init__()
        self.linear = nn.Linear(num_features, 1)
        self.embedding = nn.Embedding(num_features, embed_dim)

    def forward(self, x):
        # x: (batch, num_features) sparse input

        # 一阶
        linear_out = self.linear(x)

        # 二阶交叉（O(nk) 优化）
        embed = self.embedding.weight * x.unsqueeze(-1)  # (batch, n, k)
        sum_square = embed.sum(dim=1).pow(2)    # (Σv_i·x_i)²
        square_sum = embed.pow(2).sum(dim=1)    # Σ(v_i·x_i)²
        cross = 0.5 * (sum_square - square_sum).sum(dim=1, keepdim=True)

        return torch.sigmoid(linear_out + cross)
```

#### FFM (Field-aware FM)

```
FFM：每个特征对不同 field 学习不同的 Embedding

FM:  <v_user_age, v_item_category>
FFM: <v_user_age[field=item], v_item_category[field=user]>

优点：交叉更精细
缺点：参数量大（n × f × k，f 为 field 数）
```

### 2.3 DeepFM

```
┌──────── DeepFM (2017) ────────────────────┐
│                                            │
│        Input Features                      │
│        ┌───┬───┬───┬───┐                  │
│        │x₁ │x₂ │x₃ │...│                 │
│        └─┬─┴─┬─┴─┬─┴───┘                  │
│          │   │   │                         │
│     Embedding Layer (共享)                  │
│          │   │   │                         │
│      ┌───┘   │   └───┐                    │
│      ▼               ▼                    │
│  ┌──────┐       ┌──────┐                  │
│  │  FM  │       │  DNN │                  │
│  │二阶交叉│      │多层全连接│               │
│  └──┬───┘       └──┬───┘                  │
│     │              │                       │
│     └──── + ────── ┘                      │
│           │                                │
│        sigmoid                             │
│           │                                │
│           ŷ                                │
│                                            │
│  相比 Wide&Deep：                           │
│  - FM 替代 Wide（自动交叉，无需手工特征）    │
│  - Embedding 共享（减少参数量）              │
└────────────────────────────────────────────┘
```

```python
class DeepFM(nn.Module):
    def __init__(self, feature_configs, hidden_dims=[256, 128, 64]):
        super().__init__()
        self.embed_layer = EmbeddingLayer(feature_configs)

        total_embed_dim = sum(c['embed_dim'] for c in feature_configs.values())

        # FM 一阶
        self.fm_linear = nn.Linear(total_embed_dim, 1)

        # DNN
        layers = []
        input_dim = total_embed_dim
        for dim in hidden_dims:
            layers.extend([
                nn.Linear(input_dim, dim),
                nn.BatchNorm1d(dim),
                nn.ReLU(),
                nn.Dropout(0.1)
            ])
            input_dim = dim
        layers.append(nn.Linear(input_dim, 1))
        self.dnn = nn.Sequential(*layers)

    def forward(self, features):
        embeddings = self.embed_layer(features)  # dict of (batch, dim)
        embed_list = list(embeddings.values())

        # FM 二阶交叉
        stacked = torch.stack(embed_list, dim=1)  # (batch, n_fields, dim)
        sum_sq = stacked.sum(1).pow(2)
        sq_sum = stacked.pow(2).sum(1)
        fm_cross = 0.5 * (sum_sq - sq_sum).sum(1, keepdim=True)

        # 拼接所有 embedding
        concat = torch.cat(embed_list, dim=1)

        # FM 一阶
        fm_linear = self.fm_linear(concat)

        # DNN
        dnn_out = self.dnn(concat)

        # 融合
        logit = fm_linear + fm_cross + dnn_out
        return torch.sigmoid(logit)
```

### 2.4 DCN (Deep & Cross Network)

```
Cross Network 核心公式：
x_{l+1} = x₀ · xₗᵀ · wₗ + bₗ + xₗ

特点：
- 每层显式计算 x₀ 与 xₗ 的特征交叉
- L 层 Cross Network 可建模 L+1 阶交叉
- 参数量仅 O(d×L)（d 为输入维度，L 为层数）

DCN-V2 (2020) 改进：
x_{l+1} = x₀ ⊙ (Wₗ · xₗ + bₗ) + xₗ
- 用矩阵 W 替代向量 w
- 更强的交叉能力
```

### 3.1 DIN (Deep Interest Network)

```
┌──────── DIN 核心思想 ────────────────────────┐
│                                               │
│  用户历史行为：[鞋子A, 手机B, 衣服C, 鞋子D]     │
│  候选物品：鞋子E                               │
│                                               │
│  传统方法：对所有行为 avg pooling → 丢失差异    │
│  DIN 思想：与候选物品相关的行为权重更高          │
│                                               │
│  Attention(候选, 行为ᵢ) = softmax(f(eᵢ, eₜ))  │
│                                               │
│  鞋子A: attention=0.4  ← 与鞋子E相关          │
│  手机B: attention=0.05 ← 无关                  │
│  衣服C: attention=0.05 ← 无关                  │
│  鞋子D: attention=0.5  ← 与鞋子E相关          │
│                                               │
│  用户兴趣表示 = Σ attention_i × embedding_i    │
└───────────────────────────────────────────────┘
```

```python
class DINAttention(nn.Module):
    """DIN Activation Unit — 行为与候选物品的注意力"""
    def __init__(self, embed_dim, hidden_dim=64):
        super().__init__()
        self.mlp = nn.Sequential(
            nn.Linear(embed_dim * 4, hidden_dim),
            nn.ReLU(),
            nn.Linear(hidden_dim, 1)
        )

    def forward(self, query, keys, keys_mask=None):
        """
        query: (batch, embed_dim) — 候选物品 Embedding
        keys:  (batch, seq_len, embed_dim) — 历史行为 Embedding
        keys_mask: (batch, seq_len) — padding mask
        """
        seq_len = keys.size(1)
        query = query.unsqueeze(1).expand_as(keys)  # (batch, seq_len, dim)

        # 四元组输入：[query, key, query-key, query*key]
        din_input = torch.cat([
            query, keys, query - keys, query * keys
        ], dim=-1)

        scores = self.mlp(din_input).squeeze(-1)  # (batch, seq_len)

        if keys_mask is not None:
            scores = scores.masked_fill(~keys_mask, float('-inf'))

        weights = torch.softmax(scores, dim=-1)  # (batch, seq_len)

        # 加权求和
        output = (keys * weights.unsqueeze(-1)).sum(dim=1)  # (batch, dim)
        return output
```

### 3.2 DIEN (Deep Interest Evolution Network)

```
┌──────── DIEN 架构 ────────────────────────┐
│                                            │
│  行为序列: [b₁, b₂, b₃, ..., bₙ]          │
│                                            │
│  Layer 1: 兴趣提取层 (Interest Extractor)  │
│  ├── GRU 捕捉行为序列依赖                   │
│  ├── 辅助 Loss：预测下一个行为               │
│  └── 输出：兴趣状态序列 [h₁, h₂, ..., hₙ]  │
│                                            │
│  Layer 2: 兴趣演化层 (Interest Evolution)   │
│  ├── AUGRU (Attention-GRU)                 │
│  ├── 用候选物品做 Attention 指导演化方向     │
│  └── 只演化与候选相关的兴趣                  │
│                                            │
│  最终输出：与候选物品相关的兴趣最终状态       │
│                                            │
│  对比 DIN：DIN 是静态注意力                  │
│           DIEN 是动态兴趣演化               │
└────────────────────────────────────────────┘
```

**AUGRU (Attention-based GRU)**：

```
标准 GRU：
h_t = (1 - z_t) ⊙ h_{t-1} + z_t ⊙ h̃_t

AUGRU：
h'_t = (1 - a_t × z_t) ⊙ h_{t-1} + a_t × z_t ⊙ h̃_t

其中 a_t = attention(h_t, target_item)
效果：与候选物品无关的行为对隐状态影响减弱
```

### 3.3 BST (Behavior Sequence Transformer)

```
┌──────── BST 架构 ────────────────────────┐
│                                           │
│  DIN/DIEN 的问题：                         │
│  - DIN 无序列信息                          │
│  - DIEN 的 GRU 是顺序处理，长序列慢        │
│                                           │
│  BST 解决方案：用 Transformer 替代 GRU     │
│                                           │
│  行为序列 + 位置编码                        │
│       ↓                                   │
│  Self-Attention (多头)                     │
│       ↓                                   │
│  Feed-Forward Network                     │
│       ↓                                   │
│  × L 层                                   │
│       ↓                                   │
│  与候选物品做 target attention              │
│       ↓                                   │
│  拼接其他特征 → MLP → 预测                  │
│                                           │
│  优点：                                    │
│  - 并行计算（vs GRU 的串行）               │
│  - 长距离依赖建模更好                       │
│  - 工业界广泛使用                           │
└───────────────────────────────────────────┘
```

### 3.4 SIM (Search-based Interest Model)

```
问题：用户行为序列可能有上万条，全部输入 Transformer 太慢

SIM 两阶段方案：
┌──────────────────────────────────────────┐
│  Stage 1: General Search Unit (GSU)      │
│  ├── 用候选物品检索相关的历史行为          │
│  ├── 方法：向量相似度 / 类目匹配          │
│  └── 从 10,000 条筛选到 100 条            │
│                                          │
│  Stage 2: Exact Search Unit (ESU)        │
│  ├── 对 100 条做精细 attention            │
│  ├── 类似 DIN 的 target attention         │
│  └── 输出用户兴趣表示                     │
└──────────────────────────────────────────┘

效果：支持超长行为序列（180天），精度不降，延迟可控
```

## 4. 多目标与多任务学习

### 4.1 Shared-Bottom / MMOE / PLE

```
┌─── Shared-Bottom ───┐  ┌──── MMOE ────────┐  ┌──── PLE ─────────┐
│                      │  │                   │  │                   │
│  Shared Layers       │  │  Expert 1         │  │  Task-A Experts   │
│       ↓              │  │  Expert 2         │  │  Shared Experts   │
│  ┌────┬────┐         │  │  Expert 3         │  │  Task-B Experts   │
│  │Task│Task│         │  │     ↓             │  │     ↓             │
│  │ A  │ B  │         │  │  Gate A  Gate B   │  │  Gate A  Gate B   │
│  └────┴────┘         │  │  (soft routing)   │  │  (per-task gate)  │
│                      │  │     ↓             │  │     ↓             │
│  问题：任务间                │  ┌────┬────┐         │  │  ┌────┬────┐         │
│  可能冲突（跷跷板）   │  │Task│Task│         │  │  │Task│Task│         │
│                      │  │ A  │ B  │         │  │  │ A  │ B  │         │
│                      │  └────┴────┘         │  │  └────┴────┘         │
└──────────────────────┘  └───────────────────┘  └───────────────────┘
```

**MMOE 核心机制**：

```python
class MMOE(nn.Module):
    """Multi-gate Mixture-of-Experts"""
    def __init__(self, input_dim, expert_dim, num_experts, num_tasks):
        super().__init__()
        self.experts = nn.ModuleList([
            nn.Sequential(
                nn.Linear(input_dim, expert_dim),
                nn.ReLU(),
                nn.Linear(expert_dim, expert_dim)
            ) for _ in range(num_experts)
        ])
        self.gates = nn.ModuleList([
            nn.Sequential(
                nn.Linear(input_dim, num_experts),
                nn.Softmax(dim=-1)
            ) for _ in range(num_tasks)
        ])

    def forward(self, x):
        # 所有 expert 的输出
        expert_outs = [expert(x) for expert in self.experts]
        expert_outs = torch.stack(expert_outs, dim=1)  # (batch, n_experts, dim)

        # 每个 task 的门控权重
        task_outputs = []
        for gate in self.gates:
            gate_weights = gate(x).unsqueeze(-1)  # (batch, n_experts, 1)
            task_input = (expert_outs * gate_weights).sum(dim=1)  # (batch, dim)
            task_outputs.append(task_input)

        return task_outputs  # list of (batch, dim)
```

### 4.2 ESMM 全空间多任务

```
┌──────── ESMM 解决的问题 ────────────────┐
│                                         │
│  CVR 训练的困境：                         │
│  - CVR = P(conversion | click)           │
│  - 只有点击样本才有转化标签               │
│  - 点击样本是有偏的子集（SSB 问题）       │
│                                         │
│  ESMM 解法：                             │
│  P(conversion) = P(click) × P(conv|click)│
│       CTCVR    =   CTR    ×    CVR       │
│                                         │
│  ┌─────────────────────────────┐        │
│  │    共享 Embedding           │        │
│  │         ↓                   │        │
│  │  ┌──────────┬──────────┐   │        │
│  │  │ CTR Task │ CVR Task │   │        │
│  │  │  P(clk)  │P(conv|clk)│  │        │
│  │  └────┬─────┴────┬─────┘   │        │
│  │       └── × ─────┘         │        │
│  │           ↓                │        │
│  │        CTCVR               │        │
│  │      P(clk & conv)         │        │
│  └─────────────────────────────┘        │
│                                         │
│  优点：CVR 在全空间训练，无 SSB 问题     │
└─────────────────────────────────────────┘
```

## 5. 召回模型

### 5.1 双塔模型 (DSSM)

```python
class TwoTowerModel(nn.Module):
    """双塔模型：用户塔 + 物品塔"""
    def __init__(self, user_feature_configs, item_feature_configs, embed_dim=128):
        super().__init__()
        # 用户塔
        user_input_dim = sum(c['embed_dim'] for c in user_feature_configs.values())
        self.user_embed = EmbeddingLayer(user_feature_configs)
        self.user_tower = nn.Sequential(
            nn.Linear(user_input_dim, 256),
            nn.ReLU(),
            nn.Linear(256, embed_dim),
            nn.BatchNorm1d(embed_dim)
        )

        # 物品塔
        item_input_dim = sum(c['embed_dim'] for c in item_feature_configs.values())
        self.item_embed = EmbeddingLayer(item_feature_configs)
        self.item_tower = nn.Sequential(
            nn.Linear(item_input_dim, 256),
            nn.ReLU(),
            nn.Linear(256, embed_dim),
            nn.BatchNorm1d(embed_dim)
        )

        self.temperature = nn.Parameter(torch.tensor(0.07))

    def forward(self, user_features, item_features):
        # 用户向量
        user_embeds = self.user_embed(user_features)
        user_concat = torch.cat(list(user_embeds.values()), dim=-1)
        user_vec = self.user_tower(user_concat)
        user_vec = nn.functional.normalize(user_vec, dim=-1)

        # 物品向量
        item_embeds = self.item_embed(item_features)
        item_concat = torch.cat(list(item_embeds.values()), dim=-1)
        item_vec = self.item_tower(item_concat)
        item_vec = nn.functional.normalize(item_vec, dim=-1)

        # 相似度
        logits = torch.matmul(user_vec, item_vec.T) / self.temperature
        return logits
```

### 8.4 工业落地中的训练与服务一致性

推荐模型上线后，很多问题不是“模型不会学”，而是“训练口径和线上口径开始分叉”。高频治理点有四类：

| 问题 | 典型表现 | 常见治理手段 |
|------|----------|--------------|
| **Delayed Feedback** | 转化标签回流慢，CVR 训练样本长期不完整 | 标签回填、延迟建模、分窗口评估 |
| **Training/Serving Skew** | 离线指标高，线上效果突然变差 | 特征回放、一致性校验、统一特征定义 |
| **Embedding Drift** | 召回覆盖率下降，向量相似度失真 | 新鲜度监控、分桶重建、冷热分层更新 |
| **Explore/Exploit 失衡** | 指标短期稳定但长期增长停滞 | Bandit、探索配额、护栏指标 |

**经验原则**：
1. 模型效果问题先排“数据与特征一致性”，再排网络结构。
2. 线上收益下滑时先看召回覆盖、样本延迟、特征新鲜度，不要第一反应就是重训大模型。
3. 推荐系统不是只追求短期 CTR，而是要在探索、稳定性和长期收益之间持续找平衡。

### Q1: DeepFM 相比 Wide&Deep 的优势是什么？

**答案**：
1. **FM 替代 Wide**：Wide 需要手工构造交叉特征，FM 自动学习二阶交叉
2. **Embedding 共享**：FM 和 DNN 共享同一套 Embedding，减少参数量，训练更充分
3. **端到端训练**：不需要像 Wide&Deep 那样分别预训练 Wide 和 Deep

本质上 DeepFM = FM（自动二阶交叉）+ DNN（高阶交叉），比 Wide&Deep 更优雅。

---

### Q2: DIN 的核心思想？为什么比 avg pooling 好？

**答案**：
**核心思想**：用户的兴趣是多样的，对不同候选物品应该关注不同的历史行为。

**Avg Pooling 的问题**：将所有行为等权平均，丢失了"哪些行为与当前候选物品相关"的信息。

**DIN 的做法**：以候选物品为 query，对历史行为做 attention 加权，与候选物品相关的行为获得更高权重。

例如：用户看过 [鞋子, 手机, 衣服, 鞋子]，候选是鞋子 → 两次鞋子行为权重高，手机衣服权重低。

---

### Q3: 双塔模型从训练到 ANN 检索的闭环是怎样的？为什么它天然适合召回模型？

**答案**：
**训练阶段**：用户塔和物品塔联合训练，目标通常是让正样本用户-物品向量更接近，Loss 常见为 Sampled Softmax、BPR、InfoNCE。

**部署阶段**：
1. **离线产物料向量**：用物品塔批量产出 embedding。
2. **建 ANN 索引**：把物料向量写入 HNSW、IVF、PQ 等近似最近邻索引。
3. **在线产用户向量**：请求到来时实时算用户 embedding。
4. **向量召回**：用用户向量查索引拿 Top-K 候选。

**它为什么适合召回**：
1. **打分可拆解**：用户侧在线算一次，物品侧离线预计算，天然适合大规模候选检索。
2. **训练目标和检索形态一致**：训练时就在学“相似向量应靠近”，部署时直接用相似度做召回。
3. **延迟友好**：在线不需要逐物品复杂交叉，适合毫秒级召回预算。

这里重点是模型闭环和训练部署一致性；如果面试官追问索引新鲜度、服务降级和在线治理，优先切到 `专项知识库/07-推荐系统专项` 的双塔题。

### Q4: 推荐模型为什么需要参数服务器而不是 AllReduce？

**答案**：推荐模型的 Embedding 表非常大（百 GB 级），远超单卡显存。

| 特征 | AllReduce | Parameter Server |
|------|-----------|-----------------|
| 参数存储 | 每卡全量参数 | 分布式存储 |
| 通信 | 同步所有梯度 | 只传输稀疏梯度 |
| 适用 | 稠密模型（CV/NLP） | 稀疏大模型（推荐） |

推荐模型每个 batch 只涉及极少数 Embedding 行（万亿级 Embedding 中可能只用到万级），用参数服务器只需 pull/push 相关行，通信量远小于全量同步。

---

### Q5: ESMM 解决了什么问题？

**答案**：ESMM 解决了 CVR 模型的**样本选择偏差 (SSB)** 和**数据稀疏**问题。

**问题**：CVR = P(conv|click)，但训练数据只有点击用户的子集，这个子集是有偏的（点击用户不代表全体用户）。

**解决方案**：
- 建模等式：`P(click & conv) = P(click) × P(conv|click)`，即 `CTCVR = CTR × CVR`
- CTR 和 CVR 联合训练，**CTR 用全量曝光样本**，CVR 通过乘法关系隐式在全空间训练
- 解决了 CVR 只在点击子集训练的偏差问题

---

### Q6: 在线学习和全量训练各自的优缺点？

**答案**：

| 维度 | 全量训练 | 在线学习 |
|------|---------|---------|
| 更新频率 | 天级 | 分钟级 |
| 数据使用 | 全量历史数据 | 增量新数据 |
| 实时性 | 低（模型滞后一天） | 高（快速捕捉新趋势） |
| 稳定性 | 高 | 较低（可能遗忘/漂移） |
| 计算成本 | 高（每次全量） | 低（只处理增量） |

**工业界最佳实践**：天级全量训练（Base Model）+ 分钟级在线学习（增量更新），配合模型质量监控和异常自动回滚。

---

### Q7: 多目标学习中如何处理任务冲突？

**答案**：
1. **MMOE**：多个 Expert + 每个 task 独立 Gate，自动学习 Expert 共享程度
2. **PLE**：进一步引入 task-specific Expert，更好地处理任务差异
3. **GradNorm**：自适应调整多任务的 loss 权重，平衡梯度大小
4. **Pareto 优化**：搜索多目标的 Pareto 前沿，选择综合最优
5. **任务分组**：将相关任务放一起训练，冲突任务分开

本质上是在**共享**（减少参数/提升泛化）和**独立**（减少干扰）之间寻找平衡。

---

### Q8: 如何评估召回模型和精排模型？指标有什么区别？

**答案**：

| 模块 | 核心指标 | 说明 |
|------|---------|------|
| **召回** | Recall@K | 正样本被召回的比例 |
| **召回** | HitRate@K | 至少命中一个正样本的用户比例 |
| **精排** | AUC | 正样本排在负样本前面的概率 |
| **精排** | GAUC | 按用户分组的 AUC 加权平均 |
| **精排** | NDCG@K | 带位置折扣的排序质量 |
| **整体** | CTR/时长/留存 | 在线 AB 实验核心指标 |

**关键区别**：召回关注"不遗漏"（Recall），精排关注"排得准"（AUC/NDCG）。离线指标只是参考，**最终以在线 AB 实验为准**。

### Q9: 推荐系统出现“整体CTR升高但用户时长下降”，你会怎么解释？

**答案**：
这是典型的目标错配：模型可能过度优化了“点击”而牺牲了“消费质量”。常见原因：标题党内容被放大、短平快点击增多但深度消费减少。

治理思路：
1. 多目标建模（CTR + 时长 + 留存）而非单目标 CTR。
2. 在重排阶段加入质量约束（负反馈惩罚、内容多样性）。
3. 在线实验关注长期指标，避免被短期点击误导。

### Q10: 从模型和特征视角看，双冷启动时召回该怎么设计？

**答案**：
这题在本篇更关注“没有足够行为数据时，模型还能吃什么信号”。建议从三类特征源补足：
1. **用户侧先验**：人口属性、上下文、入口场景、设备与时间特征，先提供粗粒度兴趣。
2. **物料侧内容信号**：文本、图像、类目、标签 embedding，让新物料至少能被内容相似召回。
3. **训练侧样本构造**：用热门样本、跨域迁移样本、相似物料蒸馏样本，避免模型完全学不到新物料表示。

常见做法是“内容召回 + 热门先验 + 轻探索”组合，让模型先有可学习信号，再逐步切换到行为驱动的个性化召回。

如果要展开讲新用户引导、流量扶持、产品冷启动编排和服务兜底，那属于系统架构视角，优先看 `专项知识库/07-推荐系统专项`。

### Q11: 特征穿越（Feature Leakage）在推荐训练里怎么识别和规避？

**答案**：
识别信号：离线指标异常高、线上效果显著回落。常见穿越包括使用了曝光后才可得特征（未来信息）。

规避方法：
1. 严格按事件时间构建样本，特征时间戳必须早于标签时间。
2. 建立特征血缘审计和离线校验规则。
3. 新特征上线前做“时间回放验证”。

训练链路里时间语义不清，是推荐系统最常见的隐性事故源。

---

### Q12: 从候选集质量视角看，多路召回并行时如何做配额和去重，避免浪费精排容量？

**答案**：
这里重点不是在线服务编排，而是“给精排喂什么候选集才最有学习价值和收益空间”。

1. **先去重并保留来源特征**：同一 item 被多路召回时不要重复占坑，但要把来源路由、原始分数写回特征，供后续模型学习。
2. **基础配额 + 动态调权**：热门路保证底线，长尾路保探索空间，再按近期效果、覆盖率和重合度动态调节。
3. **惩罚高重合路**：如果某路和其他路长期高度重复，它对候选增益就低，应该下调配额。
4. **面向精排容量优化**：候选集的目标不是越多越好，而是让有限的精排预算看到“更有区分度”的候选。

如果面试官继续追问在线融合、超时控制、服务降级和缓存编排，优先切到 `专项知识库/07-推荐系统专项` 的多路召回题。

### Q13: 推荐线上效果波动时，如何区分“模型问题”和“流量问题”？

**答案**：
先做分层对比：
1. 模型侧：同流量回放下离线打分是否异常漂移。
2. 流量侧：渠道结构、活动流量、用户结构是否突变。
3. 系统侧：特征服务延迟、样本延迟、召回覆盖是否下降。

高效做法是保留“固定流量哨兵集”做持续监控，快速定位问题层级。

---

### Q14: 多目标学习里，如何避免某个大流量任务“压制”其他任务？

**答案**：
可结合三类策略：
1. 动态 loss reweight（如 GradNorm）平衡梯度规模。
2. 任务分层塔结构（MMOE/PLE）减少负迁移。
3. 关键小任务设置最低效果约束，防止被主任务吞噬。

多目标不是“把 loss 相加”就结束，必须持续监控各任务帕累托权衡。

### Q15: 推荐模型如何设计“可回滚”的发布策略？

**答案**：
建议分三层回滚：
1. 模型版本可快速切回上一个稳定 checkpoint。
2. 特征开关可按字段级回退，防止单特征事故拖垮全链路。
3. 流量层可按用户分组回退灰度。

并在发布前固定回滚演练，确保“能发版”也“能止损”。

### Q16: 你会如何给推荐团队建立“离线评估到线上收益”的闭环？

**答案**：
闭环建议包括：
1. 建立离线指标到线上指标的历史映射关系（相关性监控）。
2. 新模型必须通过小流量 AB，并设定显著性门槛。
3. 发布后持续追踪短期与长期指标分化（CTR、时长、留存）。
4. 复盘沉淀“有效指标组合”，淘汰无预测力指标。

核心不是指标越多越好，而是“指标能指导发布决策”。

### Q17: 延迟反馈（Delayed Feedback）为什么会让 CVR 训练变难？通常怎么处理？

**答案**：
CVR 标签天然存在延迟，比如曝光后几小时甚至几天才发生转化。如果太早抽取样本，很多“未来会转化”的样本会被误标成负样本，模型就会系统性低估转化率。

常见处理方式：
1. **观察窗口回撤**：训练时只使用已过足够观察窗口的样本。
2. **标签回填**：转化事件回流后修正历史样本标签。
3. **延迟感知建模**：将“曝光到转化的时间差”纳入建模或校准。
4. **分窗口评估**：区分短转化和长转化，避免单一口径误导训练。

关键不是完全消灭延迟，而是让训练数据对延迟有感知。

### Q18: 从模型一致性视角，Training/Serving Skew 怎么发现和治理？

**答案**：
Training/Serving Skew 的核心不是“系统有点不一致”，而是模型训练时学到的分布与线上打分输入已经不是同一个问题。它会直接导致离线指标好看、线上收益失真。

**发现方式**：
1. 抽样回放线上请求，复算离线特征并逐字段比对。
2. 监控特征缺失率、默认值命中率、分桶分布漂移。
3. 对关键样本做“线上打分 vs 离线打分”一致性校验。
4. 关注 embedding、特征交叉和归一化链路是否出现训练时有、线上无的情况。

**治理方式**：
1. 统一特征定义和算子实现，避免离线/在线各写一套。
2. 关键特征版本化，发布时校验 schema 和默认值。
3. 把一致性检查接入发布门禁，不通过就不发版。
4. 对模型输入分布做持续监控，而不是只在出问题后人工回放。

本题偏模型一致性和输入分布；如果要讲请求回放平台、服务发布门禁、线上链路排障，优先看 `专项知识库/07-推荐系统专项`。

### Q19: Embedding 漂移为什么会拖垮召回效果？你会怎么监控？

**答案**：
Embedding 漂移是指用户或物品向量随时间偏移，但索引、样本或特征更新节奏跟不上，导致相似度空间失真。结果通常表现为召回覆盖率下降、热门内容挤压、长尾内容几乎消失。

监控重点：
1. **向量分布**：范数、均值、分桶密度是否突变。
2. **业务指标**：Recall@K、召回后点击率、去重后候选覆盖率。
3. **新鲜度指标**：向量更新时间、索引更新时间、增量失败率。
4. **哨兵集合**：固定用户/物品样本的 TopK 是否异常漂移。

修复手段通常是重建索引、加快增量更新、对冷门向量做内容侧兜底。

### Q20: 从训练收益视角，Exploration 样本怎么设计，才不会学到一堆噪声？

**答案**：
这题在本篇重点不是流量开关，而是“探索样本怎么进入训练集，才能真正提升模型”。

1. **控制探索配额**：探索样本太少学不到东西，太多又会污染主分布，通常只在有限人群和位置开放。
2. **优先高信息量探索**：用 Bandit、不确定性采样、相似簇补洞，尽量探索“模型最缺信息的区域”，而不是纯随机。
3. **单独打标与回流**：探索样本最好带上来源标记，避免与正常流量混在一起后无法评估价值。
4. **看长期收益**：探索的目标不是当场点高，而是改善后续召回覆盖、冷启动学习和长期收益。
5. **建立护栏**：负反馈率、跳失率、投诉率超阈值时及时收缩探索范围。

如果面试官更关注流量编排、实验护栏和线上体验保护，那属于系统治理视角，优先看 `专项知识库/07-推荐系统专项`。
