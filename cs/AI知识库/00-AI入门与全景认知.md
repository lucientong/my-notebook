# AI 入门与全景认知

> 这篇不是工程速查，而是 AI 知识库的真正起点。目标只有一个：先把概念边界、系统全景和学习顺序讲明白，避免一上来就陷进 Transformer 公式、Agent 框架或微调细节。

---

## 1. 先建立一张全景图

很多人第一次学 AI，会把下面几件事混成一团：

- `模型`：LLM 为什么能生成文本，训练目标是什么
- `应用`：Prompt、RAG、Tool Calling、Agent 怎么组合
- `工程`：如何控成本、控延迟、做评估、做权限与观测
- `训练`：如何微调、对齐、量化、部署

这四层不是同一件事。更准确地说，它们是一条从底到上的链路：

```text
模型原理
  ↓
应用编排
  ↓
系统工程
  ↓
训练与优化
```

如果你还没有把前 3 层理顺，就直接去看 LoRA、DPO、蒸馏，知识结构会倒挂。

## 2. LLM 到底是什么

### 2.1 一句话定义

LLM（Large Language Model，大语言模型）本质上是一个“根据上下文预测下一个 token 的概率模型”。

它不是数据库，不是搜索引擎，也不是规则系统。它擅长的是：

- 根据上下文生成自然语言
- 压缩和重组大量语言模式
- 在训练分布内做较强的泛化

它不天然擅长的是：

- 长期、精确、可追溯的事实记忆
- 实时外部世界更新
- 对高风险任务做确定性正确执行

这也是为什么后面会出现 RAG、Tool Calling 和 Agent。

### 2.2 Token 不是“一个字”

模型并不是按“词”或“汉字”直接理解世界，而是把文本切成 token 来处理。

```text
输入文本 → 分词器（Tokenizer） → token 序列 → 模型计算概率 → 生成下一个 token
```

要先建立两个基本认识：

- token 是模型内部处理文本的最小单位，不等于字数
- 上下文窗口限制的是 token 数，不是字符数

所以很多工程问题，本质上都和 token 成本有关：

- 为什么长上下文更贵
- 为什么 prompt 写法会影响价格和延迟
- 为什么 RAG 要做 chunking

## 3. AI 应用为什么不是“调个接口”

### 3.1 最小调用链路

一个最小 AI 应用，通常至少包含下面这条链路：

```text
用户问题
  ↓
Prompt 构造
  ↓
模型调用
  ↓
结果解析
  ↓
返回答案
```

如果业务稍微复杂一点，链路会立刻变成：

```text
用户问题
  ↓
意图识别 / 路由
  ↓
补充上下文（RAG / Memory / 用户资料）
  ↓
决定是否调用工具
  ↓
模型生成
  ↓
结构化解析 / 校验 / 安全过滤
  ↓
返回结果并记录日志
```

所以 AI 应用并不只是“模型很强”，而是“围绕模型构建一个完整系统”。

### 3.2 4 个最容易混淆的概念

#### Prompt

给模型的输入组织方式。它解决的是“怎样更清楚地表达任务”。

#### RAG

先检索外部知识，再把结果放进上下文里，让模型基于检索内容生成。它解决的是“模型不知道最新信息或私有知识”。

#### Tool Calling

让模型决定是否调用外部工具，例如查数据库、发请求、执行查询。它解决的是“模型需要真实世界能力，而不是只靠生成”。

#### Agent

在 Prompt、工具、记忆、规划、多步执行之上，再包一层流程控制。它解决的是“任务不是一步回答，而是多步决策与执行”。

一个常见误区是把 Agent 理解成“更强的模型”。实际上 Agent 更像一套带状态和流程的系统。

---

## 4. 什么时候该用什么

| 问题类型 | 更合适的方式 | 原因 |
|------|------|------|
| 固定问答、规则清晰 | Prompt + 模板 | 成本低，行为稳定 |
| 依赖私有知识库 | RAG | 不需要改模型本身 |
| 需要查数据库、发请求、执行动作 | Tool Calling | 真实能力来自外部系统 |
| 需要拆步骤、反复决策、多工具协作 | Agent | 需要流程和状态管理 |
| 需要稳定学会某一类风格或任务 | 微调 | 通过训练改模型行为 |

不要一上来就上 Agent。很多问题用 `Prompt + RAG + 一个工具` 就够了。

## 5. 一套 AI 系统的核心组成

```text
前端 / API
  ↓
业务层
  ↓
模型路由层
  ↓
上下文构造层
  ├─ Prompt
  ├─ RAG
  ├─ Memory
  └─ 用户上下文
  ↓
模型调用层
  ↓
工具层 / 数据层
  ↓
评估、日志、监控、安全
```

这张图背后对应的是 5 个工程问题：

1. `效果问题`：回答是否有用、是否稳定。
2. `成本问题`：token 消耗和模型价格是否可控。
3. `延迟问题`：首 token 时间和整体响应时间是否可接受。
4. `安全问题`：是否存在提示注入、越权调用、敏感信息泄露。
5. `可运维问题`：坏结果能不能定位、评估、回滚、灰度。

## 6. 你现在最需要先搞懂的基础概念

### 6.1 模型视角

- Transformer 是什么
- Attention 在做什么
- 上下文窗口为什么有限
- Temperature / top-p 为什么会影响输出风格
- KV Cache 为什么能加速推理

### 6.2 应用视角

- Prompt 怎样约束输出格式
- RAG 为什么会“检索对了但回答还是错”
- Tool Calling 为什么要有参数 schema
- Agent 为什么需要状态与记忆

### 6.3 工程视角

- 为什么要做缓存、限流、降级
- 为什么要做离线评估和在线监控
- 为什么需要日志、trace、用户反馈闭环
- 为什么不能把模型直接暴露给任意工具权限

---

## 7. 新手最容易踩的坑

- 把 LLM 当数据库。模型会“像知道”，但不保证事实正确。
- 把 RAG 当万能药。RAG 只能补知识，不自动补推理和校验。
- 把 Prompt Engineering 理解成“咒语学”。好的 prompt 本质是任务定义、上下文控制和输出约束。
- 把 Agent 当成默认方案。Agent 增强能力的同时也增加不稳定性和成本。
- 还没把系统链路搭清楚，就先研究微调和量化。这会让你很快陷入细节泥潭。
