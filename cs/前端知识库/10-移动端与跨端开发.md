## 移动端适配方案

> ⭐⭐⭐ 移动端面试必考：Viewport → rem/vw → 1px → 安全区域 → 折叠屏，一条线串联。

### Viewport 基础

**视口概念**

移动端存在三种视口：

| 视口类型 | 说明 | 获取方式 |
|---------|------|---------|
| 布局视口 (Layout Viewport) | 浏览器默认视口，一般 980px | `document.documentElement.clientWidth` |
| 视觉视口 (Visual Viewport) | 用户实际看到的区域 | `window.innerWidth` |
| 理想视口 (Ideal Viewport) | 设备屏幕宽度 | `screen.width` |

**meta viewport 配置**

```html
<!-- 标准移动端配置 -->
<meta name="viewport"
  content="width=device-width, initial-scale=1.0, maximum-scale=1.0, minimum-scale=1.0, user-scalable=no, viewport-fit=cover">
```

各属性说明：

| 属性 | 说明 | 常用值 |
|------|------|-------|
| `width` | 视口宽度 | `device-width` |
| `initial-scale` | 初始缩放比例 | `1.0` |
| `maximum-scale` | 最大缩放比例 | `1.0`（禁止缩放） |
| `user-scalable` | 是否允许缩放 | `no` |
| `viewport-fit` | 视口填充模式 | `cover`（适配刘海屏） |

**设备像素比 (DPR)**

```javascript
// DPR = 物理像素 / 逻辑像素 (CSS 像素)
const dpr = window.devicePixelRatio; // iPhone 14: 3, 普通安卓: 2-3

// 动态设置 viewport scale（淘宝 flexible 方案核心思路）
function setViewport() {
  const dpr = window.devicePixelRatio || 1;
  const scale = 1 / dpr;
  const meta = document.querySelector('meta[name="viewport"]');
  meta.setAttribute('content',
    `width=device-width, initial-scale=${scale}, maximum-scale=${scale}, minimum-scale=${scale}, user-scalable=no`
  );
  document.documentElement.style.fontSize = `${document.documentElement.clientWidth / 10}px`;
  document.documentElement.setAttribute('data-dpr', dpr);
}
```

### rem 适配方案

**原理**：rem 单位相对于根元素 (`<html>`) 的 `font-size`，通过动态设置根字号实现等比缩放。

```javascript
// 经典 rem 方案（flexible.js 简化版）
(function flexible(window, document) {
  const docEl = document.documentElement;
  const dpr = window.devicePixelRatio || 1;

  // 设置 body 字体大小
  function setBodyFontSize() {
    if (document.body) {
      document.body.style.fontSize = 12 * dpr + 'px';
    } else {
      document.addEventListener('DOMContentLoaded', setBodyFontSize);
    }
  }
  setBodyFontSize();

  // 设置 1rem = viewWidth / 10
  function setRemUnit() {
    const rem = docEl.clientWidth / 10;
    docEl.style.fontSize = rem + 'px';
  }
  setRemUnit();

  // 页面 resize 或旋转时重新计算
  window.addEventListener('resize', setRemUnit);
  window.addEventListener('pageshow', function (e) {
    if (e.persisted) setRemUnit();
  });
})(window, document);
```

**PostCSS 自动转换 px → rem**

```javascript
// postcss.config.js
module.exports = {
  plugins: {
    'postcss-pxtorem': {
      rootValue: 37.5,       // 设计稿 375px / 10
      propList: ['*'],       // 所有属性都转换
      selectorBlackList: ['.norem'], // 排除类名
      minPixelValue: 1,      // 最小转换像素
    },
  },
};
```

```css
/* 输入 */
.box {
  width: 100px;
  height: 50px;
  font-size: 14px;
}

/* 输出（rootValue = 37.5） */
.box {
  width: 2.66667rem;
  height: 1.33333rem;
  font-size: 0.37333rem;
}
```

### vw/vh 适配方案

**原理**：`1vw = 视口宽度的 1%`，无需 JS 计算根字号，纯 CSS 方案。

```css
/* 设计稿 375px 下：1px = 100/375 vw ≈ 0.2667vw */
.box {
  width: 26.667vw;   /* 100px */
  height: 13.333vw;  /* 50px */
  font-size: 3.733vw; /* 14px */
}
```

**PostCSS 自动转换 px → vw**

```javascript
// postcss.config.js
module.exports = {
  plugins: {
    'postcss-px-to-viewport': {
      viewportWidth: 375,     // 设计稿宽度
      unitPrecision: 5,       // 精度
      viewportUnit: 'vw',     // 转换单位
      selectorBlackList: ['.novw'],
      minPixelValue: 1,
      mediaQuery: false,
      landscape: false,       // 是否处理横屏
    },
  },
};
```

**rem vs vw 对比**

| 维度 | rem | vw |
|------|-----|-----|
| 原理 | 根字号 + JS 动态计算 | 视口百分比，纯 CSS |
| JS 依赖 | 需要 flexible.js | 无需 JS |
| 兼容性 | iOS 6+ / Android 2.1+ | iOS 8+ / Android 4.4+ |
| 精度 | 存在小数四舍五入误差 | 更精确 |
| 横屏适配 | 监听 resize 重新计算 | 自动适应 |
| 最大宽度限制 | 可通过 JS 限制 | 需要 `@media` + `max-width` |
| 业界趋势 | 逐渐被 vw 取代 | ⭐ 推荐方案 |

### 1px 边框问题

**问题根源**：在 DPR=2 的设备上，CSS 的 `1px` 实际渲染为 2 个物理像素，看起来偏粗。

**方案一：transform scale（⭐ 推荐）**

```css
/* 单条边框 */
.border-bottom-1px {
  position: relative;
}
.border-bottom-1px::after {
  content: '';
  position: absolute;
  left: 0;
  bottom: 0;
  width: 100%;
  height: 1px;
  background-color: #e5e5e5;
  transform: scaleY(0.5);
  transform-origin: 0 0;
}

/* 四条边框 */
.border-1px {
  position: relative;
}
.border-1px::after {
  content: '';
  position: absolute;
  top: 0;
  left: 0;
  width: 200%;
  height: 200%;
  border: 1px solid #e5e5e5;
  border-radius: 8px;
  transform: scale(0.5);
  transform-origin: 0 0;
  pointer-events: none;
  box-sizing: border-box;
}

/* 根据 DPR 动态调整 */
@media (-webkit-min-device-pixel-ratio: 2), (min-resolution: 2dppx) {
  .border-bottom-1px::after {
    transform: scaleY(0.5);
  }
}
@media (-webkit-min-device-pixel-ratio: 3), (min-resolution: 3dppx) {
  .border-bottom-1px::after {
    transform: scaleY(0.333);
  }
}
```

**方案二：viewport + rem（整体缩放）**

```javascript
// 动态设置 viewport 的 scale = 1/dpr
const dpr = window.devicePixelRatio;
const scale = 1 / dpr;
document.querySelector('meta[name="viewport"]').setAttribute('content',
  `width=device-width, initial-scale=${scale}, maximum-scale=${scale}`
);
// 此时写 1px 就是真正的 1 物理像素
```

**方案三：box-shadow 模拟**

```css
.border-1px {
  box-shadow: 0 1px 1px -1px rgba(0, 0, 0, 0.5);
}
```

**方案四：SVG border-image**

```css
.border-1px {
  border-width: 1px;
  border-image: url("data:image/svg+xml,<svg xmlns='http://www.w3.org/2000/svg' width='1' height='1'><rect width='1' height='1' fill='%23e5e5e5'/></svg>") 1 stretch;
}
```

**1px 方案对比表**

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| transform scale | 通用性强，兼容好 | 需要伪元素 | ⭐⭐⭐ |
| viewport 缩放 | 直接写 1px | 影响全局布局 | ⭐⭐ |
| box-shadow | 实现简单 | 颜色偏淡，不精确 | ⭐ |
| SVG border-image | 精确 | 兼容性一般，改色麻烦 | ⭐⭐ |

## 移动端手势与交互

> ⭐⭐ 触摸事件模型 + 300ms 延迟优化 + 手势识别 + 常见交互组件。

### Touch 事件体系

**事件类型与触发顺序**

```
touchstart → touchmove → touchend → (300ms) → click
```

```javascript
const el = document.getElementById('target');

el.addEventListener('touchstart', (e) => {
  const touch = e.touches[0]; // 当前屏幕上所有触摸点
  console.log('起始坐标:', touch.clientX, touch.clientY);
  console.log('触摸点数:', e.touches.length);
}, { passive: true }); // passive 提升滚动性能

el.addEventListener('touchmove', (e) => {
  const touch = e.changedTouches[0];
  console.log('移动到:', touch.clientX, touch.clientY);
  // 如果需要阻止滚动, 不能使用 passive
  // e.preventDefault();
}, { passive: true });

el.addEventListener('touchend', (e) => {
  const touch = e.changedTouches[0]; // touchend 时 touches 为空
  console.log('结束坐标:', touch.clientX, touch.clientY);
});

el.addEventListener('touchcancel', (e) => {
  console.log('触摸中断（如来电）');
});
```

**Touch 对象属性**

| 属性 | 说明 |
|------|------|
| `touches` | 当前屏幕上所有触摸点 |
| `targetTouches` | 当前元素上的触摸点 |
| `changedTouches` | 触发当前事件的触摸点（touchend 必须用这个） |
| `touch.clientX/Y` | 相对视口坐标 |
| `touch.pageX/Y` | 相对文档坐标（含滚动） |
| `touch.identifier` | 触摸点唯一标识（多指触控） |

---

### 300ms 延迟问题

**原因**：移动端浏览器通过等待 300ms 判断用户是否双击缩放（double-tap-to-zoom）。

**解决方案一：meta viewport 禁止缩放（现代方案 ⭐）**

```html
<!-- Chrome 32+ / iOS 9.3+ 已原生消除 300ms 延迟 -->
<meta name="viewport" content="width=device-width, initial-scale=1.0">
```

**解决方案二：touch-action CSS**

```css
/* 告诉浏览器不需要处理双击缩放，直接触发 click */
.btn {
  touch-action: manipulation;
}

/* 或禁止所有默认触摸行为 */
html {
  touch-action: manipulation;
}
```

**解决方案三：FastClick（legacy 方案）**

```javascript
// 原理：在 touchend 时立即派发 click 事件，并阻止原生 click
// FastClick 核心思路简化版
class SimpleFastClick {
  constructor(layer) {
    this.trackingClick = false;
    this.touchStartX = 0;
    this.touchStartY = 0;

    layer.addEventListener('touchstart', this.onTouchStart.bind(this), false);
    layer.addEventListener('touchend', this.onTouchEnd.bind(this), false);
  }

  onTouchStart(e) {
    const touch = e.targetTouches[0];
    this.trackingClick = true;
    this.touchStartX = touch.pageX;
    this.touchStartY = touch.pageY;
    this.touchStartTime = Date.now();
  }

  onTouchEnd(e) {
    if (!this.trackingClick) return;

    // 如果手指移动超过阈值则不算点击
    const touch = e.changedTouches[0];
    const moveX = Math.abs(touch.pageX - this.touchStartX);
    const moveY = Math.abs(touch.pageY - this.touchStartY);
    if (moveX > 10 || moveY > 10) {
      this.trackingClick = false;
      return;
    }

    // 创建并派发模拟 click 事件
    const clickEvent = new MouseEvent('click', {
      bubbles: true,
      cancelable: true,
      view: window,
    });
    e.target.dispatchEvent(clickEvent);

    // 阻止原生 click（300ms 后触发的那个）
    e.preventDefault();
    this.trackingClick = false;
  }
}

// 使用
new SimpleFastClick(document.body);
```

### 手势识别实现

```javascript
// 通用手势识别器
class GestureRecognizer {
  constructor(el, options = {}) {
    this.el = el;
    this.options = {
      swipeThreshold: 50,
      swipeTimeout: 300,
      longPressTimeout: 750,
      doubleTapTimeout: 300,
      ...options,
    };

    this.startX = 0;
    this.startY = 0;
    this.startTime = 0;
    this.lastTapTime = 0;
    this.longPressTimer = null;

    this.bindEvents();
  }

  bindEvents() {
    this.el.addEventListener('touchstart', this.onStart.bind(this), { passive: false });
    this.el.addEventListener('touchmove', this.onMove.bind(this), { passive: false });
    this.el.addEventListener('touchend', this.onEnd.bind(this));
  }

  onStart(e) {
    const touch = e.touches[0];
    this.startX = touch.clientX;
    this.startY = touch.clientY;
    this.startTime = Date.now();
    this.moved = false;

    // 长按检测
    this.longPressTimer = setTimeout(() => {
      if (!this.moved) {
        this.emit('longpress', { x: this.startX, y: this.startY });
      }
    }, this.options.longPressTimeout);
  }

  onMove(e) {
    const touch = e.touches[0];
    const dx = touch.clientX - this.startX;
    const dy = touch.clientY - this.startY;
    if (Math.abs(dx) > 10 || Math.abs(dy) > 10) {
      this.moved = true;
      clearTimeout(this.longPressTimer);
    }
  }

  onEnd(e) {
    clearTimeout(this.longPressTimer);
    const touch = e.changedTouches[0];
    const dx = touch.clientX - this.startX;
    const dy = touch.clientY - this.startY;
    const dt = Date.now() - this.startTime;

    // 滑动检测
    if (dt < this.options.swipeTimeout && (Math.abs(dx) > this.options.swipeThreshold || Math.abs(dy) > this.options.swipeThreshold)) {
      const direction = Math.abs(dx) > Math.abs(dy)
        ? (dx > 0 ? 'right' : 'left')
        : (dy > 0 ? 'down' : 'up');
      this.emit('swipe', { direction, dx, dy });
    }

    // 双击检测
    if (!this.moved && dt < 200) {
      const now = Date.now();
      if (now - this.lastTapTime < this.options.doubleTapTimeout) {
        this.emit('doubletap', { x: touch.clientX, y: touch.clientY });
        this.lastTapTime = 0;
      } else {
        this.lastTapTime = now;
      }
    }
  }

  emit(type, detail) {
    this.el.dispatchEvent(new CustomEvent(type, { detail, bubbles: true }));
  }
}

// 使用
const gesture = new GestureRecognizer(document.getElementById('app'));
document.getElementById('app').addEventListener('swipe', (e) => {
  console.log('滑动方向:', e.detail.direction);
});
```

### 下拉刷新 / 上拉加载

```javascript
// 下拉刷新核心实现
class PullToRefresh {
  constructor(container, onRefresh) {
    this.container = container;
    this.onRefresh = onRefresh;
    this.startY = 0;
    this.pulling = false;
    this.threshold = 80; // 触发刷新的阈值

    // 创建提示元素
    this.indicator = document.createElement('div');
    this.indicator.className = 'pull-indicator';
    this.indicator.textContent = '下拉刷新';
    container.insertBefore(this.indicator, container.firstChild);

    this.bindEvents();
  }

  bindEvents() {
    this.container.addEventListener('touchstart', (e) => {
      if (this.container.scrollTop === 0) {
        this.startY = e.touches[0].clientY;
        this.pulling = true;
      }
    }, { passive: true });

    this.container.addEventListener('touchmove', (e) => {
      if (!this.pulling) return;
      const dy = e.touches[0].clientY - this.startY;
      if (dy > 0) {
        // 阻尼系数，越拉越难拉
        const distance = Math.min(dy * 0.4, 120);
        this.indicator.style.height = distance + 'px';
        this.indicator.textContent = distance > this.threshold ? '释放刷新' : '下拉刷新';
      }
    });

    this.container.addEventListener('touchend', async () => {
      if (!this.pulling) return;
      this.pulling = false;
      const height = parseInt(this.indicator.style.height) || 0;
      if (height > this.threshold) {
        this.indicator.textContent = '刷新中...';
        this.indicator.style.height = '50px';
        await this.onRefresh();
        this.indicator.textContent = '刷新完成';
      }
      setTimeout(() => { this.indicator.style.height = '0'; }, 300);
    });
  }
}

// 上拉加载（Intersection Observer 方案 ⭐）
class InfiniteScroll {
  constructor(sentinel, onLoadMore) {
    this.loading = false;
    this.observer = new IntersectionObserver(async (entries) => {
      if (entries[0].isIntersecting && !this.loading) {
        this.loading = true;
        await onLoadMore();
        this.loading = false;
      }
    }, { rootMargin: '200px' }); // 提前 200px 触发
    this.observer.observe(sentinel);
  }

  destroy() {
    this.observer.disconnect();
  }
}
```

## Hybrid App

> ⭐⭐⭐ Hybrid 是移动端面试的重点考查方向：WebView 容器 + JSBridge 通信 + 离线包方案。

### WebView 原理

**WebView 本质**：原生应用内嵌的浏览器内核组件。

| 平台 | WebView 内核 | 说明 |
|------|-------------|------|
| iOS | WKWebView | 基于 WebKit，独立进程，性能好 |
| iOS (旧) | UIWebView | 已废弃，与 App 同进程 |
| Android | Chromium WebView | Android 5.0+ 可独立更新 |
| Android (旧) | Android WebKit | 4.4 以下，性能差 |

**WKWebView vs UIWebView**

| 维度 | WKWebView | UIWebView |
|------|-----------|-----------|
| 进程模型 | 多进程（独立渲染进程） | 单进程（与 App 同进程） |
| JS 引擎 | Nitro (JIT) | JavaScriptCore (无 JIT) |

### JSBridge 通信原理

**核心需求**：JS ↔ Native 双向通信。

```
┌─────────────┐          JSBridge           ┌─────────────┐
│   WebView   │  ◄───────────────────────►  │   Native    │
│  (JS 环境)   │                             │  (OC/Swift  │
│             │   JS 调用 Native              │   /Java)    │
│             │   Native 调用 JS              │             │
└─────────────┘                             └─────────────┘
```

**JS 调用 Native 的方式**

| 方式 | 原理 | 优缺点 |
|------|------|--------|
| URL Scheme 拦截 | `iframe.src = 'jsbridge://method?params'`，Native 拦截 URL | 兼容性好，但长度限制、编码问题 |
| 注入全局 API | Native 向 JS 注入全局方法 | ⭐ 推荐，直接调用，支持回调 |
| MessageHandler (iOS) | `window.webkit.messageHandlers.xxx.postMessage()` | WKWebView 官方 API |

**方式一：URL Scheme 拦截**

```javascript
// JS 端
function callNative(method, params, callback) {
  const callbackId = 'cb_' + Date.now() + '_' + Math.random().toString(36).slice(2);

  // 注册回调
  window.__jsBridgeCallbacks = window.__jsBridgeCallbacks || {};
  window.__jsBridgeCallbacks[callbackId] = callback;

  // 构造 URL
  const url = `jsbridge://${method}?params=${encodeURIComponent(JSON.stringify(params))}&callbackId=${callbackId}`;

  // 通过 iframe 触发
  const iframe = document.createElement('iframe');
  iframe.style.display = 'none';
  iframe.src = url;
  document.body.appendChild(iframe);
  setTimeout(() => document.body.removeChild(iframe), 100);
}

// 使用
callNative('share', { title: '标题', url: 'https://example.com' }, (result) => {
  console.log('分享结果:', result);
});
```

**方式二：注入全局 API（⭐ 推荐）**

```javascript
// Native 注入到 JS 的 bridge 对象
// JS 端封装
class JSBridge {
  static callbackMap = new Map();
  static callbackId = 0;

  static call(method, params = {}) {
    return new Promise((resolve, reject) => {
      const id = ++this.callbackId;
      this.callbackMap.set(id, { resolve, reject });

      const message = JSON.stringify({ method, params, callbackId: id });

      // iOS WKWebView
      if (window.webkit?.messageHandlers?.nativeBridge) {
        window.webkit.messageHandlers.nativeBridge.postMessage(message);
      }
      // Android
      else if (window.NativeBridge) {
        window.NativeBridge.postMessage(message);
      }
    });
  }

  // Native 回调此方法
  static onCallback(callbackId, result) {
    const callback = this.callbackMap.get(callbackId);
    if (callback) {
      if (result.code === 0) {
        callback.resolve(result.data);
      } else {
        callback.reject(result);
      }
      this.callbackMap.delete(callbackId);
    }
  }

  // Native 主动调用 JS
  static registerHandler(name, handler) {
    window.__jsHandlers = window.__jsHandlers || {};
    window.__jsHandlers[name] = handler;
  }
}

// 使用示例
async function onShareClick() {
  try {
    const result = await JSBridge.call('share', {
      title: '分享标题',
      content: '分享内容',
      url: location.href,
    });
    console.log('分享成功', result);
  } catch (e) {
    console.error('分享失败', e);
  }
}

// 注册 Native → JS 的回调
JSBridge.registerHandler('onPageVisible', (data) => {
  console.log('页面可见', data);
  // 刷新数据等
});
```

### JSI (JavaScript Interface)

**核心思想**：JS 通过 C++ 层直接持有 Native 对象引用，无需 Bridge 序列化。

```cpp
// C++ 层 JSI Host Object 示例
class NativeStorage : public jsi::HostObject {
public:
  jsi::Value get(jsi::Runtime& rt, const jsi::PropNameID& name) override {
    auto propName = name.utf8(rt);
    if (propName == "getItem") {
      return jsi::Function::createFromHostFunction(rt, name, 1,
        [](jsi::Runtime& rt, const jsi::Value& thisVal,
           const jsi::Value* args, size_t count) -> jsi::Value {
          std::string key = args[0].getString(rt).utf8(rt);
          // 直接同步调用 Native 存储
          std::string value = NativeStorageImpl::getItem(key);
          return jsi::String::createFromUtf8(rt, value);
        }
      );
    }
    return jsi::Value::undefined();
  }
};
```

```javascript
      { translateY: translateY.value },
    ],
  }));
```

### 渲染管线

```

Widget Tree → Element Tree → RenderObject Tree
(配置描述)     (生命周期管理)    (布局+绘制)

1. Build   : Widget → Element（diff 更新）
```

```dart
  CirclePainter(this.progress);

  @override
  void paint(Canvas canvas, Size size) {
    final center = Offset(size.width / 2, size.height / 2);
    final radius = size.width / 2;

    // 背景圆
    canvas.drawCircle(center, radius,
```

|-------------|------|---------|

```dart
// Dart 端
class BatteryService {
  static const _methodChannel = MethodChannel('com.example/battery');
  static const _eventChannel = EventChannel('com.example/battery_stream');

  // 方法调用
  static Future<int> getBatteryLevel() async {
    final level = await _methodChannel.invokeMethod<int>('getBatteryLevel');
    return level ?? -1;
  }

  // 事件流
  static Stream<int> batteryLevelStream() {
    return _eventChannel.receiveBroadcastStream().map((level) => level as int);
  }
}
```

```kotlin
// Android 端 (Kotlin)
class MainActivity : FlutterActivity() {
    private val CHANNEL = "com.example/battery"
    private val EVENT_CHANNEL = "com.example/battery_stream"

    override fun configureFlutterEngine(flutterEngine: FlutterEngine) {
        super.configureFlutterEngine(flutterEngine)

        MethodChannel(flutterEngine.dartExecutor.binaryMessenger, CHANNEL)
            .setMethodCallHandler { call, result ->
                when (call.method) {
                    "getBatteryLevel" -> {
                        val level = getBatteryLevel()
                        if (level != -1) result.success(level)
                        else result.error("UNAVAILABLE", "Battery level unavailable", null)
                    }
                    else -> result.notImplemented()
                }
            }
    }

    private fun getBatteryLevel(): Int {
        val batteryManager = getSystemService(BATTERY_SERVICE) as BatteryManager
        return batteryManager.getIntProperty(BatteryManager.BATTERY_PROPERTY_CAPACITY)
    }
}
```

### 架构原理

**双线程模型**

```
┌──────────────────────────────────────────────────┐
│                 微信小程序架构                       │
│                                                  │
│  ┌─────────────┐           ┌─────────────┐       │
│  │  逻辑层       │  Native   │  渲染层       │       │
│  │  (JsCore)    │◄────────►│  (WebView)   │       │
│  │             │  转发消息   │              │       │
│  │  App / Page  │           │  WXML + WXSS │       │
│  │  逻辑代码     │           │  页面渲染     │       │
│  └─────────────┘           └─────────────┘       │
│         │                         │               │
│         ▼                         ▼               │
│  ┌──────────────────────────────────────┐         │
│  │           Native 层                   │         │
│  │  网络请求 / 存储 / 相机 / 定位等       │         │
│  └──────────────────────────────────────┘         │
└──────────────────────────────────────────────────┘
```

**为什么是双线程？**

| 设计考量 | 说明 |
|---------|------|
| 安全 | 逻辑层无法直接操作 DOM，避免恶意代码 |
| 性能 | 渲染和逻辑并行，不互相阻塞 |
| 管控 | 平台可审核和控制页面行为 |
| 代价 | 通信有延迟，setData 是性能瓶颈 |

**通信流程**

```javascript
// setData 通信原理（简化）
Page({
  data: { count: 0 },

  onTap() {
    // 1. JS 线程：数据变更
    this.setData({ count: this.data.count + 1 });
    // 2. 序列化 → Native 转发 → 反序列化
    // 3. WebView：虚拟 DOM diff → 更新视图
  },
});

// ⚠️ setData 性能陷阱
// 错误：传输大量数据
this.setData({ hugeList: this.data.hugeList }); // 整个列表序列化传输！

// 正确：精准更新路径
this.setData({ 'list[2].name': '新名称' }); // 只传增量
```

### 离线缓存策略

```javascript
// Service Worker 缓存策略
// sw.js
const CACHE_NAME = 'app-v2';
const PRECACHE_URLS = ['/index.html', '/static/css/main.css', '/static/js/main.js'];

// 预缓存
self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME).then((cache) => cache.addAll(PRECACHE_URLS))
  );
});

// 请求拦截 — Stale While Revalidate
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.open(CACHE_NAME).then(async (cache) => {
      const cachedResponse = await cache.match(event.request);
      const fetchPromise = fetch(event.request).then((response) => {
        if (response.ok) cache.put(event.request, response.clone());
        return response;
      }).catch(() => cachedResponse); // 离线降级

      return cachedResponse || fetchPromise;
    })
  );
});
```

**缓存策略对比**

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| Cache First | 优先缓存，无则网络 | 静态资源 (CSS/JS/图片) |
| Network First | 优先网络，失败用缓存 | API 接口 |
| Stale While Revalidate | 返回缓存同时后台更新 | ⭐ 频繁更新的资源 |

```javascript
// 路由中心 — 统一管理 Native / RN / H5 页面跳转
class UnifiedRouter {
  constructor() {
    this.routeMap = new Map();
  }

  // 注册路由
  register(path, config) {
    // config: { type: 'native' | 'rn' | 'h5', url?, module?, fallback? }
    this.routeMap.set(path, config);
  }

  // 统一跳转
  async navigate(path, params = {}) {
    const config = this.routeMap.get(path);
    if (!config) {
      console.warn(`未注册的路由: ${path}`);
      return;
    }

    switch (config.type) {
      case 'native':
        await JSBridge.call('openNativePage', { module: config.module, params });
        break;

      case 'rn':
        await JSBridge.call('openRNPage', { module: config.module, params });
        break;

      case 'h5':
        const url = new URL(config.url);
        Object.entries(params).forEach(([k, v]) => url.searchParams.set(k, v));

        // 判断是否有离线包
        const hasOffline = await JSBridge.call('checkOfflinePackage', { bizId: config.bizId });
        if (hasOffline) {
          await JSBridge.call('openWebView', { url: url.toString(), useOffline: true });
        } else {
          await JSBridge.call('openWebView', { url: url.toString() });
        }
        break;
    }
  }
}
```

### 初级高频速刷（5 题）

**Q1：什么是 Viewport？移动端为什么要设置 `meta viewport`？**

A：Viewport 是浏览器显示网页的可视区域。移动端默认布局视口约 980px，远大于屏幕宽度，导致页面缩小显示。通过 `<meta name="viewport" content="width=device-width, initial-scale=1.0">` 将布局视口设为设备宽度，实现正确的移动端适配。

**Q2：rem 和 vw 的区别？**

A：rem 相对于根元素 `font-size`，需要 JS 动态设置基准值；vw 是视口宽度的百分比（1vw = 1%），纯 CSS 方案无需 JS。vw 更推荐，但 rem 兼容性更好（支持 iOS 6+）。

**Q3：什么是 300ms 延迟？如何解决？**

A：移动浏览器为判断双击缩放，在 `touchend` 后等待 300ms 才触发 `click`。现代方案：设置 `width=device-width` 的 viewport 或 CSS `touch-action: manipulation` 即可消除。

**Q4：小程序为什么采用双线程架构？**

A：安全（逻辑层无 DOM 访问权限）+ 管控（平台可审核代码行为）+ 性能（渲染与逻辑并行）。代价是 `setData` 通信有序列化开销。

**Q5：Electron 为什么体积大？Tauri 为什么小？**

A：Electron 内嵌完整 Chromium + Node.js（~80-150MB），Tauri 使用系统 WebView + Rust 后端（~2-10MB），不打包浏览器引擎。

### 中级高频进阶（5 题）

**Q1：如何实现真正的 1px 边框？对比各方案优缺点。**

A：核心原因是 DPR > 1 时 CSS 1px 对应多个物理像素。推荐方案：伪元素 + `transform: scaleY(0.5)`，兼容性好、可精确控制；viewport 缩放方案会影响全局布局；`box-shadow` 颜色不准。具体实现见上文代码。

**Q2：JSBridge 有哪些实现方式？各自的优缺点？**

A：①URL Scheme 拦截（iframe.src）：兼容性好但有长度限制；②注入全局 API（推荐）：Native 向 JS 注入对象，直接调用，支持 Promise 回调；③WKWebView MessageHandler：iOS 官方 API，安全可靠。生产环境推荐方式②。

**Q3：React Native 新旧架构的核心区别？**

A：旧架构通过 Bridge（JSON 异步序列化）通信，有性能瓶颈。新架构引入 JSI（C++ 层直接引用 Native 对象，同步调用）、Fabric（C++ 共享 Shadow Tree，支持并发渲染）、TurboModules（懒加载 Native 模块）。消除了 Bridge 瓶颈。

**Q4：Flutter 的三棵树是什么？为什么需要三棵树？**

A：Widget Tree（配置描述，不可变，轻量重建）→ Element Tree（生命周期管理，diff 复用）→ RenderObject Tree（布局绘制，重量级）。分离关注点：Widget 声明式描述 UI，Element 管理状态和复用，RenderObject 负责昂贵的布局绘制操作。

**Q5：小程序 setData 为什么慢？如何优化？**

A：setData 经历：JS 线程序列化 → Native 转发 → WebView 反序列化 → diff → 渲染。优化：①路径更新替代全量 `setData({ 'list[0].name': 'xxx' })`；②合并多次 setData；③减少不必要数据传输；④用 WXS 在渲染层直接处理简单逻辑。

**Q3：实现一个完整的 JSBridge SDK，支持双向通信和超时处理。**

A：

```javascript
class JSBridgeSDK {
  static instance = null;
  callbackId = 0;
  callbacks = new Map();
  handlers = new Map();
  messageQueue = [];
  ready = false;

  static getInstance() {
    if (!JSBridgeSDK.instance) {
      JSBridgeSDK.instance = new JSBridgeSDK();
    }
    return JSBridgeSDK.instance;
  }

  constructor() {
    this._initReceiver();
    this._checkReady();
  }

  // JS 调用 Native
  call(method, params = {}, timeout = 10000) {
    return new Promise((resolve, reject) => {
      const id = ++this.callbackId;
      this.callbacks.set(id, { resolve, reject });

      // 超时处理
      const timer = setTimeout(() => {
        if (this.callbacks.has(id)) {
          this.callbacks.get(id).reject(new Error(`JSBridge call "${method}" timeout`));
          this.callbacks.delete(id);
        }
      }, timeout);

      const message = { method, params, callbackId: id };

      if (!this.ready) {
        this.messageQueue.push(message);
        return;
      }

      this._postMessage(message);

      // 更新 callback 加上 timer 引用
      const cb = this.callbacks.get(id);
      cb.timer = timer;
    });
  }

  // 注册 JS 供 Native 调用的方法
  register(name, handler) {
    this.handlers.set(name, handler);
  }

  // Native 回调 JS
  _onCallback(callbackId, response) {
    const cb = this.callbacks.get(callbackId);
    if (!cb) return;
    clearTimeout(cb.timer);
    if (response.code === 0) {
      cb.resolve(response.data);
    } else {
      cb.reject(response);
    }
    this.callbacks.delete(callbackId);
  }

  // Native 主动调用 JS
  _onNativeCall(name, params, nativeCallbackId) {
    const handler = this.handlers.get(name);
    if (handler) {
      Promise.resolve(handler(params)).then((result) => {
        this._postMessage({
          responseId: nativeCallbackId,
          data: result,
        });
      });
    }
  }

  _postMessage(message) {
    const msg = JSON.stringify(message);
    if (window.webkit?.messageHandlers?.bridge) {
      window.webkit.messageHandlers.bridge.postMessage(msg);
    } else if (window.NativeBridge) {
      window.NativeBridge.postMessage(msg);
    }
  }

  _initReceiver() {
    window.__jsBridgeReceive = (jsonStr) => {
      const msg = JSON.parse(jsonStr);
      if (msg.callbackId !== undefined) {
        this._onCallback(msg.callbackId, msg);
      } else if (msg.method) {
        this._onNativeCall(msg.method, msg.params, msg.nativeCallbackId);
      }
    };
  }

  _checkReady() {
    if (window.__NATIVE_BRIDGE_READY__) {
      this._onReady();
    } else {
      document.addEventListener('NativeBridgeReady', () => this._onReady());
    }
  }

  _onReady() {
    this.ready = true;
    this.messageQueue.forEach((msg) => this._postMessage(msg));
    this.messageQueue = [];
  }
}

// 使用
const bridge = JSBridgeSDK.getInstance();
bridge.register('refreshPage', () => location.reload());
const userInfo = await bridge.call('getUserInfo');
```

**Q4：React Native 的 JSI 解决了什么问题？用代码说明新旧架构的差异。**

A：

旧架构的 Bridge 问题：
1. **异步**：所有 JS↔Native 通信都是异步的，无法同步获取 Native 数据
2. **序列化**：JSON 序列化/反序列化开销大，尤其传输大数据
3. **排队**：所有消息共享一条 Bridge，高频消息（如手势）可能拥堵

```javascript
// 旧架构 — 经过 Bridge
// JS: "我要获取设备信息"
// → JSON.stringify → Bridge Queue → JSON.parse
// Native 处理
// → JSON.stringify → Bridge Queue → JSON.parse
// JS: 收到结果 (异步 callback / Promise)

import { NativeModules } from 'react-native';
const result = await NativeModules.DeviceInfo.getInfo(); // 异步！

// 新架构 — JSI 直调
// JS 通过 C++ 持有 Native 对象的引用（HostObject）
// 直接调用，无序列化，可同步返回

// C++ 层暴露的 HostObject，JS 可以直接调用
const info = global.__deviceInfo.getInfoSync(); // 同步！
```

JSI 的本质：JS 引擎（Hermes）通过 C++ 中间层直接操作 Native 对象的引用，绕开 Bridge 和 JSON 序列化。

**Q5：Flutter 的 Platform Channel 通信是同步还是异步？如何处理大量数据传输的性能问题？**

A：Platform Channel 通信是**异步**的（基于消息传递）。

大数据量优化策略：
1. **使用 BasicMessageChannel + BinaryCodec**：传输二进制数据避免编解码开销
2. **共享内存**：通过 FFI（Dart Foreign Function Interface）直接访问 C/C++ 内存
3. **分片传输**：将大数据分块传输
4. **Pigeon 代码生成**：类型安全的通信，减少手动编解码

```dart
// 使用 FFI 处理高性能场景（绕过 Channel）
import 'dart:ffi';
import 'package:ffi/ffi.dart';

// 加载 Native 库
final dylib = DynamicLibrary.open('libnative_processing.so');
final processImage = dylib.lookupFunction<
  Pointer<Uint8> Function(Pointer<Uint8>, Int32),
  Pointer<Uint8> Function(Pointer<Uint8>, int)
>('process_image');

// 直接操作内存，无 Channel 开销
Uint8List processImageData(Uint8List input) {
  final ptr = malloc<Uint8>(input.length);
  ptr.asTypedList(input.length).setAll(0, input);
  final result = processImage(ptr, input.length);
  final output = result.asTypedList(input.length).toList();
  malloc.free(ptr);
  return Uint8List.fromList(output);
}
```

**Q8：RN 热更新的原理是什么？和 Native 发版有什么区别？为什么 Flutter 不容易做热更新？**

A：

**RN 热更新原理**：RN 的 UI 由 JS Bundle 驱动。热更新 = 替换 JS Bundle 文件，App 重启后加载新 Bundle 即可。JS 是解释执行的，天然支持动态替换。

```
旧 Bundle: main.bundle.js (v1.0)
   ↓ 下载增量包
新 Bundle: main.bundle.js (v1.1)
   ↓ App 重启
加载新 Bundle → 新 UI
```

**Flutter 为什么难以热更新**：
- Flutter 使用 Dart AOT 编译为机器码（ARM binary），嵌入到 App 二进制中
- 替换机器码 = 替换可执行文件，等同于重新安装 App
- iOS 明确禁止动态下载可执行代码
- 社区方案（如 Shorebird）通过 Dart 补丁方式实现，但有限制

| 维度 | RN 热更新 | Flutter "热更新" | Native 发版 |
|------|----------|----------------|------------|
| 更新内容 | JS Bundle | Dart 补丁 (受限) | 完整 App |
| 审核 | 无需审核 | 无需审核 | 需商店审核 |
| 更新速度 | 秒级 | 秒级 | 天级 |
| 更新范围 | JS 逻辑+样式 | Dart 逻辑 | 全部 |
| 原生模块 | 无法更新 | 无法更新 | 可更新 |

**Q9：如何设计一个离线包管理系统？包括版本管理、增量更新、容灾降级。**

A：

```javascript
// 离线包管理系统设计
class OfflinePackageManager {
  constructor() {
    this.localVersions = new Map(); // bizId → version
    this.downloading = new Set();
  }

  // 1. 启动时检查更新
  async checkUpdate() {
    const config = await fetch('https://api.example.com/offline/config').then(r => r.json());

    for (const pkg of config.packages) {
      const localVersion = this.localVersions.get(pkg.bizId);

      if (!localVersion) {
        // 全量下载
        await this.downloadFull(pkg);
      } else if (localVersion !== pkg.version && pkg.patchUrl) {
        // 增量更新（bsdiff 补丁）
        await this.downloadPatch(pkg, localVersion);
      }
    }
  }

  // 2. 增量下载
  async downloadPatch(pkg, fromVersion) {
    try {
      const patch = await fetch(pkg.patchUrl).then(r => r.arrayBuffer());
      // 校验 MD5
      if (md5(patch) !== pkg.patchMd5) throw new Error('Patch MD5 mismatch');
      // Native 合并补丁
      await JSBridge.call('applyBsdiffPatch', {
        bizId: pkg.bizId,
        patch: patch,
        fromVersion: fromVersion,
        toVersion: pkg.version,
      });
      this.localVersions.set(pkg.bizId, pkg.version);
    } catch (e) {
      // 增量失败回退全量
      console.warn('Patch failed, fallback to full:', e);
      await this.downloadFull(pkg);
    }
  }

  // 3. 容灾降级
  async loadResource(bizId, path) {
    // 优先本地离线包
    const localPath = await JSBridge.call('getOfflineResource', { bizId, path });
    if (localPath) return localPath;

    // 降级到 CDN
    console.warn(`Offline miss: ${bizId}/${path}, fallback to CDN`);
    return `https://cdn.example.com/${bizId}/${path}`;
  }
}
```

关键设计要点：
- **增量更新**：使用 bsdiff 算法生成补丁包（通常只有全量包 10-20% 大小）
- **MD5 校验**：防止下载损坏或篡改
- **容灾降级**：离线包加载失败自动回退在线资源
- **灰度发布**：配置中心支持按用户百分比、设备类型、地域灰度

**Q10：移动端长列表为什么需要虚拟滚动？如何处理动态高度的虚拟列表？**

A：

**为什么需要虚拟滚动**：10000 条数据 → 10000 个 DOM 节点，导致：
- 内存占用高（低端机 OOM）
- 渲染/重排耗时长
- 滚动卡顿

**虚拟滚动原理**：只渲染可视区域 + 上下缓冲区的 DOM 节点（通常 20-50 个），通过 `transform` 模拟完整滚动。

**动态高度处理**：

```javascript
// 动态高度虚拟列表核心思路
class DynamicVirtualList {
  constructor(container, estimatedHeight = 80) {
    this.container = container;
    this.estimatedHeight = estimatedHeight;
    this.heights = new Map(); // 缓存实际高度
    this.positions = [];      // 每项的 top/bottom
  }

  // 初始化位置（使用预估高度）
  initPositions(count) {
    this.positions = Array.from({ length: count }, (_, i) => ({
      index: i,
      top: i * this.estimatedHeight,
      bottom: (i + 1) * this.estimatedHeight,
      height: this.estimatedHeight,
    }));
  }

  // 渲染后用 ResizeObserver 获取实际高度
  observeItems() {
    const observer = new ResizeObserver((entries) => {
      for (const entry of entries) {
        const index = Number(entry.target.dataset.index);
        const newHeight = entry.contentRect.height;
        const oldHeight = this.positions[index].height;

        if (Math.abs(newHeight - oldHeight) > 1) {
          this.updatePositions(index, newHeight);
        }
      }
    });

    this.container.querySelectorAll('[data-index]').forEach((el) => {
      observer.observe(el);
    });
  }

  // 更新受影响位置
  updatePositions(index, newHeight) {
    const diff = newHeight - this.positions[index].height;
    this.positions[index].height = newHeight;
    this.positions[index].bottom += diff;

    // 后续所有项位置偏移
    for (let i = index + 1; i < this.positions.length; i++) {
      this.positions[i].top += diff;
      this.positions[i].bottom += diff;
    }
  }

  // 二分查找起始索引
  getStartIndex(scrollTop) {
    let low = 0, high = this.positions.length - 1;
    while (low <= high) {
      const mid = (low + high) >>> 1;
      if (this.positions[mid].bottom <= scrollTop) low = mid + 1;
      else if (this.positions[mid].top > scrollTop) high = mid - 1;
      else return mid;
    }
    return low;
  }
}
```

**Q17：为什么说 React Native 的 Hermes 引擎是 RN 性能提升的关键一步？**

A：

Hermes 是 Meta 专为 React Native 开发的 JS 引擎，与 JavaScriptCore (JSC) 的核心区别：

| 维度 | Hermes | JSC (旧默认) | V8 |
|------|--------|-------------|-----|
| 编译方式 | **AOT 字节码** | JIT 编译 | JIT 编译 |
| 启动速度 | ⭐ 极快（跳过解析编译） | 慢（需解析+编译） | 慢 |
| 包体积 | ⭐ 更小（字节码紧凑） | 大（源码） | 大 |
| 内存 | ⭐ 低（无 JIT 缓存） | 高 | 高 |
| 峰值性能 | 中（无 JIT 优化） | 高 | 最高 |
| GC | 增量 GC，低延迟 | 分代 GC | 分代 GC |

Hermes 的核心创新：**构建时将 JS 编译为字节码**（`.hbc` 文件），运行时直接解释执行字节码，跳过了解析和编译阶段。这对移动端冷启动特别重要——用户打开 App 不需要等 JS 编译。

```
传统流程：App 启动 → 加载 JS 源码 → 解析 AST → 编译字节码 → 执行
Hermes：   App 启动 → 加载 .hbc 字节码 → 直接执行
                       省去解析+编译 → 启动快 50-80%
```

### 开放式设计题

**D1：公司要开发一个同时覆盖iOS/Android/小程序/H5的产品，技术选型怎么做？**

**参考思路**：
- 评估维度：团队技术栈（有RN经验？Flutter经验？）、性能要求、原生能力依赖、开发效率、维护成本
- 方案对比：Flutter全端（性能好但生态较新）、RN+小程序（Taro跨端）、H5+混合（灵活但体验差）
- 推荐策略：核心体验页面原生/Flutter、活动运营页面H5、小程序独立开发（生态差异大）
- 共享层：网络层/状态管理/业务逻辑层尽量跨端共享（TypeScript SDK）、UI层各端独立适配
- 关键取舍：一套代码覆盖所有端 vs 多套代码各端最优体验（大多数团队选折中方案）

**D2：React Native应用启动时间从1.5s劣化到4s，如何优化？**

**参考思路**：
- 度量：Flipper Performance插件、console.time关键节点、Systrace分析
- JS层面：Hermes引擎开启（预编译字节码）、减少require数量（lazy require）、减少初始化逻辑
- 原生层面：减少Native Modules初始化（TurboModules按需加载）、SplashScreen遮盖加载
- 资源层面：Bundle拆分（基础包+业务包）、图片资源预加载、字体异步加载
- 架构层面：预加载下一个页面的数据、渲染前先显示骨架屏
