## 性能指标体系

> ⭐⭐⭐ 性能指标是性能优化的起点——你无法优化你不能度量的东西。

### Core Web Vitals 核心指标

Google 在 2020 年正式提出 Core Web Vitals，并将其纳入搜索排名因素。2024 年 INP 正式替代 FID。

#### 1. LCP（Largest Contentful Paint）⭐⭐⭐

**定义**：视口内最大的可见内容元素完成渲染的时间点。

**衡量内容**：`<img>`、`<video>` 封面、CSS `background-image`、块级文本节点。

**阈值**：
| 评级 | 时间 |
|------|------|
| Good | ≤ 2.5s |
| Needs Improvement | 2.5s ~ 4.0s |
| Poor | > 4.0s |

**影响因素与优化方向**：
1. 服务器响应慢 → 优化 TTFB
2. 阻塞渲染的资源 → 消除关键路径上的阻塞 JS/CSS
3. 资源加载慢 → CDN / 预加载 / 压缩
4. 客户端渲染 → SSR / SSG

```javascript
// 通过 PerformanceObserver 监听 LCP
const lcpObserver = new PerformanceObserver((entryList) => {
  const entries = entryList.getEntries();
  const lastEntry = entries[entries.length - 1];
  console.log('LCP:', lastEntry.startTime, 'ms');
  console.log('LCP Element:', lastEntry.element);
  console.log('LCP URL:', lastEntry.url); // 如果是图片
  console.log('LCP Size:', lastEntry.size);
});

lcpObserver.observe({ type: 'largest-contentful-paint', buffered: true });
```

```html
<!-- 预加载 LCP 图片 -->
<link rel="preload" as="image" href="/hero-image.webp" fetchpriority="high">

<!-- fetchpriority 提升 LCP 元素优先级 -->
<img src="/hero.webp" fetchpriority="high" alt="Hero Image" width="1200" height="600">
```

#### 2. FID（First Input Delay）→ 已被 INP 替代

**定义**：用户首次与页面交互（点击、轻触、按键）到浏览器实际开始处理事件的延迟时间。

**阈值**：Good ≤ 100ms | Poor > 300ms

**局限性**：只测量首次交互，无法反映页面整体交互响应质量。2024年3月被 INP 替代。

---

#### 3. INP（Interaction to Next Paint）⭐⭐⭐

**定义**：在页面整个生命周期中，所有交互延迟中取第 98 百分位（即近乎最差的交互延迟）。

**衡量范围**：click、tap、keydown 等交互事件从触发到下一帧绘制完成的时间。

**阈值**：
| 评级 | 时间 |
|------|------|
| Good | ≤ 200ms |
| Needs Improvement | 200ms ~ 500ms |
| Poor | > 500ms |

**INP 的三个阶段**：
```
Input Delay → Processing Time → Presentation Delay
（输入延迟）    （处理时间）      （呈现延迟）
```

```javascript
// 监听 INP
const inpObserver = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    // entry.duration 是总交互时长
    // entry.processingStart - entry.startTime 是 Input Delay
    // entry.processingEnd - entry.processingStart 是 Processing Time
    // entry.startTime + entry.duration - entry.processingEnd 是 Presentation Delay

    const inputDelay = entry.processingStart - entry.startTime;
    const processingTime = entry.processingEnd - entry.processingStart;
    const presentationDelay = entry.startTime + entry.duration - entry.processingEnd;

    console.log(`INP Breakdown: 
      Input Delay: ${inputDelay}ms
      Processing: ${processingTime}ms
      Presentation: ${presentationDelay}ms
      Total: ${entry.duration}ms
    `);
  }
});

inpObserver.observe({ type: 'event', durationThreshold: 16, buffered: true });
```

**优化策略**：
```javascript
// 1. 减少 Input Delay：拆分长任务
// Bad: 长任务阻塞主线程
button.addEventListener('click', () => {
  heavyComputation(); // 200ms 阻塞
  updateUI();
});

// Good: 使用 scheduler.yield() 让出主线程
button.addEventListener('click', async () => {
  // 先处理高优先级的视觉反馈
  updateButtonState();

  // 让出控制权给浏览器
  if (scheduler.yield) {
    await scheduler.yield();
  }

  // 再处理重计算
  heavyComputation();
  updateUI();
});

// 2. 减少 Processing Time：优化事件处理函数
// Bad: 在事件处理器中做大量DOM操作
function handleScroll() {
  items.forEach(item => {
    item.style.transform = `translateY(${calculatePosition(item)}px)`;
  });
}

// Good: 批量读写DOM，避免强制同步布局
function handleScroll() {
  // 先批量读
  const positions = items.map(item => calculatePosition(item));
  // 再批量写
  requestAnimationFrame(() => {
    items.forEach((item, i) => {
      item.style.transform = `translateY(${positions[i]}px)`;
    });
  });
}
```

#### 4. CLS（Cumulative Layout Shift）⭐⭐⭐

**定义**：页面整个生命周期内所有意外布局偏移分数的总和（按会话窗口分组取最大值）。

**计算公式**：`CLS = Impact Fraction × Distance Fraction`

**阈值**：Good ≤ 0.1 | Poor > 0.25

```javascript
// 监听 CLS
let clsValue = 0;
const clsObserver = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    // 只计算非用户交互导致的偏移
    if (!entry.hadRecentInput) {
      clsValue += entry.value;
      console.log('Layout Shift:', {
        value: entry.value,
        sources: entry.sources?.map(s => ({
          node: s.node,
          previousRect: s.previousRect,
          currentRect: s.currentRect
        }))
      });
    }
  }
  console.log('CLS:', clsValue);
});

clsObserver.observe({ type: 'layout-shift', buffered: true });
```

**常见问题与修复**：
```html
<!-- 1. 图片/视频始终指定宽高 -->
<!-- Bad -->
<img src="photo.jpg">

<!-- Good：浏览器可以预留空间 -->
<img src="photo.jpg" width="800" height="600">

<!-- Better：响应式 + 固定宽高比 -->
<style>
  .img-wrapper {
    aspect-ratio: 16 / 9;
    width: 100%;
  }
  .img-wrapper img {
    width: 100%;
    height: 100%;
    object-fit: cover;
  }
</style>

<!-- 2. 动态内容预留空间 -->
<style>
  .ad-slot {
    min-height: 250px; /* 预留广告位高度 */
  }
</style>

<!-- 3. 字体加载防抖动 -->
<style>
  /* 使用 font-display: swap 或 optional */
  @font-face {
    font-family: 'CustomFont';
    src: url('/fonts/custom.woff2') format('woff2');
    font-display: swap; /* 先用系统字体，加载完切换 */
  }

  /* 更好的方案：size-adjust 减少字体切换偏移 */
  @font-face {
    font-family: 'CustomFont';
    src: url('/fonts/custom.woff2') format('woff2');
    font-display: swap;
    size-adjust: 105%;    /* 调整到与系统字体接近 */
    ascent-override: 90%;
    descent-override: 20%;
  }
</style>
```

#### 5. TTFB（Time To First Byte）⭐⭐

**定义**：从请求发出到接收第一个字节的时间，衡量服务器响应速度。

**阈值**：Good ≤ 800ms | Poor > 1800ms

```javascript
// 获取 TTFB
const [navigation] = performance.getEntriesByType('navigation');
const ttfb = navigation.responseStart - navigation.requestStart;
console.log('TTFB:', ttfb, 'ms');

// 更详细的拆解
console.log({
  dns: navigation.domainLookupEnd - navigation.domainLookupStart,
  tcp: navigation.connectEnd - navigation.connectStart,
  tls: navigation.secureConnectionStart > 0
    ? navigation.connectEnd - navigation.secureConnectionStart
    : 0,
  request: navigation.responseStart - navigation.requestStart,
  ttfb: navigation.responseStart - navigation.fetchStart
});
```

### RUM vs Synthetic Monitoring ⭐⭐

| 维度 | RUM（真实用户监控） | Synthetic（合成监控） |
|------|-------------------|---------------------|
| 数据来源 | 真实用户 | 模拟环境 |
| 覆盖场景 | 所有设备/网络/地域 | 固定配置 |
| 时效性 | 实时 | 定时执行 |
| 成本 | 需要埋点 + 数据处理 | 固定执行成本 |
| 发现问题 | 长尾问题、特定用户群 | 回归检测、基线比较 |
| 代表工具 | web-vitals库、自研SDK | Lighthouse CI、WebPageTest |
| 最佳场景 | 线上监控、告警 | CI/CD 准入、竞品分析 |

```javascript
// RUM 实践：使用 web-vitals 库
import { onLCP, onINP, onCLS, onFCP, onTTFB } from 'web-vitals';

function sendToAnalytics(metric) {
  const body = JSON.stringify({
    name: metric.name,
    value: metric.value,
    rating: metric.rating,   // 'good' | 'needs-improvement' | 'poor'
    delta: metric.delta,
    id: metric.id,
    navigationType: metric.navigationType,
    // 归因信息
    attribution: metric.attribution,
  });

  // 使用 Beacon API 确保页面卸载前也能发送
  if (navigator.sendBeacon) {
    navigator.sendBeacon('/api/vitals', body);
  } else {
    fetch('/api/vitals', { body, method: 'POST', keepalive: true });
  }
}

// 带归因信息的版本（帮助定位问题）
import { onLCP, onCLS, onINP } from 'web-vitals/attribution';

onLCP(sendToAnalytics);
onINP(sendToAnalytics);
onCLS(sendToAnalytics);
onFCP(sendToAnalytics);
onTTFB(sendToAnalytics);
```

### Tree Shaking ⭐⭐⭐

**原理**：基于 ES Module 的静态分析，在编译阶段标记和移除未被引用的导出。

**前提条件**：
1. 必须使用 ESM（`import/export`），不能是 CommonJS
2. package.json 标记 `"sideEffects": false`
3. 代码不能有副作用

```javascript
// utils.js — ES Module
export function add(a, b) {
  return a + b;
}

export function multiply(a, b) {
  return a * b;
}

export function complexUnusedFunction() {
  // 大量代码...
  return 'never used';
}

// main.js — 只导入 add
import { add } from './utils';  // multiply 和 complexUnusedFunction 会被 tree-shake
console.log(add(1, 2));
```

```json
// package.json
{
  "name": "my-lib",
  "sideEffects": false,
  // 或精确标记有副作用的文件
  "sideEffects": [
    "*.css",
    "*.scss",
    "./src/polyfills.js"
  ]
}
```

**Tree Shaking 失效的常见情况**：

```javascript
// ❌ 1. 使用 CommonJS
const { add } = require('./utils'); // CJS 无法 tree-shake

// ❌ 2. 副作用导入
import './polyfill'; // 被视为有副作用，不会 tree-shake

// ❌ 3. 导入后重新导出整个模块
export * from './utils'; // barrel file 可能阻碍 tree-shaking

// ❌ 4. 动态属性访问
import * as utils from './utils';
const fn = utils[dynamicKey]; // 无法静态分析

// ✅ 优化 barrel file
// Bad: index.js
export * from './componentA';
export * from './componentB';
export * from './componentC';

// Good: 直接导入
import { ComponentA } from './componentA';
```

### 懒加载 ⭐⭐⭐

```javascript
// 1. 原生图片懒加载
// <img src="placeholder.jpg" data-src="real.jpg" loading="lazy" alt="...">

// 2. Intersection Observer 实现懒加载
class LazyLoader {
  constructor(options = {}) {
    this.observer = new IntersectionObserver(
      (entries) => {
        entries.forEach((entry) => {
          if (entry.isIntersecting) {
            this.loadElement(entry.target);
            this.observer.unobserve(entry.target);
          }
        });
      },
      {
        rootMargin: options.rootMargin || '200px 0px', // 提前 200px 开始加载
        threshold: options.threshold || 0,
      }
    );
  }

  observe(elements) {
    elements.forEach((el) => this.observer.observe(el));
  }

  loadElement(el) {
    if (el.tagName === 'IMG') {
      el.src = el.dataset.src;
      if (el.dataset.srcset) el.srcset = el.dataset.srcset;
    } else if (el.tagName === 'VIDEO') {
      el.src = el.dataset.src;
      el.load();
    } else {
      // 背景图懒加载
      el.style.backgroundImage = `url(${el.dataset.bg})`;
    }
    el.classList.add('loaded');
  }

  destroy() {
    this.observer.disconnect();
  }
}

// 使用
const loader = new LazyLoader({ rootMargin: '300px 0px' });
loader.observe(document.querySelectorAll('[data-src]'));
```

### 预加载 preload / prefetch ⭐⭐

```html
<!-- preload: 当前页面需要的高优先级资源 -->
<link rel="preload" href="/fonts/inter.woff2" as="font" type="font/woff2" crossorigin>
<link rel="preload" href="/css/critical.css" as="style">
<link rel="preload" href="/api/initial-data" as="fetch" crossorigin>

<!-- prefetch: 下一个页面可能需要的资源（空闲时加载） -->
<link rel="prefetch" href="/js/settings-page.js">
<link rel="prefetch" href="/img/next-hero.webp">

<!-- preconnect: 提前建立连接（DNS + TCP + TLS） -->
<link rel="preconnect" href="https://cdn.example.com">
<link rel="preconnect" href="https://api.example.com">

<!-- dns-prefetch: 仅 DNS 预解析（更轻量） -->
<link rel="dns-prefetch" href="https://analytics.example.com">

<!-- modulepreload: 预加载 ES Module -->
<link rel="modulepreload" href="/js/app.mjs">
```

| 资源提示 | 优先级 | 使用场景 | 浏览器行为 |
|---------|--------|---------|-----------|
| preload | 高 | 当前页面关键资源 | 立即加载，不执行 |
| prefetch | 低 | 下一页面资源 | 空闲时加载 |
| preconnect | — | 第三方域名 | 预建TCP+TLS连接 |
| dns-prefetch | — | 第三方域名 | 仅DNS预解析 |
| modulepreload | 高 | ESM模块 | 加载+解析+编译 |

```javascript
// 动态预加载 — 路由预判
function prefetchRoute(routePath) {
  const link = document.createElement('link');
  link.rel = 'prefetch';
  link.href = routePath;
  document.head.appendChild(link);
}
```

## 网络优化

> ⭐⭐ 网络优化是性能优化的底层基座——减少延迟、增加吞吐。

### HTTP/2 多路复用 ⭐⭐

**HTTP/1.1 问题**：
- 每个域名最多 6 个 TCP 并发连接
- 队头阻塞（Head-of-Line Blocking）
- 头部未压缩，冗余传输

**HTTP/2 核心特性**：
1. **多路复用**：单个 TCP 连接上同时传输多个请求/响应
2. **头部压缩**：HPACK 算法
3. **服务端推送**：Server Push（实际已被弃用）
4. **二进制分帧**：更高效的传输格式
5. **流优先级**：控制资源加载顺序

```nginx
# Nginx 配置 HTTP/2
server {
    listen 443 ssl;
    http2 on;                       # Nginx 1.25+
    # listen 443 ssl http2;         # Nginx < 1.25

    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    # HTTP/2 优化参数
    http2_max_concurrent_streams 128;
    http2_initial_window_size 65535;
}
```

**HTTP/2 下的优化策略变化**：
| 优化策略 | HTTP/1.1 | HTTP/2 |
|---------|----------|--------|
| 域名分片 | ✅ 需要（突破6连接限制） | ❌ 不需要（反而有害） |
| 雪碧图 | ✅ 减少请求数 | ⚠️ 收益降低 |
| 文件合并 | ✅ 减少请求数 | ⚠️ 影响缓存粒度 |
| 内联小资源 | ✅ 减少请求 | ⚠️ 影响缓存 |

---

### HTTP/3 QUIC ⭐⭐

**HTTP/2 的问题**：TCP 层的队头阻塞——一个 TCP 包丢失会阻塞整个连接。

**HTTP/3 核心改进**：
1. 基于 **UDP + QUIC** 协议
2. 消除 TCP 队头阻塞（每个 QUIC 流独立）
3. **0-RTT 连接恢复**
4. 内置 TLS 1.3
5. 连接迁移（切换网络不断连）

```nginx
# Nginx 配置 HTTP/3
server {
    listen 443 ssl;
    http2 on;
    listen 443 quic reuseport;   # QUIC (HTTP/3)

    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    # 通告支持 HTTP/3
    add_header Alt-Svc 'h3=":443"; ma=86400';

    ssl_protocols TLSv1.3;
}
```

| 特性 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|------|----------|--------|--------|
| 传输协议 | TCP | TCP | QUIC (UDP) |
| 多路复用 | ❌ | ✅ | ✅ |
| 队头阻塞 | 应用层 + 传输层 | 传输层 | ❌ 无 |
| 连接建立 | 3-RTT | 3-RTT | 1-RTT / 0-RTT |
| TLS | 独立握手 | 独立握手 | 内置 TLS 1.3 |
| 连接迁移 | ❌ | ❌ | ✅ |

## 缓存策略

> ⭐⭐⭐ 缓存是性能优化中性价比最高的手段——最快的请求是不发请求。

### 缓存层级体系

```
浏览器请求资源 →
  ┌─────────────────────────────────────┐
  │  1. Memory Cache（内存缓存）          │  ← 最快，Tab 关闭即失效
  │     ↓ miss                           │
  │  2. Service Worker Cache             │  ← 可编程、离线可用
  │     ↓ miss                           │
  │  3. Disk Cache（HTTP 缓存）           │  ← 强缓存 / 协商缓存
  │     ↓ miss                           │
  │  4. Push Cache（HTTP/2）              │  ← 连接级缓存
  │     ↓ miss                           │
  │  5. CDN 边缘缓存                     │  ← 边缘节点
  │     ↓ miss                           │
  │  6. 源站                              │
  └─────────────────────────────────────┘
```

---

### 强缓存 vs 协商缓存 ⭐⭐⭐

#### 强缓存

在有效期内，浏览器直接使用缓存，不发请求。状态码 `200 (from cache)`。

```nginx
# Nginx 配置强缓存
# 带 hash 的静态资源：长期缓存
location ~* \.(js|css)$ {
    # 匹配 main.abc123.js 这样的 hash 文件名
    if ($uri ~* "\.[a-f0-9]{8,}\.(js|css)$") {
        add_header Cache-Control "public, max-age=31536000, immutable";
        break;
    }
    # 无 hash 的文件
    add_header Cache-Control "public, max-age=3600";
}

# 图片/字体：较长缓存
location ~* \.(jpg|jpeg|png|webp|avif|gif|ico|woff2|woff|ttf)$ {
    add_header Cache-Control "public, max-age=2592000"; # 30天
}

# HTML：不缓存或短缓存
location ~* \.html$ {
    add_header Cache-Control "no-cache"; # 每次都协商
}
```

#### 协商缓存

强缓存过期后，浏览器带条件请求，服务器判断是否可继续使用缓存。命中返回 `304 Not Modified`。

```
协商缓存流程：
  首次请求：
    Server → Client: 200 OK
                      ETag: "abc123"
                      Last-Modified: Wed, 21 Oct 2025 07:28:00 GMT

  再次请求（强缓存过期后）：
    Client → Server: GET /resource
                     If-None-Match: "abc123"
                     If-Modified-Since: Wed, 21 Oct 2025 07:28:00 GMT

    情况A - 未修改：
      Server → Client: 304 Not Modified (无 body)

    情况B - 已修改：
      Server → Client: 200 OK (新内容 + 新 ETag)
```

| 头字段 | 强缓存 | 协商缓存 |
|--------|--------|---------|
| Cache-Control | max-age, immutable, no-store | no-cache, must-revalidate |
| Expires | 绝对时间（已过时） | — |
| ETag / If-None-Match | — | 内容哈希比对 |
| Last-Modified / If-Modified-Since | — | 最后修改时间 |

**最佳实践缓存策略**：

```
HTML 文件         → Cache-Control: no-cache        (每次协商)
带 hash 的 JS/CSS → Cache-Control: max-age=1y, immutable (强缓存一年)
API 数据          → Cache-Control: no-store         (不缓存)
用户头像          → Cache-Control: max-age=1d       (缓存1天)
不变的库文件      → Cache-Control: max-age=1y, immutable
```

### Service Worker 缓存 ⭐⭐⭐

```javascript
// sw.js — Service Worker 缓存策略

const CACHE_NAME = 'app-v2';
const STATIC_ASSETS = [
  '/',
  '/css/main.css',
  '/js/app.js',
  '/fonts/inter.woff2',
];

// 安装：预缓存关键资源
self.addEventListener('install', (event) => {
  event.waitUntil(
    caches.open(CACHE_NAME).then((cache) => {
      return cache.addAll(STATIC_ASSETS);
    })
  );
  self.skipWaiting(); // 立即激活
});

// 激活：清理旧缓存
self.addEventListener('activate', (event) => {
  event.waitUntil(
    caches.keys().then((keys) =>
      Promise.all(
        keys.filter((key) => key !== CACHE_NAME).map((key) => caches.delete(key))
      )
    )
  );
  self.clients.claim(); // 立即接管
});

// 拦截请求：实现不同的缓存策略
self.addEventListener('fetch', (event) => {
  const { request } = event;
  const url = new URL(request.url);

  // 策略1: Cache First（静态资源）
  if (url.pathname.match(/\.(js|css|woff2|png|jpg|webp)$/)) {
    event.respondWith(cacheFirst(request));
    return;
  }

  // 策略2: Network First（API 请求）
  if (url.pathname.startsWith('/api/')) {
    event.respondWith(networkFirst(request));
    return;
  }

  // 策略3: Stale While Revalidate（HTML 页面）
  event.respondWith(staleWhileRevalidate(request));
});

// Cache First: 缓存优先，miss 后网络
async function cacheFirst(request) {
  const cached = await caches.match(request);
  if (cached) return cached;

  const response = await fetch(request);
  if (response.ok) {
    const cache = await caches.open(CACHE_NAME);
    cache.put(request, response.clone());
  }
  return response;
}

// Network First: 网络优先，失败后缓存
async function networkFirst(request) {
  try {
    const response = await fetch(request);
    if (response.ok) {
      const cache = await caches.open(CACHE_NAME);
      cache.put(request, response.clone());
    }
    return response;
  } catch (error) {
    const cached = await caches.match(request);
    return cached || new Response('Offline', { status: 503 });
  }
}

// Stale While Revalidate: 先返回缓存，同时后台更新
async function staleWhileRevalidate(request) {
  const cache = await caches.open(CACHE_NAME);
  const cached = await cache.match(request);

  const fetchPromise = fetch(request).then((response) => {
    if (response.ok) {
      cache.put(request, response.clone());
    }
    return response;
  });

  return cached || fetchPromise;
}
```

| 缓存策略 | 适用场景 | 优点 | 缺点 |
|----------|---------|------|------|
| Cache First | 静态资源、字体 | 极快，离线可用 | 更新不及时 |
| Network First | API 数据、实时内容 | 数据新鲜 | 网络慢时体验差 |
| Stale While Revalidate | HTML、非实时 API | 快且较新 | 首次看到旧数据 |
| Network Only | 实时数据、支付 | 数据一定是最新 | 离线不可用 |
| Cache Only | 预缓存的静态资源 | 最快 | 无法更新 |

## 渲染性能优化

> ⭐⭐⭐ 渲染性能直接决定用户交互体验——60fps 意味着每帧仅有 16.67ms。

### 浏览器渲染管线

```
JavaScript → Style → Layout → Paint → Composite
  (JS)      (样式计算) (布局/回流) (绘制)   (合成)

重排(Reflow)：触发 Layout 及之后的所有步骤
重绘(Repaint)：跳过 Layout，触发 Paint + Composite
合成(Composite)：只触发 Composite，最高效
```

---

### 重排与重绘 ⭐⭐⭐

**触发重排的操作**（代价最高）：
- 修改几何属性：width、height、padding、margin、border
- 修改位置：top、left、position
- 修改显示：display、overflow
- 读取布局信息：offsetTop、scrollHeight、getComputedStyle()、getBoundingClientRect()

**触发重绘的操作**：
- color、background-color、visibility、box-shadow、outline

**仅触发合成的属性**（代价最低，GPU 加速）：
- `transform`、`opacity`、`filter`、`will-change`

```javascript
// ❌ Bad: 强制同步布局（Layout Thrashing）
function resizeAll(elements) {
  elements.forEach(el => {
    const width = el.offsetWidth;     // 读（触发布局）
    el.style.width = width * 2 + 'px'; // 写（使布局失效）
    // 下一次循环读 offsetWidth 又会触发布局计算！
  });
}

// ✅ Good: 批量读写分离
function resizeAll(elements) {
  // 先批量读
  const widths = elements.map(el => el.offsetWidth);
  // 再批量写
  elements.forEach((el, i) => {
    el.style.width = widths[i] * 2 + 'px';
  });
}

// ✅ Better: 使用 requestAnimationFrame
function resizeAll(elements) {
  const widths = elements.map(el => el.offsetWidth); // 批量读
  requestAnimationFrame(() => {
    elements.forEach((el, i) => {
      el.style.width = widths[i] * 2 + 'px'; // 批量写
    });
  });
}
```

### 合成层与 GPU 加速 ⭐⭐

**提升为合成层的条件**：
- `transform: translateZ(0)` 或 `translate3d(0,0,0)`
- `will-change: transform | opacity`
- `<video>`, `<canvas>`, `<iframe>`
- CSS `animation` / `transition` 使用了 `transform` 或 `opacity`

### 虚拟列表 ⭐⭐⭐

**原理**：只渲染可视区域内的 DOM 节点，通过滚动时动态替换实现大数据列表渲染。

```javascript
// 虚拟列表核心实现
class VirtualList {
  constructor({ container, itemHeight, totalCount, renderItem, bufferSize = 5 }) {
    this.container = container;
    this.itemHeight = itemHeight;
    this.totalCount = totalCount;
    this.renderItem = renderItem;
    this.bufferSize = bufferSize;

    // 创建 DOM 结构
    this.scrollContainer = document.createElement('div');
    this.scrollContainer.style.cssText = 'overflow-y: auto; height: 100%;';

    this.phantom = document.createElement('div');
    this.phantom.style.height = `${totalCount * itemHeight}px`; // 撑开滚动高度

    this.content = document.createElement('div');
    this.content.style.cssText = 'position: relative;';

    this.scrollContainer.appendChild(this.phantom);
    this.scrollContainer.appendChild(this.content);
    this.container.appendChild(this.scrollContainer);

    this.scrollContainer.addEventListener('scroll', this.onScroll.bind(this));
    this.render();
  }

  onScroll() {
    requestAnimationFrame(() => this.render());
  }

  render() {
    const scrollTop = this.scrollContainer.scrollTop;
    const viewportHeight = this.scrollContainer.clientHeight;

    const startIndex = Math.max(0, Math.floor(scrollTop / this.itemHeight) - this.bufferSize);
    const endIndex = Math.min(
      this.totalCount - 1,
      Math.ceil((scrollTop + viewportHeight) / this.itemHeight) + this.bufferSize
    );

    // 偏移量
    this.content.style.transform = `translateY(${startIndex * this.itemHeight}px)`;

    // 生成可见区域的 DOM
    const fragment = document.createDocumentFragment();
    for (let i = startIndex; i <= endIndex; i++) {
      const item = this.renderItem(i);
      item.style.height = `${this.itemHeight}px`;
      fragment.appendChild(item);
    }

    this.content.innerHTML = '';
    this.content.appendChild(fragment);
  }
}

// 使用
new VirtualList({
  container: document.getElementById('list'),
  itemHeight: 50,
  totalCount: 100000,     // 10 万条数据
  bufferSize: 5,
  renderItem: (index) => {
    const div = document.createElement('div');
    div.textContent = `Item ${index}`;
    div.className = 'list-item';
    return div;
  },
});
```

### 时间切片 ⭐⭐

**原理**：将长任务拆分为多个小任务，分散到多个帧中执行，避免阻塞主线程。

```javascript
// 方式 1: 基于 requestIdleCallback
function timeSlice(tasks, deadline = 5) {
  return new Promise((resolve) => {
    const pending = [...tasks];

    function processBatch(idleDeadline) {
      while (pending.length > 0 && idleDeadline.timeRemaining() > deadline) {
        const task = pending.shift();
        task();
      }

      if (pending.length > 0) {
        requestIdleCallback(processBatch);
      } else {
        resolve();
      }
    }

    requestIdleCallback(processBatch);
  });
}

// 方式 2: 基于 MessageChannel（React Scheduler 使用的方式）
function scheduleTasks(tasks) {
  const pending = [...tasks];
  const channel = new MessageChannel();

  return new Promise((resolve) => {
    channel.port2.onmessage = () => {
      const startTime = performance.now();

      // 每帧执行 5ms 的任务
      while (pending.length > 0 && performance.now() - startTime < 5) {
        const task = pending.shift();
        task();
      }

      if (pending.length > 0) {
        channel.port1.postMessage(null); // 调度下一批
      } else {
        resolve();
      }
    };

    channel.port1.postMessage(null);
  });
}

// 方式 3: 使用 scheduler.yield()（Chrome 129+）
async function processLargeDataset(items) {
  for (let i = 0; i < items.length; i++) {
    processItem(items[i]);

    // 每处理 100 项让出主线程
    if (i % 100 === 0 && globalThis.scheduler?.yield) {
      await scheduler.yield();
    }
  }
}
```

```javascript
  #snapshots = [];

  takeSnapshot(label) {
    if (performance.memory) {
      this.#snapshots.push({
        label,
        timestamp: Date.now(),
        usedJSHeapSize: performance.memory.usedJSHeapSize,
        totalJSHeapSize: performance.memory.totalJSHeapSize,
      });
    }
  }

    const growth = [];
    for (let i = 1; i < this.#snapshots.length; i++) {
      const diff = this.#snapshots[i].usedJSHeapSize - this.#snapshots[i - 1].usedJSHeapSize;

        to: this.#snapshots[i].label,

        isLeak: diff > 0,
```

```javascript
// 防抖 Debounce：事件停止触发 delay 后才执行
function debounce(fn, delay, options = {}) {
  let timer = null;
  const { leading = false, maxWait = Infinity } = options;
  let lastCallTime = 0;

  const debounced = function (...args) {
    const now = Date.now();

    // leading edge 首次立即执行
    if (leading && !timer) {
      fn.apply(this, args);
    }

    clearTimeout(timer);

    // maxWait: 最大等待时间后强制执行
    if (now - lastCallTime >= maxWait) {
      fn.apply(this, args);
      lastCallTime = now;
      return;
    }

    timer = setTimeout(() => {
      if (!leading) fn.apply(this, args);
      timer = null;
      lastCallTime = now;
    }, delay);

    if (!lastCallTime) lastCallTime = now;
  };

  debounced.cancel = () => {
    clearTimeout(timer);
    timer = null;
  };

  return debounced;
}

// 节流 Throttle：固定间隔执行一次
function throttle(fn, interval, options = {}) {
  let lastTime = 0;
  let timer = null;
  const { leading = true, trailing = true } = options;

  const throttled = function (...args) {
    const now = Date.now();

    if (!leading && !lastTime) {
      lastTime = now;
    }

    const remaining = interval - (now - lastTime);

    if (remaining <= 0) {
      clearTimeout(timer);
      timer = null;
      lastTime = now;
      fn.apply(this, args);
    } else if (trailing && !timer) {
      timer = setTimeout(() => {
        lastTime = leading ? Date.now() : 0;
        timer = null;
        fn.apply(this, args);
      }, remaining);
    }
  };

  throttled.cancel = () => {
    clearTimeout(timer);
    timer = null;
    lastTime = 0;
  };

  return throttled;
}

// 使用场景
const handleSearch = debounce(async (query) => {
  const results = await searchAPI(query);
  renderResults(results);
}, 300);

const handleScroll = throttle(() => {
  updateScrollIndicator();
}, 100);
```

| 方式 | 触发频率 | 典型场景 |
|------|---------|---------|
| 防抖 | 停止触发 delay 后执行 1 次 | 搜索输入、窗口 resize |
| 节流 | 每 interval 执行 1 次 | 滚动监听、鼠标移动 |

### 字体优化 ⭐⭐

```css
/* 1. font-display 策略选择 */
@font-face {
  font-family: 'CustomFont';
  src: url('/fonts/custom.woff2') format('woff2');
  font-display: swap;       /* 先用系统字体，加载完切换 */
  /* font-display 值对比：
     auto     → 浏览器默认
     block    → 短暂隐藏文本(3s)，再显示
     swap     → 立即显示系统字体，加载完切换  ← 推荐
     fallback → 极短隐藏(100ms)，短暂 swap 期(3s)
     optional → 极短隐藏(100ms)，不切换     ← 对 CLS 最好
  */
}

/* 2. 使用 unicode-range 按需加载 */
@font-face {
  font-family: 'NotoSansSC';
  src: url('/fonts/noto-latin.woff2') format('woff2');
  unicode-range: U+0000-00FF;    /* 只加载拉丁字符 */
}
@font-face {
  font-family: 'NotoSansSC';
  src: url('/fonts/noto-cjk.woff2') format('woff2');
  unicode-range: U+4E00-9FFF;    /* 中文字符按需加载 */
}
```

```html
<!-- 3. 预加载关键字体 -->
<link
  rel="preload"
  href="/fonts/inter-var.woff2"
  as="font"
  type="font/woff2"
  crossorigin
>
```

| 策略 | FOUT | FOIT | CLS影响 | 推荐场景 |
|------|------|------|--------|---------|
| swap | 有 | 无 | 可能有 | 正文字体 |
| optional | 无 | 极短 | 几乎无 | 正文字体（弱网） |
| block | 无 | 有(3s) | 无 | 图标字体 |
| size-adjust | 减少 | 无 | 极小 | 所有场景 |

### 前端性能埋点 SDK ⭐⭐

```javascript
// 性能监控 SDK 核心实现
class PerformanceMonitor {
  constructor(config) {
    this.config = {
      reportUrl: config.reportUrl,
      sampleRate: config.sampleRate || 1,   // 采样率
      appId: config.appId,
    };
    this.metrics = {};
    this.init();
  }

  init() {
    // 采样率判断
    if (Math.random() > this.config.sampleRate) return;

    this.observeWebVitals();
    this.observeLongTasks();
    this.observeResources();
    this.observeErrors();

    // 页面卸载前上报
    window.addEventListener('visibilitychange', () => {
      if (document.visibilityState === 'hidden') {
        this.report();
      }
    });
  }

  observeWebVitals() {
    // LCP
    new PerformanceObserver((list) => {
      const entries = list.getEntries();
      this.metrics.lcp = entries[entries.length - 1].startTime;
    }).observe({ type: 'largest-contentful-paint', buffered: true });

    // CLS
    let cls = 0;
    new PerformanceObserver((list) => {
      for (const entry of list.getEntries()) {
        if (!entry.hadRecentInput) cls += entry.value;
      }
      this.metrics.cls = cls;
    }).observe({ type: 'layout-shift', buffered: true });

    // INP
    let maxINP = 0;
    new PerformanceObserver((list) => {
      for (const entry of list.getEntries()) {
        if (entry.duration > maxINP) maxINP = entry.duration;
      }
      this.metrics.inp = maxINP;
    }).observe({ type: 'event', durationThreshold: 16, buffered: true });

    // FCP
    new PerformanceObserver((list) => {
      for (const entry of list.getEntries()) {
        if (entry.name === 'first-contentful-paint') {
          this.metrics.fcp = entry.startTime;
        }
      }
    }).observe({ type: 'paint', buffered: true });

    // TTFB
    const [nav] = performance.getEntriesByType('navigation');
    if (nav) {
      this.metrics.ttfb = nav.responseStart - nav.requestStart;
    }
  }

  observeLongTasks() {
    this.metrics.longTasks = [];
    new PerformanceObserver((list) => {
      for (const entry of list.getEntries()) {
        this.metrics.longTasks.push({
          duration: entry.duration,
          startTime: entry.startTime,
        });
      }
    }).observe({ type: 'longtask', buffered: true });
  }

  observeResources() {
    this.metrics.slowResources = [];
    new PerformanceObserver((list) => {
      for (const entry of list.getEntries()) {
        if (entry.duration > 2000) {
          this.metrics.slowResources.push({
            url: entry.name,
            duration: entry.duration,
            size: entry.transferSize,
          });
        }
      }
    }).observe({ type: 'resource', buffered: true });
  }

  observeErrors() {
    this.metrics.jsErrors = [];

    window.addEventListener('error', (event) => {
      this.metrics.jsErrors.push({
        message: event.message,
        source: event.filename,
        line: event.lineno,
        col: event.colno,
      });
    });

    window.addEventListener('unhandledrejection', (event) => {
      this.metrics.jsErrors.push({
        message: event.reason?.message || String(event.reason),
        type: 'unhandledrejection',
      });
    });
  }

  report() {
    const payload = {
      appId: this.config.appId,
      url: location.href,
      userAgent: navigator.userAgent,
      timestamp: Date.now(),
      metrics: this.metrics,
      connection: navigator.connection ? {
        effectiveType: navigator.connection.effectiveType,
        downlink: navigator.connection.downlink,
        rtt: navigator.connection.rtt,
      } : null,
    };

    // 使用 sendBeacon 确保数据送达
    const blob = new Blob([JSON.stringify(payload)], { type: 'application/json' });
    navigator.sendBeacon(this.config.reportUrl, blob);
  }
}

// 使用
new PerformanceMonitor({
  reportUrl: '/api/performance',
  sampleRate: 0.1,   // 10% 采样
  appId: 'my-app',
});
```

### 初级高频速刷（5 题）

**1. 什么是重排和重绘？如何减少重排？**

重排（Reflow）：元素几何属性变化导致重新计算布局；重绘（Repaint）：外观变化但不影响布局只需重新绘制。重排一定触发重绘，重绘不一定触发重排。减少重排：批量修改 DOM、使用 transform 动画、避免频繁读取布局属性、使用 DocumentFragment。

**2. 什么是懒加载？原生如何实现？**

懒加载是延迟加载非首屏资源的技术。原生方式：`<img loading="lazy">`，浏览器会在图片进入视口附近时才加载。也可用 IntersectionObserver API 手动实现。

**3. 强缓存和协商缓存的区别？**

强缓存：资源在有效期内直接使用缓存（200 from cache），不发请求，通过 Cache-Control: max-age 控制。协商缓存：强缓存过期后，发条件请求（If-None-Match / If-Modified-Since）到服务器验证，未修改返回 304。

**4. 什么是 Code Splitting？React 中如何实现？**

Code Splitting 是将代码拆分成多个 bundle，按需加载。React 中使用 `React.lazy()` + `Suspense` 实现路由/组件级别的动态导入。

**5. Lighthouse 评分主要由哪些指标组成？**

Performance 评分由 5 个指标加权计算：FCP(10%)、SI(10%)、LCP(25%)、TBT(30%)、CLS(25%)。其中 TBT 是实验室指标，对应真实用户的 INP。

### 中级高频进阶（5 题）

**1. Core Web Vitals 有哪些指标？各自衡量什么？**

三大核心指标：LCP（最大内容绘制，衡量加载性能，阈值 ≤2.5s）、INP（交互到下次绘制，衡量交互响应性，阈值 ≤200ms，2024年替代了 FID）、CLS（累积布局偏移，衡量视觉稳定性，阈值 ≤0.1）。它们分别对应用户对"加载快不快"、"操作跟不跟手"、"页面稳不稳"的感知。

**2. 为什么用 transform 做动画比 top/left 快？**

修改 top/left 会改变元素几何属性，触发重排（Layout）→ 重绘（Paint）→ 合成（Composite）的完整渲染管线。而 transform 只触发合成层操作，由 GPU 直接处理，跳过了 Layout 和 Paint 阶段。此外，使用 transform 的元素会被提升为独立的合成层，不影响其他元素。

**3. HTTP/2 与 HTTP/1.1 的主要区别？对前端优化策略有什么影响？**

HTTP/2 核心特性：二进制分帧、多路复用（单连接并发）、头部压缩（HPACK）、流优先级。影响：不再需要域名分片（反而有害）、雪碧图和文件合并收益降低、小文件不必内联（有利于缓存粒度）。

**4. Service Worker 有哪些缓存策略？各适用什么场景？**

Cache First（静态资源、字体）、Network First（API 数据）、Stale While Revalidate（HTML、非实时API）、Network Only（支付等实时操作）、Cache Only（预缓存的不变资源）。选择取决于数据实时性要求和离线可用性需求。

**5. 虚拟列表的原理是什么？如何处理动态高度？**

虚拟列表只渲染可视区域 + 上下缓冲区的 DOM 节点，通过一个大高度容器模拟滚动。当用户滚动时，动态计算可见范围、回收和创建 DOM。动态高度处理：先用预估高度渲染，通过 ResizeObserver 获取实际高度并缓存，更新偏移量。react-window 的 VariableSizeList 支持此场景。

### 深入问答（25 题 Q&A）

**Q1: INP 和 FID 有什么区别？为什么 Google 用 INP 替代了 FID？**

A: FID 只测量首次交互的输入延迟（input delay），且不包括事件处理时间和呈现延迟。INP 测量页面整个生命周期内所有交互的全延迟（input delay + processing time + presentation delay），取 P98 值。FID 的问题：1) 只看首次交互，忽略后续交互；2) 只测 input delay，不含处理耗时；3) 首次交互通常在页面加载早期，可能不代表整体体验。INP 更全面地反映了"页面是否流畅可交互"。

```javascript
// INP 三阶段拆解
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    const inputDelay = entry.processingStart - entry.startTime;
    const processingTime = entry.processingEnd - entry.processingStart;
    const presentationDelay = (entry.startTime + entry.duration) - entry.processingEnd;

    // 哪个阶段耗时最多就优化哪个
    if (inputDelay > 100) {
      console.warn('主线程忙碌，需要拆分长任务');
    }
    if (processingTime > 100) {
      console.warn('事件处理函数太慢，需要优化逻辑');
    }
    if (presentationDelay > 100) {
      console.warn('渲染太慢，检查 DOM 操作和布局');
    }
  }
});
observer.observe({ type: 'event', durationThreshold: 16, buffered: true });
```

**Q3: CLS 的计算方式是什么？如何定位和修复布局偏移问题？**

A: CLS = 所有布局偏移分数之和（按会话窗口取最大值）。单次偏移分数 = Impact Fraction × Distance Fraction。Impact Fraction 是偏移前后两个位置的联合区域占视口的比例；Distance Fraction 是元素移动的最大距离占视口的比例。

定位方式：
```javascript
// 精确定位 CLS 来源
new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (!entry.hadRecentInput && entry.value > 0.01) {
      entry.sources?.forEach(source => {
        console.log('偏移元素:', source.node);
        console.log('偏移前位置:', source.previousRect);
        console.log('偏移后位置:', source.currentRect);
      });
    }
  }
}).observe({ type: 'layout-shift', buffered: true });
```

常见修复：图片/视频固定宽高或 aspect-ratio、动态内容预留空间、font-display: optional、避免在已有内容上方插入 DOM。

**Q4: Tree Shaking 的原理是什么？为什么 CommonJS 不能 Tree Shake？**

A: Tree Shaking 基于 ES Module 的静态结构特性。ESM 的 import/export 在编译时就能确定模块的依赖关系（静态分析），打包工具可以标记未使用的 export 并在压缩阶段移除。

CommonJS 不能 Tree Shake 的原因：
1. `require()` 是函数调用，可以在条件语句中动态执行：`if (condition) require('module')`
2. `module.exports` 可以是任意表达式：`module.exports = condition ? A : B`
3. 导出的属性可以动态访问：`require('module')[dynamicKey]`

这些都使得静态分析无法在编译时确定哪些导出被使用了。

**Q5: Webpack splitChunks 的 chunks: 'all' / 'async' / 'initial' 分别是什么意思？**

**Q6: preload 和 prefetch 的区别？使用不当会有什么问题？**

A:
- **preload**：当前页面**一定会用到**的高优先级资源，立即开始加载，不会被延迟。
- **prefetch**：下一个页面**可能会用到**的资源，浏览器空闲时低优先级加载。

使用不当的问题：
1. preload 了但没用到 → Chrome 控制台警告："The resource was preloaded but not used within a few seconds"，浪费带宽
2. 过多 preload → 与关键资源争抢带宽，反而延迟 LCP
3. prefetch 在弱网下 → 浪费用户流量
4. preload 缺少 `as` 属性 → 浏览器不知道资源类型，可能下载两次

**Q7: 如何实现一个高性能的无限滚动列表？**

A: 结合虚拟列表 + IntersectionObserver + 数据分页：

```jsx
function InfiniteVirtualList({ fetchData }) {
  const [items, setItems] = useState([]);
  const [hasMore, setHasMore] = useState(true);
  const [page, setPage] = useState(1);
  const loadMoreRef = useRef();
  const listRef = useRef();

  // IntersectionObserver 检测到底
  useEffect(() => {
    const observer = new IntersectionObserver(
      ([entry]) => {
        if (entry.isIntersecting && hasMore) {
          loadMore();
        }
      },
      { rootMargin: '300px' }
    );

    if (loadMoreRef.current) observer.observe(loadMoreRef.current);
    return () => observer.disconnect();
  }, [hasMore, page]);

  async function loadMore() {
    const newItems = await fetchData(page);
    if (newItems.length === 0) {
      setHasMore(false);
      return;
    }
    setItems(prev => [...prev, ...newItems]);
    setPage(prev => prev + 1);
  }

  return (
    <>
      <FixedSizeList
        ref={listRef}
        height={600}
        itemCount={items.length}
        itemSize={80}
      >
        {({ index, style }) => (
          <div style={style}>{items[index].name}</div>
        )}
      </FixedSizeList>
      <div ref={loadMoreRef} style={{ height: 1 }} />
      {!hasMore && <div>没有更多了</div>}
    </>
  );
}
```

**Q8: V8 的 Hidden Class 机制是什么？为什么保持对象形状一致性很重要？**

A: V8 为每个对象创建 Hidden Class（也叫 Shape/Map），记录对象的属性布局（属性名、偏移量、类型）。属性相同顺序初始化的对象共享同一个 Hidden Class，V8 通过 Inline Cache 快速访问属性。

如果对象形状频繁变化（增删属性、不同顺序初始化），V8 会创建大量 Hidden Class 并导致 Inline Cache 失效（Megamorphic），属性访问退化为字典查找，性能下降 10-100 倍。

```javascript
// ❌ Bad: 不同对象不同形状
function createPoint(x, y, z) {
  const p = { x };
  if (y !== undefined) p.y = y;  // 有时有 y，有时没有
  if (z !== undefined) p.z = z;
  return p;
}

// ✅ Good: 始终相同形状
function createPoint(x, y = 0, z = 0) {
  return { x, y, z };  // 总是三个属性，相同顺序
}
```

**Q9: 什么是内存泄漏？如何用 Chrome DevTools 检测？**

A: 内存泄漏是指程序不再需要的内存没有被垃圾回收器回收，导致内存持续增长。

Chrome DevTools 检测步骤：
1. **Performance Monitor**：观察 JS Heap Size 是否持续上涨
2. **Memory 面板 → Heap Snapshot**：拍摄多个快照对比，按 "Objects allocated between Snapshot 1 and Snapshot 2" 过滤
3. **Memory 面板 → Allocation Timeline**：记录内存分配时间线，找出未释放的分配
4. **Memory 面板 → Allocation Sampling**：低开销采样，查看哪些函数分配了最多内存

关键操作：反复进入/退出某页面，如果每次退出后内存不恢复到之前水平，就存在泄漏。查看 Heap Snapshot 中的 Detached DOM tree（已分离但未回收的 DOM 节点）。

**Q10: 防抖和节流的实现原理？什么场景用哪个？**

A: 
- **防抖（Debounce）**：每次触发重置定时器，只在最后一次触发后的 delay 时间后执行。适合：搜索输入联想、窗口 resize 后计算布局、表单验证。
- **节流（Throttle）**：固定间隔执行一次，不管触发多频繁。适合：滚动监听、鼠标移动跟踪、FPS 游戏循环。

核心区别：防抖是"等你停下来再做"，节流是"固定频率做"。防抖在高频触发时可能一直不执行（可通过 maxWait 解决）；节流保证最大间隔内必然执行一次。

**Q11: Streaming SSR 的原理是什么？相比传统 SSR 有什么优势？**

A: 传统 SSR 需要等所有数据获取完毕才能生成完整 HTML 并发送。Streaming SSR（流式渲染）允许服务器边获取数据边发送 HTML 片段。

原理：利用 HTTP chunked transfer encoding，先发送已准备好的 HTML（Shell + 占位符），然后每个数据源就绪后，以 `<script>` 标签流式发送对应的 HTML 片段和替换脚本。

优势：1) TTFB 极短（立即开始发送）；2) FCP 快（用户先看到框架）；3) 不阻塞（慢 API 不影响快 API 的展示）。React 18 的 `renderToPipeableStream` + Suspense 原生支持此模式。

**Q12: HTTP/3 基于 QUIC 协议，解决了 HTTP/2 的什么问题？**

A: HTTP/2 的核心问题是 TCP 层的队头阻塞——多路复用在一个 TCP 连接上，如果一个 TCP 包丢失，整个连接的所有流都被阻塞等待重传。

QUIC 解决方案：
1. **基于 UDP**：每个 QUIC 流独立传输，一个流丢包不影响其他流
2. **0-RTT 恢复**：已建立过连接的再次连接可以 0-RTT 发送数据
3. **内置 TLS 1.3**：握手和加密合并，减少 RTT
4. **连接迁移**：用 Connection ID 标识连接（而非 IP:Port），Wi-Fi 切 4G 不断连

**Q13: 浏览器缓存分为几层？请按优先级说明。**

A: 从高到低：
1. **Memory Cache**：存在内存中，最快（~0ms），tab 关闭失效，用于当前页面频繁访问的资源
2. **Service Worker Cache**：可编程控制，支持离线，可存放自定义策略
3. **HTTP Disk Cache**：磁盘缓存，遵循 HTTP 缓存头（强缓存/协商缓存），跨 tab 共享
4. **Push Cache**：HTTP/2 Server Push 缓存，连接级别，时间短暂（~5min）
5. **CDN 边缘缓存**：CDN 节点缓存，对所有用户共享
6. **源站/数据库缓存**：服务端缓存层（Redis等）

**Q14: 什么是关键渲染路径？如何优化它？**

A: 关键渲染路径是浏览器从接收 HTML 到首次渲染的最短必经路径：HTML → DOM + CSSOM → Render Tree → Layout → Paint。

阻塞因素：
- CSS 阻塞渲染：浏览器需要完整 CSSOM 才能渲染
- JS 阻塞 DOM 解析：默认同步脚本会暂停 DOM 构建

优化手段：
1. 关键 CSS 内联到 `<head>`，非关键 CSS 异步加载
2. JS 使用 `defer`（DOMContentLoaded 前按序执行）或 `async`（下载完立即执行）
3. 减小关键资源的体积和数量
4. preload 关键资源
5. 消除不必要的重定向

**Q15: Web Worker 和主线程是如何通信的？有什么限制？**

A: 通过 `postMessage()` 发送消息，`onmessage` 接收消息。数据通过**结构化克隆算法**（Structured Clone Algorithm）序列化传输。

限制：
1. 不能访问 DOM（没有 window、document）
2. 数据传输是拷贝（大数据有序列化开销），可用 **Transferable Objects** 转移所有权（如 ArrayBuffer）
3. 不能使用部分 API（alert、confirm 等）
4. 同源限制

```javascript
// Transferable Objects 零拷贝传输
const buffer = new ArrayBuffer(1024 * 1024 * 100); // 100MB
// 转移所有权而非拷贝，几乎零耗时
worker.postMessage({ data: buffer }, [buffer]);
// 此后主线程的 buffer.byteLength === 0
```

**Q17: Vite 为什么开发环境启动比 Webpack 快？生产构建呢？**

A: 
- **开发环境**：Vite 使用原生 ESM（浏览器直接 import），不需要打包，只需按需编译单个文件。Webpack 需要构建完整的依赖图并打包。Vite 用 esbuild（Go 编写）预构建 node_modules 依赖，极快。
- **HMR**：Vite 的 HMR 只更新变化的模块及其直接依赖，不需要重新打包。Webpack 需要重新构建受影响的 chunk。
- **生产构建**：Vite 使用 Rollup 打包，速度与 Webpack 相当。但可配置 esbuild/SWC 做压缩，比 Terser 快很多。

本质差异：Vite 是 unbundled development（开发时不打包），Webpack 是 bundled development。

**Q18: 如何实现一个性能监控 SDK？需要采集哪些数据？**

A: 核心采集维度：
1. **Web Vitals**：LCP、INP、CLS、FCP、TTFB（通过 PerformanceObserver）
2. **资源加载**：慢资源、失败资源（Resource Timing API）
3. **长任务**：>50ms 的任务（Long Tasks API）
4. **JS 错误**：window.onerror + unhandledrejection
5. **API 请求**：拦截 fetch/XHR，记录耗时和状态码
6. **用户环境**：设备类型、网络质量（navigator.connection）、浏览器版本

关键设计点：采样率控制、数据批量上报（减少请求）、sendBeacon 确保页面卸载前发送、不影响主线程（用 requestIdleCallback 处理非关键逻辑）。

**Q19: 什么是 Layout Thrashing？如何避免？**

A: Layout Thrashing（布局抖动）是指在一帧内交替读写 DOM 布局属性，导致浏览器反复进行强制同步布局计算。

```javascript
// ❌ Layout Thrashing
for (const el of elements) {
  const height = el.offsetHeight;          // 读 → 触发 layout
  el.style.height = height + 10 + 'px';   // 写 → 使 layout 失效
  // 下次循环读 offsetHeight 又会触发 layout！
}
// N 个元素 = N 次强制同步布局

// ✅ 修复：读写分离
const heights = elements.map(el => el.offsetHeight);  // 批量读（1 次 layout）
elements.forEach((el, i) => {
  el.style.height = heights[i] + 10 + 'px';           // 批量写
});
// 只触发 1 次 layout
```

也可以使用 `fastdom` 库自动批量处理读写操作。

**Q21: WebP 和 AVIF 的优劣对比？什么时候用哪个？**

A:
| 维度 | WebP | AVIF |
|------|------|------|
| 压缩率（vs JPEG） | 节省 25-35% | 节省 40-50% |
| 编码速度 | 快 | 慢 3-10x |
| 解码速度 | 快 | 较慢 |
| 最大分辨率 | 16383x16383 | 8193x4097 (Level 5.1) |
| HDR 支持 | 有限 | 完善 |
| 浏览器支持 | 97%+ | 92%+ |

选择策略：AVIF 优先（更好的压缩率），WebP 作为降级，JPEG/PNG 作为兜底。对于需要快速编码的场景（如用户上传实时处理），优先 WebP。大分辨率图片注意 AVIF 的尺寸限制。

**Q23: scheduler.yield() 是什么？它和 requestIdleCallback 有什么区别？**

A: `scheduler.yield()` 是 Scheduler API 的新提案（Chrome 129+），允许在长任务中主动让出主线程控制权给浏览器，然后以高优先级恢复执行。

与 `requestIdleCallback` 的区别：
- **requestIdleCallback**：在浏览器空闲时执行，优先级低，可能长时间不被调度
- **scheduler.yield()**：立即让出，但以**用户可见**优先级重新调度（比 rIC 高），确保快速恢复

```javascript
// scheduler.yield 最适合优化 INP
button.onclick = async () => {
  updateUI();                    // 先处理视觉反馈
  await scheduler.yield();       // 让浏览器渲染这一帧
  doHeavyWork();                 // 再做重计算
  await scheduler.yield();
  moreWork();
};
```

**Q24: 如何实现一个图片渐进式加载方案（从模糊到清晰）？**

A:

```javascript
// LQIP (Low-Quality Image Placeholder) 方案
function ProgressiveImage({ src, placeholder, alt, ...props }) {
  const [currentSrc, setCurrentSrc] = useState(placeholder);
  const [isLoaded, setIsLoaded] = useState(false);

  useEffect(() => {
    const img = new Image();
    img.onload = () => {
      setCurrentSrc(src);
      setIsLoaded(true);
    };
    img.src = src;
  }, [src]);

  return (
    <img
      src={currentSrc}
      alt={alt}
      style={{
        filter: isLoaded ? 'none' : 'blur(20px)',
        transition: 'filter 0.3s ease-out',
      }}
      {...props}
    />
  );
}

// 构建时生成 placeholder（20px 宽的 base64 缩略图）
// sharp input.jpg -resize 20 -blur 5 -quality 20 -o placeholder.jpg
// 然后 base64 内联到 HTML/JSON 中
```

另一种方案是 BlurHash：将图片编码为 20-30 字符的字符串，解码后生成模糊色块占位图，体积极小（<100 bytes）。

**Q25: 如何设计一个完整的前端性能治理体系？**

A: 完整的性能治理体系包含 5 个环节：

```
1. 度量体系（Measure）
   ├── RUM 采集（web-vitals SDK）
   ├── Synthetic 监控（Lighthouse CI）
   └── 自定义业务指标（首屏数据加载完成时间）

2. 分析平台（Analyze）
   ├── 性能大盘（P50/P75/P95 趋势图）
   ├── 多维下钻（按地域、设备、网络、页面）
   └── 竞品对比

3. 优化落地（Optimize）
   ├── 加载优化（CDN/缓存/代码分割/压缩）
   ├── 渲染优化（虚拟列表/合成动画/时间切片）
   ├── 首屏优化（SSR/关键CSS/预加载）
   └── 构建优化（Bundle分析/Tree Shaking）

4. 防劣化（Prevent）
   ├── 性能预算（LCP < 2.5s, Bundle < 200KB）
   ├── CI/CD 卡口（Lighthouse CI score >= 90）
   ├── PR 自动分析（Bundle size diff）
   └── 告警通知（P75 超标触发告警）

5. 持续迭代（Iterate）
   ├── 周报/月报
   ├── 性能回顾会
   └── 新技术跟进（HTTP/3, AVIF, Scheduler API）
```

关键原则：**先度量再优化，先优化高影响因素，持续监控防回退**。性能优化不是一次性工作，而是持续的工程实践。
