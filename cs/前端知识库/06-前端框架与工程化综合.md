|------|-------|-------|
| **响应式原理** | Object.defineProperty | Proxy |
| **组合式API** | ❌ | ✅ Composition API |

### 1.2 响应式原理

#### 1.2.1 Vue 2（Object.defineProperty）
```javascript
// Vue 2响应式实现
function defineReactive(obj, key, val) {
    const dep = new Dep();  // 依赖收集器
    
    Object.defineProperty(obj, key, {
        get() {
            if (Dep.target) {
                dep.depend();  // 收集依赖
            }
            return val;
        },
        set(newVal) {
            if (newVal !== val) {
                val = newVal;
                dep.notify();  // 通知更新
            }
        }
    });
}

// 缺陷：无法检测新增属性
const obj = { a: 1 };
defineReactive(obj, 'a', 1);
obj.b = 2;  // 不会触发更新，需要Vue.set()
```

#### 1.2.2 Vue 3（Proxy）
```javascript
// Vue 3响应式实现
function reactive(target) {
    return new Proxy(target, {
        get(target, key, receiver) {
            track(target, key);  // 收集依赖
            return Reflect.get(target, key, receiver);
        },
        set(target, key, value, receiver) {
            const result = Reflect.set(target, key, value, receiver);
            trigger(target, key);  // 触发更新
            return result;
        }
    });
}

// 优势：自动检测新增属性
const obj = reactive({ a: 1 });
obj.b = 2;  // 自动响应式
```

### 1.3 Composition API vs Options API

#### 1.3.1 Options API（Vue 2风格）
```vue
<script>
export default {
    data() {
        return {
            count: 0,
            user: null
        };
    },
    methods: {
        increment() {
            this.count++;
        },
        fetchUser() {
            // ...
        }
    },
    mounted() {
        this.fetchUser();
    }
};
</script>
```

**问题**：相关逻辑分散（count相关逻辑在data、methods、mounted中）

#### 1.3.2 Composition API（Vue 3推荐）
```vue
<script setup>
import { ref, onMounted } from 'vue';

// 计数器逻辑聚合
const count = ref(0);
const increment = () => count.value++;

// 用户逻辑聚合
const user = ref(null);
const fetchUser = async () => {
    user.value = await fetch('/api/user').then(r => r.json());
};
onMounted(fetchUser);
</script>
```

**优势**：
- 逻辑复用更容易（自定义Hook）
- TypeScript支持更好
- Tree-shaking友好

### 1.4 自定义Hooks（组合式函数）

```javascript
// useCounter.js
import { ref } from 'vue';

export function useCounter(initialValue = 0) {
    const count = ref(initialValue);
    const increment = () => count.value++;
    const decrement = () => count.value--;
    const reset = () => count.value = initialValue;
    
    return { count, increment, decrement, reset };
}

// 使用
<script setup>
import { useCounter } from './useCounter';

const { count, increment, decrement } = useCounter(10);
</script>

<template>
    <div>
        <p>Count: {{ count }}</p>
        <button @click="increment">+</button>
        <button @click="decrement">-</button>
    </div>
</template>
```

**常见Hooks**：
```javascript
// useFetch.js
import { ref, watchEffect } from 'vue';

export function useFetch(url) {
    const data = ref(null);
    const error = ref(null);
    const loading = ref(false);
    
    watchEffect(async () => {
        loading.value = true;
        try {
            const response = await fetch(url.value);
            data.value = await response.json();
        } catch (e) {
            error.value = e;
        } finally {
            loading.value = false;
        }
    });
    
    return { data, error, loading };
}
```

#### 1.7.2 ref vs reactive 深度对比

```javascript
import { ref, reactive, toRefs, isRef, isReactive, unref } from 'vue';

// ========== ref：适用于任意类型 ==========
const count = ref(0);          // 基本类型
const user = ref({ name: 'Alice' }); // 对象（内部自动调用 reactive）

// 访问/修改必须用 .value（模板中自动解包）
count.value++;
user.value.name = 'Bob';
user.value = { name: 'Charlie' }; // ✅ 可以整体替换

// ========== reactive：仅适用于对象/数组 ==========
const state = reactive({
    count: 0,
    list: [1, 2, 3],
    nested: { deep: true }
});

// 直接访问，无需 .value
state.count++;
state.list.push(4);   // ✅ 数组方法也是响应式的
state.nested.deep = false; // ✅ 深层嵌套也是响应式的

// ❌ 不要整体替换 reactive 对象——会丢失响应性
// state = reactive({ count: 1 });  // 错误！

// ========== 解构陷阱与 toRefs ==========
const { count: c } = state;       // ❌ 丢失响应性，c 只是普通值
const { count: c2 } = toRefs(state); // ✅ c2 是 Ref，保持响应性

// ========== 工具函数 ==========
console.log(isRef(count));       // true
console.log(isReactive(state));  // true
console.log(unref(count));       // 0  —— 等价于 isRef(val) ? val.value : val
```

#### 1.7.3 computed 与 watch / watchEffect 对比

```vue
<script setup>
import { ref, computed, watch, watchEffect } from 'vue';

const firstName = ref('张');
const lastName  = ref('三');

// ========== computed：派生状态，有缓存 ==========
// 只读
const fullName = computed(() => `${firstName.value} ${lastName.value}`);

// 可写
const fullNameRW = computed({
    get: () => `${firstName.value} ${lastName.value}`,
    set: (val) => {
        const [f, l] = val.split(' ');
        firstName.value = f;
        lastName.value  = l;
    }
});

// ========== watch：侦听指定数据源，惰性执行 ==========
// 侦听单个 ref
watch(firstName, (newVal, oldVal) => {
    console.log(`firstName: ${oldVal} → ${newVal}`);
});

// 侦听 reactive 对象的某个属性（需用 getter 函数）
const state = reactive({ page: 1, keyword: '' });
watch(
    () => state.page,
    (newPage) => fetchList(newPage)
);

// 侦听多个源
watch(
    [() => state.page, () => state.keyword],
    ([page, keyword], [oldPage, oldKeyword]) => {
        console.log('page or keyword changed');
    }
);

// 深层侦听 + 立即执行
watch(
    () => state,
    (val) => { /* ... */ },
    { deep: true, immediate: true }
);

// ========== watchEffect：自动收集依赖，立即执行 ==========
// 无需显式指定数据源，回调中访问到的响应式数据都会被追踪
const stopWatcher = watchEffect((onCleanup) => {
    console.log(`当前 page = ${state.page}`);
    const controller = new AbortController();
    fetch(`/api/list?page=${state.page}`, { signal: controller.signal });

    // 副作用清理（下次执行前 / 组件卸载时调用）
    onCleanup(() => controller.abort());
});

// 手动停止侦听
stopWatcher();
</script>
```

#### 1.7.4 Composition API vs Options API 对照表

| 维度 | Options API | Composition API |
|------|-------------|-----------------|
| **状态声明** | `data()` 返回对象 | `ref()` / `reactive()` |
| **计算属性** | `computed: { ... }` | `computed(() => ...)` |
| **侦听器** | `watch: { ... }` | `watch()` / `watchEffect()` |
| **方法** | `methods: { ... }` | 普通函数 |
| **生命周期** | `mounted()` 等选项 | `onMounted()` 等组合函数 |
| **逻辑复用** | Mixins（易冲突） | Composables（可组合函数） |
| **TS支持** | 需 `defineComponent` + 类型体操 | 天然友好 |
| **代码组织** | 按选项分散 | 按功能聚合 |

> **最佳实践**：新项目统一使用 `<script setup>` + Composition API；大型项目中把可复用逻辑提取为 `use***` 组合式函数。

### 1.8 Vue3 响应式原理深入

#### 1.8.1 Proxy vs Object.defineProperty 详细对比

| 维度 | Object.defineProperty（Vue2） | Proxy（Vue3） |
|------|-------------------------------|---------------|
| **拦截粒度** | 单个属性 | 整个对象 |
| **新增属性** | ❌ 需 `Vue.set()` | ✅ 自动检测 |
| **删除属性** | ❌ 需 `Vue.delete()` | ✅ 自动检测（`deleteProperty` 陷阱） |
| **数组索引** | ❌ 需重写数组方法 | ✅ 原生支持 |
| **Map/Set/WeakMap** | ❌ 不支持 | ✅ 支持 |
| **性能** | 初始化时递归遍历所有属性 | 惰性代理——访问时才递归 |
| **兼容性** | IE9+ | 不支持 IE（无法 polyfill） |

#### 1.8.2 依赖收集 / 触发更新完整机制

```
核心数据结构：
targetMap: WeakMap<target, Map<key, Set<ReactiveEffect>>>

执行流程：
┌───────────┐    get 拦截     ┌──────────┐
│  组件渲染  │ ──────────────→ │  track()  │  收集当前 activeEffect
│ (effect)  │                 │           │  到 targetMap[target][key]
└───────────┘                 └──────────┘
      ↑
      │ 重新执行                  set 拦截     ┌───────────┐
      └──────────────────────────────────────── │ trigger()  │  从 targetMap 中取出
                                               │            │  所有相关 effect 并执行
                                               └───────────┘
```

#### 1.8.3 简化版响应式系统实现

```javascript
// ========== 全局变量 ==========
let activeEffect = null;
const targetMap = new WeakMap(); // WeakMap<target, Map<key, Set<effect>>>

// ========== 依赖收集 ==========
function track(target, key) {
    if (!activeEffect) return;
    let depsMap = targetMap.get(target);
    if (!depsMap) {
        depsMap = new Map();
        targetMap.set(target, depsMap);
    }
    let dep = depsMap.get(key);
    if (!dep) {
        dep = new Set();
        depsMap.set(key, dep);
    }
    dep.add(activeEffect);        // 将当前 effect 收集到依赖集合
    activeEffect.deps.push(dep);  // 反向记录（用于 cleanup）
}

// ========== 触发更新 ==========
function trigger(target, key) {
    const depsMap = targetMap.get(target);
    if (!depsMap) return;
    const effects = depsMap.get(key);
    if (!effects) return;
    // 创建副本避免无限循环
    const effectsToRun = new Set(effects);
    effectsToRun.forEach(effect => {
        // 避免递归触发自身
        if (effect !== activeEffect) {
            if (effect.scheduler) {
                effect.scheduler(effect); // 调度器（如 queueJob 批量更新）
            } else {
                effect.run();
            }
        }
    });
}

// ========== reactive 实现 ==========
function reactive(target) {
    return new Proxy(target, {
        get(target, key, receiver) {
            const res = Reflect.get(target, key, receiver);
            track(target, key);  // 读取时收集依赖
            // 惰性递归：只有访问到的嵌套对象才会被代理
            if (typeof res === 'object' && res !== null) {
                return reactive(res);
            }
            return res;
        },
        set(target, key, value, receiver) {
            const oldVal = target[key];
            const result = Reflect.set(target, key, value, receiver);
            if (oldVal !== value) {
                trigger(target, key);  // 值变化时触发更新
            }
            return result;
        },
        deleteProperty(target, key) {
            const hadKey = Object.prototype.hasOwnProperty.call(target, key);
            const result = Reflect.deleteProperty(target, key);
            if (hadKey && result) {
                trigger(target, key);  // 删除属性也触发更新
            }
            return result;
        }
    });
}

// ========== ref 实现 ==========
function ref(value) {
    const wrapper = {
        get value() {
            track(wrapper, 'value');
            return value;
        },
        set value(newVal) {
            if (newVal !== value) {
                value = newVal;
                trigger(wrapper, 'value');
            }
        }
    };
    Object.defineProperty(wrapper, '__v_isRef', { value: true });
    return wrapper;
}

// ========== effect（副作用函数） ==========
function effect(fn, options = {}) {
    const effectFn = () => {
        cleanup(effectFn);         // 清除旧依赖，避免分支切换产生遗留依赖
        activeEffect = effectFn;
        effectStack.push(effectFn); // 支持嵌套 effect
        const res = fn();
        effectStack.pop();
        activeEffect = effectStack[effectStack.length - 1];
        return res;
    };
    effectFn.deps = [];
    effectFn.scheduler = options.scheduler;
    if (!options.lazy) {
        effectFn(); // 立即执行一次，触发 track
    }
    return effectFn;
}

const effectStack = [];
function cleanup(effectFn) {
    effectFn.deps.forEach(dep => dep.delete(effectFn));
    effectFn.deps.length = 0;
}
```

#### 1.8.4 Vue3 响应式中的关键优化

```javascript
// 1. 调度器（Scheduler）—— 批量更新，避免重复渲染
const queue = new Set();
let isFlushing = false;

function queueJob(job) {
    queue.add(job);           // Set 自动去重
    if (!isFlushing) {
        isFlushing = true;
        Promise.resolve().then(() => {   // 微任务中批量执行
            queue.forEach(job => job());
            queue.clear();
            isFlushing = false;
        });
    }
}

// 组件的渲染 effect 使用调度器
effect(renderComponent, {
    scheduler: (effect) => queueJob(effect)
});

// 2. shallowRef / shallowReactive —— 性能敏感场景只代理第一层
import { shallowRef, shallowReactive, triggerRef } from 'vue';

const bigList = shallowRef([]);       // 内部数组不会被深层代理
bigList.value = [...bigList.value, newItem]; // 需要替换引用才能触发更新
triggerRef(bigList);                   // 或手动触发

const config = shallowReactive({
    theme: 'dark',
    nested: { color: 'red' }           // nested 内部不是响应式的
});

// 3. markRaw —— 标记对象永远不会被代理
import { markRaw } from 'vue';

const heavyLib = markRaw(createHeavyInstance());
const state = reactive({
    lib: heavyLib   // 不会被 Proxy 代理，提高性能
});

// 4. toRaw —— 获取 Proxy 背后的原始对象
import { toRaw } from 'vue';

const rawState = toRaw(state);  // 跳过 Proxy，用于性能敏感操作或传给第三方库
```

### 1.9 Pinia 状态管理深入

#### 1.9.1 Pinia vs Vuex 对比

| 维度 | Vuex（3/4） | Pinia |
|------|-------------|-------|
| **Vue版本** | Vue2/3（需不同大版本） | Vue2/3 统一 |
| **API风格** | `state / getters / mutations / actions` | `state / getters / actions`（无 mutations） |
| **TypeScript** | 需大量类型声明 | 完美推导 |
| **模块** | 嵌套 modules + namespace | 扁平化 Store，按需导入 |
| **异步** | mutations 必须同步，actions 异步 | actions 同步/异步均可 |
| **体积** | ~1KB | ~1KB |
| **DevTools** | 支持 | 支持（timeline 更友好） |
| **SSR** | 需要额外处理 | 内置支持 |

> **Pinia 是 Vue 官方推荐的状态管理方案（Vuex 5）**。

#### 1.9.2 Options Store vs Setup Store

```javascript
// ============ Options Store（类似 Vuex 风格） ============
import { defineStore } from 'pinia';

export const useCounterStore = defineStore('counter', {
    state: () => ({
        count: 0,
        lastChanged: null
    }),
    getters: {
        doubleCount: (state) => state.count * 2,
        // getter 访问其他 getter
        doubleCountPlusOne() {
            return this.doubleCount + 1;
        }
    },
    actions: {
        increment() {
            this.count++;
            this.lastChanged = Date.now();
        },
        async fetchAndSet() {
            const res = await fetch('/api/count');
            this.count = await res.json();
        }
    }
});

// ============ Setup Store（Composition API 风格，更灵活） ============
export const useCounterStore2 = defineStore('counter2', () => {
    // ref() → state
    const count = ref(0);
    const lastChanged = ref(null);

    // computed() → getters
    const doubleCount = computed(() => count.value * 2);

    // function → actions
    function increment() {
        count.value++;
        lastChanged.value = Date.now();
    }

    async function fetchAndSet() {
        const res = await fetch('/api/count');
        count.value = await res.json();
    }

    // 必须返回所有需要暴露的状态和方法
    return { count, lastChanged, doubleCount, increment, fetchAndSet };
});
```

#### 1.9.3 在组件中使用 Store

```vue
<script setup>
import { storeToRefs } from 'pinia';
import { useCounterStore } from '@/stores/counter';

const counterStore = useCounterStore();

// ❌ 直接解构会丢失响应性
// const { count, doubleCount } = counterStore;

// ✅ 使用 storeToRefs 解构状态和 getters
const { count, doubleCount } = storeToRefs(counterStore);

// actions 可以直接解构（它们是普通函数）
const { increment, fetchAndSet } = counterStore;

// 批量修改状态
function resetStore() {
    counterStore.$patch({
        count: 0,
        lastChanged: null
    });
    // 或使用函数形式（适合复杂修改）
    counterStore.$patch((state) => {
        state.count = 0;
        state.lastChanged = null;
    });
}

// 重置为初始值
function fullReset() {
    counterStore.$reset();
}

// 订阅状态变化
counterStore.$subscribe((mutation, state) => {
    console.log('mutation type:', mutation.type);  // 'direct' | 'patch object' | 'patch function'
    console.log('new state:', state);
    localStorage.setItem('counter', JSON.stringify(state));
});

// 订阅 action 调用
counterStore.$onAction(({ name, args, after, onError }) => {
```

```javascript
            { path: 'role',  component: () => import('@/views/system/Role.vue'),  meta: { roles: ['admin'] } },

function filterRoutes(routes, roles) {
    return routes.filter(route => {
        if (route.meta?.roles) {
            const hasPermission = route.meta.roles.some(role => roles.includes(role));
            if (!hasPermission) return false;
        }
        if (route.children) {
            route.children = filterRoutes(route.children, roles);
        }
        return true;
    });
}

// 4. 登录后动态添加路由
async function initDynamicRoutes() {
    const authStore = useAuthStore();
    const userRoles = authStore.roles; // e.g. ['admin']

    const accessRoutes = filterRoutes(asyncRouteMap, userRoles);

    // 动态添加
    accessRoutes.forEach(route => {
        router.addRoute(route);
    });

    // 最后添加 404 兜底（必须放在最后）
    router.addRoute({ path: '/:pathMatch(.*)*', redirect: '/404' });
}

// 5. 在全局守卫中调用
let routesAdded = false;

        await initDynamicRoutes();
        routesAdded = true;
```

```javascript
// Service Worker缓存策略
// 1. Cache First（静态资源）
self.addEventListener('fetch', (event) => {
    if (event.request.url.includes('/static/')) {
        event.respondWith(
            caches.match(event.request)
                .then(cached => cached || fetch(event.request))
        );
    }
});

// 2. Network First（API请求）
self.addEventListener('fetch', (event) => {
    if (event.request.url.includes('/api/')) {
        event.respondWith(
            fetch(event.request)
                .catch(() => caches.match(event.request))
        );
    }
});

// 3. Stale While Revalidate（平衡方案）
self.addEventListener('fetch', (event) => {
    event.respondWith(
        caches.match(event.request).then(cached => {
            const fetchPromise = fetch(event.request)
                .then(response => {
                    caches.open('v1').then(cache => {
                        cache.put(event.request, response.clone());
                    });
                    return response;
                });
            return cached || fetchPromise;
        })
    );
});
```

## 9. 面试题自查

### Q1: Vue 3比Vue 2快在哪里？

**答案**：
1. **Proxy替代defineProperty**：性能更好，支持数组/对象新增属性检测
2. **编译优化**：静态提升（hoistStatic）将静态节点提升到渲染函数外；事件缓存（cacheHandlers）避免每次重新创建函数
3. **Tree-shaking**：按需引入，未使用的API不会打包
4. **Fragment支持**：组件可返回多个根节点，减少不必要的DOM包裹层
5. **Diff算法优化**：引入PatchFlag标记动态节点，只比较动态部分

### Q2: SSR vs SSG的区别和使用场景？

**答案**：
| 特性 | SSR（服务端渲染） | SSG（静态生成） |
|------|-------------------|-----------------|
| **渲染时机** | 每次请求时 | 构建时 |
| **数据** | 实时数据 | 构建时数据 |
| **服务器压力** | 高 | 低（可CDN） |
| **适用场景** | 个性化内容、实时数据 | 博客、文档、营销页 |

**ISR（增量静态再生成）**：结合两者优点，首次生成静态页面，后续按需更新。

### Q3: 微前端如何实现样式和JS隔离？

**答案**：
**样式隔离**：
- **Shadow DOM**：完全隔离，样式不会泄露或受外部影响
- **Scoped CSS**：自动添加属性选择器（如`[data-qiankun-app1]`）
- **CSS Modules**：编译时生成唯一类名
- **CSS-in-JS**：运行时生成唯一样式

**JS隔离（沙箱）**：
- **Proxy沙箱**：拦截对window的访问，创建fakeWindow
- **快照沙箱**：进入时保存window快照，退出时恢复
- **Legacy沙箱**：记录修改，退出时还原

### Q4: Vite为什么比Webpack快？

**答案**：
1. **开发模式**：
   - Webpack：启动时需要打包所有模块
   - Vite：利用浏览器原生ESM，按需编译请求的模块

2. **HMR（热更新）**：
   - Webpack：需要重新构建整个bundle
   - Vite：只需要重新编译变化的模块，利用HTTP缓存

3. **预构建**：Vite使用esbuild（Go编写）预构建依赖，比Webpack的JS打包快10-100倍

4. **生产构建**：两者都使用Rollup，性能相近

### Q5: Vue的Composition API相比Options API有什么优势？

**答案**：
1. **逻辑复用**：通过组合式函数（Composables）抽取和复用逻辑，替代Mixins避免命名冲突
2. **代码组织**：相关逻辑聚合在一起，而非分散在data、methods、computed等选项中
3. **TypeScript支持**：更好的类型推导，无需额外声明
4. **Tree-shaking**：未使用的API不会打包
5. **更小的打包体积**：压缩后变量名更短

### Q6: React和Vue的Diff算法有什么区别？

**答案**：
**相同点**：
- 都采用同层比较策略（O(n)复杂度）
- 都使用key优化列表渲染

**不同点**：
| 特性 | React | Vue |
|------|-------|-----|
| **遍历方式** | 从左到右单向 | 双端比较（头头、尾尾、头尾、尾头） |
| **移动策略** | 相对位置判断 | 最长递增子序列（Vue3） |
| **静态标记** | 无 | PatchFlag标记动态节点 |
| **更新粒度** | 组件级 | 组件级+节点级 |

### Q7: 微前端的子应用如何通信？

**答案**：
1. **全局状态**（qiankun的initGlobalState）：
```javascript
// 主应用
const actions = initGlobalState({ user: null });
actions.onGlobalStateChange((state) => console.log(state));
actions.setGlobalState({ user: { name: 'Alice' } });

// 子应用

```

2. **CustomEvent事件总线**：适合简单场景
3. **Props传递**：主应用向子应用传递数据和方法
4. **localStorage/sessionStorage**：简单但不推荐
5. **发布订阅模式**：自定义EventEmitter

### Q8: pnpm相比npm/yarn有什么优势？

**答案**：
1. **节省磁盘空间**：通过硬链接复用相同版本的包，可节省70%以上空间
2. **安装速度快**：无需重复下载，直接链接
3. **严格的依赖管理**：非扁平化node_modules结构，防止幽灵依赖（使用未声明的包）
4. **原生Monorepo支持**：内置workspace功能
5. **更好的一致性**：严格按照lockfile安装

**幽灵依赖问题**：npm/yarn扁平化会提升间接依赖，代码中可能引用未在package.json中声明的包。

### Q9: 什么是Module Federation？解决什么问题？

**答案**：
**Module Federation**（模块联邦）是Webpack 5引入的特性，允许多个独立构建的应用共享代码。

**解决的问题**：
1. 微前端场景下的代码共享（如共享React、组件库）
2. 避免重复打包公共依赖
3. 运行时动态加载远程模块
4. 独立部署、独立发布

**核心概念**：
- **Host**：消费远程模块的应用
- **Remote**：提供模块的应用
- **Shared**：共享依赖，确保单例

### Q10: Next.js的App Router和Pages Router有什么区别？

**答案**：
| 特性 | Pages Router | App Router（Next.js 13+） |
|------|-------------|-------------------------|
| **默认组件** | 客户端组件 | 服务端组件（RSC） |
| **数据获取** | getStaticProps/getServerSideProps | 组件内async/await |
| **布局** | 需要HOC或_app.js | 嵌套layout.js原生支持 |
| **流式渲染** | 不支持 | 支持Suspense流式 |
| **路由** | pages目录约定 | app目录约定 |

**RSC优势**：减少客户端JS体积，直接访问后端资源，自动代码分割。

```javascript
    rules: [{ exclude: /node_modules/ }]  // 排除不需要处理的目录
}
```

2. **缓存**：
```javascript
cache: { type: 'filesystem' }  // 持久化缓存
```

3. **并行处理**：
```javascript
use: ['thread-loader', 'babel-loader']  // 多线程
```

4. **DLL预构建**：将不常变化的库提前打包
5. **externals**：排除大型库，使用CDN

### Q13: Vue3的ref和reactive有什么区别？如何选择？

**答案**：
| 特性 | ref | reactive |
|------|-----|----------|
| **数据类型** | 任意类型 | 对象/数组 |
| **访问方式** | .value | 直接访问 |
| **解构** | 需要toRefs | 会丢失响应性 |
| **替换整体** | 可以 | 不可以 |

**选择建议**：
- 基本类型用ref
- 对象/数组且不需要替换整体用reactive
- 需要解构用ref + toRefs
- 简单场景统一用ref更方便

### Q14: 前端监控需要采集哪些数据？如何实现？

**答案**：
**采集数据**：
1. **性能数据**：FCP、LCP、FID、CLS、TTFB（Web Vitals）
2. **错误数据**：JS错误、资源加载错误、Promise未捕获错误
3. **用户行为**：PV/UV、点击、停留时长、页面路径
4. **接口数据**：请求耗时、成功率、错误码

**实现方式**：
```javascript
// 性能监控
const observer = new PerformanceObserver((list) => {
    for (const entry of list.getEntries()) {
        report({ name: entry.name, value: entry.value });
    }
});
observer.observe({ entryTypes: ['largest-contentful-paint'] });

// 错误监控
window.addEventListener('error', (e) => report({ type: 'js', message: e.message }));
window.addEventListener('unhandledrejection', (e) => report({ type: 'promise', reason: e.reason }));
```

### Q15: Monorepo有什么优缺点？如何管理？

**答案**：
**优点**：
1. 代码复用方便，共享组件/工具库
2. 统一的构建、测试、发布流程
3. 原子化提交，跨包修改一次提交
4. 依赖版本统一管理

**缺点**：
1. 仓库体积大，clone/pull慢
2. 权限管理困难，不适合开源
3. CI/CD复杂度增加
4. 需要专门工具支持

**管理工具**：
- **pnpm workspace**：依赖管理
- **Turborepo**：任务编排、缓存
- **Nx**：全功能Monorepo工具
- **Lerna**：版本管理、发布

### Q16: 如何实现前端工程的版本管理和回滚？

**答案**：
1. **构建产物版本化**：
```bash
# 文件名包含版本/hash
main.abc123.js
# 或目录版本化
/v1.2.3/main.js
```

2. **CDN多版本部署**：保留历史版本，切换时修改入口
3. **Nginx快速切换**：
```nginx
location / {
    root /var/www/versions/v1.2.3;  # 修改版本路径即可回滚
}
```

4. **配置中心**：动态下发版本号，前端加载对应资源
5. **Git Tag**：每次发布打标签，回滚时checkout对应版本重新构建

### Q17: 前端灰度发布中，如何避免“灰度流量命中错误资源版本”？

**答案**：
关键是“入口版本、静态资源、配置版本”三者绑定：
1. 灰度入口 HTML 必须引用同版本带 hash 的静态资源。
2. CDN 缓存规则按版本目录隔离，避免不同灰度批次串包。
3. 配置中心按版本下发特性开关，禁止“旧包 + 新配置”错配。

实战里最常见事故不是代码 bug，而是版本资产混用。

### Q18: 微前端落地后，如何治理“公共依赖版本漂移”问题？

**答案**：
1. 设定共享依赖白名单（如 React、路由、状态库）和版本策略。
2. 在 CI 做依赖一致性检查，超出范围阻断发布。
3. 高频公共库做“平台层统一升级窗口”，避免业务线各自升级。
4. 监控运行时冲突告警（多版本 React、重复 polyfill、样式覆盖异常）。

治理目标是“可控演进”，不是强行全仓同日升级。

### Q19: 前端监控如何从“采集很多”走向“可行动告警”？

**答案**：
建议建立分级告警：
1. P0：核心路径不可用（支付页白屏率、登录失败率）直接值班通知。
2. P1：体验劣化（LCP/P99、JS 错误率异常）进入快速处置。
3. P2：趋势告警用于优化，不打扰值班。

同时给每条告警绑定 runbook 和 owner。没有动作方案的告警应降级为看板观察。

### Q20: 工程化改造（如 Monorepo、构建迁移）怎么衡量“真的成功”？

**答案**：
用前后对比指标验收，而不是“团队感觉更好了”：
1. 构建时长、增量构建命中率、CI 成本。
2. 发布失败率、回滚耗时、跨项目改动交付周期。
3. 开发体验指标（本地启动时间、热更新时间）。

只有这些指标持续改善，工程化改造才算完成闭环。

### 开放式设计题

**D1：设计一个大型前端项目的Monorepo工程体系（20+子包），核心基建怎么搭？**

**参考思路**：
- 工具选型：pnpm workspace + Turborepo（构建缓存+并行）或 Nx（依赖图+受影响分析）
- 包划分：UI组件库/工具函数/业务SDK/应用入口 分层、明确依赖方向（上层→下层，禁止反向）
- 构建优化：增量构建（只构建变更的包）、远程缓存（Turborepo Remote Cache）、并行构建
- 版本管理：Changesets管理版本号和CHANGELOG、统一发布流程
- CI优化：affected命令只测试/构建受影响的包、Docker Layer Cache

**D2：微前端架构中子应用间样式污染如何彻底解决？**

**参考思路**：
- Shadow DOM：Web Components天然样式隔离，但第三方库兼容性差
- CSS Modules/Scoped：编译时加哈希前缀，但全局样式（reset/antd）仍需处理
- CSS-in-JS：运行时生成唯一类名（Emotion/styled-components），性能有开销
- 动态样式加载卸载：子应用mount时加载CSS、unmount时移除所有style标签
- 推荐方案：BEM命名约定（子应用前缀）+ CSS Modules + 严格禁止全局样式覆盖
