## 1. Vue框架深入

### 1.1 Vue 3 vs Vue 2

| 特性 | Vue 2 | Vue 3 |
|------|-------|-------|
| **响应式原理** | Object.defineProperty | Proxy |
| **组合式API** | ❌ | ✅ Composition API |
| **性能** | 基准 | 快2倍、内存减半 |
| **Tree-shaking** | ❌ | ✅ |
| **TypeScript** | 支持较弱 | 完整支持 |

### 1.2 响应式原理

#### 1.2.1 Vue 2（Object.defineProperty）
```javascript
// Vue 2响应式实现
function defineReactive(obj, key, val) {
    const dep = new Dep();  // 依赖收集器
    
    Object.defineProperty(obj, key, {
        get() {
            if (Dep.target) {
                dep.depend();  // 收集依赖
            }
            return val;
        },
        set(newVal) {
            if (newVal !== val) {
                val = newVal;
                dep.notify();  // 通知更新
            }
        }
    });
}

// 缺陷：无法检测新增属性
const obj = { a: 1 };
defineReactive(obj, 'a', 1);
obj.b = 2;  // 不会触发更新，需要Vue.set()
```

#### 1.2.2 Vue 3（Proxy）
```javascript
// Vue 3响应式实现
function reactive(target) {
    return new Proxy(target, {
        get(target, key, receiver) {
            track(target, key);  // 收集依赖
            return Reflect.get(target, key, receiver);
        },
        set(target, key, value, receiver) {
            const result = Reflect.set(target, key, value, receiver);
            trigger(target, key);  // 触发更新
            return result;
        }
    });
}

// 优势：自动检测新增属性
const obj = reactive({ a: 1 });
obj.b = 2;  // 自动响应式
```

### 1.3 Composition API vs Options API

#### 1.3.1 Options API（Vue 2风格）
```vue
<script>
export default {
    data() {
        return {
            count: 0,
            user: null
        };
    },
    methods: {
        increment() {
            this.count++;
        },
        fetchUser() {
            // ...
        }
    },
    mounted() {
        this.fetchUser();
    }
};
</script>
```

**问题**：相关逻辑分散（count相关逻辑在data、methods、mounted中）

#### 1.3.2 Composition API（Vue 3推荐）
```vue
<script setup>
import { ref, onMounted } from 'vue';

// 计数器逻辑聚合
const count = ref(0);
const increment = () => count.value++;

// 用户逻辑聚合
const user = ref(null);
const fetchUser = async () => {
    user.value = await fetch('/api/user').then(r => r.json());
};
onMounted(fetchUser);
</script>
```

**优势**：
- 逻辑复用更容易（自定义Hook）
- TypeScript支持更好
- Tree-shaking友好

### 1.4 自定义Hooks（组合式函数）

```javascript
// useCounter.js
import { ref } from 'vue';

export function useCounter(initialValue = 0) {
    const count = ref(initialValue);
    const increment = () => count.value++;
    const decrement = () => count.value--;
    const reset = () => count.value = initialValue;
    
    return { count, increment, decrement, reset };
}

// 使用
<script setup>
import { useCounter } from './useCounter';

const { count, increment, decrement } = useCounter(10);
</script>

<template>
    <div>
        <p>Count: {{ count }}</p>
        <button @click="increment">+</button>
        <button @click="decrement">-</button>
    </div>
</template>
```

**常见Hooks**：
```javascript
// useFetch.js
import { ref, watchEffect } from 'vue';

export function useFetch(url) {
    const data = ref(null);
    const error = ref(null);
    const loading = ref(false);
    
    watchEffect(async () => {
        loading.value = true;
        try {
            const response = await fetch(url.value);
            data.value = await response.json();
        } catch (e) {
            error.value = e;
        } finally {
            loading.value = false;
        }
    });
    
    return { data, error, loading };
}
```

```javascript
// stores/user.js
import { defineStore } from 'pinia';

export const useUserStore = defineStore('user', {
    state: () => ({
        user: null,
        token: null
    }),
    getters: {
        isLoggedIn: (state) => !!state.token,
        userName: (state) => state.user?.name || 'Guest'

            this.token = data.token;

        logout() {
            this.user = null;
            this.token = null;
        }
    }
});
```

## 2. Next.js与SSR

### 2.1 渲染模式对比

| 模式 | 渲染位置 | 首屏加载 | SEO | 交互性 |
|------|----------|----------|-----|--------|
| **CSR** | 客户端 | 慢（需下载JS） | ❌ | 快 |
| **SSR** | 服务端 | 快（返回HTML） | ✅ | 需hydration |
| **SSG** | 构建时 | 最快（静态HTML） | ✅ | 需hydration |
| **ISR** | 按需生成 | 快 | ✅ | 需hydration |

### 2.2 Next.js核心概念

#### 2.2.1 页面路由
```javascript
// pages/index.js → /
// pages/about.js → /about
// pages/blog/[slug].js → /blog/:slug
// pages/api/user.js → /api/user（API路由）

// pages/blog/[slug].js
export default function BlogPost({ post }) {
    return <div>{post.title}</div>;
}

export async function getStaticPaths() {
    // 返回所有可能的路径
    const posts = await fetchAllPosts();
    return {
        paths: posts.map(p => ({ params: { slug: p.slug } })),
        fallback: false  // 404 for non-existent paths
    };
}

export async function getStaticProps({ params }) {
    // 获取单篇文章数据
    const post = await fetchPost(params.slug);
    return {
        props: { post },
        revalidate: 60  // ISR：60秒后重新生成
    };
}
```

#### 2.2.2 数据获取方式

**SSG（静态生成）**：
```javascript
// 构建时生成HTML
export async function getStaticProps() {
    const data = await fetch('https://api.example.com/data');
    return {
        props: { data }
    };
}
```

**SSR（服务端渲染）**：
```javascript
// 每次请求时生成HTML
export async function getServerSideProps(context) {
    const { req, res, query } = context;
    const data = await fetch(`https://api.example.com/data?id=${query.id}`);
    return {
        props: { data }
    };
}
```

**ISR（增量静态再生成）**：
```javascript
// 首次访问生成，后续按需更新
export async function getStaticProps() {
    const data = await fetch('https://api.example.com/data');
    return {
        props: { data },
        revalidate: 10  // 10秒后过期，后台重新生成
    };
}
```

## 3. 微前端架构

### 3.1 微前端解决的问题

**单体前端的问题**：
- 代码库过大，构建慢
- 技术栈统一，难以升级
- 团队协作冲突

**微前端方案**：
```
┌──────────────────────────────────────┐
│         主应用（Shell）                │
├──────────────────────────────────────┤
│ ┌──────┐  ┌──────┐  ┌──────┐        │
│ │子应用A│  │子应用B│  │子应用C│        │
│ │(React)│  │(Vue) │  │(Angular)│     │
│ └──────┘  └──────┘  └──────┘        │
└──────────────────────────────────────┘
```

### 3.2 实现方案

#### 3.2.1 iframe（最简单）
```html
<!-- 主应用 -->
<div id="app">
    <iframe src="https://子应用A.com" />
    <iframe src="https://子应用B.com" />
</div>
```

**优点**：
- 完全隔离（CSS、JS）
- 实现简单

**缺点**：
- SEO不友好
- 性能差（多个独立页面）
- 通信复杂（postMessage）
- URL同步困难

#### 3.2.2 single-spa（主流方案）
```javascript
// 主应用
import { registerApplication, start } from 'single-spa';

// 注册子应用
registerApplication({
    name: 'app1',
    app: () => System.import('https://app1.com/main.js'),
    activeWhen: '/app1',  // 路由匹配时加载
    customProps: { authToken: 'xxx' }  // 传递给子应用的数据
});

registerApplication({
    name: 'app2',
    app: () => System.import('https://app2.com/main.js'),
    activeWhen: '/app2'
});

start();

// 子应用（React）
import React from 'react';
import ReactDOM from 'react-dom';
import singleSpaReact from 'single-spa-react';
import App from './App';

const lifecycles = singleSpaReact({
    React,
    ReactDOM,
    rootComponent: App,
    errorBoundary: (err) => <div>Error: {err.message}</div>
});

export const { bootstrap, mount, unmount } = lifecycles;
```

#### 3.2.3 qiankun（阿里开源，基于single-spa）

- JS沙箱（Proxy）

- 通信机制（全局状态）

#### 8.4.2 缓存策略
```javascript
// Service Worker缓存策略
// 1. Cache First（静态资源）
self.addEventListener('fetch', (event) => {
    if (event.request.url.includes('/static/')) {
        event.respondWith(
            caches.match(event.request)
                .then(cached => cached || fetch(event.request))
        );
    }
});

// 2. Network First（API请求）
self.addEventListener('fetch', (event) => {
    if (event.request.url.includes('/api/')) {
        event.respondWith(
            fetch(event.request)
                .catch(() => caches.match(event.request))
        );
    }
});

// 3. Stale While Revalidate（平衡方案）
self.addEventListener('fetch', (event) => {
    event.respondWith(
        caches.match(event.request).then(cached => {
            const fetchPromise = fetch(event.request)
                .then(response => {
                    caches.open('v1').then(cache => {
                        cache.put(event.request, response.clone());
                    });
                    return response;
                });
            return cached || fetchPromise;
        })
    );
});
```

## 9. 面试题自查

### Q1: Vue 3比Vue 2快在哪里？

**答案**：
1. **Proxy替代defineProperty**：性能更好，支持数组/对象新增属性检测
2. **编译优化**：静态提升（hoistStatic）将静态节点提升到渲染函数外；事件缓存（cacheHandlers）避免每次重新创建函数
3. **Tree-shaking**：按需引入，未使用的API不会打包
4. **Fragment支持**：组件可返回多个根节点，减少不必要的DOM包裹层
5. **Diff算法优化**：引入PatchFlag标记动态节点，只比较动态部分

### Q2: SSR vs SSG的区别和使用场景？

**答案**：
| 特性 | SSR（服务端渲染） | SSG（静态生成） |
|------|-------------------|-----------------|
| **渲染时机** | 每次请求时 | 构建时 |
| **数据** | 实时数据 | 构建时数据 |
| **服务器压力** | 高 | 低（可CDN） |
| **适用场景** | 个性化内容、实时数据 | 博客、文档、营销页 |

**ISR（增量静态再生成）**：结合两者优点，首次生成静态页面，后续按需更新。

### Q3: 微前端如何实现样式和JS隔离？

**答案**：
**样式隔离**：
- **Shadow DOM**：完全隔离，样式不会泄露或受外部影响
- **Scoped CSS**：自动添加属性选择器（如`[data-qiankun-app1]`）
- **CSS Modules**：编译时生成唯一类名
- **CSS-in-JS**：运行时生成唯一样式

**JS隔离（沙箱）**：
- **Proxy沙箱**：拦截对window的访问，创建fakeWindow
- **快照沙箱**：进入时保存window快照，退出时恢复
- **Legacy沙箱**：记录修改，退出时还原

### Q4: Vite为什么比Webpack快？

**答案**：
1. **开发模式**：
   - Webpack：启动时需要打包所有模块
   - Vite：利用浏览器原生ESM，按需编译请求的模块

2. **HMR（热更新）**：
   - Webpack：需要重新构建整个bundle
   - Vite：只需要重新编译变化的模块，利用HTTP缓存

3. **预构建**：Vite使用esbuild（Go编写）预构建依赖，比Webpack的JS打包快10-100倍

4. **生产构建**：两者都使用Rollup，性能相近

### Q5: Vue的Composition API相比Options API有什么优势？

**答案**：
1. **逻辑复用**：通过组合式函数（Composables）抽取和复用逻辑，替代Mixins避免命名冲突
2. **代码组织**：相关逻辑聚合在一起，而非分散在data、methods、computed等选项中
3. **TypeScript支持**：更好的类型推导，无需额外声明
4. **Tree-shaking**：未使用的API不会打包
5. **更小的打包体积**：压缩后变量名更短

### Q6: React和Vue的Diff算法有什么区别？

**答案**：
**相同点**：
- 都采用同层比较策略（O(n)复杂度）
- 都使用key优化列表渲染

**不同点**：
| 特性 | React | Vue |
|------|-------|-----|
| **遍历方式** | 从左到右单向 | 双端比较（头头、尾尾、头尾、尾头） |
| **移动策略** | 相对位置判断 | 最长递增子序列（Vue3） |
| **静态标记** | 无 | PatchFlag标记动态节点 |
| **更新粒度** | 组件级 | 组件级+节点级 |

### Q7: 微前端的子应用如何通信？

### Q17: 前端灰度发布中，如何避免“灰度流量命中错误资源版本”？

**答案**：
关键是“入口版本、静态资源、配置版本”三者绑定：
1. 灰度入口 HTML 必须引用同版本带 hash 的静态资源。
2. CDN 缓存规则按版本目录隔离，避免不同灰度批次串包。
3. 配置中心按版本下发特性开关，禁止“旧包 + 新配置”错配。

实战里最常见事故不是代码 bug，而是版本资产混用。

### Q18: 微前端落地后，如何治理“公共依赖版本漂移”问题？

**答案**：
1. 设定共享依赖白名单（如 React、路由、状态库）和版本策略。
2. 在 CI 做依赖一致性检查，超出范围阻断发布。
3. 高频公共库做“平台层统一升级窗口”，避免业务线各自升级。
4. 监控运行时冲突告警（多版本 React、重复 polyfill、样式覆盖异常）。

治理目标是“可控演进”，不是强行全仓同日升级。

**答案**：
建议建立分级告警：
1. P0：核心路径不可用（支付页白屏率、登录失败率）直接值班通知。
2. P1：体验劣化（LCP/P99、JS 错误率异常）进入快速处置。
3. P2：趋势告警用于优化，不打扰值班。

同时给每条告警绑定 runbook 和 owner。没有动作方案的告警应降级为看板观察。

### Q20: 工程化改造（如 Monorepo、构建迁移）怎么衡量“真的成功”？

**答案**：
用前后对比指标验收，而不是“团队感觉更好了”：
1. 构建时长、增量构建命中率、CI 成本。
2. 发布失败率、回滚耗时、跨项目改动交付周期。
3. 开发体验指标（本地启动时间、热更新时间）。

只有这些指标持续改善，工程化改造才算完成闭环。

## 总结

**现代前端全景**：

1. **框架**：Vue 3（Composition API、Pinia）、React（Hooks）、Next.js（SSR/SSG/RSC）

6. **CI/CD**：GitHub Actions、Docker多阶段构建
7. **监控**：Sentry、自定义埋点、性能监控

- Vue 3响应式原理（Proxy vs defineProperty）

- pnpm的硬链接与幽灵依赖
- Module Federation模块联邦
