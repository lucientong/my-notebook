## 1. 并发基础概念

### 1.1 并发 vs 并行

| 概念 | 说明 | 比喻 |
|------|------|------|
| **并发（Concurrency）** | 多个任务交替执行（逻辑上同时） | 一个人同时做多件事（切换） |
| **并行（Parallelism）** | 多个任务真正同时执行（物理上同时） | 多个人同时做多件事 |

**示例**：
```
并发（单核CPU）：
任务A: ---|---|---|---
任务B:   ---|---|---|---
时间: ═══════════════════>

并行（多核CPU）：
任务A: ══════════════════>
任务B: ══════════════════>
时间: ═══════════════════>
```

### 1.2 进程 vs 线程 vs 协程

| 概念 | 资源 | 调度 | 开销 | 通信 |
|------|------|------|------|------|
| **进程（Process）** | 独立内存空间 | 操作系统调度 | 大（MB级） | IPC（管道、消息队列） |
| **线程（Thread）** | 共享进程内存 | 操作系统调度 | 中（KB级） | 共享内存 |
| **协程（Goroutine/Coroutine）** | 共享线程栈 | 用户态调度 | 小（2KB起） | Channel/Queue |

**典型应用**：
- **进程**：Nginx多进程、Chrome多进程
- **线程**：Java线程池、Python线程
- **协程**：Go goroutine、Python asyncio

## 2. Go并发模型

### 2.1 Goroutine

**创建goroutine**：
```go
// 普通函数
go func() {
    fmt.Println("Hello from goroutine")
}()

// 匿名函数
go func(name string) {
    fmt.Println("Hello", name)
}("张三")

// 阻塞主goroutine
time.Sleep(1 * time.Second)
```

**goroutine特点**：
- 轻量级（初始栈2KB，可动态扩展）
- M:N调度（M个goroutine映射到N个OS线程）
- GMP模型：G（goroutine）、M（Machine线程）、P（Processor逻辑处理器）

### 2.2 Channel

**无缓冲Channel**（同步）：
```go
ch := make(chan int)

// 发送（阻塞直到接收）
go func() {
    ch <- 42
}()

// 接收（阻塞直到发送）
value := <-ch
fmt.Println(value)  // 42
```

**有缓冲Channel**（异步）：
```go
ch := make(chan int, 3)  // 缓冲大小3

ch <- 1
ch <- 2
ch <- 3
// ch <- 4  // 阻塞，缓冲已满

fmt.Println(<-ch)  // 1
fmt.Println(<-ch)  // 2
```

**关闭Channel**：
```go
ch := make(chan int, 2)
ch <- 1
ch <- 2
close(ch)  // 关闭

// 接收直到关闭
for v := range ch {
    fmt.Println(v)  // 1, 2
}

// 检查是否关闭
v, ok := <-ch
if !ok {
    fmt.Println("channel已关闭")
}
```

**单向Channel**：
```go
// 只能发送
func send(ch chan<- int) {
    ch <- 42
}

// 只能接收
func receive(ch <-chan int) {
    value := <-ch
}
```

### 2.3 Select多路复用

```go
ch1 := make(chan int)
ch2 := make(chan string)

select {
case v := <-ch1:
    fmt.Println("收到int:", v)
case v := <-ch2:
    fmt.Println("收到string:", v)
case <-time.After(1 * time.Second):
    fmt.Println("超时")
default:
    fmt.Println("无数据")
}
```

**应用场景**：
```go
// 超时控制
func doWithTimeout(fn func() error, timeout time.Duration) error {
    done := make(chan error)
    
    go func() {
        done <- fn()
    }()
    
    select {
    case err := <-done:
        return err
    case <-time.After(timeout):
        return errors.New("超时")
    }
}
```

### 2.4 WaitGroup

```go
import "sync"

var wg sync.WaitGroup

for i := 0; i < 10; i++ {
    wg.Add(1)

        defer wg.Done()
```

### 2.5 Context

**传递取消信号**：
```go
import "context"

func worker(ctx context.Context, id int) {
    for {
        select {
        case <-ctx.Done():
            fmt.Println("worker", id, "退出")
            return
        default:
            fmt.Println("worker", id, "工作中")
            time.Sleep(500 * time.Millisecond)
        }
    }
}

func main() {
    ctx, cancel := context.WithCancel(context.Background())
    
    for i := 0; i < 3; i++ {
        go worker(ctx, i)
    }
    
    time.Sleep(2 * time.Second)
    cancel()  // 取消所有worker
    time.Sleep(1 * time.Second)
}
```

**超时控制**：
```go
ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
defer cancel()

select {
case <-doSomething():
    fmt.Println("完成")
case <-ctx.Done():
    fmt.Println("超时:", ctx.Err())
}
```

**传递值**：
```go

userID := ctx.Value("user_id").(int)
```

## 3. Go Channel底层实现

### 3.1 hchan 数据结构

Channel 在 Go 运行时的底层实现是 `hchan` 结构体：

```go
// runtime/chan.go（简化）
type hchan struct {
    qcount   uint           // 当前缓冲区中的元素数量
    dataqsiz uint           // 缓冲区容量（make(chan T, N) 中的 N）
    buf      unsafe.Pointer // 环形缓冲区指针
    elemsize uint16         // 每个元素的大小
    closed   uint32         // 是否关闭（0=open, 1=closed）
    sendx    uint           // 发送索引（环形缓冲区写位置）
    recvx    uint           // 接收索引（环形缓冲区读位置）
    recvq    waitq          // 等待接收的 goroutine 队列
    sendq    waitq          // 等待发送的 goroutine 队列
    lock     mutex          // 保护 hchan 的互斥锁
}

type waitq struct {
    first *sudog  // 等待队列头
    last  *sudog  // 等待队列尾
}

type sudog struct {
    g    *g          // 等待的 goroutine
    elem unsafe.Pointer // 数据指针
    next *sudog
    prev *sudog
}
```

```
hchan 结构示意图（有缓冲 channel）：

┌──────────────────────────────────────┐
│ hchan                                 │
├──────────────────────────────────────┤
│ qcount=2  dataqsiz=4                 │
│                                       │
│ buf → [elem0][elem1][    ][    ]     │  ← 环形缓冲区
│        recvx=0       sendx=2          │
│                                       │
│ sendq → (空)                          │  ← 等待发送的goroutine
│ recvq → (空)                          │  ← 等待接收的goroutine
│                                       │
│ lock (mutex)                          │
└──────────────────────────────────────┘
```

---

### 3.2 发送与接收流程

**发送流程（`ch <- value`）**：

```
1. 加锁 hchan.lock

2. 检查 recvq 是否有等待接收的 goroutine
   ├── 有 → 直接将数据拷贝给该 goroutine（不经过缓冲区），唤醒它
   └── 无 → 继续

3. 检查缓冲区是否有空间（qcount < dataqsiz）
   ├── 有 → 将数据拷贝到 buf[sendx]，sendx++，qcount++
   └── 无 → 继续

4. 缓冲区已满（或无缓冲 channel）
   - 创建 sudog，将当前 goroutine 加入 sendq
   - 调用 gopark() 挂起当前 goroutine
   - 等待被接收方唤醒

5. 解锁
```

**接收流程（`value := <-ch`）**：

```
1. 加锁 hchan.lock

2. 检查 sendq 是否有等待发送的 goroutine
   ├── 有（无缓冲）→ 直接从发送方拷贝数据，唤醒它
   ├── 有（有缓冲且满）→ 从 buf[recvx] 取数据，将发送方数据写入 buf[sendx]，唤醒它
   └── 无 → 继续

3. 检查缓冲区是否有数据（qcount > 0）
   ├── 有 → 从 buf[recvx] 取数据，recvx++，qcount--
   └── 无 → 继续

4. 缓冲区为空
   - 创建 sudog，将当前 goroutine 加入 recvq
   - 调用 gopark() 挂起当前 goroutine
   - 等待被发送方唤醒

5. 解锁
```

> **关键优化**：当有等待方时，数据直接在两个 goroutine 之间拷贝（bypass 缓冲区），减少一次内存拷贝。

---

### 3.3 无缓冲 vs 有缓冲

| 特性 | 无缓冲 `make(chan T)` | 有缓冲 `make(chan T, N)` |
|------|----------------------|-------------------------|
| 缓冲区 | 无（`dataqsiz=0`） | 环形缓冲区大小 N |
| 发送行为 | 阻塞直到有接收方 | 缓冲未满则立即返回 |
| 接收行为 | 阻塞直到有发送方 | 缓冲非空则立即返回 |
| 同步语义 | **同步**（发送-接收配对） | **异步**（通过缓冲区解耦） |
| 适用场景 | goroutine 间同步通信 | 生产者-消费者缓冲 |

**影响**：

### 4.2 多线程（threading）

```python
import threading
import time

def worker(name):
    print(f"Worker {name} 开始")
    time.sleep(1)
    print(f"Worker {name} 完成")

threads = []
for i in range(5):
    t = threading.Thread(target=worker, args=(i,))
    threads.append(t)
    t.start()

# 等待所有线程完成
for t in threads:
    t.join()

print("全部完成")
```

**线程锁**：
```python
import threading

counter = 0
lock = threading.Lock()

def increment():
    global counter
    for _ in range(100000):
        with lock:  # 加锁
            counter += 1

threads = [threading.Thread(target=increment) for _ in range(10)]
for t in threads:
    t.start()
for t in threads:
    t.join()

print(counter)  # 1000000（正确）
```

#### 6.1.1 Mutex底层实现原理（深度剖析）⭐⭐⭐

**为什么需要了解Mutex底层？**

##### Go sync.Mutex的演进（从简单到复杂）

**最简单的Mutex实现（伪代码）**：
```go
type SimpleMutex struct {
    locked int32  // 0=未锁定, 1=已锁定
}

func (m *SimpleMutex) Lock() {
    for {
        // CAS操作：如果locked=0，设为1并返回true
        if atomic.CompareAndSwapInt32(&m.locked, 0, 1) {
            return  // 成功获取锁
        }
        // 否则，继续循环（自旋）
    }
}

func (m *SimpleMutex) Unlock() {
    atomic.StoreInt32(&m.locked, 0)
}
```

**问题**：
- ❌ CPU空转（busy-wait），浪费资源
- ❌ 不公平，可能有线程一直抢不到锁（饥饿）

---

**Go sync.Mutex的实际实现（4种模式）**

```go
type Mutex struct {
    state int32   // 状态字（锁定位、唤醒位、饥饿模式、等待者计数）
    sema  uint32  // 信号量（用于休眠/唤醒goroutine）
}

// state的位布局（32位）：
// [0]:     锁定位（1=已锁定）
// [1]:     唤醒位（1=有goroutine被唤醒）
// [2]:     饥饿模式位（1=饥饿模式）
// [3-31]:  等待者计数（最多2^29-1个等待者）
```

**Mutex的4种模式**：

```
1. 正常模式（Normal Mode）
   - goroutine先自旋（spin）几次，尝试CAS获取锁
   - 失败后进入等待队列，休眠（futex）
   - 被唤醒的goroutine与新到来的goroutine竞争
   - 新goroutine更容易获得锁（在CPU上运行，无需唤醒）

2. 饥饿模式（Starvation Mode）
   - 当goroutine等待超过1ms未获得锁，切换到饥饿模式
   - 锁直接交给队列头部的goroutine（公平）
   - 新到来的goroutine不自旋，直接入队
   - 最后一个goroutine或等待时间<1ms，退出饥饿模式

3. 自旋优化（Spinning）
   - 在单核或等待队列过长时，不自旋
   - 最多自旋4次（runtime_canSpin检查）
   - 自旋时调用PAUSE指令（减少CPU功耗）

4. 信号量休眠（Semaphore Sleep）
   - 自旋失败后，通过runtime_SemacquireMutex休眠
   - Unlock时通过runtime_Semrelease唤醒
```

**示例：Lock的执行流程**

```
场景：goroutine A持有锁，goroutine B尝试获取锁

1. B调用Lock()
2. CAS尝试获取锁（失败，state[0]=1）
3. 自旋4次（每次PAUSE + CAS）
4. 自旋失败，state[3-31] += 1（等待者+1）
5. 调用runtime_SemacquireMutex(sema)，B休眠
6. A调用Unlock()
7. state[0] = 0（解锁）
8. 调用runtime_Semrelease(sema)，唤醒B
9. B被唤醒，与新到来的C竞争
10. 如果B再次失败 && 等待>1ms → 切换到饥饿模式
```

##### CAS（Compare-And-Swap）底层原理⭐⭐⭐

**什么是CAS？**

CAS是一种**无锁（Lock-Free）的原子操作**：
```
CAS(addr, old, new):
    if *addr == old:
        *addr = new
        return true
    else:
        return false
```

**CPU指令：CMPXCHG（x86架构）**

```asm
; CAS的汇编实现（x86-64）
; RAX = old, RBX = new, [addr] = 当前值

LOCK CMPXCHG [addr], RBX

; LOCK前缀：保证原子性（锁总线或缓存行）
; CMPXCHG：比较并交换
;   - 如果 [addr] == RAX，则 [addr] = RBX，ZF=1
;   - 否则，RAX = [addr]，ZF=0
```

**为什么CAS需要LOCK前缀？**

```
多核CPU的缓存一致性问题：

CPU1 缓存:  [addr] = 0
CPU2 缓存:  [addr] = 0

时刻T1: CPU1读取[addr]=0，准备写入1
时刻T2: CPU2读取[addr]=0，准备写入1
时刻T3: CPU1写入1
时刻T4: CPU2写入1
→ 两次CAS都成功（错误！）

LOCK前缀的作用：
- 锁定总线（早期CPU）：禁止其他CPU访问内存
- 锁定缓存行（现代CPU，MESI协议）：
  - 独占该缓存行（Exclusive）
  - 其他CPU必须等待或失效
  - 保证原子性
```

**Go中的CAS示例**：

```go
import "sync/atomic"

var counter int64 = 0

// CAS实现无锁自增
func increment() {
    for {
        old := atomic.LoadInt64(&counter)
        new := old + 1
        if atomic.CompareAndSwapInt64(&counter, old, new) {
            return  // 成功
        }
        // 失败，说明其他goroutine修改了counter，重试
    }
}

// 等价于
atomic.AddInt64(&counter, 1)  // 底层也是CAS循环
```

**CAS的ABA问题**：

```
场景：栈的pop操作

初始状态：stack = A → B → C

线程1：执行pop()
  - 读取head = A
  - 准备CAS(head, A, B)
  - 被挂起...

线程2：
  - pop() → A出栈，head = B
  - pop() → B出栈，head = C
  - push(A) → A重新入栈，head = A

线程1恢复：
  - CAS(head, A, B) → 成功！（错误！）
  - 但此时A → C，不是A → B

问题：CAS只检查值是否相等，不检查是否是"同一个"对象

解决方案：
1. 版本号（Version）：CAS(head, <A, version=1>, <B, version=2>)
2. 引用计数
3. Hazard Pointer
```

##### 内存屏障（Memory Barrier）⭐⭐⭐

**为什么需要内存屏障？**

现代CPU为了性能，会对指令**重排序（Reordering）**：

```go
// 代码顺序：
a = 1        // 写操作1
b = 2        // 写操作2

// CPU可能执行顺序：
b = 2        // 先执行写操作2
a = 1        // 后执行写操作1

// 单线程下，重排序不影响结果
// 多线程下，可能导致错误！
```

**示例：双重检查锁（DCL）的错误实现**

```go
var instance *Singleton
var mu sync.Mutex

func GetInstance() *Singleton {
    if instance == nil {  // 检查1（无锁）
        mu.Lock()
        if instance == nil {  // 检查2（有锁）
            instance = &Singleton{}  // 问题：可能重排序！
        }
        mu.Unlock()
    }
    return instance
}

// 问题：instance = &Singleton{} 可能分解为：
// 1. 分配内存 → ptr
// 2. instance = ptr （先赋值！）
// 3. 初始化ptr指向的对象（后初始化！）
//
// 另一个线程读取instance != nil，但对象未初始化（野指针）！
```

**内存屏障的类型**：

```
1. LoadLoad Barrier（读-读屏障）
   Load1; LoadLoad; Load2
   → 保证Load1的数据在Load2之前加载

2. StoreStore Barrier（写-写屏障）
   Store1; StoreStore; Store2
   → 保证Store1的数据在Store2之前刷新到内存

3. LoadStore Barrier（读-写屏障）
   Load1; LoadStore; Store2
   → 保证Load1在Store2之前完成

4. StoreLoad Barrier（写-读屏障，最重的）
   Store1; StoreLoad; Load2
   → 保证Store1刷新到内存后，Load2才能读取
```

**x86架构的内存模型（TSO - Total Store Order）**：

```
x86的保证：
- LoadLoad：✅ 自动保证
- LoadStore：✅ 自动保证
- StoreStore：✅ 自动保证
- StoreLoad：❌ 需要显式屏障（MFENCE指令）

x86只需关心Store-Load重排序！
```

**Go中的内存屏障（隐式）**：

```go
// sync/atomic包的操作自动插入内存屏障

atomic.StoreInt64(&x, 1)  // 写操作 + StoreLoad屏障
// 保证x=1刷新到内存后，其他线程才能读取

atomic.LoadInt64(&x)  // LoadLoad屏障 + 读操作
// 保证读取最新的值

// 手动内存屏障（Go汇编）
sync/atomic中的实现：
TEXT ·StoreInt64(SB),NOSPLIT,$0
    MOVQ    val+8(FP), AX
    XCHGQ   AX, 0(DI)  // XCHG自带LOCK前缀（全屏障）
    RET
```

##### volatile原理（Java）⭐⭐⭐

**什么是volatile？**

Java的`volatile`关键字保证：
1. **可见性**：一个线程修改volatile变量，其他线程立即可见
2. **有序性**：禁止指令重排序

**为什么需要volatile？**

```java
// 问题：线程2可能永远看不到flag的变化（CPU缓存）
class Example {
    private boolean flag = false;  // 非volatile
    
    // 线程1
    public void writer() {
        flag = true;  // 写入可能只在CPU1缓存
    }
    
    // 线程2
    public void reader() {
        while (!flag) {  // 读取可能从CPU2缓存
            // 无限循环！
        }
    }
}
```

**volatile的底层实现（JVM）**：

```java
class Example {
    private volatile boolean flag = false;  // 添加volatile
}

// 编译后的字节码插入内存屏障：
putfield flag
StoreLoad Barrier  // 插入写屏障

getfield flag
LoadLoad Barrier   // 插入读屏障
```

**volatile的汇编实现（x86）**：

```asm
; 写volatile变量（Java）
mov DWORD PTR [flag], 1
lock addl $0, (%rsp)  ; 内存屏障（MFENCE效果）

; LOCK指令的副作用：
; 1. 刷新写缓冲区（Store Buffer）
; 2. 失效其他CPU的缓存行（MESI协议）
; 3. 保证可见性
```

**volatile vs synchronized**：

| 特性 | volatile | synchronized |
|------|----------|--------------|
| **原子性** | ❌ 只保证单次读/写原子 | ✅ 保证代码块原子 |
| **可见性** | ✅ | ✅ |
| **有序性** | ✅ 禁止重排序 | ✅ |
| **性能** | **高（无锁）** | 低（加锁） |
| **适用场景** | 标志位、状态 | 复合操作 |

**volatile的陷阱**：

```java
class Counter {
    private volatile int count = 0;
    
    public void increment() {
        count++;  // 非原子！等价于：
                  // 1. temp = count (read)
                  // 2. temp = temp + 1 (modify)
                  // 3. count = temp (write)
                  // 线程可能在步骤1-3之间切换（竞态）
    }
}

// 解决方案：
// 1. synchronized
public synchronized void increment() {
    count++;
}

// 2. AtomicInteger
private AtomicInteger count = new AtomicInteger(0);
public void increment() {
    count.incrementAndGet();  // CAS循环，无锁原子
}
```

##### 原子操作 vs 锁（性能对比）⭐⭐⭐

**原子操作（Atomic）**：
```go
// 无锁，CAS循环
var counter int64
atomic.AddInt64(&counter, 1)

// 底层实现（伪代码）：
for {
    old := load(&counter)
    new := old + 1
    if CAS(&counter, old, new) {
        break
    }
    // 自旋重试
}
```

**互斥锁（Mutex）**：
```go
var mu sync.Mutex
var counter int64

mu.Lock()
counter++
mu.Unlock()

// 竞争激烈时，可能休眠/唤醒（syscall）
```

**性能对比**（1000万次操作）：

| 场景 | 原子操作 | Mutex | 差异 |
|------|---------|-------|------|
| **无竞争** | 50ms | 100ms | **2x** |
| **低竞争（2线程）** | 100ms | 200ms | **2x** |
| **高竞争（16线程）** | 500ms | 800ms | **1.6x** |

**为什么原子操作更快？**
1. **无锁**：不需要系统调用（futex）
2. **缓存友好**：CAS操作只锁缓存行
3. **无上下文切换**：自旋不休眠

**什么时候用Mutex？**
1. 临界区较大（多条语句）
2. 需要条件变量（Cond）
3. 需要公平性保证

**什么时候用原子操作？**
1. 单个变量的简单操作（自增、交换）
2. 无竞争或低竞争场景
3. 性能敏感路径

## 7. 并发设计模式

### 7.1 Fan-Out / Fan-In（扇出/扇入）

**Fan-Out**：一个输入源分发到多个 goroutine 并行处理。
**Fan-In**：多个 goroutine 的输出汇聚到一个 channel。

```go
func fanOut(input <-chan int, workers int) []<-chan int {
    channels := make([]<-chan int, workers)
    for i := 0; i < workers; i++ {
        channels[i] = process(input) // 每个 worker 从同一个 input 读取
    }
    return channels
}

func fanIn(channels ...<-chan int) <-chan int {
    out := make(chan int)
    var wg sync.WaitGroup
    for _, ch := range channels {
        wg.Add(1)
        go func(c <-chan int) {
            defer wg.Done()
            for v := range c {
                out <- v
            }
        }(ch)
    }
    go func() {
        wg.Wait()
        close(out)
    }()
    return out
}

// 使用：input → fan-out 到 4 个 worker → fan-in 合并结果
results := fanIn(fanOut(input, 4)...)
```

---

### 7.2 Pipeline（管道模式）

多个处理阶段通过 channel 串联，每个阶段是一个 goroutine：

```go
// 阶段1：生成数字
func generate(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        for _, n := range nums {
            out <- n
        }
        close(out)
    }()
    return out
}

// 阶段2：平方
func square(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        for n := range in {
            out <- n * n
        }
        close(out)
    }()
    return out
}

// 阶段3：过滤偶数
func filterEven(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        for n := range in {
            if n%2 == 0 {
                out <- n
            }
        }
        close(out)
    }()
    return out
}

// 组装管道
pipeline := filterEven(square(generate(1, 2, 3, 4, 5)))
for v := range pipeline {
    fmt.Println(v) // 4, 16
}
```

### 7.3 Worker Pool（工作池模式）

固定数量的 worker goroutine 从共享队列获取任务：

```go
func workerPool(numWorkers int, jobs <-chan Job, results chan<- Result) {
    var wg sync.WaitGroup
    for i := 0; i < numWorkers; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            for job := range jobs {
                result := processJob(job) // 处理任务
                results <- result
            }
        }(i)
    }
    go func() {
        wg.Wait()
        close(results)
    }()
}
```

> **对比三种模式**：
> - **Pipeline**：适合串行处理流水线（每个阶段做不同操作）
> - **Fan-Out/Fan-In**：适合同一操作的并行化（多个 worker 做相同操作）
> - **Worker Pool**：适合任务队列场景（控制并发数量）

## 8. 常见并发问题

### 8.1 竞态条件（Race Condition）

**问题**：
```go
var counter int

for i := 0; i < 1000; i++ {
    go func() {
        counter++  // 非原子操作
    }()
}

time.Sleep(1 * time.Second)
fmt.Println(counter)  // 结果不确定（< 1000）
```

**解决**：
```go
// 方案1：Mutex
var mu sync.Mutex
mu.Lock()
counter++
mu.Unlock()

// 方案2：原子操作
import "sync/atomic"
var counter int64
atomic.AddInt64(&counter, 1)
```

### 8.2 死锁（Deadlock）

**问题**：
```go
var mu1, mu2 sync.Mutex

// 线程1
go func() {
    mu1.Lock()
    time.Sleep(100 * time.Millisecond)
    mu2.Lock()  // 等待mu2
    mu2.Unlock()
    mu1.Unlock()
}()

// 线程2
go func() {
    mu2.Lock()
    time.Sleep(100 * time.Millisecond)
    mu1.Lock()  // 等待mu1（死锁）
    mu1.Unlock()
    mu2.Unlock()
}()
```

**解决**：
- 锁的顺序一致（都先mu1再mu2）
- 超时机制（tryLock）
- 死锁检测工具

### 8.3 活锁（Livelock）

**问题**：两个线程互相礼让，都无法前进

**解决**：引入随机性

### 8.4 饥饿（Starvation）

**问题**：某些线程一直无法获取资源

**解决**：公平锁（FIFO）

## 9. 常见面试题

### Q1: 进程、线程、协程的区别？

**答案**：见1.2表格

**追问**：为什么协程更轻量？
- 用户态调度，无需系统调用
- 共享线程栈，初始内存小（2KB）
- 切换开销小（只需保存少量寄存器）

### Q2: Go的GMP模型？

**答案**：
- **G（Goroutine）**：协程
- **M（Machine）**：OS线程
- **P（Processor）**：逻辑处理器（GOMAXPROCS）

**调度**：
- G绑定到P的本地队列
- P绑定到M
- M执行P队列中的G
- Work Stealing：空闲P从其他P偷取G

### Q3: Python的GIL是什么？

**答案**：
- 全局解释器锁（Global Interpreter Lock）
- 同一时刻只有一个线程执行Python字节码
- 影响：多线程无法利用多核（CPU密集型）
- 解决：多进程、C扩展释放GIL

### Q4: 如何避免死锁？

**答案**：
1. **锁的顺序**：统一获取顺序
2. **超时机制**：tryLock + 超时
3. **避免嵌套锁**：尽量只持有一把锁
4. **死锁检测**：检测并打破循环等待

### Q5: synchronized和ReentrantLock的区别？

**答案**：
| 特性 | synchronized | ReentrantLock |
|------|--------------|---------------|
| **类型** | 关键字 | 类 |
| **灵活性** | 低 | 高（tryLock、超时） |
| **公平性** | 非公平 | 可选公平 |
| **条件变量** | 1个 | 多个Condition |
| **释放** | 自动 | 手动（finally） |

## 10. 实战案例

### 案例1：并发爬虫

```go
package main

import (
    "fmt"
    "sync"
    "time"
)

type Result struct {
    URL  string
    Data string
}

func crawl(url string) Result {
    time.Sleep(100 * time.Millisecond)  // 模拟网络请求
    return Result{URL: url, Data: "数据"}
}

func main() {
    urls := []string{"url1", "url2", "url3", "url4", "url5"}

    var wg sync.WaitGroup
    results := make(chan Result, len(urls))

        wg.Add(1)
        go func(u string) {
            defer wg.Done()
            results <- crawl(u)
        }(url)
    }
    
    // 关闭channel
    go func() {
        wg.Wait()
        close(results)
    }()
    
    // 收集结果
    for result := range results {
        fmt.Println(result)
    }
}
```

## 📝 面试题自查

### Q1：并发和并行有什么区别？单核 CPU 能实现并行吗？

**答**：**并发**是逻辑上同时处理多个任务（交替执行），**并行**是物理上同时执行多个任务（多核同时运行）。单核 CPU 不能并行，但可以并发——通过时间片轮转在多个任务间快速切换，给用户"同时"的感觉。Go 的 goroutine 在单核上是并发的（GMP 调度器切换），多核上既并发也并行。

---

### Q2：进程、线程、协程的区别和各自的适用场景？

**答**：**进程**：独立内存空间，OS 调度，开销大（MB 级），适合 CPU 密集型隔离（如 Nginx worker）。**线程**：共享进程内存，OS 调度，开销中等（KB 级），适合需要共享内存的并发场景。**协程**：共享线程栈，用户态调度，开销极小（2KB 起），适合高并发 IO 场景（Go goroutine、Python asyncio）。关键区别是调度方式：进程/线程是抢占式（OS 控制），协程是协作式（用户代码主动让出）。

---

### Q3：Go 的 Channel 底层是怎么实现的？发送和接收的完整流程？

**答**：Channel 底层是 `hchan` 结构体，包含环形缓冲区（`buf`）、发送/接收索引（`sendx`/`recvx`）、等待队列（`sendq`/`recvq`）和互斥锁（`lock`）。**发送流程**：加锁→如果有等待接收方则直接拷贝数据给它→否则缓冲区有空间则写入→否则加入 sendq 挂起。**接收流程**：加锁→如果有等待发送方则直接取数据→否则缓冲区有数据则读取→否则加入 recvq 挂起。关键优化：有等待方时数据直接在两个 goroutine 间拷贝，跳过缓冲区。

---

### Q4：无缓冲 Channel 和有缓冲 Channel 的区别？分别在什么场景下使用？

**答**：**无缓冲**（`make(chan T)`）：发送和接收必须同时就绪，提供**同步语义**——保证发送方数据被接收方消费后才继续。适用于 goroutine 间同步通信、信号传递。**有缓冲**（`make(chan T, N)`）：缓冲未满时发送不阻塞，提供**异步语义**——发送方和接收方解耦。适用于生产者-消费者模式、限流（缓冲大小=并发限制）。

---

### Q5：Go 的 Mutex 有哪几种模式？什么时候会进入饥饿模式？

**答**：Go Mutex 有**正常模式**和**饥饿模式**。正常模式下，被唤醒的 goroutine 和新到来的 goroutine 竞争锁，新 goroutine 更有优势（已在 CPU 上运行）。当等待方超过 **1ms** 未获得锁，切换为饥饿模式——锁直接交给队列头部 goroutine（FIFO 公平），新 goroutine 不自旋直接入队。当最后一个等待者获得锁或等待时间 < 1ms 时退出饥饿模式。此外还有自旋优化（最多 4 次 CAS 自旋）。

### Q6：什么是 CAS？它的底层是如何实现原子性的？ABA 问题是什么？

**答**：CAS（Compare-And-Swap）是无锁原子操作：如果当前值等于期望值则替换为新值。底层通过 CPU 的 **LOCK CMPXCHG** 指令实现，LOCK 前缀锁定缓存行（MESI 协议），保证操作的原子性。**ABA 问题**：值从 A 变为 B 再变回 A，CAS 认为没有变化但实际中间状态已改变。解决方案：附加版本号（每次修改 version++），CAS 同时比较值和版本号。

### Q7：什么是内存屏障？为什么 CPU 会重排序指令？

**答**：现代 CPU 为了提升流水线效率会对指令**重排序**（Store Buffer、乱序执行），单线程下结果正确但多线程下可能导致可见性问题。**内存屏障**是 CPU 指令（如 x86 的 MFENCE），强制刷新 Store Buffer 并禁止特定类型的重排序。4 种类型：LoadLoad、StoreStore、LoadStore、StoreLoad（最重）。x86（TSO 模型）自动保证前三种，只需关心 StoreLoad 重排序。Go 的 `sync/atomic` 操作隐式包含内存屏障。

---

### Q8：Java 的 volatile 底层是怎么实现的？volatile 能保证原子性吗？

**答**：volatile 通过 JVM 在字节码中插入**内存屏障**实现：写 volatile 后插入 StoreLoad 屏障（刷新 Store Buffer + 失效其他 CPU 缓存行），读 volatile 前插入 LoadLoad 屏障（从主内存加载最新值）。x86 上编译为 `LOCK addl $0, (%rsp)` 指令。volatile **不能保证原子性**——`count++` 是读-改-写三步操作，volatile 只保证每一步的可见性，但线程可能在步骤之间被切换。复合操作需要 synchronized 或 AtomicInteger。

### Q9：原子操作和互斥锁在性能上有什么区别？各自适用于什么场景？

**答**：**原子操作**：无锁（CAS 循环），不需要系统调用，只锁缓存行，无上下文切换，适合单变量的简单操作（计数器、标志位）和低竞争场景。**互斥锁**：获取失败可能触发线程休眠/唤醒（系统调用），有上下文切换开销，但适合大临界区（多条语句）、需要条件变量、需要公平性的场景。一般原子操作比锁快 1.5~2 倍。高竞争下 Mutex 的饥饿模式保证公平性优于 CAS 无限自旋。

### Q10：什么是死锁？死锁的四个必要条件？如何预防和检测？

**答**：死锁是两个或多个线程互相等待对方持有的资源，导致所有线程都无法继续。**四个必要条件**：① 互斥（资源排他使用）② 持有并等待（持有一个资源同时等待另一个）③ 不可剥夺（已持有的资源不能强制释放）④ 循环等待（A 等 B，B 等 A）。**预防**：破坏任一条件，最常用的是统一锁获取顺序（破坏循环等待）、使用 tryLock+超时。**检测**：Go 的 `go vet -race`、Java 的 `jstack` 查看线程状态、数据库死锁检测日志。

---

### Q11：Fan-Out/Fan-In、Pipeline、Worker Pool 三种并发模式的区别和适用场景？

**答**：**Pipeline（管道）**：多个阶段串联，每个阶段做不同处理（如 generate→filter→transform），适合数据流处理。**Fan-Out/Fan-In（扇出扇入）**：一个输入分发到多个 worker 并行处理相同操作，结果汇聚，适合并行化计算密集操作。**Worker Pool（工作池）**：固定数量 worker 从共享队列取任务，控制并发度，适合任务队列场景（限制 goroutine 数量防止资源耗尽）。

### Q12：Go 中如何优雅地退出多个 goroutine？Context 的取消机制是怎么实现的？

**答**：使用 `context.WithCancel` 创建可取消的 context，调用 `cancel()` 后所有监听 `ctx.Done()` 的 goroutine 收到关闭信号并退出。底层实现：context 内部维护一个 `done` channel，`cancel()` 调用 `close(done)`，所有 `<-ctx.Done()` 的 select case 同时返回。`WithTimeout`/`WithDeadline` 额外使用 `time.AfterFunc` 定时调用 cancel。context 是树形结构，父 context 取消时递归取消所有子 context。

---

### Q13：读写锁（RWMutex）的适用场景？如果写操作很频繁还适合用读写锁吗？

**答**：读写锁适合**读多写少**场景——多个读操作可以并行（共享读锁），写操作排他。如果写操作频繁，读写锁可能比普通 Mutex 更慢，因为：① 写锁需要等所有读锁释放（写饥饿）；② RWMutex 内部维护更复杂的状态（读计数器、写等待标志），锁操作开销更大。写频繁时建议用普通 Mutex 或分片加锁（ConcurrentHashMap 思路）。

---

### Q14：Go 的 select 语句在多个 case 同时就绪时会怎样？如何实现超时控制？

**答**：多个 case 同时就绪时，Go 运行时会**随机选择**一个执行（不是按顺序），保证公平性。超时控制：`select { case v := <-ch: ... case <-time.After(timeout): ... }`，`time.After` 返回一个 channel，在指定时间后发送信号。也可以用 `context.WithTimeout` 创建带截止时间的 context，在 select 中监听 `ctx.Done()`。

### Q15：如何检测 Go 程序中的数据竞争？`go vet -race` 的原理是什么？

**答**：使用 `go run -race` 或 `go test -race` 开启竞态检测器。原理是基于 **ThreadSanitizer** 算法：编译时在每个内存访问点插入检测代码，运行时维护每个内存地址的**向量时钟**（Vector Clock），记录最近的读/写线程和时间。如果检测到两个访问同一地址的操作来自不同 goroutine、至少一个是写、且没有 happens-before 关系，就报告数据竞争。开启 race detector 会使程序慢 2~10 倍，内存增加 5~10 倍，仅用于测试环境。
