## 1. 并发基础概念

### 1.1 并发 vs 并行

| 概念 | 说明 | 比喻 |
|------|------|------|
| **并发（Concurrency）** | 多个任务交替执行（逻辑上同时） | 一个人同时做多件事（切换） |
| **并行（Parallelism）** | 多个任务真正同时执行（物理上同时） | 多个人同时做多件事 |

**示例**：
```
并发（单核CPU）：
任务A: ---|---|---|---
任务B:   ---|---|---|---
时间: ═══════════════════>

并行（多核CPU）：
任务A: ══════════════════>
任务B: ══════════════════>
时间: ═══════════════════>
```

### 1.2 进程 vs 线程 vs 协程

| 概念 | 资源 | 调度 | 开销 | 通信 |
|------|------|------|------|------|
| **进程（Process）** | 独立内存空间 | 操作系统调度 | 大（MB级） | IPC（管道、消息队列） |
| **线程（Thread）** | 共享进程内存 | 操作系统调度 | 中（KB级） | 共享内存 |
| **协程（Goroutine/Coroutine）** | 共享线程栈 | 用户态调度 | 小（2KB起） | Channel/Queue |

**典型应用**：
- **进程**：Nginx多进程、Chrome多进程
- **线程**：Java线程池、Python线程
- **协程**：Go goroutine、Python asyncio

## 2. Go并发模型

### 2.1 Goroutine

**创建goroutine**：
```go
// 普通函数
go func() {
    fmt.Println("Hello from goroutine")
}()

// 匿名函数
go func(name string) {
    fmt.Println("Hello", name)
}("张三")

// 阻塞主goroutine
time.Sleep(1 * time.Second)
```

**goroutine特点**：
- 轻量级（初始栈2KB，可动态扩展）
- M:N调度（M个goroutine映射到N个OS线程）
- GMP模型：G（goroutine）、M（Machine线程）、P（Processor逻辑处理器）

### 2.2 Channel

**无缓冲Channel**（同步）：
```go
ch := make(chan int)

// 发送（阻塞直到接收）
go func() {
    ch <- 42
}()

// 接收（阻塞直到发送）
value := <-ch
fmt.Println(value)  // 42
```

**有缓冲Channel**（异步）：
```go
ch := make(chan int, 3)  // 缓冲大小3

ch <- 1
ch <- 2
ch <- 3
// ch <- 4  // 阻塞，缓冲已满

fmt.Println(<-ch)  // 1
fmt.Println(<-ch)  // 2
```

**关闭Channel**：
```go
ch := make(chan int, 2)
ch <- 1
ch <- 2
close(ch)  // 关闭

// 接收直到关闭
for v := range ch {
    fmt.Println(v)  // 1, 2
}

// 检查是否关闭
v, ok := <-ch
if !ok {
    fmt.Println("channel已关闭")
}
```

**单向Channel**：
```go
// 只能发送
func send(ch chan<- int) {
    ch <- 42
}

// 只能接收
func receive(ch <-chan int) {
    value := <-ch
}
```

### 2.3 Select多路复用

```go
ch1 := make(chan int)
ch2 := make(chan string)

select {
case v := <-ch1:
    fmt.Println("收到int:", v)
case v := <-ch2:
    fmt.Println("收到string:", v)

```

**应用场景**：
```go
// 超时控制
func doWithTimeout(fn func() error, timeout time.Duration) error {
    done := make(chan error)
    
    go func() {
        done <- fn()
    }()
    
    select {
    case err := <-done:
        return err
    case <-time.After(timeout):
        return errors.New("超时")
    }
}
```

### 2.5 Context

**传递取消信号**：
```go
import "context"

func worker(ctx context.Context, id int) {
    for {
        select {
        case <-ctx.Done():
            fmt.Println("worker", id, "退出")

    cancel()  // 取消所有worker
    time.Sleep(1 * time.Second)
}
```

## 3. Go Channel底层实现

### 3.1 hchan 数据结构

Channel 在 Go 运行时的底层实现是 `hchan` 结构体：

```go
// runtime/chan.go（简化）
type hchan struct {
    qcount   uint           // 当前缓冲区中的元素数量
    dataqsiz uint           // 缓冲区容量（make(chan T, N) 中的 N）
    buf      unsafe.Pointer // 环形缓冲区指针
    elemsize uint16         // 每个元素的大小
    closed   uint32         // 是否关闭（0=open, 1=closed）
    sendx    uint           // 发送索引（环形缓冲区写位置）
    recvx    uint           // 接收索引（环形缓冲区读位置）
    recvq    waitq          // 等待接收的 goroutine 队列
    sendq    waitq          // 等待发送的 goroutine 队列
    lock     mutex          // 保护 hchan 的互斥锁
}

type waitq struct {
    first *sudog  // 等待队列头
    last  *sudog  // 等待队列尾
}

type sudog struct {
    g    *g          // 等待的 goroutine
    elem unsafe.Pointer // 数据指针
    next *sudog
    prev *sudog
}
```

```
hchan 结构示意图（有缓冲 channel）：

┌──────────────────────────────────────┐
│ hchan                                 │
├──────────────────────────────────────┤
│ qcount=2  dataqsiz=4                 │
│                                       │
│ buf → [elem0][elem1][    ][    ]     │  ← 环形缓冲区
│        recvx=0       sendx=2          │
│                                       │
│ sendq → (空)                          │  ← 等待发送的goroutine
│ recvq → (空)                          │  ← 等待接收的goroutine
│                                       │
│ lock (mutex)                          │
└──────────────────────────────────────┘
```

---

### 3.2 发送与接收流程

**发送流程（`ch <- value`）**：

```
1. 加锁 hchan.lock

2. 检查 recvq 是否有等待接收的 goroutine
   ├── 有 → 直接将数据拷贝给该 goroutine（不经过缓冲区），唤醒它
   └── 无 → 继续

3. 检查缓冲区是否有空间（qcount < dataqsiz）
   ├── 有 → 将数据拷贝到 buf[sendx]，sendx++，qcount++
   └── 无 → 继续

4. 缓冲区已满（或无缓冲 channel）
   - 创建 sudog，将当前 goroutine 加入 sendq
   - 调用 gopark() 挂起当前 goroutine
   - 等待被接收方唤醒

5. 解锁
```

**接收流程（`value := <-ch`）**：

```
1. 加锁 hchan.lock

2. 检查 sendq 是否有等待发送的 goroutine
   ├── 有（无缓冲）→ 直接从发送方拷贝数据，唤醒它
   ├── 有（有缓冲且满）→ 从 buf[recvx] 取数据，将发送方数据写入 buf[sendx]，唤醒它
   └── 无 → 继续

3. 检查缓冲区是否有数据（qcount > 0）
   ├── 有 → 从 buf[recvx] 取数据，recvx++，qcount--
   └── 无 → 继续

4. 缓冲区为空
   - 创建 sudog，将当前 goroutine 加入 recvq
   - 调用 gopark() 挂起当前 goroutine
   - 等待被发送方唤醒

5. 解锁
```

> **关键优化**：当有等待方时，数据直接在两个 goroutine 之间拷贝（bypass 缓冲区），减少一次内存拷贝。

---

### 3.3 无缓冲 vs 有缓冲

| 特性 | 无缓冲 `make(chan T)` | 有缓冲 `make(chan T, N)` |
|------|----------------------|-------------------------|
| 缓冲区 | 无（`dataqsiz=0`） | 环形缓冲区大小 N |
| 发送行为 | 阻塞直到有接收方 | 缓冲未满则立即返回 |
| 接收行为 | 阻塞直到有发送方 | 缓冲非空则立即返回 |
| 同步语义 | **同步**（发送-接收配对） | **异步**（通过缓冲区解耦） |
| 适用场景 | goroutine 间同步通信 | 生产者-消费者缓冲 |

```python
import time

def worker(name):

threads = []
for i in range(5):
    t = threading.Thread(target=worker, args=(i,))
    threads.append(t)
    t.start()

# 等待所有线程完成
```

```python
import threading

counter = 0
lock = threading.Lock()

def increment():
    global counter
    for _ in range(100000):
        with lock:  # 加锁
            counter += 1

threads = [threading.Thread(target=increment) for _ in range(10)]
for t in threads:
    t.start()
for t in threads:
    t.join()

print(counter)  # 1000000（正确）
```

#### 6.1.1 Mutex底层实现原理（深度剖析）⭐⭐⭐

**为什么需要了解Mutex底层？**

---

##### Go sync.Mutex的演进（从简单到复杂）

**最简单的Mutex实现（伪代码）**：
```go
type SimpleMutex struct {
    locked int32  // 0=未锁定, 1=已锁定
}

func (m *SimpleMutex) Lock() {
    for {
        // CAS操作：如果locked=0，设为1并返回true
        if atomic.CompareAndSwapInt32(&m.locked, 0, 1) {
            return  // 成功获取锁
        }
        // 否则，继续循环（自旋）
    }
}

func (m *SimpleMutex) Unlock() {
    atomic.StoreInt32(&m.locked, 0)
}
```

**问题**：
- ❌ CPU空转（busy-wait），浪费资源
- ❌ 不公平，可能有线程一直抢不到锁（饥饿）

---

**Go sync.Mutex的实际实现（4种模式）**

```go
type Mutex struct {
    state int32   // 状态字（锁定位、唤醒位、饥饿模式、等待者计数）
    sema  uint32  // 信号量（用于休眠/唤醒goroutine）
}

// state的位布局（32位）：
// [0]:     锁定位（1=已锁定）
// [1]:     唤醒位（1=有goroutine被唤醒）
// [2]:     饥饿模式位（1=饥饿模式）
// [3-31]:  等待者计数（最多2^29-1个等待者）
```

**Mutex的4种模式**：

```
1. 正常模式（Normal Mode）
   - goroutine先自旋（spin）几次，尝试CAS获取锁
   - 失败后进入等待队列，休眠（futex）
   - 被唤醒的goroutine与新到来的goroutine竞争
   - 新goroutine更容易获得锁（在CPU上运行，无需唤醒）

2. 饥饿模式（Starvation Mode）
   - 当goroutine等待超过1ms未获得锁，切换到饥饿模式
   - 锁直接交给队列头部的goroutine（公平）
   - 新到来的goroutine不自旋，直接入队
   - 最后一个goroutine或等待时间<1ms，退出饥饿模式

3. 自旋优化（Spinning）
   - 在单核或等待队列过长时，不自旋
   - 最多自旋4次（runtime_canSpin检查）
   - 自旋时调用PAUSE指令（减少CPU功耗）

4. 信号量休眠（Semaphore Sleep）
   - 自旋失败后，通过runtime_SemacquireMutex休眠
   - Unlock时通过runtime_Semrelease唤醒
```

**示例：Lock的执行流程**

```
场景：goroutine A持有锁，goroutine B尝试获取锁

1. B调用Lock()
2. CAS尝试获取锁（失败，state[0]=1）
3. 自旋4次（每次PAUSE + CAS）
4. 自旋失败，state[3-31] += 1（等待者+1）
5. 调用runtime_SemacquireMutex(sema)，B休眠
6. A调用Unlock()
7. state[0] = 0（解锁）
8. 调用runtime_Semrelease(sema)，唤醒B
9. B被唤醒，与新到来的C竞争
10. 如果B再次失败 && 等待>1ms → 切换到饥饿模式
```

##### CAS（Compare-And-Swap）底层原理⭐⭐⭐

**什么是CAS？**

CAS是一种**无锁（Lock-Free）的原子操作**：
```
CAS(addr, old, new):
    if *addr == old:
        *addr = new
        return true
    else:
        return false
```

**CPU指令：CMPXCHG（x86架构）**

```asm
; CAS的汇编实现（x86-64）
; RAX = old, RBX = new, [addr] = 当前值

LOCK CMPXCHG [addr], RBX

; LOCK前缀：保证原子性（锁总线或缓存行）
; CMPXCHG：比较并交换
;   - 如果 [addr] == RAX，则 [addr] = RBX，ZF=1
;   - 否则，RAX = [addr]，ZF=0
```

**为什么CAS需要LOCK前缀？**

```
多核CPU的缓存一致性问题：

CPU1 缓存:  [addr] = 0
CPU2 缓存:  [addr] = 0

时刻T1: CPU1读取[addr]=0，准备写入1
时刻T2: CPU2读取[addr]=0，准备写入1
时刻T3: CPU1写入1
时刻T4: CPU2写入1
→ 两次CAS都成功（错误！）

LOCK前缀的作用：
- 锁定总线（早期CPU）：禁止其他CPU访问内存
- 锁定缓存行（现代CPU，MESI协议）：
  - 独占该缓存行（Exclusive）
  - 其他CPU必须等待或失效
  - 保证原子性
```

**Go中的CAS示例**：

```go
import "sync/atomic"

var counter int64 = 0

// CAS实现无锁自增
func increment() {
    for {
        old := atomic.LoadInt64(&counter)
        new := old + 1
        if atomic.CompareAndSwapInt64(&counter, old, new) {
            return  // 成功
        }
        // 失败，说明其他goroutine修改了counter，重试
    }
}

// 等价于
atomic.AddInt64(&counter, 1)  // 底层也是CAS循环
```

**CAS的ABA问题**：

```
场景：栈的pop操作

初始状态：stack = A → B → C

线程1：执行pop()
  - 读取head = A
  - 准备CAS(head, A, B)
  - 被挂起...

线程2：
  - pop() → A出栈，head = B
  - pop() → B出栈，head = C
  - push(A) → A重新入栈，head = A

线程1恢复：
  - CAS(head, A, B) → 成功！（错误！）
  - 但此时A → C，不是A → B

问题：CAS只检查值是否相等，不检查是否是"同一个"对象

解决方案：
1. 版本号（Version）：CAS(head, <A, version=1>, <B, version=2>)
2. 引用计数
3. Hazard Pointer
```

##### 内存屏障（Memory Barrier）⭐⭐⭐

**为什么需要内存屏障？**

现代CPU为了性能，会对指令**重排序（Reordering）**：

```go
// 代码顺序：
a = 1        // 写操作1
b = 2        // 写操作2

// CPU可能执行顺序：
b = 2        // 先执行写操作2
a = 1        // 后执行写操作1

// 单线程下，重排序不影响结果
// 多线程下，可能导致错误！
```

**示例：双重检查锁（DCL）的错误实现**

```go
var instance *Singleton
var mu sync.Mutex

func GetInstance() *Singleton {
    if instance == nil {  // 检查1（无锁）
        mu.Lock()
        if instance == nil {  // 检查2（有锁）
            instance = &Singleton{}  // 问题：可能重排序！
        }
        mu.Unlock()
    }
    return instance
}

// 问题：instance = &Singleton{} 可能分解为：
// 1. 分配内存 → ptr
// 2. instance = ptr （先赋值！）
// 3. 初始化ptr指向的对象（后初始化！）
//
// 另一个线程读取instance != nil，但对象未初始化（野指针）！
```

**内存屏障的类型**：

```
1. LoadLoad Barrier（读-读屏障）
   Load1; LoadLoad; Load2
   → 保证Load1的数据在Load2之前加载

2. StoreStore Barrier（写-写屏障）
   Store1; StoreStore; Store2
   → 保证Store1的数据在Store2之前刷新到内存

3. LoadStore Barrier（读-写屏障）
   Load1; LoadStore; Store2
   → 保证Load1在Store2之前完成

4. StoreLoad Barrier（写-读屏障，最重的）
   Store1; StoreLoad; Load2
   → 保证Store1刷新到内存后，Load2才能读取
```

**x86架构的内存模型（TSO - Total Store Order）**：

```
x86的保证：
- LoadLoad：✅ 自动保证
- LoadStore：✅ 自动保证
- StoreStore：✅ 自动保证
- StoreLoad：❌ 需要显式屏障（MFENCE指令）

x86只需关心Store-Load重排序！
```

**Go中的内存屏障（隐式）**：

```go
// sync/atomic包的操作自动插入内存屏障

atomic.StoreInt64(&x, 1)  // 写操作 + StoreLoad屏障
// 保证x=1刷新到内存后，其他线程才能读取

atomic.LoadInt64(&x)  // LoadLoad屏障 + 读操作
// 保证读取最新的值

// 手动内存屏障（Go汇编）
sync/atomic中的实现：
TEXT ·StoreInt64(SB),NOSPLIT,$0
    MOVQ    val+8(FP), AX
    XCHGQ   AX, 0(DI)  // XCHG自带LOCK前缀（全屏障）
    RET
```

##### volatile原理（Java）⭐⭐⭐

**什么是volatile？**

Java的`volatile`关键字保证：
1. **可见性**：一个线程修改volatile变量，其他线程立即可见
2. **有序性**：禁止指令重排序

**为什么需要volatile？**

```java
// 问题：线程2可能永远看不到flag的变化（CPU缓存）
class Example {
    private boolean flag = false;  // 非volatile
    
    // 线程1
    public void writer() {
        flag = true;  // 写入可能只在CPU1缓存
    }
    
    // 线程2
    public void reader() {
        while (!flag) {  // 读取可能从CPU2缓存
            // 无限循环！
        }
    }
}
```

**volatile的底层实现（JVM）**：

```java
class Example {
    private volatile boolean flag = false;  // 添加volatile
}

// 编译后的字节码插入内存屏障：
putfield flag
StoreLoad Barrier  // 插入写屏障

getfield flag
LoadLoad Barrier   // 插入读屏障
```

**volatile的汇编实现（x86）**：

```asm
; 写volatile变量（Java）
mov DWORD PTR [flag], 1
lock addl $0, (%rsp)  ; 内存屏障（MFENCE效果）

; LOCK指令的副作用：
; 1. 刷新写缓冲区（Store Buffer）
; 2. 失效其他CPU的缓存行（MESI协议）
; 3. 保证可见性
```

**volatile vs synchronized**：

| 特性 | volatile | synchronized |
|------|----------|--------------|
| **原子性** | ❌ 只保证单次读/写原子 | ✅ 保证代码块原子 |
| **可见性** | ✅ | ✅ |
| **有序性** | ✅ 禁止重排序 | ✅ |
| **性能** | **高（无锁）** | 低（加锁） |
| **适用场景** | 标志位、状态 | 复合操作 |

**volatile的陷阱**：

```java
class Counter {
    private volatile int count = 0;
    
    public void increment() {
        count++;  // 非原子！等价于：
                  // 1. temp = count (read)
                  // 2. temp = temp + 1 (modify)
                  // 3. count = temp (write)
                  // 线程可能在步骤1-3之间切换（竞态）
    }
}

// 解决方案：
// 1. synchronized
public synchronized void increment() {
    count++;
}

// 2. AtomicInteger
private AtomicInteger count = new AtomicInteger(0);
public void increment() {
    count.incrementAndGet();  // CAS循环，无锁原子
}
```

##### 原子操作 vs 锁（性能对比）⭐⭐⭐

**原子操作（Atomic）**：
```go
// 无锁，CAS循环
var counter int64
atomic.AddInt64(&counter, 1)

// 底层实现（伪代码）：
for {
    old := load(&counter)
    new := old + 1
    if CAS(&counter, old, new) {
        break
    }
    // 自旋重试
}
```

**互斥锁（Mutex）**：
```go
var mu sync.Mutex
var counter int64

mu.Lock()
counter++
mu.Unlock()

// 竞争激烈时，可能休眠/唤醒（syscall）
```

**性能对比**（1000万次操作）：

| 场景 | 原子操作 | Mutex | 差异 |
|------|---------|-------|------|
| **无竞争** | 50ms | 100ms | **2x** |
| **低竞争（2线程）** | 100ms | 200ms | **2x** |
| **高竞争（16线程）** | 500ms | 800ms | **1.6x** |

**为什么原子操作更快？**
1. **无锁**：不需要系统调用（futex）
2. **缓存友好**：CAS操作只锁缓存行
3. **无上下文切换**：自旋不休眠

**什么时候用Mutex？**
1. 临界区较大（多条语句）
2. 需要条件变量（Cond）
3. 需要公平性保证

**什么时候用原子操作？**
1. 单个变量的简单操作（自增、交换）
2. 无竞争或低竞争场景
3. 性能敏感路径

## 7. 并发设计模式

### 7.1 Fan-Out / Fan-In（扇出/扇入）

**Fan-Out**：一个输入源分发到多个 goroutine 并行处理。
**Fan-In**：多个 goroutine 的输出汇聚到一个 channel。

```go
func fanOut(input <-chan int, workers int) []<-chan int {
    channels := make([]<-chan int, workers)
    for i := 0; i < workers; i++ {
        channels[i] = process(input) // 每个 worker 从同一个 input 读取
    }
    return channels
}

func fanIn(channels ...<-chan int) <-chan int {
    out := make(chan int)
    var wg sync.WaitGroup
    for _, ch := range channels {
        wg.Add(1)
        go func(c <-chan int) {
            defer wg.Done()
            for v := range c {
                out <- v
            }
        }(ch)
    }
    go func() {
        wg.Wait()
        close(out)
    }()
    return out
}

// 使用：input → fan-out 到 4 个 worker → fan-in 合并结果
results := fanIn(fanOut(input, 4)...)
```

---

### 7.2 Pipeline（管道模式）

多个处理阶段通过 channel 串联，每个阶段是一个 goroutine：

```go
// 阶段1：生成数字
func generate(nums ...int) <-chan int {
    out := make(chan int)
    go func() {
        for _, n := range nums {
            out <- n
        }
        close(out)
    }()
    return out
}

// 阶段2：平方
func square(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        for n := range in {
            out <- n * n
        }
        close(out)
    }()
    return out
}

// 阶段3：过滤偶数
func filterEven(in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        for n := range in {
            if n%2 == 0 {
                out <- n
            }
        }
        close(out)
    }()
    return out
}

// 组装管道
pipeline := filterEven(square(generate(1, 2, 3, 4, 5)))
for v := range pipeline {
    fmt.Println(v) // 4, 16
}
```

### 7.3 Worker Pool（工作池模式）

固定数量的 worker goroutine 从共享队列获取任务：

```go
func workerPool(numWorkers int, jobs <-chan Job, results chan<- Result) {
    var wg sync.WaitGroup
    for i := 0; i < numWorkers; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            for job := range jobs {
                result := processJob(job) // 处理任务
                results <- result
            }
        }(i)
    }
    go func() {
        wg.Wait()
        close(results)
    }()
}
```

> **对比三种模式**：
> - **Pipeline**：适合串行处理流水线（每个阶段做不同操作）
> - **Fan-Out/Fan-In**：适合同一操作的并行化（多个 worker 做相同操作）
> - **Worker Pool**：适合任务队列场景（控制并发数量）

## 8. 常见并发问题

### 8.1 竞态条件（Race Condition）

**问题**：
```go
var counter int

for i := 0; i < 1000; i++ {
    go func() {
        counter++  // 非原子操作
    }()
}

time.Sleep(1 * time.Second)
fmt.Println(counter)  // 结果不确定（< 1000）
```

**解决**：
```go
// 方案1：Mutex
var mu sync.Mutex
mu.Lock()
counter++
mu.Unlock()

// 方案2：原子操作
import "sync/atomic"
var counter int64
atomic.AddInt64(&counter, 1)
```

### 8.2 死锁（Deadlock）

**问题**：
```go
var mu1, mu2 sync.Mutex

// 线程1
go func() {
    mu1.Lock()
    time.Sleep(100 * time.Millisecond)
    mu2.Lock()  // 等待mu2
    mu2.Unlock()
    mu1.Unlock()
}()

// 线程2
go func() {
    mu2.Lock()
    time.Sleep(100 * time.Millisecond)
    mu1.Lock()  // 等待mu1（死锁）
    mu1.Unlock()
    mu2.Unlock()
}()
```

**解决**：
- 锁的顺序一致（都先mu1再mu2）
- 超时机制（tryLock）
- 死锁检测工具

### 8.3 活锁（Livelock）

**问题**：两个线程互相礼让，都无法前进

**解决**：引入随机性

### 8.4 饥饿（Starvation）

**问题**：某些线程一直无法获取资源

**解决**：公平锁（FIFO）

## 10. Go sync.Map 实现原理

### 10.1 设计目标

`sync.Map` 是 Go 标准库为**读多写少**场景专门优化的并发安全 map。设计目标：
- 高频读、低频写时，避免每次读都加锁
- 减少锁竞争，让热路径（hot path）尽量无锁完成
- 适用场景：缓存、配置项、路由表等"写入一次、读取千次"的场景

### 10.2 双 map 结构

```go
// sync/map.go（简化）
type Map struct {
    mu     sync.Mutex          // 保护 dirty
    read   atomic.Pointer[readOnly] // 只读 map，无锁访问
    dirty  map[any]*entry      // 读写 map，需要加锁
    misses int                  // read 未命中计数
}

type readOnly struct {
    m       map[any]*entry
    amended bool  // dirty 中是否有 read 中不存在的 key
}

type entry struct {
    p atomic.Pointer[any] // 指向实际值；nil=已删除，expunged=从dirty中移除
}
```

核心思路：**read map 承接绝大部分读请求（原子操作，无锁），dirty map 承接写操作（加锁）**。

```
┌────────────────────────────────────────────────┐
│  sync.Map                                       │
│                                                 │
│  read (atomic, 无锁读)                          │
│  ┌───────────────────────┐                     │
│  │ key1 → *entry(val1)   │                     │
│  │ key2 → *entry(val2)   │                     │
│  │ amended = true         │                     │
│  └───────────────────────┘                     │
│                                                 │
│  dirty (mu保护)                                 │
│  ┌───────────────────────┐                     │
│  │ key1 → *entry(val1)   │ ← 与read共享entry   │
│  │ key2 → *entry(val2)   │                     │
│  │ key3 → *entry(val3)   │ ← read中没有        │
│  └───────────────────────┘                     │
│                                                 │
│  misses = 2                                     │
└────────────────────────────────────────────────┘
```

### 10.3 操作流程

**Load（读取）**：
```
1. 从 read map 中查找（无锁，atomic.Load）
   ├── 找到 → 返回 entry.p 的值（快速路径）
   └── 未找到 && amended=false → 返回 nil（key不存在）

2. amended=true → 加锁，从 dirty 中查找
   ├── 找到 → 返回值
   └── 未找到 → 返回 nil
   misses++
   如果 misses >= len(dirty) → promote（dirty提升为read）
```

**Store（写入）**：
```
1. 如果 key 在 read map 中存在
   → 直接 CAS 更新 entry.p（无锁快速路径）

2. 加锁
   ├── key 在 read 中存在（可能刚被删除标记为expunged）
   │   → unexpunge，更新 entry.p，加入 dirty
   ├── key 在 dirty 中存在
   │   → 直接更新 entry.p
   └── key 不存在
       → 如果 dirty==nil，从 read 复制未删除的 entry 创建 dirty
       → 在 dirty 中插入新 entry，amended=true
```

**Delete（删除）**：
```
1. 如果 key 在 read map 中存在
   → CAS 将 entry.p 置为 nil（软删除，无锁）

2. 否则加锁，从 dirty 中 delete
```

### 10.4 amended 标志与 promote 机制

- `amended=true`：表示 dirty 中有 read 中不存在的新 key，Load 未命中时需要查 dirty
- `misses` 计数器：每次 read 未命中且需要查 dirty 时 +1
- **Promote（提升）**：当 `misses >= len(dirty)` 时，直接将 dirty 提升为新的 read（`read.Store(readOnly{m: dirty})`），dirty 置 nil，misses 归零
- 提升是 O(1) 操作（只是指针赋值），但下次有新 key 写入时需要重建 dirty（从 read 复制，O(N)）

### 10.5 与 RWMutex+map 的性能对比

| 场景 | sync.Map | RWMutex+map | 建议 |
|------|----------|-------------|------|
| 读多写少（99%读） | ⚡ 极快（无锁读） | 一般（RLock开销） | ✅ sync.Map |
| 读写均衡（50/50） | 一般 | ⚡ 更快 | ✅ RWMutex+map |
| 写多读少 | ❌ 慢（频繁rebuild dirty） | ⚡ 更快 | ✅ RWMutex+map |
| key 固定不变（只更新值） | ⚡ 极快（CAS更新entry） | 一般 | ✅ sync.Map |
| key 频繁增删 | ❌ 慢 | ⚡ 更快 | ✅ RWMutex+map |

**经验法则**：如果你的 map 像"缓存"一样使用（key 集合稳定，偶尔新增），用 `sync.Map`；如果像"工作集合"一样使用（频繁增删），用 `RWMutex+map`。

### 10.6 关键结构简化代码

```go
// 简化版 sync.Map 核心逻辑
func (m *Map) Load(key any) (value any, ok bool) {
    // 快速路径：从 read 中查找（无锁）
    read := m.read.Load()
    e, ok := read.m[key]
    if !ok && read.amended {
        // 慢路径：加锁查 dirty
        m.mu.Lock()
        // double-check（加锁期间可能已 promote）
        read = m.read.Load()
        e, ok = read.m[key]
        if !ok && read.amended {
            e, ok = m.dirty[key]
            m.missLocked() // misses++，可能触发 promote
        }
        m.mu.Unlock()
    }
    if !ok { return nil, false }
    return e.load()
}

func (m *Map) missLocked() {
    m.misses++
    if m.misses < len(m.dirty) { return }
    // promote: dirty → read
    m.read.Store(&readOnly{m: m.dirty})
    m.dirty = nil
    m.misses = 0
}
```

## 11. Java ForkJoinPool 详解

### 11.1 工作窃取（Work-Stealing）算法原理

ForkJoinPool 的核心是**工作窃取**算法：
- 每个工作线程拥有一个**双端队列（Deque）**存放任务
- 线程从自己队列的**头部**取任务执行（LIFO，利用局部性）
- 当自己队列为空时，从其他线程队列的**尾部**偷取任务（FIFO，偷大任务）
- 减少线程间的竞争（头部和尾部操作分离）

```
Thread-0 Deque:    [taskA | taskB | taskC]  ← 从头部取（LIFO）
Thread-1 Deque:    [taskD | taskE]
Thread-2 Deque:    [空]  ──偷取──→ Thread-0 尾部的 taskC（FIFO）

优势：
- 任务自然负载均衡（空闲线程主动找活干）
- 减少同步开销（大部分操作在本地队列，无竞争）
- 适合递归分治任务（fork产生的子任务留在本地）
```

### 11.2 与 ThreadPoolExecutor 对比

| 特性 | ThreadPoolExecutor | ForkJoinPool |
|------|-------------------|--------------|
| **任务队列** | 全局共享队列（单点竞争） | 每线程本地队列 + 窃取 |
| **调度策略** | FIFO / 优先级 | LIFO本地 + FIFO窃取 |
| **适用任务** | 独立、异构任务 | 可分治的递归任务 |
| **负载均衡** | 依赖队列分配 | 自动（工作窃取） |
| **线程数** | 核心线程 + 最大线程 | parallelism（默认=CPU核数） |
| **阻塞处理** | 线程直接阻塞 | compensation（补偿线程） |
| **典型场景** | Web请求处理、IO任务 | 并行计算、大数组处理 |

### 11.3 RecursiveTask / RecursiveAction 使用示例

```java
import java.util.concurrent.*;

// RecursiveTask<V>：有返回值
class SumTask extends RecursiveTask<Long> {
    private final long[] array;
    private final int start, end;
    private static final int THRESHOLD = 10000;

    SumTask(long[] array, int start, int end) {
        this.array = array; this.start = start; this.end = end;
    }

    @Override
    protected Long compute() {
        if (end - start <= THRESHOLD) {
            // 基本情况：直接计算
            long sum = 0;
            for (int i = start; i < end; i++) sum += array[i];
            return sum;
        }
        // 递归分治
        int mid = (start + end) / 2;
        SumTask left = new SumTask(array, start, mid);
        SumTask right = new SumTask(array, mid, end);
        left.fork();       // 异步执行左半部分
        long rightResult = right.compute(); // 当前线程执行右半部分
        long leftResult = left.join();      // 等待左半部分结果
        return leftResult + rightResult;
    }
}

// 使用
ForkJoinPool pool = new ForkJoinPool(); // 默认 parallelism = Runtime.availableProcessors()
long[] array = new long[1_000_000];
Long result = pool.invoke(new SumTask(array, 0, array.length));

// RecursiveAction：无返回值（如并行排序）
class SortTask extends RecursiveAction {
    @Override
    protected void compute() {
        if (array.length <= THRESHOLD) { Arrays.sort(array); return; }
        // fork子任务...
    }
}
```

**compensation（补偿机制）**：

## 12. 无锁队列实际实现

### 12.1 Michael-Scott Queue 算法原理

Michael-Scott Queue（1996）是最经典的无锁 FIFO 队列算法：
- 基于单向链表 + 两个指针（head, tail）
- 使用 CAS 操作保证并发安全
- 支持多生产者多消费者（MPMC）

```
核心思想：

Enqueue（入队）：
1. 创建新节点 node
2. 读取 tail，读取 tail.next
3. 如果 tail.next == nil（tail确实是最后节点）
   → CAS(tail.next, nil, node)  // 挂载新节点
   → CAS(tail, oldTail, node)   // 推进tail（即使失败也没关系，别人会帮忙推进）
4. 如果 tail.next != nil（tail已落后）
   → CAS(tail, oldTail, tail.next) // 帮忙推进tail，然后重试

Dequeue（出队）：
1. 读取 head，读取 head.next
2. 如果 head.next == nil → 队列为空
3. CAS(head, oldHead, head.next) // 摘下head（dummy node）
4. 返回 head.next.value

关键设计：
- head 始终指向一个 dummy node（哨兵节点）
- 实际数据从 head.next 开始
- tail 可能落后（惰性更新），但enqueue会帮忙推进
```

### 12.2 CAS + 链表的 Go 实现

```go
package lockfree

import (
    "sync/atomic"
    "unsafe"
)

// 节点
type node struct {
    value any
    next  atomic.Pointer[node]
}

// 无锁队列（Michael-Scott Queue）
type LockFreeQueue struct {
    head atomic.Pointer[node]
    tail atomic.Pointer[node]
}

func NewLockFreeQueue() *LockFreeQueue {
    dummy := &node{} // 哨兵节点
    q := &LockFreeQueue{}
    q.head.Store(dummy)
    q.tail.Store(dummy)
    return q
}

// Enqueue 入队（多生产者安全）
func (q *LockFreeQueue) Enqueue(value any) {
    newNode := &node{value: value}
    for {
        tail := q.tail.Load()
        next := tail.next.Load()
        if tail == q.tail.Load() { // 一致性检查
            if next == nil {
                // tail确实是最后节点，尝试挂载
                if tail.next.CompareAndSwap(nil, newNode) {
                    // 成功挂载，尝试推进tail（失败也无妨）
                    q.tail.CompareAndSwap(tail, newNode)
                    return
                }
            } else {
                // tail落后了，帮忙推进
                q.tail.CompareAndSwap(tail, next)
            }
        }
    }
}

// Dequeue 出队（多消费者安全）
func (q *LockFreeQueue) Dequeue() (any, bool) {
    for {
        head := q.head.Load()
        tail := q.tail.Load()
        next := head.next.Load()
        if head == q.head.Load() { // 一致性检查
            if head == tail {
                if next == nil {
                    return nil, false // 队列为空
                }
                // tail落后，帮忙推进
                q.tail.CompareAndSwap(tail, next)
            } else {
                // 读取值（在CAS之前，防止被其他线程释放）
                value := next.value
                if q.head.CompareAndSwap(head, next) {
                    return value, true // 成功出队
                }
            }
        }
    }
}
```

### 12.3 ABA 问题和解决方案

**ABA 问题在无锁队列中的体现**：
```
线程1：Dequeue，读取 head=A, next=B
线程1：准备 CAS(head, A, B)...被挂起

线程2：Dequeue A，Dequeue B
线程2：Enqueue X，Enqueue A（A被重新使用/分配到同一地址）

线程1恢复：CAS(head, A, B) 成功！
→ 但B可能已经被释放，head指向悬垂指针！
```

**解决方案一：Tagged Pointer（带版本号的指针）**

```go
// 将版本号打包进指针（利用64位系统中指针高位未使用的位）
type taggedPtr struct {
    ptr unsafe.Pointer
    tag uint64  // 每次修改+1，防止ABA
}

// 或使用128位CAS（x86的CMPXCHG16B）
type taggedNode struct {
    node *node
    tag  uint64
}
// atomic.CompareAndSwap128(&head, old, new)

// Go中的实现思路：用 atomic.Uint64 打包
// 高32位=tag，低32位=index（用数组代替指针）
```

**解决方案二：Epoch-Based Reclamation（基于纪元的内存回收）**

```
核心思想：延迟释放内存，确保没有线程在使用旧节点时才回收

1. 全局维护 epoch 计数器（0, 1, 2 循环）
2. 每个线程进入临界区时记录当前 epoch
3. 线程要释放节点时，放入当前 epoch 的回收列表
4. 当所有线程都不在旧 epoch 中时，回收该 epoch 的节点

优势：批量回收，摊薄开销
劣势：如果某线程长期不退出临界区，内存无法回收
```

**解决方案三：Go 的 GC 天然解决 ABA**

```
Go 有 GC，被引用的对象不会被回收和复用。
因此上面的 Go 实现天然不存在 ABA 问题：
- 只要某线程还持有 node 指针，GC 不会回收该 node
- 新分配的 node 一定是新地址
- 这是 Go 实现无锁数据结构比 C/C++ 简单的主要原因
```

### 12.4 无锁 vs 有锁的性能对比和适用场景

| 维度 | 无锁队列 | 有锁队列（Mutex） |
|------|---------|-----------------|
| **低竞争** | ⚡ 更快（无syscall） | 一般（锁开销小但存在） |
| **中等竞争** | ⚡ 明显更快 | 慢（频繁休眠/唤醒） |
| **高竞争** | ⚠️ CAS自旋消耗CPU | 可能更好（休眠让出CPU） |
| **延迟** | 低且稳定（无阻塞） | 可能突发高延迟（等锁） |
| **吞吐** | 高（真正并行） | 受限于串行临界区 |
| **实现复杂度** | 高（ABA、内存序） | 低（加锁/解锁） |
| **调试难度** | 极高 | 一般 |
| **正确性验证** | 困难（需形式化证明） | 相对简单 |

**适用场景总结**：

| 选择 | 场景 |
|------|------|
| ✅ 用无锁队列 | 延迟敏感（交易系统）、高吞吐消息传递、中等竞争 |
| ✅ 用有锁队列 | 业务逻辑复杂、团队维护成本优先、竞争极高 |
| ✅ 用 channel（Go） | 大多数 Go 业务场景（已足够高效） |

> **经验法则**：除非 benchmark 证明锁是瓶颈，否则优先使用有锁方案。无锁编程的正确性远比性能更难保证。

## 13. 面试题自查

### 初级高频速刷

**1. 原子性、可见性、有序性分别是什么意思？**

原子性是操作不可分割；可见性是一个执行单元写入后其他执行单元能看到新值；有序性是程序关键指令不会被错误重排。并发 bug 往往不是“代码没跑”，而是这三件事里有一件没有被正确保证。

**2. 互斥锁、读写锁、信号量、条件变量分别解决什么问题？**

互斥锁解决“同一资源不能同时改”；读写锁解决“读多写少时多个读者能并发”；信号量解决“最多允许 N 个并发进入”；条件变量解决“线程要等待某个条件成立再继续”。面试官通常会追问你有没有把它们混用错场景。

**3. 为什么线程池能减少开销？线程池大小一般怎么估算？**

线程池通过复用线程减少频繁创建/销毁成本，同时能限制最大并发，避免系统被打爆。粗略上，CPU 密集型任务线程数接近 CPU 核数，IO 密集型任务可以更大，但最终还是要用压测和监控验证，而不是死背公式。

**4. `sleep`、`wait`、`park` 分别是什么语义？**

`sleep` 是“睡一会儿”，不会释放锁；`wait` 通常和条件队列/监视器配套使用，会释放监视器并等待通知；`park` 是更底层的阻塞原语，常见于并发框架内部。三者都能让执行流停下来，但语义边界完全不同。

**5. 乐观锁和悲观锁的核心区别是什么？**

悲观锁假设冲突经常发生，先加锁再操作；乐观锁假设冲突较少，先执行，提交时再校验版本。前者更保守、更容易保证一致性，后者更适合读多写少、冲突率低的场景。

### 中级高频进阶

**6. happens-before 到底在并发里意味着什么？**

happens-before 不只是“先执行”，而是“前一个操作的结果对后一个操作可见，且顺序被正确约束”。它是判断内存可见性和数据竞争的核心抽象，没有它，就无法严谨讨论多线程读写是否安全。

**7. 为什么条件变量 `wait` 通常要写在 `for` 循环里，而不是 `if`？**

因为线程被唤醒并不等于条件一定满足，可能是虚假唤醒，也可能是别的线程先一步把条件消耗掉了。所以正确写法是“被唤醒后重新检查条件”，而不是默认条件成立。

**8. 自旋什么时候可能比互斥锁更好，什么时候会更糟？**

临界区非常短、竞争不激烈、线程很快就能拿到锁时，自旋能省掉休眠唤醒成本；但如果锁持有时间长或竞争激烈，自旋只会白白烧 CPU，反而比 mutex 更差。这个取舍没有银弹，必须结合场景。

### 深入问答

#### Q1：并发和并行有什么区别？单核 CPU 能实现并行吗？

**答**：**并发**是逻辑上同时处理多个任务（交替执行），**并行**是物理上同时执行多个任务（多核同时运行）。单核 CPU 不能并行，但可以并发——通过时间片轮转在多个任务间快速切换，给用户"同时"的感觉。Go 的 goroutine 在单核上是并发的（GMP 调度器切换），多核上既并发也并行。

---

#### Q2：进程、线程、协程的区别和各自的适用场景？

**答**：**进程**：独立内存空间，OS 调度，开销大（MB 级），适合 CPU 密集型隔离（如 Nginx worker）。**线程**：共享进程内存，OS 调度，开销中等（KB 级），适合需要共享内存的并发场景。**协程**：共享线程栈，用户态调度，开销极小（2KB 起），适合高并发 IO 场景（Go goroutine、Python asyncio）。关键区别是调度方式：进程/线程是抢占式（OS 控制），协程是协作式（用户代码主动让出）。

---

#### Q3：Go 的 Channel 底层是怎么实现的？发送和接收的完整流程？

**答**：Channel 底层是 `hchan` 结构体，包含环形缓冲区（`buf`）、发送/接收索引（`sendx`/`recvx`）、等待队列（`sendq`/`recvq`）和互斥锁（`lock`）。**发送流程**：加锁→如果有等待接收方则直接拷贝数据给它→否则缓冲区有空间则写入→否则加入 sendq 挂起。**接收流程**：加锁→如果有等待发送方则直接取数据→否则缓冲区有数据则读取→否则加入 recvq 挂起。关键优化：有等待方时数据直接在两个 goroutine 间拷贝，跳过缓冲区。

---

#### Q4：无缓冲 Channel 和有缓冲 Channel 的区别？分别在什么场景下使用？

**答**：**无缓冲**（`make(chan T)`）：发送和接收必须同时就绪，提供**同步语义**——保证发送方数据被接收方消费后才继续。适用于 goroutine 间同步通信、信号传递。**有缓冲**（`make(chan T, N)`）：缓冲未满时发送不阻塞，提供**异步语义**——发送方和接收方解耦。适用于生产者-消费者模式、限流（缓冲大小=并发限制）。

---

#### Q5：Go 的 Mutex 有哪几种模式？什么时候会进入饥饿模式？

**答**：Go Mutex 有**正常模式**和**饥饿模式**。正常模式下，被唤醒的 goroutine 和新到来的 goroutine 竞争锁，新 goroutine 更有优势（已在 CPU 上运行）。当等待方超过 **1ms** 未获得锁，切换为饥饿模式——锁直接交给队列头部 goroutine（FIFO 公平），新 goroutine 不自旋直接入队。当最后一个等待者获得锁或等待时间 < 1ms 时退出饥饿模式。此外还有自旋优化（最多 4 次 CAS 自旋）。

---

#### Q6：什么是 CAS？它的底层是如何实现原子性的？ABA 问题是什么？

**答**：CAS（Compare-And-Swap）是无锁原子操作：如果当前值等于期望值则替换为新值。底层通过 CPU 的 **LOCK CMPXCHG** 指令实现，LOCK 前缀锁定缓存行（MESI 协议），保证操作的原子性。**ABA 问题**：值从 A 变为 B 再变回 A，CAS 认为没有变化但实际中间状态已改变。解决方案：附加版本号（每次修改 version++），CAS 同时比较值和版本号。

---

#### Q7：什么是内存屏障？为什么 CPU 会重排序指令？

**答**：现代 CPU 为了提升流水线效率会对指令**重排序**（Store Buffer、乱序执行），单线程下结果正确但多线程下可能导致可见性问题。**内存屏障**是 CPU 指令（如 x86 的 MFENCE），强制刷新 Store Buffer 并禁止特定类型的重排序。4 种类型：LoadLoad、StoreStore、LoadStore、StoreLoad（最重）。x86（TSO 模型）自动保证前三种，只需关心 StoreLoad 重排序。Go 的 `sync/atomic` 操作隐式包含内存屏障。

---

#### Q8：Java 的 volatile 底层是怎么实现的？volatile 能保证原子性吗？

**答**：volatile 通过 JVM 在字节码中插入**内存屏障**实现：写 volatile 后插入 StoreLoad 屏障（刷新 Store Buffer + 失效其他 CPU 缓存行），读 volatile 前插入 LoadLoad 屏障（从主内存加载最新值）。x86 上编译为 `LOCK addl $0, (%rsp)` 指令。volatile **不能保证原子性**——`count++` 是读-改-写三步操作，volatile 只保证每一步的可见性，但线程可能在步骤之间被切换。复合操作需要 synchronized 或 AtomicInteger。

#### Q9：原子操作和互斥锁在性能上有什么区别？各自适用于什么场景？

**答**：**原子操作**：无锁（CAS 循环），不需要系统调用，只锁缓存行，无上下文切换，适合单变量的简单操作（计数器、标志位）和低竞争场景。**互斥锁**：获取失败可能触发线程休眠/唤醒（系统调用），有上下文切换开销，但适合大临界区（多条语句）、需要条件变量、需要公平性的场景。一般原子操作比锁快 1.5~2 倍。高竞争下 Mutex 的饥饿模式保证公平性优于 CAS 无限自旋。

#### Q10：什么是死锁？死锁的四个必要条件？如何预防和检测？

**答**：死锁是两个或多个线程互相等待对方持有的资源，导致所有线程都无法继续。**四个必要条件**：① 互斥（资源排他使用）② 持有并等待（持有一个资源同时等待另一个）③ 不可剥夺（已持有的资源不能强制释放）④ 循环等待（A 等 B，B 等 A）。**预防**：破坏任一条件，最常用的是统一锁获取顺序（破坏循环等待）、使用 tryLock+超时。**检测**：Go 的 `go vet -race`、Java 的 `jstack` 查看线程状态、数据库死锁检测日志。

---

#### Q11：Fan-Out/Fan-In、Pipeline、Worker Pool 三种并发模式的区别和适用场景？

**答**：**Pipeline（管道）**：多个阶段串联，每个阶段做不同处理（如 generate→filter→transform），适合数据流处理。**Fan-Out/Fan-In（扇出扇入）**：一个输入分发到多个 worker 并行处理相同操作，结果汇聚，适合并行化计算密集操作。**Worker Pool（工作池）**：固定数量 worker 从共享队列取任务，控制并发度，适合任务队列场景（限制 goroutine 数量防止资源耗尽）。

#### Q12：Go 中如何优雅地退出多个 goroutine？Context 的取消机制是怎么实现的？

**答**：使用 `context.WithCancel` 创建可取消的 context，调用 `cancel()` 后所有监听 `ctx.Done()` 的 goroutine 收到关闭信号并退出。底层实现：context 内部维护一个 `done` channel，`cancel()` 调用 `close(done)`，所有 `<-ctx.Done()` 的 select case 同时返回。`WithTimeout`/`WithDeadline` 额外使用 `time.AfterFunc` 定时调用 cancel。context 是树形结构，父 context 取消时递归取消所有子 context。

---

#### Q13：读写锁（RWMutex）的适用场景？如果写操作很频繁还适合用读写锁吗？

**答**：读写锁适合**读多写少**场景——多个读操作可以并行（共享读锁），写操作排他。如果写操作频繁，读写锁可能比普通 Mutex 更慢，因为：① 写锁需要等所有读锁释放（写饥饿）；② RWMutex 内部维护更复杂的状态（读计数器、写等待标志），锁操作开销更大。写频繁时建议用普通 Mutex 或分片加锁（ConcurrentHashMap 思路）。

---

#### Q14：Go 的 select 语句在多个 case 同时就绪时会怎样？如何实现超时控制？

**答**：多个 case 同时就绪时，Go 运行时会**随机选择**一个执行（不是按顺序），保证公平性。超时控制：`select { case v := <-ch: ... case <-time.After(timeout): ... }`，`time.After` 返回一个 channel，在指定时间后发送信号。也可以用 `context.WithTimeout` 创建带截止时间的 context，在 select 中监听 `ctx.Done()`。

#### Q15：如何检测 Go 程序中的数据竞争？`go vet -race` 的原理是什么？

**答**：使用 `go run -race` 或 `go test -race` 开启竞态检测器。原理是基于 **ThreadSanitizer** 算法：编译时在每个内存访问点插入检测代码，运行时维护每个内存地址的**向量时钟**（Vector Clock），记录最近的读/写线程和时间。如果检测到两个访问同一地址的操作来自不同 goroutine、至少一个是写、且没有 happens-before 关系，就报告数据竞争。开启 race detector 会使程序慢 2~10 倍，内存增加 5~10 倍，仅用于测试环境。

#### Q16：什么是 false sharing（伪共享）？为什么它会让无锁代码也变慢？

**答**：伪共享指多个线程修改彼此独立的变量，但这些变量恰好落在同一个 CPU cache line（通常 64B）里，导致缓存一致性协议频繁失效和回写。结果是代码逻辑上没有锁竞争，硬件层面却在不断争用同一缓存行。典型场景是多个 goroutine 分别更新结构体中相邻的计数器。解决方法：做 cache line padding、按线程分片计数、减少热点共享写。

---

#### Q17：为什么说“锁的粒度”不是越细越好？如何权衡粗粒度锁和细粒度锁？

**答**：细粒度锁能提高并行度，但也会带来更高的加锁开销、更复杂的死锁风险和更差的可维护性。粗粒度锁实现简单、容易保证一致性，但在热点竞争下吞吐会受限。权衡原则：先用最简单正确的粗粒度锁跑通，再通过 profile 看是否真有锁竞争热点；只有锁等待成为瓶颈时，才拆成分段锁、读写锁或无锁结构。不要为了“理论并发度”过早引入复杂设计。

#### Q18：什么是 backpressure（背压）？并发系统里如果没有背压会发生什么？

**答**：背压是下游处理不过来时，向上游传递“减速”信号的机制。没有背压时，上游会持续生产任务，导致队列无限堆积、内存暴涨、延迟失控，最后演变成雪崩。常见实现方式：有界队列、限流、阻塞写入、丢弃策略、credit-based flow control。判断一个系统是否具备背压能力，关键看高峰时它是“稳定变慢”还是“直接堆死”。

---

#### Q19：并发程序里如何做取消（cancellation）和超时传播？为什么只靠 close(channel) 往往不够？

**答**：取消和超时要沿调用链传播，而不是只在局部 goroutine 内部生效。`context.Context` 适合做跨 goroutine、跨 RPC 的统一取消语义，因为它同时携带 deadline、取消信号和请求级元数据。只靠 `close(channel)` 往往不够，因为它只能通知本地监听者，不能自然传递到数据库、HTTP 客户端、RPC 调用等下游依赖。资深实现里一般要求：入口设置 deadline，中间层不自行吞掉 ctx，所有 IO 都要感知 ctx。

#### Q20：如何判断一个无锁算法值不值得上生产？面试里你会从哪几个维度审视它？

**答**：我会看五件事：① 是否真的有锁竞争瓶颈，还是只是”为了炫技”上无锁；② 是否正确处理 ABA、内存回收和内存序；③ 在高竞争场景下是否可能空转自旋，反而比锁更差；④ 是否容易验证和排障，团队能否长期维护；⑤ 是否有成熟实现可复用，而不是手写一套难以证明正确性的结构。无锁不是默认更快，它只是把复杂度从内核锁迁移到了算法和内存模型上。

### 开放式设计题

**D1：设计一个通用的并发任务执行框架，支持任务依赖、超时控制和优雅关闭，你会怎么设计？**

**参考思路**：
- 任务建模：DAG表示依赖关系、拓扑排序确定执行顺序、并行执行无依赖任务
- 超时控制：context.WithTimeout传播、每个任务独立超时+整体超时
- 错误处理：Fast-fail（任一任务失败则取消其余）vs Best-effort（尽量执行所有）
- 优雅关闭：signal通知→停止新任务→等待运行中任务完成（超时强杀）→清理资源
- Go实现要点：errgroup/semaphore控制并发度、channel通知完成、sync.WaitGroup等待

**D2：在一个高并发系统中，如何选择”加锁”还是”无锁”方案？给出决策框架**

**参考思路**：
- 评估维度：竞争频率（高→无锁收益大）、临界区大小（大→锁更合适）、一致性要求、代码复杂度容忍度
- 锁方案适用：临界区包含IO操作、需要复杂原子操作、团队无锁经验不足
- 无锁适用：热点计数器、简单状态机、读远多于写（atomic.Value）
- 混合策略：分段锁（ConcurrentHashMap思路）、读写分离（sync.RWMutex/atomic读+锁写）
