## 1. 大数据基础概念

### 1.1 大数据的4V特征

| 特征 | 说明 | 举例 |
|------|------|------|
| **Volume（体量）** | 数据量大（TB/PB级） | 每日10亿条日志 |
| **Velocity（速度）** | 产生速度快 | 每秒10万条消息 |
| **Variety（多样性）** | 数据类型多 | 日志、指标、事件 |
| **Value（价值）** | 价值密度低 | 99%噪音，1%有价值 |

### 1.2 Lambda架构

```
                    ┌─────────────┐
                    │  实时数据    │
                    └──────┬──────┘
                           │
            ┌──────────────┼──────────────┐
            │              │              │
    ┌───────▼──────┐  ┌────▼────┐  ┌─────▼─────┐
    │  批处理层     │  │ 速度层   │  │  服务层    │
    │ (Spark/Hive) │  │ (Flink)  │  │ (查询)     │
    └──────────────┘  └─────────┘  └───────────┘
```

**批处理层（Batch Layer）**：处理全量历史数据（Spark、Hive）  
**速度层（Speed Layer）**：处理实时数据（Flink、Storm）  
**服务层（Serving Layer）**：合并批处理和实时结果

## 2. 批处理

### 2.1 MapReduce原理深入

#### 2.1.1 MapReduce执行流程

```
┌────────────┐
│  输入数据   │  HDFS上的大文件
└──────┬─────┘
       │ split（切分成块）
┌──────▼─────┐
│  Input     │  Block 1: 64MB, Block 2: 64MB, Block 3: 64MB
│  Splits    │
└──────┬─────┘
       │ 并行处理
┌──────▼──────────────────────────────┐
│         Map阶段（并行）              │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐│
│  │ Mapper1 │ │ Mapper2 │ │ Mapper3 ││
│  │(Block1) │ │(Block2) │ │(Block3) ││
│  └────┬────┘ └────┬────┘ └────┬────┘│
│       │           │           │     │
│     (k1,v1)     (k2,v2)     (k3,v3) │
└───────┼───────────┼───────────┼─────┘
        │           │           │
        └───────────┼───────────┘
                    │ Shuffle & Sort
┌───────────────────▼─────────────────┐
│      Shuffle阶段（网络传输）         │
│  - 分区（Partitioner）               │
│  - 排序（Sort）                      │
│  - 合并（Combine）                   │
│  - 分组（Group）                     │
└───────────────────┬─────────────────┘
                    │
┌───────────────────▼─────────────────┐
│        Reduce阶段（并行）            │
│  ┌──────────┐ ┌──────────┐          │
│  │ Reducer1 │ │ Reducer2 │          │
│  │(key:a-m) │ │(key:n-z) │          │
│  └─────┬────┘ └─────┬────┘          │
│        │            │               │
│   output_part1  output_part2        │
└────────┼────────────┼────────────────┘
         │            │
         └────────────┘
         ↓
    ┌─────────┐
    │ HDFS输出 │
    └─────────┘
```

#### 2.1.2 Map阶段详解

```python
# Map函数：处理单条记录
def map_function(key, value):
    """
    输入：
    - key: 文件偏移量（通常忽略）
    - value: 一行文本
    
    输出：
    - 多个(key, value)对
    """
    words = value.split()
    for word in words:
        emit(word, 1)  # 输出中间结果

# Map内部流程
class Mapper:
    def __init__(self):
        self.buffer = []  # 内存缓冲区
        self.buffer_size = 100  # 缓冲区大小（简化）
    
    def map(self, key, value):
        # 1. 处理输入
        results = map_function(key, value)
        
        # 2. 写入缓冲区
        for k, v in results:
            self.buffer.append((k, v))
            
            # 3. 缓冲区满了，溢写到磁盘
            if len(self.buffer) >= self.buffer_size:
                self.spill_to_disk()
    
    def spill_to_disk(self):
        # 4. 分区：根据key分配到不同Reducer
        partitions = {}
        for k, v in self.buffer:
            partition_id = hash(k) % num_reducers
            if partition_id not in partitions:
                partitions[partition_id] = []
            partitions[partition_id].append((k, v))
        
        # 5. 排序：每个分区内按key排序
        for partition_id, kvs in partitions.items():
            kvs.sort(key=lambda x: x[0])
            write_to_disk(f"map_output_{partition_id}", kvs)
        
        self.buffer = []  # 清空缓冲区
```

**Map输出示例**：
```
Map Input:
  "hello world"
  "hello spark"

Map Output（分区0）:
  (hello, 1)
  (hello, 1)

Map Output（分区1）:
  (world, 1)
  (spark, 1)
```

#### 2.1.3 Shuffle阶段详解

**Shuffle是MapReduce的核心**（决定性能）

```python
class Shuffle:
    def shuffle(self):
        # 1. Copy阶段：从所有Mapper拉取数据
        for mapper_id in range(num_mappers):
            partition_file = f"mapper_{mapper_id}_partition_{self.reducer_id}"
            data = fetch_from_mapper(mapper_id, partition_file)
            self.merge_data(data)
        
        # 2. Sort阶段：归并排序
        self.sorted_data = merge_sort(self.data)
        
        # 3. Group阶段：相同key的value分组
        self.grouped_data = group_by_key(self.sorted_data)
        # 例如：[(hello, [1, 1, 1]), (world, [1, 1])]

# 网络传输优化
class ShuffleOptimizer:
    def __init__(self):
        self.combiner = Combiner()  # 本地聚合
    
    def optimize(self, map_output):
        # Combiner：Map端预聚合，减少网络传输
        # 例如：(hello, 1), (hello, 1) → (hello, 2)
        combined = self.combiner.combine(map_output)
        return combined

# 示例：WordCount的Combiner
def combiner(key, values):
    # Map输出：(hello, 1), (hello, 1), (hello, 1)
    # Combiner输出：(hello, 3)
    return key, sum(values)

# 效果：
# 不用Combiner：网络传输3条记录
# 用Combiner：网络传输1条记录（减少66%）
```

#### 2.1.4 Reduce阶段详解

```python
class Reducer:
    def reduce(self, key, values):
        """
        输入：
        - key: 单词
        - values: 该单词的所有计数 [1, 1, 1, ...]
        
        输出：
        - (key, aggregated_value)
        """
        total = sum(values)
        emit(key, total)

# Reduce内部流程
class ReduceTask:
    def run(self):
        # 1. 从Shuffle获取分组数据
        for key, values in self.grouped_data:
            # 2. 调用用户定义的reduce函数
            result = reduce_function(key, values)
            
            # 3. 写入HDFS
            self.output.write(result)

# 输出示例
# part-00000（Reducer 0的输出）:
# hello   3
# world   2

# part-00001（Reducer 1的输出）:
# spark   1
# python  2
```

#### 2.1.5 数据倾斜问题

**问题**：某个key的数据量特别大，导致单个Reducer处理时间过长

```python
# 示例：统计用户订单量
# 问题：VIP用户order_count=1000000，普通用户order_count=10
orders = [
    ("user_1", 1),    # VIP用户
    ("user_1", 1),
    # ... 100万条
    ("user_2", 1),    # 普通用户
    ("user_2", 1),
]

# 默认分区：user_1全部发送到一个Reducer（数据倾斜）
```

**解决方案1：加盐（Salting）**

```python
# Map阶段：给热key添加随机后缀
def map_with_salt(key, value):
    if is_hot_key(key):
        # 热key：user_1 → user_1_0, user_1_1, ..., user_1_9
        salt = random.randint(0, 9)
        emit(f"{key}_{salt}", value)
    else:
        emit(key, value)

# Reduce阶段1：部分聚合
# user_1_0 → 100000
# user_1_1 → 100000
# ...

# Reduce阶段2：去盐，最终聚合
def final_reduce(key, value):
    original_key = key.rsplit('_', 1)[0]  # user_1_0 → user_1
    emit(original_key, value)
```

**解决方案2：两阶段聚合**

```python
# 第1阶段：Map端预聚合
map_output = [
    ("user_1_mapper_0", 50000),  # Mapper 0处理的user_1数据
    ("user_1_mapper_1", 50000),  # Mapper 1处理的user_1数据
]

# 第2阶段：Reduce端汇总
reduce_output = [
    ("user_1", 100000)
]
```

### 2.2 Spark原理深入

#### 2.2.1 RDD（弹性分布式数据集）

**RDD特性**：
1. **不可变**：创建后不能修改，只能通过转换生成新RDD
2. **分区**：数据分布在多个节点
3. **容错**：记录Lineage（血统），丢失后可重新计算

```python
# RDD转换（Transformation）：惰性执行
rdd1 = sc.textFile("file.txt")       # 读取文件
rdd2 = rdd1.flatMap(lambda x: x.split())  # 分词
rdd3 = rdd2.map(lambda x: (x, 1))    # 转换为(word, 1)
rdd4 = rdd3.reduceByKey(lambda a, b: a + b)  # 聚合

# 此时还没有真正执行，只是记录了转换关系（Lineage）

# RDD行动（Action）：触发计算
result = rdd4.collect()  # 收集结果到Driver

# Lineage（血统）记录：
# rdd4 = rdd3.reduceByKey(...)
# rdd3 = rdd2.map(...)
# rdd2 = rdd1.flatMap(...)
# rdd1 = sc.textFile(...)

# 如果rdd3丢失，可以根据Lineage重新计算
```

**RDD内部结构**：
```python
class RDD:
    def __init__(self):
        self.partitions = []      # 分区列表
        self.dependencies = []    # 依赖的父RDD
        self.partitioner = None   # 分区器
        self.compute_func = None  # 计算函数
    
    def compute(self, partition):
        """计算单个分区"""
        # 1. 获取父RDD的分区数据
        parent_data = self.dependencies[0].compute(partition)
        
        # 2. 应用转换函数
        result = self.compute_func(parent_data)
        
        return result
```

#### 2.2.2 宽依赖vs窄依赖

```
窄依赖（Narrow Dependency）：
  父RDD的一个分区 → 子RDD的一个分区
  
  例如：map、filter
  
  RDD1: [P0] [P1] [P2]
         ↓    ↓    ↓
  RDD2: [P0] [P1] [P2]

宽依赖（Wide Dependency）：
  父RDD的多个分区 → 子RDD的一个分区（需要Shuffle）
  
  例如：reduceByKey、groupByKey
  
  RDD1: [P0] [P1] [P2]
         ↓  ↙↓↘  ↓
  RDD2:   [P0] [P1]
```

**为什么区分宽窄依赖？**
- **窄依赖**：可以流水线执行（Pipeline），不需要等待
- **宽依赖**：需要Shuffle，产生Stage边界

#### 2.2.3 DAG调度

```python
# Spark将作业划分为Stage
# Stage边界：宽依赖处

# 示例：WordCount的DAG
# textFile → flatMap → map → reduceByKey → collect
#                                ↑ Shuffle（Stage边界）

# Stage 0（Map阶段）:
#   textFile → flatMap → map

# Stage 1（Reduce阶段）:
#   reduceByKey → collect

# 调度器
class DAGScheduler:
    def submit_job(self, rdd):
        # 1. 构建DAG
        dag = build_dag(rdd)
        
        # 2. 划分Stage（从后往前遍历，遇到Shuffle就切分）
        stages = split_stages(dag)
        
        # 3. 提交Stage（按依赖顺序）
        for stage in stages:
            self.submit_stage(stage)
    
    def submit_stage(self, stage):
        # 4. 创建Task（每个分区一个Task）
        tasks = []
        for partition in stage.partitions:
            task = Task(stage, partition)
            tasks.append(task)
        
        # 5. 提交Task到Executor执行
        self.task_scheduler.submit_tasks(tasks)
```

## 3. 流式计算

### 3.1 流式计算基础概念

#### 3.1.1 批处理vs流处理

| 维度 | 批处理 | 流处理 |
|------|--------|--------|
| **数据特征** | 有界数据集 | 无界数据流 |
| **处理延迟** | 分钟~小时 | 毫秒~秒 |
| **处理方式** | 一次性处理全部数据 | 持续处理到达的数据 |
| **状态管理** | 简单（无需维护状态） | 复杂（需要维护状态） |
| **结果完整性** | 完整 | 近似（基于窗口） |

#### 3.1.2 事件时间vs处理时间

```
事件时间（Event Time）：数据产生的时间
处理时间（Processing Time）：数据被处理的时间

示例：
  事件A: 发生于 10:00:00，到达系统 10:00:05（延迟5秒）
  事件B: 发生于 10:00:01，到达系统 10:00:02（延迟1秒）

处理时间顺序：B → A
事件时间顺序：A → B

问题：网络延迟、系统故障导致事件乱序到达
```

**为什么需要事件时间？**
```python
# 场景：统计每小时的订单量

# 使用处理时间（错误）：
# 10:59:59生成的订单，11:00:01才到达
# → 被统计到11点的数据中（错误）

# 使用事件时间（正确）：
# 根据订单的创建时间统计
# → 无论何时到达，都统计到10点的数据中
```

### 3.2 Flink核心概念

#### 3.2.1 水位线（Watermark）

**水位线的作用**：处理乱序数据

```
数据流（带时间戳）:
  事件A(t=1) → 事件B(t=3) → 事件C(t=2) → 事件D(t=5)
                                ↑ 乱序

水位线（Watermark）：表示"时间t之前的数据都已到达"

Watermark(t=3) 表示：
  - 时间3之前的数据（A、C）都已到达
  - 可以触发[0, 3]窗口的计算

水位线生成策略：
  Watermark = 最大事件时间 - 允许延迟时间
```

**代码示例**：
```java
// Flink水位线生成
DataStream<Event> stream = env.addSource(...)
    .assignTimestampsAndWatermarks(
        WatermarkStrategy
            .<Event>forBoundedOutOfOrderness(Duration.ofSeconds(5))  // 允许5秒乱序
            .withTimestampAssigner((event, timestamp) -> event.getTimestamp())
    );

// 工作原理
class BoundedOutOfOrdernessWatermarkGenerator {
    private long maxTimestamp = Long.MIN_VALUE;
    private long outOfOrdernessMillis = 5000;  // 5秒
    
    public void onEvent(Event event, long eventTimestamp) {
        maxTimestamp = Math.max(maxTimestamp, eventTimestamp);
    }
    
    public Watermark getCurrentWatermark() {
        // Watermark = 最大事件时间 - 5秒
        return new Watermark(maxTimestamp - outOfOrdernessMillis);
    }
}

// 示例：
// 事件流：t=1, t=3, t=2, t=7, t=9
// maxTimestamp: 1 → 3 → 3 → 7 → 9
// Watermark:   -4 → -2 → -2 → 2 → 4

// 当Watermark=2时：
//   - 窗口[0, 3)可以触发计算（所有数据都到齐了）
//   - 窗口[3, 6)还在等待
```

**水位线传播**：
```
Source1 → [Watermark=5] ─┐
                          ├→ Join Operator
Source2 → [Watermark=3] ─┘     ↓
                            [Watermark=min(5,3)=3]
                            （取最小值，保守策略）
```

#### 3.2.2 窗口（Window）

**窗口类型**：

```
1. 滚动窗口（Tumbling Window）：固定大小，不重叠
   [0, 5) [5, 10) [10, 15)
   ─────  ─────   ─────

2. 滑动窗口（Sliding Window）：固定大小，可重叠
   [0, 10)
        [5, 15)
             [10, 20)
   ──────────
        ──────────
             ──────────

3. 会话窗口（Session Window）：动态大小，根据间隔分组
   事件：t=1, t=2, t=3 ... t=10, t=11
   会话1: [1, 3]（间隔<5）
   会话2: [10, 11]（间隔>5，新会话）
```

**代码示例**：
```java
// 1. 滚动窗口：每5分钟统计一次
stream
    .keyBy(event -> event.getUserId())
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .aggregate(new CountAggregateFunction());

// 2. 滑动窗口：每30秒统计最近1分钟
stream
    .keyBy(event -> event.getUserId())
    .window(SlidingEventTimeWindows.of(
        Time.minutes(1),    // 窗口大小
        Time.seconds(30)    // 滑动步长
    ))
    .aggregate(new AvgAggregateFunction());

// 3. 会话窗口：间隔30分钟
stream
    .keyBy(event -> event.getUserId())
    .window(EventTimeSessionWindows.withGap(Time.minutes(30)))
    .aggregate(new SessionAggregateFunction());
```

**窗口触发机制**：

#### 3.2.3 状态管理

**为什么需要状态？**
```java
// 场景：实时计算每个用户的累计消费金额

// 无状态（错误）：
public class SumFunction extends MapFunction {
    public Double map(Transaction txn) {
        return txn.getAmount();  // 只返回当前金额，无法累加
    }
}

// 有状态（正确）：
public class StatefulSumFunction extends RichMapFunction {
    private ValueState<Double> sumState;  // 状态：累计金额
    
    @Override
    public void open(Configuration config) {
        // 初始化状态
        sumState = getRuntimeContext().getState(
            new ValueStateDescriptor<>("sum", Double.class)
        );
    }
    
    @Override
    public Double map(Transaction txn) throws Exception {
        Double currentSum = sumState.value();
        if (currentSum == null) {
            currentSum = 0.0;
        }
        
        Double newSum = currentSum + txn.getAmount();
        sumState.update(newSum);  // 更新状态
        
        return newSum;
    }
}
```

#### 3.2.4 Checkpoint（检查点）

**容错机制**：定期保存状态快照

```
时间轴：
  t0 ────── t1 ────── t2 ────── t3 ────── t4
         Checkpoint1  故障↓   恢复↑  Checkpoint2
         
状态快照：
  Checkpoint1: {user1: 100, user2: 200}
  
故障恢复：
  1. 从最近的Checkpoint1恢复状态
  2. 从Kafka重新消费数据（offset保存在Checkpoint中）
  3. 重新计算
```

**Checkpoint原理（Chandy-Lamport算法）**：
```
1. JobManager触发Checkpoint
   ↓
2. Source插入Barrier（栅栏）到数据流
   Source → [数据1] [Barrier-1] [数据2] [Barrier-2] ...
   ↓
3. Operator收到Barrier时：
   - 保存当前状态到StateBackend
   - 向下游转发Barrier
   ↓
4. Sink收到Barrier时：
   - 确认Checkpoint完成
   - 通知JobManager
   ↓
5. 所有Operator都确认后，Checkpoint完成
```

**代码配置**：
```java
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

// 启用Checkpoint
env.enableCheckpointing(60000);  // 每60秒一次

// Checkpoint配置
CheckpointConfig config = env.getCheckpointConfig();
config.setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);  // 精确一次
config.setMinPauseBetweenCheckpoints(30000);  // 两次Checkpoint间隔至少30秒
config.setCheckpointTimeout(600000);  // 超时时间10分钟
config.setMaxConcurrentCheckpoints(1);  // 最多1个Checkpoint并发

// StateBackend：状态存储后端
env.setStateBackend(new RocksDBStateBackend("hdfs://checkpoints"));
// 选项：
// - MemoryStateBackend：内存（小状态）
// - FsStateBackend：文件系统（中等状态）
// - RocksDBStateBackend：RocksDB（大状态）
```

#### 3.2.5 Exactly-Once语义

**问题**：如何保证消息不重不丢？

```
At-Most-Once（最多一次）：
  - 可能丢失数据
  - 发送后不等待确认

At-Least-Once（至少一次）：
  - 可能重复数据
  - 失败后重试

Exactly-Once（精确一次）：
  - 不重不丢
  - Flink通过Checkpoint + 两阶段提交实现
```

**Exactly-Once实现（Kafka Sink）**：
```java
// 两阶段提交协议
class KafkaSink {
    private KafkaProducer producer;
    private String transactionId;
    
    public void processElement(Record record) {
        // 1. 写入Kafka（事务中）
        producer.beginTransaction();
        producer.send(record);
        // 不提交，等待Checkpoint
    }
    
    public void snapshotState(CheckpointContext context) {
        // 2. Checkpoint时：预提交
        producer.flush();
        // 还未真正提交到Kafka
    }
    
    public void notifyCheckpointComplete(long checkpointId) {
        // 3. Checkpoint成功后：提交事务
        producer.commitTransaction();
        // 数据真正可见
    }
    
    public void notifyCheckpointAborted(long checkpointId) {
        // 4. Checkpoint失败：回滚事务
        producer.abortTransaction();
    }
}
```

```python
# Python版本（简化）
from collections import defaultdict
from datetime import datetime, timedelta

class AlertAggregator:
    def __init__(self, window_size=300):  # 5分钟
        self.window_size = window_size
        self.alerts = defaultdict(list)
    
    def add_alert(self, alert):
        key = (alert['service'], alert['metric'])
        self.alerts[key].append(alert)
        
        # 清理过期告警
        self.cleanup()
    
    def cleanup(self):
        now = datetime.now()
        for key in list(self.alerts.keys()):
            self.alerts[key] = [
                alert for alert in self.alerts[key]
                if (now - alert['timestamp']).seconds < self.window_size
            ]
            if not self.alerts[key]:
                del self.alerts[key]
    
    def get_aggregated_alerts(self):
        result = []
        for (service, metric), alerts in self.alerts.items():
            if len(alerts) > 0:
                result.append({
                    'service': service,
                    'metric': metric,
                    'count': len(alerts),
                    'first_occurrence': min(a['timestamp'] for a in alerts),
                    'last_occurrence': max(a['timestamp'] for a in alerts),
                    'sample_alerts': alerts[:3]  # 只保留3个样本
                })
        return result
```

```python
print(f"Predicted root cause: {predicted_cause[0]}")

def find_correlated_alerts(target_alert, all_alerts, time_window=300):
    """找出与目标告警时间接近的其他告警（可能有因果关系）"""
    target_time = target_alert['timestamp']

        if time_diff < time_window and alert != target_alert:

    return correlated

# 示例
```

## 7. 面试题自查

### 数据链路治理补充

- 大数据系统的关键不只是“算得动”，而是“链路稳定、口径一致、可回放、可追溯”。
- 设计实时链路时要先定好延迟目标、准确性目标和成本上限，再做引擎选型。
- 实时和离线并存时，优先定义统一口径和对账机制，避免同一个指标多份答案。

### 初级高频速刷

**1. 批处理和流处理的区别是什么？**

批处理面向有边界的数据集，追求吞吐；流处理面向持续到来的无界数据，追求低延迟和实时性。两者不是谁替代谁，而是场景不同。

**2. HDFS 的 Block 和副本机制分别在解决什么问题？**

Block 把大文件切分成分布式存储和并行处理的基本单位；副本机制保证数据可靠性和容错。理解这两点，才能解释 HDFS 为什么适合大数据场景。

**3. Map 和 Reduce 两个阶段分别做什么？**

Map 负责把输入数据转换为中间键值对，Reduce 负责按 key 聚合和汇总结果。真正的性能难点通常不在这两个词本身，而在中间的 Shuffle。

**4. Spark 和 Hadoop MapReduce 的定位有什么区别？**

MapReduce 是经典离线批处理模型，Spark 是更通用的分布式计算引擎，支持批处理、SQL、机器学习和流式处理。很多面试会先从这个基础问题切入，再追问为什么 Spark 更快。

**5. Checkpoint 和 Savepoint 的区别是什么？**

Checkpoint 更多用于故障恢复，通常由系统周期性自动生成；Savepoint 更偏人工控制，常用于作业迁移、版本升级和状态保留。两者都保存状态，但生命周期和用途不同。

### 中级高频进阶

**6. 为什么很多日志分析系统会同时保留明细表和聚合表？**

因为明细表适合做追溯、抽样排查和灵活分析，聚合表适合支撑高频 dashboard 和告警查询。只留明细会让查询太慢，只留聚合又会丢掉排查细节。

**7. 宽表为什么常见于数仓和实时分析系统？**

因为它把常用维度和指标预先组织到一起，减少查询时的 join 和数据搬运成本。在分析场景里，空间冗余换查询效率通常是值得的。

**8. 为什么 Flink 这类流式系统必须强调状态管理？**

因为窗口聚合、去重、会话分析、Exactly-Once 这些能力本质上都依赖“记住过去发生了什么”。没有状态管理，流处理就只能做简单的无状态变换，价值会大幅下降。

**9. 为什么大数据平台必须明确“事件时间”和“处理时间”的边界？**

因为事件延迟、乱序、补数据是常态。若口径混用，实时看板和离线报表会长期不一致，团队会在“到底哪个数是真的”上反复争论，排障和业务决策都会失真。

**10. 为什么数仓里经常强调幂等写入和可回放能力？**

因为链路故障、任务重跑、下游修复都可能触发重复写入或补写历史。没有幂等和回放机制，数据修复只能靠人工脚本，风险高且不可审计。

### 深入问答

#### Q1: MapReduce的Shuffle过程是怎样的？为什么Shuffle是性能瓶颈？

**答案**：
**Shuffle过程**：
1. **Map端**：输出写入环形缓冲区 → 分区 → 排序 → 溢写到磁盘 → 合并
2. **Copy阶段**：Reducer从所有Mapper拉取属于自己分区的数据
3. **Sort阶段**：归并排序所有拉取的数据
4. **Group阶段**：相同key的value分组

**性能瓶颈原因**：
- 大量磁盘IO（Map溢写、Reduce拉取）
- 网络传输（跨节点数据传输）
- 排序开销

**优化**：Combiner预聚合、压缩传输、调大缓冲区

---

#### Q2: Spark为什么比MapReduce快？

**答案**：
1. **内存计算**：中间结果存内存，MapReduce每次写HDFS
2. **DAG调度**：宽依赖才切Stage，窄依赖流水线执行
3. **RDD惰性求值**：只有Action才触发计算，优化执行计划
4. **缓存机制**：重复使用的RDD可以cache到内存

**量化对比**：迭代计算（如PageRank），Spark比MapReduce快10-100倍

---

#### Q3: 什么是RDD的宽依赖和窄依赖？为什么要区分？

**答案**：
**窄依赖**：父RDD的一个分区 → 子RDD的一个分区（map、filter）
**宽依赖**：父RDD的多个分区 → 子RDD的一个分区（reduceByKey、groupByKey）

**区分原因**：
1. **Stage划分**：宽依赖是Stage边界，需要Shuffle
2. **容错差异**：窄依赖只需重算一个分区，宽依赖需要重算所有父分区
3. **Pipeline执行**：窄依赖可以流水线执行，无需等待

---

#### Q4: 如何处理大数据中的数据倾斜问题？

**答案**：
**问题**：某个key的数据量特别大，单个Reducer处理时间过长

**解决方案**：
1. **加盐（Salting）**：给热key添加随机后缀，打散到多个Reducer
2. **两阶段聚合**：先局部聚合，再全局聚合
3. **广播Join**：小表广播到所有节点，避免Shuffle
4. **过滤倾斜key**：单独处理热key

**代码示例（加盐）**：
```python
# 原始key="hot_user"
# 加盐后="hot_user_0", "hot_user_1", ..., "hot_user_9"
salted = rdd.map(lambda (k,v): (f"{k}_{random.randint(0,9)}", v))
partial = salted.reduceByKey(add)
final = partial.map(lambda (k,v): (k.rsplit('_',1)[0], v)).reduceByKey(add)
```

#### Q5: Flink的Watermark是什么？如何处理乱序数据？

**答案**：
**Watermark定义**：表示"时间t之前的数据都已到达"的标记

**处理乱序**：

- 迟到数据可以丢弃或发送到侧输出流

**示例**：允许5秒乱序

#### Q6: Flink如何实现Exactly-Once语义？

**答案**：
**实现机制**：Checkpoint + 两阶段提交

**流程**：
1. Source插入Barrier到数据流
2. Operator收到Barrier时保存状态
3. Sink预提交数据（不可见）

5. Sink提交事务，数据可见

**关键点**：
- Barrier对齐：多输入时等待所有输入的Barrier
- 状态后端：RocksDB支持大状态

#### Q16: Spark 的 Shuffle 和 Q1 的 MapReduce Shuffle 有什么实战差异？

```java
#### Q21: 实时数仓为什么要分 ODS / DWD / DWS / ADS？

**答案**：
分层的价值在于“口径沉淀和复用”，而不是为了层数好看。
1. ODS 保留原始数据，便于回放和追溯。
2. DWD 做明细标准化，统一字段和事件语义。
3. DWS 做主题聚合，服务高频分析。
4. ADS 面向业务指标和看板，直接给运营和管理层使用。

不分层的常见后果是：指标定义分散、重复开发严重、问题定位链路断裂。

#### Q22: 如何保证实时指标和离线指标口径一致？

**答案**：
可以用“三统一”原则：
1. 统一口径定义：指标计算逻辑写成可复用规范或 UDF/UDAF。
2. 统一维表和主数据：避免实时/离线使用不同维度映射。
3. 统一对账流程：按天/小时做差异校验，设定可接受误差阈值。

工程上建议把“口径差异率”纳入平台指标，避免长期依赖人工发现。

#### Q23: 数据回填（Backfill）怎么做才不会冲垮实时链路？

**答案**：
回填的核心是“限速、隔离、幂等”。
1. 限速：限制回填任务并发与吞吐，避免抢占在线资源。
2. 隔离：回填作业使用独立队列或资源池，和实时任务分开。
3. 幂等：确保重放同一批数据不会重复累计。

同时要明确回填窗口、影响范围和回滚方案，避免一次回填引发全链路重算。

#### Q24: 什么情况下应该优先选择 Iceberg/Hudi/Delta 这类湖仓格式？

**答案**：
当你需要“对象存储上的可更新大表 + ACID + 增量读取 + 时间旅行”时，湖仓格式通常比纯文件分区更合适。

典型场景：

2. 流批共用同一份数据资产，要求统一元数据和快照管理。
3. 需要按快照回溯历史口径，用于审计和问题追查。

#### Q25: 大数据实时链路里，迟到数据占比突然上升，你先改 watermark 还是先查上游？

**答案**：
先查上游，再决定是否临时调整 watermark。因为迟到占比突升往往是链路异常信号（采集延迟、消息堆积、网络抖动、上游批量补发），盲目放宽 watermark 可能掩盖真实故障并拉高状态成本。

建议顺序：
1. 查上游生产延迟和 Kafka lag。
2. 查链路拥塞点（反压、checkpoint 变慢、下游写入抖动）。

#### Q26: 实时和离线口径长期偏差 1%-2%，你会如何建立“自动对账 + 追责闭环”？

**答案**：
长期小偏差通常是流程问题而不是一次性 bug。闭环可按四层建立：
1. **自动对账**：按天/小时对核心指标做差异计算并分维度下钻。
2. **阈值分级**：定义可接受偏差区间，超阈值自动告警与阻断下游发布。
3. **责任归属**：差异单自动挂到具体链路 owner（采集、加工、维表、口径）。
4. **修复验收**：修复后连续观察周期内差异回归，并纳入复盘指标。

关键是把”偏差处理”产品化，而不是每次靠人工拉群比 SQL。
```

### 开放式设计题

**D1：设计一个实时数据分析平台，支持秒级延迟的多维度数据查询，你会怎么架构？**

**参考思路**：
- 数据采集：应用埋点→Kafka→Flink实时ETL→ClickHouse/Druid
- 存储层：预聚合物化视图（分钟级/小时级/天级）、明细数据保留7天供下钻
- 查询层：ClickHouse做OLAP查询（秒级）、Redis缓存热门Dashboard结果
- 数据建模：星型模型（事实表+维度表）、预计算Cube减少查询时聚合
- 关键指标：查询P99<3s、数据延迟<30s、日处理量>100亿条

**D2：Flink作业频繁反压（Backpressure），如何排查和优化？**

**参考思路**：
- 定位瓶颈算子：Flink Web UI → Backpressure tab → 找到反压源头算子
- 常见原因：外部IO慢（DB/ES写入）、数据倾斜（某个Key的数据量远超其他）、状态过大（RocksDB慢）
- 优化：异步IO(AsyncFunction)、数据打散（加盐Key）、增大并行度、状态TTL清理、Checkpoint间隔调大

## 总结

**大数据技术栈**：

1. **存储**：HDFS、ClickHouse、Elasticsearch、Parquet/ORC
2. **批处理**：MapReduce原理、Spark（RDD/DataFrame/SQL）、Hive
3. **流处理**：Flink（Watermark/窗口/状态/Checkpoint）、Kafka Streams
4. **消息队列**：Kafka
5. **可视化**：Kibana、Grafana

**核心原理**：
- MapReduce Shuffle机制
- Spark DAG调度与内存计算
- Flink Exactly-Once与状态管理
- 数据倾斜处理

**典型场景**：
- **日志分析**：ELK、Spark批处理
- **实时监控**：Flink + ClickHouse
- **告警分析**：去重聚合、根因分析
