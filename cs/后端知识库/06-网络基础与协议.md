## 2. TCP协议详解

### 2.1 TCP vs UDP

| 特性 | TCP | UDP |
|------|-----|-----|
| **连接** | 面向连接（三次握手） | 无连接 |
| **可靠性** | 可靠（确认、重传） | 不可靠 |
| **顺序** | 保证顺序 | 不保证顺序 |
| **速度** | 慢 | 快 |
| **头部开销** | 20字节 | 8字节 |
| **应用场景** | HTTP、FTP、邮件 | DNS、视频、**游戏** |

### 2.2 TCP三次握手

```
客户端                     服务器
  |                          |
  |  ①SYN seq=x             |
  |------------------------->|
  |                          |
  |  ②SYN-ACK seq=y ack=x+1 |
  |<-------------------------|
  |                          |
  |  ③ACK seq=x+1 ack=y+1   |
  |------------------------->|
  |                          |
  |   [连接建立]              |
```

**为什么需要三次握手？**
- **两次不够**：服务器无法确认客户端能收到SYN-ACK
- **四次太多**：第②③次可以合并（SYN-ACK）

**状态变化**：
```
客户端：CLOSED → SYN_SENT → ESTABLISHED
服务器：CLOSED → LISTEN → SYN_RECV → ESTABLISHED
```

### 2.2.1 三次握手内核实现要点

> 以下内容基于Linux 5.x内核源码分析

**为什么必须三次？** 核心目的：①确认对端在线能即时响应（面向连接）；②双方确认数据传输的起始序列号（双向的seq确认需要三次交互）。两次握手无法让服务端确认客户端收到了自己的序列号。

**SYN Flood攻击与syncookie**：

```bash
# 半连接队列上限
cat /proc/sys/net/ipv4/tcp_max_syn_backlog  # 默认1024-2048

# syncookie开关（防SYN Flood的核心机制）
cat /proc/sys/net/ipv4/tcp_syncookies
# 0=关闭, 1=默认(半连接队列耗尽后启用), 2=忽略队列直接使用cookie
```

**syncookie原理**：服务端不存储SYN信息，而是将四元组+时间戳做hash运算生成cookie放入SYN-ACK的seq中返回给客户端，客户端在第三次握手ACK中带回cookie（seq+1），服务端验证通过后建立连接。将内存攻击转化为CPU运算，大幅削弱SYN Flood效果。

**listen backlog — 全连接队列**：

```bash
# listen(fd, backlog) 的backlog限制的是已完成三次握手但未被accept的连接队列长度
# 实际生效值 = min(backlog, somaxconn)
cat /proc/sys/net/core/somaxconn  # 默认128

# 查看全连接队列使用情况
ss -lnt
# LISTEN  Recv-Q  Send-Q  Local:Port
# LISTEN  3       128     0.0.0.0:80
#         ↑当前排队数    ↑最大长度

# 队列溢出时：新连接SYN无回应，客户端重试直到超时
# 溢出计数器
netstat -s | grep "listen queue"  # 或 nstat | grep ListenOverflows
```

**SYN/SYN-ACK重试次数**：

```bash
cat /proc/sys/net/ipv4/tcp_syn_retries      # 默认6（最长等待63秒，2^n指数退避）
cat /proc/sys/net/ipv4/tcp_synack_retries    # 默认5（最长等待31秒）
```

**TFO（TCP Fast Open）**：允许在SYN包中携带应用数据，减少一次RTT。

```bash
# 开启TFO
echo 3 > /proc/sys/net/ipv4/tcp_fastopen
# 0=关闭, 1=客户端, 2=服务端, 3=双向

# 工作流程：
# 首次连接：客户端SYN带空cookie请求 → 服务端返回cookie
# 后续连接：客户端SYN+cookie+数据 → 服务端验证cookie后直接处理数据（省1个RTT）

# nginx开启TFO
# listen 80 fastopen=128;
```

### 2.3 TCP四次挥手

```
客户端                     服务器
  |                          |
  |  ①FIN seq=u             |
  |------------------------->|  [客户端主动关闭]
  |                          |
  |  ②ACK seq=v ack=u+1     |
  |<-------------------------|  [服务器确认]
  |                          |
  |  ③FIN seq=w ack=u+1     |
  |<-------------------------|  [服务器关闭]
  |                          |
  |  ④ACK seq=u+1 ack=w+1   |
  |------------------------->|  [客户端确认]
  |                          |
  |   [连接关闭]              |
```

**TIME_WAIT状态**：
- 主动关闭方等待2MSL（Maximum Segment Lifetime，通常60秒）
- 原因：确保最后的ACK到达，防止旧连接数据干扰新连接

---

#### 为什么TIME_WAIT是2MSL？底层原理深度剖析⭐⭐⭐

**为什么需要了解TIME_WAIT？**
- 面试高频：为什么是2MSL而不是1MSL或3MSL？
- 性能调优：TIME_WAIT过多导致端口耗尽
- 故障排查：大量TIME_WAIT的原因和解决方案

---

**什么是MSL？**

```
MSL（Maximum Segment Lifetime）：报文段最大生存时间
- Linux默认：60秒
- Windows默认：120秒
- RFC标准建议：2分钟

含义：IP数据包在网络中存活的最长时间
- 超过MSL，路由器丢弃数据包（TTL=0）
```

**为什么需要TIME_WAIT？（2个原因）**

**原因1：确保最后的ACK到达对方**

```
四次挥手过程：
客户端                              服务器
  |                                    |
  | ①FIN(seq=u)                        |
  |─────────────────────────────────→  |
  |                                    |
  | ②ACK(ack=u+1)                      |
  |←─────────────────────────────────  |
  |                                    |
  | ③FIN(seq=v)                        |
  |←─────────────────────────────────  |
  |                                    |
  | ④ACK(ack=v+1)                      |
  |─────────────────────────────────→  |
  |                                    |
  | [TIME_WAIT: 2MSL]                  |
  |                                    |

问题场景：
- 客户端发送最后的ACK（④）
- ACK在网络中丢失
- 服务器没收到ACK，超时重传FIN（③）
- 如果客户端立即关闭，无法响应重传的FIN
- 服务器会一直重传FIN，直到超时（可能等待数分钟）

TIME_WAIT的作用：
- 客户端等待2MSL期间，可以接收并响应服务器重传的FIN
- 2MSL = 1MSL（FIN重传到达）+ 1MSL（ACK返回）
```

**原因2：防止旧连接数据干扰新连接**

```
场景：快速复用四元组（相同的src_ip:port → dst_ip:port）

时刻0：连接A关闭
      客户端:1234 → 服务器:80
      最后发送的数据包：seq=1000

时刻1：立即创建连接B（相同四元组）
      客户端:1234 → 服务器:80
      新连接的seq从1开始

时刻2：旧连接A的数据包（seq=1000）延迟到达
      问题：服务器如何区分这是旧连接A的数据，还是新连接B的数据？
      → 可能导致数据混乱

TIME_WAIT的作用：
- 等待2MSL，确保旧连接的所有数据包都已消失（TTL超时）
- 2MSL = 1MSL（旧数据发送）+ 1MSL（旧数据返回）
- 新连接不会收到旧连接的数据
```

**为什么是2MSL，而不是1MSL或3MSL？**

```
1MSL不够：
- 无法覆盖"发送FIN重传 + 接收ACK"的往返时间
- 旧连接的返回数据可能干扰新连接

3MSL太长：
- 浪费资源（端口、内存）
- 不必要（2MSL已足够覆盖往返）

2MSL刚好：
- 去程1MSL：对方重传的FIN到达（或旧数据到达）
- 返程1MSL：我方的ACK返回（或旧数据返回）
- 覆盖所有可能的网络延迟情况
```

---

**TIME_WAIT过多的问题**：

**问题1：端口耗尽**

```
客户端短连接场景：
- 客户端发起100,000次HTTP请求
- 每次请求使用一个临时端口（1024-65535，约64000个）
- 每个连接关闭后，端口进入TIME_WAIT（60秒）
- 60秒内发起64000次请求 → 端口耗尽

错误信息：
connect: Cannot assign requested address
```

**问题2：内存占用**

```
每个TIME_WAIT连接占用约4KB内存：
- TCP控制块（TCB）
- 套接字缓冲区

100,000个TIME_WAIT连接 = 400MB内存
```

**解决方案**：

```bash
# 方案1：启用TIME_WAIT端口复用（推荐）⭐⭐⭐
sysctl -w net.ipv4.tcp_tw_reuse=1
# 允许将TIME_WAIT端口分配给新的出站连接
# 注意：只对客户端有效（出站连接）

# 方案2：缩短FIN_WAIT_2超时
sysctl -w net.ipv4.tcp_fin_timeout=30
# 默认60秒 → 30秒

# 方案3：启用TIME_WAIT快速回收（危险，不推荐）
sysctl -w net.ipv4.tcp_tw_recycle=1
# 已在Linux 4.12+内核中移除（不安全）
# 原因：NAT环境下可能拒绝合法连接

# 方案4：增大端口范围
sysctl -w net.ipv4.ip_local_port_range="1024 65535"
# 增加可用临时端口数量

# 方案5：使用连接池（最佳方案）⭐⭐⭐
# HTTP连接池（复用连接，避免频繁创建/销毁）
http.Transport{
    MaxIdleConns:        100,
    MaxIdleConnsPerHost: 10,
    IdleConnTimeout:     90 * time.Second,
}
```

**为什么tcp_tw_reuse安全，tcp_tw_recycle不安全？**

```
tcp_tw_reuse（安全）：
- 只复用客户端的TIME_WAIT端口
- 检查新连接的时间戳（timestamp）
- 如果新连接timestamp > 旧连接timestamp，允许复用
- 前提：双方都启用TCP timestamp（默认启用）

tcp_tw_recycle（不安全）：
- 快速回收TIME_WAIT连接（< 2MSL）
- 依赖对方的timestamp递增
- NAT环境问题：
  - NAT后多个客户端共享同一个公网IP
  - 客户端A的timestamp=100，客户端B的timestamp=50
  - 服务器认为B的连接是旧连接（timestamp倒退），拒绝
  → 导致合法连接被拒绝

Linux 4.12+已移除tcp_tw_recycle
```

**TIME_WAIT过多的根因分析**：

```bash
# 查看TIME_WAIT数量
ss -s | grep TIME-WAIT
# 或
netstat -an | grep TIME_WAIT | wc -l

# 查看哪个进程产生TIME_WAIT（主动关闭方）
ss -tapn | grep TIME-WAIT

# 常见原因
1. 短连接频繁（HTTP/1.0无Keep-Alive）
   → 解决：使用HTTP/1.1 Keep-Alive或HTTP/2
   
2. 应用主动关闭连接（客户端）
   → 解决：让服务器主动关闭（TIME_WAIT在服务器）
   
3. 负载均衡健康检查
   → 解决：增大检查间隔、使用UDP健康检查
```

**最佳实践**：

```
1. 客户端（高并发爬虫、压测工具）
   ✅ 启用tcp_tw_reuse
   ✅ 使用连接池
   ✅ 减少短连接（使用Keep-Alive）

2. 服务器（Web服务器、API网关）
   ✅ 增大文件描述符限制（ulimit -n 100000）
   ✅ 增大端口范围
   ✅ 让客户端主动关闭连接（TIME_WAIT在客户端）
   ❌ 不要启用tcp_tw_recycle（已移除）

3. 监控TIME_WAIT数量
   ✅ 阈值告警（> 10000）
   ✅ 定期检查（Prometheus + Grafana）
```

### 2.3.1 四次挥手内核实现要点

**Linux上实际常为三次挥手**：由于Linux的延迟ACK机制，被动关闭方收到FIN后不会立即发送ACK，而是等待延迟发送。如果在延迟期间应用层也确认关闭连接，FIN和ACK会合并为一个包发出，四次挥手变为三次。

**TIME_WAIT内核实现**：

```bash
# Linux固定等待60秒（非RFC的2×MSL=4分钟）
# 内核代码：#define TCP_TIMEWAIT_LEN (60*HZ)

# 限制TIME_WAIT总数（防止内存耗尽）
cat /proc/sys/net/ipv4/tcp_max_tw_buckets  # 超限后直接RST关闭

# TIME_WAIT每个连接占用约4KB slab内存
# 大量TIME_WAIT还会增加CPU消耗（定时器处理、哈希查找）
```

**各关闭状态驻留问题**：

| 状态 | 驻留原因 | 超时机制 | 调优手段 |
|------|----------|----------|----------|
| **FIN_WAIT1** | 对端未回ACK | tcp重传定时器，受tcp_orphan_retries限制（默认0→实际8次） | 减小tcp_orphan_retries |
| **FIN_WAIT2** | 对端未发FIN | tcp_fin_timeout（默认60秒） | 减小tcp_fin_timeout |
| **CLOSE_WAIT** | **本端应用层未close** | 无内核超时，完全依赖应用 | 修复应用代码BUG |
| **LAST_ACK** | 对端未回最后ACK | tcp重传定时器 | 同FIN_WAIT1 |
| **TIME_WAIT** | 等待旧包消失 | 固定60秒 | tcp_max_tw_buckets / tcp_tw_reuse |

```bash
# tcp orphan（已脱离应用的tcp连接）数量限制
cat /proc/sys/net/ipv4/tcp_max_orphans  # 超限后RST关闭

# FIN重传次数（同时影响FIN_WAIT1和LAST_ACK）
cat /proc/sys/net/ipv4/tcp_orphan_retries  # 默认0=内核实际用8次
```

**SO_LINGER选项**：对socket设置SO_LINGER后，close时直接发送RST关闭连接，跳过所有状态直接进入CLOSED。激进方案，慎用。

### 2.4 TCP滑动窗口与流量控制⭐⭐⭐

> 滑动窗口是TCP实现可靠传输和流量控制的核心机制，也是理解拥塞控制的前提。

#### 为什么需要滑动窗口？

```
没有窗口（停等协议）：
发送方发1个包 → 等ACK → 发下一个
效率极低：每个RTT只发一个包

有窗口（流水线发送）：
发送方连续发N个包（N=窗口大小）→ 收到ACK后窗口向前滑动
效率高：一个RTT内发N个包
```

#### 发送窗口与接收窗口

```
发送方维护的窗口：
┌───────────────────────────────────────────────────────┐
│  已发送已确认  │  已发送未确认  │  可以发送  │  不可发送  │
│  (可以释放)    │  (在窗口内)    │ (在窗口内)  │ (窗口外)  │
└───────────────────────────────────────────────────────┘
                  ← ────── 发送窗口 ────── →

发送窗口大小 = min(cwnd, rwnd)
- cwnd（拥塞窗口）：由拥塞控制算法决定，反映网络承受能力
- rwnd（接收窗口）：接收方在TCP报头中通告，反映接收方缓冲区剩余空间

接收方维护的窗口：
┌───────────────────────────────────────────────────────┐
│  已接收已确认  │  等待接收（接收窗口）  │  不可接收    │
└───────────────────────────────────────────────────────┘
                  ← ── rwnd（通告给发送方）── →
```

#### 滑动窗口运作过程

```
初始：rwnd = 4000字节，MSS = 1000字节

时刻1：发送方发送seq=1000~4000（4个MSS）
  已发送未确认：[1000, 2000, 3000, 4000]
  可以发送：无（窗口满）

时刻2：接收方确认ACK=3000（前2个包已收到）
  接收方rwnd更新：rwnd = 4000（缓冲区已被应用层读走）
  → 发送方窗口右移：释放[1000, 2000]，新增可发送空间
  已发送未确认：[3000, 4000]
  可以发送：[5000, 6000]

时刻3：接收方缓冲区满（应用层来不及读取）
  接收方通告：rwnd = 0
  → 发送方停止发送，等待窗口更新
```

#### 零窗口与窗口探测

```
问题场景：
1. 接收方缓冲区满 → 通告 rwnd = 0
2. 发送方停止发送
3. 接收方缓冲区释放 → 通告 rwnd > 0
4. 但这个窗口更新ACK丢失了！
5. 发送方永远不知道可以发送 → 死锁

解决方案：零窗口探测（Zero Window Probe）
- 发送方收到rwnd=0后，启动持续定时器（Persist Timer）
- 定时发送1字节的探测报文（ZWP, Zero Window Probe）
- 接收方收到探测后回复当前rwnd
- 探测间隔：指数退避（类似TCP重传），最大60秒
```

```bash
# Linux中零窗口探测相关参数
# 探测次数上限（默认15次，超过后RST关闭连接）
cat /proc/sys/net/ipv4/tcp_retries2
```

#### 流量控制 vs 拥塞控制

| 维度 | 流量控制 | 拥塞控制 |
|------|----------|----------|
| **解决什么问题** | 发送方不要淹没接收方 | 发送方不要淹没网络 |
| **控制变量** | rwnd（接收方通告窗口） | cwnd（拥塞窗口） |
| **谁决定** | 接收方（根据缓冲区剩余空间） | 发送方（根据网络反馈） |
| **反馈机制** | TCP报头的Window字段 | 丢包（超时/重复ACK） |
| **协作关系** | 实际发送窗口 = min(cwnd, rwnd) | 两者共同约束发送速率 |

```
发送速率的双重约束：

接收方缓冲区充足，但网络拥塞：
  rwnd = 100KB, cwnd = 10KB → 发送窗口 = 10KB（受拥塞控制限制）

网络畅通，但接收方处理慢：
  rwnd = 5KB, cwnd = 100KB → 发送窗口 = 5KB（受流量控制限制）
```

#### Nagle算法与延迟ACK的交互

```
Nagle算法（发送方）：
- 目的：减少小包数量（避免大量1字节TCP段）
- 规则：如果有已发送未确认的数据，新数据积累到MSS后再发送
- 如果没有未确认数据，立即发送（即使很小）

延迟ACK（接收方）：
- 目的：减少ACK数量（攒一批数据一起确认）
- 规则：收到数据后不立即回ACK，等待最多200ms（或收到第二个包后立即ACK）

两者叠加可能的问题：
- 发送方发了一个小包，等ACK才发下一个（Nagle）
- 接收方等200ms才回ACK（延迟ACK）
- → 每个小包都有200ms额外延迟，交互式应用（SSH/游戏）卡顿

解决：
setsockopt(fd, IPPROTO_TCP, TCP_NODELAY, &on, sizeof(on));
// TCP_NODELAY=1 关闭Nagle算法，小包立即发送
// 适用场景：游戏、SSH、实时交互
```

### 2.5 TCP拥塞控制

**四大算法**：

**1. 慢启动（Slow Start）**
- 初始窗口小（cwnd=1 MSS）
- 每收到一个ACK，cwnd翻倍
- 指数增长至ssthresh（慢启动阈值）

**2. 拥塞避免（Congestion Avoidance）**
- cwnd > ssthresh后，每收到一个ACK，cwnd+1
- 线性增长

**3. 快速重传（Fast Retransmit）**
- 收到3个重复ACK，立即重传（不等超时）

**4. 快速恢复（Fast Recovery）**
- 快速重传后，ssthresh = cwnd/2，cwnd = ssthresh
- 进入拥塞避免

---

#### TCP拥塞控制底层原理深度剖析⭐⭐⭐

**为什么需要拥塞控制？**
- 面试高频：慢启动为什么是指数增长？拥塞避免为什么是线性增长？
- 性能调优：如何避免慢启动影响性能？
- 原理理解：TCP如何平衡速度和稳定性？

---

**核心概念**：

```
cwnd（Congestion Window，拥塞窗口）：
- 发送方维护的窗口大小
- 表示"我认为网络能承受多少数据"

ssthresh（Slow Start Threshold，慢启动阈值）：
- 初始值：通常很大（65535字节）
- 作用：区分慢启动和拥塞避免的分界线

实际发送窗口 = min(cwnd, rwnd)
- rwnd：接收方通告窗口（流量控制）
- cwnd：拥塞窗口（拥塞控制）
```

---

##### 1. 慢启动（Slow Start）详解⭐⭐⭐

**为什么叫"慢启动"？**

```
相对于"直接发送最大窗口"，慢启动是"慢"的：
- 不慢启动：初始cwnd=65535字节，可能导致网络拥塞
- 慢启动：初始cwnd=1 MSS（约1460字节），逐步探测网络容量
```

**慢启动的指数增长过程**：

```
初始：cwnd = 1 MSS，ssthresh = 65535字节

第1轮RTT：
发送：1个MSS（1460字节）
收到ACK：cwnd = 2 MSS

第2轮RTT：
发送：2个MSS
收到2个ACK：cwnd = 2 + 2 = 4 MSS

第3轮RTT：
发送：4个MSS
收到4个ACK：cwnd = 4 + 4 = 8 MSS

第4轮RTT：
发送：8个MSS
收到8个ACK：cwnd = 8 + 8 = 16 MSS

每个RTT，cwnd翻倍（指数增长）
1 → 2 → 4 → 8 → 16 → 32 → 64 → ...
```

**为什么是指数增长？**

```
目标：快速探测网络的可用带宽

线性增长（+1）太慢：
1 → 2 → 3 → 4 → 5 → ... → 100（需要100个RTT）

指数增长（×2）很快：
1 → 2 → 4 → 8 → 16 → 32 → 64 → 128（只需7个RTT）

优势：
- 快速达到合理的发送速率
- 避免长时间低效传输
```

**慢启动何时结束？**

```
条件1：cwnd >= ssthresh
→ 转入拥塞避免阶段

条件2：发生丢包（超时或3个重复ACK）
→ 重新设置ssthresh，重新慢启动
```

---

##### 2. 拥塞避免（Congestion Avoidance）详解⭐⭐⭐

**为什么需要拥塞避免？**

```
慢启动的指数增长太快：
- 接近网络容量时，再翻倍可能导致拥塞
- 需要更谨慎地探测网络容量
```

**拥塞避免的线性增长过程**：

```
进入条件：cwnd >= ssthresh

示例：cwnd = 16 MSS，ssthresh = 16 MSS

第1轮RTT：
发送：16个MSS
收到16个ACK：每个ACK增加 1/16 MSS
cwnd = 16 + 16×(1/16) = 17 MSS

第2轮RTT：
发送：17个MSS
收到17个ACK：每个ACK增加 1/17 MSS
cwnd = 17 + 17×(1/17) = 18 MSS

每个RTT，cwnd只增加1 MSS（线性增长）
16 → 17 → 18 → 19 → 20 → ...
```

**为什么是线性增长？**

```
目标：谨慎探测网络容量，避免拥塞

指数增长（×2）太激进：
16 → 32 → 64（可能立即拥塞）

线性增长（+1）刚好：
16 → 17 → 18（缓慢接近容量上限）

优势：
- 避免突然的流量激增
- 网络拥塞时能快速察觉（丢包）
```

---

##### 3. 快速重传（Fast Retransmit）详解⭐⭐⭐

**为什么需要快速重传？**

```
传统重传：依赖超时（RTO，通常1秒）
问题：超时时间太长，影响性能

快速重传：收到3个重复ACK，立即重传
优势：不等超时，立即恢复
```

**快速重传的触发过程**：

```
发送方：
发送seq=1000, 1500, 2000, 2500, 3000

接收方：
收到seq=1000 → 返回ACK=1500（期待1500）
收到seq=2000 → seq=1500丢失，返回ACK=1500（重复1）
收到seq=2500 → seq=1500仍丢失，返回ACK=1500（重复2）
收到seq=3000 → seq=1500仍丢失，返回ACK=1500（重复3）

发送方：
收到3个重复ACK=1500 → 立即重传seq=1500
（不等超时）

为什么是3个重复ACK？
- 1-2个重复ACK：可能是乱序（不是丢包）
- 3个重复ACK：大概率是丢包
- 权衡：误判率 vs 恢复速度
```

---

##### 4. 快速恢复（Fast Recovery）详解⭐⭐⭐

**快速恢复的核心思想**：

```
传统方式（Tahoe算法）：
丢包 → ssthresh = cwnd/2，cwnd = 1
→ 重新慢启动（太慢）

快速恢复（Reno算法）：
丢包 → ssthresh = cwnd/2，cwnd = ssthresh
→ 直接进入拥塞避免（更快）

为什么更快？
- 收到3个重复ACK，说明网络仍有传输能力（不是完全拥塞）
- 没必要降到cwnd=1，直接减半即可
```

**快速恢复的详细过程**：

```
初始：cwnd = 20 MSS，ssthresh = 40 MSS

步骤1：收到3个重复ACK
→ ssthresh = cwnd / 2 = 10 MSS
→ cwnd = ssthresh = 10 MSS
→ 重传丢失的数据包

步骤2：继续收到重复ACK（表示后续包已到达）
→ cwnd = cwnd + 1（临时膨胀，允许发送新数据）

步骤3：收到新的ACK（确认重传的包）
→ cwnd = ssthresh = 10 MSS
→ 进入拥塞避免阶段
```

---

##### 5. 完整的拥塞控制状态机⭐⭐⭐

**完整流程示例**：

```
时刻0：连接建立
cwnd = 1 MSS，ssthresh = 65535字节

阶段1：慢启动（指数增长）
RTT 1: cwnd = 1 → 2
RTT 2: cwnd = 2 → 4
RTT 3: cwnd = 4 → 8
RTT 4: cwnd = 8 → 16（达到ssthresh=16）

阶段2：拥塞避免（线性增长）
RTT 5: cwnd = 16 → 17
RTT 6: cwnd = 17 → 18
RTT 7: cwnd = 18 → 19
RTT 8: cwnd = 19 → 20

阶段3：丢包（3个重复ACK）
cwnd = 20，ssthresh = 20
→ ssthresh = 20/2 = 10
→ cwnd = 10（快速恢复）

阶段4：拥塞避免（线性增长）
RTT 9: cwnd = 10 → 11
RTT 10: cwnd = 11 → 12
...

阶段5：超时（网络严重拥塞）
cwnd = 15，ssthresh = 10
→ ssthresh = 15/2 = 7
→ cwnd = 1（重新慢启动）

阶段6：慢启动（指数增长）
RTT 11: cwnd = 1 → 2
RTT 12: cwnd = 2 → 4
RTT 13: cwnd = 4 → 8（达到ssthresh=7）

阶段7：拥塞避免...
```

**状态转换图**：

```
慢启动 ─────────────→ 拥塞避免
  ↑        cwnd>=ssthresh    │
  │                          │
  │                          │ 3个重复ACK
  │                          ↓
  └────────超时────────── 快速恢复
       (cwnd=1)           (cwnd=ssthresh)
```

##### 6. 拥塞控制算法演进⭐⭐

**Tahoe（1988）**：
```
丢包 → cwnd = 1，重新慢启动
缺点：太保守，性能差
```

**Reno（1990）**：
```
3个重复ACK → 快速恢复（cwnd = ssthresh）
超时 → cwnd = 1，慢启动
改进：减少慢启动次数
```

**NewReno（1999）**：
```
改进快速恢复：处理多个丢包
当前主流算法
```

**CUBIC（2008，Linux默认）**：
```
目标：高带宽网络（如10Gbps）
特点：窗口增长函数是三次函数（不是线性）
优势：更快达到高带宽
```

**BBR（2016，Google）**：
```
革命性：不基于丢包，而是基于带宽和RTT
目标：最大化带宽利用率，最小化延迟
YouTube、Google Cloud使用
```

**算法选择指南**：

**慢启动优化**：

```bash
# 增大初始拥塞窗口（Linux 3.0+默认10 MSS）

# 禁用慢启动重启（长时间空闲后不重新慢启动）
sysctl -w net.ipv4.tcp_slow_start_after_idle=0
```

##### 8. 慢启动内核实现要点⭐⭐

**初始拥塞窗口（initcwnd）**：

内核中 `tcp_init_cwnd()` 决定初始cwnd，默认 `TCP_INIT_CWND = 10`（即10个MSS ≈ 14480字节）。这就是为什么新连接发送约14KB后就会暂停等待ACK。

```bash
# 通过路由表修改initcwnd（不需要改内核代码）
ip route change 10.0.0.0/24 dev eth0 initcwnd 20

# 实际发送窗口 = min(cwnd, 接收方通告窗口rwnd)
# 即使cwnd调到100，也受限于对端第一个SYN中通告的窗口（通常64240字节）
```

**慢启动对中等数据的性能影响**（实验数据）：

| 网络延迟(RTT) | 512KB数据传输耗时 | 慢启动中断次数 |
|---------------|-------------------|----------------|
| ~0.3ms | 8ms | 6次（几乎无感知） |
| ~3ms | 44ms | 6次 |
| ~7ms | 63ms | 6次（延时影响显著） |

**关键结论**：中等大小数据（几百KB）在RTT稍有增加时，传输延时会放大数倍。原因是每次窗口等待都要消耗一个RTT，6次中断×7ms RTT = 42ms额外开销。

**tcp_slow_start_after_idle**：

```bash
# 默认为1：长连接空闲后重新慢启动（cwnd重置为initcwnd）
# 设为0：空闲后复用之前的大窗口，第二次传输可大幅加速
sysctl -w net.ipv4.tcp_slow_start_after_idle=0

# 实验：512KB数据在长连接中连续两次传输（RTT=3ms）
# 开启(默认)：两次均耗时~19ms
# 关闭后：第一次19ms，第二次仅5ms（窗口延续）
```

### 2.6 TCP粘包/拆包问题

**粘包/拆包原因**：
- TCP是字节流协议，无消息边界
- Nagle算法（合并小包）
- MSS限制（拆分大包）

**解决方案**：
1. **固定长度**：每个包固定N字节
2. **分隔符**：用特殊字符（如\n）分隔
3. **消息头+长度**（推荐）：

```go
// 编码
func Encode(data []byte) []byte {
    length := uint32(len(data))
    buf := make([]byte, 4+len(data))
    binary.BigEndian.PutUint32(buf[:4], length)
    copy(buf[4:], data)
    return buf
}

// 解码
func Decode(buf []byte) ([]byte, int) {
    if len(buf) < 4 {
        return nil, 0  // 数据不完整
    }
    
    length := binary.BigEndian.Uint32(buf[:4])
    if len(buf) < int(4+length) {
        return nil, 0  // 数据不完整
    }
    
    data := buf[4 : 4+length]
    return data, int(4 + length)
}
```

### 2.7 Socket缓存与TCP性能调优

> Socket缓存大小直接影响TCP在高延时网络上的带宽利用率

#### 2.7.1 缓存参数体系

```bash
# 全局Socket缓存（影响TCP和UDP）
cat /proc/sys/net/core/rmem_default  # 接收缓存默认值 212992
cat /proc/sys/net/core/rmem_max      # setsockopt可设最大值
cat /proc/sys/net/core/wmem_default  # 发送缓存默认值
cat /proc/sys/net/core/wmem_max      # setsockopt可设最大值

# TCP专用缓存（覆盖core下的默认值）
cat /proc/sys/net/ipv4/tcp_rmem  # min default max
# 4096  131072  6291456
cat /proc/sys/net/ipv4/tcp_wmem  # min default max
# 4096  16384   4194304

# default：TCP socket创建时的默认缓存大小
# min-max：tcp_moderate_rcvbuf开启时的动态调整范围
# 注意：setsockopt设置后，动态调整被关闭，上限仍受rmem_max/wmem_max限制
# 注意：setsockopt实际生效值 = 设置值 × 2
```

#### 2.7.2 高延时网络的带宽利用问题

**核心问题**：TCP发送数据需等待ACK确认，在高延时网络上，等待时间远大于数据发送时间，导致带宽利用率极低。

**带宽延时积（BDP）公式**：

```
BDP = 带宽 × RTT
   = 125MB/s × 0.4s = 50MB

# BDP就是"管道"中同时在传输的数据量
# Socket缓存必须 >= BDP，否则发送方会因缓存满而暂停发送
```

**调优实验**（1000Mbit/s带宽 + 400ms RTT）：

| 配置 | 下载速度 | 带宽利用率 |
|------|----------|-----------|
| 默认缓存（16KB wmem） | 13.4 MB/s | ~10% |
| 仅调大发送方wmem=50MB | 14.9 MB/s | ~12% |
| 同时调大接收方rmem=100MB | **92.7 MB/s** | **~74%** |

**关键发现**：仅调发送方缓存不够！TCP发送窗口受接收方通告窗口（rwnd）限制，而rwnd由接收方**剩余**rmem决定。所以**必须同时调整两端缓存**。

#### 2.7.3 调优指南

```bash
# 计算目标缓存大小
# 发送方tcp_wmem max >= BDP
# 接收方tcp_rmem max >= 2 × BDP（留余量给overhead）

# 示例：1Gbps带宽，200ms RTT
# BDP = 125MB/s × 0.2s = 25MB
echo "4096 131072 52428800" > /proc/sys/net/ipv4/tcp_wmem   # max=50MB
echo "4096 131072 104857600" > /proc/sys/net/ipv4/tcp_rmem  # max=100MB

# 确保动态调整已开启
cat /proc/sys/net/ipv4/tcp_moderate_rcvbuf  # 1=开启(默认)

# tcp_adv_win_scale：控制缓存中overhead占比
cat /proc/sys/net/ipv4/tcp_adv_win_scale
# 默认1：1/2缓存用于overhead，2：1/4用于overhead
```

**调优优先级**：
1. **低延时网络（RTT<5ms）**：缓存调整对性能几乎无影响，瓶颈在带宽本身
2. **中等延时（5-50ms）**：适当调大缓存，关注慢启动影响
3. **高延时（>50ms）**：缓存调整效果显著，需计算BDP精确调整
4. **高延时+丢包**：缓存调整+选择BBR拥塞控制算法（对丢包不敏感）

## 3. UDP协议与游戏场景

### 3.1 UDP特点

**优势**：
- ✅ **无连接**：无需握手，减少延迟
- ✅ **快速**：无确认、无重传
- ✅ **低开销**：头部只有8字节
- ✅ **支持广播/组播**

**劣势**：
- ❌ **不可靠**：丢包、乱序
- ❌ **无流量控制**
- ❌ **无拥塞控制**

### 3.2 UDP在游戏中的应用

**为什么游戏适合UDP？**
1. **实时性要求高**：FPS、MOBA需要低延迟（<100ms）
2. **允许丢包**：丢弃过时的位置数据，用最新的
3. **高频更新**：20-60次/秒发送位置，TCP重传会导致卡顿

**游戏场景选择**：
| 场景 | 协议 | 原因 |
|------|------|------|
| **登录、充值** | TCP | 必须可靠 |
| **聊天** | TCP | 必须可靠 |
| **移动、攻击** | UDP | 实时性优先 |
| **游戏结果** | TCP | 必须可靠 |

### 3.3 基于UDP的可靠传输（KCP、QUIC）

**问题**：UDP不可靠，如何在应用层实现可靠传输？

**KCP协议**（快速可靠协议）：
- 比TCP快30%-40%
- ARQ（自动重传请求）
- 快速重传、快速恢复
- 牺牲带宽换速度

**Go示例（KCP）**：
```go
import "github.com/xtaci/kcp-go"

// 服务端
listener, _ := kcp.ListenWithOptions(":8080", nil, 10, 3)
for {
    conn, _ := listener.AcceptKCP()
    go handleKCPConn(conn)
}

// 客户端
conn, _ := kcp.DialWithOptions("127.0.0.1:8080", nil, 10, 3)
conn.Write([]byte("Hello KCP"))
```

**QUIC协议**（HTTP/3基础）：
- 基于UDP
- 0-RTT握手（减少延迟）
- 多路复用、无队头阻塞
- 连接迁移（网络切换无需重连）

### 4.4 HTTP缓存策略⭐⭐⭐

> HTTP缓存是Web性能优化的核心手段，分为强缓存和协商缓存两种机制。

#### 强缓存（不需要请求服务器）

```
命中强缓存 → 浏览器直接从本地读取，不发送请求，状态码200 (from cache)

两个控制字段（优先级：Cache-Control > Expires）：

1. Cache-Control（HTTP/1.1，推荐）
   常用指令：
   - max-age=3600     → 资源有效期3600秒（从响应生成时间算起）
   - no-cache         → 不使用强缓存，每次都要走协商缓存验证
   - no-store         → 完全不缓存（敏感数据：银行页面等）
   - public           → 允许CDN/代理服务器缓存
   - private          → 只允许浏览器缓存（默认值）
   - s-maxage=3600    → CDN/代理的有效期（覆盖max-age）
   - must-revalidate  → 缓存过期后必须重新验证，不能使用过期缓存

2. Expires（HTTP/1.0，已过时）
   - 值为绝对时间：Expires: Thu, 26 Mar 2026 08:00:00 GMT
   - 缺点：依赖客户端时钟准确，客户端时间偏差会导致缓存失效
```

#### 协商缓存（需要请求服务器验证）

```
强缓存失效后 → 浏览器发请求到服务器验证资源是否变化

命中协商缓存 → 服务器返回304 Not Modified，浏览器用本地缓存
未命中 → 服务器返回200 + 新资源

两组验证字段（优先级：ETag > Last-Modified）：

1. ETag / If-None-Match（推荐，精确）
   响应头：ETag: "5d8c72a5edda8"（资源的唯一标识，通常是内容hash）
   请求头：If-None-Match: "5d8c72a5edda8"
   - 服务器对比ETag，相同返回304，不同返回200+新资源
   - 优点：精确到内容变化，即使时间变了但内容没变也不会失效
   - 缺点：服务器需要计算hash，有性能开销

2. Last-Modified / If-Modified-Since（兼容性好）
   响应头：Last-Modified: Thu, 26 Mar 2026 08:00:00 GMT
   请求头：If-Modified-Since: Thu, 26 Mar 2026 08:00:00 GMT
   - 服务器对比文件修改时间，未修改返回304
   - 缺点：精度只到秒（1秒内多次修改无法感知）
   - 缺点：文件被touch但内容没变也会失效
```

#### 缓存决策流程

```
浏览器请求资源
  │
  ├─ 有缓存？
  │   ├─ 否 → 请求服务器 → 缓存响应
  │   │
  │   └─ 是 → 检查强缓存（Cache-Control / Expires）
  │         ├─ 未过期 → 200 (from cache)，不发请求 ✅
  │         │
  │         └─ 已过期 → 发起协商缓存请求
  │               │   携带 If-None-Match / If-Modified-Since
  │               │
  │               ├─ 资源未变 → 304 Not Modified，用本地缓存 ✅
  │               │
  │               └─ 资源已变 → 200 + 新资源，更新缓存
```

#### 实际应用策略

```
不同资源的缓存策略：

HTML页面：
  Cache-Control: no-cache
  → 每次都协商验证（保证用户看到最新版本）

CSS/JS/图片（带hash文件名，如 app.a1b2c3.js）：
  Cache-Control: public, max-age=31536000 (1年)
  → 长期强缓存（文件名含hash，内容变了hash变，自动失效）

API接口：
  Cache-Control: no-store
  → 完全不缓存（数据实时性要求高）

CDN资源：
  Cache-Control: public, s-maxage=86400, max-age=3600
  → CDN缓存1天，浏览器缓存1小时
```

```go
import "github.com/gorilla/websocket"

var upgrader = websocket.Upgrader{
    CheckOrigin: func(r *http.Request) bool { return true },
}

func handleWebSocket(w http.ResponseWriter, r *http.Request) {
    conn, _ := upgrader.Upgrade(w, r, nil)
    defer conn.Close()
    
    for {
        messageType, message, err := conn.ReadMessage()
        if err != nil {

        }
        conn.WriteMessage(messageType, message)
    }
```

## 9. QUIC 与 HTTP/3 深入

### 9.1 QUIC 设计动机

TCP 作为传输层协议服役数十年，在现代网络环境中暴露出三个核心痛点：

| 痛点 | 具体表现 | 影响 |
|------|----------|------|
| **TCP 队头阻塞** | TCP 是单一有序字节流，一个包丢失会阻塞后续所有数据 | HTTP/2 多路复用的收益被 TCP 层队头阻塞抵消 |
| **握手延迟叠加** | TCP 三次握手（1 RTT）+ TLS 1.3 握手（1 RTT）= 至少 2 RTT | 移动网络首次连接体感慢（RTT 常 100ms+） |
| **连接绑定四元组** | TCP 连接 = (src_ip, src_port, dst_ip, dst_port) | Wi-Fi ↔ 4G 切换 IP 变化即断连，需重新握手 |

### 9.2 QUIC 核心特性

#### 0-RTT / 1-RTT 握手

```
首次连接（1-RTT）：
客户端                              服务器
  |  Initial: ClientHello + Key Share  |
  |──────────────────────────────────→ |
  |                                    |
  |  Initial: ServerHello + Handshake  |
  |  + EncryptedExtensions + Cert      |
  |←────────────────────────────────── |
  |                                    |
  |  [此时已可发送加密应用数据]          |

恢复连接（0-RTT）：
客户端                              服务器
  |  Initial + 0-RTT应用数据（PSK）    |
  |──────────────────────────────────→ |
  |  ← 服务器直接处理数据并响应         |
  |                                    |
  省去1个RTT！首包即携带业务数据
```

> **0-RTT 安全注意**：0-RTT 数据不具备前向保密性且可被重放（Replay），应仅用于幂等请求（如 GET）。

#### 多路复用 — 无队头阻塞

```
HTTP/2 over TCP：
流A: [帧1] [帧2] ─────────────────→  TCP字节流
流B: [帧3] [帧4] ─────────────────→  （共享同一条TCP连接）

帧2丢失 → TCP层等待重传 → 帧3、帧4也被阻塞（即使属于不同流）

HTTP/3 over QUIC：
流A: [帧1] [帧2] ── 独立QUIC流 ──→  各流独立排序
流B: [帧3] [帧4] ── 独立QUIC流 ──→

帧2丢失 → 只有流A等待重传 → 流B不受影响，正常交付
```

#### 连接 ID 迁移

```
QUIC连接标识 = Connection ID（而非四元组）

场景：手机从Wi-Fi切换到4G
- IP地址变化：192.168.1.100 → 10.0.0.50
- TCP：连接断开，需重新握手 + TLS协商
- QUIC：Connection ID不变，验证Path后继续传输，用户无感知

实现：客户端在新路径发送PATH_CHALLENGE帧，服务端回PATH_RESPONSE验证后继续通信
```

#### 内置 TLS 1.3

- QUIC **强制**加密，TLS 1.3 是协议不可分割的一部分（而非像 TCP+TLS 那样分层叠加）
- 连接建立和加密协商合并在同一次握手中完成
- 报头也部分加密，中间设备无法篡改传输参数

### 9.3 QUIC vs TCP+TLS 对比

| 维度 | TCP + TLS 1.3 | QUIC |
|------|---------------|------|
| **首次握手延迟** | 2 RTT（TCP 1 + TLS 1） | 1 RTT（合并握手） |
| **恢复连接延迟** | 1-2 RTT | 0 RTT（携带应用数据） |
| **队头阻塞** | 有（TCP 层） | 无（流级独立排序） |
| **连接迁移** | 不支持（四元组绑定） | 支持（Connection ID） |
| **拥塞控制** | 内核实现，升级需改内核 | 用户态实现，可快速迭代 |
| **NAT 穿越** | 成熟 | UDP 可能被限流/阻断 |
| **加密范围** | 载荷加密，头部明文 | 载荷 + 部分头部加密 |
| **中间设备兼容** | 极好（几十年积累） | 部分企业防火墙/ISP 限制 UDP |

### 9.4 HTTP/3 = HTTP 语义 + QUIC 传输

```
协议栈对比：

HTTP/2                          HTTP/3
┌─────────────┐                ┌─────────────┐
│  HTTP/2     │                │  HTTP/3     │
│  (HPACK)    │                │  (QPACK)    │
├─────────────┤                ├─────────────┤
│  TLS 1.2/1.3│                │   QUIC      │
├─────────────┤                │  (内置TLS   │
│   TCP       │                │    1.3)     │
├─────────────┤                ├─────────────┤
│   IP        │                │   UDP / IP  │
└─────────────┘                └─────────────┘
```

**HTTP/3 vs HTTP/2 关键区别**：

| 区别 | HTTP/2 | HTTP/3 |
|------|--------|--------|
| 头部压缩 | HPACK（依赖有序交付） | QPACK（适配无序交付，单向流传编码表） |
| 流控 | HTTP 帧级 + TCP 级双层流控 | QUIC 流级 + 连接级流控 |
| 优先级 | 复杂的依赖树模型（实践中少有实现） | 简化为 Urgency + Incremental 两个参数 |
| 服务器推送 | 支持（但实践中使用极少） | 保留但进一步弱化 |

### 9.5 QUIC 帧结构简图

```
QUIC Packet（UDP载荷）：
┌────────────────────────────────────────────────┐
│ Header                                         │
│  ├─ Header Form (1 bit): Long/Short            │
│  ├─ Connection ID (0-20 bytes)                 │
│  ├─ Packet Number (1-4 bytes, 加密)             │
│  └─ Version (Long Header only)                 │
├────────────────────────────────────────────────┤
│ Encrypted Payload (一个或多个QUIC帧)             │
│  ┌──────────────┐                              │
│  │ STREAM帧      │ → 携带应用数据（Stream ID +  │
│  │               │   Offset + Length + Data）   │
│  ├──────────────┤                              │
│  │ ACK帧         │ → 确认收到的Packet Number范围 │
│  ├──────────────┤                              │
│  │ CRYPTO帧      │ → TLS握手数据               │
│  ├──────────────┤                              │
│  │ 其他帧...     │ → PATH_CHALLENGE/RESPONSE,  │
│  │               │   NEW_CONNECTION_ID,         │
│  │               │   MAX_STREAM_DATA 等         │
│  └──────────────┘                              │
└────────────────────────────────────────────────┘

关键设计：
- Packet Number 单调递增，不复用（区别于TCP seq），简化丢包判断
- 一个UDP包可携带多个QUIC帧，减少头部开销
- STREAM帧携带Stream ID + Offset，实现流级独立排序
```

### 9.6 实际部署现状与注意事项

**部署现状**（截至2025）：
- Google、Meta、Cloudflare 等大厂已大规模部署 HTTP/3
- 主流浏览器（Chrome、Firefox、Safari、Edge）均已支持
- Nginx 1.25+、Caddy、LiteSpeed 等服务器已支持
- 全球约 30% 的 Web 流量使用 HTTP/3

**部署注意事项**：

```bash
# 1. UDP 限流问题
# 很多企业防火墙和ISP默认限制或阻断UDP流量
# 部署前需确认链路上UDP 443端口畅通
# HTTP/3客户端通常会回退：先尝试QUIC，失败后降级到HTTP/2（Alt-Svc机制）

# 2. 中间设备兼容性
# NAT设备对UDP连接的超时通常比TCP短（30s vs 数小时）
# QUIC需要定期发送keep-alive防止NAT映射过期
# 部分DPI设备无法识别QUIC流量，可能误判为异常

# 3. 服务器配置示例（Nginx）
# nginx.conf
server {
    listen 443 quic reuseport;  # QUIC监听
    listen 443 ssl;              # TLS回退
    http3 on;
    ssl_certificate     /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;
    add_header Alt-Svc 'h3=":443"; ma=86400';  # 告知客户端支持HTTP/3
}

# 4. 调试工具
# - Chrome: chrome://net-internals/#quic 查看QUIC连接状态
# - curl 7.66+: curl --http3 https://example.com
# - Wireshark 3.x+: 支持QUIC解析（需导入TLS密钥日志）
# - qlog: QUIC标准化日志格式，配合qvis可视化工具分析
```

## 10. TCP BBR 拥塞控制算法

### 10.1 传统拥塞控制的问题

传统拥塞控制算法（如 CUBIC、Reno）**以丢包作为拥塞信号**，存在根本性缺陷：

```
理想网络模型：
                          ← BtlBw（瓶颈带宽）
发送速率 ──────────────────┬────────────────
                           │
                           │ ← 缓冲区开始填充
                           │
丢包点 ─────────────────── │ ──────── ← 缓冲区溢出才丢包
                           │
                        实际吞吐量

问题：CUBIC等待丢包才降速，但丢包意味着缓冲区已满
      → 缓冲区长期处于满/近满状态（Bufferbloat）
      → RTT被缓冲区排队延迟大幅抬高
      → 高延迟链路（如跨洋、卫星）表现极差
```

**Bufferbloat（缓冲区膨胀）效应**：
- 现代网络设备缓冲区越来越大
- CUBIC 会填满缓冲区才触发丢包 → RTT 从 50ms 飙到 500ms+
- 用户体感：带宽够但延迟高、交互卡顿

### 10.2 BBR 核心理念

BBR（Bottleneck Bandwidth and Round-trip propagation time）由 Google 于 2016 年提出。

**核心思想：不以丢包为信号，而是持续探测两个关键参数**：

```
BtlBw（Bottleneck Bandwidth）：瓶颈链路的最大带宽
  → 通过观测已确认数据量 / ACK间隔 来估算
  → 取最近10个RTT周期内的最大值

RTprop（Round-trip propagation time）：最小往返传播延迟
  → 通过观测最小RTT来估算（排除排队延迟）
  → 取最近10秒内的最小值

最优发送速率 = BtlBw
最优在途数据量（inflight） = BtlBw × RTprop

目标：发送速率刚好等于瓶颈带宽，在途数据量刚好填满管道但不溢出缓冲区
      → 最大吞吐 + 最小延迟（Kleinrock最优工作点）
```

### 10.3 BBR 四个阶段

```
┌─────────────────────────────────────────────────────────┐
│                  BBR 状态机                               │
│                                                         │
│  ① Startup ──→ ② Drain ──→ ③ ProbeBW ←──→ ④ ProbeRTT  │
│  (指数探测带宽)  (排空队列)   (稳态周期探测)   (探测最小RTT) │
└─────────────────────────────────────────────────────────┘
```

**① Startup（启动阶段）**：
- 类似慢启动，以 2/ln2 ≈ 2.89 倍速率指数增长
- 目标：快速发现瓶颈带宽 BtlBw
- 退出条件：连续 3 轮 RTT 带宽增长不超过 25%（说明已找到瓶颈）

**② Drain（排空阶段）**：
- Startup 结束时缓冲区已积压数据（超发了约 BDP 的数据）
- 降低发送速率（pacing_gain < 1），排空多余队列
- 退出条件：在途数据量 ≤ BtlBw × RTprop（即 1 个 BDP）

**③ ProbeBW（带宽探测 — 稳态阶段）**：
- 大部分时间处于此阶段
- 以 8 轮 RTT 为一个周期，周期性地微调发送速率：
  - 1 轮：pacing_gain = 1.25（略微超发，探测是否有更多带宽）
  - 1 轮：pacing_gain = 0.75（回退，排空可能的队列积压）
  - 6 轮：pacing_gain = 1.0（以估算的 BtlBw 稳定发送）

**④ ProbeRTT（RTT 探测阶段）**：
- 每隔约 10 秒触发一次（如果 RTprop 长期未更新）
- 将在途数据量降至 4 个包，持续 200ms
- 目标：排空缓冲区队列，测量真实的最小传播延迟
- 短暂降低吞吐，但确保 RTprop 估算准确

### 10.4 BBR vs CUBIC 对比

| 维度 | CUBIC | BBR |
|------|-------|-----|
| **拥塞信号** | 丢包 | 带宽和RTT |
| **带宽利用率** | 中等（受丢包率影响大） | 高（接近瓶颈带宽） |
| **延迟表现** | 高（填满缓冲区） | 低（主动控制在途数据量） |
| **高丢包环境** | 吞吐急剧下降 | 受影响小（不把丢包当拥塞信号） |
| **公平性** | 与同类算法公平 | BBRv1与CUBIC共存时可能抢占带宽 |
| **适用场景** | 通用、低丢包链路 | 高延迟/高丢包/跨地域链路 |
| **实现位置** | 内核 | 内核（但设计上易于用户态实现） |

### 10.5 实际效果

Google 于 2016 年在其全球骨干网部署 BBR 后的效果：

```
YouTube 实测数据（Google 2016 公开数据）：
- 全球平均吞吐提升 4%，部分地区提升 14%+
- 全球平均 RTT 降低 33%，部分地区降低 50%+
- 重缓冲率（播放卡顿）降低显著

原因分析：
- YouTube 大量用户在发展中国家（高延迟/高丢包网络）
- CUBIC 在这些网络上因频繁丢包而严重降速
- BBR 不受丢包率影响，持续以瓶颈带宽发送
```

### 10.6 Linux 开启 BBR

```bash
# 1. 检查内核版本（需要 4.9+）
uname -r

# 2. 加载 BBR 模块
modprobe tcp_bbr

# 3. 设置 BBR 为默认拥塞控制算法
sysctl -w net.ipv4.tcp_congestion_control=bbr

# 4. 验证
sysctl net.ipv4.tcp_congestion_control
# 输出：net.ipv4.tcp_congestion_control = bbr

# 5. 确认 BBR 已加载
lsmod | grep bbr
# 输出：tcp_bbr  ...

# 6. 永久生效（写入配置文件）
cat >> /etc/sysctl.conf << 'EOF'
net.core.default_qdisc = fq
net.ipv4.tcp_congestion_control = bbr
EOF
sysctl -p

# 注意：BBR 推荐搭配 fq（Fair Queuing）队列调度器
# fq 提供精确的 pacing（包间距控制），是 BBR 发挥最佳效果的前提
```

### 10.7 BBR 的争议与演进

**公平性争议（BBRv1）**：

```
BBRv1 与 CUBIC 共存时的不公平问题：

场景：10Mbps瓶颈链路，1个BBR流 + 1个CUBIC流共享
- 理论公平：各占5Mbps
- 实测：BBR占约7-8Mbps，CUBIC只剩2-3Mbps

原因：
- BBR不以丢包为信号，持续高速发送
- CUBIC检测到丢包后主动降速
- BBR的"不怕丢包"变成了对CUBIC的带宽侵占

多条BBR流共存时公平性尚可，但与基于丢包的算法共存时存在挤占问题
```

**BBRv2 改进（2019年开始开发，持续迭代中）**：

```
BBRv2 核心改进：
1. 引入丢包和ECN作为辅助信号（不再完全忽略丢包）
   - 显式拥塞通知（ECN）比丢包更早、更精确
2. 更保守的Startup阶段（减少初始超发）
3. 改进与CUBIC的公平共存
4. 更精确的带宽探测（减少ProbeRTT的吞吐波动）

BBRv3（实验中）：
- 进一步优化公平性和收敛速度
- 更好的ECN支持
- Google内部持续测试迭代

当前建议：
- 生产环境优先使用 BBRv1（稳定、内核原生支持）
- 关注 BBRv2/v3 的合入进度
- 如果链路上BBR和CUBIC流混合，需评估公平性影响
```

## 12. 面试题自查

### 网络排障补充

- 网络题不应只背协议定义，必须能落到故障定位路径：现象分层、抓包验证、指标交叉。
- 排障时建议固定顺序：DNS -> 连接建立 -> 传输质量 -> 应用协议语义，避免盲目抓日志。
- 线上高频问题通常不是“连不上”，而是“偶发慢、偶发丢、偶发重试风暴”。

### 初级高频速刷

**1. OSI 七层模型和 TCP/IP 四层模型怎么对应？**

OSI 七层更细，TCP/IP 四层更贴近实际网络实现。大致可以理解为：OSI 的应用层/表示层/会话层合并成 TCP/IP 的应用层，传输层对应传输层，网络层对应网络层，数据链路层和物理层合并成链路层。

**2. `GET` 和 `POST` 的区别是什么？**

`GET` 语义上用于获取资源，天然适合幂等查询；`POST` 语义上用于提交数据并触发服务端处理。真正的区别首先是语义和缓存/幂等约定，而不是“一个能传参一个不能传参”这种过时回答。

**3. `301`、`302`、`307`、`308` 分别表示什么？**

`301` 永久重定向，`302` 临时重定向，历史上很多客户端会把 `302` 改成 GET；`307` 临时重定向但要求保持原请求方法；`308` 永久重定向且保持原请求方法。这个问题常用来筛掉只会背“301永久302临时”的回答。

**4. Cookie、Session、Token 的区别是什么？**

Cookie 是浏览器存储载体，Session 是服务端保存的会话状态，Token 是认证凭证。它们不是互斥关系，常见组合是 Cookie 里带 SessionID，或者前端持有 JWT Token 访问后端。

**5. 常见 HTTP 状态码 `200/301/302/400/401/403/404/500/502/503` 分别表示什么？**

`200` 成功，`301/302` 重定向，`400` 请求参数错误，`401` 未认证，`403` 禁止访问，`404` 资源不存在，`500` 服务端内部错误，`502` 网关拿到上游无效响应，`503` 服务暂时不可用。真正面试时，经常会接着追问 `502` 和 `503` 的排障差异。

**6. `TCP` 和 `UDP` 的核心差异是什么？**

`TCP` 提供连接、可靠传输、拥塞控制和流量控制；`UDP` 无连接、尽力而为、开销小。真正选型要看业务目标，是要可靠有序，还是要更低时延和更高实时性。

### 中级高频进阶

**7. HTTP Keep-Alive、TCP keepalive、应用层心跳有什么区别？**

HTTP Keep-Alive 解决的是“连接复用”；TCP keepalive 解决的是“内核层面判断长连接是否失活”；应用层心跳解决的是“业务协议自己定义健康探测和超时控制”。三者名字相近，但解决的问题并不在同一层。

**8. 为什么很多 RPC 框架会选择 HTTP/2 或 gRPC？**

因为 HTTP/2 支持多路复用、头部压缩、服务端推送和更适合长连接的传输模型，能减少连接数和队头阻塞问题。gRPC 再叠加 Protobuf 的强约束和高效序列化，非常适合服务间通信。

**9. `401` 和 `403` 在语义上到底怎么区分？**

`401` 是“你还没完成合法认证”，`403` 是“你是谁我知道，但你没有权限做这件事”。这个区分看似细节，实际在网关、鉴权中间件和安全审计里都很重要。

**10. `502`、`503`、`504` 在网关侧分别说明什么？**

`502` 更偏上游返回了无效响应，`503` 更偏服务当前不可用或被限流，`504` 更偏网关等待上游超时。把这三者区分清楚，排障入口会快很多。

### 深入问答

#### Q1: TCP三次握手每一步的作用是什么？为什么不能两次或四次？

**答**：第一次SYN让服务端确认客户端发送能力正常；第二次SYN-ACK让客户端确认服务端收发能力正常；第三次ACK让服务端确认客户端接收能力正常。两次不够是因为服务端无法确认客户端能收到自己的SYN-ACK（即无法确认客户端的接收能力和自己的发送能力），四次多余是因为第二步SYN和ACK可以合并。核心目的是双方互相确认对方的seq序列号。

#### Q2: SYN Flood攻击原理是什么？syncookie如何防御？

**答**：攻击者大量发送SYN但不回ACK，耗尽服务端半连接队列。syncookie的防御原理是服务端不保存SYN状态，而是将四元组+时间戳做hash生成cookie放入SYN-ACK的seq字段，客户端ACK带回cookie（seq+1），服务端验证通过后直接建立连接，将内存攻击转化为CPU运算。

#### Q3: TIME_WAIT状态为什么是2MSL？TIME_WAIT过多怎么解决？

**答**：2MSL = 1MSL（对方重传FIN的最大延迟）+ 1MSL（我方ACK的最大延迟），确保①最后的ACK能到达对方（否则对方会重传FIN）②旧连接的所有游荡报文消失，不干扰新连接。解决TIME_WAIT过多：启用`tcp_tw_reuse`复用端口（安全，依赖timestamp比较）、使用连接池（最佳方案）、增大端口范围、使用HTTP Keep-Alive减少短连接。

#### Q4: CLOSE_WAIT大量积压说明什么问题？如何排查？

**答**：CLOSE_WAIT说明对端已经关闭连接（发了FIN），但本端应用程序没有调用close()。这是**应用层BUG**，没有内核超时机制保护，必须修复代码。排查：`ss -tapn | grep CLOSE-WAIT`找到对应进程，检查代码中是否有未关闭的连接（如忘记defer conn.Close()、异常分支没有关闭等）。

#### Q5: TCP滑动窗口的作用是什么？发送窗口大小由什么决定？

**答**：滑动窗口允许发送方在不等每个ACK的情况下连续发送多个数据段，实现流水线传输，提高带宽利用率。发送窗口大小 = min(cwnd, rwnd)，受拥塞控制（cwnd拥塞窗口，反映网络能力）和流量控制（rwnd接收窗口，反映接收方缓冲区剩余空间）的双重约束。

#### Q6: 零窗口探测是什么？解决了什么问题？

**答**：当接收方缓冲区满时通告rwnd=0，发送方停止发送。如果后续接收方窗口更新ACK丢失，双方会死锁。零窗口探测（ZWP）解决此问题：发送方启动持续定时器，定期发送1字节探测报文，接收方回复当前rwnd。探测间隔指数退避，超过tcp_retries2次后RST关闭连接。

#### Q7: 流量控制和拥塞控制的区别是什么？

**答**：流量控制防止发送方淹没**接收方**，通过接收方通告rwnd实现，反映接收方缓冲区状态；拥塞控制防止发送方淹没**网络**，通过发送方维护的cwnd实现，根据丢包等网络反馈动态调整。两者协作：实际发送窗口 = min(cwnd, rwnd)。

#### Q8: TCP慢启动为什么是指数增长？到达ssthresh后为什么改为线性增长？

**答**：慢启动阶段指数增长（每RTT cwnd翻倍）是为了快速探测网络可用带宽，避免线性增长时长时间低效传输。到达ssthresh后转为拥塞避免的线性增长（每RTT cwnd+1），因为此时已接近网络容量上限，继续翻倍可能立即引发拥塞，线性增长可以谨慎逼近实际容量。

#### Q9: 快速重传和快速恢复相比传统Tahoe算法有什么改进？

**答**：传统Tahoe收到丢包信号后cwnd=1重新慢启动，过于保守。快速重传：收到3个重复ACK立即重传，不等超时（超时通常1秒）。快速恢复（Reno）：丢包后ssthresh=cwnd/2，cwnd=ssthresh直接进入拥塞避免，跳过慢启动。理由是收到3个重复ACK说明网络仍有传输能力（后续包能到达），没必要重置到cwnd=1。

#### Q10: CUBIC和BBR拥塞控制算法各有什么特点？如何选择？

**答**：CUBIC（Linux默认）：窗口增长函数为三次函数，适合高带宽网络，对丢包敏感。BBR（Google）：不基于丢包而是持续测量带宽和RTT，目标是最大化带宽利用率、最小化延迟。选择：高带宽低延迟选BBR，高丢包率网络CUBIC更稳定，数据中心内部用DCTCP。修改：`sysctl -w net.ipv4.tcp_congestion_control=bbr`。

#### Q11: Nagle算法和延迟ACK叠加会导致什么问题？如何解决？

**答**：Nagle算法让发送方在有未确认数据时积累小包；延迟ACK让接收方等200ms才回ACK。两者叠加：发送方发了小包等ACK（Nagle），接收方等200ms才回（延迟ACK），导致每个小包有200ms额外延迟。解决：设置TCP_NODELAY关闭Nagle算法，适用于游戏、SSH等实时交互场景。

#### Q12: HTTP强缓存和协商缓存的区别是什么？各自的判断字段是什么？

**答**：强缓存直接从本地读取不发请求（200 from cache），由Cache-Control（max-age/no-cache/no-store）和Expires控制，优先级Cache-Control > Expires。协商缓存需要向服务器验证资源是否变化（304 Not Modified），由ETag/If-None-Match和Last-Modified/If-Modified-Since控制，优先级ETag > Last-Modified。ETag基于内容hash更精确，Last-Modified精度只到秒。

#### Q13: HTTP/1.1、HTTP/2、HTTP/3的核心区别是什么？

**答**：HTTP/1.1基于TCP，队头阻塞严重（请求串行或有限并行）。HTTP/2仍基于TCP，引入多路复用（单连接多流）、头部压缩（HPACK）、服务端推送，但TCP层仍有队头阻塞。HTTP/3基于QUIC（UDP），彻底解决队头阻塞（丢包只影响单个流），支持0-RTT握手和连接迁移（网络切换IP变化不断连）。

#### Q14: HTTPS的TLS握手过程是怎样的？TLS 1.3相比1.2有何改进？

**答**：TLS 1.2需要2-RTT：Client Hello → Server Hello+证书 → 密钥交换 → Finished。TLS 1.3简化为1-RTT：Client Hello携带Key Share → Server Hello携带Key Share+证书，之后立即加密通信。改进：①减少一次RTT②移除不安全的加密套件③支持0-RTT恢复（重连场景）。

#### Q15: WebSocket和HTTP长轮询有什么区别？WebSocket如何保活？

**答**：HTTP长轮询：客户端发请求，服务器hold住直到有数据或超时返回，半双工。WebSocket：通过HTTP Upgrade握手后建立全双工连接，双方可随时推送，头部开销小（2-10字节 vs HTTP几百字节）。保活：通过Ping/Pong帧实现心跳，发送方定时（如30s）发Ping，接收方自动回Pong，超时未收到则判定断连。

#### Q16: DNS解析的完整流程是什么？如何优化DNS性能？

**答**：浏览器缓存 → 系统hosts缓存 → 本地DNS服务器（递归查询）→ 根DNS → 顶级域DNS(.com) → 权威DNS → 返回IP。优化：①DNS预解析（`<link rel="dns-prefetch">`）②合理设置TTL③使用本地DNS缓存④减少DNS查找次数（合并域名）⑤使用HTTPDNS绕过运营商DNS劫持。

#### Q17: BDP（带宽延时积）是什么？为什么高延时网络需要调整Socket缓存？

**答**：BDP = 带宽 × RTT，表示网络"管道"中同时在传的数据量。TCP发送窗口受rwnd限制，rwnd由接收方缓冲区剩余空间决定。如果Socket缓存 < BDP，发送方会因缓冲区满而暂停发送，导致带宽利用率低。例如1Gbps+400ms RTT，BDP=50MB，默认16KB缓存只能利用~10%带宽，需要**同时**调大收发两端缓存。

#### Q18: TCP粘包/拆包问题的原因是什么？有哪些解决方案？

**答**：TCP是字节流协议无消息边界，加上Nagle算法合并小包、MSS拆分大包，接收方无法区分消息边界。解决方案：①固定长度（每个消息固定N字节，简单但浪费）②分隔符（如\n分隔，需要转义）③**长度前缀**（推荐，消息头4字节表示消息体长度，解码时先读长度再读对应字节数）。

#### Q19: 为什么tcp_tw_reuse安全而tcp_tw_recycle不安全（已被移除）？

**答**：`tcp_tw_reuse`只复用客户端出站连接的TIME_WAIT端口，且检查新连接的TCP timestamp > 旧连接timestamp才允许复用，安全。`tcp_tw_recycle`快速回收TIME_WAIT（不等2MSL），在NAT环境下有致命问题：NAT后多个客户端共享公网IP但timestamp各异，服务器可能因为timestamp回退而拒绝合法连接。已在Linux 4.12+移除。

#### Q20: 如果你是一个后端开发，如何排查线上服务大量TIME_WAIT/CLOSE_WAIT？

**答**：**TIME_WAIT过多**：`ss -s | grep TIME-WAIT`查看数量 → `ss -tapn | grep TIME-WAIT`定位进程 → 常见原因：短连接频繁（改用Keep-Alive/连接池）、客户端主动关闭过多（让服务器主动关闭）、健康检查过频 → 临时缓解：`tcp_tw_reuse=1`、增大端口范围。**CLOSE_WAIT过多**：一定是应用代码BUG（没有close连接），查找对应进程的代码，检查异常路径和资源释放。

#### Q21: 四层负载均衡（L4）和七层负载均衡（L7）如何取舍？

**答**：L4 基于 IP/端口转发，性能高、开销小，但对业务语义感知弱；L7 能按 Host/Path/Header/Cookie 路由，支持灰度、鉴权、重写和可观测，但引入更多处理开销和治理复杂度。实践里通常是边缘入口用 L7 做策略治理，内部高吞吐链路保留 L4 以控制时延。

#### Q22: RTT、吞吐、P99 延迟三者是什么关系？为什么“带宽够大”不代表请求快？

**答**：RTT 决定单次往返成本，吞吐决定单位时间可传输总量，P99 体现尾部请求体验。带宽高只能说明“管道宽”，如果 RTT 高、重传多、排队长，P99 依然会很差。面试回答建议强调：容量问题和时延问题是两条不同维度，不能只看带宽。

#### Q23: 线上偶发超时时，如何用抓包快速判断是重传、乱序还是应用处理慢？

**答**：先按时间窗口抓双端包：
1. 看是否出现大量 `Dup ACK`、`Retransmission`，判断是否链路丢包或拥塞。
2. 看请求到达时间和响应发出时间差，区分网络慢还是应用处理慢。
3. 对比客户端与服务端时间线，确认慢点在客户端发送前、网络传输中还是服务端处理阶段。

抓包只是证据源，最终要和应用日志、网关指标、内核网络统计一起交叉验证。

#### Q24: DNS 成为性能瓶颈时，常见治理手段有哪些？

**答**：常见做法有四类：
1. 缓存治理：合理 TTL、本地缓存、连接池复用避免频繁解析。
2. 拓扑治理：就近解析、权威 DNS 高可用、避免单解析链路。
3. 容灾治理：多解析服务商、超时回退策略、熔断异常解析结果。
4. 观测治理：监控解析成功率、延迟分位、NXDOMAIN 比例和上游错误码。

如果只做“加缓存”不做可观测，问题会从高延迟变成“缓存错误放大”。

#### Q25: HTTP/3(QUIC) 在什么场景下明显优于 HTTP/2，什么场景收益不明显？

**答**：
明显收益场景：高丢包、弱网、移动网络切换频繁（IP 变化）的场景。QUIC 在单连接多流下避免 TCP 队头阻塞，且支持连接迁移，能显著改善尾延迟和重连成本。

收益不明显场景：低丢包、低 RTT、同机房或高质量内网链路下，HTTP/2 已经足够稳定，HTTP/3 的收益可能主要体现在边缘场景而非核心链路。

评估应以真实链路分层指标为准，不要只看理论协议优势。

#### Q26: 抓包看到重传不多，但 P99 还是高，下一步你查应用还是内核？为什么？

**答**：
先查**应用侧等待链路**，再下探内核。因为“重传不多”只说明显性丢包不是主因，但高 P99 更常见于应用排队、锁竞争、下游慢调用、连接池等待。

建议顺序：
1. 应用侧：线程池队列、连接池等待、慢接口分布、依赖超时。
2. 内核侧：run queue、软中断、socket backlog、磁盘/网络队列。

这样能先定位最可能的瓶颈层，再决定是否深入内核调优。

### 开放式设计题

**D1：设计一个全球化部署的实时通信系统（如视频会议），网络层需要考虑什么？**

**参考思路**：
- 传输协议：实时音视频用UDP（RTP/RTCP）、信令用WebSocket/gRPC、考虑QUIC替代UDP+自研可靠层
- 全球加速：Anycast接入→就近POP节点→专线骨干网转发→目的地区POP出口（类似GSLB+SD-WAN）
- 弱网对抗：FEC前向纠错、Jitter Buffer、自适应码率（根据RTT和丢包率动态调整）
- NAT穿越：STUN/TURN/ICE、Relay兜底
- 质量监控：RTT/丢包率/抖动实时采集→动态选路（多路径同时传输选最优）

**D2：TCP连接数突然从正常的1万暴涨到50万，导致服务不可用，如何排查和应对？**

**参考思路**：
- 紧急止血：限制新连接速率（iptables/Nginx limit_conn）、重启负载均衡摘除问题节点
- 排查：ss -s查连接状态分布、大量CLOSE_WAIT说明应用未关闭连接、大量TIME_WAIT说明大量短连接
- CLOSE_WAIT：定位哪个goroutine/线程持有未关闭的conn（pprof/jstack）、常见原因（HTTP Response Body未读完/未Close）
- TIME_WAIT：开启SO_REUSEADDR/SO_REUSEPORT、调整tcp_tw_reuse、改用长连接池
- 根因治理：连接池化、超时设置、连接数监控告警
