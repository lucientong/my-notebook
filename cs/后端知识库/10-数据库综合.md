## 关系型数据库

### PostgreSQL

#### 核心特性

**1. MVCC实现**（与MySQL不同）

```
PostgreSQL MVCC：
- 每个事务有唯一的事务ID（XID）
- 每行记录有 xmin（创建该版本的事务ID）和 xmax（删除该版本的事务ID）
- 通过比较XID判断可见性

MySQL MVCC：
- 使用 undo log 版本链
- 通过 Read View 判断可见性
```

**2. 复杂查询支持**

```sql
-- 递归查询（树形结构）
WITH RECURSIVE org_tree AS (
  SELECT id, name, parent_id, 1 AS level
  FROM departments
  WHERE parent_id IS NULL
  
  UNION ALL
  
  SELECT d.id, d.name, d.parent_id, ot.level + 1
  FROM departments d
  JOIN org_tree ot ON d.parent_id = ot.id
)
SELECT * FROM org_tree;

-- 窗口函数（更强大）
SELECT 
  user_id,
  order_date,
  amount,
  SUM(amount) OVER (
    PARTITION BY user_id 
    ORDER BY order_date
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS cumulative_amount
FROM orders;
```

**3. JSON支持**

```sql
-- JSON字段查询
CREATE TABLE users (
  id SERIAL PRIMARY KEY,
  profile JSONB  -- JSONB：二进制JSON，支持索引
);

-- 插入JSON
INSERT INTO users (profile) VALUES 
('{"name": "Alice", "age": 30, "tags": ["tech", "finance"]}');

-- 查询JSON
SELECT profile->>'name' AS name,
       profile->>'age' AS age
FROM users
WHERE profile->>'age' = '30';

-- JSON索引
CREATE INDEX idx_profile_age ON users USING gin ((profile->'age'));
```

**4. 全文搜索**

```sql
-- 创建全文搜索索引
CREATE INDEX idx_content_fts ON articles USING gin(to_tsvector('english', content));

-- 全文搜索
SELECT * FROM articles
WHERE to_tsvector('english', content) @@ to_tsquery('english', 'postgresql & performance');

-- 中文全文搜索（需要 zhparser 插件）
CREATE TEXT SEARCH CONFIGURATION chinese (PARSER = zhparser);

SELECT * FROM articles
WHERE to_tsvector('chinese', content) @@ to_tsquery('chinese', '数据库 & 优化');
```

#### PostgreSQL vs MySQL

| 特性 | PostgreSQL | MySQL |
|------|-----------|-------|
| **ACID支持** | 完整支持 | InnoDB完整支持 |
| **并发控制** | MVCC（基于XID） | MVCC（基于undo log） |
| **复杂查询** | ✅ 递归查询、窗口函数强大 | ⚠️ 窗口函数支持有限 |
| **JSON支持** | ✅ JSONB（二进制存储+索引） | ⚠️ JSON（文本存储） |
| **全文搜索** | ✅ 内置全文搜索引擎 | ⚠️ 需要额外配置 |
| **GIS支持** | ✅ PostGIS（地理信息系统） | ⚠️ 需要额外扩展 |
| **复制** | ✅ 流复制、逻辑复制 | ✅ binlog复制 |
| **分区表** | ✅ 声明式分区（10+） | ✅ 原生分区（5.7+） |
| **性能** | 复杂查询快 | 简单查询快 |
| **适用场景** | 金融、数据分析、GIS | Web应用、高并发读写 |

#### PostgreSQL 索引类型

**1. B-Tree索引**（默认）
```sql
CREATE INDEX idx_name ON users(name);
-- 适用：等值查询、范围查询、排序
```

**2. Hash索引**
```sql
CREATE INDEX idx_email ON users USING hash(email);
-- 适用：等值查询（不支持范围查询）
```

**3. GIN索引**（倒排索引）
```sql
CREATE INDEX idx_tags ON articles USING gin(tags);
-- 适用：数组、JSON、全文搜索
```

**4. GiST索引**（通用搜索树）
```sql
CREATE INDEX idx_location ON stores USING gist(location);
-- 适用：几何类型、全文搜索、范围类型
```

**5. BRIN索引**（块范围索引）
```sql
CREATE INDEX idx_created_at ON logs USING brin(created_at);
-- 适用：大表的顺序存储数据（如时间序列）
-- 索引体积小，适合TB级数据
```

### Oracle

#### 核心特性

**1. RAC（Real Application Cluster）**

```
Oracle RAC架构：
┌─────────┐  ┌─────────┐
│ 节点1    │  │ 节点2    │
│ Instance│  │ Instance│
└────┬────┘  └────┬────┘
     │            │
     └────┬───────┘
          │ 共享存储
     ┌────┴────┐
     │ 数据文件 │
     └─────────┘

特点：
- 多个实例共享同一份数据
- 负载均衡、高可用
- 适用：金融核心系统
```

**2. 分区表**

```sql
-- 范围分区
CREATE TABLE orders (
  order_id NUMBER PRIMARY KEY,
  order_date DATE,
  amount NUMBER
)
PARTITION BY RANGE (order_date) (
  PARTITION p2023 VALUES LESS THAN (TO_DATE('2024-01-01', 'YYYY-MM-DD')),
  PARTITION p2024 VALUES LESS THAN (TO_DATE('2025-01-01', 'YYYY-MM-DD'))
);

-- 列表分区
CREATE TABLE users (
  user_id NUMBER PRIMARY KEY,
  country VARCHAR2(20)
)
PARTITION BY LIST (country) (
  PARTITION p_china VALUES ('CN'),
  PARTITION p_usa VALUES ('US'),
  PARTITION p_others VALUES (DEFAULT)
);

-- Hash分区
CREATE TABLE transactions (
  txn_id NUMBER PRIMARY KEY,
  user_id NUMBER
)
PARTITION BY HASH (user_id)
PARTITIONS 16;
```

**3. Flashback（闪回）**

```sql
-- 查询历史数据（5分钟前）
SELECT * FROM orders AS OF TIMESTAMP (SYSTIMESTAMP - INTERVAL '5' MINUTE);

-- 恢复误删的表
FLASHBACK TABLE orders TO BEFORE DROP;

-- 恢复误删的数据
FLASHBACK TABLE orders TO TIMESTAMP (SYSTIMESTAMP - INTERVAL '1' HOUR);
```

**4. 物化视图**

```sql
-- 创建物化视图（预计算结果）
CREATE MATERIALIZED VIEW mv_daily_sales
BUILD IMMEDIATE
REFRESH FAST ON COMMIT
AS
SELECT 
  product_id,
  TRUNC(order_date) AS day,
  SUM(amount) AS total_sales
FROM orders
GROUP BY product_id, TRUNC(order_date);

-- 查询物化视图（速度快）
SELECT * FROM mv_daily_sales WHERE day = TRUNC(SYSDATE);
```

### SQL Server

#### 核心特性

**1. 列存储索引**

```sql
-- 创建列存储索引（适合数据仓库）
CREATE COLUMNSTORE INDEX idx_sales_cs ON sales;

-- 性能对比
-- 行存储：查询特定列需要扫描所有列（I/O高）
-- 列存储：只读取需要的列（I/O低），压缩率高
```

**2. 内存优化表**

```sql
-- 创建内存优化表
CREATE TABLE orders_memory (
  order_id INT PRIMARY KEY NONCLUSTERED HASH WITH (BUCKET_COUNT = 1000000),
  user_id INT NOT NULL,
  amount DECIMAL(10,2) NOT NULL
) WITH (MEMORY_OPTIMIZED = ON, DURABILITY = SCHEMA_AND_DATA);

-- 特点：
-- - 数据全部存储在内存中
-- - 无锁并发控制
-- - 适用：高并发OLTP（如证券交易）
```

**3. 时态表**（Temporal Table）

```sql
-- 创建时态表（自动记录历史版本）
CREATE TABLE employees (
  employee_id INT PRIMARY KEY,
  name NVARCHAR(50),
  salary DECIMAL(10,2),
  valid_from DATETIME2 GENERATED ALWAYS AS ROW START,
  valid_to DATETIME2 GENERATED ALWAYS AS ROW END,
  PERIOD FOR SYSTEM_TIME (valid_from, valid_to)
) WITH (SYSTEM_VERSIONING = ON);

-- 查询历史数据
SELECT * FROM employees
FOR SYSTEM_TIME AS OF '2024-01-01 10:00:00';

-- 查询某个时间段的变化
SELECT * FROM employees
FOR SYSTEM_TIME BETWEEN '2024-01-01' AND '2024-12-31';
```

**4. Always On 可用性组**

```
主副本（Primary）：读写
├─ 同步副本1（Synchronous）：自动故障转移
├─ 同步副本2（Synchronous）：自动故障转移
└─ 异步副本（Asynchronous）：只读，手动故障转移

特点：
- 高可用（RPO=0，RTO<30s）
- 读写分离（异步副本处理只读查询）
```

## 列存储数据库

### 列存储原理

#### 行存储 vs 列存储

**行存储**（Row-Based Storage）：
```
磁盘存储：
Row1: [id=1, name='Alice', age=30, city='Beijing']
Row2: [id=2, name='Bob', age=25, city='Shanghai']
Row3: [id=3, name='Charlie', age=35, city='Beijing']

查询：SELECT AVG(age) FROM users WHERE city='Beijing';
- 需要读取所有列的数据（包括不需要的 name）
- I/O开销大
```

**列存储**（Column-Based Storage）：
```
磁盘存储：
Column id: [1, 2, 3]
Column name: ['Alice', 'Bob', 'Charlie']
Column age: [30, 25, 35]
Column city: ['Beijing', 'Shanghai', 'Beijing']

查询：SELECT AVG(age) FROM users WHERE city='Beijing';
- 只读取 age 和 city 两列
- I/O开销小
- 相同列的数据连续存储，压缩率高（如 city 列：'Beijing', 'Beijing'）
```

**适用场景对比**：

| 特性 | 行存储 | 列存储 |
|------|--------|--------|
| **查询** | 查询少量行的多列数据 | 查询大量行的少量列数据 |
| **场景** | OLTP（交易系统） | OLAP（数据分析） |
| **示例SQL** | `SELECT * FROM users WHERE id=1` | `SELECT AVG(age) FROM users` |
| **写入性能** | 快（按行写入） | 慢（需要写入多个列文件） |
| **压缩率** | 低 | 高（同列数据相似度高） |

### ClickHouse

#### 核心特性

**1. 表引擎**

```sql
-- MergeTree（最常用）
CREATE TABLE events (
  event_date Date,
  user_id UInt64,
  event_type String,
  count UInt32
) ENGINE = MergeTree()
PARTITION BY toYYYYMM(event_date)  -- 按月分区
ORDER BY (event_date, user_id);    -- 主键排序

-- ReplacingMergeTree（去重）
CREATE TABLE users (
  user_id UInt64,
  name String,
  version UInt64
) ENGINE = ReplacingMergeTree(version)  -- 按 version 去重，保留最新
ORDER BY user_id;

-- SummingMergeTree（自动聚合）
CREATE TABLE metrics (
  date Date,
  metric_name String,
  value Float64
) ENGINE = SummingMergeTree()
PARTITION BY toYYYYMM(date)
ORDER BY (date, metric_name);

-- 自动合并相同 (date, metric_name) 的行，SUM(value)
```

**2. 物化视图**

```sql
-- 原始数据表
CREATE TABLE raw_events (
  event_time DateTime,
  user_id UInt64,
  event_type String
) ENGINE = MergeTree()
ORDER BY event_time;

-- 物化视图（实时聚合）
CREATE MATERIALIZED VIEW daily_events_mv
ENGINE = SummingMergeTree()
PARTITION BY toYYYYMMDD(event_date)
ORDER BY (event_date, event_type)
AS
SELECT 
  toDate(event_time) AS event_date,
  event_type,
  count() AS event_count
FROM raw_events
GROUP BY event_date, event_type;

-- 写入原始数据时，物化视图自动更新
INSERT INTO raw_events VALUES (now(), 123, 'click');

-- 查询聚合结果（速度快）
SELECT * FROM daily_events_mv WHERE event_date = today();
```

**3. 高性能查询**

```sql
-- 向量化执行（SIMD）
SELECT AVG(age) FROM users;  -- 一次处理多行数据（如8行）

-- 列压缩
-- 同一列的数据相似度高，压缩率可达 10:1 甚至 100:1

-- 并行查询
SELECT count() FROM events;  -- 自动并行扫描多个分区
```

**4. 分布式查询**

```sql
-- 创建分布式表
CREATE TABLE events_dist AS events
ENGINE = Distributed(cluster_name, database, events, rand());

-- 查询自动路由到多个节点
SELECT count() FROM events_dist;
-- ClickHouse自动：
-- 1. 将查询发送到所有节点
-- 2. 各节点并行执行
-- 3. 汇总结果
```

---

#### ClickHouse 使用场景

**适用**：
- ✅ 日志分析（如访问日志、操作日志）
- ✅ 实时报表（如大屏展示）
- ✅ 用户行为分析（如漏斗分析、留存分析）
- ✅ 监控指标聚合（如QPS、错误率）

**不适用**：
- ❌ 高频更新（列存储更新慢）
- ❌ 事务处理（不支持ACID）
- ❌ 主键查询（相比MySQL慢）

### HBase

#### 核心架构

```
HBase 架构：
┌──────────────┐
│   Client     │
└──────┬───────┘
       │
┌──────┴───────────────────────────┐
│          ZooKeeper               │  ← 元数据管理
└──────┬───────────────────────────┘
       │
┌──────┴───────┐
│ HMaster      │  ← 负载均衡、故障恢复
└──────┬───────┘
       │
┌──────┴───────┬──────────┬────────┐
│ RegionServer│RegionServer│RegionServer│  ← 数据存储
└──────┬───────┴──────────┴────────┘
       │
┌──────┴───────┐
│     HDFS     │  ← 底层存储
└──────────────┘
```

**核心概念**：
- **Row Key**：主键，按字典序排序
- **Column Family**：列族（物理存储单位）
- **Column Qualifier**：列名
- **Timestamp**：版本号

```
HBase 数据模型：
Row Key | Column Family: cf1       | Column Family: cf2
--------|--------------------------|------------------
user001 | cf1:name=Alice          | cf2:age=30
        | cf1:email=alice@a.com   | cf2:city=Beijing
user002 | cf1:name=Bob            | cf2:age=25
```

---

#### HBase 操作

```java
// 创建表
HTableDescriptor table = new HTableDescriptor(TableName.valueOf("users"));
table.addFamily(new HColumnDescriptor("info"));
table.addFamily(new HColumnDescriptor("stats"));
admin.createTable(table);

// 插入数据
Put put = new Put(Bytes.toBytes("user001"));  // Row Key
put.addColumn(Bytes.toBytes("info"), Bytes.toBytes("name"), Bytes.toBytes("Alice"));
put.addColumn(Bytes.toBytes("info"), Bytes.toBytes("age"), Bytes.toBytes("30"));
table.put(put);

// 查询单行
Get get = new Get(Bytes.toBytes("user001"));
Result result = table.get(get);
String name = Bytes.toString(result.getValue(Bytes.toBytes("info"), Bytes.toBytes("name")));

// 扫描范围
Scan scan = new Scan();
scan.setStartRow(Bytes.toBytes("user000"));
scan.setStopRow(Bytes.toBytes("user999"));
ResultScanner scanner = table.getScanner(scan);
for (Result result : scanner) {
  // 处理结果
}
```

---

#### HBase Row Key 设计（重要⭐⭐⭐）

**设计原则**：
1. **均匀分布**（避免热点）
2. **支持范围查询**
3. **唯一性**

**反例：使用时间戳作为Row Key**
```
错误设计：rowkey = timestamp
20240101000000_user001
20240101000001_user002
20240101000002_user003

问题：
- 写入集中在一个Region（热点写入）
- 无法充分利用分布式存储
```

**正例：Hash + 时间戳**
```
正确设计：rowkey = MD5(user_id).substring(0,4) + timestamp + user_id
a3f1_20240101000000_user001
b7e2_20240101000001_user002
3c9d_20240101000002_user003

优势：
- Hash前缀：分散到不同Region（均匀写入）
- 时间戳：支持范围查询
- user_id：唯一性
```

## 时序数据库

### 时序数据特点

**定义**：时间序列数据（Time Series Data）是按时间顺序排列的数据点序列。

**特点**：
1. **时间戳**：每条数据都有时间戳
2. **只增不改**：数据写入后很少更新
3. **高写入量**：每秒数万至数百万条
4. **定期查询**：按时间范围查询（如最近1小时）
5. **降采样**：历史数据可以降精度（如1分钟粒度→1小时粒度）

**应用场景**：
- ✅ 监控系统（如Prometheus + Grafana）
- ✅ IoT传感器数据
- ✅ 金融行情数据（K线图）
- ✅ 应用性能监控（APM）

### InfluxDB

#### 核心概念

```
InfluxDB 数据模型：
measurement（表） = cpu
tags（索引字段） = host=server01, region=us-west
fields（非索引字段） = usage_idle=90.5, usage_system=5.2
timestamp = 1609459200000000000

示例数据：
cpu,host=server01,region=us-west usage_idle=90.5,usage_system=5.2 1609459200000000000
cpu,host=server02,region=us-east usage_idle=85.0,usage_system=10.0 1609459200000000000
```

**重要概念**：
- **measurement**：类似表名
- **tags**：索引字段（用于快速查询）
- **fields**：数据字段（不索引）
- **timestamp**：时间戳（纳秒精度）

---

#### InfluxDB 操作

**写入数据**（Line Protocol）：
```bash
# 格式：measurement,tag1=value1,tag2=value2 field1=value1,field2=value2 timestamp
curl -XPOST 'http://localhost:8086/write?db=mydb' \
  -d 'cpu,host=server01,region=us-west usage_idle=90.5,usage_system=5.2 1609459200000000000'
```

**查询数据**（InfluxQL）：
```sql
-- 查询最近1小时的平均CPU使用率
SELECT MEAN(usage_system) 
FROM cpu 
WHERE time > now() - 1h 
  AND host = 'server01'
GROUP BY time(1m);  -- 按1分钟聚合

-- 查询结果：
time                 mean
----                 ----
2024-01-01T10:00:00Z 5.2
2024-01-01T10:01:00Z 5.5
2024-01-01T10:02:00Z 5.1
```

**降采样**（Continuous Query）：
```sql
-- 创建连续查询：每小时计算1小时的平均值
CREATE CONTINUOUS QUERY cq_cpu_hour ON mydb
BEGIN
  SELECT MEAN(usage_system) AS usage_system_mean
  INTO cpu_hour
  FROM cpu
  GROUP BY time(1h), host, region
END;

-- 自动生成降采样数据：
-- cpu_hour 表：1小时粒度的数据（存储1年）
-- cpu 表：1分钟粒度的数据（存储30天后删除）
```

**数据保留策略**（Retention Policy）：
```sql
-- 创建保留策略：数据保留30天
CREATE RETENTION POLICY rp_30d ON mydb DURATION 30d REPLICATION 1 DEFAULT;

-- 创建保留策略：数据保留1年（降采样后的数据）
CREATE RETENTION POLICY rp_1y ON mydb DURATION 365d REPLICATION 1;
```

### TimescaleDB

#### 核心特性

**定义**：TimescaleDB是PostgreSQL的扩展，将PostgreSQL变成时序数据库。

**优势**：
- ✅ 完全兼容PostgreSQL（SQL语法、生态工具）
- ✅ 支持JOIN、事务（InfluxDB不支持）
- ✅ 自动分区（Hypertable）

---

#### Hypertable（超表）

```sql
-- 创建普通表
CREATE TABLE conditions (
  time TIMESTAMPTZ NOT NULL,
  location TEXT NOT NULL,
  temperature DOUBLE PRECISION NULL
);

-- 转换为超表（自动按时间分区）
SELECT create_hypertable('conditions', 'time');

-- TimescaleDB 自动创建分区（chunk）：
-- conditions_1: 2024-01-01 00:00:00 ~ 2024-01-08 00:00:00
-- conditions_2: 2024-01-08 00:00:00 ~ 2024-01-15 00:00:00
-- conditions_3: 2024-01-15 00:00:00 ~ 2024-01-22 00:00:00

-- 插入数据（与普通表相同）
INSERT INTO conditions VALUES (now(), 'Beijing', 25.5);

-- 查询数据（自动选择相关分区）
SELECT * FROM conditions
WHERE time > now() - INTERVAL '1 hour'
  AND location = 'Beijing';
```

---

#### 连续聚合（Continuous Aggregates）

```sql
-- 创建连续聚合视图（类似物化视图）
CREATE MATERIALIZED VIEW conditions_hourly
WITH (timescaledb.continuous) AS
SELECT 
  time_bucket('1 hour', time) AS bucket,
  location,
  AVG(temperature) AS avg_temp,
  MAX(temperature) AS max_temp,
  MIN(temperature) AS min_temp
FROM conditions
GROUP BY bucket, location;

-- 自动刷新策略
SELECT add_continuous_aggregate_policy('conditions_hourly',
  start_offset => INTERVAL '3 hours',  -- 从3小时前开始聚合
  end_offset => INTERVAL '1 hour',     -- 到1小时前结束
  schedule_interval => INTERVAL '1 hour');  -- 每小时执行一次

-- 查询聚合数据（速度快）
SELECT * FROM conditions_hourly
```

## 向量数据库

### 向量数据库原理

#### 为什么需要向量数据库？

**传统数据库**：
- 存储结构化数据（数字、字符串）
- 查询方式：精确匹配、范围查询

**向量数据库**：
- 存储高维向量（如768维）
- 查询方式：相似度搜索（近似最近邻，ANN）

```
示例：文本相似度搜索

文本1: "如何学习Python"
向量1: [0.12, 0.45, 0.67, ..., 0.89]  # 768维

文本2: "Python入门教程"
向量2: [0.15, 0.43, 0.65, ..., 0.87]  # 768维

相似度计算：
cosine_similarity(向量1, 向量2) = 0.92  # 高相似度
```

---

#### 向量相似度计算

**1. 欧氏距离**（L2 Distance）
```python
distance = sqrt((x1-x2)^2 + (y1-y2)^2 + ... + (zn-zn)^2)
# 距离越小，越相似
```

**2. 余弦相似度**（Cosine Similarity）
```python
similarity = (A·B) / (||A|| * ||B||)
# 范围：[-1, 1]，值越大越相似
```

**3. 点积**（Dot Product）
```python
similarity = A·B = a1*b1 + a2*b2 + ... + an*bn
# 值越大越相似（需要归一化向量）
```

---

#### ANN算法（近似最近邻）

**暴力搜索**（Brute Force）：
```python
# 计算查询向量与所有向量的距离，找Top-K
for vec in database:
  distance = calculate_distance(query_vec, vec)
# 时间复杂度：O(N * D)，N=向量数量，D=向量维度
# 问题：百万级向量，查询很慢
```

**HNSW**（Hierarchical Navigable Small World）：
```
HNSW 结构（多层图）：
Layer 2: [1] ---------> [100]
         |               |
Layer 1: [1]--[50]--[100]--[200]
         |    |     |     |
Layer 0: [1]-[10]-[50]-[100]-[150]-[200]-[250]

搜索过程：
1. 从最顶层开始
2. 找到最近的邻居，跳到下一层
3. 重复，直到最底层
4. 在最底层找到Top-K

时间复杂度：O(log N)
准确率：95%+
```

**IVF**（Inverted File Index）：
```
IVF 结构：
1. 将向量聚类成多个簇（Cluster）
2. 查询时，先找到最近的几个簇
3. 只在这些簇内搜索

Cluster 1: [vec1, vec2, vec3, ...]
Cluster 2: [vec4, vec5, vec6, ...]
Cluster 3: [vec7, vec8, vec9, ...]

时间复杂度：O(N/K)，K=簇数量
```

## NewSQL数据库

> NewSQL = 分布式架构 + 传统SQL + 强一致事务，解决传统关系型数据库水平扩展困难的问题。

### TiDB

**架构概览**：

```
TiDB 架构（计算与存储分离）：

┌──────────────────────────────────────────────────┐
│                   TiDB Server                    │
│           （无状态SQL计算层，水平扩展）              │
│         解析SQL → 生成执行计划 → 分发KV请求          │
└─────────────────────┬────────────────────────────┘
                      │
          ┌───────────┴───────────┐
          │                       │
┌─────────┴──────────┐  ┌────────┴──────────┐
│       TiKV          │  │     TiFlash        │
│  （行存储引擎）       │  │  （列存储引擎）      │
│  Raft一致性协议      │  │  实时同步TiKV数据    │
│  分布式事务(Percolator)│ │  OLAP加速          │
│  Region自动分裂/合并  │  │  MPP执行引擎        │
└─────────────────────┘  └────────────────────┘
          │
┌─────────┴──────────┐
│        PD           │
│  （Placement Driver）│
│  元数据管理          │
│  时间戳分配(TSO)     │
│  负载均衡调度        │
└─────────────────────┘
```

**核心特性**：

| 特性 | 说明 |
|------|------|
| **MySQL兼容** | 兼容MySQL协议和语法，应用迁移成本低 |
| **水平扩展** | 计算层/存储层独立扩缩容，容量弹性伸缩 |
| **强一致事务** | 基于Percolator模型实现分布式事务，支持Snapshot Isolation |
| **HTAP** | TiKV(行存OLTP) + TiFlash(列存OLAP) 同一套系统兼顾TP和AP |
| **高可用** | 数据默认3副本，Raft协议保证一致性，Region级别自动故障转移 |

**分布式事务原理（Percolator模型）**：

```
两阶段提交流程：

1. Prewrite阶段：
   - 选择一个Key作为Primary Key
   - 对所有Key加锁并写入数据到TiKV
   - 如果发现锁冲突则回滚

2. Commit阶段：
   - 向PD获取Commit TS
   - 提交Primary Key（原子操作，决定事务成败）
   - 异步提交Secondary Keys
   - 清理锁

关键点：
- Primary Key的提交是原子的 → 整个事务的原子性保证
- 使用TSO（Timestamp Oracle）保证全局一致性快照
```

**适用场景**：
- MySQL单机容量/性能瓶颈，需要水平扩展
- 需要分布式事务 + SQL的场景（如金融、电商订单）
- 需要HTAP混合负载（避免ETL同步到单独的分析库）

**核心架构**：

```
Elasticsearch 集群架构：

┌─────────────────────────────────────────────────┐
│                   ES Cluster                     │
│                                                  │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │  Node 1   │  │  Node 2   │  │  Node 3   │      │
│  │ (Master)  │  │ (Data)    │  │ (Data)    │      │
│  │           │  │           │  │           │      │
│  │ Shard P0  │  │ Shard P1  │  │ Shard P2  │      │
│  │ Shard R1  │  │ Shard R2  │  │ Shard R0  │      │
│  └──────────┘  └──────────┘  └──────────┘      │
│                                                  │
│  P = Primary Shard    R = Replica Shard          │
└─────────────────────────────────────────────────┘
```

**倒排索引原理**：

```
正排索引（Forward Index）：
  Doc1 → "Go语言是一种编译型语言"
  Doc2 → "Java是一种面向对象的语言"

倒排索引（Inverted Index）：
  "Go语言"  → [Doc1]
  "编译型"  → [Doc1]
  "Java"   → [Doc2]
  "面向对象" → [Doc2]
  "语言"   → [Doc1, Doc2]

查询 "Go语言" → 直接定位到 Doc1（O(1)查找）

底层结构：
- Term Dictionary：词项字典（排序后的所有词项）
- Term Index：词项索引（FST有限状态转换器，前缀压缩，常驻内存）
- Posting List：倒排列表（DocID列表 + 词频TF + 位置信息）
  - 使用FOR/Roaring Bitmap压缩，减少磁盘和内存占用
```

**与ClickHouse的对比**：

| 维度 | Elasticsearch | ClickHouse |
|------|---------------|------------|
| **核心能力** | 全文搜索 + 聚合分析 | 列存储OLAP分析 |
| **数据写入** | 近实时（refresh_interval 1s） | 批量写入（MergeTree后台合并） |
| **查询语言** | DSL JSON / SQL | SQL |
| **聚合分析** | 适合中等规模 | 大规模聚合极快 |
| **全文搜索** | ✅ 核心能力 | ❌ 不擅长 |
| **适用场景** | 日志分析、全文搜索、APM | BI报表、用户行为分析、实时数仓 |
| **存储成本** | 高（倒排+正排双重存储） | 低（列存压缩率高） |

**典型应用场景**：
- **日志分析**：ELK Stack（Elasticsearch + Logstash + Kibana）
- **全文搜索**：电商商品搜索、站内搜索
- **APM监控**：应用性能监控（如Elastic APM）
- **安全分析**：SIEM安全信息和事件管理

## 实战案例：多数据库混合架构设计

> 以一个中大型电商平台为例，展示如何按数据特征选型，设计多数据库协同架构。

---

### 1. 业务场景分析：按数据特征选型

电商平台的数据可以按**访问模式**和**一致性要求**划分为六大类，每类对应最合适的数据库：

| 业务场景 | 数据特征 | 选型 | 选型理由 |
|----------|----------|------|----------|
| 核心交易（订单/支付/库存） | 强一致性、高并发CRUD、事务 | **MySQL**（主从+分库分表） | ACID事务保证资金安全，InnoDB行锁支持高并发 |
| 商品搜索 | 全文检索、多条件筛选、聚合 | **Elasticsearch** | 倒排索引实现毫秒级全文搜索，支持相关性评分 |
| 用户行为/推荐分析 | 海量写入、聚合分析、宽表 | **ClickHouse** | 列存压缩+向量化执行，亿级数据秒级聚合 |
| 会话/缓存/排行榜 | 高速读写、TTL过期、排序 | **Redis** | 内存存储微秒级响应，ZSet天然支持排行榜 |
| 用户画像/标签 | 灵活Schema、嵌套结构、频繁扩展 | **MongoDB** | 文档模型无需DDL变更，标签体系随业务灵活扩展 |
| 时序监控数据 | 时间序列、高写入、自动过期 | **InfluxDB/Prometheus** | 时序压缩+保留策略，写入性能远超关系型数据库 |

**选型核心原则**：让每个数据库只做它最擅长的事，而不是让一个数据库承担所有负载。

### 3. 数据同步方案

#### 3.1 MySQL → Elasticsearch：Binlog + Canal/Debezium

```
同步链路：
  MySQL(Binlog) → Canal → Kafka → ES Consumer → Elasticsearch

Canal 工作原理：
  1. Canal伪装为MySQL从库，向主库发送dump请求
  2. 主库推送Binlog给Canal
  3. Canal解析Binlog为结构化事件（INSERT/UPDATE/DELETE）
  4. 投递到Kafka，ES消费端消费后写入ES

关键配置：
  - Kafka分区键 = 商品ID（保证同一商品的变更有序）
  - ES写入采用bulk API批量提交（提升吞吐）
  - 消费端幂等：用MySQL主键作为ES的_id，重复消费自动覆盖
```

#### 3.2 MySQL → ClickHouse：Flink CDC

```
同步链路：
  MySQL(Binlog) → Flink CDC Source → Flink Transform → ClickHouse Sink

Flink CDC 优势（相比Canal）：
  - 全量+增量一体化（自动完成历史数据快照+增量追踪）
  - Exactly-Once语义（Checkpoint机制保证不丢不重）
  - 支持实时ETL（在流中做字段映射、过滤、聚合）

ClickHouse写入注意：
  - 使用ReplacingMergeTree引擎处理UPDATE（通过版本号去重）
  - 批量写入（每批≥1000行），避免频繁小写入导致Part过多
  - 配合物化视图实现实时预聚合
```

#### 3.3 缓存一致性：Cache Aside + 延迟双删

```
Cache Aside 标准流程：
  读：先查Redis → 命中返回 → 未命中查MySQL → 写入Redis → 返回
  写：先更新MySQL → 再删除Redis缓存

延迟双删（解决并发导致的脏缓存）：
  1. 删除Redis缓存
  2. 更新MySQL
  3. 延迟500ms（等待可能的并发读回填旧值）
  4. 再次删除Redis缓存

伪代码：
  func UpdateProduct(id, data):
      redis.Del(key)              // 第一次删除
      mysql.Update(id, data)      // 更新DB
      go func():                  // 异步延迟双删
          time.Sleep(500ms)
          redis.Del(key)          // 第二次删除

为什么不用"先更新缓存再更新DB"：
  - 两个并发写入可能导致缓存和DB数据不一致
  - 删除缓存是幂等操作，更安全
```

### 4. Go 代码示例：统一数据访问层（Repository Pattern）

```go
package repository

import (
    "context"
    "encoding/json"
    "fmt"
    "time"

    "github.com/go-redis/redis/v8"
    "github.com/olivere/elastic/v7"
    "go.mongodb.org/mongo-driver/mongo"
    "gorm.io/gorm"
)

// ==================== 领域模型 ====================

type Product struct {
    ID          int64     `json:"id" gorm:"primaryKey"`
    Name        string    `json:"name"`
    Description string    `json:"description"`
    Price       int64     `json:"price"`       // 单位：分
    Stock       int32     `json:"stock"`
    CategoryID  int64     `json:"category_id"`
    Tags        []string  `json:"tags" gorm:"-"` // ES/Mongo中使用
    UpdatedAt   time.Time `json:"updated_at"`
}

type ProductSearchResult struct {
    Products   []Product `json:"products"`
    Total      int64     `json:"total"`
    TookMs     int64     `json:"took_ms"`
}

// ==================== 统一接口定义 ====================

// ProductRepository 统一数据访问接口
// 上层业务只依赖接口，不关心底层用了几个数据库
type ProductRepository interface {
    // 核心交易操作（MySQL）
    GetByID(ctx context.Context, id int64) (*Product, error)
    Create(ctx context.Context, p *Product) error
    UpdateStock(ctx context.Context, id int64, delta int32) error

    // 搜索操作（Elasticsearch）
    Search(ctx context.Context, keyword string, categoryID int64,
        minPrice, maxPrice int64, page, size int) (*ProductSearchResult, error)

    // 缓存操作（Redis）
    GetCached(ctx context.Context, id int64) (*Product, error)
    InvalidateCache(ctx context.Context, id int64) error
}

// ==================== 多数据源实现 ====================

type productRepo struct {
    db    *gorm.DB          // MySQL
    rdb   *redis.Client     // Redis
    es    *elastic.Client   // Elasticsearch
    mongo *mongo.Collection // MongoDB（用户画像等场景扩展）
}

func NewProductRepository(
    db *gorm.DB,
    rdb *redis.Client,
    es *elastic.Client,
    mongoColl *mongo.Collection,
) ProductRepository {
    return &productRepo{db: db, rdb: rdb, es: es, mongo: mongoColl}
}

// ---------- MySQL：核心交易数据 ----------

func (r *productRepo) GetByID(ctx context.Context, id int64) (*Product, error) {
    var p Product
    if err := r.db.WithContext(ctx).First(&p, id).Error; err != nil {
        return nil, fmt.Errorf("mysql get product %d: %w", id, err)
    }
    return &p, nil
}

func (r *productRepo) Create(ctx context.Context, p *Product) error {
    return r.db.WithContext(ctx).Create(p).Error
}

// UpdateStock 使用乐观锁扣减库存，防止超卖
func (r *productRepo) UpdateStock(ctx context.Context, id int64, delta int32) error {
    result := r.db.WithContext(ctx).
        Model(&Product{}).
        Where("id = ? AND stock + ? >= 0", id, delta).
        Update("stock", gorm.Expr("stock + ?", delta))
    if result.RowsAffected == 0 {
        return fmt.Errorf("stock not enough or product %d not found", id)
    }
    return result.Error
}

// ---------- Redis：缓存层（Cache Aside） ----------

func (r *productRepo) cacheKey(id int64) string {
    return fmt.Sprintf("product:%d", id)
}

func (r *productRepo) GetCached(ctx context.Context, id int64) (*Product, error) {
    // 1. 先查缓存
    val, err := r.rdb.Get(ctx, r.cacheKey(id)).Result()
    if err == nil {
        var p Product
        if json.Unmarshal([]byte(val), &p) == nil {
            return &p, nil // 缓存命中
        }
    }

    // 2. 缓存未命中，查MySQL
    p, err := r.GetByID(ctx, id)
    if err != nil {
        return nil, err
    }

    // 3. 回填缓存（TTL 10分钟，防止缓存雪崩加随机偏移）
    data, _ := json.Marshal(p)
    ttl := 10*time.Minute + time.Duration(id%60)*time.Second
    r.rdb.Set(ctx, r.cacheKey(id), data, ttl)

    return p, nil
}

// InvalidateCache 延迟双删
func (r *productRepo) InvalidateCache(ctx context.Context, id int64) error {
    key := r.cacheKey(id)
    r.rdb.Del(ctx, key) // 第一次删除
    go func() {
        time.Sleep(500 * time.Millisecond)
        r.rdb.Del(context.Background(), key) // 延迟第二次删除
    }()
    return nil
}

// ---------- Elasticsearch：商品搜索 ----------

func (r *productRepo) Search(ctx context.Context, keyword string,
    categoryID int64, minPrice, maxPrice int64, page, size int,
) (*ProductSearchResult, error) {
    query := elastic.NewBoolQuery()

    // 全文搜索（商品名+描述）
    if keyword != "" {
        query.Must(elastic.NewMultiMatchQuery(keyword, "name", "description").
            Type("best_fields").
            Fuzziness("AUTO"))
    }

    // 分类过滤
    if categoryID > 0 {
        query.Filter(elastic.NewTermQuery("category_id", categoryID))
    }

    // 价格范围
    if minPrice > 0 || maxPrice > 0 {
        rangeQ := elastic.NewRangeQuery("price")
        if minPrice > 0 {
            rangeQ.Gte(minPrice)
        }
        if maxPrice > 0 {
            rangeQ.Lte(maxPrice)
        }
        query.Filter(rangeQ)
    }

    // 执行搜索
    result, err := r.es.Search().
        Index("products").
        Query(query).
        From((page - 1) * size).
        Size(size).
        SortBy(elastic.NewScoreSort()).     // 相关性排序
        SortBy(elastic.NewFieldSort("updated_at").Desc()). // 次排序
        Do(ctx)
    if err != nil {
        return nil, fmt.Errorf("es search: %w", err)
    }

    // 解析结果
    products := make([]Product, 0, len(result.Hits.Hits))
    for _, hit := range result.Hits.Hits {
        var p Product
        if err := json.Unmarshal(hit.Source, &p); err == nil {
            products = append(products, p)
        }
    }

    return &ProductSearchResult{
        Products: products,
        Total:    result.TotalHits(),
        TookMs:   result.TookInMillis,
    }, nil
}
```

### 6. 常见坑和解决方案

#### 坑1：数据不一致（MySQL和ES数据不同步）

```
现象：
  用户修改了商品价格，MySQL已更新，但ES中搜索出来的还是旧价格。

根因：
  - Canal/Debezium同步延迟或消费堆积
  - ES的refresh_interval导致写入后不可立即搜索
  - 消费端异常导致部分消息丢失

解决方案：
  1. 监控Kafka消费Lag，设置告警阈值（如Lag > 10000）
  2. 定时对账任务：每小时比对MySQL和ES的关键字段（价格/库存/状态）
  3. 补偿机制：发现不一致时，触发全量/增量重建
  4. 关键操作双写兜底：价格变更时，同步调ES更新API（降级方案）
```

#### 坑2：缓存与数据库不一致

```
现象：
  库存已扣减为0，但缓存中还显示有库存，导致超卖。

根因：
  - 并发场景下，读请求在「删除缓存」和「更新DB」之间回填了旧值
  - 延迟双删的延迟时间不够，或异步删除失败

解决方案：
  1. 库存等资金敏感数据不走缓存，直接查MySQL（或用Redis原子操作）
  2. 延迟双删的第二次删除走消息队列，失败可重试
  3. 缓存TTL设置合理上限（如10分钟），即使不一致也会自动修复
  4. 核心链路使用Redis Lua脚本保证原子性：
     -- 扣减库存Lua脚本
     local stock = redis.call('GET', KEYS[1])
     if tonumber(stock) >= tonumber(ARGV[1]) then
         return redis.call('DECRBY', KEYS[1], ARGV[1])
     end
     return -1
```

#### 坑3：同步延迟导致业务问题

```
现象：
  用户下单后立刻搜索"我的订单"，搜不到（因为ES同步有延迟）。

根因：
  MySQL → Kafka → ES 链路正常延迟1-3秒，高峰期可能到10秒+

解决方案：
  1. 读写分离策略：写入后短时间内强制读主库（Session级别的读写路由）
  2. 前端乐观展示：下单成功后前端直接展示该订单，不依赖搜索结果
  3. "我的订单"查询走MySQL而非ES（这是精确查询，不需要全文搜索）
  4. 区分场景：列表搜索可以接受延迟，详情页必须走主库
```

#### 坑4：运维复杂度爆炸

```
现象：
  6个数据库 + 3条同步链路，任何一个组件故障都可能导致数据问题。

根因：
  - 每个数据库有独立的运维体系（备份/监控/扩容/升级）
  - 同步链路的监控和告警不完善
  - 团队对部分数据库不熟悉

解决方案：
  1. 统一可观测性：所有数据库接入统一监控平台（Grafana），关键指标：
     - MySQL: QPS/慢查询/主从延迟/连接数
     - ES: 索引延迟/搜索延迟/JVM内存/分片状态
     - ClickHouse: 查询耗时/MergeTree合并速度/磁盘使用
     - Redis: 命中率/内存使用/大Key/慢命令
     - Kafka: 消费Lag/分区数/ISR状态
  2. 渐进式引入：不要一次上6个数据库，按优先级分阶段
     Phase 1: MySQL + Redis（覆盖80%需求）
     Phase 2: + Elasticsearch（搜索需求明确后）
     Phase 3: + ClickHouse（数据量达到MySQL分析瓶颈后）
  3. 基础设施即代码：用Terraform/Helm管理所有数据库的部署和配置
  4. 混沌工程：定期模拟同步链路中断，验证降级和恢复能力
```

#### 坑5：数据孤岛和口径不一致

```
现象：
  MySQL中的订单金额和ClickHouse中的GMV报表对不上。

根因：
  - 同步链路丢数据（Kafka消费异常、ClickHouse写入失败）
  - ETL逻辑与业务逻辑不一致（如退款的处理方式不同）
  - 时间窗口不对齐（MySQL按事务提交时间，ClickHouse按事件时间）

解决方案：
  1. 建立数据血缘：记录每个字段从源到目标的完整转换链路
  2. 自动化对账：
     - 每日T+1全量对账（MySQL SUM vs ClickHouse SUM）
     - 实时抽样校验（每分钟随机抽10条比对）
  3. 统一时间语义：所有系统使用事件发生时间（event_time），而非处理时间
  4. 在数据仓库层建立统一指标定义（指标字典），避免各系统各自计算
```

## 面试题自查

### 数据平台治理补充

- 数据库选型不是一次性决策，而是随着访问模式变化持续演进的架构能力。
- 交易、搜索、分析、向量召回要有明确分工和数据同步路径，避免“一个库承载所有语义”。
- 除了性能，还要同时评估一致性、运维复杂度、团队技能和成本。

### 初级高频速刷

**1. 关系型数据库和 NoSQL 的核心区别是什么？**

关系型数据库强调结构化模式、事务和 SQL 能力；NoSQL 更强调灵活模型、水平扩展和针对特定场景优化。真正的差异不是“一个先进一个落后”，而是数据模型和 trade-off 不同。

**2. 主从复制、分片、副本三者分别在解决什么问题？**

主从复制解决读扩展和故障切换，分片解决单节点容量与吞吐瓶颈，多副本解决数据可靠性和高可用。很多人把这三个词混为一谈，这在数据库综合面试里是很容易被追问的点。

**3. 什么是多模型数据库，为什么现实系统里经常不是“一库走天下”？**

多模型数据库是指针对不同数据模型和访问模式选择不同存储引擎，例如关系型、文档、列式、搜索、向量、时序。现实系统里经常按场景分工，因为交易、搜索、分析、推荐的访问模式完全不同，很难让一个数据库把所有事情都做到最好。

**4. 为什么数据库选型不能只看“支持不支持”，还要看访问模式？**

因为很多数据库在功能上都“能做”，但性能路径和成本完全不同。一个系统每天是高频点查、批量写、全文搜索、聚合分析还是向量召回，决定了底层索引、压缩、事务和扩展方式的取舍。

**5. 为什么很多架构会把交易、搜索、分析拆到不同数据库？**

因为交易需要强事务和低延迟写入，搜索需要倒排索引和相关性排序，分析需要列存和聚合扫描。把这三类负载硬压到一个数据库里，通常意味着每类场景都做得不够好。

### 中级高频进阶

**6. 为什么文档数据库适合“聚合根”式建模，而关系库更适合强事务拆分？**

因为文档数据库鼓励把一组经常一起读写的数据放进同一文档中，减少跨对象 join；关系库则更擅长通过规范化建模和事务约束来维护复杂一致性。两者不是谁替代谁，而是建模哲学不同。

**7. 跨数据库（交易库 + 搜索库 + 分析库）时，数据链路应该怎么设计才不失控？**

核心是把“同步路径、延迟目标、补偿机制”先定义清楚，而不是先把库建出来。常见做法是交易库作为事实源，通过 CDC/MQ 同步到搜索和分析系统，并为链路配置重放、对账和回补能力，避免故障后只能人工补数据。

**8. 向量数据库为什么经常和关系型数据库一起出现，而不是单独落地？**

因为向量数据库擅长“找相似”，但不擅长管理完整业务实体、事务和复杂过滤条件。实际系统里常见做法是向量库负责召回候选集，关系型或搜索系统负责元数据过滤、排序和最终展示。

**9. 为什么“先定选型、再想数据链路”通常会踩坑？**

因为多数据库系统真正的难点往往不在单库性能，而在数据如何同步、延迟如何控制、出错如何补偿。只谈单库能力不谈数据链路，最终很容易在一致性和运维上失控。

**10. 为什么数据库平台要做“按场景分层”，而不是所有业务直接连主库？**

因为在线交易、报表分析、搜索检索、模型召回的负载形态完全不同。按场景分层可以把冲突流量隔离开，避免分析查询拖慢交易链路，也让容量规划和故障定位更清晰。

### 深入问答

#### 1. PostgreSQL的MVCC和MySQL的MVCC有什么区别？各有什么优缺点？

**答**：PostgreSQL通过在元组头部记录 `xmin/xmax` 事务ID实现多版本，旧版本直接存储在堆表中，需要VACUUM清理死元组；MySQL(InnoDB)通过undo log版本链存储旧版本，用Read View判断可见性。PostgreSQL的优点是实现简单、回滚快（不需要回放undo），缺点是表膨胀需要定期VACUUM；MySQL的优点是主表不膨胀，缺点是长事务会导致undo log链过长。

---

#### 2. PostgreSQL支持哪些索引类型？分别适合什么场景？

**答**：**B-Tree**：默认索引，适合等值/范围查询；**Hash**：仅等值查询，PG10+才支持WAL（之前不推荐）；**GIN（通用倒排索引）**：适合全文搜索、JSONB字段、数组包含查询；**GiST（通用搜索树）**：适合地理空间（PostGIS）、范围类型、几何数据；**BRIN（块范围索引）**：适合物理有序的大表（如时间序列），索引极小。

---

#### 3. 为什么列存储适合OLAP？行存储和列存储的本质区别是什么？

**答**：行存储按行连续存储，一次IO读取完整行，适合事务型的整行读写（OLTP）；列存储按列连续存储，一次IO只读需要的列。OLAP场景通常只查询少数几列但扫描大量行，列存储减少了90%以上的无效IO。此外，同列数据类型一致、值域相似，压缩率极高（10:1甚至100:1），进一步减少IO。列存储还天然适合SIMD向量化执行，CPU并行处理效率更高。

---

#### 4. ClickHouse的MergeTree引擎原理是什么？为什么查询性能极高？

**答**：MergeTree是ClickHouse核心引擎，写入时按主键排序生成不可变的数据Part，后台异步合并小Part为大Part（类似LSM-Tree思想）。高性能原因：①列存储减少IO；②主键稀疏索引（每8192行一个标记），快速定位数据范围；③数据压缩（LZ4/ZSTD），减少磁盘和内存开销；④向量化执行引擎，利用SIMD批量处理数据；⑤分区裁剪（Partition Pruning），按分区键跳过无关数据。

---

#### 5. HBase的Row Key如何设计？设计不当会有什么后果？

**答**：Row Key设计原则：①**避免热点**：不要用时间戳或单调递增ID作前缀（导致写入集中在单个Region），可用Hash/反转/加盐前缀打散；②**满足查询**：Row Key是唯一索引，常用查询条件应编码进Row Key；③**长度适中**：建议10-100字节，过长浪费存储，过短信息不足。设计不当的后果：热点问题导致单台RegionServer负载过高，集群资源利用不均，严重时写入TPS断崖式下降。

---

#### 6. 时序数据库（InfluxDB/TimescaleDB）相比MySQL存储时序数据有什么优势？

**答**：①**写入性能**：时序DB针对追加写入优化（LSM-Tree/批量写入），MySQL随机B+Tree插入性能差；②**压缩率**：时序DB利用时间戳的单调递增特性做Delta/Delta-of-Delta编码，压缩率可达10:1以上；③**自动过期**：内建保留策略（Retention Policy），自动删除过期数据，MySQL需要手动DELETE；④**降采样**：内建连续查询/聚合（如1秒→1分钟），MySQL需要额外ETL；⑤**查询优化**：原生支持时间范围查询、窗口函数、插值填充等时序特有操作。

---

#### 7. 向量数据库中HNSW和IVF两种ANN索引算法有什么区别？如何选择？

**答**：**HNSW（分层可导航小世界图）**：构建多层图结构，查询时从顶层逐层向下搜索，优点是查询速度极快（O(log N)）且召回率高，缺点是内存占用大（需要存储图结构）、构建速度慢。**IVF（倒排文件索引）**：将向量空间划分为多个聚类（Voronoi Cell），查询时只搜索最近的几个聚类，优点是内存占用小、支持磁盘索引，缺点是召回率略低于HNSW。选择建议：数据量小（<1000万）且对延迟要求极高→HNSW；数据量大（亿级）且需要控制成本→IVF+PQ（乘积量化）。

---

#### 8. MongoDB的文档模型什么时候该用嵌入（Embedding）什么时候该用引用（Reference）？

**答**：**用嵌入**：①数据天然是一对一或一对少量（如用户+地址）；②子文档总是和父文档一起查询（减少JOIN）；③子文档不会独立增长到很大。**用引用**：①一对多且"多"的数量很大或无上限（如用户+订单）；②子数据需要被多个父文档共享（如标签、分类）；③子文档需要独立查询和更新。关键考量：MongoDB单文档16MB限制，嵌入过深会导致文档膨胀和更新效率下降。

---

#### 9. MongoDB如何选择分片键（Shard Key）？有哪些常见的坑？

**答**：分片键选择原则：①**高基数**（Cardinality）：分片键值域要足够大，避免数据集中在少数chunk；②**写分散**：避免单调递增键（如ObjectId/时间戳）作分片键，否则写入热点集中在尾部chunk；③**查询对齐**：常用查询条件包含分片键，避免scatter-gather查询。常见方案：Hash分片（写均匀但范围查询差）、复合分片键（如`{user_id: 1, timestamp: 1}`兼顾均匀和范围）。常见坑：①分片键一旦设定不可更改；②低基数分片键导致jumbo chunk无法迁移。

---

#### 10. OLTP和OLAP有什么本质区别？为什么不能用MySQL直接做分析？

**答**：OLTP面向事务处理，特点是高并发、小数据量、随机读写、强一致性；OLAP面向数据分析，特点是低并发、大数据量、顺序扫描、聚合计算。MySQL不适合OLAP的原因：①行存储做全表扫描浪费IO（读取大量无用列）；②B+Tree索引对大范围扫描无优化；③没有向量化执行和列式压缩；④单机计算能力有限，无法水平扩展分析负载；⑤大查询会阻塞OLTP事务。

---

#### 11. TiDB的架构是怎样的？它如何实现分布式事务？

**答**：TiDB三大组件：**TiDB Server**（无状态SQL计算层，兼容MySQL协议）、**TiKV**（分布式KV存储，Raft一致性，数据按Region分片）、**PD**（元数据管理+时间戳分配+负载调度）。分布式事务采用Percolator模型（两阶段提交）：Prewrite阶段对所有Key加锁并写入数据，选一个Primary Key；Commit阶段原子提交Primary Key（决定事务成败），然后异步提交Secondary Keys。通过PD分配全局时间戳（TSO）保证一致性快照。

---

#### 12. Elasticsearch的倒排索引原理是什么？为什么全文搜索比MySQL快？

**答**：倒排索引将"词项→文档列表"建立映射。底层由三部分组成：Term Index（FST前缀树，常驻内存，快速定位Term Dictionary的位置）、Term Dictionary（排序的词项字典，存储在磁盘）、Posting List（每个词项对应的文档ID列表+词频+位置信息，用Frame of Reference/Roaring Bitmap压缩）。MySQL的B+Tree索引对 `LIKE '%keyword%'` 无法使用索引导致全表扫描；ES通过倒排索引可以O(1)定位到包含关键词的所有文档，且支持分词、模糊匹配、相关性评分等全文搜索能力。

---

#### 13. Elasticsearch和ClickHouse都能做日志分析，如何选择？

**答**：**选ES**：需要全文搜索（关键词检索、模糊匹配）、日志文本分析、实时告警、APM场景；配合Kibana提供强大的可视化和日志探索能力。**选ClickHouse**：侧重结构化数据的聚合统计（如PV/UV、TopN、趋势分析）、数据量极大需要高压缩率、查询以SQL聚合为主。核心区别：ES擅长"搜"（找到特定日志），ClickHouse擅长"算"（统计分析指标）。实际中常两者配合使用：ES做实时日志搜索，ClickHouse做离线统计分析。

---

#### 14. Prometheus适合什么场景？它和InfluxDB有什么区别？

**答**：Prometheus是拉取（Pull）模式的监控时序数据库，最适合基础设施和应用监控告警。区别：①**数据模型**：Prometheus是标签化指标模型（Metric + Labels），InfluxDB是Measurement+Tag+Field更灵活；②**采集方式**：Prometheus主动拉取（Pull），InfluxDB被动接收推送（Push）；③**查询语言**：Prometheus用PromQL（函数式），InfluxDB用Flux/InfluxQL（类SQL）；④**长期存储**：Prometheus本地存储有限，长期存储需要Thanos/Cortex；InfluxDB原生支持保留策略和降采样。选择建议：监控告警场景→Prometheus；IoT/业务时序数据存储分析→InfluxDB/TimescaleDB。

---

#### 15. 什么是NewSQL数据库？它解决了什么问题？TiDB和CockroachDB有什么区别？

**答**：NewSQL = 传统SQL能力（事务+SQL语法）+ 分布式架构（水平扩展+高可用），解决传统关系型数据库无法水平扩展的问题，同时避免NoSQL放弃SQL和事务的代价。TiDB vs CockroachDB：①兼容协议不同（TiDB→MySQL，CockroachDB→PostgreSQL）；②TiDB支持HTAP（TiFlash列存引擎），CockroachDB没有内置OLAP加速；③CockroachDB支持Serializable隔离，TiDB默认Snapshot Isolation；④CockroachDB核心卖点是全球分布式部署（Geo-Partitioning），TiDB在国内生态更好。

---

#### 16. 如何在PostgreSQL和MySQL之间做技术选型？

**答**：**选PostgreSQL**：①需要复杂SQL（递归CTE、窗口函数、lateral join）；②需要JSONB存储和查询；③GIS空间数据（PostGIS）；④数据完整性要求高（Check/Exclusion约束丰富）；⑤需要自定义类型/函数/扩展能力。**选MySQL**：①简单CRUD为主的高并发Web应用；②团队MySQL经验丰富；③生态成熟（中间件多：ProxySQL、MyCat、Canal等）；④互联网公司主流选型，招聘容易。本质区别：PostgreSQL更"学院派"功能全面，MySQL更"工程派"简单高效。

---

#### 17. 什么是Lambda架构？在多数据库协同中如何应用？

**答**：Lambda架构将数据处理分为三层：**批处理层**（Batch Layer）用Hadoop/Spark处理全量历史数据生成批视图；**速度层**（Speed Layer）用Flink/ClickHouse处理实时增量数据生成实时视图；**服务层**（Serving Layer）合并批视图和实时视图对外提供查询。多数据库协同示例：MySQL(OLTP事务) → Kafka(数据管道) → ClickHouse(实时分析) + Hadoop(离线分析) → Redis(查询缓存) → 应用层。缺点是维护两套处理逻辑复杂，Kappa架构（纯流处理）是其简化替代方案。

---

#### 18. 向量数据库中的量化（Quantization）技术是什么？为什么重要？

**答**：量化是将高精度向量（如float32）压缩为低精度表示（如int8/binary）的技术，目的是减少内存占用和加速距离计算。常见方法：**标量量化（SQ）**：将每个维度从float32量化为int8，内存减少4倍；**乘积量化（PQ）**：将向量切分为多个子向量分别量化，极大压缩（如128维float32→16字节）；**二值量化**：每维只保留0/1，用汉明距离加速。量化的重要性：亿级向量的原始存储需要几百GB内存，量化后可以缩减到几十GB，使得大规模向量检索在有限硬件资源下成为可能，代价是损失少量召回精度。

---

#### 19. Oracle RAC和TiDB相比，在高可用方面有什么区别？

**答**：**Oracle RAC**（Real Application Clusters）是共享存储架构（Shared-Everything），多个计算节点共享同一份SAN存储，通过Cache Fusion在节点间同步缓存。优点是强一致、成熟稳定；缺点是存储是单点瓶颈、硬件成本极高（专用SAN/光纤）、扩展能力有限（通常2-4节点）。**TiDB**是Shared-Nothing架构，每个TiKV节点存储独立的数据分片（Region），通过Raft协议保证3副本一致性。优点是水平扩展能力强、廉价硬件、Region级故障自愈；缺点是分布式事务有额外延迟、SQL兼容性不如Oracle完整。

---

#### 20. 多数据库架构里如何做数据一致性治理？

**答**：核心是分层定义一致性目标，而不是追求全链路强一致。交易主链路通常以主库事务为准，搜索/分析/推荐链路通过 CDC 或消息异步同步，接受秒级到分钟级延迟。工程上要配套三件事：①链路级延迟和丢失监控；②对账与补偿机制；③关键字段的幂等更新策略。没有这三件事，多库架构很容易演变成“数据看起来在流动，但没人能保证正确”。

---

#### 21. 如何设计一条可观测的数据同步链路（MySQL -> Kafka -> ES/ClickHouse）？

**答**：至少要观测四类指标：采集端（binlog 位点推进）、传输端（Kafka lag、失败重试）、消费端（写入成功率、延迟分位）、业务端（查询新鲜度）。并且要让每条数据可追踪到源位点和目标版本，出现数据缺口时能快速定位是采集、传输还是落库阶段出问题。只监控“服务活着”不监控“数据新鲜度”，对数据平台来说等于盲飞。

---

#### 22. 向量检索上线后，如何平衡召回率、延迟和成本？

**答**：这是典型三角约束。提高召回率通常意味着更大的索引和更高的查询开销；压低延迟通常要牺牲部分召回；控制成本则需要量化压缩和分层存储。常见做法是分层检索：先向量库做快速粗召回，再在关系库/搜索层做精排和过滤，同时按业务场景设置不同的召回阈值和 SLA，而不是全站统一一套参数。

---

#### 23. 数据库选型评审时，如何避免“功能对比表很好看，落地却失败”？

**答**：评审不能只看功能清单，要做带负载模型的 PoC：真实数据规模、真实查询模式、真实写入峰值、真实故障演练。并把迁移成本、运维人力、生态工具链、回滚路径纳入同一张决策表。很多失败选型都不是功能不够，而是迁移和治理成本被低估。

---

#### 24. 如果你来设计一个电商系统的数据库架构，会怎么选型？

**答**：**核心交易**（订单/支付/库存）：MySQL主从集群（强事务+高并发CRUD），数据量大时考虑分库分表或迁移TiDB；**商品搜索**：Elasticsearch（全文搜索+多条件筛选+相关性排序）；**用户行为分析**：ClickHouse（PV/UV/转化率等聚合分析）；**推荐系统**：Milvus向量数据库（用户/商品Embedding相似度检索）；**缓存层**：Redis（热点商品/购物车/会话）；**监控告警**：Prometheus + Grafana。数据流转：MySQL → Canal → Kafka → ES/ClickHouse/数据湖。关键原则：核心OLTP用关系型数据库保证事务，分析/搜索/缓存各用专业工具，通过消息队列异步同步。

---

#### 25. 分库分表后，跨分片分页和排序你怎么设计，才能不把数据库拖垮？

**答**：
跨分片最怕“全分片扫 + 内存归并 + 深分页”。可执行设计通常是：
1. **优先按分片键查询**，尽量把请求收敛到少量分片。
2. **用游标分页替代 offset 深分页**，降低扫描代价。
3. **排序字段前置索引化**，避免每分片大量临时排序。
4. **汇总层限流与超时**，超过阈值降级为近似结果或异步导出。

核心原则：把“强一致全局有序分页”当作昂贵能力，仅给必要场景开放。

---

#### 26. CDC 链路里 schema 变更（加列/改类型）如何做到不停机演进？

**答**：
不停机演进要走“前向兼容 + 双读写过渡”：
1. 先做兼容发布：消费者先支持新旧字段并行解析。
2. 再做 schema 变更：加列优先、改类型尽量走新列迁移而不是原地替换。
3. 过渡期双写：生产端同时写旧字段和新字段，确保下游平滑切换。
4. 最后清理：确认全链路完成切换后再下线旧字段。

同时要在 CDC 平台加 schema registry/版本校验，防止不兼容变更直接冲垮消费端。

### 开放式设计题

**D1：如果让你为一个初创公司选技术栈，从0到100万用户，数据库选型策略是什么？**

**参考思路**：
- 0-1万用户：单机PostgreSQL搞定一切（JSONB支持灵活Schema、全文搜索够用）
- 1-10万：加Redis缓存热点数据、读写分离（1主2从）、开始引入ES做搜索
- 10-100万：分库分表（垂直拆分先行）、引入ClickHouse做分析报表、MongoDB处理非结构化数据
- 关键原则：单一数据库坚持到不得不拆、每次只加一个新组件、同步链路比数据库本身更容易出问题
- 避免过早优化：不要Day 1就上微服务+10种数据库，运维成本会压垮小团队

**D2：ClickHouse查询突然变慢，从秒级劣化到分钟级，排查思路是什么？**

**参考思路**：
- system.query_log分析慢查询、EXPLAIN查看执行计划
- 常见原因：数据倾斜（某个分区数据过多）、Join大表（ClickHouse不擅长Join）、内存不足溢出到磁盘
- 优化方向：物化视图预聚合、合理分区键（避免过多小分区）、Join改为字典表或预宽表
