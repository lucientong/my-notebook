## 1. SOLID 原则 ⭐⭐⭐

> SOLID 是面向对象/面向接口设计的五大基础原则，在 Go 的接口与组合体系中同样适用。

### 1.1 单一职责原则（SRP — Single Responsibility Principle）

**原理**：一个模块/结构体应该只有一个引起它变化的原因。如果一个结构体同时承担「业务逻辑」和「持久化」两项职责，那么任一变化都会迫使修改该结构体，导致耦合。

**UML 文字描述**：
```
[反例] UserService 类同时持有 sendEmail() 和 saveToDB() 方法
         ┌──────────────────┐
         │   UserService    │
         ├──────────────────┤
         │ + CreateUser()   │
         │ + SendEmail()    │ ← 职责 1: 通知
         │ + SaveToDB()     │ ← 职责 2: 持久化
         └──────────────────┘

[正例] 拆分为 UserService / EmailNotifier / UserRepository
         ┌──────────────┐    ┌───────────────┐    ┌────────────────┐
         │ UserService   │───▶│ EmailNotifier │    │ UserRepository │
         │ + CreateUser()│───▶│ + Send()      │    │ + Save()       │
         └──────────────┘    └───────────────┘    └────────────────┘
```

**Go — 反例**：
```go
// BAD: UserService 承担了发邮件 + 存储两项职责
type UserService struct {
    db   *sql.DB
    smtp *smtp.Client
}

func (s *UserService) CreateUser(name, email string) error {
    // 1. 存储用户
    _, err := s.db.Exec("INSERT INTO users(name,email) VALUES(?,?)", name, email)
    if err != nil {
        return err
    }
    // 2. 发送欢迎邮件 — 与用户创建逻辑强耦合
    return s.smtp.Mail(email)
}
```

**Go — 正例**：
```go
// GOOD: 拆分为独立职责

// UserRepository 只管持久化
type UserRepository struct {
    db *sql.DB
}

func (r *UserRepository) Save(name, email string) error {
    _, err := r.db.Exec("INSERT INTO users(name,email) VALUES(?,?)", name, email)
    return err
}

// Notifier 接口 — 只管通知
type Notifier interface {
    Notify(email string) error
}

type EmailNotifier struct {
    smtp *smtp.Client
}

func (n *EmailNotifier) Notify(email string) error {
    return n.smtp.Mail(email)
}

// UserService 编排流程
type UserService struct {
    repo     *UserRepository
    notifier Notifier
}

func (s *UserService) CreateUser(name, email string) error {
    if err := s.repo.Save(name, email); err != nil {
        return err
    }
    return s.notifier.Notify(email)
}
```

**框架应用**：Gin 的 `HandlerFunc` 只处理 HTTP 请求转换，业务逻辑交给 Service 层，持久化交给 Repository 层 — 典型三层分离。

### 1.2 开放封闭原则（OCP — Open/Closed Principle）

**原理**：软件实体应对扩展开放、对修改关闭。通过接口/多态实现新功能，而无需修改已有代码。

**Go — 反例**：
```go
// BAD: 每新增一种日志输出都要修改 Log 函数
func Log(target string, msg string) {
    switch target {
    case "console":
        fmt.Println(msg)
    case "file":
        os.WriteFile("app.log", []byte(msg), 0644)
    // 新增 "kafka" ? 必须改这个 switch
    }
}
```

**Go — 正例**：
```go
// GOOD: 定义 Logger 接口, 新增输出方式只需新增实现

type Logger interface {
    Log(msg string) error
}

type ConsoleLogger struct{}

func (l *ConsoleLogger) Log(msg string) error {
    fmt.Println(msg)
    return nil
}

type FileLogger struct {
    Path string
}

func (l *FileLogger) Log(msg string) error {
    return os.WriteFile(l.Path, []byte(msg), 0644)
}

// 新增 KafkaLogger 不需要修改任何已有代码
type KafkaLogger struct {
    Producer sarama.SyncProducer
    Topic    string
}

func (l *KafkaLogger) Log(msg string) error {
    _, _, err := l.Producer.SendMessage(&sarama.ProducerMessage{
        Topic: l.Topic,
        Value: sarama.StringEncoder(msg),
    })
    return err
}

// 使用方依赖接口
func ProcessOrder(logger Logger, orderID string) {
    // ...
    logger.Log("order processed: " + orderID)
}
```

**框架应用**：gRPC 拦截器（Interceptor）机制：框架核心代码不变，用户通过注册拦截器扩展 RPC 的前/后处理逻辑。

---

### 1.3 里氏替换原则（LSP — Liskov Substitution Principle）

**原理**：子类型必须能替换其基类型而不破坏程序的正确性。在 Go 中体现为：实现接口的结构体必须完全遵守接口的语义契约，不仅仅是签名匹配。

**Go — 反例**：
```go
// BAD: ReadOnlyFile 实现了 io.Writer 但违反了语义
type ReadOnlyFile struct{}

func (f *ReadOnlyFile) Write(p []byte) (n int, err error) {
    return 0, errors.New("read-only, cannot write") // 永远失败
}

// 调用方期望 io.Writer 能正常写入, ReadOnlyFile 违反了 LSP
func SaveData(w io.Writer, data []byte) error {
    _, err := w.Write(data)
    return err
}
```

**Go — 正例**：
```go
// GOOD: 用更细粒度的接口区分能力

type Reader interface {
    Read(p []byte) (n int, err error)
}

type Writer interface {
    Write(p []byte) (n int, err error)
}

type ReadWriteFile struct { /* ... */ }

func (f *ReadWriteFile) Read(p []byte) (int, error)  { /* ... */ return 0, nil }
func (f *ReadWriteFile) Write(p []byte) (int, error) { /* ... */ return len(p), nil }

type ReadOnlyFile struct { /* ... */ }

func (f *ReadOnlyFile) Read(p []byte) (int, error) { /* ... */ return 0, nil }
// ReadOnlyFile 不实现 Writer, 编译期就能防止误用

func SaveData(w Writer, data []byte) error {
    _, err := w.Write(data)
    return err
}
```

**框架应用**：Go 标准库 `io.Reader` / `io.Writer` / `io.ReadWriter` 的分离就是 LSP 的经典体现。

---

### 1.4 接口隔离原则（ISP — Interface Segregation Principle）

**原理**：不应该强迫客户端依赖它不需要的方法。Go 天然鼓励小接口（1-3 个方法），这是 ISP 的最佳实践。

**Go — 反例**：
```go
// BAD: 过大的接口
type Animal interface {
    Walk()
    Swim()
    Fly()
}

// 狗不会飞, 但被迫实现 Fly()
type Dog struct{}

func (d *Dog) Walk() { fmt.Println("walking") }
func (d *Dog) Swim() { fmt.Println("swimming") }
func (d *Dog) Fly()  { panic("dogs can't fly!") } // 无意义实现
```

**Go — 正例**：
```go
// GOOD: 按能力拆分接口
type Walker interface {
    Walk()
}

type Swimmer interface {
    Swim()
}

type Flyer interface {
    Fly()
}

type Dog struct{}

func (d *Dog) Walk() { fmt.Println("walking") }
func (d *Dog) Swim() { fmt.Println("swimming") }
// Dog 无需实现 Flyer

type Bird struct{}

func (b *Bird) Walk() { fmt.Println("walking") }
func (b *Bird) Fly()  { fmt.Println("flying") }

// 需要时可以组合
type Amphibian interface {
    Walker
    Swimmer
}
```

**Go 惯例**：`io.Reader`（1 个方法）、`fmt.Stringer`（1 个方法）、`sort.Interface`（3 个方法）— 标准库接口普遍不超过 3 个方法。

### 1.5 依赖倒置原则（DIP — Dependency Inversion Principle）

**原理**：高层模块不应依赖低层模块，两者都应依赖抽象（接口）。抽象不应依赖细节，细节应依赖抽象。

**Go — 反例**：
```go
// BAD: OrderService 直接依赖 MySQLRepository 具体实现
type MySQLRepository struct {
    db *sql.DB
}

func (r *MySQLRepository) SaveOrder(order Order) error { /* ... */ return nil }

type OrderService struct {
    repo *MySQLRepository // 直接依赖具体类型
}

func (s *OrderService) PlaceOrder(order Order) error {
    return s.repo.SaveOrder(order)
}
```

**Go — 正例**：
```go
// GOOD: 依赖接口

type OrderRepository interface {
    SaveOrder(order Order) error
}

type MySQLOrderRepo struct {
    db *sql.DB
}

func (r *MySQLOrderRepo) SaveOrder(order Order) error { /* ... */ return nil }

type MongoOrderRepo struct {
    coll *mongo.Collection
}

func (r *MongoOrderRepo) SaveOrder(order Order) error { /* ... */ return nil }

// OrderService 依赖接口, 不关心具体存储
type OrderService struct {
    repo OrderRepository // 依赖接口
}

func NewOrderService(repo OrderRepository) *OrderService {
    return &OrderService{repo: repo}
}

func (s *OrderService) PlaceOrder(order Order) error {
    // 业务验证...
    return s.repo.SaveOrder(order)
}

// 测试时可以注入 mock
type MockOrderRepo struct {
    Orders []Order
}

func (m *MockOrderRepo) SaveOrder(order Order) error {
    m.Orders = append(m.Orders, order)
    return nil
}
```

**框架应用**：Go 的 `database/sql` 包定义了 `driver.Driver` 接口，各数据库驱动（mysql、pgx）实现该接口。上层代码通过 `sql.Open` 使用，不依赖具体驱动。

## 2. 创建型模式

### 2.1 单例模式（Singleton） ⭐⭐⭐

**原理**：确保一个类/结构体在全局只有一个实例，并提供一个全局访问点。常用于配置管理器、连接池、日志器等。

**UML 文字描述**：
```
┌──────────────────────┐
│     Singleton         │
├──────────────────────┤
│ - instance: *Singleton│
│ - once: sync.Once     │
├──────────────────────┤
│ + GetInstance(): *S    │
└──────────────────────┘
    唯一实例通过 GetInstance 获取
```

**Go 实现（sync.Once — 推荐）**：
```go
package config

import "sync"

type Config struct {
    DBHost string
    DBPort int
    Debug  bool
}

var (
    instance *Config
    once     sync.Once
)

// GetInstance 线程安全的懒加载单例
func GetInstance() *Config {
    once.Do(func() {
        instance = &Config{
            DBHost: "localhost",
            DBPort: 3306,
            Debug:  false,
        }
        // 实际项目中这里可以读取配置文件/环境变量
    })
    return instance
}
```

**Go 实现（init 函数 — 饿汉式）**：
```go
package logger

import "log"

type AppLogger struct {
    *log.Logger
}

var globalLogger *AppLogger

func init() {
    globalLogger = &AppLogger{
        Logger: log.Default(),
    }
}

func GetLogger() *AppLogger {
    return globalLogger
}
```

**Java 实现（双重检查锁）**：
```java
public class DatabasePool {
    private static volatile DatabasePool instance;

    private DatabasePool() {
        // 初始化连接池
    }

    public static DatabasePool getInstance() {
        if (instance == null) {
            synchronized (DatabasePool.class) {
                if (instance == null) {
                    instance = new DatabasePool();
                }
            }
        }
        return instance;
    }
}
```

**Java 实现（枚举单例 — 推荐）**：
```java
public enum ConfigSingleton {
    INSTANCE;

    private String dbHost = "localhost";

    public String getDbHost() {
        return dbHost;
    }
}
```

**框架应用**：
- Go `sync.Pool` 内部使用了类似单例的全局管理
- Java Spring 中 Bean 默认 scope 为 singleton
- Go 标准库 `http.DefaultClient`、`http.DefaultServeMux` 都是包级单例

### 2.2 工厂方法模式（Factory Method） ⭐⭐⭐

**原理**：定义一个创建对象的接口，让子类决定实例化哪个类。将对象创建延迟到子类/具体工厂。

**UML 文字描述**：
```
         ┌────────────┐
         │  Creator    │ (接口/抽象)
         │ +Create(): P│
         └─────┬──────┘
        ┌──────┴──────┐
   ┌────┴─────┐  ┌────┴─────┐
   │CreatorA   │  │CreatorB   │
   │+Create():P│  │+Create():P│
   └───────────┘  └───────────┘
         ▼              ▼
    ProductA       ProductB
```

**Go 实现**：
```go
package payment

import "fmt"

// Product 接口
type PaymentProcessor interface {
    Pay(amount float64) error
    Refund(orderID string) error
}

// 具体产品 — 支付宝
type AlipayProcessor struct{}

func (p *AlipayProcessor) Pay(amount float64) error {
    fmt.Printf("Alipay: paying %.2f\n", amount)
    return nil
}

func (p *AlipayProcessor) Refund(orderID string) error {
    fmt.Printf("Alipay: refunding order %s\n", orderID)
    return nil
}

// 具体产品 — 微信支付
type WechatProcessor struct{}

func (p *WechatProcessor) Pay(amount float64) error {
    fmt.Printf("Wechat: paying %.2f\n", amount)
    return nil
}

func (p *WechatProcessor) Refund(orderID string) error {
    fmt.Printf("Wechat: refunding order %s\n", orderID)
    return nil
}

// 工厂方法 — 用函数而非类，Go 惯用写法
func NewPaymentProcessor(method string) (PaymentProcessor, error) {
    switch method {
    case "alipay":
        return &AlipayProcessor{}, nil
    case "wechat":
        return &WechatProcessor{}, nil
    default:
        return nil, fmt.Errorf("unsupported payment method: %s", method)
    }
}

// 更 Go 风格：用注册表模式
var registry = map[string]func() PaymentProcessor{}

func Register(name string, factory func() PaymentProcessor) {
    registry[name] = factory
}

func Create(name string) (PaymentProcessor, error) {
    factory, ok := registry[name]
    if !ok {
        return nil, fmt.Errorf("unknown processor: %s", name)
    }
    return factory(), nil
}

func init() {
    Register("alipay", func() PaymentProcessor { return &AlipayProcessor{} })
    Register("wechat", func() PaymentProcessor { return &WechatProcessor{} })
}
```

**Java 实现**：
```java
// 抽象工厂方法
public abstract class PaymentFactory {
    public abstract PaymentProcessor createProcessor();

    public void processPayment(double amount) {
        PaymentProcessor processor = createProcessor();
        processor.pay(amount);
    }
}

public class AlipayFactory extends PaymentFactory {
    @Override
    public PaymentProcessor createProcessor() {
        return new AlipayProcessor();
    }
}

public class WechatFactory extends PaymentFactory {
    @Override
    public PaymentProcessor createProcessor() {
        return new WechatProcessor();
    }
}
```

**框架应用**：
- Go `database/sql.Register()` + `sql.Open()` 就是注册表工厂模式
- Java `Calendar.getInstance()` / `NumberFormat.getInstance()`

### 2.3 抽象工厂模式（Abstract Factory） ⭐⭐⭐

**原理**：提供一个创建一族相关对象的接口，而无需指定具体类。当系统需要在多个「产品族」间切换时使用。

**UML 文字描述**：
```
    ┌─────────────────────┐
    │  UIFactory (接口)     │
    │ + CreateButton(): B   │
    │ + CreateInput(): I    │
    └──────┬──────────────┘
    ┌──────┴──────┐
┌───┴────┐   ┌───┴────┐
│DarkUI   │   │LightUI  │
│Factory  │   │Factory  │
└─────────┘   └─────────┘
  ▼    ▼        ▼    ▼
DarkBtn DarkInput LightBtn LightInput
```

**Go 实现**：
```go
package ui

import "fmt"

// ---- 产品接口 ----
type Button interface {
    Render() string
}

type Input interface {
    Render() string
}

// ---- 抽象工厂 ----
type UIFactory interface {
    CreateButton() Button
    CreateInput() Input
}

// ---- Dark 产品族 ----
type DarkButton struct{}

func (b *DarkButton) Render() string { return "<button class='dark'>...</button>" }

type DarkInput struct{}

func (i *DarkInput) Render() string { return "<input class='dark' />" }

type DarkUIFactory struct{}

func (f *DarkUIFactory) CreateButton() Button { return &DarkButton{} }
func (f *DarkUIFactory) CreateInput() Input   { return &DarkInput{} }

// ---- Light 产品族 ----
type LightButton struct{}

func (b *LightButton) Render() string { return "<button class='light'>...</button>" }

type LightInput struct{}

func (i *LightInput) Render() string { return "<input class='light' />" }

type LightUIFactory struct{}

func (f *LightUIFactory) CreateButton() Button { return &LightButton{} }
func (f *LightUIFactory) CreateInput() Input   { return &LightInput{} }

// ---- 客户端代码 ----
func RenderPage(factory UIFactory) {
    btn := factory.CreateButton()
    input := factory.CreateInput()
    fmt.Println(btn.Render())
    fmt.Println(input.Render())
}
```

**Java 实现**：
```java
public interface UIFactory {
    Button createButton();
    Input createInput();
}

public class DarkUIFactory implements UIFactory {
    public Button createButton() { return new DarkButton(); }
    public Input createInput()   { return new DarkInput(); }
}

// 使用
UIFactory factory = new DarkUIFactory();
Button btn = factory.createButton();
```

**框架应用**：Java AWT 的 `Toolkit` 就是抽象工厂 — 在不同平台返回不同的 Button、TextField 实现。

### 2.4 建造者模式（Builder） ⭐⭐⭐

**原理**：将一个复杂对象的构建与它的表示分离，使得同样的构建过程可以创建不同的表示。适用于参数多、有多种组合的对象。

**UML 文字描述**：
```
    Client
      │
      ▼
  ┌─────────┐     ┌──────────────┐
  │ Director │────▶│ Builder(接口) │
  └─────────┘     │ +SetA()       │
                  │ +SetB()       │
                  │ +Build(): P   │
                  └──────┬───────┘
                  ┌──────┴───────┐
                  │ConcreteBuilder│
                  └──────────────┘
```

**Go 实现（链式调用 — 推荐）**：
```go
package server

import (
    "time"
)

type Server struct {
    Host         string
    Port         int
    ReadTimeout  time.Duration
    WriteTimeout time.Duration
    MaxConns     int
    TLS          bool
}

type ServerBuilder struct {
    server Server
}

func NewServerBuilder(host string, port int) *ServerBuilder {
    return &ServerBuilder{
        server: Server{
            Host:         host,
            Port:         port,
            ReadTimeout:  30 * time.Second,
            WriteTimeout: 30 * time.Second,
            MaxConns:     1000,
        },
    }
}

func (b *ServerBuilder) ReadTimeout(d time.Duration) *ServerBuilder {
    b.server.ReadTimeout = d
    return b
}

func (b *ServerBuilder) WriteTimeout(d time.Duration) *ServerBuilder {
    b.server.WriteTimeout = d
    return b
}

func (b *ServerBuilder) MaxConns(n int) *ServerBuilder {
    b.server.MaxConns = n
    return b
}

func (b *ServerBuilder) EnableTLS() *ServerBuilder {
    b.server.TLS = true
    return b
}

func (b *ServerBuilder) Build() Server {
    return b.server
}

// 使用
// srv := NewServerBuilder("0.0.0.0", 8080).
//     ReadTimeout(10 * time.Second).
```

### 2.5 原型模式（Prototype） ⭐⭐

**原理**：通过拷贝已有实例来创建新对象，避免昂贵的初始化操作。适用于对象创建代价大（如需要从数据库加载、复杂计算）的场景。

**Go 实现**：
```go
package prototype

import "fmt"

type Cloneable interface {
    Clone() Cloneable
}

// Document 可克隆的文档
type Document struct {
    Title   string
    Content string
    Tags    []string
}

func (d *Document) Clone() Cloneable {
    // 深拷贝 slice
    tags := make([]string, len(d.Tags))
    copy(tags, d.Tags)
    return &Document{
        Title:   d.Title,
        Content: d.Content,
        Tags:    tags,
    }
}

// 原型注册表
type PrototypeRegistry struct {
    prototypes map[string]Cloneable
}

func NewRegistry() *PrototypeRegistry {
    return &PrototypeRegistry{
        prototypes: make(map[string]Cloneable),
    }
}

func (r *PrototypeRegistry) Register(name string, proto Cloneable) {
    r.prototypes[name] = proto
}

func (r *PrototypeRegistry) Get(name string) (Cloneable, error) {
    proto, ok := r.prototypes[name]
    if !ok {
        return nil, fmt.Errorf("prototype %q not found", name)
    }
    return proto.Clone(), nil
}
```

**Java 实现**：
```java
public class Document implements Cloneable {
    private String title;
    private String content;
    private List<String> tags;

    @Override
    public Document clone() {
        try {
            Document cloned = (Document) super.clone();
            cloned.tags = new ArrayList<>(this.tags); // 深拷贝
            return cloned;
        } catch (CloneNotSupportedException e) {
            throw new RuntimeException(e);
        }
    }
}
```

**框架应用**：
- Go `proto.Clone()` 在 protobuf 中用于深拷贝消息
- Java Spring 中 Bean scope 设为 `prototype` 时，每次获取都创建新实例

## 3. 结构型模式

### 3.1 适配器模式（Adapter） ⭐⭐⭐

**原理**：将一个接口转换为客户端期望的另一个接口。使原本不兼容的类可以协同工作。Go 中常用组合 + 接口实现。

**UML 文字描述**：
```
Client ──▶ Target(接口)
               ▲
               │
          ┌────┴─────┐
          │  Adapter   │──▶ Adaptee (旧接口)
          └───────────┘
```

**Go 实现**：
```go
package adapter

import "fmt"

// 目标接口 — 系统期望的统一日志接口
type Logger interface {
    Info(msg string)
    Error(msg string)
}

// 旧的第三方日志库（不可修改）
type LegacyLogger struct{}

func (l *LegacyLogger) WriteLog(level int, message string) {
    fmt.Printf("[Legacy] level=%d msg=%s\n", level, message)
}

// 适配器 — 将 LegacyLogger 适配为 Logger 接口
type LegacyLoggerAdapter struct {
    legacy *LegacyLogger
}

func NewLegacyLoggerAdapter(l *LegacyLogger) Logger {
    return &LegacyLoggerAdapter{legacy: l}
}

func (a *LegacyLoggerAdapter) Info(msg string) {
    a.legacy.WriteLog(1, msg)
}

func (a *LegacyLoggerAdapter) Error(msg string) {
    a.legacy.WriteLog(3, msg)
}

// 客户端只依赖 Logger 接口
func ProcessRequest(logger Logger) {
    logger.Info("processing request")
    // ...
    logger.Error("something went wrong")
}
```

**Java 实现**：
```java
// 目标接口
public interface MediaPlayer {
    void play(String filename);
}

// 高级播放器（Adaptee）
public class VLCPlayer {
    public void playVLC(String filename) {
        System.out.println("Playing VLC: " + filename);
    }
}

// 适配器
public class VLCAdapter implements MediaPlayer {
    private VLCPlayer vlcPlayer = new VLCPlayer();

    @Override
    public void play(String filename) {
        vlcPlayer.playVLC(filename);
    }
}
```

**框架应用**：
- Go `io.NopCloser` 将 `io.Reader` 适配为 `io.ReadCloser`
- Go `http.HandlerFunc` 是函数到接口的适配器
- Java `Arrays.asList()` 将数组适配为 List 接口

### 3.2 装饰器模式（Decorator） ⭐⭐⭐

**原理**：动态地给对象附加额外的职责，比子类继承更灵活。通过组合而非继承来扩展功能，支持叠加多个装饰。

**UML 文字描述**：
```
      ┌──────────────┐
      │  Component    │ (接口)
      │ + Execute()   │
      └──┬───────┬───┘
  ┌──────┴──┐  ┌─┴────────────┐
  │Concrete  │  │BaseDecorator  │
  │Component │  │-wrapped: Comp │
  └─────────┘  │+Execute()     │
               └──┬────────┬──┘
            ┌─────┴──┐  ┌──┴──────┐
            │LogDeco  │  │CacheDeco │
            └────────┘  └─────────┘
```

**Go 实现**：
```go
package decorator

import (
    "fmt"
    "log"
    "time"
)

// 核心接口
type DataFetcher interface {
    Fetch(key string) (string, error)
}

// 基础实现
type DBFetcher struct{}

func (f *DBFetcher) Fetch(key string) (string, error) {
    // 模拟数据库查询
    time.Sleep(100 * time.Millisecond)
    return fmt.Sprintf("data_for_%s", key), nil
}

// 装饰器 1: 日志
type LoggingDecorator struct {
    wrapped DataFetcher
}

func NewLoggingDecorator(f DataFetcher) DataFetcher {
    return &LoggingDecorator{wrapped: f}
}

func (d *LoggingDecorator) Fetch(key string) (string, error) {
    start := time.Now()
    result, err := d.wrapped.Fetch(key)
    log.Printf("[Fetch] key=%s duration=%v err=%v", key, time.Since(start), err)
    return result, err
}

// 装饰器 2: 缓存
type CacheDecorator struct {
    wrapped DataFetcher
    cache   map[string]string
}

func NewCacheDecorator(f DataFetcher) DataFetcher {
    return &CacheDecorator{
        wrapped: f,
        cache:   make(map[string]string),
    }
}

func (d *CacheDecorator) Fetch(key string) (string, error) {
    if val, ok := d.cache[key]; ok {
        return val, nil // 命中缓存
    }
    result, err := d.wrapped.Fetch(key)
    if err == nil {
        d.cache[key] = result
    }
    return result, err
}

// 装饰器 3: 重试
type RetryDecorator struct {
    wrapped    DataFetcher
    maxRetries int
}

func NewRetryDecorator(f DataFetcher, retries int) DataFetcher {
    return &RetryDecorator{wrapped: f, maxRetries: retries}
}

func (d *RetryDecorator) Fetch(key string) (string, error) {
    var lastErr error
    for i := 0; i <= d.maxRetries; i++ {
        result, err := d.wrapped.Fetch(key)
        if err == nil {
            return result, nil
        }
        lastErr = err
        time.Sleep(time.Duration(i*100) * time.Millisecond)
    }
    return "", fmt.Errorf("all %d retries failed: %w", d.maxRetries, lastErr)
}

// 使用: 可自由叠加装饰器
// fetcher := NewLoggingDecorator(
//     NewCacheDecorator(
//         NewRetryDecorator(&DBFetcher{}, 3),
//     ),
// )
// result, _ := fetcher.Fetch("user:123")
```

**Java 实现**：
```java
// Java I/O 就是装饰器模式的经典应用
InputStream is = new BufferedInputStream(
    new FileInputStream("file.txt")
);

// 自定义装饰器
public class LoggingFetcher implements DataFetcher {
    private final DataFetcher wrapped;

    public LoggingFetcher(DataFetcher wrapped) {
        this.wrapped = wrapped;
    }

    @Override
    public String fetch(String key) {
        long start = System.currentTimeMillis();
        String result = wrapped.fetch(key);
        System.out.printf("Fetch %s took %dms%n", key, System.currentTimeMillis() - start);
        return result;
    }
}
```

**框架应用**：
- Go `http.Handler` 中间件本质就是装饰器模式
- Java `BufferedReader(new FileReader(...))` — 经典 I/O 装饰链

### 3.3 代理模式（Proxy） ⭐⭐⭐

**原理**：为另一个对象提供一个替身/占位符，以控制对这个对象的访问。常见类型：远程代理、虚拟代理（延迟加载）、保护代理（权限控制）。

**Go 实现**：
```go
package proxy

import (
    "fmt"
    "sync"
)

// 服务接口
type Database interface {
    Query(sql string) (string, error)
    Execute(sql string) error
}

// 真实服务
type RealDatabase struct {
    dsn string
}

func (db *RealDatabase) Query(sql string) (string, error) {
    return fmt.Sprintf("result of: %s", sql), nil
}

func (db *RealDatabase) Execute(sql string) error {
    fmt.Printf("Executed: %s\n", sql)
    return nil
}

// 保护代理 — 权限控制
type AccessControlProxy struct {
    db   Database
    role string
}

func NewAccessControlProxy(db Database, role string) Database {
    return &AccessControlProxy{db: db, role: role}
}

func (p *AccessControlProxy) Query(sql string) (string, error) {
    return p.db.Query(sql) // 所有角色都可以查询
}

func (p *AccessControlProxy) Execute(sql string) error {
    if p.role != "admin" {
        return fmt.Errorf("permission denied: role %q cannot execute", p.role)
    }
    return p.db.Execute(sql)
}

// 延迟初始化代理
type LazyDBProxy struct {
    dsn  string
    db   Database
    once sync.Once
}

func NewLazyDBProxy(dsn string) Database {
    return &LazyDBProxy{dsn: dsn}
}

func (p *LazyDBProxy) init() {
    p.once.Do(func() {
        fmt.Println("Initializing database connection...")
        p.db = &RealDatabase{dsn: p.dsn}
    })
}

func (p *LazyDBProxy) Query(sql string) (string, error) {
    p.init()
    return p.db.Query(sql)
}

func (p *LazyDBProxy) Execute(sql string) error {
    p.init()
    return p.db.Execute(sql)
}
```

**Java 实现（动态代理）**：
```java
public class LoggingProxy implements InvocationHandler {
    private final Object target;

    public LoggingProxy(Object target) {
        this.target = target;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("Before: " + method.getName());
        Object result = method.invoke(target, args);
        System.out.println("After: " + method.getName());
        return result;
    }

    @SuppressWarnings("unchecked")
    public static <T> T create(T target, Class<T> iface) {
        return (T) Proxy.newProxyInstance(
            iface.getClassLoader(),
            new Class[]{iface},
            new LoggingProxy(target)
        );
    }
}
```

**代理 vs 装饰器 对比**：

| 维度 | 代理模式 | 装饰器模式 |
|------|---------|-----------|
| **目的** | 控制访问 | 增强功能 |
| **关系** | 代理自己创建/管理被代理对象 | 装饰器由外部传入被装饰对象 |
| **生命周期** | 代理控制目标的生命周期 | 客户端控制 |
| **典型场景** | 权限、延迟加载、远程调用 | 日志、缓存、加密 |

**框架应用**：
- Go `httputil.ReverseProxy` 是 HTTP 远程代理
- Java Spring AOP 的 `@Transactional` 底层是动态代理

### 3.4 外观模式（Facade） ⭐⭐

**原理**：为子系统中的一组接口提供一个统一的高层接口，降低子系统的使用复杂度。

**Go 实现**：
```go
package facade

import "fmt"

// ---- 子系统 ----
type UserService struct{}

func (s *UserService) GetUser(id int) string { return fmt.Sprintf("User_%d", id) }

type InventoryService struct{}

func (s *InventoryService) CheckStock(itemID int) bool { return true }
func (s *InventoryService) DeductStock(itemID int, qty int) error { return nil }

type PaymentService struct{}

func (s *PaymentService) Charge(userID int, amount float64) error { return nil }

type NotificationService struct{}

func (s *NotificationService) SendOrderConfirmation(userID int, orderID string) error {
    return nil
}

// ---- Facade ----
type OrderFacade struct {
    user      *UserService
    inventory *InventoryService
    payment   *PaymentService
    notify    *NotificationService
}

func NewOrderFacade() *OrderFacade {
    return &OrderFacade{
        user:      &UserService{},
        inventory: &InventoryService{},
        payment:   &PaymentService{},
        notify:    &NotificationService{},
    }
}

// PlaceOrder 将多个子系统调用封装为一个简单操作
func (f *OrderFacade) PlaceOrder(userID, itemID, qty int, amount float64) (string, error) {
    // 1. 验证用户
    _ = f.user.GetUser(userID)

    // 2. 检查库存
    if !f.inventory.CheckStock(itemID) {
        return "", fmt.Errorf("item %d out of stock", itemID)
    }

    // 3. 扣减库存
    if err := f.inventory.DeductStock(itemID, qty); err != nil {
        return "", fmt.Errorf("deduct stock failed: %w", err)
    }

    // 4. 扣款
    if err := f.payment.Charge(userID, amount); err != nil {
        return "", fmt.Errorf("payment failed: %w", err)
    }

    // 5. 发送通知
    orderID := fmt.Sprintf("ORD-%d-%d", userID, itemID)
    _ = f.notify.SendOrderConfirmation(userID, orderID)

    return orderID, nil
}
```

**框架应用**：
- Go `net/http` 包中 `http.ListenAndServe()` 是创建服务器的外观
- Java `SLF4J` 是日志系统的外观

### 3.5 桥接模式（Bridge） ⭐⭐

**原理**：将抽象与实现分离，使它们可以独立变化。当一个类有两个或更多独立变化的维度时使用。

**UML 文字描述**：
```
┌─────────────┐        ┌──────────────┐
│ Abstraction  │───────▶│ Implementor   │ (接口)
│ -impl: Impl  │        └──┬───────┬──┘
└──┬──────────┘     ┌─────┴──┐ ┌──┴─────┐
┌──┴──────────┐     │ImplA    │ │ImplB    │
│Refined       │     └────────┘ └────────┘
│Abstraction   │
└─────────────┘
```

**Go 实现**：
```go
package bridge

import "fmt"

// 实现维度: 消息发送方式
type MessageSender interface {
    Send(to, content string) error
}

type SMSSender struct{}

func (s *SMSSender) Send(to, content string) error {

    return nil
}

    fmt.Printf("[Email] To: %s, Content: %s\n", to, content)
    return nil
}

type WechatSender struct{}

    fmt.Printf("[Wechat] To: %s, Content: %s\n", to, content)

}

// 抽象维度: 消息类型
```

### 3.6 组合模式（Composite） ⭐⭐

**原理**：将对象组合成树形结构以表示「部分-整体」的层次关系，使客户端对单个对象和组合对象的使用具有一致性。

**Go 实现**：
```go
package composite

import "fmt"

// 统一接口
type FileSystemNode interface {
    Name() string
    Size() int64
    Print(indent string)
}

// 叶子节点: 文件
type File struct {
    name string
    size int64
}

func NewFile(name string, size int64) *File {
    return &File{name: name, size: size}
}

func (f *File) Name() string { return f.name }
func (f *File) Size() int64  { return f.size }
func (f *File) Print(indent string) {
    fmt.Printf("%s📄 %s (%d bytes)\n", indent, f.name, f.size)
}

// 组合节点: 目录
type Directory struct {
    name     string
    children []FileSystemNode
}

func NewDirectory(name string) *Directory {
    return &Directory{name: name}
}

func (d *Directory) Name() string { return d.name }

func (d *Directory) Size() int64 {
    var total int64
    for _, child := range d.children {
        total += child.Size()
    }
    return total
}

func (d *Directory) Add(node FileSystemNode) {
    d.children = append(d.children, node)
}

func (d *Directory) Print(indent string) {
    fmt.Printf("%s📁 %s/ (%d bytes)\n", indent, d.name, d.Size())
    for _, child := range d.children {
        child.Print(indent + "  ")
    }
}

// 使用:
// root := NewDirectory("root")
// src := NewDirectory("src")
// src.Add(NewFile("main.go", 1024))
// src.Add(NewFile("util.go", 512))
// root.Add(src)
// root.Add(NewFile("README.md", 256))
// root.Print("")
// 输出:
// 📁 root/ (1792 bytes)
//   📁 src/ (1536 bytes)
//     📄 main.go (1024 bytes)
//     📄 util.go (512 bytes)
//   📄 README.md (256 bytes)
```

**框架应用**：
- Go `net/http.ServeMux` 可以嵌套路由 — 组合模式
- Java Swing 的 `Component`/`Container` 体系

### 3.7 享元模式（Flyweight） ⭐⭐

**原理**：运用共享技术有效地支持大量细粒度对象。将对象的状态分为内部状态（共享）和外部状态（不共享），减少内存占用。

**Go 实现**：
```go
package flyweight

import (
    "fmt"
    "sync"
)

// 享元对象 — 内部状态: 字体属性
type TextStyle struct {
    FontName string
    FontSize int
    Bold     bool
    Italic   bool
}

func (s *TextStyle) Render(content string, x, y int) string {
    return fmt.Sprintf("[%s %dpt B:%v I:%v] '%s' at (%d,%d)",
        s.FontName, s.FontSize, s.Bold, s.Italic, content, x, y)
}

// 享元工厂
type StyleFactory struct {
    mu     sync.RWMutex
    styles map[string]*TextStyle
}

func NewStyleFactory() *StyleFactory {
    return &StyleFactory{styles: make(map[string]*TextStyle)}
}

func (f *StyleFactory) GetStyle(fontName string, fontSize int, bold, italic bool) *TextStyle {
    key := fmt.Sprintf("%s-%d-%v-%v", fontName, fontSize, bold, italic)

    f.mu.RLock()
    if style, ok := f.styles[key]; ok {
        f.mu.RUnlock()
        return style
    }
    f.mu.RUnlock()

    f.mu.Lock()
    defer f.mu.Unlock()

    // 二次检查
    if style, ok := f.styles[key]; ok {
        return style
    }

    style := &TextStyle{
        FontName: fontName,
        FontSize: fontSize,
        Bold:     bold,
        Italic:   italic,
    }
    f.styles[key] = style
    return style
}

func (f *StyleFactory) Count() int {
    f.mu.RLock()
    defer f.mu.RUnlock()
    return len(f.styles)
}

// 使用:
// factory := NewStyleFactory()
// s1 := factory.GetStyle("Arial", 12, false, false)
// s2 := factory.GetStyle("Arial", 12, false, false) // 返回同一个对象
// fmt.Println(s1 == s2)  // true — 共享了
```

**框架应用**：
- Go `sync.Pool` 是享元思想的应用 — 复用临时对象
- Java `Integer.valueOf(-128~127)` 缓存、`String` 常量池

## 4. 行为型模式

### 4.1 策略模式（Strategy） ⭐⭐⭐

**原理**：定义一系列算法，把它们封装起来，并使它们可以互相替换。让算法的变化独立于使用它的客户端。在 Go 中，函数类型是天然的策略。

**UML 文字描述**：
```
    ┌──────────┐      ┌────────────┐
    │  Context  │─────▶│ Strategy   │ (接口)
    │           │      │ +Execute() │
    └──────────┘      └──┬─────┬──┘
                    ┌────┴──┐ ┌┴──────┐
                    │StratA  │ │StratB  │
                    └───────┘ └───────┘
```

**Go 实现（接口版）**：
```go
package strategy

import (
    "fmt"
    "sort"
)

// 策略接口: 压缩算法
type Compressor interface {
    Compress(data []byte) ([]byte, error)
    Name() string
}

type GzipCompressor struct{}

func (c *GzipCompressor) Compress(data []byte) ([]byte, error) {
    fmt.Println("Compressing with gzip...")
    // 实际压缩逻辑
    return data, nil
}
func (c *GzipCompressor) Name() string { return "gzip" }

type Lz4Compressor struct{}

func (c *Lz4Compressor) Compress(data []byte) ([]byte, error) {
    fmt.Println("Compressing with lz4...")
```

### 4.2 观察者模式（Observer） ⭐⭐⭐

**原理**：定义对象间一对多的依赖关系，当一个对象状态改变时，所有依赖者都收到通知并自动更新。也叫发布-订阅（Pub/Sub）。

**Go 实现**：
```go
package observer

import (
    "fmt"
    "sync"
)

// 事件类型
type Event struct {
    Type string
    Data interface{}
}

// 观察者接口
type Observer interface {
    OnNotify(event Event)
    ID() string
}

// 被观察者 (EventBus)
type EventBus struct {
    mu        sync.RWMutex
    observers map[string][]Observer // eventType -> observers
}

func NewEventBus() *EventBus {
    return &EventBus{
        observers: make(map[string][]Observer),
    }
}

func (bus *EventBus) Subscribe(eventType string, observer Observer) {
    bus.mu.Lock()
    defer bus.mu.Unlock()
    bus.observers[eventType] = append(bus.observers[eventType], observer)
}

func (bus *EventBus) Unsubscribe(eventType string, observerID string) {
    bus.mu.Lock()
    defer bus.mu.Unlock()
    observers := bus.observers[eventType]
    for i, o := range observers {
        if o.ID() == observerID {
            bus.observers[eventType] = append(observers[:i], observers[i+1:]...)
            return
        }
    }
}

func (bus *EventBus) Publish(event Event) {
    bus.mu.RLock()
    observers := make([]Observer, len(bus.observers[event.Type]))
    copy(observers, bus.observers[event.Type])
    bus.mu.RUnlock()

    for _, observer := range observers {
        observer.OnNotify(event)
    }
}

// 异步发布
func (bus *EventBus) PublishAsync(event Event) {
    bus.mu.RLock()
    observers := make([]Observer, len(bus.observers[event.Type]))
    copy(observers, bus.observers[event.Type])
    bus.mu.RUnlock()

    var wg sync.WaitGroup
    for _, observer := range observers {
        wg.Add(1)
        go func(o Observer) {
            defer wg.Done()
            o.OnNotify(event)
        }(observer)
    }
    wg.Wait()
}

// 具体观察者
type EmailObserver struct {
    id string
}

func (o *EmailObserver) ID() string { return o.id }
func (o *EmailObserver) OnNotify(event Event) {
    fmt.Printf("[Email %s] Received event: %s, data: %v\n", o.id, event.Type, event.Data)
}

type LogObserver struct {
    id string
}

func (o *LogObserver) ID() string { return o.id }
func (o *LogObserver) OnNotify(event Event) {
    fmt.Printf("[Log %s] Event: %s, data: %v\n", o.id, event.Type, event.Data)
}

// 使用:
// bus := NewEventBus()
// bus.Subscribe("order.created", &EmailObserver{id: "email-1"})
// bus.Subscribe("order.created", &LogObserver{id: "log-1"})
// bus.Publish(Event{Type: "order.created", Data: map[string]string{"orderID": "123"}})
```

**Go 实现（channel 版 — Go 惯用）**：
```go
type ChannelEventBus struct {
    mu          sync.RWMutex
    subscribers map[string][]chan Event
}

func NewChannelEventBus() *ChannelEventBus {
    return &ChannelEventBus{
        subscribers: make(map[string][]chan Event),
    }
}

func (bus *ChannelEventBus) Subscribe(eventType string) <-chan Event {
    ch := make(chan Event, 16)
    bus.mu.Lock()
    bus.subscribers[eventType] = append(bus.subscribers[eventType], ch)
    bus.mu.Unlock()
    return ch
}

func (bus *ChannelEventBus) Publish(event Event) {
    bus.mu.RLock()
    defer bus.mu.RUnlock()
    for _, ch := range bus.subscribers[event.Type] {
        // 非阻塞发送
        select {
        case ch <- event:
        default:
            // channel 满了, 丢弃或记录日志
        }
    }
}
```

- Redis Pub/Sub、Kafka Consumer Group

### 4.3 模板方法模式（Template Method） ⭐⭐⭐

**原理**：在基类中定义算法的骨架，将某些步骤延迟到子类实现。Go 没有继承，通过接口 + 组合来实现。

**Go 实现**：
```go
package template

import (
    "fmt"
    "time"
)

// 定义可变步骤的接口
type DataMiner interface {
    Extract(source string) ([]byte, error)
    Transform(raw []byte) ([]map[string]interface{}, error)
    Load(data []map[string]interface{}) error
}

// 模板: 固定的 ETL 流程
func RunETL(miner DataMiner, source string) error {
    fmt.Println("=== ETL Pipeline Start ===")
    start := time.Now()

    // Step 1: Extract
    raw, err := miner.Extract(source)
    if err != nil {
        return fmt.Errorf("extract failed: %w", err)
    }
    fmt.Printf("Extracted %d bytes\n", len(raw))

    // Step 2: Transform
    data, err := miner.Transform(raw)
    if err != nil {
        return fmt.Errorf("transform failed: %w", err)
    }
    fmt.Printf("Transformed into %d records\n", len(data))

    // Step 3: Load
    if err := miner.Load(data); err != nil {
        return fmt.Errorf("load failed: %w", err)
    }

    fmt.Printf("=== ETL Pipeline Done (%v) ===\n", time.Since(start))
    return nil
}
```

### 4.4 责任链模式（Chain of Responsibility） ⭐⭐⭐

**原理**：将请求沿链传递，直到有一个处理者处理它。解耦请求发送者和接收者。链中的每个节点决定处理还是传递。

**Go 实现**：
```go
package chain

import (
    "fmt"
    "net/http"
)

// 审批系统示例
type PurchaseRequest struct {
    Amount      float64
    Description string
}

type Approver interface {
    SetNext(approver Approver) Approver
    Approve(req PurchaseRequest) string
}

// 基础审批者（提供链的逻辑复用）
type BaseApprover struct {
    next Approver
}

func (b *BaseApprover) SetNext(next Approver) Approver {
    b.next = next
    return next
}

func (b *BaseApprover) ApproveNext(req PurchaseRequest) string {
    if b.next != nil {
        return b.next.Approve(req)
    }
    return "REJECTED: no approver found"
}

// 组长: 可批 < 1000
type TeamLead struct {
    BaseApprover
}

func (t *TeamLead) Approve(req PurchaseRequest) string {
    if req.Amount < 1000 {
        return fmt.Sprintf("TeamLead approved: %s ($%.2f)", req.Description, req.Amount)
    }
    return t.ApproveNext(req)
}

// 经理: 可批 < 10000
type Manager struct {
    BaseApprover
}

func (m *Manager) Approve(req PurchaseRequest) string {
    if req.Amount < 10000 {
        return fmt.Sprintf("Manager approved: %s ($%.2f)", req.Description, req.Amount)
    }
    return m.ApproveNext(req)
}

// VP: 可批 < 100000
type VP struct {
    BaseApprover
}

func (v *VP) Approve(req PurchaseRequest) string {
    if req.Amount < 100000 {
        return fmt.Sprintf("VP approved: %s ($%.2f)", req.Description, req.Amount)
    }
    return v.ApproveNext(req)
}

// 使用:
// lead := &TeamLead{}
// lead.SetNext(&Manager{}).SetNext(&VP{})
// result := lead.Approve(PurchaseRequest{Amount: 5000, Description: "Server"})
// fmt.Println(result) // "Manager approved: Server ($5000.00)"
```

**Go HTTP 中间件链（最常见的责任链）**：
```go
type Middleware func(http.Handler) http.Handler

func Chain(handler http.Handler, middlewares ...Middleware) http.Handler {
    // 从后往前包装
    for i := len(middlewares) - 1; i >= 0; i-- {
        handler = middlewares[i](handler)
    }
    return handler
}

func LoggingMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        fmt.Printf("%s %s\n", r.Method, r.URL.Path)
        next.ServeHTTP(w, r)
    })
}

func AuthMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        token := r.Header.Get("Authorization")
        if token == "" {
            http.Error(w, "unauthorized", http.StatusUnauthorized)
            return // 不调用 next, 链终止
        }
        next.ServeHTTP(w, r)
    })
}
```

### 4.5 命令模式（Command） ⭐⭐

**原理**：将请求封装为对象，从而支持参数化请求、队列化、日志记录和撤销操作。

**Go 实现**：
```go
package command

import "fmt"

// 命令接口
type Command interface {
    Execute() error
    Undo() error
    Description() string
}

// 接收者: 文本编辑器
type TextEditor struct {
    content string
}

func (e *TextEditor) Insert(text string, pos int) {
    if pos > len(e.content) {
        pos = len(e.content)
    }
    e.content = e.content[:pos] + text + e.content[pos:]
}

func (e *TextEditor) Delete(pos, length int) string {
    if pos+length > len(e.content) {
        length = len(e.content) - pos
    }
    deleted := e.content[pos : pos+length]
    e.content = e.content[:pos] + e.content[pos+length:]
    return deleted
}

func (e *TextEditor) Content() string { return e.content }

// 具体命令: 插入
type InsertCommand struct {
    editor *TextEditor
    text   string
    pos    int
}

func (c *InsertCommand) Execute() error {
    c.editor.Insert(c.text, c.pos)
    return nil
}

func (c *InsertCommand) Undo() error {
    c.editor.Delete(c.pos, len(c.text))
    return nil
}

func (c *InsertCommand) Description() string {
    return fmt.Sprintf("Insert '%s' at pos %d", c.text, c.pos)
}

// 具体命令: 删除
type DeleteCommand struct {
    editor  *TextEditor
    pos     int
    length  int
    deleted string // 用于 Undo
}

func (c *DeleteCommand) Execute() error {
    c.deleted = c.editor.Delete(c.pos, c.length)
    return nil
}

func (c *DeleteCommand) Undo() error {
    c.editor.Insert(c.deleted, c.pos)
    return nil
}

func (c *DeleteCommand) Description() string {
    return fmt.Sprintf("Delete %d chars at pos %d", c.length, c.pos)
}

// 调用者: 命令历史管理
type CommandInvoker struct {
    history []Command
    undone  []Command
}

func (inv *CommandInvoker) Execute(cmd Command) error {
    if err := cmd.Execute(); err != nil {
        return err
    }
    inv.history = append(inv.history, cmd)
    inv.undone = nil // 执行新命令后清空 redo 栈
    return nil
}

func (inv *CommandInvoker) Undo() error {
    if len(inv.history) == 0 {
        return fmt.Errorf("nothing to undo")
    }
    cmd := inv.history[len(inv.history)-1]
    inv.history = inv.history[:len(inv.history)-1]
    if err := cmd.Undo(); err != nil {
        return err
    }
    inv.undone = append(inv.undone, cmd)
    return nil
}

func (inv *CommandInvoker) Redo() error {
    if len(inv.undone) == 0 {
        return fmt.Errorf("nothing to redo")
    }
    cmd := inv.undone[len(inv.undone)-1]
    inv.undone = inv.undone[:len(inv.undone)-1]
    return inv.Execute(cmd)
}
```

**框架应用**：
- 数据库事务日志（WAL）本质是命令模式
- Redis 的 MULTI/EXEC 事务
- GUI 应用中的 Undo/Redo

**原理**：允许对象在内部状态改变时改变其行为，看起来好像修改了它的类。将状态相关的行为抽取到独立的状态类中。

### 4.7 迭代器模式（Iterator） ⭐⭐

**原理**：提供一种方法顺序访问聚合对象中的各个元素，而不暴露其内部表示。

**Go 实现**：
```go
package iterator

// Go 1.23+ 的 iter 包提供了标准的迭代器协议
// 这里展示自定义迭代器

type Iterator[T any] interface {
    HasNext() bool
    Next() T
}

// 链表节点
type Node[T any] struct {
    Value T
    Next  *Node[T]
}

// 链表
type LinkedList[T any] struct {
    Head *Node[T]
    size int
}

func (l *LinkedList[T]) Add(val T) {
    node := &Node[T]{Value: val, Next: l.Head}
    l.Head = node
    l.size++
}

// 链表迭代器
type LinkedListIterator[T any] struct {
    current *Node[T]
}

func (l *LinkedList[T]) Iterator() Iterator[T] {
    return &LinkedListIterator[T]{current: l.Head}
}

func (it *LinkedListIterator[T]) HasNext() bool {
    return it.current != nil
}

func (it *LinkedListIterator[T]) Next() T {
    val := it.current.Value
    it.current = it.current.Next
    return val
}

// 使用 channel 实现迭代器（Go 惯用方式）
func (l *LinkedList[T]) Iter() <-chan T {
    ch := make(chan T)
    go func() {
        defer close(ch)
        current := l.Head
        for current != nil {
            ch <- current.Value
            current = current.Next
        }
    }()
    return ch
}

// 使用:
// list := &LinkedList[int]{}
// list.Add(1); list.Add(2); list.Add(3)
// for val := range list.Iter() {
//     fmt.Println(val)
// }
```

- Go `sql.Rows` 的 `Next()` + `Scan()` 也是迭代器
- Java `Iterator<T>` 接口

### 4.8 中介者模式（Mediator） ⭐⭐

**原理**：用一个中介对象封装一系列对象的交互。中介者使各对象不需要显式地互相引用，从而松散耦合。

**Go 实现**：
```go
package mediator

    mediator ChatMediator

}

    fmt.Printf("[%s] Received from %s: %s\n", u.Name, from, msg)
}
```

## 5. Go 特色模式

### 5.1 函数选项模式（Functional Options） ⭐⭐⭐

**原理**：利用 Go 的一等函数和可变参数，为构造函数提供灵活、可扩展的配置方式。比 Builder 模式更轻量，比配置结构体更灵活（可以有默认值、验证逻辑）。

> 这是 Go 社区最受推崇的构造模式，由 Dave Cheney 和 Rob Pike 推广。

**Go 实现**：
```go
package server

import (
    "crypto/tls"
    "log"
    "time"
)

type Server struct {
    addr         string
    port         int
    readTimeout  time.Duration
    writeTimeout time.Duration
    maxConns     int
    tlsConfig    *tls.Config
    logger       *log.Logger
}

// Option 是一个函数类型, 接收 *Server 并修改它
type Option func(*Server)

// 每个选项是一个返回 Option 的函数
func WithPort(port int) Option {
    return func(s *Server) {
        s.port = port
    }
}

func WithReadTimeout(d time.Duration) Option {
    return func(s *Server) {
        s.readTimeout = d
    }
}

func WithWriteTimeout(d time.Duration) Option {
    return func(s *Server) {
        s.writeTimeout = d
    }
}

func WithMaxConns(n int) Option {
    return func(s *Server) {
        s.maxConns = n
    }
}

func WithTLS(config *tls.Config) Option {
    return func(s *Server) {
        s.tlsConfig = config
    }
}

func WithLogger(logger *log.Logger) Option {
    return func(s *Server) {
        s.logger = logger
    }
}

// 构造函数: 必选参数放在签名中, 可选参数通过 Option
func NewServer(addr string, opts ...Option) *Server {
    s := &Server{
        addr:         addr,
        port:         8080,             // 默认值
        readTimeout:  30 * time.Second, // 默认值
        writeTimeout: 30 * time.Second,
        maxConns:     1000,
    }

    for _, opt := range opts {
        opt(s)
    }

    return s
}
```

### 5.2 中间件模式（Middleware） ⭐⭐⭐

**原理**：通过函数嵌套实现请求/响应的前后处理。每个中间件接收一个 handler 并返回一个增强的 handler，形成洋葱模型。

**Go 实现（HTTP 中间件）**：
```go
package middleware

import (
    "context"
    "fmt"
    "log"
    "net/http"
    "time"
)

// 标准中间件签名
type Middleware func(http.Handler) http.Handler

// 日志中间件
func Logger() Middleware {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            start := time.Now()
            // 前处理
            log.Printf("--> %s %s", r.Method, r.URL.Path)

            next.ServeHTTP(w, r) // 调用下一个

            // 后处理
            log.Printf("<-- %s %s (%v)", r.Method, r.URL.Path, time.Since(start))
        })
    }
}

// 认证中间件
func Auth(tokenValidator func(string) bool) Middleware {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            token := r.Header.Get("Authorization")
            if !tokenValidator(token) {
                http.Error(w, "Unauthorized", http.StatusUnauthorized)
                return // 短路: 不调用 next
            }
            // 将用户信息注入 context
            ctx := context.WithValue(r.Context(), "user", "authenticated_user")
            next.ServeHTTP(w, r.WithContext(ctx))
        })
    }
}

// Recover 中间件
func Recover() Middleware {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            defer func() {
                if err := recover(); err != nil {
                    log.Printf("PANIC: %v", err)
                    http.Error(w, "Internal Server Error", http.StatusInternalServerError)
                }
            }()
            next.ServeHTTP(w, r)
        })
    }
}

// CORS 中间件
func CORS(allowedOrigins []string) Middleware {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            origin := r.Header.Get("Origin")
            for _, allowed := range allowedOrigins {
                if origin == allowed || allowed == "*" {
                    w.Header().Set("Access-Control-Allow-Origin", origin)
                    w.Header().Set("Access-Control-Allow-Methods", "GET, POST, PUT, DELETE")
                    w.Header().Set("Access-Control-Allow-Headers", "Content-Type, Authorization")
                    break
                }
            }
            if r.Method == "OPTIONS" {
                w.WriteHeader(http.StatusOK)
                return
            }
            next.ServeHTTP(w, r)
        })
    }
}

// 链式组合
func Chain(handler http.Handler, middlewares ...Middleware) http.Handler {
    for i := len(middlewares) - 1; i >= 0; i-- {
        handler = middlewares[i](handler)
    }
    return handler
}

// 使用:
// mux := http.NewServeMux()
// mux.HandleFunc("/api/data", dataHandler)
//
// handler := Chain(mux,
//     Recover(),
//     Logger(),
//     CORS([]string{"*"}),
//     Auth(validateToken),
// )
// http.ListenAndServe(":8080", handler)
```

### 5.3 Pipeline 模式 ⭐⭐⭐

**原理**：将数据处理拆分为多个阶段（stage），每个阶段通过 channel 连接，数据沿管道流动。每个阶段可以并发处理。

**Go 实现**：
```go
package pipeline

import (
    "context"
    "fmt"
    "math"
    "strings"
)

// Stage 是一个管道阶段, 输入 channel 输出 channel
// 通用管道阶段类型
type Stage[In, Out any] func(ctx context.Context, in <-chan In) <-chan Out

// Generator: 生成数据的第一个阶段
func Generator(ctx context.Context, data ...int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for _, d := range data {
            select {
            case <-ctx.Done():
                return
            case out <- d:
            }
        }
    }()
    return out
}

// Filter: 过滤阶段
func Filter(ctx context.Context, in <-chan int, predicate func(int) bool) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {
            if predicate(n) {
                select {
                case <-ctx.Done():
                    return
                case out <- n:
                }
            }
        }
    }()
    return out
}

// Map: 转换阶段
func Map(ctx context.Context, in <-chan int, transform func(int) int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out)
        for n := range in {
            select {
            case <-ctx.Done():
                return
            case out <- transform(n):
            }
        }
    }()
    return out
}

// Reduce: 聚合阶段
func Reduce(ctx context.Context, in <-chan int, initial int, fn func(int, int) int) int {
    result := initial
    for n := range in {
        select {
        case <-ctx.Done():
            return result
        default:
            result = fn(result, n)
        }
    }
    return result
}

// 使用: 找出 1~100 中所有偶数的平方和
// ctx := context.Background()
// nums := Generator(ctx, 1, 2, 3, ..., 100)
// evens := Filter(ctx, nums, func(n int) bool { return n%2 == 0 })
// squares := Map(ctx, evens, func(n int) int { return n * n })
// sum := Reduce(ctx, squares, 0, func(a, b int) int { return a + b })

// 完整文本处理 Pipeline 示例
func TextPipeline(ctx context.Context, texts []string) <-chan string {
    // Stage 1: 生成
    stage1 := func() <-chan string {
        out := make(chan string)
        go func() {
            defer close(out)
            for _, t := range texts {
                select {
                case <-ctx.Done():
                    return
                case out <- t:
                }
            }
        }()
        return out
    }()

    // Stage 2: 转小写
    stage2 := func(in <-chan string) <-chan string {
        out := make(chan string)
        go func() {
            defer close(out)
            for s := range in {
                select {
                case <-ctx.Done():
                    return
                case out <- strings.ToLower(s):
                }
            }
        }()
        return out
    }(stage1)

    // Stage 3: 去除空白
    stage3 := func(in <-chan string) <-chan string {
        out := make(chan string)
        go func() {
            defer close(out)
            for s := range in {
                trimmed := strings.TrimSpace(s)
                if trimmed != "" {
                    select {
                    case <-ctx.Done():
                        return
                    case out <- trimmed:
                    }
                }
            }
        }()
        return out
    }(stage2)

    return stage3
}

// 判断素数的 Pipeline 示例
func PrimesPipeline(ctx context.Context, limit int) <-chan int {
    // Stage 1: 生成 2 到 limit
    nums := make(chan int)
    go func() {
        defer close(nums)
        for i := 2; i <= limit; i++ {
            select {
            case <-ctx.Done():
                return
            case nums <- i:
            }
        }
    }()

    // Stage 2: 筛选素数
    primes := make(chan int)
    go func() {
        defer close(primes)
        for n := range nums {
            if isPrime(n) {
                select {
                case <-ctx.Done():
                    return
                case primes <- n:
                }
            }
        }
    }()

    return primes
}

func isPrime(n int) bool {
    if n < 2 {
        return false
    }
    for i := 2; i <= int(math.Sqrt(float64(n))); i++ {
        if n%i == 0 {
            return false
        }
    }
    return true
}
```

### 5.4 Fan-out / Fan-in 模式 ⭐⭐⭐

**原理**：
- **Fan-out**：一个 channel 的数据分发给多个 goroutine 并发处理
- **Fan-in**：多个 channel 的结果合并到一个 channel

适用于 CPU 密集型或 I/O 密集型的并发处理场景。

**Go 实现**：
```go
package fanout

import (
    "context"
    "fmt"
    "sync"
)

// Fan-out: 启动 n 个 worker 从同一个 channel 读取
func FanOut[T, R any](
    ctx context.Context,
    input <-chan T,
    workerCount int,
    process func(T) R,
) []<-chan R {
    outputs := make([]<-chan R, workerCount)
    for i := 0; i < workerCount; i++ {
        outputs[i] = worker(ctx, input, process)
    }
    return outputs
}

func worker[T, R any](ctx context.Context, input <-chan T, process func(T) R) <-chan R {
    out := make(chan R)
    go func() {
        defer close(out)
        for item := range input {
            select {
            case <-ctx.Done():
                return
            case out <- process(item):
            }
        }
    }()
    return out
}

// Fan-in: 将多个 channel 合并为一个
func FanIn[T any](ctx context.Context, channels ...<-chan T) <-chan T {
    merged := make(chan T)
    var wg sync.WaitGroup

    for _, ch := range channels {
        wg.Add(1)
        go func(c <-chan T) {
            defer wg.Done()
            for item := range c {
                select {
                case <-ctx.Done():
                    return
                case merged <- item:
                }
            }
        }(ch)
    }

    go func() {
        wg.Wait()
        close(merged)
    }()

    return merged
}

// 完整示例: 并发处理 URL 列表
type URLResult struct {
    URL    string
    Status int
    Err    error
}

func ProcessURLs(ctx context.Context, urls []string, concurrency int) <-chan URLResult {
    // 1. 生成任务
    input := make(chan string)
    go func() {
        defer close(input)
        for _, url := range urls {
            select {
            case <-ctx.Done():
                return
            case input <- url:
            }
        }
    }()

    // 2. Fan-out: 多个 worker 并发处理
    process := func(url string) URLResult {
        // 模拟 HTTP 请求
        return URLResult{URL: url, Status: 200}
    }
    workers := FanOut(ctx, input, concurrency, process)

    // 3. Fan-in: 合并结果
    return FanIn(ctx, workers...)
}

// 带限流的 Worker Pool
func WorkerPool[T, R any](
    ctx context.Context,
    tasks []T,
    workerCount int,
    process func(context.Context, T) (R, error),
) ([]R, []error) {
    taskCh := make(chan T, len(tasks))
    type result struct {
        val R
        err error
    }
    resultCh := make(chan result, len(tasks))

    // 启动 worker
    var wg sync.WaitGroup
    for i := 0; i < workerCount; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for task := range taskCh {
                val, err := process(ctx, task)
                resultCh <- result{val: val, err: err}
            }
        }()
    }

    // 发送任务
    for _, task := range tasks {
        taskCh <- task
    }
    close(taskCh)

    // 等待完成
    go func() {
        wg.Wait()
        close(resultCh)
    }()

    // 收集结果
    var results []R
    var errs []error
    for r := range resultCh {
        if r.err != nil {
            errs = append(errs, r.err)
        } else {
            results = append(results, r.val)
        }
    }
    return results, errs
}
```

### 6.2 实体（Entity）

**原理**：有唯一标识的对象，通过 ID 而非属性来区分。同一个人改了名字还是同一个人。

```go
package domain

import (
    "time"

    "github.com/google/uuid"

// 实体: 有唯一 ID
type User struct {
    id        string
    name      string
    email     string
    createdAt time.Time
    updatedAt time.Time
}

func NewUser(name, email string) *User {
    return &User{
        id:        uuid.New().String(),
        name:      name,
        email:     email,
        createdAt: time.Now(),
        updatedAt: time.Now(),
    }
}

func (u *User) ID() string       { return u.id }
func (u *User) Name() string     { return u.name }
func (u *User) Email() string    { return u.email }

func (u *User) ChangeName(name string) {
    u.name = name
    u.updatedAt = time.Now()
}

// 实体通过 ID 比较相等性
```

### 6.3 值对象（Value Object）

**原理**：没有唯一标识，通过所有属性值来判等。值对象应该是不可变的。

```go
// 值对象: Money
type Currency string

const (
    CNY Currency = "CNY"
    USD Currency = "USD"
)

type Money struct {
    amount   int64    // 用最小单位(分)避免浮点问题
    currency Currency
}

func NewMoney(amount int64, currency Currency) Money {
    return Money{amount: amount, currency: currency}
}

func (m Money) Amount() int64      { return m.amount }
func (m Money) Currency() Currency { return m.currency }

// 值对象通过所有属性比较
func (m Money) Equals(other Money) bool {
    return m.amount == other.amount && m.currency == other.currency
}

// 值对象的操作返回新对象(不可变)
func (m Money) Add(other Money) (Money, error) {
    if m.currency != other.currency {
        return Money{}, fmt.Errorf("currency mismatch: %s vs %s", m.currency, other.currency)
    }
    return NewMoney(m.amount+other.amount, m.currency), nil
}

func (m Money) Multiply(factor int64) Money {
    return NewMoney(m.amount*factor, m.currency)
}

// 值对象: Address
type Address struct {
    province string
    city     string
    district string
    street   string
    zipCode  string
}

func NewAddress(province, city, district, street, zipCode string) Address {
    return Address{
        province: province,
        city:     city,
        district: district,
        street:   street,
        zipCode:  zipCode,
    }
}

func (a Address) FullAddress() string {
    return fmt.Sprintf("%s%s%s%s", a.province, a.city, a.district, a.street)
}

func (a Address) Equals(other Address) bool {
    return a == other // struct 比较
}
```

### 6.4 聚合根（Aggregate Root）

**原理**：聚合是一组相关对象的集合，聚合根是外部访问聚合内部对象的唯一入口。聚合根负责维护聚合的一致性约束。

```go
// 聚合根: Order
type OrderStatus string

const (
    OrderStatusPending   OrderStatus = "PENDING"
    OrderStatusConfirmed OrderStatus = "CONFIRMED"
    OrderStatusCancelled OrderStatus = "CANCELLED"
)

type OrderItem struct {
    ProductID string
    Name      string
    Price     Money
    Quantity  int
}

func (item OrderItem) Subtotal() Money {
    return item.Price.Multiply(int64(item.Quantity))
}

type Order struct {
    id        string
    userID    string
    items     []OrderItem
    status    OrderStatus
    address   Address   // 值对象
    createdAt time.Time
    events    []DomainEvent // 领域事件
}

func NewOrder(userID string, address Address) *Order {
    order := &Order{
        id:        uuid.New().String(),
        userID:    userID,
        status:    OrderStatusPending,
        address:   address,
        createdAt: time.Now(),
    }
    order.addEvent(OrderCreatedEvent{
        OrderID: order.id,
        UserID:  userID,
        At:      order.createdAt,
    })
    return order
}

// 聚合根方法 — 维护业务不变量
func (o *Order) AddItem(productID, name string, price Money, qty int) error {
    if o.status != OrderStatusPending {
        return fmt.Errorf("cannot add items to %s order", o.status)
    }
    if qty <= 0 {
        return fmt.Errorf("quantity must be positive")
    }
    o.items = append(o.items, OrderItem{
        ProductID: productID,
        Name:      name,
        Price:     price,
        Quantity:  qty,
    })
    return nil
}

func (o *Order) TotalAmount() Money {
    total := NewMoney(0, CNY)
    for _, item := range o.items {
        subtotal := item.Subtotal()
        total, _ = total.Add(subtotal)
    }
    return total
}

func (o *Order) Confirm() error {
    if o.status != OrderStatusPending {
        return fmt.Errorf("can only confirm pending orders")
    }
    if len(o.items) == 0 {
        return fmt.Errorf("cannot confirm empty order")
    }
    o.status = OrderStatusConfirmed
    o.addEvent(OrderConfirmedEvent{
        OrderID: o.id,
        Total:   o.TotalAmount(),
        At:      time.Now(),
    })
    return nil
}

func (o *Order) Cancel() error {
    if o.status == OrderStatusCancelled {
        return fmt.Errorf("already cancelled")
    }
    o.status = OrderStatusCancelled
    o.addEvent(OrderCancelledEvent{OrderID: o.id, At: time.Now()})
    return nil
}

func (o *Order) addEvent(event DomainEvent) {
    o.events = append(o.events, event)
}

func (o *Order) DomainEvents() []DomainEvent {
    return o.events
}

func (o *Order) ClearEvents() {
    o.events = nil
}
```

### 6.5 领域服务（Domain Service）

**原理**：当业务逻辑不自然地属于任何实体或值对象时，放在领域服务中。

```go

    total := order.TotalAmount()
    var discountRate int64

    switch userLevel {
    case "VIP":
        discountRate = 20 // 8折
    case "SVIP":
        discountRate = 30 // 7折
    default:
        discountRate = 0
    }

    discount := NewMoney(total.Amount()*discountRate/100, total.Currency())
    return discount

// 转账服务 — 涉及两个账户聚合
type TransferService struct {
    accountRepo AccountRepository
}

func (s *TransferService) Transfer(fromID, toID string, amount Money) error {
    from, err := s.accountRepo.FindByID(fromID)
    if err != nil {
        return err
    }
    to, err := s.accountRepo.FindByID(toID)
    if err != nil {
        return err
    }

    if err := from.Debit(amount); err != nil {
        return fmt.Errorf("debit failed: %w", err)
    }
    if err := to.Credit(amount); err != nil {
        return fmt.Errorf("credit failed: %w", err)
    }

    // 持久化
    if err := s.accountRepo.Save(from); err != nil {
        return err
    }
    return s.accountRepo.Save(to)
}
```

## 7. 依赖注入 ⭐⭐

### 7.1 原理

**依赖注入（DI）**：将对象的依赖关系从内部创建转移到外部注入。是依赖倒置原则（DIP）的具体实现手段。

```
[无 DI] Service 自己创建依赖
    type Service struct{}
    func (s *Service) Do() {
        repo := &MySQLRepo{} // 硬编码依赖
    }

[有 DI] 依赖由外部注入
    type Service struct {
        repo Repository // 接口
    }
    func NewService(repo Repository) *Service {
        return &Service{repo: repo}
    }
```

**三种注入方式**：

```go
// 1. 构造函数注入 (推荐)
func NewService(repo Repository, logger Logger) *Service {
    return &Service{repo: repo, logger: logger}
}

// 2. 方法注入
func (s *Service) SetRepo(repo Repository) {
    s.repo = repo
}

// 3. 接口注入 (Go 中少见)
type RepositoryAware interface {
    SetRepository(repo Repository)
}
```

### 7.2 Go Wire（编译时 DI）

**原理**：Google 的 Wire 通过代码生成在编译时解析依赖图，没有运行时开销。

```go
// wire.go (Wire 配置文件)
//go:build wireinject

package main

import "github.com/google/wire"

// 定义 Provider — 提供某个类型的函数
func NewMySQLRepo(dsn string) *MySQLRepo {
    return &MySQLRepo{dsn: dsn}
}

func NewOrderService(repo OrderRepository) *OrderService {
    return &OrderService{repo: repo}
}

func NewOrderHandler(svc *OrderService) *OrderHandler {
    return &OrderHandler{svc: svc}
}

// Provider Set — 将相关的 Provider 分组
var RepoSet = wire.NewSet(
    NewMySQLRepo,
    wire.Bind(new(OrderRepository), new(*MySQLRepo)), // 接口绑定
)

var ServiceSet = wire.NewSet(
    NewOrderService,
)

var HandlerSet = wire.NewSet(
    NewOrderHandler,
)

// Injector — Wire 根据这个函数生成代码
func InitializeApp(dsn string) (*OrderHandler, error) {
    wire.Build(RepoSet, ServiceSet, HandlerSet)
    return nil, nil // Wire 会替换这个返回
}
```

### 7.3 Go Fx（运行时 DI）

**原理**：Uber 的 Fx 框架通过反射在运行时解析依赖，支持生命周期管理。

```go
package main

import (
    "context"
    "net/http"

    "go.uber.org/fx"
    "go.uber.org/zap"
)

func main() {
    app := fx.New(
        // 提供依赖
        fx.Provide(
            NewConfig,
            NewLogger,
            NewMySQLRepo,
            NewOrderService,
            NewHTTPServer,
        ),

        // 绑定接口
        fx.Provide(
            fx.Annotate(
                NewMySQLRepo,
                fx.As(new(OrderRepository)),
            ),
        ),

        // 触发启动
        fx.Invoke(StartHTTPServer),
    )

    app.Run()
}

func NewConfig() *Config {
    return &Config{DSN: "root:pass@tcp(localhost:3306)/mydb"}
}

func NewLogger() (*zap.Logger, error) {
    return zap.NewProduction()
}

func NewHTTPServer(lc fx.Lifecycle, handler *OrderHandler) *http.Server {
    srv := &http.Server{Addr: ":8080", Handler: handler}

    lc.Append(fx.Hook{
        OnStart: func(ctx context.Context) error {
            go srv.ListenAndServe()
            return nil
        },
        OnStop: func(ctx context.Context) error {
            return srv.Shutdown(ctx)
        },
    })

    return srv
}

func StartHTTPServer(*http.Server) {} // 触发 Server 的创建
```

## 8. 反模式 ⭐⭐

### 8.1 God Object（上帝对象）

**特征**：一个结构体/类承担了过多职责，知道太多、做太多。

```go
// BAD: God Object
type Application struct {
    db            *sql.DB
    cache         *redis.Client
    emailClient   *smtp.Client
    config        map[string]string
    users         map[string]*User
    orders        map[string]*Order
    // ... 100+ fields
}

func (app *Application) CreateUser(...)   {}
func (app *Application) SendEmail(...)    {}
func (app *Application) ProcessOrder(...) {}
func (app *Application) UpdateCache(...)  {}
func (app *Application) ValidateToken(...) {}
// ... 200+ methods
```

**重构方法**：按职责拆分为多个服务。
```go
// GOOD: 拆分
type UserService struct { repo UserRepository }
type OrderService struct { repo OrderRepository }
type EmailService struct { client *smtp.Client }
type CacheService struct { client *redis.Client }
```

### 8.2 Spaghetti Code（意大利面条代码）

**特征**：逻辑纠缠不清，大量嵌套 if/else，缺乏结构。

```go
// BAD: 深层嵌套, 逻辑难以跟踪
func ProcessOrder(order Order) error {
    if order.Status == "pending" {
        if order.Total > 0 {
            if order.UserID != "" {
                user, err := getUser(order.UserID)
                if err == nil {
                    if user.Active {
                        if user.Balance >= order.Total {
                            // 终于到了业务逻辑...
                            // 5-6 层嵌套
                        }
                    }
                }
            }
        }
    }
    return nil
}
```

**重构方法**：早返回 + 拆分函数。
```go
// GOOD: 早返回, 扁平化
func ProcessOrder(order Order) error {
    if order.Status != "pending" {
        return fmt.Errorf("order not pending")
    }
    if order.Total <= 0 {
        return fmt.Errorf("invalid total")
    }
    if order.UserID == "" {
        return fmt.Errorf("missing user ID")
    }

    user, err := getUser(order.UserID)
    if err != nil {
        return fmt.Errorf("get user: %w", err)
    }
    if !user.Active {
        return fmt.Errorf("user inactive")
    }
    if user.Balance < order.Total {
        return fmt.Errorf("insufficient balance")
    }

    return executeOrder(order, user)
}
```

### 8.3 过度设计（Over-Engineering）

**特征**：为未来可能不存在的需求创建过多的抽象层。

```go
// BAD: 只有一种实现却创建了三层抽象
type DataAccessor interface {
    Access() DataProvider
}

type DataProvider interface {
    Provide() DataTransformer
}

type DataTransformer interface {
    Transform() DataPersister
}

type DataPersister interface {
    Persist(data []byte) error
}

// 实际上只需要:
type Storage interface {
    Save(data []byte) error
}
```

**判断标准**：
- YAGNI（You Ain't Gonna Need It）— 不需要的就不做
- 三次法则：同样的逻辑重复三次再抽象
- 如果接口只有一个实现，考虑是否真的需要接口

### 8.4 循环依赖（Circular Dependency）

**特征**：包 A 依赖包 B，包 B 又依赖包 A。Go 编译器会直接报错。

```
// BAD: 循环依赖
package order  →  import "myapp/user"
package user   →  import "myapp/order"  // 编译错误!
```

**重构方法**：

```go
// 方法 1: 提取接口到独立包
// package interfaces
type UserGetter interface {
    GetUser(id string) (*User, error)
}

// package order — 依赖接口包
type OrderService struct {
    userGetter interfaces.UserGetter
}

// 方法 2: 依赖倒置 — 在高层定义接口
// package order
type UserProvider interface {
    GetUser(id string) (*User, error)
}

// package user — 实现 order 包定义的接口
type UserService struct{}
func (s *UserService) GetUser(id string) (*User, error) { /* ... */ return nil, nil }

// 方法 3: 合并（如果两个包确实紧密相关）
```

### 8.5 反模式识别清单

| 反模式 | 信号 | 重构方向 |
|--------|------|---------|
| God Object | 文件 > 1000 行, 方法 > 20 个 | 拆分职责 |
| Spaghetti Code | 嵌套 > 3 层, 函数 > 50 行 | 早返回, 提取函数 |
| 过度设计 | 接口只有 1 个实现, 层数 > 需要 | 简化, YAGNI |
| 循环依赖 | Go 编译报错 | 提取接口, 依赖倒置 |
| Shotgun Surgery | 改一个功能要改 10+ 文件 | 内聚, 减少散弹式修改 |
| Feature Envy | 方法大量访问其他结构体字段 | 移动方法到数据所在处 |
| Primitive Obsession | 到处用 string/int 表示业务概念 | 引入值对象 |

### 9.2 重构后（设计模式组合）

```go
// ============================================================
// 领域层: domain/order/
// ============================================================

// 值对象
type Money struct {
    amount   int64
    currency string
}

func NewMoney(amount int64, currency string) Money {
    return Money{amount: amount, currency: currency}
}

func (m Money) Add(other Money) Money {
    return NewMoney(m.amount+other.amount, m.currency)
}

func (m Money) MultiplyFloat(rate float64) Money {
    return NewMoney(int64(float64(m.amount)*rate), m.currency)
}

// 实体 + 聚合根
type OrderItem struct {
    ProductID string
    Name      string
    Price     Money
    Quantity  int
}

type Order struct {
    ID        string
    UserID    string
    Items     []OrderItem
    Status    string
    Discount  Money
    CreatedAt time.Time
    events    []DomainEvent
}

func NewOrder(userID string, items []OrderItem) (*Order, error) {
    if len(items) == 0 {
        return nil, fmt.Errorf("order must have at least one item")
    }
    o := &Order{
        ID:        uuid.New().String(),
        UserID:    userID,
        Items:     items,
        Status:    "pending",
        CreatedAt: time.Now(),
    }
    o.events = append(o.events, OrderCreatedEvent{OrderID: o.ID, UserID: userID})
    return o, nil
}

func (o *Order) Total() Money {
    total := NewMoney(0, "CNY")
    for _, item := range o.Items {
        total = total.Add(item.Price.MultiplyFloat(float64(item.Quantity)))
    }
    // 减去折扣
    return NewMoney(total.amount-o.Discount.amount, total.currency)
}

func (o *Order) ApplyDiscount(discount Money) {
    o.Discount = discount
}

// 仓储接口 (领域层定义)
type OrderRepository interface {
    Save(order *Order) error
    FindByID(id string) (*Order, error)
}

// ============================================================
// 策略模式: 折扣计算
// ============================================================

type DiscountStrategy interface {
    Calculate(total Money, userLevel int) Money
}

type NormalDiscount struct{}

func (d *NormalDiscount) Calculate(total Money, userLevel int) Money {
    return NewMoney(0, total.currency)
}

type VIPDiscount struct{}

func (d *VIPDiscount) Calculate(total Money, userLevel int) Money {
    return total.MultiplyFloat(0.1) // 9折
}

type SVIPDiscount struct{}

func (d *SVIPDiscount) Calculate(total Money, userLevel int) Money {
    return total.MultiplyFloat(0.2) // 8折
}

// 工厂方法: 根据用户等级选择策略
func NewDiscountStrategy(userLevel int) DiscountStrategy {
    switch userLevel {
    case 1:
        return &VIPDiscount{}
    case 2:
        return &SVIPDiscount{}
    default:
        return &NormalDiscount{}
    }
}

// ============================================================
// 策略模式: 支付

    }
```

## 10. 面试题自查

### 10.1 治理补充（3 条经验）

1. **模式不是银弹** — 在引入设计模式前，先确认问题确实存在。过早引入模式会导致过度设计。遵循"三次法则"：同样的代码重复三次时才考虑抽象。

2. **Go 的惯用模式 > 经典 OOP 模式** — Go 没有继承、泛型（1.18 前），应优先使用函数选项模式、中间件模式、channel 模式，而不是生搬 Java 的经典 GoF 实现。

3. **可测试性是设计好坏的试金石** — 如果代码难以编写单元测试，通常意味着耦合度过高。通过接口 + 依赖注入解耦，使用 mock 替换外部依赖。

### 10.2 初级高频速刷（5 题）

**Q1: 什么是单例模式？Go 中如何实现线程安全的单例？**
A: 单例确保全局只有一个实例。Go 用 `sync.Once` + 包级变量实现，`once.Do()` 保证初始化函数只执行一次，天然线程安全。

**Q2: 工厂模式和抽象工厂模式的区别？**
A: 工厂方法创建一种产品（如 `NewPaymentProcessor("alipay")`）；抽象工厂创建一族相关产品（如 `UIFactory` 同时创建 Button + Input，保证风格一致）。

**Q3: 代理模式和装饰器模式的区别？**
A: 代理控制访问（权限、延迟加载），自己管理被代理对象的生命周期；装饰器增强功能，被装饰对象由外部传入。代理关注"能不能"，装饰器关注"加什么"。

**Q4: 策略模式和状态模式的区别？**
A: 策略由客户端选择使用哪个算法，策略间互不知晓；状态模式中状态对象自己决定何时转换到下一个状态，状态间有关联。

**Q5: SOLID 中哪个原则在 Go 中体现最明显？**
A: ISP（接口隔离）。Go 标准库推崇小接口：`io.Reader`（1 方法）、`fmt.Stringer`（1 方法）。Go 的隐式接口实现天然鼓励按需定义小接口。

---

### 10.3 中级高频进阶（5 题）

**Q1: Go 的函数选项模式和 Builder 模式各自适用什么场景？**
A: 函数选项模式更 Go 风格，适合构造函数参数多但类型明确的场景（如 `NewServer(addr, WithPort(8080), WithTLS())`）。Builder 模式适合构建步骤有序、需要分步验证的场景（如 SQL 查询构建器）。函数选项更轻量，Builder 可以保存中间状态。

**Q2: 如何在 Go 中实现装饰器模式而不用继承？**
A: 利用接口 + 组合。装饰器结构体持有一个接口类型的字段（被装饰对象），自身也实现同一接口。在方法中先/后执行增强逻辑，中间调用被装饰对象的方法。多个装饰器可以嵌套叠加。

**Q3: DDD 中聚合根的设计原则？**
A: ① 聚合根是外部访问聚合的唯一入口 ② 聚合内保持事务一致性 ③ 聚合间通过 ID 引用而非对象引用 ④ 聚合尽量小 ⑤ 修改聚合内部对象必须通过聚合根的方法。

**Q4: 中间件模式的"洋葱模型"是什么？**
A: 请求从外向内依次经过每个中间件的前处理逻辑，到达最内层的 handler，然后从内向外依次经过每个中间件的后处理逻辑。形成 `M1前 → M2前 → Handler → M2后 → M1后` 的调用链。

**Q5: Wire 和 Fx 应该怎么选？**
A: Wire 编译时生成代码，零运行时开销，错误在编译期发现，适合性能敏感或中小项目。Fx 运行时反射解析，支持生命周期钩子（OnStart/OnStop），适合需要优雅启停的复杂微服务。

**Q1: Go 没有继承，如何实现模板方法模式？**

A: 通过接口定义可变步骤 + 普通函数定义模板骨架。

```go
// 定义可变步骤接口
type Steps interface {
    Step1() error
    Step2() error
    Step3() error
}

// 模板函数 — 定义固定骨架
func RunTemplate(s Steps) error {
    if err := s.Step1(); err != nil {
        return err
    }
    // 固定逻辑...
    if err := s.Step2(); err != nil {
        return err
    }
    // 固定逻辑...
    return s.Step3()
}
```

与 Java 的区别：Java 用 abstract class + final 模板方法 + abstract 钩子方法；Go 用接口 + 顶层函数，更灵活（不需要继承关系）。

**Q2: `sync.Once` 实现单例有什么潜在问题？如何解决？**

A: 问题：`once.Do(f)` 如果 f 中发生 panic，后续调用不会再执行 f（`Once` 标记为已完成），导致 instance 为 nil。

```go
var (
    instance *Config
    once     sync.Once
    initErr  error
)

func GetInstance() (*Config, error) {
    once.Do(func() {
        instance, initErr = loadConfig() // 可能失败
    })
    // 如果 loadConfig 失败, 后续调用都返回 nil + error
    return instance, initErr
}
```

如果需要重试初始化，不要用 `sync.Once`，改用 `sync.Mutex` + 状态标志。

**Q3: 观察者模式在高并发下可能有什么问题？如何优化？**

A: 问题：
1. 同步通知时，慢的观察者阻塞整个发布流程
2. 观察者列表的并发读写需要加锁
3. 观察者中 panic 导致后续观察者收不到通知

优化方案：
```go
func (bus *EventBus) PublishAsync(event Event) {
    bus.mu.RLock()
    observers := make([]Observer, len(bus.observers[event.Type]))
    copy(observers, bus.observers[event.Type]) // 快照, 减少锁持有时间
    bus.mu.RUnlock()

    for _, observer := range observers {
        go func(o Observer) {
            defer func() {
                if r := recover(); r != nil {
                    log.Printf("observer panic: %v", r)
                }
            }()
            o.OnNotify(event)
        }(observer)
    }
}
```

或使用 channel 解耦：每个观察者有独立的 buffered channel，发布者非阻塞写入。

**Q4: 责任链模式和中间件模式的关系是什么？**

A: 中间件模式是责任链模式在 Web 框架中的具体应用。

区别：
- 经典责任链：每个节点决定处理或传递，只有一个节点最终处理请求
- 中间件链：每个中间件都可以做前处理和后处理，请求经过所有中间件（除非短路）

中间件是"增强型责任链" — 洋葱模型允许在 `next()` 调用前后都添加逻辑。

**Q5: 如何用 Go channel 实现观察者模式？相比接口回调有什么优缺点？**

A:
```go
type EventBus struct {
    subscribers map[string][]chan Event
    mu          sync.RWMutex
}

func (bus *EventBus) Subscribe(topic string) <-chan Event {
    ch := make(chan Event, 32) // buffered 防阻塞
    bus.mu.Lock()
    bus.subscribers[topic] = append(bus.subscribers[topic], ch)
    bus.mu.Unlock()
    return ch
}

func (bus *EventBus) Publish(topic string, event Event) {
    bus.mu.RLock()
    defer bus.mu.RUnlock()
    for _, ch := range bus.subscribers[topic] {
        select {
        case ch <- event:
        default: // 满了就丢弃
        }
    }
}
```

| 维度 | Channel | 接口回调 |
|------|---------|---------|
| **异步** | 天然异步 | 需要额外 goroutine |
| **解耦** | 更彻底（不需要定义接口） | 需要定义 Observer 接口 |
| **背压** | 有 buffer 可控 | 无内置背压 |
| **取消订阅** | 需要维护 channel 关闭逻辑 | 从列表移除即可 |
| **调试** | 较难追踪数据流 | 调用栈清晰 |

**Q6: Go 的接口隐式实现如何体现 ISP 原则？**

A: Go 不需要 `implements` 关键字，任何类型只要实现了接口的所有方法就自动满足该接口。这意味着：

1. 消费方可以在自己的包里定义最小接口（而不是依赖提供方的大接口）
2. 同一个结构体可以满足不同包中定义的不同小接口

```go
// package A 只需要读
type Reader interface { Read(p []byte) (int, error) }

// package B 只需要关闭
type Closer interface { Close() error }

// os.File 同时满足两个接口, 但 A 和 B 只依赖自己需要的部分
```

这种"消费方定义接口"的模式是 Go 的 ISP 最佳实践。

**Q7: Fan-out/Fan-in 模式中如何处理错误？如何实现优雅退出？**

A:
```go
func FanOutWithCancel(ctx context.Context, urls []string, workers int) ([]Result, error) {
    ctx, cancel := context.WithCancel(ctx)
    defer cancel()

    g, ctx := errgroup.WithContext(ctx)
    taskCh := make(chan string, len(urls))
    resultCh := make(chan Result, len(urls))

    // 发送任务
    go func() {
        defer close(taskCh)
        for _, url := range urls {
            select {
            case <-ctx.Done():
                return
            case taskCh <- url:
            }
        }
    }()

    // Fan-out workers
    for i := 0; i < workers; i++ {
        g.Go(func() error {
            for url := range taskCh {
                result, err := fetch(ctx, url)
                if err != nil {
                    return err // errgroup 会 cancel context
                }
                select {
                case resultCh <- result:
                case <-ctx.Done():
                    return ctx.Err()
                }
            }
            return nil
        })
    }

    // 关闭结果 channel
    go func() {
        g.Wait()
        close(resultCh)
    }()

    // 收集结果
    var results []Result
    for r := range resultCh {
        results = append(results, r)
    }

    return results, g.Wait()
}
```

关键点：用 `errgroup` + `context.WithCancel` 实现"一个 worker 失败则全部取消"。

**Q8: DDD 中值对象和实体的区别？什么时候用值对象？**

**Q9: 工厂方法模式中，Go 用注册表模式比 switch/case 好在哪里？**

```go
// switch/case — 违反 OCP, 新增类型必须修改此函数

    case "a": return &A{}
```

**Q11: Pipeline 模式中 channel 满了怎么办？如何避免 goroutine 泄漏？**

A:
1. **channel 满了**：用 select + context.Done 设置超时/取消
2. **goroutine 泄漏**：确保所有 goroutine 都能感知到 context 取消

```go
func Stage(ctx context.Context, in <-chan int) <-chan int {
    out := make(chan int)
    go func() {
        defer close(out) // 确保下游能感知到结束
        for {
            select {
            case <-ctx.Done():
                return // 感知取消, 退出 goroutine
            case val, ok := <-in:
                if !ok {
                    return // 上游关闭, 退出
                }
                select {
                case <-ctx.Done():
                    return
                case out <- val * 2:
                }
            }
        }
    }()
    return out
}
```

规则：① 每个 stage 的 goroutine 都要监听 `ctx.Done()` ② 输出 channel 在 goroutine 退出时 `close` ③ 用 `defer` 确保资源释放。

**Q13: 享元模式和对象池（sync.Pool）的区别？**

A:
- **享元**：共享不可变的内部状态，同一个享元对象被多个客户端同时使用（只读共享）
- **对象池**：复用可变对象，一次只有一个使用者，用完归还

```go
// 享元: TextStyle 被多处引用, 共享读
style := factory.GetStyle("Arial", 12) // 返回同一个 *TextStyle

buf := pool.Get().(*bytes.Buffer)
buf.Reset()
// ... 使用
pool.Put(buf) // 归还
```

**Q15: Go 中如何实现不可变值对象？**

**Q16: 组合模式中如何防止循环引用（目录 A 包含目录 B，B 又包含 A）？**

A:
```go
func (d *Directory) Add(node FileSystemNode) error {
    // 检查是否会形成循环
    if dir, ok := node.(*Directory); ok {
        if d.isDescendantOf(dir) || d == dir {
            return fmt.Errorf("circular reference detected")
        }
    }
    d.children = append(d.children, node)
    return nil
}

func (d *Directory) isDescendantOf(ancestor *Directory) bool {
    for _, child := range ancestor.children {
        if child == d {
            return true
        }
        if dir, ok := child.(*Directory); ok {
            if d.isDescendantOf(dir) {
                return true
            }
        }
    }
    return false
}
```

**Q18: 命令模式中的 Undo 如何保证与 Execute 的一致性？**

A: 关键是在 Execute 时保存足够的"反向操作信息"：

```go
type DeleteCommand struct {
    editor  *TextEditor
    pos     int
    length  int
    deleted string // Execute 时保存被删除的内容
}

func (c *DeleteCommand) Execute() error {
    c.deleted = c.editor.Delete(c.pos, c.length) // 保存
    return nil
}

func (c *DeleteCommand) Undo() error {
    c.editor.Insert(c.deleted, c.pos) // 恢复
    return nil
}
```

对于复杂操作，可以使用 Memento 模式（保存完整快照）配合 Command。

**Q19: 如何选择策略模式的接口版和函数版？**

```go
// 函数版: 策略逻辑简单, 无状态
type HashFunc func(key string) uint32

// 接口版: 策略有多个方法或需要状态
```

**Q20: DDD 的聚合应该多大？如何划分聚合边界？**

A: 原则：
1. **尽量小** — 一个聚合只包含必须保持一致性的对象
2. **事务边界** — 一次事务只修改一个聚合
3. **通过 ID 引用** — 聚合间通过 ID 而非对象引用关联
4. **业务不变量** — 聚合根负责维护的约束决定了边界

```go
// BAD: Order 聚合包含了 User
type Order struct {
    User  *User       // 对象引用, 聚合太大
    Items []OrderItem
}

// GOOD: Order 聚合只通过 ID 引用 User
type Order struct {
    UserID string      // ID 引用
    Items  []OrderItem
}
```

**Q21: Go 中如何实现线程安全的注册表工厂？**

A:
```go
type Registry[T any] struct {
    mu       sync.RWMutex
    creators map[string]func() T
}

func NewRegistry[T any]() *Registry[T] {
    return &Registry[T]{creators: make(map[string]func() T)}
}

func (r *Registry[T]) Register(name string, creator func() T) {
    r.mu.Lock()
    defer r.mu.Unlock()
    r.creators[name] = creator
}

func (r *Registry[T]) Create(name string) (T, error) {
    r.mu.RLock()
    defer r.mu.RUnlock()
    creator, ok := r.creators[name]
    if !ok {
        var zero T
        return zero, fmt.Errorf("unknown type: %s", name)
    }
    return creator(), nil
}
```

注意：如果注册只在 `init()` 阶段发生（Go 保证 init 串行执行），读操作不需要加锁。

**Q22: 代理模式在微服务中有哪些应用？**

A:
1. **API Gateway** — 远程代理，转发请求到后端服务
2. **Service Mesh (Envoy/Istio)** — sidecar 代理，透明拦截网络流量
3. **熔断器** — 保护代理，当下游故障率过高时直接返回错误
4. **缓存代理** — 在 Redis 前加一层本地缓存

```go
// 熔断器代理示例
type CircuitBreakerProxy struct {
    service    Service
    failures   int32
    threshold  int32
    lastFail   time.Time
    resetAfter time.Duration
}

func (p *CircuitBreakerProxy) Call(req Request) (Response, error) {
    if atomic.LoadInt32(&p.failures) >= p.threshold {
        if time.Since(p.lastFail) < p.resetAfter {
            return Response{}, ErrCircuitOpen
        }
        atomic.StoreInt32(&p.failures, 0) // 半开状态
    }

    resp, err := p.service.Call(req)
    if err != nil {
        atomic.AddInt32(&p.failures, 1)
        p.lastFail = time.Now()
    }
    return resp, err
}
```

**Q24: 如何判断是否应该引入设计模式？**

A: 判断标准（YAGNI + 三次法则 + 痛点驱动）：

1. **有没有痛点**？代码难改、难测、难读 → 需要重构
2. **痛点频率**？改动频繁的部分才值得投入 → 稳定的代码不要动
3. **三次法则**：同样的变化模式出现三次 → 抽象
4. **增加的复杂度是否值得**？引入模式后代码是更清晰了还是更晦涩了
5. **团队能理解吗**？过于"聪明"的设计不如简单直接的实现

Go 社区格言："A little copying is better than a little dependency."

**Q25: 如何在 Go 项目中落地 DDD？有哪些常见的坑？**

A: 落地步骤：
1. **事件风暴（Event Storming）**：与业务方一起识别领域事件和聚合边界
2. **分层架构**：domain / application / infrastructure / interfaces 四层
3. **领域层零外部依赖**：repository 接口在 domain 层定义，实现在 infrastructure 层

常见的坑：
```go
// 坑 1: 贫血模型 — 聚合根变成纯数据结构
type Order struct {
    ID     string
    Status string
    Items  []OrderItem
    // 没有任何业务方法... 逻辑全在 Service 里
}

// 坑 2: 聚合太大 — 把不相关的东西塞进一个聚合
type Order struct {
    User     *User     // 应该用 UserID
    Products []Product // 应该用 ProductID
    Payments []Payment // 应该是独立聚合
}

// 坑 3: 滥用领域事件 — 同步事件导致性能问题
// 应区分: 聚合内用直接调用, 聚合间用异步事件

// 坑 4: 过早引入 CQRS/Event Sourcing
// 先用简单的 DDD 分层, 等复杂度真的需要时再引入
```

### 开放式设计题

**D1：你接手了一个没有任何设计模式、代码高度耦合的遗留系统（God Object + Spaghetti Code），如何渐进式重构？**

**参考思路**：
- 第一步：加测试覆盖（先有安全网再动代码）、关键路径写集成测试、核心函数写单元测试
- 第二步：提取接口（依赖反转），把直接依赖改为接口依赖，方便后续替换实现
- 第三步：按职责拆分God Object（每次提取一个清晰的职责类）、用Facade维持旧API不变
- 第四步：引入策略模式/模板方法替换大量if-else/switch-case
- 原则：Strangler Fig（逐步替代而非重写）、每次重构都通过CI验证、不追求完美只追求方向正确

**D2：在Go项目中如何落地DDD？Go的语言特性对DDD有哪些独特的影响？**

**参考思路**：
- 项目结构：domain/（实体+值对象+仓储接口）、application/（用例编排）、infrastructure/（仓储实现+外部服务）、interfaces/（HTTP/gRPC入口）
- Go特色：无继承→聚合组合代替、interface隐式实现→天然的端口适配器、值类型结构体→天然的值对象
- 领域事件：Go channel或消息队列实现事件驱动、注意channel在同进程内、跨服务用Kafka
- 常见坑：贫血模型（实体只有字段没有行为）、聚合过大（一个聚合包含太多实体）、过度设计
