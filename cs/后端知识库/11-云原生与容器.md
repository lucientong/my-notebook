## 1. Docker容器技术

### 1.1 什么是容器？

**容器 vs 虚拟机**：

| 特性 | 容器 | 虚拟机 |
|------|------|--------|
| **隔离级别** | 进程级别 | 操作系统级别 |
| **启动速度** | 秒级 | 分钟级 |
| **资源占用** | MB级别 | GB级别 |
| **性能** | 接近原生 | 有损耗 |
| **可移植性** | 高（镜像） | 低 |

### 1.2 Docker底层原理⭐⭐⭐

**为什么需要了解Docker底层？**
- 面试高频：容器是如何实现隔离的？
- 故障排查：为什么容器内看不到宿主机进程？
- 性能调优：如何限制容器资源？

#### Namespace（命名空间）：隔离视图

**6种Namespace**：

```
1. PID Namespace：进程隔离
   - 容器内进程看到的PID从1开始
   - 宿主机看到真实的PID

2. Network Namespace：网络隔离
   - 独立的网卡、IP、路由表、iptables规则

3. Mount Namespace：文件系统隔离
   - 独立的文件系统挂载点

4. UTS Namespace：主机名隔离
   - 独立的hostname和domainname

5. IPC Namespace：进程间通信隔离
   - 独立的消息队列、信号量、共享内存

6. User Namespace：用户隔离
   - 容器内root ≠ 宿主机root
```

**PID Namespace详解**：

```c
// 创建PID Namespace
int child_pid = clone(child_func, stack, 
                      CLONE_NEWPID | SIGCHLD, NULL);

示例：
宿主机视角：
PID 1234 ← 容器进程

容器内视角：
PID 1 ← 容器内的第一个进程（通常是init）
PID 2 ← 应用进程

为什么容器内进程看不到宿主机进程？
- PID Namespace隔离了进程树
- 容器内只能看到自己Namespace的进程
```

**Network Namespace详解**：

```bash
# 创建Network Namespace
ip netns add container1

# 查看网络设备（容器内）
ip netns exec container1 ip link
# 输出：只有lo（本地回环）

# 宿主机
ip link
# 输出：eth0, docker0, veth123...

容器网络模式：
1. Bridge模式（默认）：
   容器 ← veth pair → docker0 bridge → eth0 → 外网
   
2. Host模式：
   容器直接使用宿主机网络（无隔离）
   
3. None模式：
   容器无网络
```

**为什么容器启动快？**

```
虚拟机启动：
1. 启动Hypervisor
2. 加载Guest OS内核
3. 初始化Guest OS（init进程、系统服务）
4. 启动应用
总时间：1-2分钟

容器启动：
1. 创建Namespace（微秒级）
2. 配置Cgroup（微秒级）
3. 挂载文件系统（毫秒级）
4. 启动应用进程
总时间：秒级

关键：容器共享宿主机内核，无需启动OS
```

#### Cgroup（控制组）：资源限制

**Cgroup控制的资源**：

```
1. CPU：限制CPU使用率
2. Memory：限制内存使用量
3. Disk I/O：限制磁盘读写速度
4. Network：限制网络带宽（需要tc工具）
```

**CPU限制**：

```bash
# 限制容器使用1个CPU核心
docker run --cpus=1 nginx

# 底层实现（Cgroup）
echo 100000 > /sys/fs/cgroup/cpu/docker/<container_id>/cpu.cfs_quota_us
echo 100000 > /sys/fs/cgroup/cpu/docker/<container_id>/cpu.cfs_period_us
# quota/period = 100000/100000 = 1个CPU

# 限制0.5个CPU
echo 50000 > cpu.cfs_quota_us  # quota/period = 50000/100000 = 0.5
```

**内存限制**：

```bash
# 限制容器使用512MB内存
docker run -m 512m nginx

# 底层实现（Cgroup）
echo 536870912 > /sys/fs/cgroup/memory/docker/<container_id>/memory.limit_in_bytes
# 512MB = 536870912字节

# 超过限制会怎样？
# OOM Killer杀死容器进程
```

**为什么需要资源限制？**

```
场景：单台宿主机运行10个容器
- 无限制：某个容器占用90% CPU，其他容器饿死
- 有限制：每个容器最多10% CPU，公平调度

Cgroup的作用：
- 防止"吵闹邻居"问题（Noisy Neighbor）
- 保证资源公平性
- 提高资源利用率（超售）
```

#### Union FS（联合文件系统）：镜像分层

**为什么需要分层？**

```
传统方式（单层镜像）：
Ubuntu镜像：2GB
每个应用镜像：2GB（Ubuntu）+ 100MB（应用）
10个应用 = 20GB + 1GB = 21GB

分层方式（Union FS）：
Ubuntu镜像：2GB（共享）
应用层1：100MB
应用层2：100MB
...
10个应用 = 2GB + 1GB = 3GB

节省：21GB → 3GB（节省86%）
```

**Docker镜像分层结构**：

```
Dockerfile:
FROM ubuntu:20.04          ← Layer 1（基础层，只读）
RUN apt-get update         ← Layer 2（只读）
RUN apt-get install nginx  ← Layer 3（只读）
COPY app.py /app/          ← Layer 4（只读）
CMD ["python", "app.py"]   ← 元数据

容器启动后：
┌─────────────────────┐
│  容器层（读写）      │ ← 可写（容器内的修改）
├─────────────────────┤
│  Layer 4: app.py    │ ← 只读
├─────────────────────┤
│  Layer 3: nginx     │ ← 只读
├─────────────────────┤
│  Layer 2: update    │ ← 只读
├─────────────────────┤
│  Layer 1: ubuntu    │ ← 只读
└─────────────────────┘
```

**写时复制（Copy-on-Write）**：

```
读取文件：
1. 从上往下查找文件
2. 找到后直接读取（不复制）

修改文件：
1. 从上往下查找文件
2. 复制到容器层（可写层）
3. 修改容器层的副本
4. 原文件不变（只读层）

示例：
容器内执行：echo "hello" >> /etc/nginx/nginx.conf

步骤：
1. 查找nginx.conf（在Layer 3）
2. 复制到容器层
3. 修改容器层的nginx.conf
4. Layer 3的原文件不变

为什么这样设计？
- 保护只读层（可以安全共享）
- 隔离容器修改（互不影响）
- 支持容器快速创建（不需要复制整个镜像）
```

**OverlayFS工作原理**：

```
OverlayFS结构：
lowerdir（只读层）：镜像层1、层2、层3...
upperdir（可写层）：容器层
merged（合并视图）：容器内看到的文件系统

示例：
lowerdir=/layer1:/layer2:/layer3
upperdir=/container_rw
workdir=/work
merged=/merged

容器内看到的路径：/merged
实际存储：
- 只读文件：lowerdir
- 新建/修改文件：upperdir
```

### 1.3 Docker网络

**Bridge模式详解**：

```
默认网络架构：
容器A（172.17.0.2）
   |
   veth0 ← veth pair → veth123
                        |
                     docker0 bridge（172.17.0.1）
                        |
                     veth456 ← veth pair → veth1
   |
容器B（172.17.0.3）

数据流：
容器A → 容器B：
1. 容器A发送到172.17.0.3
2. veth0 → veth123 → docker0 bridge
3. docker0查找MAC地址表，转发到veth456
4. veth456 → veth1 → 容器B

容器A → 外网：
1. 容器A发送到8.8.8.8
2. docker0 → iptables NAT（SNAT，源地址转换）
3. 宿主机eth0 → 外网
4. 外网响应 → eth0 → iptables（DNAT）→ docker0 → 容器A
```

**端口映射原理**：

```bash
# 映射容器80端口到宿主机8080
docker run -p 8080:80 nginx

# 底层实现（iptables规则）
iptables -t nat -A DOCKER -p tcp --dport 8080 \
         -j DNAT --to-destination 172.17.0.2:80

数据流：
外网 → 宿主机8080 → iptables DNAT → 172.17.0.2:80 → 容器
```

```dockerfile
# ❌ 不推荐（镜像大，层多）

# 镜像大小：800MB → 10MB
```

## 2. Kubernetes核心原理

### 2.1 K8s架构

**Master节点（控制平面）**：

```
1. API Server：
   - 集群的唯一入口
   - RESTful API
   - 认证、授权、准入控制

2. etcd：
   - 分布式KV存储
   - 存储集群状态（Pod、Service等）
   - 使用Raft协议保证一致性

3. Scheduler：
   - 调度Pod到合适的Node
   - 考虑资源、亲和性、污点等

4. Controller Manager：
   - Deployment Controller
   - ReplicaSet Controller
   - Node Controller
   - ...
```

**Worker节点（数据平面）**：

```
1. Kubelet：
   - Node上的Agent
   - 管理Pod生命周期
   - 上报Node状态

2. Kube-proxy：
   - 实现Service网络
   - iptables/IPVS规则

3. Container Runtime：
   - Docker/containerd/CRI-O
   - 实际运行容器
```

### 2.2 Pod调度原理⭐⭐⭐

**调度流程**：

```
步骤1：过滤（Filtering）
- 筛选出满足条件的Node
- 条件：资源充足、污点容忍、节点选择器等

示例：
请求：CPU=2核，内存=4GB
Node1: CPU=4核（剩余3核），内存=8GB（剩余6GB）✅
Node2: CPU=4核（剩余1核），内存=8GB（剩余6GB）❌ CPU不足
Node3: CPU=8核（剩余5核），内存=16GB（剩余10GB）✅

候选节点：Node1、Node3

步骤2：打分（Scoring）
- 对候选Node评分
- 选择得分最高的Node

评分维度：
1. 资源剩余量（LeastRequestedPriority）
   Node1: (3-2)/4 + (6-4)/8 = 0.25 + 0.25 = 0.5
   Node3: (5-2)/8 + (10-4)/16 = 0.375 + 0.375 = 0.75 ✅

2. 负载均衡（BalancedResourceAllocation）
   - 尽量让CPU和内存使用率接近

3. 亲和性（Affinity）
   - 尽量调度到亲和的Node

步骤3：绑定（Binding）
- 创建Binding对象
- Kubelet拉起Pod
```

**为什么需要调度器？**

```
场景：100个Node，1000个Pod
- 无调度器：随机分配，可能Node1满载，Node2空闲
- 有调度器：根据资源、亲和性等智能分配，负载均衡

K8s调度器的优势：
- 资源利用率最大化
- 满足业务需求（亲和性、反亲和性）
- 支持自定义调度策略
```

### 2.3 Service网络原理⭐⭐⭐

**为什么需要Service？**

```
问题：
- Pod IP不固定（重启后变化）
- 多个Pod副本，如何负载均衡？

Service的作用：
- 提供稳定的虚拟IP（ClusterIP）
- 自动负载均衡（iptables/IPVS）
- 服务发现（DNS）
```

**ClusterIP实现原理**：

```
示例：
Service ClusterIP: 10.96.0.10:80
后端Pod: 
- 172.16.1.5:8080
- 172.16.2.6:8080
- 172.16.3.7:8080

iptables规则（简化）：
1. 访问10.96.0.10:80，跳转到负载均衡链
   -A KUBE-SERVICES -d 10.96.0.10/32 -p tcp --dport 80 -j KUBE-SVC-ABC

2. 负载均衡链（随机选择一个Pod）
   -A KUBE-SVC-ABC -m statistic --mode random --probability 0.33 -j KUBE-SEP-1
   -A KUBE-SVC-ABC -m statistic --mode random --probability 0.50 -j KUBE-SEP-2
   -A KUBE-SVC-ABC -j KUBE-SEP-3

3. DNAT到Pod IP
   -A KUBE-SEP-1 -p tcp -j DNAT --to-destination 172.16.1.5:8080
   -A KUBE-SEP-2 -p tcp -j DNAT --to-destination 172.16.2.6:8080
   -A KUBE-SEP-3 -p tcp -j DNAT --to-destination 172.16.3.7:8080

数据流：
客户端 → 10.96.0.10:80 → iptables DNAT → 172.16.1.5:8080（Pod）
```

**IPVS vs iptables**：

| 特性 | iptables | IPVS |
|------|----------|------|
| **性能** | O(n)（规则数） | O(1)（哈希表） |
| **负载均衡算法** | 随机 | 轮询、最小连接、源地址哈希等 |
| **规则数量** | < 1000 Service | > 10000 Service |
| **成熟度** | 高（默认） | 中（K8s 1.11+） |

```bash
# 启用IPVS模式
kubectl edit configmap kube-proxy -n kube-system
# 修改mode: ipvs
```

### 2.4 Pod网络（CNI）

**Flannel网络原理**：

```
VXLAN模式：
Node1（10.0.1.0/24）        Node2（10.0.2.0/24）
  Pod1（10.0.1.5）            Pod2（10.0.2.6）
    |                           |
  cni0 bridge                 cni0 bridge
    |                           |
  flannel.1（VXLAN）          flannel.1（VXLAN）
    |                           |
  eth0（192.168.1.10）        eth0（192.168.1.20）
    └────────────┬──────────────┘
                 │
             物理网络

数据流（Pod1 → Pod2）：
1. Pod1发送到10.0.2.6
2. 路由表：10.0.2.0/24 via flannel.1
3. flannel.1封装VXLAN（外层IP：192.168.1.10 → 192.168.1.20）
4. 通过物理网络传输
5. Node2的flannel.1解封装
6. 路由到Pod2
```

## 3. Service Mesh

### 3.1 什么是Service Mesh？

**传统微服务 vs Service Mesh**：

```
传统微服务：
应用A → [负载均衡/熔断/重试/监控] → 应用B
         ↑ 每个应用都要实现这些功能（代码耦合）

Service Mesh：
应用A → Sidecar Proxy → Sidecar Proxy → 应用B
         ↑ 负载均衡/熔断/重试/监控
         
优势：
- 应用代码无侵入
- 统一的流量管理
- 多语言支持（Go/Java/Python...）
```

### 4.2 容器安全⭐⭐

**容器安全威胁模型**：

```
1. 镜像安全：
   - 恶意镜像（后门、挖矿程序）
   - 已知漏洞（CVE）
   - 敏感信息泄露（密钥、密码）

2. 运行时安全：
   - 容器逃逸（突破隔离）
   - 特权容器滥用
   - 资源耗尽攻击

3. 网络安全：
   - 容器间未授权访问
   - 敏感数据传输
```

**镜像安全最佳实践**：

```bash
# 1. 使用可信基础镜像
FROM alpine:3.18  # 官方镜像，最小化

# 2. 镜像漏洞扫描
trivy image nginx:latest
# 输出：CVE-2023-xxxx: HIGH (libssl)

# 3. 多阶段构建，减少攻击面
FROM golang:1.20 AS builder
RUN go build -o app
FROM scratch  # 空镜像，无shell、无包管理器
COPY --from=builder /app /app

# 4. 使用非root用户
FROM node:18-alpine
RUN adduser -D appuser
USER appuser  # 以非root运行

# 5. 只读文件系统
docker run --read-only nginx
```

**容器逃逸防护**：

```
常见逃逸方式：
1. 特权容器：--privileged 可访问宿主机所有设备
2. 挂载敏感目录：-v /:/host 可读写宿主机文件系统
3. CAP_SYS_ADMIN：允许mount、chroot等危险操作
4. 内核漏洞：利用宿主机内核漏洞逃逸

防护措施：
1. 避免特权容器（生产环境禁止）
2. 使用用户命名空间（User Namespace）
3. 删除不必要的capabilities
4. 启用Seccomp和AppArmor
5. 及时更新内核和容器运行时
```

```yaml
# Kubernetes安全上下文
apiVersion: v1
kind: Pod
spec:
  securityContext:
    runAsNonRoot: true
```

### 4.3 Kubernetes存储⭐⭐

**存储架构**：

```
PersistentVolume（PV）：集群级存储资源
    ↑ 绑定
PersistentVolumeClaim（PVC）：命名空间级存储请求
    ↑ 挂载
Pod：使用存储

StorageClass：动态创建PV的模板
```

**静态供给 vs 动态供给**：

```yaml
# 静态供给：管理员预先创建PV
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-static
spec:
  capacity:
    storage: 10Gi
  accessModes:
    - ReadWriteOnce
  hostPath:
    path: /data/pv-static

---
# 动态供给：StorageClass自动创建PV
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: kubernetes.io/aws-ebs
parameters:
  type: gp3
  iopsPerGB: "50"
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer

---
# PVC请求动态创建
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
    - ReadWriteOnce
  storageClassName: fast-ssd  # 指定StorageClass
  resources:
    requests:
      storage: 20Gi
```

**CSI（Container Storage Interface）**：

```
为什么需要CSI？
- 存储插件与K8s核心代码解耦
- 支持任意存储后端（AWS EBS、Ceph、NFS...）
- 插件独立更新，无需升级K8s

CSI组件：
1. CSI Controller：创建/删除卷（运行在某个Node）
2. CSI Node：挂载/卸载卷（运行在每个Node）
3. CSI Driver：实现具体存储操作

工作流程：
1. 用户创建PVC
2. Controller调用CSI Controller创建卷
3. Pod调度到Node
4. Kubelet调用CSI Node挂载卷
5. Pod使用存储
```

**存储访问模式**：

```
ReadWriteOnce (RWO)：单Node读写（块存储）
ReadOnlyMany (ROX)：多Node只读（适合配置文件）
ReadWriteMany (RWX)：多Node读写（NFS、CephFS）
ReadWriteOncePod (RWOP)：单Pod读写（K8s 1.22+）
```

### 4.4 Kubernetes资源管理⭐⭐

**Request和Limit**：

```yaml
apiVersion: v1
kind: Pod
spec:
  containers:
  - name: app
    resources:
      requests:  # 调度依据，保证最低资源
        memory: "256Mi"
        cpu: "250m"     # 0.25核
      limits:    # 资源上限，超过会被限制/OOM
        memory: "512Mi"
        cpu: "500m"     # 0.5核
```

```
Request和Limit的关系：
┌─────────────────────────────────────────┐
│                Limit                     │
│   ┌─────────────────────────────────┐   │
│   │         可突发使用               │   │
│   ├─────────────────────────────────┤   │
│   │         Request（保证）          │   │
│   └─────────────────────────────────┘   │
└─────────────────────────────────────────┘

- Request: 调度器保证节点有这么多资源
- Limit: 容器最多使用这么多资源
- 超过Limit：CPU被限流，内存被OOM Kill
```

**QoS等级**：

```
1. Guaranteed（最高优先级）：
   - 所有容器都设置了request和limit
   - request == limit
   - OOM时最后被杀

2. Burstable（中等优先级）：
   - 至少一个容器设置了request或limit
   - request != limit
   - OOM时中等概率被杀

3. BestEffort（最低优先级）：
   - 没有任何容器设置request或limit
   - OOM时最先被杀

建议：生产环境使用Guaranteed，避免被OOM Kill
```

**HPA（Horizontal Pod Autoscaler）**：

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: nginx-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: nginx
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70  # CPU > 70% 扩容
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80  # 内存 > 80% 扩容
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # 缩容冷却期
```

**VPA（Vertical Pod Autoscaler）**：

```
HPA vs VPA：
- HPA：水平扩展（增加Pod数量）
- VPA：垂直扩展（增加Pod资源）

VPA工作模式：
1. Off：只推荐，不自动调整
2. Initial：只在Pod创建时设置
3. Auto：自动调整（会重启Pod）

适用场景：
- HPA：无状态服务，快速扩缩
- VPA：有状态服务，资源难以预估
- 两者不建议同时用于CPU/内存
```

**ResourceQuota和LimitRange**：

```yaml
# ResourceQuota：限制命名空间总资源
apiVersion: v1
kind: ResourceQuota
metadata:
  name: team-a-quota
  namespace: team-a
spec:
  hard:
    requests.cpu: "10"      # 总CPU请求10核
    requests.memory: 20Gi   # 总内存请求20GB
    limits.cpu: "20"        # 总CPU限制20核
    limits.memory: 40Gi     # 总内存限制40GB
    pods: "50"              # 最多50个Pod
    services: "20"          # 最多20个Service

---
# LimitRange：限制单个资源的默认值和范围
apiVersion: v1
kind: LimitRange
metadata:
  name: default-limits
spec:
  limits:
  - type: Container
    default:          # 默认limit
      memory: "512Mi"
      cpu: "500m"
    defaultRequest:   # 默认request
      memory: "256Mi"
      cpu: "250m"
    min:              # 最小值
      memory: "128Mi"
      cpu: "100m"
    max:              # 最大值
      memory: "2Gi"
      cpu: "2"
```

```
                    ┌──────────────┐
```

## 5. GitOps与持续交付

### 5.1 GitOps原理⭐⭐⭐

**什么是GitOps？**

```
GitOps是一种以Git仓库为核心的运维方法论，
核心思想：Git是基础设施和应用配置的唯一事实来源（Single Source of Truth）。

传统运维 vs GitOps：

传统运维：
开发者 → 手动kubectl apply → K8s集群
        → SSH到服务器修改配置
        → 通过CI直接推送到集群
问题：
- 谁改了什么？不知道
- 当前集群状态和配置一致吗？不确定
- 如何回滚到上一个版本？手动操作

GitOps：
开发者 → Git Push → Git仓库 → GitOps Operator → K8s集群
                      ↑                  ↓
                      │            持续对账（Reconciliation）
                      │                  ↓
                      └──── 漂移检测 ←── 集群实际状态

优势：
- 所有变更有Git记录（谁、什么时候、改了什么）
- 回滚 = Git Revert
- 集群状态始终与Git一致
```

**声明式配置**：

```
命令式 vs 声明式：

命令式（Imperative）：
  kubectl run nginx --image=nginx:1.20
  kubectl scale deployment nginx --replicas=3
  kubectl set image deployment/nginx nginx=nginx:1.21
  问题：执行顺序依赖、状态难追踪、不幂等

声明式（Declarative）：
  # nginx-deployment.yaml
  apiVersion: apps/v1
  kind: Deployment
  metadata:
    name: nginx
  spec:
    replicas: 3
    template:
      spec:
        containers:
        - name: nginx
          image: nginx:1.21
  
  kubectl apply -f nginx-deployment.yaml
  优势：幂等、可版本化、可审计

GitOps要求所有配置必须是声明式的：
- K8s资源 → YAML/Kustomize/Helm
- 基础设施 → Terraform
- 策略 → OPA/Kyverno
```

**Git作为单一事实来源**：

```
仓库结构设计（推荐的Monorepo方式）：

infrastructure-repo/
├── base/                    # 基础配置（跨环境共享）
│   ├── deployment.yaml
│   ├── service.yaml
│   └── kustomization.yaml
├── overlays/                # 环境差异化配置
│   ├── dev/
│   │   ├── kustomization.yaml
│   │   └── patch-replicas.yaml
│   ├── staging/
│   │   ├── kustomization.yaml
│   │   └── patch-replicas.yaml
│   └── prod/
│       ├── kustomization.yaml
│       ├── patch-replicas.yaml
│       └── patch-resources.yaml
└── apps/                    # 多应用配置
    ├── order-service/
    ├── user-service/
    └── payment-service/

分仓策略：
- 应用代码仓库（App Repo）：业务代码 + Dockerfile
- 配置仓库（Config Repo）：K8s YAML + Helm values
- 分离好处：应用变更和配置变更解耦，审批流不同

Push模式 vs Pull模式：
Push模式：CI流水线推送到集群（传统CI/CD）
  → 需要集群凭证暴露给CI
  → 安全风险高

Pull模式：GitOps Operator从Git拉取配置（ArgoCD/Flux）
  → Operator在集群内部运行
  → 无需暴露集群凭证
  → 安全性更高（推荐）
```

### 5.2 ArgoCD⭐⭐⭐

**ArgoCD核心架构**：

```
┌───────────────────────────────────────────────────┐
│                   ArgoCD架构                       │
│                                                   │
│  ┌─────────────┐    ┌──────────────┐             │
│  │  API Server  │←──│   Web UI     │             │
│  └──────┬──────┘    └──────────────┘             │
│         │                                         │
│  ┌──────▼──────┐    ┌──────────────┐             │
│  │ Application │←──│   Repo       │             │
│  │ Controller  │    │   Server     │ ← Git拉取   │
│  └──────┬──────┘    └──────────────┘             │
│         │                                         │
│  ┌──────▼──────┐                                 │
│  │  K8s API    │ ← 同步资源到集群                │
│  └─────────────┘                                 │
└───────────────────────────────────────────────────┘

核心流程：
1. Repo Server 从Git仓库拉取配置
2. Application Controller 比较期望状态与实际状态
3. 检测到漂移（Drift）→ 触发同步
4. 通过K8s API将资源同步到集群
```

**Application CRD详解**：

```yaml
# ArgoCD Application：定义一个需要管理的应用
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: order-service
  namespace: argocd
  # 可选：级联删除策略
  finalizers:
  - resources-finalizer.argocd.argoproj.io
spec:
  # 所属项目（权限隔离）
  project: production
  
  # 配置来源
  source:
    repoURL: https://github.com/myorg/infra-config.git
    targetRevision: main          # 分支/Tag/Commit
    path: apps/order-service/overlays/prod  # 路径
    
    # 如果使用Helm Chart
    # helm:
    #   valueFiles:
    #   - values-prod.yaml
    #   parameters:
    #   - name: image.tag
    #     value: "v2.1.0"
  
  # 目标集群和命名空间
  destination:
    server: https://kubernetes.default.svc  # 集群地址
    namespace: production
  
  # 同步策略
  syncPolicy:
    automated:
      prune: true        # 自动删除Git中不存在的资源
      selfHeal: true     # 集群状态漂移时自动修复
      allowEmpty: false  # 不允许同步空目录
    syncOptions:
    - CreateNamespace=true     # 自动创建命名空间
    - PrunePropagationPolicy=foreground
    - PruneLast=true           # 删除操作最后执行
    retry:
      limit: 5                 # 重试次数
      backoff:
        duration: 5s
        factor: 2
        maxDuration: 3m
  
  # 忽略某些字段的差异（避免误判漂移）
  ignoreDifferences:
  - group: apps
    kind: Deployment
    jsonPointers:
    - /spec/replicas          # 忽略HPA导致的副本数变化
```

**同步策略深入**：

```
ArgoCD同步模式：

1. 手动同步（Manual Sync）：
   - 适合生产环境，变更需要人工确认
   - 流程：Git Push → ArgoCD检测到差异 → 通知 → 人工点击同步

2. 自动同步（Auto Sync）：
   - 适合开发/测试环境
   - 流程：Git Push → ArgoCD自动同步到集群
   - selfHeal：集群被手动修改后自动恢复为Git状态
   - prune：Git中删除的资源自动从集群删除

3. 同步窗口（Sync Windows）：
   - 限制同步的时间窗口
   - 如：只允许工作日10:00-18:00同步

同步状态：
- Synced：集群状态 = Git状态
- OutOfSync：集群状态 ≠ Git状态
- Unknown：无法确定状态

健康状态：
- Healthy：所有资源运行正常
- Degraded：部分资源异常
- Progressing：资源正在更新中
- Missing：资源不存在
```

## 6. K8s 排障实战

### 6.1 Pod 常见异常状态排查 SOP

#### CrashLoopBackOff

**含义**：容器反复启动又崩溃，Kubelet 按指数退避重启（10s→20s→40s→...→5min）。

**排查流程**：

```
步骤1：查看 Pod 事件，定位最近的异常事件
  kubectl describe pod <pod-name> -n <namespace>
  重点关注 Events 部分：
  - Back-off restarting failed container → 确认是 CrashLoopBackOff
  - 查看 Last State 中的 Exit Code 和 Reason

步骤2：查看容器日志，定位崩溃原因
  # 查看当前容器日志
  kubectl logs <pod-name> -n <namespace> -c <container-name>
  # 查看上一次崩溃的日志（关键！当前容器可能还没输出就崩了）
  kubectl logs <pod-name> -n <namespace> -c <container-name> --previous

步骤3：检查容器退出码
  kubectl get pod <pod-name> -n <namespace> -o jsonpath='{.status.containerStatuses[0].lastState}'
  常见退出码：
  - Exit Code 0：正常退出（但不应该退出，检查 CMD/ENTRYPOINT）
  - Exit Code 1：应用异常（代码错误、未捕获异常）
  - Exit Code 137：OOMKilled（被内核杀死）或 kill -9
  - Exit Code 139：段错误（SIGSEGV）
  - Exit Code 143：SIGTERM 正常终止

步骤4：进入容器内调试（如果容器能短暂运行）
  kubectl exec -it <pod-name> -n <namespace> -- /bin/sh
  # 容器秒崩无法 exec 时，用临时调试容器
  kubectl debug -it <pod-name> -n <namespace> --image=busybox --target=<container-name>
```

**常见原因与解决方案**：

```
原因1：OOM（内存不足）
  现象：Exit Code 137，describe 中有 OOMKilled
  解决：增大 resources.limits.memory，或排查内存泄漏

原因2：配置错误
  现象：Exit Code 1，日志中有配置解析失败
  解决：检查 ConfigMap/Secret 挂载、环境变量注入是否正确
  排查：kubectl get configmap <name> -o yaml
        kubectl get secret <name> -o jsonpath='{.data}' | base64 -d

原因3：依赖服务未就绪
  现象：日志中有连接超时（Connection refused / timeout）
  解决：
  - 添加 initContainer 等待依赖就绪
  - 应用内实现重试机制和优雅降级
  - 检查 Service/Endpoint 是否正常

原因4：启动命令错误
  现象：Exit Code 127（command not found）或 126（权限不足）
  解决：检查 Dockerfile CMD/ENTRYPOINT，检查镜像中是否包含所需二进制
  排查：kubectl get pod <pod-name> -o jsonpath='{.spec.containers[0].command}'
```

---

#### OOMKilled

**含义**：容器使用的内存超过了 limits 限制，被内核 OOM Killer 杀死。

**识别方式**：

```
步骤1：确认 OOMKilled 状态
  kubectl get pod <pod-name> -n <namespace> -o jsonpath='{.status.containerStatuses[0].lastState.terminated}'
  # 输出中 reason: OOMKilled, exitCode: 137

步骤2：查看当前内存使用（需要 metrics-server）
  kubectl top pod <pod-name> -n <namespace>
  kubectl top pod -n <namespace> --sort-by=memory

步骤3：查看 Pod 资源配置
  kubectl get pod <pod-name> -n <namespace> -o jsonpath='{.spec.containers[0].resources}'
  # 对比实际使用量与 limits 设置

步骤4：查看节点级内存压力
  kubectl describe node <node-name> | grep -A5 "Conditions"
  kubectl top node
```

**优化方案**：

```
方案1：调整资源限制（短期）
  适当增大 memory limits，建议 limits = 实际峰值 × 1.2~1.5
  spec:
    containers:
    - resources:
        requests:
          memory: "512Mi"
        limits:
          memory: "1Gi"    # 留有余量

方案2：排查内存泄漏（长期）
  - Java 应用：检查 -Xmx 是否超过 limits，配合 JFR/MAT 分析堆内存
  - Go 应用：pprof 抓取 heap profile
  - Node.js 应用：检查 --max-old-space-size 设置

方案3：使用 VPA 自动推荐资源配置
  kubectl get vpa <name> -o jsonpath='{.status.recommendation}'
```

---

#### ImagePullBackOff

**含义**：Kubelet 无法拉取容器镜像，进入退避重试。

**排查流程**：

```
步骤1：查看具体错误信息
  kubectl describe pod <pod-name> -n <namespace>
  # Events 中常见错误：
  # - ErrImagePull / ImagePullBackOff
  # - repository does not exist → 镜像名/Tag 错误
  # - unauthorized: authentication required → 认证失败
  # - context deadline exceeded → 网络超时

步骤2：验证镜像是否存在
  # 在能访问 Registry 的机器上执行
  docker pull <image:tag>
  # 或使用 crane/skopeo 检查
  crane manifest <image:tag>

步骤3：检查镜像拉取凭证
  kubectl get pod <pod-name> -o jsonpath='{.spec.imagePullSecrets}'
  kubectl get secret <secret-name> -n <namespace> -o jsonpath='{.data.\.dockerconfigjson}' | base64 -d

步骤4：检查节点网络连通性
  # 在问题节点上测试
  kubectl debug node/<node-name> -it --image=busybox
  # 进入后测试 Registry 连通性
  nslookup <registry-host>
  wget -qO- https://<registry-host>/v2/
```

**常见原因**：

```
原因1：镜像名或 Tag 拼写错误 → 仔细检查 image 字段
原因2：私有仓库未配置 imagePullSecrets → 创建 docker-registry 类型 Secret 并关联到 Pod/ServiceAccount
原因3：镜像仓库网络不可达 → 检查防火墙、代理配置、DNS 解析
原因4：镜像仓库限流（如 Docker Hub rate limit）→ 使用镜像代理或私有仓库
```

---

#### Pending

**含义**：Pod 已创建但无法被调度到任何节点。

**排查流程**：

```
步骤1：查看 Pod 事件，确定 Pending 原因
  kubectl describe pod <pod-name> -n <namespace>
  # 常见事件消息：
  # - Insufficient cpu / Insufficient memory → 资源不足
  # - didn't match Pod's node affinity/selector → 亲和性/选择器无法满足
  # - persistentvolumeclaim is not bound → PVC 未绑定
  # - had taint that the pod didn't tolerate → 污点不容忍

步骤2：检查集群资源余量
  kubectl describe nodes | grep -A5 "Allocated resources"
  kubectl top nodes
  # 对比 Pod 的 requests 与各节点剩余资源

步骤3：检查调度约束
  # 查看 nodeSelector
  kubectl get pod <pod-name> -o jsonpath='{.spec.nodeSelector}'
  # 查看 affinity
  kubectl get pod <pod-name> -o jsonpath='{.spec.affinity}'
  # 查看 tolerations
  kubectl get pod <pod-name> -o jsonpath='{.spec.tolerations}'
  # 查看节点 taints
  kubectl describe nodes | grep Taints

步骤4：检查 PVC 绑定状态（存储相关 Pending）
  kubectl get pvc -n <namespace>
  kubectl describe pvc <pvc-name> -n <namespace>
  # 确认 StorageClass 是否存在、Provisioner 是否正常
  kubectl get storageclass
```

### 6.2 K8s 网络排障

#### Service 无法访问

**排查流程（自上而下逐层排查）**：

```
步骤1：验证 DNS 解析（CoreDNS 层）
  # 在集群内 Pod 中测试
  kubectl run debug-dns --rm -it --image=busybox --restart=Never -- nslookup <service-name>.<namespace>.svc.cluster.local
  
  # 如果 DNS 解析失败：
  kubectl get pods -n kube-system -l k8s-app=kube-dns
  kubectl logs -n kube-system -l k8s-app=kube-dns
  # 检查 CoreDNS ConfigMap
  kubectl get configmap coredns -n kube-system -o yaml

步骤2：检查 Service 和 Endpoint
  kubectl get svc <service-name> -n <namespace>
  kubectl get endpoints <service-name> -n <namespace>
  # Endpoint 列表为空 → Service selector 与 Pod label 不匹配
  # 验证标签匹配
  kubectl get pods -n <namespace> --show-labels
  kubectl get svc <service-name> -n <namespace> -o jsonpath='{.spec.selector}'

步骤3：检查 kube-proxy 规则
  # iptables 模式
  kubectl exec -it <kube-proxy-pod> -n kube-system -- iptables-save | grep <service-clusterip>
  # IPVS 模式
  kubectl exec -it <kube-proxy-pod> -n kube-system -- ipvsadm -Ln | grep <service-clusterip>
  # 检查 kube-proxy 日志
  kubectl logs -n kube-system -l k8s-app=kube-proxy

步骤4：检查 CNI 网络层
  # 验证 Pod IP 是否可达
  kubectl exec -it <debug-pod> -- ping <target-pod-ip>
  kubectl exec -it <debug-pod> -- wget -qO- http://<target-pod-ip>:<port>
  # 如果 Pod IP 不可达，问题在 CNI 层，检查 CNI 插件状态
  kubectl get pods -n kube-system | grep -E "calico|flannel|cilium|weave"
```

#### Pod 间通信异常

**跨节点通信排查**：

```
步骤1：确认同节点 Pod 通信是否正常
  # 找两个在同一节点的 Pod 互相访问
  kubectl get pods -o wide -n <namespace>
  # 同节点通信走 Linux Bridge，如果也不通说明 CNI 基础配置有问题

步骤2：排查跨节点路由
  # 检查节点路由表
  kubectl debug node/<node-name> -it --image=busybox -- ip route
  # 检查是否有到目标 Pod 子网的路由
  # Flannel：应有 10.244.x.0/24 via flannel.1 的路由
  # Calico：应有 BGP 通告的路由

步骤3：排查 VXLAN/Overlay 隧道
  # 检查 VXLAN 设备状态
  kubectl debug node/<node-name> -it --image=nicolaka/netshoot -- ip -d link show flannel.1
  # 检查 VXLAN FDB 表
  kubectl debug node/<node-name> -it --image=nicolaka/netshoot -- bridge fdb show dev flannel.1

步骤4：排查 NetworkPolicy 拦截
  kubectl get networkpolicy -n <namespace>
  kubectl describe networkpolicy <policy-name> -n <namespace>
  # 临时删除 NetworkPolicy 验证是否是策略导致
  # 注意：仅在测试环境操作
```

### 6.3 K8s 性能问题排查

#### API Server 响应慢

**排查流程**：

```
步骤1：确认 API Server 延迟指标
  # 查看 API Server 请求延迟（需要 Prometheus）
  kubectl get --raw /metrics | grep apiserver_request_duration_seconds
  # 或直接测试 API 响应时间
  time kubectl get nodes
  time kubectl get pods -A

步骤2：排查 etcd 延迟
  # etcd 是 API Server 的后端存储，etcd 慢 = API Server 慢
  # 查看 etcd 延迟指标
  kubectl get --raw /metrics | grep etcd_request_duration_seconds
  # 检查 etcd 健康状态
  kubectl exec -it etcd-<master-node> -n kube-system -- etcdctl endpoint health --cluster
  kubectl exec -it etcd-<master-node> -n kube-system -- etcdctl endpoint status --cluster -w table
  # etcd 磁盘延迟过高（>10ms）通常是性能瓶颈
  # 解决：使用 SSD、独立部署 etcd、减少大 Key（如大 ConfigMap/Secret）

步骤3：检查审计日志开销
  # 审计策略过于详细会显著增加 API Server 负载
  kubectl get pods -n kube-system -l component=kube-apiserver -o yaml | grep audit
  # 优化：降低审计级别（Metadata → None），排除高频低价值请求

步骤4：检查请求限流（APF - API Priority and Fairness）
  kubectl get --raw /metrics | grep apiserver_flowcontrol
  kubectl get flowschema
  kubectl get prioritylevelconfiguration
  # 查看是否有请求被限流排队
  # 现象：大量 LIST/WATCH 请求导致排队，影响正常操作
```

---

#### 调度延迟

**排查流程**：

```
步骤1：观察调度耗时
  # 查看 Pod 从创建到调度完成的时间
  kubectl describe pod <pod-name> | grep -A2 "Events"
  # 关注 Scheduled 事件的时间差
  # 正常：毫秒到秒级；异常：分钟级

步骤2：检查 Scheduler 状态和队列
  kubectl get pods -n kube-system -l component=kube-scheduler
  kubectl logs -n kube-system -l component=kube-scheduler --tail=200
  # 查看调度失败原因
  # 常见：0/10 nodes are available: 10 Insufficient cpu

步骤3：排查 Predicates（过滤阶段）失败
  kubectl describe pod <pod-name> -n <namespace>
  # Events 中的 FailedScheduling 消息会列出每个节点被过滤掉的原因：
  # - Insufficient cpu/memory → 扩容节点或减少 requests
  # - node(s) didn't match Pod's node affinity → 检查 nodeSelector/affinity
  # - node(s) had taint → 添加 tolerations
  # - node(s) unschedulable → kubectl uncordon <node>

步骤4：资源碎片化排查
  # 节点总资源充足但单个节点剩余不够
  kubectl describe nodes | grep -A10 "Allocated resources"
  # 解决：使用 Descheduler 重新平衡、调整 Pod 资源请求粒度
```

#### 节点 NotReady

**排查流程**：

```
步骤1：查看节点状态和 Conditions
  kubectl get nodes
  kubectl describe node <node-name>
  # 关注 Conditions 部分：
  # - Ready=False → kubelet 停止上报或上报异常
  # - MemoryPressure=True → 节点内存不足
  # - DiskPressure=True → 节点磁盘不足
  # - PIDPressure=True → 节点 PID 耗尽
  # - NetworkUnavailable=True → 节点网络异常

步骤2：检查 kubelet 健康状态
  # SSH 到问题节点（或使用 kubectl debug node）
  systemctl status kubelet
  journalctl -u kubelet --since "30 minutes ago" --no-pager | tail -100
  # 常见错误：
  # - PLEG is not healthy → 容器运行时卡死
  # - certificate has expired → 证书过期
  # - failed to get cgroup → cgroup 驱动不匹配

步骤3：排查磁盘压力
  # 检查磁盘使用
  df -h
  # kubelet 默认阈值：
  # - imagefs.available < 15% → 开始驱逐 Pod
  # - nodefs.available < 10% → 节点变为 DiskPressure
  # 清理方案：
  docker system prune -a    # 清理未使用的镜像和容器
  crictl rmi --prune         # containerd 清理

步骤4：排查 PID 压力
  # 检查系统 PID 使用情况
  ls /proc | grep -c '^[0-9]'      # 当前进程数
  cat /proc/sys/kernel/pid_max     # PID 上限
  # 找出 PID 消耗大户
  ps aux --sort=-nlwp | head -20   # 按线程数排序
  # kubelet 默认 PID 阈值：节点级 pid.available < 某值时标记 PIDPressure
  # 解决：找出泄漏进程并修复，适当调大 pid_max

步骤5：排查容器运行时
  # containerd 状态
  systemctl status containerd
  crictl ps                         # 列出运行中容器
  crictl pods                       # 列出 Pod 沙箱
  # Docker 状态
  systemctl status docker
  docker ps
  # PLEG（Pod Lifecycle Event Generator）不健康通常是运行时卡住
  # 解决：重启 containerd/docker，严重时重启 kubelet
```

## 7. 面试题自查

### 使用建议

> 刷题顺序建议：
> 1. 初级高频速刷先过容器和 K8s 的基础对象。
> 2. 详细 Q&A 再看调度、网络、存储、安全、网格和发布治理。
> 3. 如果你平时只会用，不会说原理，这一节要重点刷。

### 初级高频速刷

**1. 镜像（Image）和容器（Container）的区别是什么？**

镜像是静态只读模板，容器是镜像启动后的运行实例。一个镜像可以起多个容器，面试问这个通常是想确认你不是把两者混成一个概念。

**2. Pod、Deployment、Service 分别是什么？**

Pod 是最小调度单元，Deployment 管理无状态应用副本和滚动更新，Service 给一组 Pod 提供稳定的访问入口。K8s 基础概念里，这三个对象几乎必问。

**3. `livenessProbe` 和 `readinessProbe` 的区别是什么？**

`livenessProbe` 失败一般表示容器该重启了；`readinessProbe` 失败表示容器还活着，但暂时不应该接收流量。很多线上故障就是把两者配置反了。

**4. `ClusterIP`、`NodePort`、`LoadBalancer` 三种 Service 有什么区别？**

`ClusterIP` 只在集群内部访问，`NodePort` 暴露到每个节点固定端口，`LoadBalancer` 通常通过云负载均衡对外提供服务。回答时最好顺带说明实际生产中大多还会叠加 Ingress/Gateway。

**5. `Deployment` 和 `StatefulSet` 分别适合什么场景？**

Deployment 适合无状态服务，副本可随意替换；StatefulSet 适合有稳定身份和持久化需求的服务，比如 MySQL、Kafka、ZooKeeper。区别核心不是“一个高级一个普通”，而是身份和存储语义不同。

### 中级高频进阶

**6. `CrashLoopBackOff`、`OOMKilled`、`Evicted`、`ImagePullBackOff` 分别意味着什么？**

`CrashLoopBackOff` 通常是应用启动后反复崩溃；`OOMKilled` 是内存超限被内核杀死；`Evicted` 是节点资源压力大被调度系统驱逐；`ImagePullBackOff` 是镜像拉取失败。它们都表现为 Pod 起不来，但根因完全不同。

**7. `requests` 和 `limits` 配得不合理，会带来哪些具体问题？**

`requests` 过低会让调度过于激进，节点实际运行时容易资源争抢；`limits` 过低会导致频繁 throttling 或 OOMKilled；两者都不设则可能影响 QoS 和资源隔离。K8s 很多稳定性问题，本质上是资源模型配置不准确。

**8. 为什么有些 Pod 看起来进程还活着，却一直不应该接流量？**

因为“活着”和“准备好服务用户”是两件事。比如应用刚启动完但缓存尚未预热、依赖尚未连通、数据尚未同步完成，此时 liveness 可能正常，但 readiness 应该失败，避免把坏流量直接导入实例。

**9. `Service`、`Ingress`、`Gateway` 三者各自负责什么？为什么不能混着理解？**

`Service` 主要解决服务发现和四层转发，`Ingress` 主要面向 HTTP 七层入口路由，`Gateway API` 则把入口流量治理做成更清晰、可扩展的资源模型。很多人会把它们都说成“对外暴露服务”，但真正工程上它们承担的是不同层次的职责。

**10. `PodDisruptionBudget` 为什么不是高可用银弹？**

因为它只约束自愿驱逐，比如节点排空、版本升级、集群维护时不要一下子赶走太多 Pod；如果是节点突然宕机、容器崩溃、应用自身异常，PDB 并不能兜底。它是高可用治理的一部分，不是完整答案。

### 深入问答

#### Q1: 容器和虚拟机的本质区别是什么？为什么容器启动更快？

**答案**：
- **隔离层级不同**：虚拟机在硬件层虚拟化，每个VM运行独立的Guest OS；容器在操作系统层虚拟化，共享宿主机内核
- **启动过程差异**：
  - 虚拟机：启动Hypervisor → 加载Guest OS内核 → 初始化系统服务 → 启动应用（1-2分钟）
  - 容器：创建Namespace（微秒级）→ 配置Cgroup（微秒级）→ 挂载文件系统（毫秒级）→ 启动应用进程（秒级）
- **关键点**：容器无需启动完整OS，共享宿主机内核是快速启动的核心原因

#### Q2: Docker的Namespace有哪些类型？各自隔离什么资源？

**答案**：
Linux提供6种Namespace实现容器隔离：
1. **PID Namespace**：进程隔离，容器内PID从1开始，看不到宿主机进程
2. **Network Namespace**：网络隔离，独立的网卡、IP、路由表、iptables规则
3. **Mount Namespace**：文件系统隔离，独立的挂载点视图
4. **UTS Namespace**：主机名隔离，独立的hostname和domainname
5. **IPC Namespace**：进程间通信隔离，独立的消息队列、信号量、共享内存
6. **User Namespace**：用户隔离，容器内root不等于宿主机root

#### Q3: Cgroup如何实现容器的资源限制？超过限制会怎样？

**答案**：
**资源限制机制**：
- **CPU**：通过`cpu.cfs_quota_us`和`cpu.cfs_period_us`控制时间片，quota/period即为CPU核数
- **内存**：通过`memory.limit_in_bytes`限制最大内存使用

**超限行为**：
- CPU超限：进程被限流（Throttle），不会被杀死，但执行变慢
- 内存超限：触发OOM Killer，容器进程被杀死

**实际意义**：防止"吵闹邻居"问题，保证多容器环境下的资源公平性

---

#### Q4: Docker镜像的分层存储和写时复制机制是如何工作的？

**答案**：
**分层存储**：
- Dockerfile每条指令创建一个只读层
- 多个容器可共享相同的只读层，只有容器层可写
- 大幅节省存储空间（例如10个基于Ubuntu的应用可共享Ubuntu基础层）

**写时复制（CoW）**：
- **读取**：从上往下查找文件，找到后直接读取，无需复制
- **写入**：将文件从只读层复制到可写的容器层，修改容器层副本
- **好处**：保护只读层可安全共享，支持容器快速创建（无需复制整个镜像）

#### Q5: Kubernetes的调度流程是怎样的？调度器如何选择最优节点？

**答案**：
调度分为三个阶段：

**1. 过滤（Filtering）**：
- 筛选出满足硬性条件的节点
- 条件包括：资源充足、污点容忍、节点选择器、亲和性要求等

**2. 打分（Scoring）**：
- 对候选节点评分，常用策略：
  - LeastRequestedPriority：资源剩余越多分数越高
  - BalancedResourceAllocation：CPU和内存使用率越接近分数越高
  - NodeAffinityPriority：满足亲和性偏好的节点分数更高

**3. 绑定（Binding）**：
- 选择得分最高的节点，创建Binding对象
- Kubelet拉起Pod

#### Q6: K8s Service的ClusterIP是如何实现负载均衡的？iptables和IPVS有什么区别？

**答案**：
**ClusterIP实现原理**：
- kube-proxy在每个节点上创建iptables/IPVS规则
- 访问ClusterIP时，通过DNAT规则转发到后端Pod
- iptables使用随机概率实现负载均衡

**iptables vs IPVS**：
| 特性 | iptables | IPVS |
|------|----------|------|
| 性能 | O(n)，规则越多越慢 | O(1)，哈希表查找 |
| 算法 | 仅随机 | 轮询、最小连接、源地址哈希等 |
| 规模 | <1000 Service | >10000 Service |

**生产建议**：大规模集群（>1000 Service）应启用IPVS模式

#### Q7: 如何理解K8s的Pod网络？CNI插件（如Flannel）是如何实现跨节点Pod通信的？

**答案**：
**Pod网络要求**：
- 每个Pod有独立IP，Pod间可直接通信（无需NAT）
- 同节点Pod通过Linux Bridge通信
- 跨节点Pod需要CNI插件实现

**Flannel VXLAN模式**：
1. 每个节点分配一个子网（如10.0.1.0/24）
2. flannel.1是VXLAN设备，封装/解封装数据包
3. 跨节点通信：Pod1 → cni0 bridge → flannel.1（VXLAN封装）→ 物理网络 → 目标节点flannel.1（解封装）→ cni0 → Pod2

**其他CNI插件**：Calico（BGP路由）、Cilium（eBPF）、Weave等

#### Q8: K8s Deployment的滚动更新是如何保证零停机的？

**答案**：
**滚动更新策略参数**：
- `maxUnavailable`：更新过程中最多有几个Pod不可用
- `maxSurge`：更新过程中最多额外创建几个Pod

**更新流程**（以replicas=10, maxUnavailable=2, maxSurge=2为例）：
1. 创建2个新版本Pod（v2）
2. 等待新Pod就绪
3. 删除2个旧版本Pod（v1）
4. 重复直到全部更新完成

**保证可用性**：
- 始终有`replicas - maxUnavailable`个Pod运行
- 配合ReadinessProbe确保新Pod真正就绪后才接收流量
- 发现问题可立即回滚（`kubectl rollout undo`）

#### Q9: 什么是Service Mesh？Istio的Sidecar模式有什么优势？

**答案**：
**Service Mesh定义**：专用的基础设施层，处理服务间通信，包括负载均衡、熔断、重试、监控等

**Sidecar模式优势**：
1. **应用无侵入**：流量管理逻辑在Sidecar中，应用代码无需修改
2. **语言无关**：Go/Java/Python应用都可使用相同的流量管理能力
3. **统一管控**：通过控制平面统一配置所有Sidecar
4. **可观测性**：Sidecar自动收集metrics、traces、logs

**Istio架构**：
- 数据平面：Envoy Sidecar拦截所有进出流量
- 控制平面：Pilot（流量管理）、Citadel（安全）、Galley（配置）

#### Q10: 如何实现灰度发布/金丝雀发布？流量切分有哪些方式？

**答案**：
**Istio流量切分方式**：

1. **按比例切分**：
```yaml
route:
- destination: { host: reviews, subset: v1 }
  weight: 90
- destination: { host: reviews, subset: v2 }
  weight: 10  # 10%流量到新版本
```

2. **按请求头切分**：
```yaml
match:
- headers:
    user: { exact: "vip" }
route:
- destination: { host: reviews, subset: v2 }  # VIP用户使用新版本
```

3. **按Cookie切分**：A/B测试常用

**渐进式发布流程**：1% → 5% → 10% → 25% → 50% → 100%，每阶段观察错误率和延迟

#### Q11: K8s的PV/PVC/StorageClass分别是什么？动态供给是如何工作的？

**答案**：
**存储抽象层次**：
- **PersistentVolume (PV)**：集群级存储资源，由管理员创建或动态供给
- **PersistentVolumeClaim (PVC)**：用户的存储请求，绑定到PV
- **StorageClass**：定义动态创建PV的模板和参数

**动态供给流程**：
1. 用户创建PVC，指定StorageClass
2. StorageClass的Provisioner自动创建符合要求的PV
3. PVC绑定到新创建的PV
4. Pod挂载PVC使用存储

**访问模式**：RWO（单节点读写）、ROX（多节点只读）、RWX（多节点读写）

#### Q12: 容器安全有哪些常见威胁？如何防止容器逃逸？

**答案**：
**安全威胁**：
1. 镜像漏洞（已知CVE）
2. 容器逃逸（突破隔离访问宿主机）
3. 特权容器滥用
4. 敏感信息泄露

**防止容器逃逸**：
1. **禁止特权容器**：`--privileged=false`
2. **使用非root用户**：`runAsNonRoot: true`
3. **只读文件系统**：`readOnlyRootFilesystem: true`
4. **删除危险capabilities**：`drop: [ALL]`
5. **启用Seccomp/AppArmor**：限制系统调用
6. **使用用户命名空间**：容器内root映射为宿主机普通用户

#### Q13: K8s的Request和Limit有什么区别？什么是QoS等级？

**答案**：
**Request vs Limit**：
- **Request**：调度依据，保证节点有这么多资源可分配
- **Limit**：资源上限，超过会被限制（CPU）或OOM Kill（内存）

**QoS等级**（OOM优先级从低到高）：
1. **BestEffort**：未设置任何request/limit，OOM时最先被杀
2. **Burstable**：设置了request/limit但不相等，中等优先级
3. **Guaranteed**：所有容器request==limit，OOM时最后被杀

**生产建议**：关键服务使用Guaranteed，配置合理的request和limit

---

#### Q14: HPA和VPA分别适用于什么场景？如何配置基于自定义指标的自动扩缩容？

**答案**：
**HPA（水平扩展）**：
- 适用场景：无状态服务，可快速横向扩展
- 触发条件：CPU/内存使用率、自定义指标（QPS、队列长度等）
- 配置：`minReplicas`、`maxReplicas`、`targetUtilization`

**VPA（垂直扩展）**：
- 适用场景：有状态服务，难以横向扩展
- 工作方式：自动调整Pod的resource request/limit
- 注意：会重启Pod，不适合敏感服务

**自定义指标HPA**：
1. 部署Prometheus Adapter
2. 配置custom metrics API
3. HPA中引用自定义指标（如requests_per_second）

#### Q15: etcd在K8s中的作用是什么？如何保证etcd的高可用？

**答案**：
**etcd作用**：
- 分布式KV存储，存储所有集群状态（Pod、Service、ConfigMap等）
- 使用Raft协议保证一致性
- 是K8s控制平面唯一的持久化存储

**高可用配置**：
1. **奇数节点**：3节点容忍1故障，5节点容忍2故障
2. **SSD存储**：写入延迟关键
3. **独立部署**：不与其他组件混部
4. **定期备份**：`etcdctl snapshot save`
5. **监控**：磁盘IOPS、网络延迟、Leader选举频率

#### Q16: 什么是Helm？它解决了什么问题？

**Helm定义**：Kubernetes的包管理器（类似apt/yum）

**解决的问题**：

3. **版本管理**：支持upgrade、rollback，跟踪发布历史

**核心概念**：
- **Chart**：K8s资源的模板集合
- **Values**：自定义配置参数
- **Release**：Chart的一次安装实例

#### Q17: GitOps是什么？ArgoCD的工作原理是怎样的？

**答案**：
**GitOps核心原则**：

#### Q18: Envoy代理的核心概念有哪些？什么是xDS协议？

#### Q19: Service Mesh中的mTLS是什么？为什么需要它？

**答案**：
**mTLS（双向TLS）**：

**为什么需要**：

3. **流量加密**：防止中间人攻击和窃听
4. **细粒度授权**：基于服务身份做访问控制

**Istio mTLS实现**：

- Envoy Sidecar自动处理证书轮换

#### Q21: NetworkPolicy 是什么？为什么很多团队配了策略却发现根本没生效？

**答案**：
`NetworkPolicy` 是 Kubernetes 提供的 Pod 网络访问控制模型，用来限制某组 Pod 的 ingress/egress 流量。它本身只是声明式策略，真正执行要依赖底层 CNI 插件，比如 Calico、Cilium 等。

常见失效原因有四类：
1. **CNI 不支持**：集群虽然能跑 Pod，但网络插件并不实现 NetworkPolicy。
2. **选择器没选中目标**：`podSelector`、`namespaceSelector` 标签写错，实际没命中任何 Pod。
3. **只配了 ingress，漏了 egress**：结果 DNS、数据库、外部 API 全被挡住，或反过来根本没限制出站。
4. **没有把 DNS 放行**：应用看起来像“网络不通”，实际是域名解析被策略拦住了。

面试里更成熟的回答是：NetworkPolicy 不是“写了 YAML 就安全”，而是要结合流量拓扑、DNS、跨命名空间调用和 CNI 落地能力一起验证。

#### Q22: PodDisruptionBudget（PDB）到底保护什么？它能保证哪些事，不能保证哪些事？

**答案**：
PDB 保护的是**自愿中断**下的最小可用副本数，比如节点维护、集群升级、`kubectl drain`、滚动发布这类由控制面主动发起的驱逐动作。它常用 `minAvailable` 或 `maxUnavailable` 来约束同一时间最多能少几个 Pod。

它能保证的事：
1. 节点排空和维护时，不会把关键服务一次性驱逐过多副本。
2. 集群运维动作会尊重服务的最小可用性要求。

它不能保证的事：
1. **节点突然宕机**、宿主机断电、网络分区，这些不是自愿驱逐，PDB 管不到。
2. 应用自己崩溃、探针失败、OOMKilled，PDB 也不会阻止。
3. 如果副本数本来就不够，PDB 也无法凭空创造可用实例。

所以 PDB 要和多副本、拓扑分散、探针治理、容量冗余一起使用，单独配置一个 PDB 并不等于高可用已经做好。

#### Q23: K8s 里 readiness 正常但延迟很高，通常先查哪三类指标？

**答案**：
readiness 只说明“可接流量”，不代表“服务性能健康”。优先看三类：
1. **应用层 RED**：请求量、错误率、P95/P99 延迟是否在特定接口或版本集中。
2. **容器资源层**：CPU throttle、内存回收/缺页、GC 停顿、线程池/连接池等待。
3. **依赖层**：下游数据库/MQ/外部 API 的 RT、超时率、连接饱和度。

排障目标是定位“慢在本服务还是慢在依赖”，而不是只看探针状态。

---

#### Q24: 为什么 PDB 配了也会出现业务不可用？请给一个真实故障路径。

**答案**：
常见故障路径：服务只有 2 副本，PDB 设 `minAvailable: 1`；一次节点维护触发 `drain` 时确实只驱逐 1 个 Pod，但剩余 Pod 恰好因下游抖动进入高延迟，readiness 虽通过但请求大量超时，业务实际不可用。

这类事故说明：PDB 只约束“驱逐数量”，不保证“剩余副本性能充足”。需要和容量余量、反亲和、依赖健康门禁一起设计。

#### Q25: NetworkPolicy 全开后 DNS 间歇失败，怎么快速定位是策略问题还是 CoreDNS 容量问题？

**答案**：
建议先分层排查：
1. **策略面**：检查 egress 是否放行到 DNS Service/Pod，确认 UDP/TCP 53 都放通。
2. **服务面**：看 CoreDNS QPS、延迟、错误码（SERVFAIL/NXDOMAIN）与 Pod 重启情况。
3. **链路面**：在问题 Pod 内做 `nslookup/dig`，结合 tcpdump 看是否有请求发出但无响应。

若策略放行后立即恢复，多半是策略配置问题；若放行后仍高失败且 CoreDNS 指标飙升，多半是容量或上游解析问题。

### 开放式设计题

**D1：设计一个支持日均1000次部署的CI/CD平台，核心架构是什么？**

**参考思路**：
- 流水线设计：代码提交→单元测试→镜像构建→安全扫描→灰度部署→全量发布
- 构建优化：多阶段Docker构建、分层缓存、并行构建多服务
- 部署策略：GitOps(ArgoCD)声明式部署、金丝雀发布+自动Rollback（基于Prometheus指标）
- 环境管理：Namespace隔离Dev/Staging/Prod、Kustomize overlay差异化配置
- 安全：镜像签名(Cosign)、准入控制(OPA/Kyverno)、RBAC最小权限
- 可观测：部署频率/失败率/回滚率/Lead Time四个DORA指标

**D2：K8s集群中某个Deployment的Pod频繁被驱逐（Evicted），如何定位和解决？**

**参考思路**：
- 确认驱逐原因：kubectl describe pod查Events、节点上kubectl describe node查Conditions
- 磁盘压力驱逐：nodefs/imagefs使用率超阈值→清理无用镜像和日志→调整GC策略
- 内存压力驱逐：节点可分配内存不足→检查是否有Pod未设limits导致内存无限增长→设置ResourceQuota
- QoS等级：Guaranteed > Burstable > BestEffort，优先驱逐BestEffort→确保核心服务设置requests=limits

**健康检查**：
- 配置livenessProbe（存活检查）和readinessProbe（就绪检查）
- 合理设置initialDelaySeconds和超时时间

**安全**：
- 使用非root用户运行容器
- 只读文件系统，删除不必要的capabilities
- 使用NetworkPolicy限制Pod间通信
- 敏感信息使用Secret，考虑Vault等外部密钥管理

**高可用**：
- 多副本部署，配置PodDisruptionBudget
- 使用Pod反亲和性分散到不同节点
- 跨可用区部署

**可观测性**：
- 结构化日志输出到stdout
- 暴露metrics供Prometheus采集
- 集成分布式追踪（Jaeger/Zipkin）
