## 1. Docker容器技术

### 1.1 什么是容器？

**容器 vs 虚拟机**：

| 特性 | 容器 | 虚拟机 |
|------|------|--------|
| **隔离级别** | 进程级别 | 操作系统级别 |
| **启动速度** | 秒级 | 分钟级 |
| **资源占用** | MB级别 | GB级别 |
| **性能** | 接近原生 | 有损耗 |
| **可移植性** | 高（镜像） | 低 |

### 1.2 Docker底层原理⭐⭐⭐

**为什么需要了解Docker底层？**
- 面试高频：容器是如何实现隔离的？

#### Namespace（命名空间）：隔离视图

**6种Namespace**：

```
1. PID Namespace：进程隔离
   - 容器内进程看到的PID从1开始
   - 宿主机看到真实的PID

2. Network Namespace：网络隔离
   - 独立的网卡、IP、路由表、iptables规则

3. Mount Namespace：文件系统隔离
   - 独立的文件系统挂载点

4. UTS Namespace：主机名隔离
   - 独立的hostname和domainname

5. IPC Namespace：进程间通信隔离
   - 独立的消息队列、信号量、共享内存

6. User Namespace：用户隔离
   - 容器内root ≠ 宿主机root
```

**PID Namespace详解**：

```c
// 创建PID Namespace
int child_pid = clone(child_func, stack, 
                      CLONE_NEWPID | SIGCHLD, NULL);

示例：
宿主机视角：
PID 1234 ← 容器进程

容器内视角：
PID 1 ← 容器内的第一个进程（通常是init）
PID 2 ← 应用进程

为什么容器内进程看不到宿主机进程？
- PID Namespace隔离了进程树
- 容器内只能看到自己Namespace的进程
```

**Network Namespace详解**：

```bash
# 创建Network Namespace
ip netns add container1

# 查看网络设备（容器内）
ip netns exec container1 ip link
# 输出：只有lo（本地回环）

# 宿主机
ip link
# 输出：eth0, docker0, veth123...

容器网络模式：
1. Bridge模式（默认）：
   容器 ← veth pair → docker0 bridge → eth0 → 外网
   
2. Host模式：
   容器直接使用宿主机网络（无隔离）
   
3. None模式：
   容器无网络
```

**为什么容器启动快？**

```
虚拟机启动：
1. 启动Hypervisor
2. 加载Guest OS内核
3. 初始化Guest OS（init进程、系统服务）
4. 启动应用
总时间：1-2分钟

容器启动：
1. 创建Namespace（微秒级）
2. 配置Cgroup（微秒级）
3. 挂载文件系统（毫秒级）
4. 启动应用进程
总时间：秒级

关键：容器共享宿主机内核，无需启动OS
```

#### Cgroup（控制组）：资源限制

**Cgroup控制的资源**：

```
1. CPU：限制CPU使用率
2. Memory：限制内存使用量
3. Disk I/O：限制磁盘读写速度
4. Network：限制网络带宽（需要tc工具）
```

**CPU限制**：

```bash
# 限制容器使用1个CPU核心
docker run --cpus=1 nginx

# 底层实现（Cgroup）
echo 100000 > /sys/fs/cgroup/cpu/docker/<container_id>/cpu.cfs_quota_us
echo 100000 > /sys/fs/cgroup/cpu/docker/<container_id>/cpu.cfs_period_us
# quota/period = 100000/100000 = 1个CPU

# 限制0.5个CPU
echo 50000 > cpu.cfs_quota_us  # quota/period = 50000/100000 = 0.5
```

**内存限制**：

```bash
# 限制容器使用512MB内存
docker run -m 512m nginx

# 底层实现（Cgroup）
echo 536870912 > /sys/fs/cgroup/memory/docker/<container_id>/memory.limit_in_bytes
# 512MB = 536870912字节

# 超过限制会怎样？
# OOM Killer杀死容器进程
```

**为什么需要资源限制？**

```
场景：单台宿主机运行10个容器
- 无限制：某个容器占用90% CPU，其他容器饿死
- 有限制：每个容器最多10% CPU，公平调度

Cgroup的作用：
- 防止"吵闹邻居"问题（Noisy Neighbor）
- 保证资源公平性
- 提高资源利用率（超售）
```

#### Union FS（联合文件系统）：镜像分层

**为什么需要分层？**

```
传统方式（单层镜像）：
Ubuntu镜像：2GB
每个应用镜像：2GB（Ubuntu）+ 100MB（应用）
10个应用 = 20GB + 1GB = 21GB

分层方式（Union FS）：
Ubuntu镜像：2GB（共享）
应用层1：100MB
应用层2：100MB
...
10个应用 = 2GB + 1GB = 3GB

节省：21GB → 3GB（节省86%）
```

**Docker镜像分层结构**：

```
Dockerfile:
FROM ubuntu:20.04          ← Layer 1（基础层，只读）
RUN apt-get update         ← Layer 2（只读）
RUN apt-get install nginx  ← Layer 3（只读）
COPY app.py /app/          ← Layer 4（只读）
CMD ["python", "app.py"]   ← 元数据

容器启动后：
┌─────────────────────┐
│  容器层（读写）      │ ← 可写（容器内的修改）
├─────────────────────┤
│  Layer 4: app.py    │ ← 只读
├─────────────────────┤
│  Layer 3: nginx     │ ← 只读
├─────────────────────┤
│  Layer 2: update    │ ← 只读
├─────────────────────┤
│  Layer 1: ubuntu    │ ← 只读
└─────────────────────┘
```

**写时复制（Copy-on-Write）**：

```
读取文件：
1. 从上往下查找文件
2. 找到后直接读取（不复制）

修改文件：
1. 从上往下查找文件
2. 复制到容器层（可写层）
3. 修改容器层的副本
4. 原文件不变（只读层）

示例：
容器内执行：echo "hello" >> /etc/nginx/nginx.conf

步骤：
1. 查找nginx.conf（在Layer 3）
2. 复制到容器层
3. 修改容器层的nginx.conf
4. Layer 3的原文件不变

为什么这样设计？
- 保护只读层（可以安全共享）
- 隔离容器修改（互不影响）
- 支持容器快速创建（不需要复制整个镜像）
```

**OverlayFS工作原理**：

```
OverlayFS结构：
lowerdir（只读层）：镜像层1、层2、层3...
upperdir（可写层）：容器层
merged（合并视图）：容器内看到的文件系统

示例：
lowerdir=/layer1:/layer2:/layer3
upperdir=/container_rw
workdir=/work
merged=/merged

容器内看到的路径：/merged
实际存储：
- 只读文件：lowerdir
- 新建/修改文件：upperdir
```

### 1.3 Docker网络

**Bridge模式详解**：

```
默认网络架构：
容器A（172.17.0.2）
   |
   veth0 ← veth pair → veth123
                        |
                     docker0 bridge（172.17.0.1）
                        |
                     veth456 ← veth pair → veth1
   |
容器B（172.17.0.3）

数据流：
容器A → 容器B：
1. 容器A发送到172.17.0.3
2. veth0 → veth123 → docker0 bridge
3. docker0查找MAC地址表，转发到veth456
4. veth456 → veth1 → 容器B

容器A → 外网：
1. 容器A发送到8.8.8.8
2. docker0 → iptables NAT（SNAT，源地址转换）
3. 宿主机eth0 → 外网
4. 外网响应 → eth0 → iptables（DNAT）→ docker0 → 容器A
```

**端口映射原理**：

```bash
# 映射容器80端口到宿主机8080
docker run -p 8080:80 nginx

# 底层实现（iptables规则）
iptables -t nat -A DOCKER -p tcp --dport 8080 \
         -j DNAT --to-destination 172.17.0.2:80

数据流：
外网 → 宿主机8080 → iptables DNAT → 172.17.0.2:80 → 容器
```

## 2. Kubernetes核心原理

### 2.1 K8s架构

**Master节点（控制平面）**：

```
1. API Server：
   - 集群的唯一入口
   - RESTful API
   - 认证、授权、准入控制

2. etcd：
   - 分布式KV存储
   - 存储集群状态（Pod、Service等）
   - 使用Raft协议保证一致性

3. Scheduler：
   - 调度Pod到合适的Node
   - 考虑资源、亲和性、污点等

4. Controller Manager：
   - Deployment Controller
   - ReplicaSet Controller
   - Node Controller
   - ...
```

**Worker节点（数据平面）**：

```
1. Kubelet：
   - Node上的Agent
   - 管理Pod生命周期
   - 上报Node状态

2. Kube-proxy：
   - 实现Service网络
   - iptables/IPVS规则

3. Container Runtime：
   - Docker/containerd/CRI-O
   - 实际运行容器
```

### 2.2 Pod调度原理⭐⭐⭐

**调度流程**：

```
步骤1：过滤（Filtering）
- 筛选出满足条件的Node
- 条件：资源充足、污点容忍、节点选择器等

示例：
请求：CPU=2核，内存=4GB
Node1: CPU=4核（剩余3核），内存=8GB（剩余6GB）✅
Node2: CPU=4核（剩余1核），内存=8GB（剩余6GB）❌ CPU不足
Node3: CPU=8核（剩余5核），内存=16GB（剩余10GB）✅

候选节点：Node1、Node3

步骤2：打分（Scoring）
- 对候选Node评分
- 选择得分最高的Node

评分维度：
1. 资源剩余量（LeastRequestedPriority）
   Node1: (3-2)/4 + (6-4)/8 = 0.25 + 0.25 = 0.5
   Node3: (5-2)/8 + (10-4)/16 = 0.375 + 0.375 = 0.75 ✅

2. 负载均衡（BalancedResourceAllocation）
   - 尽量让CPU和内存使用率接近

3. 亲和性（Affinity）
   - 尽量调度到亲和的Node

步骤3：绑定（Binding）
- 创建Binding对象
- Kubelet拉起Pod
```

**为什么需要调度器？**

```
场景：100个Node，1000个Pod
- 无调度器：随机分配，可能Node1满载，Node2空闲
- 有调度器：根据资源、亲和性等智能分配，负载均衡

K8s调度器的优势：
- 资源利用率最大化
- 满足业务需求（亲和性、反亲和性）
- 支持自定义调度策略
```

### 2.3 Service网络原理⭐⭐⭐

**为什么需要Service？**

```
问题：
- Pod IP不固定（重启后变化）
- 多个Pod副本，如何负载均衡？

Service的作用：
- 提供稳定的虚拟IP（ClusterIP）
- 自动负载均衡（iptables/IPVS）
- 服务发现（DNS）
```

**ClusterIP实现原理**：

```
示例：
Service ClusterIP: 10.96.0.10:80
后端Pod: 
- 172.16.1.5:8080
- 172.16.2.6:8080
- 172.16.3.7:8080

iptables规则（简化）：
1. 访问10.96.0.10:80，跳转到负载均衡链
   -A KUBE-SERVICES -d 10.96.0.10/32 -p tcp --dport 80 -j KUBE-SVC-ABC

2. 负载均衡链（随机选择一个Pod）
   -A KUBE-SVC-ABC -m statistic --mode random --probability 0.33 -j KUBE-SEP-1
   -A KUBE-SVC-ABC -m statistic --mode random --probability 0.50 -j KUBE-SEP-2
   -A KUBE-SVC-ABC -j KUBE-SEP-3

3. DNAT到Pod IP
   -A KUBE-SEP-1 -p tcp -j DNAT --to-destination 172.16.1.5:8080
   -A KUBE-SEP-2 -p tcp -j DNAT --to-destination 172.16.2.6:8080
   -A KUBE-SEP-3 -p tcp -j DNAT --to-destination 172.16.3.7:8080

数据流：
客户端 → 10.96.0.10:80 → iptables DNAT → 172.16.1.5:8080（Pod）
```

**IPVS vs iptables**：

| 特性 | iptables | IPVS |
|------|----------|------|
| **性能** | O(n)（规则数） | O(1)（哈希表） |
| **负载均衡算法** | 随机 | 轮询、最小连接、源地址哈希等 |
| **规则数量** | < 1000 Service | > 10000 Service |
| **成熟度** | 高（默认） | 中（K8s 1.11+） |

```bash
# 启用IPVS模式
kubectl edit configmap kube-proxy -n kube-system
# 修改mode: ipvs
```

### 2.4 Pod网络（CNI）

**Flannel网络原理**：

```
VXLAN模式：
Node1（10.0.1.0/24）        Node2（10.0.2.0/24）
  Pod1（10.0.1.5）            Pod2（10.0.2.6）
    |                           |
  cni0 bridge                 cni0 bridge
    |                           |
  flannel.1（VXLAN）          flannel.1（VXLAN）
    |                           |
  eth0（192.168.1.10）        eth0（192.168.1.20）
    └────────────┬──────────────┘
                 │
             物理网络

数据流（Pod1 → Pod2）：
1. Pod1发送到10.0.2.6
2. 路由表：10.0.2.0/24 via flannel.1
3. flannel.1封装VXLAN（外层IP：192.168.1.10 → 192.168.1.20）
4. 通过物理网络传输
5. Node2的flannel.1解封装
6. 路由到Pod2
```

### 2.5 Deployment滚动更新

**滚动更新过程**：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 10
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 2  # 最多2个不可用
      maxSurge: 2        # 最多额外2个Pod
```

**更新流程**：

```
初始：10个Pod（v1）

步骤1：创建2个新Pod（v2）
v1: ████████████ (10个)
v2: ██ (2个)

步骤2：删除2个旧Pod（v1）
v1: ████████ (8个)
v2: ██ (2个)

步骤3：创建2个新Pod（v2）
v1: ████████ (8个)
v2: ████ (4个)

...（循环）

最终：10个Pod（v2）
v1: (0个)
v2: ████████████ (10个)

为什么这样设计？
- 保证可用性（始终有8-12个Pod运行）
- 快速回滚（发现问题立即停止）
- 零停机部署
```

## 3. Service Mesh

### 3.1 什么是Service Mesh？

**传统微服务 vs Service Mesh**：

```
传统微服务：
应用A → [负载均衡/熔断/重试/监控] → 应用B
         ↑ 每个应用都要实现这些功能（代码耦合）

Service Mesh：
应用A → Sidecar Proxy → Sidecar Proxy → 应用B
         ↑ 负载均衡/熔断/重试/监控
         
优势：
- 应用代码无侵入
- 统一的流量管理
- 多语言支持（Go/Java/Python...）
```

## 5. 常见面试题

### Q1: 容器和虚拟机的区别？

**答案**：
- **隔离级别**：容器（进程级别），虚拟机（OS级别）
- **启动速度**：容器（秒级），虚拟机（分钟级）
- **资源占用**：容器（MB），虚拟机（GB）
- **底层技术**：容器（Namespace+Cgroup），虚拟机（Hypervisor）

### Q2: Docker镜像为什么分层？

**答案**：
- **节省存储**：共享基础层（Ubuntu层只需一份）
- **加速构建**：只重建变化的层（利用缓存）
- **加速分发**：只传输变化的层

### Q3: K8s如何实现服务发现？

**答案**：
- **DNS**：每个Service自动注册DNS（如`nginx.default.svc.cluster.local`）
- **环境变量**：Pod启动时注入Service的IP和端口
- **API Server**：直接查询API

### Q4: K8s如何实现负载均衡？

**答案**：
- **ClusterIP**：通过iptables/IPVS规则DNAT到后端Pod
- **NodePort**：每个Node监听相同端口，转发到ClusterIP
- **LoadBalancer**：云厂商的LB，分发到NodePort
