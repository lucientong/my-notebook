## GMP调度模型

### 为什么需要 GMP？（从问题出发理解设计）⭐⭐⭐

在理解 GMP 之前，先理解 Go 面临的核心问题：

**操作系统线程的代价太大**：

```
创建一个 OS 线程：
- 栈空间：默认 1MB~8MB（固定分配）
- 创建耗时：约 10~100 微秒（需要内核态切换）
- 上下文切换：约 1~10 微秒（保存/恢复寄存器、刷新 TLB）
- 数量上限：受内存限制，通常几千~几万个

创建一个 Goroutine：
- 栈空间：初始 2KB（按需增长，最大 1GB）
- 创建耗时：约 0.3 微秒（纯用户态操作）
- 上下文切换：约 0.1~0.2 微秒（不经过内核）
- 数量上限：轻松创建百万级
```

**核心矛盾**：Go 希望支持百万级并发，但操作系统只能高效管理几千个线程。

**解决方案**：在用户态实现一个调度器，用少量 OS 线程运行大量 Goroutine → 这就是 GMP 模型。

> **类比**：把 GMP 想象成一家公司——
> - **G（Goroutine）= 待办任务**（工单），数量巨大
> - **M（Machine）= 员工**（OS 线程），数量有限
> - **P（Processor）= 工位 + 工具箱**（执行环境），员工必须坐在工位上才能干活
>
> 公司不可能为每个任务雇一个员工（太贵），所以用少量员工轮流处理大量任务。

### 核心概念（G、M、P 详解）⭐⭐⭐

#### G — Goroutine（任务）

```go
// 每一个 go 关键字都会创建一个 G
go func() {
    fmt.Println("我是一个 G")
}()
```

G 的内部结构（简化）：

```go
// runtime/runtime2.go（简化）
type g struct {
    stack       stack    // 栈内存（初始 2KB，可增长）
    stackguard0 uintptr  // 栈溢出检测（用于抢占调度）
    sched       gobuf    // 保存的寄存器状态（SP、PC、BP 等）
    atomicstatus uint32  // G 的状态
    goid         int64   // goroutine ID
    m            *m      // 当前绑定的 M（正在执行时）
    // ...
}
```

**G 的状态流转**：

```
                  go func()
                     │
                     ▼
    ┌──────────── _Gidle ──────────────┐
    │           （刚分配，未初始化）       │
    │                │                  │
    │                ▼                  │
    │         _Grunnable ◄─────────────┤
    │         （在队列中等待）            │
    │                │                  │
    │       被 P 调度 │                  │
    │                ▼                  │
    │          _Grunning               │
    │          （正在执行）               │
    │           │      │               │
    │    系统调用 │      │ channel/IO    │
    │           ▼      ▼               │
    │     _Gsyscall  _Gwaiting ────────┘
    │     （系统调用） （等待唤醒）  唤醒后回到 _Grunnable
    │           │
    │           ▼
    │     系统调用返回 → _Grunnable
    │
    │     函数返回
    │         │
    │         ▼
    └──→  _Gdead（执行完毕，可复用）
```

> **关键理解**：G 不是线程，它只是一个"任务描述"（保存了函数入口、参数、栈），需要被调度到 M 上才能真正执行。

---

#### M — Machine（OS 线程）

M 是真正的操作系统线程，是 CPU 调度的基本单位。

```go
// runtime/runtime2.go（简化）
type m struct {
    g0      *g     // 调度栈（每个 M 都有一个 g0，专门用于执行调度代码）
    curg    *g     // 当前正在执行的 G
    p       puintptr // 当前绑定的 P（M 必须持有 P 才能执行 G）
    spinning bool   // 是否处于自旋状态
    // ...
}
```

**M 的关键特性**：
- M 的数量**没有硬上限**（默认最大 10000，可通过 `runtime/debug.SetMaxThreads` 调整）
- **M 必须持有 P 才能执行 G**（没有 P 的 M 要么在执行系统调用，要么在休眠）
- M 有一个特殊的 `g0`，不执行用户代码，专门负责调度（在 G 之间切换）

```
M 的工作循环：

     ┌──────────────────────────────────┐
     │                                  │
     ▼                                  │
  g0 栈：schedule()                     │
     │  寻找可运行的 G                   │
     │  找到 G                          │
     ▼                                  │
  切换到 G 的栈：执行用户代码             │
     │  G 执行完毕 / 被抢占 / 阻塞       │
     │                                  │
     └──── 切回 g0 栈 ─────────────────┘
```

> **类比**：M 就像员工的双手——真正干活的部分。`g0` 是员工的"大脑"，负责决定接下来干哪个任务。

---

#### P — Processor（执行环境）

P 是 Go 调度器最精妙的设计，它**不是**CPU、不是线程，而是一个"执行上下文"。

```go
// runtime/runtime2.go（简化）
type p struct {
    id          int32
    status      uint32   // P 的状态
    m           muintptr // 当前绑定的 M
    runqhead    uint32   // 本地队列头
    runqtail    uint32   // 本地队列尾
    runq        [256]guintptr // 本地运行队列（固定 256 个槽位）
    runnext     guintptr // 下一个优先执行的 G（快速通道）
    // 还包含：内存分配缓存(mcache)、defer池、sudog缓存等
}
```

**为什么需要 P？（不能直接让 M 执行 G 吗？）**

```
假设没有 P，只有 G 和 M：

方案A：一个全局队列
┌────────────────────────────────────┐
│ 全局队列：G1 G2 G3 G4 G5 G6 ... │  ← 所有 M 竞争同一把锁
└────────────────────────────────────┘
      ↑   ↑   ↑   ↑
     M0  M1  M2  M3

问题：所有 M 抢同一把锁，高并发下锁竞争严重，性能急剧下降

方案B：引入 P（每个 P 有本地队列）
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ P0       │  │ P1       │  │ P2       │  │ P3       │
│ G1 G2 G3 │  │ G4 G5 G6 │  │ G7 G8 G9 │  │ G10 G11  │
└─────┬────┘  └─────┬────┘  └─────┬────┘  └─────┬────┘
      │             │             │              │
     M0            M1            M2             M3

优势：
- 大部分情况下 M 从自己 P 的本地队列取 G，无锁操作！
- 本地队列空了才去全局队列或偷其他 P 的
- P 还携带了内存缓存(mcache)等资源，减少全局竞争
```

> **P 的数量** = `GOMAXPROCS`（默认 = CPU 核心数），表示最多有多少个 G 能**同时**执行。

### 调度流程（一个 Goroutine 的一生）⭐⭐⭐

#### 第一步：创建 G

```go
go func() {
    fmt.Println("Hello")
}()
```

**底层发生了什么？**

```go
// 编译器将 go func() 转换为 runtime.newproc 调用
// runtime/proc.go（简化）
func newproc(fn *funcval) {
    // 1. 获取当前 G 和 P
    gp := getg()
    pp := gp.m.p.ptr()

    // 2. 尝试从 P 的空闲 G 缓存中获取（复用，避免频繁分配）
    newg := gfget(pp)
    if newg == nil {
        // 3. 缓存没有，则新建一个 G（分配 2KB 栈）
        newg = malg(stackMin) // stackMin = 2048 bytes
    }

    // 4. 初始化 G 的状态（设置 PC 指向 fn，SP 指向栈顶）
    newg.sched.pc = fn.fn  // 程序计数器 → 函数入口
    newg.sched.sp = ...    // 栈指针

    // 5. 放入 P 的队列
    runqput(pp, newg, true) // true 表示优先放 runnext

    // 6. 如果有空闲 P，唤醒一个 M 来执行
    if atomic.Load(&sched.npidle) != 0 {
        wakep()
    }
}
```

**G 被放入队列的优先级**：

```
go func() 创建新 G
    │
    ▼
① P.runnext（快速通道，只能放1个）
    │ 如果 runnext 已有旧 G → 旧 G 被踢到本地队列
    ▼
② P.runq（本地队列，容量 256）
    │ 如果满了（256个）
    ▼
③ Global Run Queue（全局队列，需要加锁）
    会把本地队列一半的 G + 新 G 一起移到全局队列
```

> **runnext 的设计用意**：刚创建的 G 通常会很快被执行（时间局部性），放在 runnext 可以减少一次队列操作。

---

#### 第二步：M 获取 G 并执行

M 从绑定的 P 中找 G，找不到就扩大搜索范围：

```go
// runtime/proc.go（简化）
func schedule() {
    pp := getg().m.p.ptr()

    // === 第一优先级：每 61 次调度检查一次全局队列 ===
    // 为什么是 61？质数，避免与其他周期性行为产生共振
    if pp.schedtick%61 == 0 {
        if gp := globrunqget(pp, 1); gp != nil {
            execute(gp)
            return
        }
    }

    // === 第二优先级：P 的本地队列（无锁，最快） ===
    if gp := runqget(pp); gp != nil {
        execute(gp)
        return
    }

    // === 第三优先级：扩大搜索范围 ===
    // 这是一个复杂的函数，按顺序尝试：
    gp := findrunnable() // 会阻塞直到找到
    execute(gp)
}

func findrunnable() *g {
    // 1. 再次检查本地队列（防止并发放入）
    // 2. 检查全局队列
    // 3. 检查 netpoller（网络IO就绪的G）
    // 4. 从其他 P 偷取（Work Stealing）
    // 5. 都没有 → 休眠等待
}
```

**图解查找过程**：

```
M 需要一个 G 来执行：

  ① 本地队列有吗？ ───有──→ 拿到，执行！（无锁，最快）
         │
         无
         ▼
  ② 全局队列有吗？ ───有──→ 拿到（加锁），执行！
         │                   顺便多拿一批到本地队列
         无
         ▼
  ③ netpoller 有就绪的？─有→ 拿到，执行！
         │
         无
         ▼
  ④ 从其他 P 偷取     ───有──→ 偷一半，执行！
         │
         无
         ▼
  ⑤ 休眠（等待被唤醒）
```

#### 第三步：抢占调度（确保公平性）

**问题**：如果一个 G 一直执行不让出 CPU，其他 G 就会饿死。

**Go 1.14 之前：协作式抢占**

```go
// 编译器在函数入口插入检查代码
func someFunc() {
    // 编译器自动插入：
    if stackguard0 == stackPreempt {
        // 被标记为"需要抢占"，主动让出
        gopreempt()
    }
    // ... 用户代码 ...
}
```

依赖函数调用时的栈检查 → **死循环没有函数调用就无法抢占**：

```go
// Go 1.13 及之前，这个 goroutine 会霸占 CPU
func evil() {
    for {
        // 纯计算，无函数调用，永远不会触发栈检查
        // 其他 goroutine 全部饿死！
    }
}
```

**Go 1.14+：基于信号的异步抢占**

```
工作流程：

Sysmon（后台监控线程，每 10ms~10s 检查一次）
    │
    │ 发现 G 运行超过 10ms
    │
    ▼
发送 SIGURG 信号给 G 所在的 M
    │
    │ M 收到信号，进入信号处理函数
    ▼
asyncPreempt()
    │ 保存当前 G 的所有寄存器状态
    │ 将 G 状态设为 _Grunnable，放回队列
    │
    ▼
M 切回 g0，调用 schedule() 执行下一个 G
```

```go
// Go 1.14+，死循环也能被抢占了
func notSoEvil() {
    for {
        // 运行超过 10ms 会被 SIGURG 信号强制中断
        // 保存状态，放回队列，其他 G 有机会执行
    }
}
```

> **为什么选 SIGURG？** 因为 SIGURG 几乎没有其他用途（原本用于 TCP 带外数据），不会和用户程序的信号处理冲突。

#### Sysmon 监控线程

Sysmon 是一个特殊的后台 M，**不需要绑定 P**，独立运行，负责"兜底"：

```go
// runtime/proc.go（简化）
func sysmon() {
    for {
        // 动态调整检查频率：空闲时慢（10ms→10s），忙碌时快
        usleep(delay)

        // 1. 抢占长时间运行的 G（> 10ms）
        retake(now)

        // 2. 检查 netpoller（有网络IO就绪的G需要唤醒）
        if netpollinited() {
            list := netpoll(0) // 非阻塞检查
            injectglist(&list) // 将就绪的 G 放入全局队列
        }

        // 3. 强制 GC（超过 2 分钟未 GC）
        if t := (gcTrigger{kind: gcTriggerTime, now: now}); t.test() {
            forcegchelper()
        }

        // 4. 释放长时间 syscall 中的 P（Hand Off）
        // 如果 M 被系统调用阻塞了超过 10ms，
        // 把 P 从这个 M 上摘下来给其他 M 用
    }
}
```

### 重要机制详解⭐⭐⭐

#### 1. Work Stealing（工作窃取）

当 P 的本地队列空了，先去全局队列找，还没有就去**偷**其他 P 的任务。

```
偷取之前：                              偷取之后：
┌────────────┐  ┌────────────┐       ┌────────────┐  ┌────────────┐
│ P0         │  │ P1         │       │ P0         │  │ P1         │
│ G1 G2 G3 G4│  │ (空)       │       │ G1 G2      │  │ G3 G4      │
│ 很忙！      │  │ 闲着没事   │  →    │ 还在忙     │  │ 有活干了   │
└────────────┘  └────────────┘       └────────────┘  └────────────┘
```

```go
// runtime/proc.go（简化）
func stealWork(pp *p) (gp *g, inheritTime bool) {
    // 随机选择一个 P 开始偷（避免总偷同一个）
    for i := 0; i < 4; i++ {  // 尝试 4 次
        victim := allp[fastrand()%len(allp)]
        if victim == pp {
            continue
        }
        // 偷取 victim 本地队列的一半
        if gp := runqsteal(pp, victim, stealRunNextG); gp != nil {
            return gp, false
        }
    }
    return nil, false
}
```

**设计要点**：
- 偷一半（而不是一个），减少偷取次数
- 随机选择目标 P，避免热点竞争
- 无锁 CAS 实现，偷取过程高效

---

#### 2. Hand Off（P 移交）

**问题**：M 执行系统调用（如文件 IO）会阻塞，如果 P 跟着一起阻塞，P 上排队的其他 G 全部停滞。

```
系统调用前：                   Hand Off 后：
┌──────────┐                 ┌──────────┐  ┌──────────┐
│ M0 + P0  │                 │ M0 (无P) │  │ M2 + P0  │
│ G1(执行)  │                 │ G1(阻塞)  │  │ G2(执行)  │
│ G2 G3    │  ───────→       │ syscall  │  │ G3       │
│ (等待中)  │                 │ 中...    │  │ 继续运行  │
└──────────┘                 └──────────┘  └──────────┘

                             M0 系统调用返回后：
                             - 尝试获取一个空闲 P
                             - 有 → 绑定 P，继续执行 G1
                             - 没有 → 将 G1 放入全局队列，M0 休眠
```

```go
// 当 G 进入系统调用时
func entersyscall() {
    // 1. 保存 G 的状态
    save(gp.sched.pc, gp.sched.sp)
    // 2. 标记 P 为 _Psyscall
    pp.status = _Psyscall
    // 3. 解除 M 和 P 的绑定（逻辑上）
    // Sysmon 发现后会把 P 交给其他 M
}
```

> **为什么不在进入 syscall 时立即 Hand Off？** 因为大部分系统调用很快就返回了，立即 Hand Off 反而浪费。所以由 Sysmon 在 10ms 后检查，如果还在阻塞才执行 Hand Off。

---

#### 3. Spinning Thread（自旋线程）

当 M 暂时找不到 G 时，不立即休眠，先**自旋**（空转）一段时间。

```
为什么自旋而不是直接休眠？

时间线（没有自旋）：
M 空闲 → 休眠（耗时~10μs）→ 新 G 来了 → 唤醒 M（耗时~10μs）→ 开始执行
                                                               总延迟 ≈ 20μs

时间线（有自旋）：
M 空闲 → 自旋等待（消耗少量 CPU）→ 新 G 来了 → 立即执行
                                                总延迟 ≈ 0
```

**限制**：最多 `GOMAXPROCS` 个 M 同时自旋（避免所有 CPU 都在空转浪费电）。

### 面试高频问题⭐⭐⭐

**Q1: GOMAXPROCS 是什么？如何设置？**

```go
// GOMAXPROCS 控制 P 的数量（≠ M 的数量！≠ G 的数量！）
// P 的数量 = 最大同时并行执行的 G 的数量

// 设置最大 P 数量（默认等于 CPU 核心数）
runtime.GOMAXPROCS(8)

// 查看当前值
n := runtime.GOMAXPROCS(0) // 传 0 不修改，只查询
fmt.Println("GOMAXPROCS:", n)
```

**原则**：
- **CPU密集型**：设置为 CPU 核心数（默认值就好）
- **IO密集型**：可以略大于 CPU 核心数（更多 P 意味着更多 G 能同时发起 IO）
- **容器环境**：注意 cgroup 限制，Go 默认读 `runtime.NumCPU()` 可能拿到宿主机核心数

```go
// 容器环境推荐使用 uber 的库自动适配
import _ "go.uber.org/automaxprocs"

func main() {
    // 自动根据 cgroup 限制设置 GOMAXPROCS
    // 例如容器限制 4 核，就设置为 4（而不是宿主机的 64）
}
```

**Q2: G、M、P 的数量关系？**

```
P 的数量：GOMAXPROCS（默认 = CPU核心数），固定的
M 的数量：按需创建，最多 10000（有系统调用阻塞时会创建新 M）
G 的数量：用户创建多少就有多少，百万级都没问题

关系：
- 同时运行的 G 数量 ≤ P 数量 ≤ M 数量
- M 可能 > P（有些 M 在执行系统调用，没有绑定 P）
- G >> P（大量 G 在队列中等待调度）
```

### Go 协程 vs Python 协程⭐⭐⭐

> Go 和 Python 都有"协程"概念，但设计理念和实现方式**完全不同**。

#### 核心差异对比

| 维度 | Go Goroutine | Python Coroutine (`async/await`) |
|------|-------------|----------------------------------|
| **调度方式** | **抢占式**（Go 1.14+，运行时自动切换） | **协作式**（必须遇到 `await` 才让出） |
| **调度器** | Go runtime 内置 GMP 调度器（用户态） | 事件循环 `asyncio.EventLoop`（用户态） |
| **并行能力** | ✅ 真正并行（多个 M 绑定多核 CPU） | ❌ 单线程并发（受 GIL 限制，同一时刻只有一个协程执行） |
| **栈模型** | **有栈协程**（每个 G 有独立栈，2KB 起步，可增长） | **无栈协程**（基于生成器/状态机，无独立栈） |
| **创建方式** | `go func()` | `async def` + `await` |
| **通信方式** | Channel（CSP 模型） | `asyncio.Queue` / `Future` |
| **适用场景** | CPU 密集 + IO 密集（通用） | IO 密集（网络请求、文件IO） |
| **创建成本** | 极低（~0.3μs，2KB内存） | 低（~1μs，但比 goroutine 略重） |
| **数量上限** | 百万级 | 万级~十万级（受事件循环性能限制） |

---

#### 调度方式差异（最关键的区别）

**Go：运行时自动调度，开发者不用操心**

```go
func main() {
    // 开发者只管 go，调度器自动处理一切
    for i := 0; i < 100000; i++ {
        go func(id int) {
            // 无论是 CPU 计算还是 IO，都会被自动调度
            result := heavyComputation(id) // CPU密集 → 运行10ms后被抢占
            resp := httpGet(url)           // IO密集 → 阻塞时自动切换
            fmt.Println(result, resp)
        }(i)
    }
    select {} // 阻塞主协程
}
// 开发者完全不需要关心"什么时候让出 CPU"
```

**Python：开发者必须显式用 `await` 让出控制权**

```python
import asyncio

async def main():
    tasks = []
    for i in range(1000):
        tasks.append(fetch_data(i))
    await asyncio.gather(*tasks)

async def fetch_data(id):
    # ✅ 这里会让出控制权，事件循环可以切换到其他协程
    resp = await aiohttp.get(f"http://api.com/{id}")

    # ❌ 这里不会让出！如果计算很久，整个事件循环都卡住
    result = heavy_computation(id)  # 阻塞整个线程！

    return result

# 如果忘记写 await，协程就不会让出，其他协程全部饿死
```

> **关键区别**：Go 的 goroutine 即使在做 CPU 密集计算也会被抢占调度（10ms 强制切换），Python 的协程只有遇到 `await` 才会让出，CPU 密集计算会阻塞整个事件循环。

#### 并行能力差异

```
Go（真正并行）：
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ CPU 核心0 │  │ CPU 核心1 │  │ CPU 核心2 │  │ CPU 核心3 │
│ M0 → G1  │  │ M1 → G2  │  │ M2 → G3  │  │ M3 → G4  │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
→ 4 个 goroutine 在 4 个核心上同时执行（并行）

Python asyncio（只能并发，不能并行）：
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ CPU 核心0 │  │ CPU 核心1 │  │ CPU 核心2 │  │ CPU 核心3 │
│ 事件循环  │  │ (空闲)   │  │ (空闲)   │  │ (空闲)   │
│ → 协程1   │  │          │  │          │  │          │
└──────────┘  └──────────┘  └──────────┘  └──────────┘
→ 所有协程在同一个线程中交替执行（并发但不并行）
→ 受 GIL 限制，即使用多线程也无法并行执行 Python 字节码
```

#### 有栈 vs 无栈协程

```
Go（有栈协程）：
每个 Goroutine 有自己独立的栈（初始 2KB，可增长到 1GB）
┌─────────┐  ┌─────────┐  ┌─────────┐
│  G1 栈   │  │  G2 栈   │  │  G3 栈   │
│ 2KB~1GB │  │ 2KB~1GB │  │ 2KB~1GB │
│ 函数调用 │  │ 函数调用 │  │ 函数调用 │
│ 局部变量 │  │ 局部变量 │  │ 局部变量 │
└─────────┘  └─────────┘  └─────────┘

优势：
- 可以在任意位置暂停/恢复（包括深层函数调用中）
- 编程模型简单：写同步代码，享受异步性能

Python（无栈协程）：
协程本质是一个状态机对象，没有独立栈
┌──────────────────────┐
│  协程对象（状态机）    │
│  状态: SUSPENDED     │
│  下一步: await 之后   │
│  局部变量: {...}      │
└──────────────────────┘

优势：
- 内存开销更小（不需要预分配栈）
限制：
- 只能在 await 点暂停（不能在普通函数调用中暂停）
- 需要 async/await 语法传染（调用链上所有函数都要标记 async）
```

**"async 传染性"问题**（Python 特有）：

```python
# Python 的 async 会"传染"整个调用链
async def top_level():
    result = await middle_layer()   # middle_layer 必须是 async
    return result

async def middle_layer():
    data = await bottom_layer()     # bottom_layer 必须是 async
    return process(data)

async def bottom_layer():
    return await aiohttp.get(url)   # 最底层的IO操作

# 如果底层要改成异步，整个调用链都要改成 async
# 这就是"函数颜色"问题（What Color is Your Function?）
```

```go
// Go 没有这个问题——所有函数都是"同一颜色"
func topLevel() Result {
    return middleLayer()  // 普通调用
}

func middleLayer() Data {
    return bottomLayer()  // 普通调用
}

func bottomLayer() Data {
    resp, _ := http.Get(url)  // IO 操作，goroutine 自动挂起
    // 看起来是同步代码，实际上 runtime 自动处理了异步
    return parse(resp)
}
```

**Q2: Goroutine 泄漏如何发现和解决？**

**场景1：channel 永久阻塞**

```go
// ❌ 泄漏
func leak() {
    ch := make(chan int)
    go func() {
        val := <-ch // 永远阻塞，goroutine 泄漏
        fmt.Println(val)
    }()
    // ch 没有写入，goroutine 永远等待
}

// ✅ 修复：使用 context 超时
func fixed() {
    ch := make(chan int)
    ctx, cancel := context.WithTimeout(context.Background(), 1*time.Second)
    defer cancel()
    
    go func() {
        select {
        case val := <-ch:
            fmt.Println(val)
        case <-ctx.Done():
            fmt.Println("timeout")
            return
        }
    }()
}
```

**场景2：无限循环不退出**

```go
// ❌ 泄漏
func poll() {
    go func() {
        for {
            // 无退出条件
            time.Sleep(1 * time.Second)
        }
    }()
}

// ✅ 修复：使用 done channel
func fixed() {
    done := make(chan struct{})
    
    go func() {
        for {
            select {
            case <-done:
                return
            default:
                time.Sleep(1 * time.Second)
            }
        }
    }()
    
    // 停止时关闭 done
    close(done)
}
```

**2. Goroutine 池（Worker Pool）**

```go
func workerPool(tasks <-chan Task, results chan<- Result, workers int) {
    var wg sync.WaitGroup
    
    for i := 0; i < workers; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for task := range tasks {
                result := process(task)
                results <- result
            }
        }()
    }
    
    wg.Wait()
    close(results)
}

// 使用
tasks := make(chan Task, 100)
results := make(chan Result, 100)

go workerPool(tasks, results, 10) // 10 个 worker

    tasks <- task
}

    fmt.Println(result)
}
```

### Channel 深入

**1. Channel 底层结构**

```go
type hchan struct {
    qcount   uint           // 当前队列中的元素数量
    dataqsiz uint           // 环形队列大小
    buf      unsafe.Pointer // 环形队列指针
    elemsize uint16         // 元素大小
    closed   uint32         // 是否关闭
    sendx    uint           // 发送索引
    recvx    uint           // 接收索引
    recvq    waitq          // 接收者队列
    sendq    waitq          // 发送者队列
    lock     mutex          // 互斥锁
```

---

**2. Channel 操作**

**发送数据**：
```go
ch <- value
```

流程：
1. 获取锁
2. 如果有等待的接收者（recvq），直接传递
3. 否则，放入环形队列（buf）
4. 队列满了，发送者进入 sendq 等待
5. 释放锁

**接收数据**：
```go
value := <-ch
```

流程：
1. 获取锁
2. 如果有等待的发送者（sendq），直接接收
3. 否则，从环形队列（buf）取出
4. 队列空了，接收者进入 recvq 等待
5. 释放锁

---

**3. Channel 关闭规则**

**黄金法则**：
- ✅ **发送方关闭 channel**
- ❌ 接收方不要关闭 channel
- ❌ 不要向已关闭的 channel 发送数据（panic）

```go
// ✅ 正确
func producer(ch chan<- int) {
    for i := 0; i < 10; i++ {
        ch <- i
    }
    close(ch) // 发送方关闭
}

func consumer(ch <-chan int) {
    for val := range ch { // range 会在 channel 关闭时退出
        fmt.Println(val)
    }
}

// ❌ 错误
func consumer(ch chan int) {
    for val := range ch {
        fmt.Println(val)
    }
    close(ch) // 接收方不应该关闭
}
```

**检查 channel 是否关闭**：

```go
val, ok := <-ch
if !ok {
    fmt.Println("channel closed")
}
```

**6. 并发模式**

**Fan-Out / Fan-In**

```go
// Fan-Out：一个生产者，多个消费者
func fanOut(input <-chan int, workers int) []<-chan int {
    outputs := make([]<-chan int, workers)
    
    for i := 0; i < workers; i++ {
        out := make(chan int)
        outputs[i] = out
        
        go func(out chan<- int) {
            defer close(out)
            for val := range input {
                out <- val * 2 // 处理
            }
        }(out)
    }
    
    return outputs
}

// Fan-In：多个生产者，一个消费者
func fanIn(inputs ...<-chan int) <-chan int {
    out := make(chan int)
    var wg sync.WaitGroup
    
    for _, input := range inputs {
        wg.Add(1)
        go func(ch <-chan int) {
            defer wg.Done()
            for val := range ch {
                out <- val
            }
        }(input)
    }
    
    go func() {
        wg.Wait()
        close(out)
    }()
    
    return out
}
```

**Pipeline**

```go
func pipeline() {
    // Stage 1: 生成数据
    gen := func() <-chan int {
        out := make(chan int)
        go func() {
            defer close(out)
            for i := 0; i < 10; i++ {
                out <- i
            }
        }()
        return out
    }

        out := make(chan int)

            for val := range in {
                out <- val * val

            for val := range in {
                if val%2 == 0 {
                    out <- val
                }
            }
        }()

    }

    }
}
```

### Channel vs Mutex

**什么时候用 Channel？**
- 数据传递、流式处理
- 多个 goroutine 协作
- 事件通知

**什么时候用 Mutex？**
- 共享状态修改
- 临界区很小
- 需要高性能

```go
// ✅ 使用 Channel（传递数据）
func process(tasks <-chan Task) {
    for task := range tasks {
        // 处理
    }
}

// ✅ 使用 Mutex（共享状态）
type Counter struct {
    mu    sync.Mutex
    count int
}

func (c *Counter) Increment() {
    c.mu.Lock()
    c.count++
    c.mu.Unlock()
}
```

**口诀**：
> **Don't communicate by sharing memory; share memory by communicating.**
> 
> 不要通过共享内存来通信，通过通信来共享内存。

### 底层实现原理

#### 1. emptyCtx（根Context）

**源码解析**：
```go
// context包内部定义
type emptyCtx int

```

**为什么需要emptyCtx？**
- 作为Context树的根节点
- 永不超时、永不取消，作为长期存在的顶层Context
- 性能优化：空实现，零开销

---

#### 2. cancelCtx（可取消Context）

**源码解析**：
```go
type cancelCtx struct {
    Context  // 内嵌父Context

    mu       sync.Mutex            // 保护以下字段
    done     chan struct{}         // 懒加载，第一次调用Done()时创建
    children map[canceler]struct{} // 所有子Context（形成取消链）
    err      error                 // 取消原因
}

func (c *cancelCtx) Done() <-chan struct{} {
    c.mu.Lock()
    if c.done == nil {
        c.done = make(chan struct{})  // 懒加载
    }
    d := c.done
    c.mu.Unlock()
    return d
}

func (c *cancelCtx) Err() error {
    c.mu.Lock()
    err := c.err
    c.mu.Unlock()
    return err
}
```

**取消机制**：
```go
func (c *cancelCtx) cancel(removeFromParent bool, err error) {
    if err == nil {
        panic("context: internal error: missing cancel error")
    }
    
    c.mu.Lock()
    if c.err != nil {
        c.mu.Unlock()
        return  // 已经被取消
    }
    c.err = err
    
    if c.done == nil {
        c.done = closedchan  // 直接使用预先关闭的channel
    } else {
        close(c.done)  // 关闭channel，通知所有等待者
    }
    
    // ⭐ 关键：递归取消所有子Context
    for child := range c.children {
        child.cancel(false, err)
    }
    c.children = nil
    c.mu.Unlock()
    
    if removeFromParent {
        removeChild(c.Context, c)  // 从父Context移除
    }
}
```

**为什么取消是递归的？**
- Context形成树状结构：父→子→孙
- 父Context取消时，所有子孙Context必须同步取消
- 通过`children map`维护子Context集合，取消时遍历通知

---

#### 3. timerCtx（带超时Context）

**源码解析**：
```go
type timerCtx struct {
    cancelCtx        // 内嵌cancelCtx
    timer *time.Timer  // 定时器
    deadline time.Time // 截止时间
}

func (c *timerCtx) Deadline() (deadline time.Time, ok bool) {
    return c.deadline, true
}

func (c *timerCtx) cancel(removeFromParent bool, err error) {
    c.cancelCtx.cancel(false, err)
    if removeFromParent {
        removeChild(c.cancelCtx.Context, c)
    }
    c.mu.Lock()
    if c.timer != nil {
        c.timer.Stop()  // 停止定时器，避免泄漏
        c.timer = nil
    }
    c.mu.Unlock()
}
```

**WithTimeout实现**：
```go
func WithTimeout(parent Context, timeout time.Duration) (Context, CancelFunc) {
    return WithDeadline(parent, time.Now().Add(timeout))
}

func WithDeadline(parent Context, d time.Time) (Context, CancelFunc) {
    // 如果父Context的截止时间更早，直接使用父Context
    if cur, ok := parent.Deadline(); ok && cur.Before(d) {
        return WithCancel(parent)
    }
    
    c := &timerCtx{
        cancelCtx: newCancelCtx(parent),
        deadline:  d,
    }
    propagateCancel(parent, c)  // 父取消时，子也取消
    
    dur := time.Until(d)
    if dur <= 0 {
        c.cancel(true, DeadlineExceeded)  // 已经超时
        return c, func() { c.cancel(false, Canceled) }
    }
    
    c.mu.Lock()
    defer c.mu.Unlock()
    if c.err == nil {
        // ⭐ 启动定时器，到时自动取消
        c.timer = time.AfterFunc(dur, func() {
            c.cancel(true, DeadlineExceeded)
        })
    }
    return c, func() { c.cancel(true, Canceled) }
}
```

**为什么超时后自动取消？**
- `time.AfterFunc`启动后台goroutine，到时调用`c.cancel`
- 自动关闭`Done()` channel，通知所有等待者
- 避免goroutine泄漏

#### 4. valueCtx（携带值Context）

**源码解析**：
```go
type valueCtx struct {
    Context          // 内嵌父Context
    key, val interface{}  // 只存一个key-value对
}

func (c *valueCtx) Value(key interface{}) interface{} {
    if c.key == key {
        return c.val  // 当前层匹配
    }
    return c.Context.Value(key)  // ⭐ 向父Context递归查找
}
```

**WithValue实现**：
```go
func WithValue(parent Context, key, val interface{}) Context {
    if key == nil {
        panic("nil key")
    }
    if !reflectlite.TypeOf(key).Comparable() {
        panic("key is not comparable")
    }
    return &valueCtx{parent, key, val}
}
```

**为什么只存一个key-value？**
- Context设计为不可变（immutable）
- 每次WithValue创建新节点，形成链表结构
- 查找Value需要O(n)遍历（n为层级深度）
- **不适合存储大量数据，只用于请求域的少量元数据**

### 为什么不要存储Context？⭐⭐⭐

这是一个**非常重要的最佳实践**，原因如下：

#### 原因1：生命周期不匹配

```go
// ❌ 错误示例
type Server struct {
    ctx context.Context  // 存储在结构体中
}

func NewServer() *Server {
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    // ⚠️ 问题：5秒后ctx自动取消，但Server可能存活更久
    return &Server{ctx: ctx}
}

func (s *Server) Handle() {
    // 可能在ctx已取消后调用，导致所有操作立即失败
    if err := s.doSomething(s.ctx); err != nil {
        // 总是返回"context canceled"
    }
}
```

**底层原因**：
- Context的生命周期通常是**请求级别**（request-scoped）
- 结构体的生命周期通常是**应用级别**（application-scoped）
- 存储后，短生命周期的Context可能已取消，但结构体仍在使用

---

#### 原因2：内存泄漏

```go
// ❌ 错误示例
type Cache struct {
    ctx context.Context
}

func NewCache(ctx context.Context) *Cache {
    return &Cache{ctx: ctx}
}

// ⚠️ 问题：如果ctx是timerCtx，内部有time.Timer
// Cache存在期间，Timer的goroutine一直运行，无法被GC
```

**底层原因**：
- `timerCtx`内部有`time.Timer`，会启动后台goroutine
- 如果Context被长期存储，Timer goroutine无法结束
- 导致goroutine泄漏和内存泄漏

**验证泄漏**：
```go
func TestContextLeak(t *testing.T) {
    before := runtime.NumGoroutine()
    
    for i := 0; i < 1000; i++ {
        ctx, cancel := context.WithTimeout(context.Background(), 1*time.Hour)
        _ = ctx
        cancel()  // ✅ 必须调用cancel释放资源
    }
    
    time.Sleep(100 * time.Millisecond)
    after := runtime.NumGoroutine()
    
    // 如果不调用cancel，after会比before多很多
    if after-before > 10 {
        t.Errorf("goroutine leak: %d -> %d", before, after)
    }
}
```

---

#### 原因3：并发安全问题

```go
// ❌ 错误示例
type Worker struct {
    ctx context.Context
    mu  sync.Mutex
}

func (w *Worker) Process() {
    // ⚠️ 问题：多个goroutine共享同一个ctx
    // 如果其中一个调用cancel，所有goroutine都会受影响
    go w.doWork(w.ctx)
    go w.doWork(w.ctx)
}
```

**底层原因**：
- Context设计为**只读**，但cancel操作是**可写**
- 多个goroutine共享同一个Context，任何一个调用cancel都会影响所有goroutine
- 存储后很难控制取消的影响范围

---

#### 原因4：违反Context设计原则

Context设计原则（来自官方文档）：
> "Context should be passed as the first parameter, not stored in structs."

**为什么这样设计？**
1. **显式传递**：让调用链清晰可见
2. **按需创建**：每个请求创建独立的Context树
3. **自动清理**：请求结束，Context自动释放
4. **避免全局状态**：Context不是全局变量

---

### ✅ 正确的使用方式

#### 方式1：作为函数参数传递

## sync同步原语

### Mutex（互斥锁）

**使用场景**：保护临界区，同一时间只有一个 goroutine 访问。

```go
type Counter struct {
    mu    sync.Mutex
    count int
}

func (c *Counter) Increment() {
    c.mu.Lock()
    c.count++
    c.mu.Unlock()
}

func (c *Counter) Get() int {
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.count
}
```

**注意**：
- Mutex 不可复制（使用指针传递）
- Lock 和 Unlock 必须成对出现
- 避免重复 Lock（死锁）

---

### RWMutex（读写锁）

**使用场景**：读多写少的场景，允许多个读者，但只允许一个写者。

```go
type Cache struct {
    mu   sync.RWMutex
    data map[string]string
}

func (c *Cache) Get(key string) (string, bool) {
    c.mu.RLock() // 读锁
    defer c.mu.RUnlock()
    val, ok := c.data[key]
    return val, ok
}

func (c *Cache) Set(key, value string) {
    c.mu.Lock() // 写锁
    defer c.mu.Unlock()
    c.data[key] = value
}
```

**性能对比**：
- 读多写少：RWMutex > Mutex
- 写多：RWMutex ≈ Mutex（因为写锁需要等待所有读锁释放）

---

### WaitGroup（等待组）

**使用场景**：等待多个 goroutine 完成。

```go
func process(urls []string) {
    var wg sync.WaitGroup
    
    for _, url := range urls {
        wg.Add(1)
        go func(url string) {
            defer wg.Done()
            fetch(url)
        }(url)
    }
    
    wg.Wait() // 等待所有 goroutine 完成
    fmt.Println("all done")
}
```

**注意**：
- Add 必须在 goroutine 启动前调用
- Done 等价于 Add(-1)
- Wait 会阻塞直到计数器归零

---

### Once（单次执行）

**使用场景**：确保某个操作只执行一次（初始化、单例）。

```go
var (
    instance *Singleton
    once     sync.Once
)

        instance = &Singleton{}
        fmt.Println("init singleton")
    })

```

**特性**：
- 线程安全
- 即使 panic 也只会执行一次（不会重试）

### Pool（对象池）

**使用场景**：复用对象，减少 GC 压力。

```go
var bufferPool = sync.Pool{
    New: func() interface{} {
        return new(bytes.Buffer)
    },
}

func process(data []byte) {
    buf := bufferPool.Get().(*bytes.Buffer)
    defer bufferPool.Put(buf)
    
    buf.Reset() // 清理
    buf.Write(data)
    // 使用 buf
}
```

**注意**：
- Pool 中的对象可能随时被 GC 回收
- 使用前必须重置对象状态
- 适合临时对象（buffer、临时结构体）

### Map（并发安全 Map）

**使用场景**：并发读写 map。

```go
var cache sync.Map

// 读取

    fmt.Println(val)
}

// 删除

}

    fmt.Println(key, value)

```

**适用场景**：
- 键值对基本固定（读多写少）
- 多个 goroutine 读写不重叠的键

**不适用场景**：
- 频繁增删键
- 需要复杂的逻辑（此时用 Mutex + map）

## 核心类型底层实现

### slice 底层结构与扩容⭐⭐⭐

#### 底层结构

```go
// runtime/slice.go
type slice struct {
    array unsafe.Pointer // 指向底层数组的指针
    len   int            // 当前长度
    cap   int            // 容量
}
```

```
slice 变量（24字节）          底层数组（连续内存）
┌─────────────────┐         ┌───┬───┬───┬───┬───┐
│ array ──────────────────→ │ 1 │ 2 │ 3 │ 0 │ 0 │
│ len = 3         │         └───┴───┴───┴───┴───┘
│ cap = 5         │
└─────────────────┘
```

**关键理解**：slice 是引用类型，但 slice 变量本身是值（包含指针+len+cap），传参时复制的是这三个字段，不是底层数组。

#### 创建方式

```go
// 1. 字面量
s := []int{1, 2, 3}

// 2. make（推荐，可以控制 len 和 cap）
s := make([]int, 3, 5) // len=3, cap=5

// 3. 从数组切片（共享底层数组！）
arr := [5]int{1, 2, 3, 4, 5}
s := arr[1:3] // s = [2, 3], len=2, cap=4（从索引1到数组末尾）
```

#### 扩容机制（Go 1.18+ 版本）

```go
// runtime/slice.go（简化）
func growslice(et *_type, old slice, cap int) slice {
    newcap := old.cap
    doublecap := newcap + newcap

    if cap > doublecap {
        newcap = cap
    } else {
        const threshold = 256
        if old.cap < threshold {
            // 小于 256：直接翻倍
            newcap = doublecap
        } else {
            // 大于等于 256：每次增长 25% + 192（平滑过渡）
            for 0 < newcap && newcap < cap {
                newcap += (newcap + 3*threshold) / 4
            }
            if newcap <= 0 {
                newcap = cap
            }
        }
    }
    // 最终 newcap 会做内存对齐（向上取整到内存分配器的 size class）
}
```

```
扩容策略总结：
┌──────────────────────────────────────────────────┐
│ 容量 < 256  →  翻倍扩容                           │
│ 容量 ≥ 256  →  每次增长约 25%（渐进式增长）         │
│ 无论哪种，最终容量都会做内存对齐                      │
└──────────────────────────────────────────────────┘

示例：
cap=4 → append → cap=8（翻倍）
cap=512 → append → cap=832 左右（≈ 512 + 512*25% + 192）
```

> **注意**：Go 1.18 之前的分界点是 1024（而非 256），增长策略是 `<1024 翻倍，≥1024 增长 25%`，面试中需要说明版本差异。

#### slice 的常见陷阱

**陷阱1：切片共享底层数组**

```go
a := []int{1, 2, 3, 4, 5}
b := a[1:3] // b = [2, 3]，与 a 共享底层数组

b[0] = 99   // 修改 b 也会影响 a！
fmt.Println(a) // [1, 99, 3, 4, 5]

// ✅ 安全做法：使用 copy
c := make([]int, len(b))
copy(c, b) // c 有独立的底层数组
```

**陷阱2：append 可能导致底层数组不共享**

```go
a := make([]int, 3, 5)  // len=3, cap=5
b := a[0:3]

b = append(b, 4) // cap 够用，a 和 b 仍共享底层数组
b = append(b, 5) // cap 够用
b = append(b, 6) // cap 不够，b 扩容，分配新数组！a 和 b 不再共享
```

**陷阱3：nil slice 与空 slice**

```go
var s1 []int       // nil slice: s1 == nil, len=0, cap=0
s2 := []int{}      // 空 slice: s2 != nil, len=0, cap=0
s3 := make([]int, 0) // 空 slice: s3 != nil, len=0, cap=0

// 行为上三者都可以 append、range，但 JSON 序列化不同：
// nil slice → null
// 空 slice → []
```

### map 底层实现⭐⭐⭐

#### 底层结构

```go
// runtime/map.go（简化）
type hmap struct {
    count     int    // 元素数量（len()返回的值）
    flags     uint8  // 状态标记（是否在写入等）
    B         uint8  // 桶的数量 = 2^B
    noverflow uint16 // 溢出桶数量
    hash0     uint32 // 哈希种子（随机化，防止哈希碰撞攻击）
    buckets    unsafe.Pointer // 桶数组（2^B个桶）
    oldbuckets unsafe.Pointer // 扩容时的旧桶数组
    nevacuate  uintptr        // 扩容进度
    // ...
}

// 每个桶最多存 8 个 key-value 对
type bmap struct {
    tophash [8]uint8   // 每个 key 哈希值的高8位（快速比较）
    // 后面紧跟 8 个 key 和 8 个 value（编译期确定布局）
    // keys   [8]keytype
    // values [8]valuetype
    // overflow *bmap     // 溢出桶指针
}
```

```
hmap
┌──────────────┐
│ count = 5    │
│ B = 2        │  → 2^2 = 4 个桶
│ buckets ─────────→ ┌────────────┐
│              │     │ bucket 0   │
│              │     │ tophash[8] │
│              │     │ keys[8]    │
│              │     │ values[8]  │
│              │     │ overflow ──────→ 溢出桶
│              │     ├────────────┤
│              │     │ bucket 1   │
│              │     ├────────────┤
│              │     │ bucket 2   │
│              │     ├────────────┤
│              │     │ bucket 3   │
│              │     └────────────┘
└──────────────┘
```

#### 查找过程

```
key = "hello"
    │
    ▼
1. 计算 hash = hash("hello", hash0)
    │
    ▼
2. 定位桶：bucket_index = hash & (2^B - 1)   // 低 B 位
    │
    ▼
3. 定位槽：tophash = hash >> (64 - 8)         // 高 8 位
    │
    ▼
4. 在桶中遍历 tophash[0..7]
   ├── tophash 匹配 → 比较完整 key → 相等则返回 value
   ├── tophash 不匹配 → 继续
   └── 遍历完 → 查溢出桶 → 都没有 → 返回零值
```

#### 扩容机制

**触发条件**：

```
条件1：负载因子 > 6.5（count / 2^B > 6.5）
→ 翻倍扩容（2^B → 2^(B+1)）

条件2：溢出桶过多（noverflow > 2^B 且 B < 15，或 noverflow > 2^15）
→ 等量扩容（桶数量不变，重新整理数据，消除稀疏的溢出桶）
```

**渐进式扩容**（关键设计）：

```
扩容不是一次性完成的（避免长时间暂停），而是在每次读写 map 时，
顺便迁移 1~2 个旧桶的数据到新桶 → 分摊扩容成本

类似 Redis 的 rehash 策略
```

#### map 的并发安全问题

```go
// ❌ map 不是并发安全的！并发读写会 panic
m := map[string]int{}
go func() { m["a"] = 1 }()
go func() { _ = m["a"] }()
// fatal error: concurrent map read and map write

// ✅ 方案1：sync.Mutex
// ✅ 方案2：sync.RWMutex（读多写少）
// ✅ 方案3：sync.Map（特定场景）
```

> **面试陷阱**：Go 的 map 在运行时会检测并发写入（通过 `flags` 标记位），一旦检测到直接 `throw("concurrent map writes")`，这是 **fatal error 不可恢复**，不是 panic。

### interface 底层实现⭐⭐⭐

#### 两种 interface 结构

Go 的 interface 在底层有两种结构：

```go
// 1. eface —— 空接口 interface{}
type eface struct {
    _type *_type          // 类型信息
    data  unsafe.Pointer  // 数据指针
}

// 2. iface —— 非空接口（有方法的接口）
type iface struct {
    tab  *itab            // 接口表（类型+方法集）
    data unsafe.Pointer   // 数据指针
}

type itab struct {
    inter *interfacetype  // 接口的类型信息
    _type *_type          // 实际类型信息
    hash  uint32          // 类型哈希（用于类型断言快速比较）
    _     [4]byte
    fun   [1]uintptr      // 方法地址表（变长数组）
}
```

```
var w io.Writer = os.Stdout

iface
┌──────────────────┐
│ tab ──→ itab     │
│         ┌────────────────────┐
│         │ inter: *Writer     │  接口类型
│         │ _type: *os.File    │  实际类型
│         │ fun: [Write]       │  方法地址表
│         └────────────────────┘
│ data ──→ os.Stdout 实例       │
└──────────────────┘
```

#### 接口的 nil 判断陷阱

```go
// ⚠️ 经典陷阱
type MyError struct{}
func (e *MyError) Error() string { return "error" }

func getError() error {
    var err *MyError = nil
    return err  // 返回的 interface 不是 nil！
}

func main() {
    err := getError()
    fmt.Println(err == nil) // false！

    // 原因：err 是 iface{tab: *itab(MyError), data: nil}
    // tab 不为 nil，所以 err != nil
    // 即使 data 是 nil，只要 tab 有值，interface 就不等于 nil
}

// ✅ 正确做法：直接返回 nil，而不是返回带类型的 nil 指针
func getError() error {
    var err *MyError = nil
    if err == nil {
        return nil  // 返回真正的 nil interface
    }
    return err
}
```

```go
// 类型断言（O(1)，通过 itab.hash 快速比较）
```

### string 与 []byte⭐⭐

#### string 底层结构

```go
// runtime/string.go
type stringStruct struct {
    str unsafe.Pointer // 指向底层字节数组（不可变）
    len int            // 字节长度（不是字符数！）
}
```

**string 的不可变性**：

```go
s := "hello"
// s[0] = 'H'  // 编译错误！string 是不可变的

// 修改 string 必须转换为 []byte
b := []byte(s) // ⚠️ 会发生内存拷贝
b[0] = 'H'
```

#### string 与 []byte 的零拷贝转换

```go
// Go 1.22+ 推荐使用 unsafe.String / unsafe.StringData
// 或者用 unsafe.Pointer 手动转换（需谨慎）

// ⚠️ 高性能场景的零拷贝（慎用，破坏了 string 不可变性的约定）
func stringToBytes(s string) []byte {
    return unsafe.Slice(unsafe.StringData(s), len(s))
}

func bytesToString(b []byte) string {
    return unsafe.String(&b[0], len(b))
}
```

**编译器优化**：以下场景编译器会自动避免 `string ↔ []byte` 的拷贝：
- `map` 查找：`m[string(b)]` 不会拷贝
- 字符串拼接：`"prefix" + string(b)` 可能优化
- `for range []byte(s)` 不会拷贝

## defer机制

### defer 执行顺序⭐⭐⭐

**核心规则**：LIFO（后进先出），函数返回前按注册的逆序执行。

```go
func main() {
    fmt.Println("start")
    defer fmt.Println("first")
    defer fmt.Println("second")
    defer fmt.Println("third")
    fmt.Println("end")
}
// 输出：
// start
// end
// third
// second
// first
```

### defer 与 return 的执行顺序⭐⭐⭐

**关键理解**：`return xxx` 不是原子操作，实际分三步：

```
1. 返回值赋值：result = xxx
2. 执行 defer 函数
3. RET 指令（真正返回）
```

```go
// 命名返回值 + defer 可以修改返回值
func f() (result int) {
    defer func() {
        result++  // 在 return 之后、RET 之前执行
    }()
    return 0
}
// 返回 1（不是 0！）
// 执行过程：result = 0 → defer: result++ → RET(result=1)

// 匿名返回值 + defer 不能修改返回值
func g() int {
    result := 0
    defer func() {
        result++  // 修改的是局部变量，不是返回值
    }()
    return result
}
// 返回 0
// 执行过程：_retval = result(0) → defer: result++ → RET(_retval=0)
```

### defer 参数预计算⭐⭐

```go
func main() {
    x := 1
    defer fmt.Println(x) // x 在 defer 注册时就被求值为 1
    x = 2
}
// 输出 1（不是 2）

// 如果要捕获最新值，用闭包
func main() {
    x := 1
    defer func() {
        fmt.Println(x) // 闭包引用，取 defer 执行时的值
    }()
    x = 2
}
// 输出 2
```

### defer 的底层实现演进

```
Go 1.12 及之前：堆上分配（heap-allocated defer）
- 每个 defer 在堆上分配一个 _defer 结构体，链到 goroutine 的 defer 链表
- 性能较差：涉及堆分配和 GC

Go 1.13：编译器优化（open-coded defer）
- 对于确定不会在循环中的 defer，编译器直接在栈上分配
- 性能提升约 30%

Go 1.14+：内联 defer
- 编译器将 defer 直接内联为函数末尾的代码调用
- 性能接近直接调用，几乎零开销
- 但循环中的 defer 仍然使用堆分配

性能对比：
直接调用：~0.5ns
Go 1.14 内联 defer：~1ns
Go 1.12 堆 defer：~35ns
```

### defer 在 panic/recover 中的作用

```go
func safeFunc() {
    defer func() {
        if r := recover(); r != nil {
            fmt.Println("recovered:", r)
        }
    }()

    panic("something went wrong")
}

// recover 只能在 defer 函数中调用才有效
// 直接调用 recover()（不在 defer 中）永远返回 nil
```

> **重要**：`recover()` 只能捕获当前 goroutine 的 panic。子 goroutine 的 panic 无法被父 goroutine 的 recover 捕获，会导致整个进程崩溃。

## 错误处理

### Go 的错误哲学⭐⭐

Go 选择显式错误处理（而非异常机制）的原因：
1. **错误是值**：`error` 是一个接口，可以携带丰富的上下文信息
2. **显式处理**：调用者必须决定如何处理错误，不会被隐藏
3. **没有隐藏的控制流**：不像 try/catch 那样可能跳过中间代码

```go
type error interface {
    Error() string
}
```

### 错误包装与链式错误（Go 1.13+）⭐⭐⭐

```go
import (
    "errors"
    "fmt"
)

// 包装错误（添加上下文信息）
func readConfig(path string) error {
    data, err := os.ReadFile(path)
    if err != nil {
        return fmt.Errorf("read config %s: %w", path, err) // %w 包装错误
    }
    // ...
    return nil
}

// errors.Is —— 判断错误链中是否包含特定错误
if errors.Is(err, os.ErrNotExist) {
    fmt.Println("file not found")
}

// errors.As —— 从错误链中提取特定类型的错误
var pathErr *os.PathError
if errors.As(err, &pathErr) {
    fmt.Println("path:", pathErr.Path)
}
```

**errors.Is vs == 的区别**：

```go
err := fmt.Errorf("outer: %w", os.ErrNotExist)

err == os.ErrNotExist        // false（err 是包装后的新错误）
errors.Is(err, os.ErrNotExist) // true（递归解包检查整条错误链）
```

### 自定义错误类型

```go
// 业务错误码
type BizError struct {
    Code    int
    Message string
    Cause   error
}

func (e *BizError) Error() string {
    if e.Cause != nil {
        return fmt.Sprintf("[%d] %s: %v", e.Code, e.Message, e.Cause)
    }
    return fmt.Sprintf("[%d] %s", e.Code, e.Message)
}

func (e *BizError) Unwrap() error {
    return e.Cause // 支持 errors.Is / errors.As 解包
}

// 使用
var ErrUserNotFound = &BizError{Code: 404, Message: "user not found"}

func getUser(id int) (*User, error) {
    user, err := db.Find(id)
    if err != nil {
        return nil, &BizError{
            Code:    500,
            Message: "query user failed",
            Cause:   err,
        }
    }
    if user == nil {
        return nil, ErrUserNotFound
    }
    return user, nil
}
```

### panic 与 recover 的使用原则

```
何时使用 panic：
✅ 不可恢复的程序错误（不该出现的状态）
✅ 初始化失败（配置文件不存在、端口被占用）
❌ 业务逻辑错误（用 error 返回）
❌ 外部输入验证失败（用 error 返回）

何时使用 recover：
✅ HTTP 框架的兜底中间件（防止一个请求 panic 导致服务崩溃）
✅ goroutine 的顶层保护（防止子任务 panic 影响主流程）
❌ 不要用 panic/recover 模拟 try/catch
```

## 内存管理与GC

### 内存分配

**逃逸分析（Escape Analysis）**

编译器决定变量分配在栈还是堆：
- **栈**：函数内部，生命周期短，自动回收
- **堆**：逃逸到函数外，需要 GC 回收

**逃逸场景**：
1. 返回指针
2. 闭包引用
3. 接口类型
4. 切片/map 容量过大

```go
// 逃逸分析
func escape() *int {
    x := 42
    return &x // x 逃逸到堆
}

func noEscape() int {
    x := 42
    return x // x 分配在栈
}

// 查看逃逸分析
// go build -gcflags="-m" main.go
```

**优化建议**：
- 避免不必要的指针返回
- 使用值传递（小对象）
- 复用对象（sync.Pool）

---

### GC 机制

**三色标记法**（Tri-color Marking）

**三种颜色**：
- **白色**：未标记，GC 候选对象
- **灰色**：已标记，但其引用的对象未完全标记
- **黑色**：已标记，其引用的对象也已标记

**流程**：
1. **标记准备**：所有对象标记为白色，根对象（全局变量、栈上变量）标记为灰色
2. **并发标记**：
   - 从灰色对象开始，标记其引用的对象为灰色
   - 标记完成后，自己变为黑色
   - 重复直到没有灰色对象
3. **清除**：回收白色对象

**并发问题**：标记过程中，程序继续运行，可能出现：
- **黑色对象引用白色对象**（应该标记但未标记，会被误回收）

**解决方案：写屏障**（Write Barrier）
- 在黑色对象引用白色对象时，将白色对象标记为灰色

---

**GC 触发条件**：
1. **主动触发**：`runtime.GC()`
2. **内存增长**：堆内存增长超过上次 GC 后的 100%（GOGC=100）
3. **定时触发**：2 分钟未 GC

```go
func TimeoutMiddleware(timeout time.Duration) gin.HandlerFunc {
    return func(c *gin.Context) {

            close(finished)
```

| 特性 | Goroutine | 线程 |
|------|-----------|------|
| 创建成本 | 2KB 初始栈 | 1-2MB 栈 |
| 调度 | 用户态（GMP） | 内核态（OS） |
| 切换成本 | 纳秒级 | 微秒级 |
| 数量 | 百万级 | 千级 |
| 调度器 | 协作式抢占 | 抢占式 |

---

**2. Channel 和 Mutex 如何选择？**

**使用 Channel**：
- 数据传递、流式处理
- 多个 goroutine 协作
- 事件通知

**使用 Mutex**：
- 共享状态修改
- 临界区很小
- 需要高性能

---

**3. 如何避免 Goroutine 泄漏？**

1. 使用 context 超时/取消
2. 使用 done channel 通知退出
3. 确保 channel 能正常关闭
4. 定期检查 `runtime.NumGoroutine()`

---

**4. GC 如何优化？**

1. 减少堆分配（使用 sync.Pool、对象复用）
2. 避免不必要的指针（减少扫描）
3. 调整 GOGC 参数
4. 使用 pprof 定位问题

---

**5. Context 如何传递？**

1. 作为函数第一个参数
2. 不要存储在结构体中
3. 超时预算层层传递
4. 取消信号会向下传播

```go
type ConfigManager struct {
    mu      sync.RWMutex
    configs map[string]*ActivityConfig
    version int64
}

func (cm *ConfigManager) LoadConfig(ctx context.Context) error {
    // 从数据库加载最新配置
    configs, version, err := db.FetchConfigs(ctx)
    if err != nil {
        return err
    }
    
    cm.mu.Lock()
    cm.configs = configs
    cm.version = version
    cm.mu.Unlock()
    
    log.Printf("config loaded, version=%d", version)
    return nil
}

func (cm *ConfigManager) GetConfig(activityID string) (*ActivityConfig, bool) {
    cm.mu.RLock()
    defer cm.mu.RUnlock()
    config, ok := cm.configs[activityID]
    return config, ok
}

// 定时热更新
func (cm *ConfigManager) Watch(ctx context.Context) {
    ticker := time.NewTicker(10 * time.Second)
    defer ticker.Stop()
    
    for {
        select {
        case <-ticker.C:
            if err := cm.LoadConfig(ctx); err != nil {
                log.Printf("load config failed: %v", err)
            }
        case <-ctx.Done():
            log.Println("config watcher stopped")
            return
        }
    }
}
```

```go
import "github.com/go-redis/redis/v8"

type RedisLock struct {
    client *redis.Client
    key    string
    value  string
    ttl    time.Duration
}

func NewRedisLock(client *redis.Client, key string, ttl time.Duration) *RedisLock {
    return &RedisLock{
        client: client,
        key:    key,
        value:  uuid.New().String(),
        ttl:    ttl,
    }
}

func (l *RedisLock) Lock(ctx context.Context) error {
    // SET key value NX EX ttl
    ok, err := l.client.SetNX(ctx, l.key, l.value, l.ttl).Result()
    if err != nil {
        return err
    }
    if !ok {
        return errors.New("lock failed")
    }
    return nil
}

func (l *RedisLock) Unlock(ctx context.Context) error {
    // Lua 脚本：只删除自己持有的锁
    script := `
        if redis.call("get", KEYS[1]) == ARGV[1] then
            return redis.call("del", KEYS[1])
        else
            return 0
        end
    `
    return l.client.Eval(ctx, script, []string{l.key}, l.value).Err()
}

// 使用
func deductBalance(ctx context.Context, userID string, amount int) error {
    lock := NewRedisLock(redisClient, fmt.Sprintf("lock:user:%s", userID), 5*time.Second)
    
    if err := lock.Lock(ctx); err != nil {
        return err
    }
    defer lock.Unlock(ctx)
    
    // 查询余额
    balance, err := db.GetBalance(ctx, userID)
    if err != nil {
        return err
    }
    
    // 扣款
    if balance < amount {
        return errors.New("insufficient balance")
    }
    
    return db.UpdateBalance(ctx, userID, balance-amount)
}
```

## 📝 面试题自查

> 以下问题可作为学习本文后的自查清单，以资深面试官视角设计。每题附有简要回答，详细原理请回到文章对应章节深入学习。

---

**Q1: 请完整描述 GMP 模型中 G、M、P 各自的角色，以及为什么需要引入 P？**

G 是 Goroutine（用户态协程），代表一个待执行的任务，保存了函数入口、参数和栈；M 是 Machine（OS线程），是真正执行代码的载体；P 是 Processor（逻辑处理器），持有本地运行队列和内存缓存 mcache，M 必须绑定 P 才能执行 G。引入 P 的核心原因是：如果没有 P，所有 M 共享一个全局队列需要加锁，高并发下锁竞争严重。P 提供了本地无锁队列（容量256），大部分情况下 M 从自己的 P 本地队列取 G，只在本地队列空时才访问全局队列或偷取其他 P 的任务，大幅减少了锁竞争。

---

**Q2: Go 1.14 之前和之后的抢占调度有什么区别？为什么选用 SIGURG 信号？**

Go 1.14 之前是**协作式抢占**，依赖编译器在函数入口插入栈检查代码，只有函数调用时才有机会被抢占。缺陷是纯计算的死循环（没有函数调用）无法被抢占，会导致其他 goroutine 饿死。Go 1.14+ 引入**基于信号的异步抢占**：Sysmon 发现 G 运行超过 10ms 后，向 G 所在的 M 发送 SIGURG 信号，M 收到信号后保存 G 的寄存器状态，将 G 设为 `_Grunnable` 放回队列，然后执行下一个 G。选用 SIGURG 是因为该信号几乎没有其他用途（原本用于 TCP 带外数据），不会与用户程序的信号处理冲突。

---

**Q3: Work Stealing 机制是如何工作的？为什么偷一半而不是偷一个？**

当 P 的本地队列为空时，M 会随机选择另一个 P，偷取其本地队列中一半的 G 到自己的队列。偷一半而非一个的原因是：偷取操作本身有成本（需要 CAS 原子操作），偷一半可以减少偷取次数，让窃取者有足够的任务执行一段时间，避免频繁偷取带来的开销。随机选择目标 P 则避免了总偷同一个 P 造成的热点竞争。

---

**Q4: slice 的底层结构是什么？append 操作可能导致什么问题？**

slice 底层是一个包含三个字段的结构体：`array`（指向底层数组的指针）、`len`（当前长度）、`cap`（容量）。append 操作的关键问题：1）当 cap 不够时会分配新的底层数组，导致原来共享同一底层数组的多个 slice 不再共享，修改互不影响；2）当 cap 够用时直接在原数组上追加，多个共享底层数组的 slice 之间可能互相覆盖数据。Go 1.18+ 的扩容策略：容量 <256 时翻倍，≥256 时每次增长约 25%+192，最终做内存对齐。

---

**Q5: map 为什么不是并发安全的？并发写 map 会发生什么？**

Go 的 map 底层是哈希表（hmap），出于性能考虑没有内置锁。并发写 map 时，运行时通过 `flags` 标记位检测到并发写入会直接 `throw("concurrent map writes")`，这是一个 **fatal error 而非 panic**，无法被 recover 捕获，会导致整个进程崩溃。解决方案有三种：`sync.Mutex`（通用）、`sync.RWMutex`（读多写少）、`sync.Map`（键值基本固定或多 goroutine 读写不重叠的键）。

---

**Q6: interface 的底层实现 eface 和 iface 有什么区别？interface 的 nil 判断有什么陷阱？**

空接口 `interface{}` 底层是 `eface`，包含 `_type`（类型信息）和 `data`（数据指针）两个字段。非空接口底层是 `iface`，包含 `tab`（指向 itab，存储接口类型+实际类型+方法地址表）和 `data`。nil 陷阱：当一个具体类型的 nil 指针赋值给 interface 变量时，interface 的 `tab`/`_type` 字段不为 nil（记录了类型信息），只有 `data` 为 nil，因此 `interface != nil`。正确做法是直接返回 `nil` 而非带类型的 nil 指针。

---

**Q7: channel 的底层数据结构是什么？发送和接收数据的完整流程是怎样的？**

channel 底层是 `hchan` 结构体，核心字段：环形缓冲区（`buf`）、发送/接收索引（`sendx`/`recvx`）、等待队列（`sendq`/`recvq`，双向链表存储阻塞的 goroutine）、互斥锁（`lock`）。发送流程：加锁 → 如果 recvq 有等待的接收者，直接将数据拷贝给它（唤醒该 goroutine）→ 否则放入环形缓冲区 → 缓冲区满则将当前 goroutine 封装为 sudog 挂入 sendq 等待 → 解锁。接收流程类似但方向相反。

---

**Q8: context 的底层有几种实现？WithValue 查找的时间复杂度是多少？为什么不要把 context 存储在 struct 中？**

context 有四种底层实现：`emptyCtx`（根节点，Background/TODO）、`cancelCtx`（可取消）、`timerCtx`（带超时）、`valueCtx`（携带值）。WithValue 查找是 O(n)，因为 valueCtx 每次只存一个 key-value 对，查找时沿着 Context 链（子→父）逐层比较。不要存储 context 的原因：1）生命周期不匹配（context 是请求级，struct 可能是应用级）；2）timerCtx 内部有 timer goroutine，长期存储导致泄漏；3）多 goroutine 共享同一 context，任何一方 cancel 影响所有方。

---

**Q9: defer 的执行顺序是什么？defer 与 return 的关系是怎样的？能否用 defer 修改函数返回值？**

defer 按 LIFO（后进先出）顺序执行。`return xxx` 实际分三步：① 返回值赋值 `result = xxx` → ② 执行 defer 函数 → ③ RET 指令。因此命名返回值 + defer 可以修改返回值（defer 中修改的是已命名的返回值变量，在 RET 之前生效），而匿名返回值 + defer 不能（defer 修改的是局部变量，返回值已经被拷贝到返回寄存器了）。另外 defer 的参数在注册时就已求值，如需捕获最新值应使用闭包。

---

**Q10: Go 的 GC 使用什么算法？什么是写屏障？GC 的触发条件有哪些？**

Go 使用**三色标记法 + 写屏障**的并发 GC。三色标记：白色（未标记，待回收）、灰色（已标记但引用未处理完）、黑色（已标记且引用处理完）。并发标记期间程序继续运行，可能出现黑色对象新引用白色对象（被误回收）的问题，写屏障在赋值时拦截这种情况，将白色对象标记为灰色。触发条件：① 堆内存增长超过 GOGC 阈值（默认100%，即翻倍）；② 2分钟未 GC（Sysmon 强制触发）；③ 手动调用 `runtime.GC()`。

---

**Q11: 什么是逃逸分析？有哪些常见的逃逸场景？如何查看变量是否逃逸？**

逃逸分析是编译器在编译期决定变量分配在栈还是堆上的静态分析。分配在栈上不需要 GC，分配在堆上需要 GC 回收。常见逃逸场景：① 返回局部变量的指针；② 闭包引用外部变量；③ 赋值给接口类型（interface）；④ 切片/map 容量过大或动态大小。查看方式：`go build -gcflags="-m"` 会输出逃逸分析结果。优化建议：避免不必要的指针返回、小对象用值传递、使用 sync.Pool 复用。

---

**Q12: sync.Pool 的作用是什么？对象什么时候会被回收？适合什么场景？**

sync.Pool 是对象复用池，用于缓存临时对象，减少频繁的堆分配和 GC 压力。核心方法：`Get()` 获取对象（池空则调用 `New` 创建）、`Put()` 归还对象。Pool 中的对象不保证持久存在——**每次 GC 时，Pool 中的对象可能被全部清除**。因此 Pool 只适合缓存**临时、无状态**的对象（如 bytes.Buffer、临时结构体），使用前必须重置对象状态。不适合缓存需要持久化的连接、文件句柄等资源。

---

**Q13: Go 协程和 Python 协程有什么本质区别？什么是"函数颜色"问题？**

核心区别有三：① **调度方式**：Go 是抢占式（运行时自动调度），Python asyncio 是协作式（必须遇到 await 才让出）；② **并行能力**：Go goroutine 可真正并行（多 M 绑定多核 CPU），Python 受 GIL 限制只能并发不能并行；③ **栈模型**：Go 是有栈协程（每个 G 独立栈，可在任意位置挂起），Python 是无栈协程（状态机，只能在 await 点挂起）。"函数颜色"问题是 Python 特有的：如果底层函数改成 async，整个调用链上的所有函数都要标记为 async（传染性），Go 没有这个问题——所有函数都是"同一颜色"。

---

**Q14: 如何发现和定位 Goroutine 泄漏？有哪些常见原因？**

定位方法：① `runtime.NumGoroutine()` 监控 goroutine 数量变化；② `pprof.Lookup("goroutine").WriteTo(os.Stdout, 1)` 导出堆栈；③ HTTP pprof 端点 `GET /debug/pprof/goroutine?debug=1` 查看详细堆栈。常见原因：① channel 永久阻塞（发送/接收无对端）；② 无限循环缺少退出条件；③ context 未取消导致 goroutine 一直等待；④ 第三方库内部 goroutine 未正确关闭。修复手段：用 context 超时/取消控制 goroutine 生命周期，用 done channel 通知退出，确保 channel 在不需要时能被关闭。

---

**Q15: errors.Is 和 errors.As 有什么区别？为什么 Go 1.13 之后推荐用 %w 包装错误？**

`errors.Is(err, target)` 沿着错误链递归解包，判断链中是否存在与 target **值相等**的错误，替代了 `==` 比较（`==` 只能比较最外层，无法穿透 wrap 层）。`errors.As(err, &target)` 沿着错误链递归解包，提取链中第一个能**类型断言**为 target 类型的错误。`%w` 包装错误的意义：保留原始错误信息的同时添加上下文（如"read config /etc/app.conf: file not found"），调用方可以通过 errors.Is/As 检查根因，又不失去上层的描述信息。这比 Go 1.13 之前用 `%v` 丢失错误链要好得多。

---

**Q16: GOMAXPROCS 控制的是什么？设置过大或过小有什么影响？在容器环境下有什么注意事项？**

GOMAXPROCS 控制 P 的数量，即最多有多少个 Goroutine 可以**同时并行**执行（≠ M 的数量，≠ G 的数量）。设置过小：CPU 利用率不足，吞吐量受限。设置过大：增加调度开销和上下文切换次数，对 CPU 密集型任务反而降低性能。默认值等于 `runtime.NumCPU()`。容器环境注意：Go 默认读到的是**宿主机**的 CPU 核心数而非 cgroup 限制的核心数，可能导致 GOMAXPROCS 远大于容器实际可用核心，建议使用 `uber-go/automaxprocs` 库自动适配容器的 CPU 限制。

---

**Q17: select 语句的执行机制是什么？所有 case 都不满足时会怎样？nil channel 在 select 中有什么作用？**

select 用于同时监听多个 channel 操作。执行规则：① 如果多个 case 同时满足，**随机**选择一个执行（不是按顺序）；② 如果没有 case 满足且有 default，执行 default；③ 没有 default 则阻塞等待。nil channel 的特殊作用：向 nil channel 发送或接收永远阻塞，因此将 channel 变量设为 nil 可以**动态禁用 select 的某个 case 分支**，这是一个常见的编程技巧，比如收到停止信号后设 tasks channel 为 nil 来停止接收新任务但继续处理其他分支。

---

**Q18: sync.Once 的实现原理是什么？如果 Once.Do 中的函数 panic 了，再次调用 Do 会重新执行吗？**

sync.Once 底层使用 `done uint32` 原子标记 + `Mutex` 互斥锁实现。第一次调用 `Do(f)` 时获取锁、执行 f、将 done 标记为 1；后续调用先用 `atomic.LoadUint32` 检查 done 标记，为 1 直接返回（快速路径，无锁）。如果 f 执行过程中 panic 了，done **仍然被标记为已完成**，后续再调用 Do **不会重新执行** f。这意味着如果初始化函数可能 panic，需要在函数内部自行处理恢复逻辑，或使用其他机制保证初始化成功。

---

**Q19: Gin 框架的中间件机制是怎样的？c.Next() 和 c.Abort() 分别做了什么？**

Gin 的中间件采用**洋葱模型**（类似 Koa）：请求依次经过每个中间件的前半段 → 到达处理函数 → 再依次经过每个中间件的后半段（逆序）。`c.Next()` 的作用是将控制权交给下一个中间件/处理函数，当下游全部执行完后再继续执行 `c.Next()` 之后的代码。`c.Abort()` 则终止后续中间件的执行（设置 index 为 abortIndex），但不会中断当前中间件函数自身的后续代码。两者组合使用可以实现认证拦截、超时控制、统一错误处理等模式。

---

**Q20: Go 的 pprof 工具能分析哪些维度的性能？如何在线上环境安全地使用？**

pprof 支持多个维度：① **CPU Profile**（函数 CPU 耗时占比）；② **Heap Profile**（堆内存分配热点）；③ **Goroutine Profile**（goroutine 数量和堆栈）；④ **Block Profile**（channel/锁阻塞时间）；⑤ **Mutex Profile**（锁竞争耗时）。线上安全使用方式：通过 `import _ "net/http/pprof"` 注册 HTTP 端点，但要**限制访问**（绑定内部端口、加认证中间件，不要暴露到公网）。采集时指定 `seconds=30` 限制采样时间，避免长时间 profiling 影响性能。火焰图（`go tool pprof -http=:8080`）是最直观的分析方式。
