## 1. 消息队列基础

### 1.1 为什么需要消息队列？

**核心作用**：

| 作用 | 说明 | 示例 |
|------|------|------|
| **解耦** | 生产者和消费者独立 | 订单服务 → MQ → 库存服务 |
| **异步** | 非核心流程异步处理 | 注册成功 → MQ → 发邮件 |
| **削峰** | 缓冲流量高峰 | 秒杀 → MQ → 慢慢处理 |
| **广播** | 一条消息多个消费者 | 订单创建 → 多个下游服务 |

**示例：同步 vs 异步**
```
同步（串行）：
用户注册 → 写数据库(50ms) → 发邮件(200ms) → 发短信(150ms) = 400ms

异步（MQ）：
用户注册 → 写数据库(50ms) → 发送MQ(5ms) = 55ms
                              ↓
                          后台异步处理（发邮件、发短信）
```

### 1.2 核心概念

| 概念 | 说明 |
|------|------|
| **Producer（生产者）** | 发送消息 |
| **Consumer（消费者）** | 接收消息 |
| **Broker** | 消息服务器 |
| **Topic** | 消息主题/队列 |
| **Partition** | 分区（Kafka） |
| **Consumer Group** | 消费者组 |
| **Offset** | 消费位置 |

### 1.3 消息模型

**点对点（Queue）**：
```
[Producer] → [Queue] → [Consumer]
```
- 一条消息只被一个消费者消费
- 消费后消息删除

**发布订阅（Topic）**：
```
[Producer] → [Topic] → [Consumer A]
                    → [Consumer B]
                    → [Consumer C]
```
- 一条消息可被多个消费者消费
- 消息不删除（保留一段时间）

## 2. Kafka详解

### 2.1 架构

```
[Producer] → [Broker1] → [Consumer Group A]
          → [Broker2] → [Consumer Group B]
          → [Broker3]
          
ZooKeeper: 元数据管理、选举
```

**核心组件**：
- **Producer**：生产者
- **Broker**：Kafka服务器（多个组成集群）
- **Topic**：主题（逻辑概念）
- **Partition**：分区（物理存储）
- **Consumer Group**：消费者组

### 2.2 Topic与Partition

**Topic分区**：
```
Topic: orders
├─ Partition 0: [msg0, msg3, msg6, ...]
├─ Partition 1: [msg1, msg4, msg7, ...]
└─ Partition 2: [msg2, msg5, msg8, ...]
```

**分区策略**：
1. **轮询**：依次分配到各分区
2. **Key Hash**：相同key的消息进入同一分区（保证顺序）
3. **自定义**：指定分区

**副本机制**：
```
Partition 0
├─ Leader (Broker1)  ← 读写
├─ Follower (Broker2)  ← 同步
└─ Follower (Broker3)  ← 同步
```

### 2.3 生产者

**Go示例**：
```go
import "github.com/segmentio/kafka-go"

func main() {
    writer := kafka.NewWriter(kafka.WriterConfig{
        Brokers:  []string{"localhost:9092"},
        Topic:    "orders",
        Balancer: &kafka.Hash{},  // 按key分区
    })
    defer writer.Close()
    
    // 发送消息
    err := writer.WriteMessages(context.Background(),
        kafka.Message{
            Key:   []byte("order_1001"),
            Value: []byte(`{"id":1001,"amount":100}`),
        },
    )
    if err != nil {
        log.Fatal(err)
    }
}
```

**Java示例**：
```java
import org.apache.kafka.clients.producer.*;

Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("key.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("value.serializer", "org.apache.kafka.common.serialization.StringSerializer");
props.put("acks", "all");  // 等待所有副本确认

Producer<String, String> producer = new KafkaProducer<>(props);

ProducerRecord<String, String> record = new ProducerRecord<>(
    "orders", 
    "order_1001", 
    "{\"id\":1001,\"amount\":100}"
);

// 同步发送
producer.send(record).get();

// 异步发送
producer.send(record, (metadata, exception) -> {
    if (exception == null) {
        System.out.println("发送成功: " + metadata.offset());
    } else {
        exception.printStackTrace();
    }
});

producer.close();
```

### 2.4 消费者

**Go示例**：
```go
func main() {
    reader := kafka.NewReader(kafka.ReaderConfig{
        Brokers: []string{"localhost:9092"},
        Topic:   "orders",
        GroupID: "order-service",  // 消费者组
        MinBytes: 10e3,  // 10KB
        MaxBytes: 10e6,  // 10MB
    })
    defer reader.Close()
    
    for {
        msg, err := reader.ReadMessage(context.Background())
        if err != nil {
            log.Fatal(err)
        }
        
        fmt.Printf("Partition: %d, Offset: %d, Key: %s, Value: %s\n",
            msg.Partition, msg.Offset, string(msg.Key), string(msg.Value))
        
        // 处理消息
        processOrder(msg.Value)
    }
}
```

**Java示例**：
```java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("group.id", "order-service");
props.put("key.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");
props.put("value.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");
props.put("enable.auto.commit", "false");  // 手动提交offset

KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(Arrays.asList("orders"));

while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    
    for (ConsumerRecord<String, String> record : records) {
        System.out.printf("Partition: %d, Offset: %d, Key: %s, Value: %s\n",
            record.partition(), record.offset(), record.key(), record.value());
        
        // 处理消息
        processOrder(record.value());
    }
    
    // 手动提交offset
    consumer.commitSync();
}
```

### 2.5 Consumer Group

**负载均衡**：
```
Topic: orders (3个分区)

Consumer Group A:
├─ Consumer 1 → Partition 0, 1
└─ Consumer 2 → Partition 2

Consumer Group B:
├─ Consumer 1 → Partition 0
├─ Consumer 2 → Partition 1
└─ Consumer 3 → Partition 2
```

**Rebalance（重平衡）**：
- 消费者加入/退出
- 分区数量变化
- 触发时短暂停止消费

### 2.6 Offset管理

**三种提交方式**：

**1. 自动提交**（不推荐）：
```java
props.put("enable.auto.commit", "true");
props.put("auto.commit.interval.ms", "1000");
```
- 问题：可能丢失消息（消费前提交）

**2. 手动提交（同步）**：
```java
consumer.commitSync();  // 阻塞直到提交成功
```

**3. 手动提交（异步）**：
```java
consumer.commitAsync((offsets, exception) -> {
    if (exception != null) {
        log.error("提交失败", exception);
    }
});
```

**指定offset消费**：
```java
// 从头开始
consumer.seekToBeginning(partitions);

// 从末尾开始
consumer.seekToEnd(partitions);

// 指定offset
consumer.seek(partition, offset);
```

## 3. RabbitMQ详解

### 3.1 架构

```
[Producer] → [Exchange] → [Queue] → [Consumer]
```

**核心组件**：
- **Producer**：生产者
- **Exchange**：交换机（路由）
- **Queue**：队列
- **Binding**：绑定（Exchange → Queue）
- **Consumer**：消费者

### 3.2 Exchange类型

**1. Direct（直连）**：
```
Exchange (direct)
├─ routing_key: "order.create" → Queue A
└─ routing_key: "order.pay"    → Queue B
```

**2. Fanout（广播）**：
```
Exchange (fanout)
├─ Queue A
├─ Queue B
└─ Queue C
```

**3. Topic（主题）**：
```
Exchange (topic)
├─ routing_key: "order.*"      → Queue A  (order.create, order.pay)
├─ routing_key: "order.create" → Queue B
└─ routing_key: "#"            → Queue C  (所有消息)
```

**4. Headers（头部匹配）**：根据消息头属性路由

### 3.3 生产者

**Go示例**：
```go
import "github.com/streadway/amqp"

func main() {
    conn, _ := amqp.Dial("amqp://guest:guest@localhost:5672/")
    defer conn.Close()
    
    ch, _ := conn.Channel()
    defer ch.Close()
    
    // 声明Exchange
    ch.ExchangeDeclare(
        "orders",  // name
        "direct",  // type
        true,      // durable
        false,     // auto-deleted
        false,     // internal
        false,     // no-wait
        nil,       // arguments
    )
    
    // 发送消息
    ch.Publish(
        "orders",        // exchange
        "order.create",  // routing key
        false,           // mandatory
        false,           // immediate
        amqp.Publishing{
            ContentType: "application/json",
            Body:        []byte(`{"id":1001}`),
        },
    )
}
```

### 3.4 消费者

```go
func main() {
    conn, _ := amqp.Dial("amqp://guest:guest@localhost:5672/")
    defer conn.Close()
    
    ch, _ := conn.Channel()
    defer ch.Close()
    
    // 声明队列
    q, _ := ch.QueueDeclare(
        "order_queue",  // name
        true,           // durable
        false,          // delete when unused
        false,          // exclusive
        false,          // no-wait
        nil,            // arguments
    )
    
    // 绑定到Exchange
    ch.QueueBind(
        q.Name,         // queue name
        "order.create", // routing key
        "orders",       // exchange
        false,
        nil,
    )
    
    // 消费消息
    msgs, _ := ch.Consume(
        q.Name, // queue
        "",     // consumer
        false,  // auto-ack
        false,  // exclusive
        false,  // no-local
        false,  // no-wait
        nil,    // args
    )
    
    for msg := range msgs {
        fmt.Println("收到消息:", string(msg.Body))
        
        // 处理消息
        processOrder(msg.Body)
        
        // 手动确认
        msg.Ack(false)
    }
}
```

### 3.5 消息确认

**生产者确认**（Confirm模式）：
```go
ch.Confirm(false)  // 开启确认模式

ch.Publish(...)

select {
case confirm := <-ch.NotifyPublish(make(chan amqp.Confirmation, 1)):
    if confirm.Ack {
        fmt.Println("消息已确认")
    } else {
        fmt.Println("消息未确认")
    }
}
```

**消费者确认**：
```go
// 手动确认
msg.Ack(false)

// 拒绝并重新入队
msg.Nack(false, true)

// 拒绝不重新入队
msg.Reject(false)
```

## 4. 消息可靠性

### 4.1 生产者可靠性

**问题**：消息发送失败

**解决方案**：
1. **同步发送 + 重试**：
```go
func sendWithRetry(producer *kafka.Writer, msg kafka.Message, maxRetries int) error {
    for i := 0; i < maxRetries; i++ {
        err := producer.WriteMessages(context.Background(), msg)
        if err == nil {
            return nil
        }
        time.Sleep(time.Second * time.Duration(i+1))
    }
    return errors.New("发送失败")
}
```

2. **本地消息表**（最可靠）：
```sql
CREATE TABLE local_messages (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    topic VARCHAR(100),
    message TEXT,
    status VARCHAR(20),  -- 'pending', 'sent', 'failed'
    retry_count INT DEFAULT 0,
    created_at TIMESTAMP
);
```

```go
// 1. 业务逻辑 + 本地消息（同一事务）
tx.Exec("INSERT INTO orders ...")
tx.Exec("INSERT INTO local_messages ...")
tx.Commit()

// 2. 定时扫描发送
func sendPendingMessages() {
    rows := db.Query("SELECT * FROM local_messages WHERE status = 'pending'")
    for rows.Next() {
        // 发送到MQ
        err := kafka.WriteMessages(...)
        if err == nil {
            db.Exec("UPDATE local_messages SET status = 'sent' WHERE id = ?", id)
        }
    }
}
```

### 4.2 Broker可靠性

**Kafka**：
1. **副本机制**：`replication.factor=3`（3个副本）
2. **ISR**（In-Sync Replicas）：同步副本集合
3. **acks配置**：
   - `acks=0`：不等待确认（最快，可能丢失）
   - `acks=1`：Leader确认（可能丢失）
   - `acks=all`：所有ISR确认（最可靠）

**RabbitMQ**：
1. **持久化**：Exchange、Queue、Message都设置durable
2. **镜像队列**：多个节点存储副本

### 4.3 消费者可靠性

**问题**：消费失败、重复消费

**解决方案**：

**1. 手动提交offset**（Kafka）：
```go
// 先处理，再提交
processMessage(msg)
reader.CommitMessages(ctx, msg)
```

**2. 幂等处理**：
```go
func processOrder(orderID string) {
    // 检查是否已处理
    if db.Exists("processed:" + orderID) {
        return  // 幂等，跳过
    }
    
    // 处理业务
    doBusinessLogic()
    
    // 标记已处理
    db.Set("processed:" + orderID, 1)
}
```

**3. 死信队列（DLQ）**：
```go
// 处理失败，发送到死信队列
if err := processMessage(msg); err != nil {
    sendToDLQ(msg)
    msg.Ack(false)  // 确认原消息
}
```

### 4.4 消息顺序性

**问题**：消息乱序

**Kafka解决方案**：
1. **单分区**：同一key的消息进入同一分区
2. **单消费者**：一个分区只有一个消费者

```go
// 生产者：按订单ID分区
writer.WriteMessages(context.Background(),
    kafka.Message{
        Key:   []byte(orderID),  // 相同订单ID进入同一分区
        Value: []byte(data),
    },
)
```

**RabbitMQ解决方案**：
1. **单队列**：消息串行处理
2. **消息ID**：自己维护顺序

### 4.5 Kafka Rebalance机制⭐⭐

> Consumer Group中的分区分配发生变化时触发Rebalance，理解其原理是避免线上消费中断的关键。

**Rebalance触发条件**：
```
1. 消费者加入 Consumer Group（如新实例上线）
2. 消费者退出（进程崩溃、心跳超时、主动离开）
3. Topic的分区数量变化
4. 消费者订阅的Topic发生变化
```

**Rebalance流程（Eager协议）**：
```
1. 所有消费者停止消费，释放所有分区（Stop-The-World）
2. Group Coordinator 收集所有消费者信息
3. 选举一个消费者作为Leader
4. Leader根据分配策略重新分配分区
5. 分配结果通过Coordinator下发给所有消费者
6. 消费者重新开始消费

问题：Rebalance期间整个Consumer Group停止消费！
```

**增量Rebalance（Cooperative Sticky，Kafka 2.4+推荐）**：
```
改进：不再一次性释放所有分区
1. 第一轮：只释放需要移交的分区
2. 第二轮：将释放的分区分配给新消费者
好处：未受影响的分区消费不中断
```

**如何减少不必要的Rebalance**：
```bash
# 常见问题：消费者处理慢被误判为离开
# 增大心跳超时和处理超时
session.timeout.ms=45000        # 心跳超时（默认10s，调大）
heartbeat.interval.ms=15000     # 心跳间隔（建议为session.timeout的1/3）
max.poll.interval.ms=600000     # 两次poll的最大间隔（默认5分钟，调大）
max.poll.records=50             # 每次poll的最大消息数（减少单次处理时间）
```

### 4.6 延迟队列⭐⭐

> 延迟队列允许消息在指定延迟后才被消费，常用于：订单超时取消、定时任务、重试退避。

**RabbitMQ实现延迟队列**：
```
方案：TTL（消息存活时间）+ DLX（死信交换机）

1. 消息发送到普通队列，设置TTL=30分钟
2. 30分钟后消息过期，自动路由到DLX（死信交换机）
3. DLX将消息路由到真正的消费队列
4. 消费者从消费队列拉取消息处理

流程：
Producer → [普通队列, TTL=30min] → 到期 → [DLX] → [消费队列] → Consumer
```

**Kafka实现延迟队列**：
```
Kafka原生不支持延迟队列，常见方案：

方案1：多级延迟Topic
- 创建多个Topic：delay-5s, delay-30s, delay-5min, delay-30min
- 消息发送到对应延迟级别的Topic
- 延迟服务定时扫描，到期后转发到目标Topic

方案2：时间轮
- 自己实现时间轮数据结构
- 消息先入时间轮，到期后推送到Kafka目标Topic

方案3：第三方组件
- 使用Redis ZSet存储延迟消息（score=到期时间戳）
- 定时扫描到期消息，推送到Kafka
```

### 4.7 Kafka幂等与Exactly-Once语义⭐⭐

**Producer端幂等（enable.idempotence）**：
```
问题：网络抖动导致Producer重试，Broker收到重复消息

开启幂等：
enable.idempotence=true  （Kafka 0.11+）

原理：
1. Producer启动时分配一个PID（Producer ID）
2. 每条消息携带PID + Sequence Number（单调递增）
3. Broker检查<PID, Partition, SeqNum>是否已存在
4. 重复则直接返回成功，不重复写入

限制：
- 只保证单分区内幂等（跨分区不保证）
- 只保证单会话幂等（Producer重启PID变化）
```

**Exactly-Once语义（事务）**：
```
端到端 Exactly-Once = 幂等Producer + 事务 + 消费者read_committed

事务流程：
1. Producer开启事务
2. 发送消息到多个Topic/Partition
3. 发送消费者offset
4. 提交事务（原子性：要么全成功要么全失败）

适用场景：Kafka Streams流处理（consume-transform-produce模式）
代价：性能下降约20%
```

## 5. 消息队列选型

### 5.1 Kafka vs RabbitMQ

| 特性 | Kafka | RabbitMQ |
|------|-------|----------|
| **吞吐量** | 百万级/秒 | 万级/秒 |
| **延迟** | ms级 | μs级 |
| **持久化** | 磁盘（顺序写） | 内存+磁盘 |
| **消息顺序** | 分区有序 | 队列有序 |
| **消息路由** | 简单（Topic/Partition） | 灵活（Exchange） |
| **适用场景** | 日志、大数据、流处理 | 任务队列、RPC |

### 5.2 选型建议

**选Kafka**：
- 大数据量（百万级QPS）
- 日志收集、监控数据
- 流处理（Kafka Streams）
- 消息回溯（按offset读取历史）

**选RabbitMQ**：
- 复杂路由（Exchange）
- 低延迟要求
- 任务队列、异步RPC
- 优先级队列、延迟队列

**选Redis（Pub/Sub + Stream）**：
- 轻量级消息
- 简单场景
- 已有Redis基础设施

## 6. 实战案例

### 案例1：订单系统异步处理

**架构**：
```
[订单服务] → Kafka(orders) → [库存服务]
                           → [积分服务]
                           → [通知服务]
```

**订单服务（生产者）**：
```go
func CreateOrder(order *Order) error {
    // 1. 写数据库
    tx, _ := db.Begin()
    tx.Exec("INSERT INTO orders ...")
    
    // 2. 写本地消息表
    tx.Exec("INSERT INTO local_messages (topic, message) VALUES ('orders', ?)", order.JSON())
    
    tx.Commit()
    
    // 3. 异步发送MQ
    go sendToKafka("orders", order)
    
    return nil
}
```

**库存服务（消费者）**：
```go
func main() {
    reader := kafka.NewReader(kafka.ReaderConfig{
        Brokers: []string{"localhost:9092"},
        Topic:   "orders",
        GroupID: "inventory-service",
    })
    
    for {
        msg, _ := reader.ReadMessage(context.Background())
        
        // 幂等处理
        if processed(msg.Key) {
            continue
        }
        
        // 扣减库存
        err := deductInventory(msg.Value)
        if err != nil {
            // 发送到死信队列
            sendToDLQ(msg)
        }
        
        // 标记已处理
        markProcessed(msg.Key)
    }
}
```

### 案例2：削峰填谷（秒杀）

**架构**：
```
[秒杀请求] → Redis预扣库存 → MQ → [订单服务慢慢处理]
```

**秒杀接口**：
```go
func Seckill(userID, productID int64) error {
    // 1. 限流
    if !limiter.Allow() {
        return ErrTooManyRequests
    }
    
    // 2. Redis预扣库存
    stock := redis.Decr(fmt.Sprintf("stock:%d", productID)).Val()
    if stock < 0 {
        redis.Incr(fmt.Sprintf("stock:%d", productID))
        return ErrOutOfStock
    }
    
    // 3. 发送到MQ
    kafka.WriteMessages(context.Background(), kafka.Message{
        Key:   []byte(fmt.Sprintf("%d", userID)),
        Value: []byte(fmt.Sprintf(`{"user_id":%d,"product_id":%d}`, userID, productID)),
    })
    
    return nil
}
```

**订单服务（消费者）**：
```go
func processOrder(msg kafka.Message) {

    order := parseMessage(msg.Value)
    db.Exec("INSERT INTO orders ...")
    
    // 2. 扣减库存（真正扣减）
    db.Exec("UPDATE inventory SET stock = stock - 1 WHERE product_id = ? AND stock > 0", order.ProductID)
```

## 总结

消息队列核心：
1. **基础**：解耦、异步、削峰
2. **Kafka**：高吞吐、分区、Consumer Group
3. **RabbitMQ**：灵活路由、低延迟
4. **可靠性**：生产者（本地消息表）、Broker（副本）、消费者（幂等）
5. **选型**：Kafka（大数据）、RabbitMQ（任务队列）

## 7. 面试题自查

### 消息治理补充

- MQ 不是“异步化万金油”，它把同步复杂度转成了消息语义、重试语义和运营治理复杂度。
- 生产可用的 MQ 方案必须同时定义：幂等策略、重试预算、死信处理、积压预案和回放方案。
- 没有消费可观测性（lag、失败率、重试次数、DLQ 增长），消息系统迟早会从“解耦工具”变成“故障放大器”。

### 初级高频速刷

**1. Producer、Broker、Consumer 分别扮演什么角色？**

Producer 生产消息，Broker 存储并转发消息，Consumer 拉取或接收消息后执行业务。这个问题看似基础，实际在考你是否理解消息系统是“职责拆分”，而不只是“中间加个队列”。

**2. Topic 和 Queue 的区别是什么？**

Queue 更偏点对点语义，一条消息通常被一个消费者组内实例处理；Topic 更偏发布订阅，多个订阅方可以各自消费同一类消息。不同 MQ 产品命名不同，但核心差异是“消费关系”而不是单纯术语差异。

**3. Push 消费和 Pull 消费的区别是什么？**

Push 是 Broker 主动把消息推给消费者，使用简单但消费者背压控制更难；Pull 是消费者主动拉取，节奏更可控，也更适合 Kafka 这种高吞吐日志流模型。工程上不是谁更先进，而是谁更匹配系统目标。

**4. 消息确认（ACK）是什么意思？不 ACK 会发生什么？**

ACK 表示消费者确认“这条消息我已经处理完成”。如果不 ACK，Broker 往往会认为消息尚未成功处理，从而触发重投、保留未提交 offset 或进入重试流程。所以 ACK 的时机直接决定了不丢和不重的平衡点。

**5. `At Most Once`、`At Least Once`、`Exactly Once` 三种语义分别是什么？**

最多一次表示消息可能丢但不会重复；至少一次表示尽量不丢，但可能重复；恰好一次表示既不丢也不重，但实现复杂度和性能代价最高。绝大多数业务真正落地的方案其实是“至少一次 + 消费端幂等”。

### 中级高频进阶

**6. Consumer 的 offset 什么时候提交更合理？**

如果先提交 offset 再处理业务，处理失败时消息就真的丢了；如果处理成功后再提交，会更安全，但崩溃重启时可能重复消费。所以工程上通常选择后者，再用幂等兜底，而不是追求理论上的绝对不重不丢。

**7. Retry Queue 和死信队列（DLQ）有什么区别？**

Retry Queue 是给“可能恢复”的失败场景做延迟重试，DLQ 是给“多次重试后仍失败”的消息做兜底和人工排查。它们的目标完全不同，混用会让重试策略和故障分析都变得混乱。

**8. 为什么扩容 Kafka 分区可能会影响顺序性和 key 路由？**

因为很多生产者会用 `key % partition_num` 之类的路由逻辑，分区数一变，同一个 key 可能被映射到新分区，原有顺序边界被打破。扩容能提升吞吐，但不意味着对业务透明，这一点常被忽视。

**9. 为什么“先上 MQ 再说”常常会把系统复杂度推高？**

因为一旦引入 MQ，你就必须补齐至少一次语义下的幂等、失败重试、死信治理、消息回放、积压监控，否则只是把同步调用失败换成了异步链路不可见失败。MQ 能解耦，但前提是你有能力治理它。

**10. 为什么重试策略必须有预算，而不是默认无限重试？**

无限重试会把下游故障放大成流量风暴，尤其在生产者重试、消费者重试、RPC 重试叠加时。重试应当是“有上限、有退避、有抖动、可熔断”的受控行为，而不是默认兜底。

### 深入问答

#### Q1: 消息队列的核心作用是什么？什么场景下需要引入MQ？

**答**：核心作用：①解耦（上下游服务独立演进）②异步（非核心流程异步处理，降低响应时间）③削峰（缓冲流量高峰，保护下游）④广播（一条消息多个消费者）。引入场景：注册后发邮件/短信（异步）、订单创建通知多个下游（解耦）、秒杀（削峰）、日志收集（大数据量异步传输）。

#### Q2: Kafka的Topic和Partition是什么关系？为什么需要分区？

**答**：Topic是逻辑概念，Partition是物理存储单元。一个Topic被划分为多个Partition，分布在不同Broker上。分区的作用：①并行读写提高吞吐（多个Producer/Consumer可以并行操作不同分区）②水平扩展（增加分区数可提升吞吐）③保证分区内有序（同一key的消息进入同一分区）。分区选择策略：轮询、Key Hash、自定义。

#### Q3: Kafka Consumer Group的工作机制是什么？Rebalance有什么问题？

**答**：同一Consumer Group内的消费者平分Topic的Partition，每个Partition只被一个消费者消费。Rebalance在消费者加入/退出或分区变化时触发。问题：Eager协议Rebalance期间Stop-The-World（所有消费者停止消费）。缓解：使用Cooperative Sticky分配策略（增量Rebalance，Kafka 2.4+），调大session.timeout.ms和max.poll.interval.ms避免误判。

#### Q4: Kafka为什么吞吐量能达到百万级？有哪些底层优化？

**答**：5大优化：①顺序写磁盘（Append-Only Log，比随机写内存还快）②零拷贝（sendfile系统调用，数据从磁盘PageCache直接DMA到网卡，避免CPU拷贝和用户态切换）③批量发送/接收（batch.size+linger.ms，减少网络往返）④PageCache（利用OS页缓存，Consumer读取命中内存，多Consumer共享缓存）⑤分区并行（多Partition多Consumer并行读写）。

#### Q5: 零拷贝的原理是什么？传统IO和sendfile有什么区别？

**答**：传统IO：磁盘→内核缓冲区（DMA拷贝）→用户缓冲区（CPU拷贝）→Socket缓冲区（CPU拷贝）→网卡（DMA拷贝），共4次拷贝2次系统调用。sendfile零拷贝：磁盘→内核缓冲区（DMA拷贝）→网卡（DMA拷贝），共2次拷贝1次系统调用，数据不经过用户空间，节省CPU和内存。Kafka通过FileChannel.transferTo()实现。

#### Q6: 如何保证消息不丢失？从Producer、Broker、Consumer三个环节分别说。

**答**：**Producer**：同步发送+重试+本地消息表（事务保证本地DB和消息表一致，定时扫描补偿发送）。**Broker**：Kafka设置acks=all（所有ISR副本确认）+ replication.factor≥3 + min.insync.replicas≥2；RabbitMQ开启持久化+镜像队列。**Consumer**：手动提交offset（先处理后提交），处理失败不提交或发送到死信队列。

#### Q7: 如何保证消息不重复消费？幂等处理有哪些方案？

**答**：消息重复不可能完全避免（网络重试），必须在消费端做幂等。方案：①唯一ID去重（消息携带唯一ID，用Redis SET NX或数据库唯一索引判重）②状态机（如订单 created→paid，重复paid操作无效）③数据库乐观锁（UPDATE ... WHERE version=N）④Kafka Producer端幂等（enable.idempotence=true，<PID, Partition, SeqNum>去重，单分区单会话有效）。

#### Q8: 如何保证消息顺序性？全局有序和分区有序的区别？

**答**：全局有序：单分区+单消费者，牺牲并发性。分区有序（推荐）：按业务key（如订单ID）hash到同一分区，同一分区内消息有序，单分区只被一个消费者消费。RabbitMQ：单队列串行消费。注意：消费者并发处理时可能破坏顺序，需要单线程处理或者内存中按key排序后批量处理。

#### Q9: Kafka和RabbitMQ的核心区别是什么？如何选型？

**答**：Kafka：百万级吞吐、顺序写磁盘、分区并行、消息可回溯（按offset）、适合大数据/日志/流处理。RabbitMQ：万级吞吐、μs级低延迟、灵活路由（Exchange 4种类型）、支持优先级/延迟队列、适合任务队列/异步RPC。选型：大数据量/日志/流处理选Kafka，复杂路由/低延迟/任务队列选RabbitMQ，轻量场景可用Redis Stream。

#### Q10: RabbitMQ的Exchange有哪几种类型？各自的路由规则是什么？

**答**：4种：①Direct：按routing_key精确匹配路由到Queue②Fanout：广播到所有绑定的Queue（忽略routing_key）③Topic：通配符匹配（*匹配一个单词，#匹配零或多个单词，如order.*匹配order.create）④Headers：根据消息头属性匹配路由。常用Direct（点对点）和Topic（灵活订阅）。

#### Q11: 延迟队列的实现方式有哪些？各有什么优缺点？

**答**：RabbitMQ：TTL+DLX（消息设TTL过期后进入死信交换机再路由到消费队列），原生支持但同一队列TTL必须相同（不同延迟需多个队列），或使用rabbitmq_delayed_message_exchange插件。Kafka无原生支持：方案1多级延迟Topic（delay-5s/delay-30s等），简单但延迟级别有限；方案2 Redis ZSet（score=到期时间戳定时扫描），灵活但引入额外组件；方案3自实现时间轮。

#### Q12: 消息积压怎么处理？如何快速消化积压消息？

**答**：①增加消费者实例数（Consumer Group扩容，但不能超过分区数）②增加分区数提高并发度③优化消费逻辑（异步处理、批量写DB、减少单条处理时间）④临时紧急方案：创建临时Topic扩大分区数，消费者只做转发不处理业务，另起大量消费者消费临时Topic⑤降级：跳过非核心消息，优先处理关键业务。

#### Q13: Kafka的ISR机制是什么？为什么不用所有副本确认？

**答**：ISR（In-Sync Replicas）是与Leader保持同步的副本集合。Follower如果落后太多（replica.lag.time.max.ms超时）会被踢出ISR。acks=all只要求ISR内所有副本确认。为什么不要求所有副本：如果某个Follower故障或网络分区，等待它确认会导致写入阻塞。ISR机制平衡了可靠性和可用性。min.insync.replicas控制ISR最小副本数，低于此值Producer写入失败。

#### Q14: Kafka的Exactly-Once语义是如何实现的？有什么代价？

**答**：三层保证：①Producer幂等（enable.idempotence=true，<PID,Partition,SeqNum>去重，保证单分区单会话不重复）②事务（transactional.id，跨分区原子写入+消费者offset提交原子绑定）③Consumer read_committed（只读已提交事务的消息）。适用场景：Kafka Streams的consume-transform-produce模式。代价：性能下降约20%（事务协调、两阶段提交开销）。

#### Q15: 本地消息表模式是什么？它解决了什么问题？

**答**：解决分布式事务中"业务操作成功但消息发送失败"的问题。实现：①业务操作和写入本地消息表在同一个数据库事务中（保证一致性）②消息表记录status=pending③异步定时扫描pending消息发送到MQ，成功后更新status=sent④失败重试，超过次数标记failed人工介入。本质是将分布式事务拆解为本地事务+最终一致性补偿。

#### Q16: Kafka 为什么不适合做传统任务队列？如果硬要做，会踩哪些坑？

**答**：Kafka 的设计目标是高吞吐日志流，而不是“任务被某个 worker 精确取走并执行一次”。它缺少任务队列常见的 ack/nack、单条优先级、延迟重投、单消息级死信处理等能力。硬要做时常见问题有：失败重试语义粗糙、积压时延迟不可控、一个慢消费者会拖住分区、顺序和并发难同时满足。适合 Kafka 的是事件流、日志、埋点、CDC，不适合复杂任务编排。

#### Q17: Consumer Rebalance 为什么经常成为线上抖动源？怎么降低它的影响？

**答**：Rebalance 会重新分配分区，期间消费者可能暂停消费、撤销本地状态、重新拉取数据，导致吞吐抖动和延迟尖峰。触发源包括实例频繁重启、GC 停顿、`max.poll.interval.ms` 配置过小、扩缩容过于频繁。降低影响的方法：使用 Cooperative Sticky 分配策略、控制滚动发布节奏、避免单条消息处理时间过长、将慢处理改为异步、把消费者实例做成稳定长连接而不是频繁抖动。

#### Q18: 死信队列（DLQ）应该怎么设计？为什么“所有失败都进死信”是个坏主意？

**答**：DLQ 适合承接“自动重试后仍失败、且需要人工分析”的消息，而不是把所有异常一股脑丢进去。设计时要明确：进入条件、保留时长、重放方式、去重策略、责任归属。所有失败都进死信的问题在于会把临时性故障、下游抖动和真正坏消息混在一起，最终没人能处理。更合理的做法是先区分可重试错误和不可重试错误，再决定是延迟重试、丢弃、告警还是进 DLQ。

#### Q19: 为什么消息系统里“重试”本身也可能是故障放大器？

**答**：下游已经过载或故障时，盲目重试会把一次失败放大成多次请求，进一步压垮下游，形成 retry storm。尤其是生产者重试、消费者重试、RPC 重试叠加时，放大倍数非常可怕。正确做法是给重试设预算：限制次数、指数退避加抖动、区分可重试错误码，并和熔断、限流、DLQ 配合使用。面试里如果只会说“失败就重试”，通常说明没有线上故障经验。

#### Q20: 如果一条业务链路要求“下单成功后必须最终发送优惠券”，你会怎么选 MQ 方案？

**答**：这是典型的最终一致性问题，重点不是追求分布式强事务，而是保证“业务成功后消息一定能补发”。工程上优先考虑本地消息表或事务消息：下单事务提交成功后，消息进入待发送状态，异步投递到 MQ，消费者用幂等方式发券。再配合重试、告警、对账任务保证最终送达。资深面试官会继续追问你如何处理重复发券、补偿扫描、人工兜底，这些都要提前想清楚。

#### Q21: 如何设计“可重放（Replay）”能力，避免线上修复只能手工补数据？

**答**：可重放设计的核心是消息可定位、可筛选、可幂等重放。通常要做三件事：①消息体带稳定业务主键和事件版本，支持按时间段和条件筛选；②消费端天然幂等，重放不会重复扣款/发券；③提供受控回放通道（限速、灰度、审计日志），避免一次重放把下游再次打挂。没有可重放能力的 MQ 系统，事故恢复会长期依赖人工脚本和数据修补。

#### Q22: Kafka 的分区键（Partition Key）怎么设计更稳妥？

**答**：分区键需要在“顺序性”和“负载均衡”之间取平衡。若业务要求同一实体有序（如订单状态流转），用实体 ID 作为 key；若业务不要求严格有序，优先选分布更均匀的 key 防热点。还要提前考虑扩容影响：分区数变化会改变路由结果，关键链路应避免把顺序性建立在“可变分区数 + 简单取模”之上。

#### Q23: 消费端如何做到“业务成功才算消费成功”？

**答**：原则是“先完成业务事务，再确认消费进度”。Kafka 场景里通常是业务处理成功后再提交 offset；RabbitMQ 场景里是处理成功后 ack，失败时 nack/requeue 或进重试队列。并且要有幂等防线兜底，避免崩溃重启导致重复处理。先提交再处理虽然看起来吞吐更高，但会把失败路径直接变成消息丢失。

#### Q24: 为什么很多团队有了 DLQ 仍然处理不好坏消息？

**答**：因为只有“入口”没有“运营闭环”。DLQ 真正有效需要同时具备：明确进入条件、按错误类型分桶、可观测告警、可重放工具、责任人和 SLA。否则 DLQ 只会变成消息坟场，积压越来越多却没人敢动。面试里如果能把 DLQ 讲成“治理流程”而不是“一个队列”，基本就有实战深度。

#### Q25: Outbox 已经做了，为什么还会出现重复消费导致资损？

**答**：Outbox 只能保证“业务数据和消息最终都会发出”，不能天然保证“下游只执行一次”。重复资损通常来自三类问题：
1. 投递层重试导致同一事件被消费多次。
2. 消费端没有业务幂等（只做了消息 ID 去重，但没有状态机约束）。
3. 幂等记录和业务写入不在同一事务，出现“先扣款后写幂等标记失败”。

治理要点：消费端幂等必须落在业务主键和状态机上，并与核心写操作同事务提交。
