## 1. 进程与线程管理

### 1.1 进程vs线程

| 维度 | 进程 | 线程 |
|------|------|------|
| **定义** | 资源分配的基本单位 | CPU调度的基本单位 |
| **内存** | 独立地址空间 | 共享地址空间 |
| **通信** | IPC（管道、消息队列、共享内存） | 直接访问全局变量 |
| **开销** | 创建/切换开销大（3-5ms） | 创建/切换开销小（<1μs） |
| **稳定性** | 一个进程崩溃不影响其他进程 | 一个线程崩溃整个进程崩溃 |

### 1.2 进程状态转换

```
┌──────┐  fork()  ┌─────────┐  调度  ┌─────────┐
│ 创建 │ ───────→ │ 就绪队列 │ ─────→ │  运行   │
└──────┘          └─────────┘        └─────────┘
                       ↑                   │
                       │                   │ 阻塞（IO/锁）
                       │                   ↓
                       │              ┌─────────┐
                       └──────────────│  阻塞   │
                          唤醒         └─────────┘
```

**查看进程状态**：
```bash
# ps命令
ps aux | head -5
# STAT列：
# R: Running（运行）
# S: Sleeping（睡眠）
# D: Disk Sleep（不可中断的睡眠，等待IO）
# Z: Zombie（僵尸进程）
# T: Stopped（暂停）

# 示例
USER  PID  STAT  COMMAND
root  1    Ss    /sbin/init
root  123  R     ./myapp
root  456  D     cp /data /backup  # IO密集
```

### 1.3 进程调度算法

#### 1.3.1 CFS（Completely Fair Scheduler）
Linux默认调度器，核心思想：**让每个进程获得公平的CPU时间**

```c
// 虚拟运行时间（vruntime）
vruntime += delta_exec * (NICE_0_LOAD / weight)

// weight根据nice值计算：
// nice=-20（最高优先级）：weight=88761
// nice=0（默认）：weight=1024
// nice=19（最低优先级）：weight=15
```

**示例**：
```bash
# 进程A：nice=0, weight=1024
# 进程B：nice=5, weight=335
# 
# 运行10ms后：
# A的vruntime += 10 * (1024/1024) = 10ms
# B的vruntime += 10 * (1024/335) = 30.5ms
# 
# 下次调度选择vruntime最小的（进程A）
```

**调整优先级**：
```bash
# 降低优先级（nice值越大，优先级越低）
nice -n 10 ./myapp

# 提高优先级（需要root）
nice -n -10 ./myapp

# 修改正在运行的进程
renice -n 5 -p 1234
```

#### 1.3.2 实时调度（RT）
```bash
# 查看实时进程
ps -eo pid,class,rtprio,comm | grep RR
# PID  CLS  RTPRIO  COMMAND
# 123  RR   50      realtime-app

# 设置实时优先级（0-99，99最高）
chrt -r -p 50 1234
```

### 1.4 进程间通信（IPC）

#### 1.4.1 管道（Pipe）
```go
// 匿名管道（只能用于父子进程）
cmd := exec.Command("ls", "/var/log")
stdout, _ := cmd.StdoutPipe()
cmd.Start()

scanner := bufio.NewScanner(stdout)
for scanner.Scan() {
    fmt.Println(scanner.Text())
}
```

**命名管道（FIFO）**：
```bash
# 创建命名管道
mkfifo /tmp/mypipe

# 进程A：写入
echo "hello" > /tmp/mypipe

# 进程B：读取
cat /tmp/mypipe  # 输出: hello
```

#### 1.4.2 共享内存（最快的IPC）
```c
#include <sys/shm.h>

// 进程A：创建共享内存
int shmid = shmget(IPC_PRIVATE, 1024, IPC_CREAT | 0666);
char *shmaddr = (char *)shmat(shmid, NULL, 0);
strcpy(shmaddr, "hello from process A");

// 进程B：读取共享内存
char *shmaddr = (char *)shmat(shmid, NULL, 0);
printf("%s\n", shmaddr);  // 输出: hello from process A

// 清理
shmdt(shmaddr);
shmctl(shmid, IPC_RMID, NULL);
```

**Go语言使用mmap**：
```go
import "golang.org/x/exp/mmap"

// 进程A：写入
f, _ := os.Create("/tmp/shared.dat")
f.Write([]byte("hello"))
f.Close()

// 进程B：mmap读取
at, _ := mmap.Open("/tmp/shared.dat")
data := make([]byte, 5)
at.ReadAt(data, 0)
fmt.Println(string(data))  // 输出: hello
```

#### 1.4.3 信号（Signal）
```bash
# 常用信号
SIGTERM (15)  # 优雅退出
SIGKILL (9)   # 强制杀死（无法捕获）
SIGHUP (1)    # 终端断开
SIGUSR1 (10)  # 用户自定义

# 发送信号
kill -15 1234  # 发送SIGTERM
kill -9 1234   # 发送SIGKILL
```

**Go语言捕获信号**：
```go
import (
    "os"
    "os/signal"
    "syscall"
)

func main() {
    sigChan := make(chan os.Signal, 1)
    signal.Notify(sigChan, syscall.SIGTERM, syscall.SIGINT)
    
    go func() {
        <-sigChan
        log.Println("收到退出信号，开始清理资源...")
        // 关闭数据库连接、保存状态等
        cleanup()
        os.Exit(0)
    }()
    
    // 主逻辑
    run()
}
```

### 1.5 线程模型

#### 1.5.1 用户级线程 vs 内核级线程

| 类型 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| **用户级线程** | 用户空间调度（Go runtime） | 轻量、切换快 | 无法利用多核 |
| **内核级线程** | 内核调度（pthread） | 利用多核 | 创建/切换开销大 |
| **混合模型** | M:N映射（Go的M:P:G） | 兼顾两者 | 实现复杂 |

#### 1.5.2 Go的GMP模型

```
┌─────────────────────────────────────┐
│         G (Goroutine)               │
│  轻量级线程，2KB初始栈               │
└──────────┬──────────────────────────┘
           │ 调度
┌──────────↓──────────────────────────┐
│         P (Processor)               │
│  逻辑处理器，GOMAXPROCS数量          │
└──────────┬──────────────────────────┘
           │ 绑定
┌──────────↓──────────────────────────┐
│         M (Machine)                 │
│  内核线程，真正执行代码               │
└─────────────────────────────────────┘
```

**工作窃取（Work Stealing）**：
```go
// P1的G队列：[G1, G2, G3]
// P2的G队列：[]（空闲）

// P2会从P1偷一半任务
// P1的G队列：[G1]
// P2的G队列：[G2, G3]
```

**查看Goroutine数量**：
```go
import "runtime"

func main() {
    fmt.Println("Goroutines:", runtime.NumGoroutine())
    
    // 创建10000个goroutine
    for i := 0; i < 10000; i++ {
        go func() {
            time.Sleep(1 * time.Hour)
        }()
    }
    
    fmt.Println("Goroutines:", runtime.NumGoroutine())  // 输出: 10001
}
```

## 2. 内存管理

### 2.1 虚拟内存

**为什么需要虚拟内存？**
1. **隔离**：每个进程有独立地址空间，互不干扰
2. **扩展**：虚拟内存可以大于物理内存（通过swap）
3. **共享**：多个进程共享同一物理页（如libc.so）

**地址转换**：
```
虚拟地址 (48位) = 页号 (36位) + 页内偏移 (12位)
                  ↓ 查页表
物理地址 (48位) = 页框号 (36位) + 页内偏移 (12位)
```

**查看进程内存布局**：
```bash
cat /proc/1234/maps
# 输出：
# 地址范围              权限  偏移  设备  inode  路径
# 00400000-00452000  r-xp  0     08:01  123    /usr/bin/myapp  # 代码段
# 00651000-00652000  r--p  51000 08:01  123    /usr/bin/myapp  # 只读数据
# 00652000-00653000  rw-p  52000 08:01  123    /usr/bin/myapp  # 读写数据
# 7ffff7a0d000-...   r-xp  0     08:01  456    /lib/libc.so.6  # 共享库
# 7ffffffde000-...   rw-p  0     00:00  0      [stack]         # 栈
# 7ffff7ff9000-...   rw-p  0     00:00  0      [heap]          # 堆
```

### 2.2 页表与TLB

**多级页表（x86_64）**：
```
虚拟地址 (48位)
├─ PGD索引 (9位) ────→ 页全局目录
├─ PUD索引 (9位) ────→ 页上层目录
├─ PMD索引 (9位) ────→ 页中间目录
├─ PTE索引 (9位) ────→ 页表项
└─ 页内偏移 (12位) ──→ 物理页内偏移
```

**TLB（Translation Lookaside Buffer）**：
- 缓存虚拟地址→物理地址的映射
- 命中时无需查页表（快100倍）
- TLB Miss时触发Page Walk（查多级页表）

**查看TLB Miss率**：
```bash
perf stat -e dTLB-load-misses,dTLB-loads ./myapp
# 输出：
# 1,234,567  dTLB-loads
# 12,345     dTLB-load-misses  # Miss率=1%
```

**优化TLB Miss**：
```c
// ❌ 跨步访问（TLB Miss多）
for (int i = 0; i < N; i += 1024) {
    sum += arr[i];
}

// ✅ 顺序访问（TLB命中高）
for (int i = 0; i < N; i++) {
    sum += arr[i];
}
```

### 2.3 缺页中断（Page Fault）

**分类**：
1. **Minor Page Fault**：页在内存中，但未映射到页表
2. **Major Page Fault**：页不在内存，需要从磁盘读取

**查看Page Fault**：
```bash
ps -o pid,min_flt,maj_flt,comm
# PID  MINFLT  MAJFLT  COMMAND
# 1234  12345   10      myapp

# strace跟踪
strace -c ./myapp
# % time  calls  errors  syscall
# 45.00   10     0       mmap      # 内存映射
# 30.00   10     0       page_fault # 缺页中断
```

**减少Page Fault**：
```go
// 预分配内存，减少缺页中断
slice := make([]int, 0, 1000000)  // 预分配100万个元素的容量
for i := 0; i < 1000000; i++ {
    slice = append(slice, i)
}
```

### 2.4 内存分配器

#### 2.4.1 Buddy System（Linux）
```
伙伴系统：内存块大小为2^n（4KB、8KB、16KB...）

示例：申请6KB
1. 找到最小的满足条件的块（8KB）
2. 如果没有8KB，则分裂16KB块 → 两个8KB块
3. 分配一个8KB块，另一个8KB块放回空闲链表
```

**查看内存碎片**：
```bash
cat /proc/buddyinfo
# Node 0, zone DMA      1   1   1   0   2   1   1   0   1   1   3
# Node 0, zone Normal   100 50  20  10  5   2   1   0   0   0   0
#                       4KB 8KB 16K 32K 64K 128K...
```

#### 2.4.2 TCMalloc（Thread-Caching Malloc）
Go、Chrome使用

```
┌────────────────────────┐
│   Thread Cache         │  线程私有，无锁
│  小对象(<32KB)快速分配  │
└───────┬────────────────┘
        │ 不够时申请
┌───────↓────────────────┐
│   Central Cache        │  全局，有锁
│  中等对象(32KB-256KB)   │
└───────┬────────────────┘
        │ 不够时申请
┌───────↓────────────────┐
│   Page Heap            │  全局，管理大块内存
│  大对象(>256KB)         │
└────────────────────────┘
```

**优势**：
- 小对象分配无锁（线程缓存）
- 减少内存碎片（size class）

### 2.5 内存泄漏排查

#### 2.5.1 Go内存泄漏
```go
// ❌ Goroutine泄漏
func leak() {
    ch := make(chan int)
    go func() {
        <-ch  // 永远阻塞，goroutine泄漏
    }()
    // 忘记关闭channel
}

// ✅ 正确做法
func noLeak() {
    ch := make(chan int)
    go func() {
        select {
        case <-ch:
        case <-time.After(5 * time.Second):
            return  // 超时退出
        }
    }()
    close(ch)
}
```

**pprof排查**：

```bash
curl http://localhost:6060/debug/pprof/heap > heap2.prof

go tool pprof -base heap1.prof heap2.prof
(pprof) top
# 显示增长最多的内存分配点
```

## 3. IO模型与异步IO

### 3.1 五种IO模型

| 模型 | 阻塞 | 说明 |
|------|------|------|
| **阻塞IO** | ✅ | read()阻塞直到数据到达 |
| **非阻塞IO** | ❌ | read()立即返回，需轮询 |
| **IO多路复用** | ✅ | select/epoll阻塞，监听多个fd |
| **信号驱动IO** | ❌ | 数据到达时发送信号 |
| **异步IO** | ❌ | 内核完成IO后通知应用 |

#### 3.1.1 阻塞IO
```c
// 阻塞直到收到数据
char buf[1024];
int n = read(sockfd, buf, sizeof(buf));  // 阻塞
printf("received: %s\n", buf);
```

#### 3.1.2 非阻塞IO
```c
// 设置为非阻塞
fcntl(sockfd, F_SETFL, O_NONBLOCK);

// 轮询
while (1) {
    int n = read(sockfd, buf, sizeof(buf));
    if (n > 0) {
        printf("received: %s\n", buf);
        break;
    } else if (errno == EAGAIN) {
        // 数据还没到，继续轮询
        usleep(1000);
    }
}
```

#### 3.1.3 IO多路复用（epoll）
```c
int epfd = epoll_create1(0);

// 添加监听的fd
struct epoll_event ev;
ev.events = EPOLLIN;
ev.data.fd = sockfd;
epoll_ctl(epfd, EPOLL_CTL_ADD, sockfd, &ev);

// 等待事件
struct epoll_event events[10];
while (1) {
    int nfds = epoll_wait(epfd, events, 10, -1);  // 阻塞
    for (int i = 0; i < nfds; i++) {
        if (events[i].events & EPOLLIN) {
            // 可读
            int fd = events[i].data.fd;
            read(fd, buf, sizeof(buf));
        }
    }
}
```

**Go的netpoller**：
```go
// Go运行时自动使用epoll（Linux）/kqueue（macOS）
listener, _ := net.Listen("tcp", ":8080")
for {
    conn, _ := listener.Accept()  // 底层使用epoll
    go handleConn(conn)
}
```

### 3.2 epoll vs select

| 特性 | select | epoll |
|------|--------|-------|
| **fd数量限制** | 1024 | 无限制 |
| **性能** | O(n) | O(1) |
| **内存拷贝** | 每次拷贝fd_set | 共享内存 |
| **适用场景** | 少量连接 | 高并发（C10K） |

**select缺陷**：
```c
fd_set readfds;
FD_ZERO(&readfds);
FD_SET(fd1, &readfds);
FD_SET(fd2, &readfds);
// ... 添加1000个fd

select(maxfd+1, &readfds, NULL, NULL, NULL);

// 需要遍历所有fd，找到就绪的（O(n)）
for (int fd = 0; fd <= maxfd; fd++) {
    if (FD_ISSET(fd, &readfds)) {
        read(fd, buf, sizeof(buf));
    }
}
```

**epoll优化**：
```c
epoll_wait(epfd, events, 10, -1);  // 返回就绪的events

// 只遍历就绪的fd（O(1)）
for (int i = 0; i < nfds; i++) {
    int fd = events[i].data.fd;
    read(fd, buf, sizeof(buf));
}
```

### 3.3 零拷贝（Zero Copy）

#### 3.3.1 传统拷贝（4次拷贝）
```
磁盘 → 内核缓冲区（DMA拷贝）
     → 用户空间（CPU拷贝）
     → Socket缓冲区（CPU拷贝）
     → 网卡（DMA拷贝）
```

#### 3.3.2 sendfile（2次拷贝）
```c
// Linux sendfile
int filefd = open("file.txt", O_RDONLY);
int sockfd = socket(AF_INET, SOCK_STREAM, 0);

// 直接从文件fd发送到socket fd（零拷贝）
sendfile(sockfd, filefd, NULL, file_size);

// 内核直接传输，无需拷贝到用户空间
// 磁盘 → 内核缓冲区 → 网卡
```

**Go语言**：
```go
// Go自动使用sendfile优化（Linux）
func ServeFile(w http.ResponseWriter, r *http.Request) {
    f, _ := os.Open("large_file.dat")
    defer f.Close()
    io.Copy(w, f)  // 底层使用sendfile
}
```

#### 3.3.3 mmap（内存映射）
```c
// 将文件映射到内存
int fd = open("file.txt", O_RDONLY);
char *data = mmap(NULL, file_size, PROT_READ, MAP_PRIVATE, fd, 0);

// 直接访问，像访问内存一样
printf("%s\n", data);  // 无需read()

munmap(data, file_size);
```

**优点**：
- 避免read()系统调用
- 多个进程共享同一物理页
- 内核自动管理缓存

**Nginx使用mmap**：
```nginx
location /static/ {
    # 开启sendfile和mmap
    sendfile on;
    tcp_nopush on;
}
```

### 3.4 直接IO（Direct IO）

**绕过页缓存，直接读写磁盘**：
```c
int fd = open("file.txt", O_DIRECT | O_RDONLY);
// 要求：buf地址对齐、大小为512字节的倍数
char *buf = aligned_alloc(512, 4096);
read(fd, buf, 4096);
```

**使用场景**：
- 数据库（自己管理缓存，不需要页缓存）
- 流媒体（大文件，不需要缓存）

**对比**：
```bash
# 普通IO（使用页缓存）
dd if=/dev/zero of=test bs=1M count=1000
# 1000+0 records in
# 1000+0 records out
# 1048576000 bytes (1.0 GB) copied, 0.5 s, 2.0 GB/s

# Direct IO（绕过缓存）
dd if=/dev/zero of=test oflag=direct bs=1M count=1000
# 1000+0 records in
# 1000+0 records out
# 1048576000 bytes (1.0 GB) copied, 2.0 s, 500 MB/s
```

## 4. 文件系统

### 4.1 inode与目录结构

**inode（索引节点）**：
```bash
ls -li /etc/hosts
# 123456 -rw-r--r-- 1 root root 123 Jan 1 00:00 /etc/hosts
#   ↑ inode号

stat /etc/hosts
# File: /etc/hosts
# Inode: 123456
# Links: 1
# Access: (0644/-rw-r--r--)
# Size: 123
# Blocks: 8
```

**inode包含的信息**：
- 文件类型（普通文件、目录、符号链接）
- 权限
- 所有者
- 大小
- 时间戳（atime、mtime、ctime）
- **数据块指针**（指向实际数据的位置）

**硬链接vs软链接**：
```bash
# 硬链接：多个文件名指向同一个inode
ln /etc/hosts /tmp/hosts_link
ls -li /etc/hosts /tmp/hosts_link
# 123456 -rw-r--r-- 2 root root 123 Jan 1 00:00 /etc/hosts
# 123456 -rw-r--r-- 2 root root 123 Jan 1 00:00 /tmp/hosts_link
# 同一个inode，Links=2

# 软链接：新文件，inode不同，内容是目标路径
ln -s /etc/hosts /tmp/hosts_symlink
ls -li /tmp/hosts_symlink
# 789012 lrwxrwxrwx 1 root root 10 Jan 1 00:00 /tmp/hosts_symlink -> /etc/hosts
```

### 4.2 文件系统布局（ext4）

```
┌──────────────────────────────────────────┐
│  Super Block（超级块）                    │
```

### 4.4 文件系统性能优化

## 6. Cgroup资源隔离

> **Cgroup（Control Groups）** 是Linux内核提供的资源限制、控制与隔离机制，是容器技术（Docker/K8s）的核心基石。

### 6.1 Cgroup概述

**四大资源子系统**：
| 子系统 | 控制对象 | 关键文件/工具 |
|--------|----------|---------------|
| **CPU** | 计算时间、核心绑定 | cpuset, cpu.cfs_quota_us, cpu.shares |
| **Memory** | 内存用量、Swap | memory.limit_in_bytes, memory.oom_control |
| **BlkIO** | 磁盘带宽、IOPS | blkio.throttle.*, blkio.weight |
| **Net** | 网络带宽 | net_cls + tc（流量控制） |

**查看系统Cgroup版本**：
```bash
# Cgroup V1（传统，各子系统独立挂载）
mount | grep cgroup
# cgroup on /sys/fs/cgroup/cpu type cgroup (rw,...,cpu,cpuacct)
# cgroup on /sys/fs/cgroup/memory type cgroup (rw,...,memory)

# Cgroup V2（unified hierarchy，单一挂载点）
mount | grep cgroup2
# cgroup2 on /sys/fs/cgroup type cgroup2 (rw,...,nsdelegate)
```

### 6.2 CPU资源隔离

Linux通过CFS（完全公平调度器）实现CPU时间的分配，Cgroup提供三种隔离方式：

#### 6.2.1 cpuset — 核心绑定

将进程绑定到指定CPU核心运行，实现**物理隔离**：

```bash
# 查看/设置cgroup绑定的CPU核心
cat /sys/fs/cgroup/cpuset/mygroup/cpuset.cpus
echo "0-3" > /sys/fs/cgroup/cpuset/mygroup/cpuset.cpus

# 必须同时设置内存节点（NUMA）
echo "0" > /sys/fs/cgroup/cpuset/mygroup/cpuset.mems
```

**超线程陷阱**：在开启超线程的服务器上，需注意核心编号的物理对应关系：
```bash
# 查看物理核心与逻辑核心的映射
cat /proc/cpuinfo | grep -e "core id" -e "physical id"
# physical id: 0, core id: 0  → 逻辑核心 0 和 12 是同一物理核心
# 绑定 0-5 和 12-17 才是"半数物理核心"
# 绑定 0-11 看似一半，实际覆盖了所有物理核心（超线程部分无法倍增算力）
```

**实验数据对比**（24逻辑核心服务器，筛质数测试）：
| 配置 | 核心数 | 执行时间 |
|------|--------|----------|
| 0-23（全部） | 24 | 8.9s |
| 0-11（全物理核心） | 12 | 9.5s ≈ 无变化！ |
| 0-5,12-17（半物理核心） | 12 | 17.8s ≈ 翻倍 |

#### 6.2.2 cpuquota — 配额限制（硬限制）

按时间比例限制CPU使用上限，**即使系统空闲也不能超额**：

```bash
# 时间周期（默认100ms = 100000μs）
cat /sys/fs/cgroup/cpu/mygroup/cpu.cfs_period_us   # 100000

# 配额（-1表示不限制）
# 限制单核50%：echo 50000
# 限制4核满载：echo 400000
# 限制24核50%：echo 1200000
echo "50000" > /sys/fs/cgroup/cpu/mygroup/cpu.cfs_quota_us
```

**计算公式**：`quota = 目标百分比 × period × CPU核心数`

#### 6.2.3 cpushares — 权重比例（弹性限制）

按权重比例分配CPU时间，**系统空闲时可借用其他cgroup的资源**：

```bash
# 权重范围：2-262144，默认1024
echo "1024" > /sys/fs/cgroup/cpu/groupA/cpu.shares  # A
echo "2048" > /sys/fs/cgroup/cpu/groupB/cpu.shares  # B

# 效果：
# 竞争时：A占1/3，B占2/3
# B空闲时：A可占满所有CPU
```

**三种方式对比**：
| 特性 | cpuset | cpuquota | cpushares |
|------|--------|----------|-----------|
| **隔离类型** | 物理核心隔离 | 时间配额硬限制 | 时间权重弹性分配 |
| **空闲借用** | ❌ 不能 | ❌ 不能 | ✅ 可以 |
| **最小粒度** | 1个核心 | 任意百分比 | 任意比例 |
| **隔离强度** | 最强（核心独占） | 强（硬上限） | 弱（按比例） |
| **适用场景** | 延迟敏感型、NUMA | 多租户硬隔离 | 弹性共享、超卖 |

**内核资源冲突的影响**：当多个cgroup的进程访问相同的内核资源（如同一文件的自旋锁竞争），cpuset由于物理隔离表现最优，cpuquota在此场景下性能损失最大。

### 6.3 内存资源隔离

#### 6.3.1 Linux内存管理基础回顾

```
进程申请内存 → malloc/mmap（仅分配虚拟地址）
     ↓ 实际访问时
缺页异常（Page Fault）
     ├─ Minor Fault：页在内存但未映射（无磁盘IO）
     └─ Major Fault：页在Swap/磁盘（需磁盘IO）
     ↓
alloc_pages → 伙伴系统分配物理页
     ↓ 内存不足时进入 slowpath
1. 唤醒 kswapd 交换守护进程
2. 尝试回收 Page Cache（进程置D状态）
3. 触发 OOM Killer 杀进程
4. 循环重试
```

#### 6.3.2 内存限制配置

```bash
# 硬限制：内存使用上限（超出触发OOM或挂起）
echo "512M" > /sys/fs/cgroup/memory/mygroup/memory.limit_in_bytes

# 内存+Swap总量限制（设为与limit相同 = 禁用Swap）
echo "512M" > /sys/fs/cgroup/memory/mygroup/memory.memsw.limit_in_bytes

# 软限制：用于内存超卖场景，全局内存紧张时按此值回收
echo "256M" > /sys/fs/cgroup/memory/mygroup/memory.soft_limit_in_bytes

# 查看当前使用量
cat /sys/fs/cgroup/memory/mygroup/memory.usage_in_bytes
```

#### 6.3.3 OOM控制

```bash
cat /sys/fs/cgroup/memory/mygroup/memory.oom_control
# oom_kill_disable 0   → 0=启用OOM Killer（默认），1=禁用（进程挂起为D状态）
# under_oom 0          → 0=正常，1=当前已OOM

# 禁用OOM Killer（进程挂起而非被杀）
echo 1 > /sys/fs/cgroup/memory/mygroup/memory.oom_control
# 恢复方式：增加内存限制、杀进程、迁移进程到其他cgroup
```

#### 6.3.4 内存压力通知（eventfd）

```c
// 通过 eventfd + memory.pressure_level 实现内存压力监控
int efd = eventfd(0, 0);
int cfd = open("/sys/fs/cgroup/memory/mygroup/memory.pressure_level", O_RDONLY);
// 写入 cgroup.event_control: "<efd> <cfd> <level>"
// level: "low" | "medium" | "critical"

// 压力级别：
// low     → 开始内存回收
// medium  → 开始Swap换出、活跃Cache写回
// critical → 触发OOM Killer
```

### 6.4 IO资源隔离

#### 6.4.1 Linux IO体系7层架构

```
应用程序 read()/write()
    ↓
① VFS层         — 虚拟文件系统，统一接口
    ↓
② 文件系统层    — ext4/xfs/btrfs 具体实现
    ↓
③ 页缓存层      — Page Cache（Buffered IO在此缓冲）
    ↓
④ 通用块层      — Block IO请求标准化 ← 【带宽/IOPS限速在此实现】
    ↓
⑤ IO调度层      — cfq/deadline/noop ← 【权重限速在此实现】
    ↓
⑥ 块设备驱动层  — 设备操作接口
    ↓
⑦ 块设备层      — 物理设备
```

#### 6.4.2 IO调度算法

| 算法 | 特点 | 适用场景 |
|------|------|----------|
| **cfq** | 完全公平，按进程分时间片 | 通用服务器，支持cgroup权重 |
| **deadline** | 保证请求最终期限，防止饿死 | 数据库等IO密集型 |
| **noop** | 无调度，仅FIFO合并 | SSD固态硬盘（最佳） |

```bash
# 查看/切换调度算法
cat /sys/block/sda/queue/scheduler    # noop deadline [cfq]
echo deadline > /sys/block/sda/queue/scheduler
```

#### 6.4.3 blkio限速配置

```bash
# 带宽限制（基于通用块层，需 Direct IO 才对应用可感知）
echo "8:0 1048576" > /sys/fs/cgroup/blkio/mygroup/blkio.throttle.write_bps_device  # 写1MB/s
echo "8:0 1048576" > /sys/fs/cgroup/blkio/mygroup/blkio.throttle.read_bps_device   # 读1MB/s

# IOPS限制
echo "8:0 100" > /sys/fs/cgroup/blkio/mygroup/blkio.throttle.write_iops_device

# 权重限制（需cfq调度器，同样需 Direct IO）
echo 500 > /sys/fs/cgroup/blkio/mygroup/blkio.weight  # 范围10-1000
```

**重要限制**：Cgroup V1的blkio对 **Buffered IO无法限速**！因为Buffered IO的write-back由内核异步完成，无法追踪所属cgroup。此问题在Cgroup V2中已解决。

### 6.5 网络资源隔离

Cgroup网络隔离复用Linux的 **tc（Traffic Control）** 机制，通过 `net_cls` 子系统给数据包打标签，配合tc队列规则实现限速。

#### 6.5.1 tc队列规则（qdisc）概览

| 类型 | qdisc | 核心机制 | 特点 |
|------|-------|----------|------|
| **不可分类** | pfifo_fast | 3优先级FIFO | 旧版默认，不可调参 |
| **不可分类** | fq_codel | 公平队列+CoDel | **新版默认**，解决Bufferbloat |
| **不可分类** | tbf | 令牌桶 | 简单限速 |
| **可分类** | htb | 分层令牌桶 | **推荐**，支持带宽借用和分类 |

**Bufferbloat问题**：过大的网络缓冲区导致数据包驻留时间超过超时阈值，引发全部请求超时失败，系统看似满载实则不可用。CoDel算法通过控制**驻留时间**而非队列长度来解决此问题。

#### 6.5.2 使用cgroup限制网络带宽

```bash
# 1. 创建cgroup并标记classid
mkdir /sys/fs/cgroup/net_cls/mygroup
echo 0x00010010 > /sys/fs/cgroup/net_cls/mygroup/net_cls.classid
# 0xAAAABBBB → AAAA:BBBB = 1:10

# 2. 配置HTB队列规则
tc qd add dev eth0 root handle 1: htb default 100
tc cl add dev eth0 parent 1: classid 1:1 htb rate 100mbit burst 20k
tc cl add dev eth0 parent 1:1 classid 1:10 htb rate 10mbit ceil 50mbit burst 20k
tc qd add dev eth0 parent 1:10 handle 10: fq_codel

# 3. 添加cgroup过滤器
tc fi add dev eth0 parent 1: protocol ip prio 1 handle 1: cgroup

# 4. 将进程加入cgroup
echo $PID > /sys/fs/cgroup/net_cls/mygroup/tasks
```

### 6.6 Cgroup V2（Unified Hierarchy）

Cgroup V2将所有资源控制器统一到单一层级，简化配置并新增重要功能：

**开启方法**：
```bash
# 内核引导参数
grubby --update-kernel=ALL --args=systemd.unified_cgroup_hierarchy=1
```

#### 6.6.1 V2核心变化

| 特性 | V1 | V2 |
|------|----|----|
| 层级结构 | 每个资源独立层级 | **统一层级** |
| Buffer IO限速 | ❌ 不支持 | ✅ **支持** |
| 压力通知 | eventfd（仅memory） | **PSI机制**（cpu/memory/io） |
| 线程级控制 | ❌ | ✅ threaded模式 |
| 配置复杂度 | 高 | **大幅简化** |

#### 6.6.2 V2配置示例

```bash
# 创建cgroup
mkdir /sys/fs/cgroup/mygroup

# 启用资源控制器
echo '+cpu +memory +io +pids' > /sys/fs/cgroup/cgroup.subtree_control

# CPU配额（max period格式）
echo "50000 100000" > /sys/fs/cgroup/mygroup/cpu.max      # 单核50%
echo "100" > /sys/fs/cgroup/mygroup/cpu.weight             # 权重（1-10000）

# 内存限制
echo "512M" > /sys/fs/cgroup/mygroup/memory.max            # 硬限制
echo "256M" > /sys/fs/cgroup/mygroup/memory.high           # 软限制（触发回收）
echo "128M" > /sys/fs/cgroup/mygroup/memory.min            # 硬保护（不被回收）
echo "1" > /sys/fs/cgroup/mygroup/memory.oom.group         # 整组OOM

# IO限速（支持Buffer IO！）
echo "253:0 wbps=2097152" > /sys/fs/cgroup/mygroup/io.max  # 写2MB/s
echo "253:0 rbps=10485760 wiops=100" > /sys/fs/cgroup/mygroup/io.max

# PIDS限制
echo "1000" > /sys/fs/cgroup/mygroup/pids.max

# 添加进程
echo $PID > /sys/fs/cgroup/mygroup/cgroup.procs
```

#### 6.6.3 PSI（Pressure Stall Information）

V2引入的**统一压力监控机制**，可监控cpu/memory/io的资源压力：

```bash
cat /sys/fs/cgroup/mygroup/cpu.pressure
# some avg10=4.56 avg60=2.30 avg300=1.15 total=234567
# full avg10=0.00 avg60=0.00 avg300=0.00 total=0

# some：至少有一个任务在等待资源的时间比例
# full：所有任务都在等待资源的时间比例（cpu无此项）
```

## 7. 系统调用与性能

### 7.1 系统调用开销

**用户态→内核态切换开销**：约100ns

```c
// strace统计系统调用
strace -c ./myapp
# % time  seconds  usecs/call  calls  errors  syscall
# 50.00   0.500000  50          10000   0      read
# 30.00   0.300000  30          10000   0      write
# 10.00   0.100000  100         1000    0      open
```

**减少系统调用**：
```go
// ❌ 频繁系统调用
for i := 0; i < 10000; i++ {
    syscall.Write(fd, []byte("x"))  // 10000次系统调用
}

// ✅ 批量写入
var buf bytes.Buffer
for i := 0; i < 10000; i++ {
    buf.WriteByte('x')
}
syscall.Write(fd, buf.Bytes())  // 1次系统调用
```

### 7.2 vDSO（虚拟动态共享对象）

**无需陷入内核的系统调用**：
```c
// gettimeofday()通过vDSO实现（无系统调用）
struct timeval tv;
gettimeofday(&tv, NULL);  // 直接读取内核在用户空间映射的时间页
```

**查看vDSO**：
```bash
ldd /bin/ls
# linux-vdso.so.1 => (0x00007ffe123ab000)  # vDSO
# libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
```

## 8. 常见面试题

### Q1: 进程和线程的区别？

**答案**：
- **进程**：资源分配单位，独立地址空间，通信需IPC
- **线程**：CPU调度单位，共享地址空间，通信通过全局变量
- **开销**：进程创建/切换开销大（3-5ms），线程小（<1μs）
- **稳定性**：进程崩溃不影响其他进程，线程崩溃整个进程崩溃

### Q2: 虚拟内存的作用？

**答案**：
1. **隔离**：每个进程独立地址空间
2. **扩展**：虚拟内存可大于物理内存（swap）
3. **共享**：多个进程共享同一物理页（如libc.so）
4. **保护**：非法访问触发缺页中断

### Q3: epoll为什么比select快？

**答案**：
1. **无fd数量限制**：select限制1024，epoll无限制
2. **O(1)复杂度**：select需遍历所有fd（O(n)），epoll只遍历就绪fd（O(1)）
3. **无内存拷贝**：select每次拷贝fd_set，epoll共享内存

### Q4: 零拷贝的原理？

**答案**：
- **传统拷贝**：磁盘→内核→用户空间→Socket→网卡（4次）
- **sendfile**：磁盘→内核→网卡（2次），无需拷贝到用户空间
- **mmap**：将文件映射到内存，直接访问，无需read()

### Q5: TCP三次握手，为什么不是两次？

**答案**：
- **防止旧连接**：如果只有两次，旧的SYN包可能导致错误建立连接
- **确认双方都能收发**：
  - 第1次：客户端发送能力OK
  - 第2次：服务端接收+发送能力OK
  - 第3次：客户端接收能力OK

## 总结

**操作系统核心知识点**：

1. **进程/线程**：调度（CFS/RT）、IPC（管道/共享内存/信号）、GMP模型
2. **内存**：虚拟内存、多级页表、TLB、Buddy System、TCMalloc
3. **IO**：epoll vs select、零拷贝（sendfile/mmap）、Direct IO
4. **文件系统**：inode、page cache、fsync、ext4布局
5. **网络**：TCP握手、拥塞控制、Socket优化
6. **Cgroup资源隔离**：CPU（cpuset/cpuquota/cpushares）、内存（OOM/压力通知）、IO（blkio/调度算法）、网络（tc/HTB）、V2统一层级与PSI

**性能优化方向**：
- 减少系统调用（批量操作、vDSO）
- 减少内存拷贝（零拷贝）
- 利用page cache
- Cgroup资源隔离与限制
- 异步IO

掌握这些底层原理，能帮你设计出高性能、低延迟的系统！🚀
