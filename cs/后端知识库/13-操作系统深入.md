## 1. 进程与线程管理

### 1.1 进程vs线程

| 维度 | 进程 | 线程 |
|------|------|------|
| **定义** | 资源分配的基本单位 | CPU调度的基本单位 |
| **内存** | 独立地址空间 | 共享地址空间 |
| **通信** | IPC（管道、消息队列、共享内存） | 直接访问全局变量 |
| **开销** | 创建/切换开销大（3-5ms） | 创建/切换开销小（<1μs） |
| **稳定性** | 一个进程崩溃不影响其他进程 | 一个线程崩溃整个进程崩溃 |

### 1.2 进程状态转换

```
┌──────┐  fork()  ┌─────────┐  调度  ┌─────────┐
│ 创建 │ ───────→ │ 就绪队列 │ ─────→ │  运行   │
└──────┘          └─────────┘        └─────────┘
                       ↑                   │
                       │                   │ 阻塞（IO/锁）
                       │                   ↓
                       │              ┌─────────┐
                       └──────────────│  阻塞   │
                          唤醒         └─────────┘
```

**查看进程状态**：
```bash

# R: Running（运行）
# S: Sleeping（睡眠）

# 示例

```

### 1.3 进程调度算法

#### 1.3.1 CFS（Completely Fair Scheduler）
Linux默认调度器，核心思想：**让每个进程获得公平的CPU时间**

```c
// 虚拟运行时间（vruntime）
vruntime += delta_exec * (NICE_0_LOAD / weight)

// weight根据nice值计算：
// nice=-20（最高优先级）：weight=88761
// nice=0（默认）：weight=1024
// nice=19（最低优先级）：weight=15
```

**示例**：
```bash
# 进程A：nice=0, weight=1024
# 进程B：nice=5, weight=335
# 
# 运行10ms后：
# A的vruntime += 10 * (1024/1024) = 10ms
# B的vruntime += 10 * (1024/335) = 30.5ms
# 
# 下次调度选择vruntime最小的（进程A）
```

**调整优先级**：
```bash
# 降低优先级（nice值越大，优先级越低）
nice -n 10 ./myapp

# 提高优先级（需要root）
nice -n -10 ./myapp

# 修改正在运行的进程
renice -n 5 -p 1234
```

#### 1.3.2 实时调度（RT）
```bash
# 查看实时进程
ps -eo pid,class,rtprio,comm | grep RR
# PID  CLS  RTPRIO  COMMAND
# 123  RR   50      realtime-app

# 设置实时优先级（0-99，99最高）
chrt -r -p 50 1234
```

### 1.4 进程间通信（IPC）

#### 1.4.1 管道（Pipe）
```go
// 匿名管道（只能用于父子进程）
cmd := exec.Command("ls", "/var/log")
stdout, _ := cmd.StdoutPipe()
cmd.Start()

scanner := bufio.NewScanner(stdout)
for scanner.Scan() {
    fmt.Println(scanner.Text())
}
```

**命名管道（FIFO）**：
```bash
# 创建命名管道
mkfifo /tmp/mypipe

# 进程A：写入
echo "hello" > /tmp/mypipe

# 进程B：读取
cat /tmp/mypipe  # 输出: hello
```

#### 1.4.2 共享内存（最快的IPC）
```c
#include <sys/shm.h>

// 进程A：创建共享内存
int shmid = shmget(IPC_PRIVATE, 1024, IPC_CREAT | 0666);
char *shmaddr = (char *)shmat(shmid, NULL, 0);
strcpy(shmaddr, "hello from process A");

// 进程B：读取共享内存
char *shmaddr = (char *)shmat(shmid, NULL, 0);
printf("%s\n", shmaddr);  // 输出: hello from process A

// 清理
shmdt(shmaddr);
shmctl(shmid, IPC_RMID, NULL);
```

**Go语言使用mmap**：
```go
import "golang.org/x/exp/mmap"

// 进程A：写入
f, _ := os.Create("/tmp/shared.dat")
f.Write([]byte("hello"))
f.Close()

// 进程B：mmap读取
at, _ := mmap.Open("/tmp/shared.dat")
data := make([]byte, 5)
at.ReadAt(data, 0)
fmt.Println(string(data))  // 输出: hello
```

```go
    "os"
    "os/signal"
    "syscall"

}
```

### 1.5 线程模型

#### 1.5.1 用户级线程 vs 内核级线程

| 类型 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| **用户级线程** | 用户空间调度（Go runtime） | 轻量、切换快 | 无法利用多核 |
| **内核级线程** | 内核调度（pthread） | 利用多核 | 创建/切换开销大 |
| **混合模型** | M:N映射（Go的M:P:G） | 兼顾两者 | 实现复杂 |

#### 1.5.2 Go的GMP模型

```
┌─────────────────────────────────────┐
│         G (Goroutine)               │
│  轻量级线程，2KB初始栈               │
└──────────┬──────────────────────────┘
           │ 调度
┌──────────↓──────────────────────────┐
│         P (Processor)               │
│  逻辑处理器，GOMAXPROCS数量          │
└──────────┬──────────────────────────┘
           │ 绑定
┌──────────↓──────────────────────────┐
│         M (Machine)                 │
│  内核线程，真正执行代码               │
└─────────────────────────────────────┘
```

**工作窃取（Work Stealing）**：
```go
// P1的G队列：[G1, G2, G3]
// P2的G队列：[]（空闲）

// P2会从P1偷一半任务
// P1的G队列：[G1]
// P2的G队列：[G2, G3]
```

**查看Goroutine数量**：
```go
import "runtime"

func main() {
    fmt.Println("Goroutines:", runtime.NumGoroutine())
    
    // 创建10000个goroutine
    for i := 0; i < 10000; i++ {
        go func() {
            time.Sleep(1 * time.Hour)
        }()
    }
    
    fmt.Println("Goroutines:", runtime.NumGoroutine())  // 输出: 10001
}
```

### 1.6 僵尸进程与孤儿进程

**僵尸进程（Zombie）**：

```
产生原因：
- 子进程退出后，父进程未调用wait()回收

状态：
- 进程已结束，但进程表项未删除
- 占用PID资源（PID有限）

查看僵尸进程：
ps aux | grep Z
# USER  PID  STAT  COMMAND
# root  123  Z     [myapp] <defunct>

解决方法：
1. 父进程调用wait()/waitpid()
2. 杀死父进程（僵尸进程变成孤儿进程，由init回收）
```

**孤儿进程**：

```
产生原因：
- 父进程退出，子进程仍在运行

处理方式：
- 由init进程（PID=1）收养
- init进程会调用wait()回收

孤儿进程是正常的，不会造成资源泄漏
```

**Go代码示例**：

```go
// ❌ 可能产生僵尸进程
func badExample() {
    cmd := exec.Command("sleep", "10")
    cmd.Start()
    // 忘记调用 cmd.Wait()
}

// ✅ 正确做法
func goodExample() {
    cmd := exec.Command("sleep", "10")
    cmd.Start()
    cmd.Wait()  // 回收子进程
}
```

### 1.7 上下文切换

**什么是上下文切换？**

```
进程上下文切换：
1. 保存当前进程的CPU寄存器、程序计数器、栈指针
2. 保存页表基址（TLB失效！）
3. 切换到新进程的地址空间
4. 恢复新进程的CPU状态

开销：约3-5微秒（主要是TLB失效）

线程上下文切换：
- 同一进程内切换，无需切换地址空间
- TLB不失效
- 开销约1微秒
```

**查看上下文切换次数**：

```bash
# 系统级别
vmstat 1
# procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
#  r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
#  1  0      0 1234567  1234  12345    0    0     0     0  100  500  5  2 93  0  0
#                                                              ↑
#                                                      context switches/s

# 进程级别
pidstat -w -p 1234 1
# PID   cswch/s nvcswch/s  Command
# 1234  100     50         myapp
#       ↑       ↑
#       自愿切换（等待IO/锁）
#               非自愿切换（时间片用完）
```

**减少上下文切换**：

```go
// ❌ 频繁切换（阻塞式）
for i := 0; i < 10000; i++ {
    go func() {
        time.Sleep(1 * time.Second)  // 每次都切换
    }()
}

// ✅ 批量处理（减少切换）
go func() {
    for i := 0; i < 10000; i++ {
        // 批量处理，减少goroutine数量
    }
}()
```

### 1.8 CPU缓存与内存屏障⭐⭐

**CPU缓存层级**：

```
CPU核心
  ↓
L1 Cache（32-64KB，1-2周期）← 每核独占
  ↓
L2 Cache（256KB-1MB，10周期）← 每核独占
  ↓
L3 Cache（8-64MB，30周期）← 多核共享
  ↓
主内存（100周期）

速度：L1 > L2 > L3 > 内存
延迟：1ns < 4ns < 15ns < 100ns
```

**缓存行（Cache Line）**：

```
- 缓存的最小单位：64字节
- 读取一个字节，会加载整个缓存行

伪共享（False Sharing）问题：
// ❌ 两个变量在同一缓存行，频繁失效
type Counter struct {
    a int64  // 8字节
    b int64  // 8字节
}
// a和b可能在同一缓存行（64字节）
// 线程1修改a，导致线程2的b缓存失效

// ✅ 缓存行填充（Padding）
type Counter struct {
    a int64
    _ [56]byte  // 填充到64字节
    b int64
    _ [56]byte
}
```

**MESI缓存一致性协议**：

```
四种状态：
M (Modified)：已修改，独占，需写回
E (Exclusive)：独占，未修改
S (Shared)：共享，只读
I (Invalid)：无效

状态转换示例：
1. CPU0读取变量X → X在CPU0为E状态
2. CPU1读取变量X → X在CPU0、CPU1都为S状态
3. CPU0修改变量X → X在CPU0为M状态，CPU1为I状态
4. CPU1读取变量X → CPU0写回内存，X在两者都为S状态
```

**内存屏障（Memory Barrier）**：

```go
// CPU和编译器可能重排指令

// 示例：双重检查锁定（DCL）
var instance *Singleton
var once sync.Once

func GetInstance() *Singleton {
    once.Do(func() {
        instance = &Singleton{}  // 可能重排为：分配内存→赋值→初始化
    })
    return instance
}

// Go的sync.Once内部使用atomic操作保证可见性
// atomic.StorePointer 隐含内存屏障
```

**查看缓存命中率**：

```bash
perf stat -e cache-references,cache-misses ./myapp
# 1,000,000 cache-references
# 50,000    cache-misses   # 5% miss rate
```

## 2. 内存管理

### 2.1 虚拟内存

**为什么需要虚拟内存？**
1. **隔离**：每个进程有独立地址空间，互不干扰
2. **扩展**：虚拟内存可以大于物理内存（通过swap）
3. **共享**：多个进程共享同一物理页（如libc.so）

**地址转换**：
```
虚拟地址 (48位) = 页号 (36位) + 页内偏移 (12位)
                  ↓ 查页表
物理地址 (48位) = 页框号 (36位) + 页内偏移 (12位)
```

**查看进程内存布局**：
```bash
cat /proc/1234/maps
# 输出：
# 地址范围              权限  偏移  设备  inode  路径
# 00400000-00452000  r-xp  0     08:01  123    /usr/bin/myapp  # 代码段
# 00651000-00652000  r--p  51000 08:01  123    /usr/bin/myapp  # 只读数据
# 00652000-00653000  rw-p  52000 08:01  123    /usr/bin/myapp  # 读写数据
# 7ffff7a0d000-...   r-xp  0     08:01  456    /lib/libc.so.6  # 共享库
# 7ffffffde000-...   rw-p  0     00:00  0      [stack]         # 栈
# 7ffff7ff9000-...   rw-p  0     00:00  0      [heap]          # 堆
```

### 2.2 页表与TLB

**多级页表（x86_64）**：
```
虚拟地址 (48位)
├─ PGD索引 (9位) ────→ 页全局目录
├─ PUD索引 (9位) ────→ 页上层目录
├─ PMD索引 (9位) ────→ 页中间目录
├─ PTE索引 (9位) ────→ 页表项
└─ 页内偏移 (12位) ──→ 物理页内偏移
```

**TLB（Translation Lookaside Buffer）**：
- 缓存虚拟地址→物理地址的映射
- 命中时无需查页表（快100倍）
- TLB Miss时触发Page Walk（查多级页表）

**查看TLB Miss率**：
```bash
perf stat -e dTLB-load-misses,dTLB-loads ./myapp
# 输出：
# 1,234,567  dTLB-loads
# 12,345     dTLB-load-misses  # Miss率=1%
```

**优化TLB Miss**：
```c
// ❌ 跨步访问（TLB Miss多）
for (int i = 0; i < N; i += 1024) {
    sum += arr[i];
}

// ✅ 顺序访问（TLB命中高）
for (int i = 0; i < N; i++) {
    sum += arr[i];
}
```

### 2.3 缺页中断（Page Fault）

**分类**：
1. **Minor Page Fault**：页在内存中，但未映射到页表
2. **Major Page Fault**：页不在内存，需要从磁盘读取

**查看Page Fault**：
```bash

```

**减少Page Fault**：
```go

    slice = append(slice, i)
}
```

### 2.4 内存分配器

#### 2.4.1 Buddy System（Linux）
```
伙伴系统：内存块大小为2^n（4KB、8KB、16KB...）

示例：申请6KB
1. 找到最小的满足条件的块（8KB）
2. 如果没有8KB，则分裂16KB块 → 两个8KB块
3. 分配一个8KB块，另一个8KB块放回空闲链表
```

**查看内存碎片**：
```bash

# Node 0, zone DMA      1   1   1   0   2   1   1   0   1   1   3
# Node 0, zone Normal   100 50  20  10  5   2   1   0   0   0   0
#                       4KB 8KB 16K 32K 64K 128K...
```

#### 2.4.2 TCMalloc（Thread-Caching Malloc）
Go、Chrome使用

```
┌────────────────────────┐
│   Thread Cache         │  线程私有，无锁
│  小对象(<32KB)快速分配  │
└───────┬────────────────┘
        │ 不够时申请
┌───────↓────────────────┐
│   Central Cache        │  全局，有锁
│  中等对象(32KB-256KB)   │
└───────┬────────────────┘
        │ 不够时申请
┌───────↓────────────────┐
│   Page Heap            │  全局，管理大块内存
│  大对象(>256KB)         │
└────────────────────────┘
```

**优势**：
- 小对象分配无锁（线程缓存）
- 减少内存碎片（size class）

### 2.5 内存泄漏排查

## 3. IO模型与异步IO

### 3.1 五种IO模型

| 模型 | 阻塞 | 说明 |
|------|------|------|
| **阻塞IO** | ✅ | read()阻塞直到数据到达 |
| **非阻塞IO** | ❌ | read()立即返回，需轮询 |
| **IO多路复用** | ✅ | select/epoll阻塞，监听多个fd |
| **信号驱动IO** | ❌ | 数据到达时发送信号 |
| **异步IO** | ❌ | 内核完成IO后通知应用 |

#### 3.1.1 阻塞IO
```c
// 阻塞直到收到数据
char buf[1024];
int n = read(sockfd, buf, sizeof(buf));  // 阻塞
printf("received: %s\n", buf);
```

#### 3.1.2 非阻塞IO
```c
// 设置为非阻塞
fcntl(sockfd, F_SETFL, O_NONBLOCK);

// 轮询
while (1) {
    int n = read(sockfd, buf, sizeof(buf));
    if (n > 0) {
        printf("received: %s\n", buf);
        break;
    } else if (errno == EAGAIN) {
        // 数据还没到，继续轮询
        usleep(1000);
    }
}
```

#### 3.1.3 IO多路复用（epoll）
```c
int epfd = epoll_create1(0);

// 添加监听的fd
struct epoll_event ev;
ev.events = EPOLLIN;
ev.data.fd = sockfd;
epoll_ctl(epfd, EPOLL_CTL_ADD, sockfd, &ev);

// 等待事件
struct epoll_event events[10];
while (1) {
    int nfds = epoll_wait(epfd, events, 10, -1);  // 阻塞
    for (int i = 0; i < nfds; i++) {
        if (events[i].events & EPOLLIN) {
            // 可读
            int fd = events[i].data.fd;
            read(fd, buf, sizeof(buf));
        }
    }
}
```

**Go的netpoller**：
```go
// Go运行时自动使用epoll（Linux）/kqueue（macOS）
listener, _ := net.Listen("tcp", ":8080")
for {
    conn, _ := listener.Accept()  // 底层使用epoll
    go handleConn(conn)
}
```

### 3.2 epoll vs select

| 特性 | select | epoll |
|------|--------|-------|
| **fd数量限制** | 1024 | 无限制 |
| **性能** | O(n) | O(1) |
| **内存拷贝** | 每次拷贝fd_set | 共享内存 |
| **适用场景** | 少量连接 | 高并发（C10K） |

**select缺陷**：
```c
fd_set readfds;
FD_ZERO(&readfds);
FD_SET(fd1, &readfds);
FD_SET(fd2, &readfds);
// ... 添加1000个fd

select(maxfd+1, &readfds, NULL, NULL, NULL);

// 需要遍历所有fd，找到就绪的（O(n)）
for (int fd = 0; fd <= maxfd; fd++) {
    if (FD_ISSET(fd, &readfds)) {
        read(fd, buf, sizeof(buf));
    }
}
```

**epoll优化**：
```c
epoll_wait(epfd, events, 10, -1);  // 返回就绪的events

// 只遍历就绪的fd（O(1)）
for (int i = 0; i < nfds; i++) {
    int fd = events[i].data.fd;
    read(fd, buf, sizeof(buf));
}
```

### 3.3 零拷贝（Zero Copy）

#### 3.3.1 传统拷贝（4次拷贝）
```
磁盘 → 内核缓冲区（DMA拷贝）
     → 用户空间（CPU拷贝）
     → Socket缓冲区（CPU拷贝）
     → 网卡（DMA拷贝）
```

#### 3.3.2 sendfile（2次拷贝）
```c
// Linux sendfile
int filefd = open("file.txt", O_RDONLY);
int sockfd = socket(AF_INET, SOCK_STREAM, 0);

// 直接从文件fd发送到socket fd（零拷贝）
sendfile(sockfd, filefd, NULL, file_size);

// 内核直接传输，无需拷贝到用户空间
// 磁盘 → 内核缓冲区 → 网卡
```

**Go语言**：
```go
// Go自动使用sendfile优化（Linux）
func ServeFile(w http.ResponseWriter, r *http.Request) {
    f, _ := os.Open("large_file.dat")
    defer f.Close()
    io.Copy(w, f)  // 底层使用sendfile
}
```

#### 3.3.3 mmap（内存映射）
```c
// 将文件映射到内存
int fd = open("file.txt", O_RDONLY);
char *data = mmap(NULL, file_size, PROT_READ, MAP_PRIVATE, fd, 0);

// 直接访问，像访问内存一样
printf("%s\n", data);  // 无需read()

munmap(data, file_size);
```

**优点**：
- 避免read()系统调用
- 多个进程共享同一物理页
- 内核自动管理缓存

**Nginx使用mmap**：
```nginx
location /static/ {
    # 开启sendfile和mmap
    sendfile on;
    tcp_nopush on;
}
```

### 3.4 直接IO（Direct IO）

**绕过页缓存，直接读写磁盘**：
```c
int fd = open("file.txt", O_DIRECT | O_RDONLY);
// 要求：buf地址对齐、大小为512字节的倍数
char *buf = aligned_alloc(512, 4096);
read(fd, buf, 4096);
```

**使用场景**：
- 数据库（自己管理缓存，不需要页缓存）
- 流媒体（大文件，不需要缓存）

**对比**：
```bash

dd if=/dev/zero of=test bs=1M count=1000
# 1000+0 records in
# 1000+0 records out
# 1048576000 bytes (1.0 GB) copied, 0.5 s, 2.0 GB/s

# Direct IO（绕过缓存）
dd if=/dev/zero of=test oflag=direct bs=1M count=1000
# 1000+0 records in
# 1000+0 records out
# 1048576000 bytes (1.0 GB) copied, 2.0 s, 500 MB/s
```

## 6. Cgroup资源隔离

> **Cgroup（Control Groups）** 是Linux内核提供的资源限制、控制与隔离机制，是容器技术（Docker/K8s）的核心基石。

### 6.1 Cgroup概述

**四大资源子系统**：
| 子系统 | 控制对象 | 关键文件/工具 |
|--------|----------|---------------|
| **CPU** | 计算时间、核心绑定 | cpuset, cpu.cfs_quota_us, cpu.shares |
| **Memory** | 内存用量、Swap | memory.limit_in_bytes, memory.oom_control |
| **BlkIO** | 磁盘带宽、IOPS | blkio.throttle.*, blkio.weight |
| **Net** | 网络带宽 | net_cls + tc（流量控制） |

**查看系统Cgroup版本**：
```bash
# Cgroup V1（传统，各子系统独立挂载）
mount | grep cgroup
# cgroup on /sys/fs/cgroup/cpu type cgroup (rw,...,cpu,cpuacct)
# cgroup on /sys/fs/cgroup/memory type cgroup (rw,...,memory)

# Cgroup V2（unified hierarchy，单一挂载点）
mount | grep cgroup2
# cgroup2 on /sys/fs/cgroup type cgroup2 (rw,...,nsdelegate)
```

### 6.2 CPU资源隔离

Linux通过CFS（完全公平调度器）实现CPU时间的分配，Cgroup提供三种隔离方式：

#### 6.2.1 cpuset — 核心绑定

将进程绑定到指定CPU核心运行，实现**物理隔离**：

```bash
# 查看/设置cgroup绑定的CPU核心
cat /sys/fs/cgroup/cpuset/mygroup/cpuset.cpus
echo "0-3" > /sys/fs/cgroup/cpuset/mygroup/cpuset.cpus

# 必须同时设置内存节点（NUMA）
echo "0" > /sys/fs/cgroup/cpuset/mygroup/cpuset.mems
```

**超线程陷阱**：在开启超线程的服务器上，需注意核心编号的物理对应关系：
```bash
# 查看物理核心与逻辑核心的映射
cat /proc/cpuinfo | grep -e "core id" -e "physical id"
# physical id: 0, core id: 0  → 逻辑核心 0 和 12 是同一物理核心
# 绑定 0-5 和 12-17 才是"半数物理核心"
# 绑定 0-11 看似一半，实际覆盖了所有物理核心（超线程部分无法倍增算力）
```

**实验数据对比**（24逻辑核心服务器，筛质数测试）：
| 配置 | 核心数 | 执行时间 |
|------|--------|----------|
| 0-23（全部） | 24 | 8.9s |
| 0-11（全物理核心） | 12 | 9.5s ≈ 无变化！ |
| 0-5,12-17（半物理核心） | 12 | 17.8s ≈ 翻倍 |

#### 6.2.2 cpuquota — 配额限制（硬限制）

按时间比例限制CPU使用上限，**即使系统空闲也不能超额**：

```bash
# 时间周期（默认100ms = 100000μs）
cat /sys/fs/cgroup/cpu/mygroup/cpu.cfs_period_us   # 100000

# 配额（-1表示不限制）
# 限制单核50%：echo 50000
# 限制4核满载：echo 400000
# 限制24核50%：echo 1200000
echo "50000" > /sys/fs/cgroup/cpu/mygroup/cpu.cfs_quota_us
```

**计算公式**：`quota = 目标百分比 × period × CPU核心数`

#### 6.2.3 cpushares — 权重比例（弹性限制）

按权重比例分配CPU时间，**系统空闲时可借用其他cgroup的资源**：

```bash
# 权重范围：2-262144，默认1024
echo "1024" > /sys/fs/cgroup/cpu/groupA/cpu.shares  # A
echo "2048" > /sys/fs/cgroup/cpu/groupB/cpu.shares  # B

# 效果：
# 竞争时：A占1/3，B占2/3
# B空闲时：A可占满所有CPU
```

**三种方式对比**：
| 特性 | cpuset | cpuquota | cpushares |
|------|--------|----------|-----------|
| **隔离类型** | 物理核心隔离 | 时间配额硬限制 | 时间权重弹性分配 |
| **空闲借用** | ❌ 不能 | ❌ 不能 | ✅ 可以 |
| **最小粒度** | 1个核心 | 任意百分比 | 任意比例 |
| **隔离强度** | 最强（核心独占） | 强（硬上限） | 弱（按比例） |
| **适用场景** | 延迟敏感型、NUMA | 多租户硬隔离 | 弹性共享、超卖 |

**内核资源冲突的影响**：当多个cgroup的进程访问相同的内核资源（如同一文件的自旋锁竞争），cpuset由于物理隔离表现最优，cpuquota在此场景下性能损失最大。

### 6.3 内存资源隔离

#### 6.3.1 Linux内存管理基础回顾

```
进程申请内存 → malloc/mmap（仅分配虚拟地址）
     ↓ 实际访问时
缺页异常（Page Fault）
     ├─ Minor Fault：页在内存但未映射（无磁盘IO）
     └─ Major Fault：页在Swap/磁盘（需磁盘IO）
     ↓
alloc_pages → 伙伴系统分配物理页
     ↓ 内存不足时进入 slowpath
1. 唤醒 kswapd 交换守护进程
2. 尝试回收 Page Cache（进程置D状态）
3. 触发 OOM Killer 杀进程
4. 循环重试
```

#### 6.3.2 内存限制配置

```bash
# 硬限制：内存使用上限（超出触发OOM或挂起）
echo "512M" > /sys/fs/cgroup/memory/mygroup/memory.limit_in_bytes

# 内存+Swap总量限制（设为与limit相同 = 禁用Swap）
echo "512M" > /sys/fs/cgroup/memory/mygroup/memory.memsw.limit_in_bytes

# 软限制：用于内存超卖场景，全局内存紧张时按此值回收
echo "256M" > /sys/fs/cgroup/memory/mygroup/memory.soft_limit_in_bytes

# 查看当前使用量
cat /sys/fs/cgroup/memory/mygroup/memory.usage_in_bytes
```

#### 6.3.3 OOM控制

```bash
cat /sys/fs/cgroup/memory/mygroup/memory.oom_control
# oom_kill_disable 0   → 0=启用OOM Killer（默认），1=禁用（进程挂起为D状态）
# under_oom 0          → 0=正常，1=当前已OOM

# 禁用OOM Killer（进程挂起而非被杀）
echo 1 > /sys/fs/cgroup/memory/mygroup/memory.oom_control
# 恢复方式：增加内存限制、杀进程、迁移进程到其他cgroup
```

#### 6.3.4 内存压力通知（eventfd）

```c
// 通过 eventfd + memory.pressure_level 实现内存压力监控
int efd = eventfd(0, 0);
int cfd = open("/sys/fs/cgroup/memory/mygroup/memory.pressure_level", O_RDONLY);
// 写入 cgroup.event_control: "<efd> <cfd> <level>"
// level: "low" | "medium" | "critical"

// 压力级别：
// low     → 开始内存回收
// medium  → 开始Swap换出、活跃Cache写回
// critical → 触发OOM Killer
```

### 6.4 IO资源隔离

#### 6.4.1 Linux IO体系7层架构

```
应用程序 read()/write()
    ↓
① VFS层         — 虚拟文件系统，统一接口
    ↓
② 文件系统层    — ext4/xfs/btrfs 具体实现
    ↓
③ 页缓存层      — Page Cache（Buffered IO在此缓冲）
    ↓
④ 通用块层      — Block IO请求标准化 ← 【带宽/IOPS限速在此实现】
    ↓
⑤ IO调度层      — cfq/deadline/noop ← 【权重限速在此实现】
    ↓
⑥ 块设备驱动层  — 设备操作接口
    ↓
⑦ 块设备层      — 物理设备
```

#### 6.4.2 IO调度算法

| 算法 | 特点 | 适用场景 |
|------|------|----------|
| **cfq** | 完全公平，按进程分时间片 | 通用服务器，支持cgroup权重 |
| **deadline** | 保证请求最终期限，防止饿死 | 数据库等IO密集型 |
| **noop** | 无调度，仅FIFO合并 | SSD固态硬盘（最佳） |

```bash
# 查看/切换调度算法
cat /sys/block/sda/queue/scheduler    # noop deadline [cfq]
echo deadline > /sys/block/sda/queue/scheduler
```

#### 6.4.3 blkio限速配置

```bash
# 带宽限制（基于通用块层，需 Direct IO 才对应用可感知）
echo "8:0 1048576" > /sys/fs/cgroup/blkio/mygroup/blkio.throttle.write_bps_device  # 写1MB/s
echo "8:0 1048576" > /sys/fs/cgroup/blkio/mygroup/blkio.throttle.read_bps_device   # 读1MB/s

# IOPS限制
echo "8:0 100" > /sys/fs/cgroup/blkio/mygroup/blkio.throttle.write_iops_device

# 权重限制（需cfq调度器，同样需 Direct IO）
echo 500 > /sys/fs/cgroup/blkio/mygroup/blkio.weight  # 范围10-1000
```

**重要限制**：Cgroup V1的blkio对 **Buffered IO无法限速**！因为Buffered IO的write-back由内核异步完成，无法追踪所属cgroup。此问题在Cgroup V2中已解决。

### 6.5 网络资源隔离

Cgroup网络隔离复用Linux的 **tc（Traffic Control）** 机制，通过 `net_cls` 子系统给数据包打标签，配合tc队列规则实现限速。

#### 6.5.1 tc队列规则（qdisc）概览

| 类型 | qdisc | 核心机制 | 特点 |
|------|-------|----------|------|
| **不可分类** | pfifo_fast | 3优先级FIFO | 旧版默认，不可调参 |
| **不可分类** | fq_codel | 公平队列+CoDel | **新版默认**，解决Bufferbloat |
| **不可分类** | tbf | 令牌桶 | 简单限速 |
| **可分类** | htb | 分层令牌桶 | **推荐**，支持带宽借用和分类 |

**Bufferbloat问题**：过大的网络缓冲区导致数据包驻留时间超过超时阈值，引发全部请求超时失败，系统看似满载实则不可用。CoDel算法通过控制**驻留时间**而非队列长度来解决此问题。

#### 6.5.2 使用cgroup限制网络带宽

```bash
# 1. 创建cgroup并标记classid
mkdir /sys/fs/cgroup/net_cls/mygroup
echo 0x00010010 > /sys/fs/cgroup/net_cls/mygroup/net_cls.classid
# 0xAAAABBBB → AAAA:BBBB = 1:10

# 2. 配置HTB队列规则
tc qd add dev eth0 root handle 1: htb default 100
tc cl add dev eth0 parent 1: classid 1:1 htb rate 100mbit burst 20k
tc cl add dev eth0 parent 1:1 classid 1:10 htb rate 10mbit ceil 50mbit burst 20k
tc qd add dev eth0 parent 1:10 handle 10: fq_codel

# 3. 添加cgroup过滤器
tc fi add dev eth0 parent 1: protocol ip prio 1 handle 1: cgroup

# 4. 将进程加入cgroup
echo $PID > /sys/fs/cgroup/net_cls/mygroup/tasks
```

## 8. Linux内核深入

> 本节覆盖Linux内核的核心机制，包括内存寻址模型、中断与异常处理、内核同步机制、内核抢占、lockup检测、D状态与load计算、系统调用传参方式等，帮助深入理解操作系统底层行为。

### 8.1 内存寻址：物理地址、逻辑地址、线性地址

**三种地址的关系**：

```
逻辑地址（Logical Address）
    = 段选择符(Segment Selector) : 段内偏移(Offset)
        ↓ 分段机制（Segmentation）
线性地址（Linear Address / 虚拟地址）
    = 段基址 + 偏移量
        ↓ 分页机制（Paging）
物理地址（Physical Address）
    = 页框号 + 页内偏移
```

**x86_64下的实际情况**：
- Linux在x86_64下**几乎绕过了分段**：所有段基址设为0，段限制设为最大值
- 因此：逻辑地址 ≈ 线性地址 ≈ 虚拟地址（平坦内存模型）
- 分页是实际使用的地址转换机制

#### 分段（Segmentation）

```
# x86中的段描述符（GDT / LDT）
段描述符包含：
- Base Address：段基地址（32位 or 64位下为0）
- Limit：段界限
- DPL：描述符特权级（0=内核，3=用户）
- Type：代码段/数据段/系统段

# Linux的平坦模型：
__KERNEL_CS: base=0, limit=0xFFFFFFFF, DPL=0  # 内核代码段
__KERNEL_DS: base=0, limit=0xFFFFFFFF, DPL=0  # 内核数据段
__USER_CS:   base=0, limit=0xFFFFFFFF, DPL=3  # 用户代码段
__USER_DS:   base=0, limit=0xFFFFFFFF, DPL=3  # 用户数据段

# 所有段基地址为0 → 逻辑偏移 = 线性地址
```

#### 分页（Paging）

**x86_32 vs x86_64页表结构**：

| 特性 | x86_32（普通） | x86_32（PAE） | x86_64 |
|------|-------------|-------------|--------|
| **级数** | 2级 | 3级 | 4级（PML4） |
| **虚拟地址位** | 32位（4GB） | 32位（4GB） | 48位（256TB） |
| **物理地址位** | 32位（4GB） | 36位（64GB） | 52位（4PB） |
| **页表项大小** | 4字节 | 8字节 | 8字节 |
| **目录结构** | PGD→PTE | PGD→PMD→PTE | PGD→PUD→PMD→PTE |

**x86_64 四级页表**：
```
虚拟地址(48位) = PGD[9] + PUD[9] + PMD[9] + PTE[9] + Offset[12]

页表项标记位（PTE flags）：
- P (Present)：页是否在内存中
- R/W (Read/Write)：读写权限
- U/S (User/Supervisor)：用户态/内核态访问权限
- A (Accessed)：是否被访问过（供LRU页面回收参考）
- D (Dirty)：是否被修改过（决定是否需要写回磁盘）
- NX (No Execute)：禁止执行（安全防护，防止栈溢出攻击）
```

**页表切换**：
```c
// 进程切换时，通过加载CR3寄存器切换页表
// CR3 存储当前进程的PGD（页全局目录）物理地址
mov cr3, <新进程PGD物理地址>

// 切换CR3会导致TLB全部失效！
// 这是进程上下文切换开销大的主要原因之一

// 优化：PCID（Process Context ID）
// 支持PCID的CPU可以为不同进程的TLB条目打标签
// 切换CR3时不需要刷新整个TLB
```

### 8.2 中断与异常处理

#### 中断 vs 异常

| 类型 | 来源 | 示例 |
|------|------|------|
| **硬中断（Hardware Interrupt）** | 外部设备 | 网卡收包、磁盘IO完成、时钟中断 |
| **软中断（Software Interrupt）** | 内核自身 | NET_RX_SOFTIRQ、TIMER_SOFTIRQ |
| **异常（Exception）** | CPU执行指令时 | 缺页异常(#PF)、除零(#DE)、越界(#GP) |

#### 中断处理完整流程

```
硬件设备产生中断
    ↓
中断控制器（APIC）→ 选择目标CPU
    ↓
CPU暂停当前任务
    ↓ 保存现场（压栈：RIP/CS/RFLAGS/RSP/SS）
    ↓
查IDT（中断描述符表）→ 找到中断处理函数入口
    ↓
┌─────────────────────────────────┐
│ 上半部（Top Half / Hardirq）    │
│ - 关中断执行，必须快速完成      │
│ - 读取设备状态、清中断标志      │
│ - 将数据放入缓冲区             │
│ - 如需延迟处理，标记下半部      │
└─────────────────────────────────┘
    ↓ 退出上半部，开中断
┌─────────────────────────────────┐
│ 下半部（Bottom Half）           │
│ 负责耗时的后续处理              │
│ - softirq / tasklet / workqueue │
└─────────────────────────────────┘
    ↓
恢复现场，继续执行被中断的任务
```

#### 三种下半部机制对比

| 机制 | 执行上下文 | 并发性 | 可否睡眠 | 适用场景 |
|------|-----------|--------|---------|---------|
| **softirq** | 软中断上下文 | 同类型可多CPU并发 | ❌ 不可 | 网络收发包(NET_RX/TX)、块设备 |
| **tasklet** | 软中断上下文 | 同实例不可并发 | ❌ 不可 | 驱动中较简单的延迟处理 |
| **workqueue** | 内核线程 | 可并发 | ✅ 可以 | 需要睡眠的操作（分配内存、IO等） |

```c
// softirq示例（网络收包）
// 定义在 kernel/softirq.c
enum {
    HI_SOFTIRQ = 0,           // 高优先级tasklet
    TIMER_SOFTIRQ,             // 定时器
    NET_TX_SOFTIRQ,            // 网络发送
    NET_RX_SOFTIRQ,            // 网络接收 ← 最常见
    BLOCK_SOFTIRQ,             // 块设备
    TASKLET_SOFTIRQ,           // 普通tasklet
    SCHED_SOFTIRQ,             // 调度器
    RCU_SOFTIRQ,               // RCU回调
};

// tasklet示例
struct tasklet_struct my_tasklet;
void my_tasklet_handler(unsigned long data) {
    // 延迟处理逻辑（不可睡眠）
}
tasklet_init(&my_tasklet, my_tasklet_handler, 0);
tasklet_schedule(&my_tasklet);  // 调度执行

// workqueue示例
struct work_struct my_work;
void my_work_handler(struct work_struct *work) {
    // 可以睡眠的处理逻辑
    kmalloc(1024, GFP_KERNEL);  // 可以分配内存
}
INIT_WORK(&my_work, my_work_handler);
schedule_work(&my_work);  // 放入系统默认workqueue
```

### 8.3 内核锁机制

| 锁类型 | 可否睡眠 | 适用场景 | 开销 |
|--------|---------|---------|------|
| **spinlock** | ❌ 忙等 | 短临界区、中断上下文 | 极低（CPU自旋） |
| **mutex** | ✅ 可睡眠 | 长临界区、进程上下文 | 中等（涉及调度） |
| **rwlock** | ❌ 忙等 | 读多写少 | 低 |
| **rwsem** | ✅ 可睡眠 | 读多写少、可能睡眠 | 中等 |
| **RCU** | ✅/❌ | 读极多写极少 | 读端零开销 |

#### spinlock（自旋锁）

```c
spinlock_t lock;
spin_lock_init(&lock);

// 普通自旋锁（进程上下文）
spin_lock(&lock);
// 临界区（必须极短，不能睡眠！）
spin_unlock(&lock);

// 中断安全版本（中断上下文也要用此锁时）
spin_lock_irqsave(&lock, flags);    // 关本地中断 + 加锁
// 临界区
spin_unlock_irqrestore(&lock, flags); // 恢复中断 + 解锁

// ⚠️ 在持有spinlock时绝不能睡眠/调度
// 否则会导致死锁（持锁CPU被调度走，其他CPU自旋等待）
```

#### mutex（互斥锁）

```c
struct mutex lock;
mutex_init(&lock);

mutex_lock(&lock);      // 获取不到则睡眠等待
// 临界区（可以做耗时操作、可以睡眠）
mutex_unlock(&lock);

// ⚠️ 不能在中断上下文使用（中断中不能睡眠）
// ⚠️ 用户态持有mutex锁时，进程处于S状态（可中断睡眠），不是D状态
```

#### RCU（Read-Copy-Update）

**核心思想**：读端零开销，写端负责延迟释放旧数据

```c
// 读端（零开销，无锁！）
rcu_read_lock();           // 仅禁止抢占（不是真正的锁）
p = rcu_dereference(gp);  // 读取受RCU保护的指针
// 使用 p 访问数据
rcu_read_unlock();

// 写端（Copy-on-Write模式）
struct data *new_data = kmalloc(sizeof(*new_data), GFP_KERNEL);
*new_data = *old_data;     // 复制旧数据
new_data->field = new_val; // 修改新副本
rcu_assign_pointer(gp, new_data);  // 原子替换指针
synchronize_rcu();         // 等待所有读端退出（Grace Period）
kfree(old_data);           // 安全释放旧数据
```

**RCU原理图**：
```
写者：           读者A（旧数据）    读者B（新数据）
修改 → 复制新副本 ──→ 原子替换指针 ──→ 等待Grace Period ──→ 释放旧数据
                      │                    │
                      └── 此时读者A仍在读旧数据，安全
                                           └── 读者A退出后，旧数据可安全释放
```

**应用场景**：路由表、进程列表、文件系统dcache等读远多于写的数据结构

### 8.4 内核抢占（Kernel Preemption）

**什么是内核抢占？**
- 普通进程在**用户态**可被调度器抢占（时间片用完、更高优先级任务到来）
- **内核抢占**指：进程在**内核态**执行系统调用/中断处理时，也可被高优先级任务抢占

**三种内核抢占模型**：

| 模型 | CONFIG选项 | 特点 | 适用场景 |
|------|-----------|------|---------|
| **不可抢占** | PREEMPT_NONE | 内核代码执行完才让出CPU | 服务器（吞吐优先） |
| **自愿抢占** | PREEMPT_VOLUNTARY | 在内核中的显式检查点可抢占 | 通用桌面 |
| **完全抢占** | PREEMPT | 除持锁外随时可抢占 | 低延迟（实时性） |

```bash
# 查看当前内核的抢占配置
cat /boot/config-$(uname -r) | grep PREEMPT
# CONFIG_PREEMPT_NONE=y        → 不可抢占（大多数服务器内核）
# CONFIG_PREEMPT_VOLUNTARY=y   → 自愿抢占
# CONFIG_PREEMPT=y             → 完全抢占

# 运行时查看
uname -v
# #1 SMP PREEMPT Tue Jan 1 00:00:00 UTC 2024  → 有PREEMPT字样=开启
```

**抢占计数器（preempt_count）**：
```c
// 内核通过preempt_count判断是否可以抢占
// preempt_count > 0 时不可抢占

preempt_disable();   // preempt_count++
// 临界区
preempt_enable();    // preempt_count--，检查是否需要重新调度

// 持有spinlock时自动禁用抢占
spin_lock(&lock);    // 隐含 preempt_disable()
// ...
spin_unlock(&lock);  // 隐含 preempt_enable()
```

### 8.5 Lockup检测：Hard Lockup与Soft Lockup

#### Soft Lockup（软锁定）

**定义**：CPU被内核代码**持续占用超过阈值**（默认20秒），不调度其他任务

**检测原理**：
```
watchdog内核线程（每个CPU一个，优先级最高的SCHED_FIFO）
    - 正常情况下：watchdog线程周期性被调度执行，更新时间戳
    - 异常情况下：某个任务长时间占用CPU（如死循环、长时间持锁）
                  watchdog线程无法被调度
                  时钟中断检查发现watchdog时间戳过期
                  → 触发 soft lockup 警告
```

```bash
# 内核日志
dmesg | grep -i lockup
# BUG: soft lockup - CPU#2 stuck for 22s! [kworker/2:1:1234]

# 调整阈值
sysctl -w kernel.watchdog_thresh=30  # 默认10（soft lockup在2倍=20s触发）
```

#### Hard Lockup（硬锁定）

**定义**：CPU被**中断也无法打断**的代码持续占用（通常是关中断后死循环）

**检测原理**：
```
NMI watchdog（不可屏蔽中断）
    - 基于PMU（Performance Monitoring Unit）的NMI中断
    - NMI不受关中断（cli）影响
    - 如果soft lockup的watchdog线程超过阈值未更新，
      NMI watchdog检测到 → 触发 hard lockup panic
```

```bash
# 内核日志
# Watchdog detected hard LOCKUP on cpu 3

# 检查NMI watchdog是否启用
cat /proc/sys/kernel/nmi_watchdog  # 1=启用

# hard lockup常见原因：
# 1. 关中断后进入死循环
# 2. 内核spinlock死锁（A等B，B等A）
# 3. 硬件故障（如PCIe设备卡死）
```

**对比总结**：

| 特性 | Soft Lockup | Hard Lockup |
|------|------------|-------------|
| **定义** | CPU不调度其他任务 | CPU连中断都无法响应 |
| **检测机制** | 时钟中断 + watchdog线程 | NMI watchdog（PMU） |
| **默认阈值** | 20秒 | 10秒 |
| **关中断** | 可以响应中断 | 中断也被屏蔽 |
| **严重程度** | 警告（可恢复） | Panic（通常不可恢复） |
| **常见原因** | 长时间持锁、死循环（未关中断） | spinlock死锁、关中断死循环 |

### 8.6 进程D状态与Load Average

#### D状态详解

**D状态（TASK_UNINTERRUPTIBLE）**：不可中断的睡眠

```
进程状态对比：
S (Sleeping)    - 可中断睡眠，可被信号唤醒（如等待用户输入）
D (Disk Sleep)  - 不可中断睡眠，不能被信号唤醒（如等待磁盘IO）
                  → kill -9 也杀不死D状态进程！

为什么需要D状态？
- 保护内核数据结构的一致性
- 例如：进程正在通过NFS读取文件
  如果此时被信号中断，内核中的IO状态可能不一致
  因此设为D状态，直到IO操作完成

常见触发D状态的场景：
1. 磁盘IO等待（read/write系统调用进入内核后）
2. NFS/CIFS网络文件系统访问（网络延迟导致长时间D状态）
3. 内核中等待mutex锁（部分内核路径）
4. 页面换出/换入（swap操作）
```

```bash
# 查看D状态进程
ps aux | awk '$8 ~ /D/ {print}'

# 查看D状态进程的内核栈（定位卡在哪里）
cat /proc/<pid>/stack
# [<0>] nfs_wait_bit_killable+0x1c/0x30 [nfs]
# [<0>] __wait_on_bit+0x42/0x80
# [<0>] out_of_line_wait_on_bit+0x89/0xb0
# → 说明卡在NFS等待上
```

**注意**：用户态持有mutex锁的进程**不处于D状态**，而是S状态或R状态。只有**内核态**的不可中断等待才是D状态。

#### Load Average计算原理

```
Linux的load average = 处于R状态 + 处于D状态的任务数的指数移动平均

公式：
load(t) = load(t-1) * e^(-1/τ) + n * (1 - e^(-1/τ))

其中：
- n = 当前R + D状态的任务数
- τ = 时间常数（1分钟/5分钟/15分钟）
- 每5秒（HZ/12 tick）采样一次
```

```bash
# 查看load average
uptime
# 14:00:00 up 10 days,  load average: 2.50, 3.20, 4.10
#                                     1分钟  5分钟  15分钟

cat /proc/loadavg
# 2.50 3.20 4.10 3/256 12345
# ↑               ↑     ↑
# load avg        R/总进程数  最新PID

# 判断标准：
# load / CPU核心数 > 1 → 过载
# 例：4核CPU，load=4.0 表示刚好满载
# load=8.0 表示有一半任务在等待
```

**为什么D状态计入load？**
- D状态进程虽然不占CPU，但占用了系统资源（IO子系统繁忙）
- 大量D状态进程说明IO是瓶颈（即使CPU空闲，load也会很高）
- 这是Linux特有的设计（其他Unix系统load只计R状态）

### 8.7 系统调用传参方式

**x86_64系统调用的三种触发方式**：

| 方式 | 指令 | 适用架构 | 特点 |
|------|------|---------|------|
| **int 0x80** | int $0x80 | x86_32 | 软中断方式，开销大（~400 cycles） |
| **sysenter/sysexit** | sysenter | x86_32（Pentium II+） | 快速系统调用（~200 cycles） |
| **syscall/sysret** | syscall | x86_64 | 64位快速系统调用（~100 cycles） |

**x86_64系统调用传参规则**：
```asm
; 系统调用号 → rax
; 参数1     → rdi
; 参数2     → rsi
; 参数3     → rdx
; 参数4     → r10  （注意：不是rcx！因为syscall指令会用rcx保存返回地址）
; 参数5     → r8
; 参数6     → r9
; 返回值    → rax

; 示例：write(1, "hello\n", 6)
; write的系统调用号 = 1
mov rax, 1          ; 系统调用号 (SYS_write)
mov rdi, 1          ; fd = 1 (stdout)
lea rsi, [msg]      ; buf = "hello\n"
mov rdx, 6          ; count = 6
syscall             ; 触发系统调用
```

**普通函数 vs 系统调用的传参对比**：
```
普通函数调用（System V AMD64 ABI）：
参数顺序：rdi, rsi, rdx, rcx, r8, r9（第4个用rcx）

系统调用：
参数顺序：rdi, rsi, rdx, r10, r8, r9（第4个用r10）

差别在第4个参数：函数用rcx，系统调用用r10
原因：syscall指令隐式使用rcx保存返回地址（RIP）
```

**系统调用完整流程（x86_64）**：
```
用户态                              内核态
────────                            ────────
syscall指令
  ├─ RCX = 返回地址(RIP)
  ├─ R11 = RFLAGS
  ├─ 加载MSR_LSTAR → RIP           → entry_SYSCALL_64
  │                                    ├─ 保存用户态寄存器到pt_regs
  │                                    ├─ 查sys_call_table[rax]
  │                                    ├─ 调用具体系统调用函数
  │                                    └─ sysret返回用户态
  ↓
sysret指令
  ├─ 从RCX恢复RIP
  └─ 从R11恢复RFLAGS
```

### 9.1 CPU 100% 排查完整SOP

#### Step 1：确认现象——谁在吃CPU？

```bash
# 1. 整体概览：确认是哪个进程在吃CPU
top -b -n 1 | head -20          # 批处理模式取一次快照
# 关注 %Cpu(s) 行：
#   us (user)    — 用户态CPU
#   sy (system)  — 内核态CPU
#   wa (iowait)  — IO等待
#   si (softirq) — 软中断

# 2. 拿到进程PID后确认进程信息

# 3. 进一步区分：到底是用户态高还是内核态高？

# 看 %usr 和 %system 列
```

**判断分支**：
- `us` 高 → 业务代码/GC/正则回溯等，走 **用户态路径**
- `sy` 高 → 系统调用频繁/锁竞争/网络收发包，走 **内核态路径**

#### Step 2a：Go 服务排查路径（用户态CPU高）

```bash
# 1. 开启 pprof（通常已在服务中注册）
#    import _ "net/http/pprof"
#    go func() { http.ListenAndServe(":6060", nil) }()

# 2. 采集 30 秒 CPU Profile
go tool pprof -http=:8080 http://localhost:6060/debug/pprof/profile?seconds=30

# 3. 查看火焰图，定位热点函数
# 在浏览器中 View → Flame Graph

# 4. 同时抓 goroutine 堆栈，看有无异常积压
curl http://localhost:6060/debug/pprof/goroutine?debug=2 > goroutine.txt
wc -l goroutine.txt              # goroutine 数量异常多说明可能泄漏
grep -c "^goroutine" goroutine.txt

# 5. 查看GC情况
curl http://localhost:6060/debug/pprof/heap?debug=1 | grep -A5 "gc"
# 或通过 GODEBUG=gctrace=1 观察GC日志
```

**Go 服务 CPU 高常见根因**：

| 根因 | 特征 | 解决方案 |
|------|------|----------|
| 死循环/忙等待 | 火焰图某函数占比 >80% | 修复循环逻辑，加退出条件 |
| GC 频繁 | `runtime.gcBgMarkWorker` 占比高 | 减少小对象分配，用 sync.Pool |
| 正则回溯 | `regexp.(*machine).match` 占比高 | 预编译正则，避免贪婪匹配 |
| JSON 序列化 | `encoding/json` 相关函数 | 换用 jsoniter / sonic |
| 锁竞争 | `runtime.lock` / `sync.Mutex` | 减小临界区，换用分段锁或无锁结构 |

#### Step 2b：Java 服务排查路径（用户态CPU高）

```bash
# 1. 找到 Java 进程 PID
jps -lv | grep <服务名>

# 2. 找到进程内最耗CPU的线程
top -Hp <PID>                    # 注意 -H 表示显示线程
# 记录高CPU的线程TID，例如 12345

# 3. TID 转 16 进制（jstack 用 nid=0x 匹配）
printf "%x\n" 12345              # 输出 3039

# 4. 抓线程堆栈
jstack <PID> > jstack.txt

# 5. 搜索对应线程
grep -A 30 "nid=0x3039" jstack.txt

# 6. 多次采样对比（间隔3-5秒，取3次）
for i in 1 2 3; do
    jstack <PID> > jstack_$i.txt
    sleep 3
done
# 如果同一线程始终在同一位置 → 死循环或长时间计算
```

**Java 服务 CPU 高常见根因**：

| 根因 | 特征 | 解决方案 |
|------|------|----------|
| Full GC 频繁 | 线程在 `VM Thread` / `GC task` | 调大堆、排查内存泄漏 |
| 死锁 | jstack 底部显示 `Found deadlock` | 调整加锁顺序 |
| 忙等待/自旋 | 同一线程反复执行同一代码段 | 改为阻塞等待或加 sleep |
| 正则回溯 | `java.util.regex` 相关堆栈 | 修正正则表达式 |
| 日志序列化 | `toString()` / `JSON.toJSONString` | 降低日志级别，异步打印 |

#### Step 2c：内核态CPU高排查

```bash
# 1. 用 perf 查看内核热点
perf top -g                      # 实时查看内核态热点函数

# 2. 采集并生成报告
perf record -g -p <PID> -- sleep 30
perf report

# 3. 系统调用统计
strace -c -p <PID> -e trace=all  # 统计系统调用频率
# 关注 calls 列最高的系统调用

# 4. 软中断分布
watch -n 1 'cat /proc/softirqs'  # 观察 NET_RX/NET_TX 增长速度
```

**内核态 CPU 高常见根因**：

| 根因 | 特征 | 解决方案 |
|------|------|----------|
| 大量短连接 | `sys_connect`/`tcp_v4_connect` 频繁 | 改用连接池/长连接 |
| 频繁内存分配 | `__alloc_pages` / `kmalloc` 热点 | 减少 mmap，用内存池 |
| 锁竞争（内核锁） | `_raw_spin_lock` 占比高 | 减少并发写同一文件/socket |
| 网络收包风暴 | `NET_RX` 软中断单核打满 | 开启 RSS/RPS 多队列 |

### 9.2 内存OOM排查完整SOP

#### Step 1：确认现象——是不是被OOM Killer杀了？

```bash
# 1. 检查系统日志中的 OOM 记录
dmesg | grep -i "oom" | tail -20
# 或
journalctl -k | grep -i "oom"

# 2. 查看被杀进程的详细信息
dmesg | grep -i "killed process"
# 输出示例：
# Out of memory: Killed process 12345 (java) total-vm:8192000kB ...

# 3. 当前内存概览
free -h                          # 总体内存使用
cat /proc/meminfo | head -20     # 详细内存分布

# 4. 查看各进程内存占用排名
ps aux --sort=-%mem | head -20
# 或
smem -r -s pss | tail -20       # 按 PSS（比例分摊共享内存）排序
```

**关键指标解读**：
- `MemFree` 接近 0 不一定有问题（Linux会用空闲内存做page cache）
- 看 `MemAvailable`：这才是真正可用内存
- `SwapUsed` 持续增长 → 物理内存确实不够了

#### Step 2a：Go 服务内存泄漏排查

```bash
# 1. 查看堆内存概况
go tool pprof http://localhost:6060/debug/pprof/heap

# 在 pprof 交互界面中：
#   top 20          — 查看内存分配 Top 20 函数
#   list <函数名>   — 查看函数逐行分配
#   web             — 生成调用图（需安装 graphviz）

# 2. 对比两个时间点的 heap profile（差分分析）
curl http://localhost:6060/debug/pprof/heap > heap_t1.prof
# 等待一段时间（如5分钟）
curl http://localhost:6060/debug/pprof/heap > heap_t2.prof
go tool pprof -base heap_t1.prof heap_t2.prof
# top 查看增长最多的分配点

# 3. goroutine 泄漏检测（goroutine 泄漏 = 间接内存泄漏）
curl http://localhost:6060/debug/pprof/goroutine?debug=1 | head -30
# 如果 goroutine 数量持续增长，定位创建它们的代码

# 4. 运行时内存统计
curl http://localhost:6060/debug/pprof/heap?debug=1 | grep -E "Alloc|Sys|HeapIdle|HeapReleased|NumGC"
```

**Go 服务内存泄漏常见根因**：

| 根因 | 特征 | 解决方案 |
|------|------|----------|
| goroutine 泄漏 | goroutine 数量持续增长 | 确保 channel 有发送/接收对、用 context 控制生命周期 |
| 未关闭 HTTP Response Body | `net/http` 相关分配增长 | 必须 `defer resp.Body.Close()` |
| time.After 在循环中使用 | `time` 包分配增长 | 改用 `time.NewTimer` + `Reset` |
| 全局 map 无限增长 | 堆分配持续增长 | 加 TTL / LRU 淘汰策略 |
| slice 底层数组未释放 | 大 slice 切片后仍持有底层数组 | `copy` 到新 slice |
| cgo 内存泄漏 | Go heap 正常但 RSS 持续增长 | 检查 C 代码的 malloc/free |

#### Step 2b：Java 服务内存泄漏排查

```bash
# 1. 查看堆内存概况
jmap -heap <PID>

# 2. 导出堆转储（注意：会触发STW，生产环境谨慎）
jmap -dump:format=b,file=heap.hprof <PID>
# 或在启动参数中预设自动导出：
# -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/

# 3. 使用 MAT（Eclipse Memory Analyzer）分析
# 打开 heap.hprof → Leak Suspects Report → 自动定位可疑对象

# 4. 快速查看对象统计（不dump全量）
jmap -histo <PID> | head -30    # 查看实例数和大小 Top 30

# 5. GC 日志分析
# 启动参数加：-Xloggc:/tmp/gc.log -XX:+PrintGCDetails
# 使用 GCViewer 或 gceasy.io 分析

# 6. 堆外内存排查（NIO DirectByteBuffer 等）
jcmd <PID> VM.native_memory summary
# 需要启动参数 -XX:NativeMemoryTracking=summary
```

**Java 服务内存泄漏常见根因**：

| 根因 | 特征 | 解决方案 |
|------|------|----------|
| ThreadLocal 未清理 | 线程池中 ThreadLocal 累积 | 用完 `remove()`，或用 TransmittableThreadLocal |
| 缓存无上限 | HashMap/ConcurrentHashMap 持续增长 | 改用 Guava Cache / Caffeine 设置 maxSize |
| 类加载器泄漏 | Metaspace 持续增长 | 排查动态类生成（CGLIB/反射），限制 `-XX:MaxMetaspaceSize` |
| 未关闭的连接/流 | `Finalizer` 队列积压 | 用 try-with-resources |
| 大对象直接进老年代 | Old Gen 快速填满 | 调整 `-XX:PretenureSizeThreshold` |
| 堆外内存泄漏 | 堆内正常但 RSS 异常增长 | 检查 DirectByteBuffer，开启 NMT |

#### Step 2c：系统级内存排查

```bash
# 1. 内存详细分布
cat /proc/meminfo
# 关注：
#   MemAvailable  — 真正可用内存
#   Buffers       — 块设备缓存
#   Cached        — 页缓存（文件缓存）
#   Slab          — 内核 slab 分配器
#   SReclaimable  — 可回收 slab
#   SUnreclaim    — 不可回收 slab

# 2. Slab 缓存详情（内核态内存泄漏排查）
slabtop -o | head -20            # 查看 slab 对象排名
cat /proc/slabinfo | sort -k 3 -nr | head -20

# 3. Page Cache 分析
# 如果 Cached 很大但 MemAvailable 也够，不是问题
# 如果要手动释放 page cache（应急用）：
sync && echo 3 > /proc/sys/vm/drop_caches

# 4. 各进程实际内存占用
# /proc/<PID>/smaps 提供最精确的内存分布
cat /proc/<PID>/smaps_rollup
# 或
pmap -x <PID> | tail -5         # 查看内存映射汇总

# 5. 交换分区分析
swapon -s                        # 查看 swap 使用
vmstat 1 5                       # 关注 si/so 列（swap in/out）
# si/so 持续 >0 说明物理内存严重不足

# 6. NUMA 内存不均衡
numastat                         # 查看各 NUMA 节点内存分配
# numa_miss 高说明跨节点分配频繁，影响性能
```

#### Step 3：OOM Score 调整和保护措施

```bash
# 1. 查看进程的 OOM Score（分数越高越容易被杀）
cat /proc/<PID>/oom_score
cat /proc/<PID>/oom_score_adj    # 可调整值：-1000 ~ 1000

# 2. 保护关键进程（降低被 OOM Kill 的概率）
echo -1000 > /proc/<PID>/oom_score_adj   # -1000 = 永不被杀

# 3. 让不重要的进程优先被杀
echo 1000 > /proc/<PID>/oom_score_adj    # 1000 = 最先被杀

# 4. 系统级 OOM 策略调整
# 0 = 默认启发式策略（可能过度提交）
# 1 = 允许过度提交（不检查）
# 2 = 禁止过度提交（分配超过 swap + ratio% * 物理内存 则失败）
cat /proc/sys/vm/overcommit_memory
echo 2 > /proc/sys/vm/overcommit_memory   # 生产建议：禁止过度提交
echo 80 > /proc/sys/vm/overcommit_ratio   # 允许分配 swap + 80% 物理内存

# 5. 设置内存 cgroup 限制（更精细的保护）
# 限制单个服务的内存使用，避免一个服务OOM拖垮整台机器
# cgroup v2 示例：
echo 4G > /sys/fs/cgroup/<服务>/memory.max
echo 3G > /sys/fs/cgroup/<服务>/memory.high  # 超过 high 触发回收，超过 max 触发 OOM
```

**生产环境OOM防护清单**：
- [ ] 关键服务设置 `oom_score_adj = -1000`
- [ ] 所有服务通过 cgroup 限制内存上限
- [ ] 配置 `-XX:+HeapDumpOnOutOfMemoryError`（Java）
- [ ] 接入内存监控告警（>80% 预警，>90% 告警）
- [ ] 定期检查 goroutine / 线程数趋势
- [ ] `vm.overcommit_memory = 2`，杜绝内存过度提交

### 9.3 磁盘IO慢排查SOP

#### Step 1：确认现象——IO到底慢在哪？

```bash
# 1. 整体IO概览
iostat -x 1 5                    # 每秒采样，共5次
# 关键指标：
#   %util   — 磁盘繁忙度（>80% 说明磁盘接近饱和）
#   await   — 平均IO等待时间（ms），包含队列等待+实际服务时间
#   r_await — 读等待    w_await — 写等待
#   avgqu-sz — 平均队列长度（>2 说明IO积压）
#   rrqm/s wrqm/s — 合并读写请求数（合并多说明顺序IO较多）

# 2. 定位到具体磁盘后，找出哪个进程在做IO
iotop -o -P                      # -o 只显示有IO的进程
# 或
pidstat -d 1 5                   # 每秒采样进程级IO统计

# 3. 查看进程打开的文件
lsof -p <PID> | grep -E "REG|DIR"
# 或
ls -la /proc/<PID>/fd/ | wc -l  # 打开的文件描述符数量
```

#### Step 2：iowait高的常见原因和解决方案

```bash
# 确认是否是 iowait 导致 CPU 看起来"很忙"
vmstat 1 5
# wa 列 > 20% → 说明大量CPU时间在等IO

# 排查 iowait 高的具体原因：
# 1) 大量同步写日志
strace -e trace=write -c -p <PID>
# 如果 write 调用极频繁 → 考虑异步日志/批量写

# 2) 数据库大量随机IO
iostat -x 1 | grep <磁盘设备>
# 如果 rrqm/s 和 wrqm/s 都很低，但 IOPS 很高 → 随机IO
# 解决方案：加内存做缓存、换SSD、优化查询减少随机读

# 3) Swap 导致的 IO（内存不足引发）
vmstat 1 5    # si/so > 0 → 正在换入换出
# 解决方案：增加物理内存、优化应用内存使用
```

**iowait 高常见原因总结**：

| 根因 | 诊断方法 | 解决方案 |
|------|----------|----------|
| 同步写日志过多 | strace 统计 write 调用 | 异步日志、日志级别调高、批量刷盘 |
| 数据库随机IO | iostat 看合并率低+IOPS高 | 加缓存、换SSD、优化索引 |
| Swap IO（内存不足）| vmstat 看 si/so 列 | 加内存、优化应用、关闭 swap |
| 大文件顺序读写 | iotop 定位进程+文件 | 限速、IO调度器调优、分时段执行 |
| RAID 重建/磁盘坏道 | dmesg 看硬件错误 | 更换磁盘、RAID 修复 |

#### Step 3：文件系统层面排查

```bash
# 1. 磁盘空间是否写满
df -h                            # 查看各分区使用率
# 使用率 100% → 写操作会失败
# 找出大文件：
du -sh /* 2>/dev/null | sort -rh | head -10
# 找出大日志文件（常见元凶）：
find /var/log -type f -size +100M -exec ls -lh {} \;

# 2. inode 是否耗尽（空间有剩余但无法创建文件）
df -i                            # 查看 inode 使用率
# inode 100% → 大量小文件导致
# 定位小文件目录：
find / -xdev -type d -exec sh -c 'echo "$(find "$1" -maxdepth 1 | wc -l) $1"' _ {} \; | sort -rn | head -10

# 3. 文件系统错误检查
dmesg | grep -i "ext4\|xfs\|error\|readonly"
# 如果看到 "Remounting filesystem read-only" → 文件系统损坏
# 修复（需卸载分区）：fsck -y /dev/sdX

# 4. 已删除但未释放的文件（进程仍持有文件句柄）
lsof | grep deleted | sort -k 7 -rn | head -10
# 解决方案：重启对应进程释放句柄

# 5. IO 调度器优化
cat /sys/block/sda/queue/scheduler    # 查看当前调度器
# SSD 建议：none 或 mq-deadline
# HDD 建议：mq-deadline 或 bfq
echo "mq-deadline" > /sys/block/sda/queue/scheduler
```

**磁盘IO排查速查流程**：

```
           ┌─────────────────┐
           │  IO慢/iowait高  │
           └────────┬────────┘
                    │
        ┌───────────┴───────────┐
        ▼                       ▼
   iostat -x              df -h / df -i
   磁盘级指标              文件系统层面
        │                       │
   ┌────┴────┐            ┌─────┴─────┐
   ▼         ▼            ▼           ▼
%util高   await高       空间满      inode满
   │         │            │           │
   ▼         ▼            ▼           ▼
iotop     队列积压     清理日志    清理小文件
定位进程   avgqu-sz    du排查      find定位
```

## 10. 面试题自查

### 操作系统排障补充

- 面试里的操作系统题最终都会落到可观测和排障，建议把 `top/vmstat/iostat/pidstat/perf` 串成固定排查路径。
- 只背概念很容易在追问里失分，必须能解释“某个指标异常时下一步查什么”。
- 常见线上故障是 CPU 不高但延迟上升，这类问题往往在调度等待、IO 等待、锁竞争或缺页抖动。

### 初级高频速刷

**1. 用户态和内核态的区别是什么？系统调用为什么慢？**

用户态权限低，内核态权限高；系统调用慢，不是因为“函数本身复杂”，而是因为涉及特权级切换、上下文保存恢复以及可能的调度和缓存扰动。这个问题是理解操作系统性能开销的起点。

**2. 进程和线程的区别是什么？**

进程是资源分配基本单位，线程是 CPU 调度基本单位。同一进程内线程共享地址空间和大部分资源，这让通信更轻量，但也带来了同步复杂度。

**3. Page Cache 是什么？为什么它会影响 IO 性能判断？**

Page Cache 是内核缓存文件页的数据结构。很多文件读取之所以快，不是因为磁盘快，而是因为数据已经在内存里了，所以做 IO 压测和排障时必须先判断是不是 cache hit。

**4. `mmap` 和 `read/write` 的区别是什么？**

`read/write` 是显式系统调用拷贝数据，`mmap` 是把文件映射到进程地址空间，用内存访问语义操作文件。两者不是简单的“谁更快”，而是访问模式、缺页行为和拷贝路径不同。

**5. 阻塞 / 非阻塞 / 同步 / 异步 这四个概念怎么区分？**

阻塞和非阻塞描述调用时线程是否被挂起；同步和异步描述结果完成后是谁来通知。比如 `epoll` 常被说成异步，其实很多场景下它仍然是同步 IO 多路复用，这正是面试最爱考的概念陷阱。

### 中级高频进阶

**6. 文件描述符（fd）是什么？fd 泄漏一般会怎么表现？**

fd 是进程访问内核对象的引用句柄，文件、socket、管道都可能占用 fd。泄漏后常见表现是新连接建立失败、日志文件打不开、报 `too many open files`，而且问题通常会随着运行时间持续恶化。

**7. `fork` 为什么通常看起来很快？**

因为现代操作系统并不会在 `fork` 时立刻复制整份父进程地址空间，而是采用 Copy-On-Write。父子进程先共享只读页，等任意一方真正写某页时再复制，这大大降低了初始开销。

**8. 为什么用户态等 mutex 不会进入 D 状态，而磁盘 IO 可能会？**

因为用户态 mutex 等待通常属于可中断或可调度的睡眠，线程仍然在正常调度体系内；而 D 状态是内核态不可中断睡眠，常见于等待磁盘、NFS 等内核资源。这个问题本质上在考你是否理解调度状态和锁等待的边界。

**9. NUMA 是什么？为什么跨 NUMA 访问会拉高延迟？**

NUMA 表示“非一致内存访问”，CPU 访问本地节点内存更快，访问远端节点内存更慢。高并发服务如果线程和内存分配不亲和，容易出现跨节点访存，表现为尾延迟上升、CPU 利用率看起来正常但吞吐下降。

**10. 为什么线上会出现“CPU 不高但请求变慢”？**

因为延迟不只由 CPU 使用率决定，还可能卡在 run queue 排队、磁盘/NFS 等待、锁竞争、缺页中断或网络重传。排障时应先用 RED 指标定位受损链路，再分解到 CPU 调度、内存、IO、网络四条路径，而不是只盯 `top`。

### 深入问答

#### Q1: 进程和线程的本质区别是什么？什么时候用多进程，什么时候用多线程？

**答案**：
**本质区别**：
- **进程**：资源分配的基本单位，有独立的地址空间，进程间通信需要IPC
- **线程**：CPU调度的基本单位，共享进程地址空间，可直接访问全局变量

**使用场景**：
- **多进程**：需要强隔离（一个崩溃不影响其他）、利用多核、安全敏感
- **多线程**：需要共享大量数据、通信频繁、创建/销毁频繁

**Go语言建议**：使用Goroutine，它是轻量级线程，结合了多线程的易用性和多进程的安全性

---

#### Q2: 什么是僵尸进程和孤儿进程？如何处理？

**答案**：
**僵尸进程**：
- 子进程退出后，父进程未调用wait()回收
- 占用PID资源，可能导致PID耗尽
- 处理：父进程调用wait()，或杀死父进程让init回收

**孤儿进程**：
- 父进程退出，子进程仍在运行
- 由init进程（PID=1）收养并回收
- 是正常现象，不会造成资源泄漏

---

#### Q3: 虚拟内存有什么作用？页表是如何工作的？

**答案**：
**虚拟内存作用**：
1. **隔离**：每个进程独立地址空间，互不干扰
2. **扩展**：虚拟内存可大于物理内存（通过swap）
3. **共享**：多个进程共享同一物理页（如共享库）

**页表工作原理**：
- 虚拟地址通过多级页表（x86_64为4级）转换为物理地址
- TLB缓存常用的虚拟→物理映射，命中时无需查页表
- TLB Miss时触发Page Walk，遍历多级页表

---

#### Q4: 什么是缺页中断？Minor和Major Page Fault有什么区别？

**答案**：
**缺页中断**：访问的虚拟页未映射到物理页时触发

**Minor Page Fault**：
- 页在内存中，但未映射到进程页表
- 只需更新页表，无需磁盘IO
- 常见于首次访问匿名映射

**Major Page Fault**：
- 页不在内存，需要从磁盘（swap或文件）读取
- 需要磁盘IO，开销大
- 应尽量避免（增加物理内存、减少swap使用）

---

#### Q5: epoll和select有什么区别？为什么epoll更高效？

**答案**：
| 特性 | select | epoll |
|------|--------|-------|
| fd数量限制 | 1024 | 无限制 |
| 性能 | O(n)，遍历所有fd | O(1)，只遍历就绪fd |
| 内存拷贝 | 每次拷贝fd_set | 共享内存，无拷贝 |

**epoll高效原因**：
1. 使用红黑树管理fd，添加/删除O(logN)
2. 使用就绪链表，只返回就绪的fd
3. 使用mmap共享内存，避免拷贝

---

#### Q6: 什么是零拷贝？有哪些实现方式？

**答案**：
**传统拷贝**：磁盘→内核→用户空间→Socket→网卡（4次拷贝，2次用户态/内核态切换）

**零拷贝实现**：
1. **sendfile**：磁盘→内核→网卡（2次拷贝），无需拷贝到用户空间
2. **mmap**：将文件映射到用户空间，避免read系统调用
3. **splice**：在两个fd之间直接传输数据，无需经过用户空间

**应用场景**：静态文件服务（Nginx）、Kafka消息传输

---

#### Q7: 什么是伪共享（False Sharing）？如何避免？

**答案**：
**伪共享**：
- 多个线程访问不同变量，但变量在同一缓存行（64字节）
- 一个线程修改变量，导致其他线程的缓存行失效
- 严重影响多线程性能

**避免方法**：
1. 缓存行填充（Padding）：变量之间填充64字节
2. Go中使用独立的goroutine本地变量
3. 按缓存行对齐数据结构

---

#### Q8: TCP三次握手的过程？为什么不是两次或四次？

**答案**：
**三次握手**：
1. 客户端→服务端：SYN（我要建立连接）
2. 服务端→客户端：SYN+ACK（同意，我也要建立连接）
3. 客户端→服务端：ACK（同意）

**为什么不是两次**：
- 防止旧的SYN包导致错误建立连接
- 确认双方都能收发

**为什么不是四次**：
- 第二步合并了SYN和ACK，没必要分开

#### Q9: 什么是上下文切换？如何减少上下文切换？

**答案**：
**上下文切换**：
- 保存当前任务的CPU状态，恢复另一个任务的状态
- 进程切换：3-5μs（需要切换页表，TLB失效）
- 线程切换：1μs（同进程内，无需切换页表）

**减少方法**：
1. 使用协程（goroutine）替代线程
2. 减少锁竞争（无锁数据结构）
3. 使用IO多路复用（epoll）替代阻塞IO
4. 批量处理替代频繁调度

#### Q10: Cgroup的CPU隔离有哪些方式？各有什么特点？

**答案**：
| 方式 | 机制 | 特点 |
|------|------|------|
| cpuset | 核心绑定 | 物理隔离，最强隔离性 |
| cpuquota | 时间配额 | 硬限制，不能超额使用 |
| cpushares | 权重比例 | 弹性限制，空闲时可借用 |

**选择建议**：
- 延迟敏感：cpuset（独占核心）
- 多租户硬隔离：cpuquota
- 弹性共享/超卖：cpushares

#### Q11: Linux的IO调度算法有哪些？各适用什么场景？

**答案**：
| 算法 | 特点 | 适用场景 |
|------|------|----------|
| cfq | 完全公平，按进程分时间片 | 通用服务器 |
| deadline | 保证请求期限，防止饿死 | 数据库等IO密集型 |
| noop | 无调度，仅FIFO | SSD固态硬盘 |

**SSD为什么用noop**：
- SSD无寻道时间，不需要重排请求
- noop减少调度开销

#### Q12: 什么是CFS调度器？nice值如何影响调度？

**答案**：
**CFS（完全公平调度器）**：
- Linux默认调度器
- 通过虚拟运行时间（vruntime）保证公平
- vruntime最小的进程优先调度

**nice值影响**：
- nice范围：-20（最高优先级）到19（最低优先级）
- nice值越小，权重越大，vruntime增长越慢
- nice=-10的进程获得的CPU时间约为nice=0的3倍

#### Q13: `mmap`、`sendfile`、`read/write` 三种数据路径怎么选？

**答案**：
**选择逻辑**：
1. **`read/write`**：通用方案，语义清晰，适合大多数业务读写路径。
2. **`mmap`**：适合随机访问和多进程共享页缓存，但要注意缺页抖动和映射管理复杂度。
3. **`sendfile`**：适合“文件到 socket”直传，减少用户态拷贝和上下文切换。

**工程取舍**：
- 静态资源分发优先 `sendfile`
- 索引类随机读场景可考虑 `mmap`
- 需要精细控制缓存与刷盘策略时，通常回到 `read/write` 或配合 `O_DIRECT`

#### Q14: 什么是OOM Killer？如何避免被OOM Kill？

**答案**：
**OOM Killer**：
- 系统内存不足时，内核选择杀死进程释放内存
- 根据oom_score选择（分数越高越容易被杀）

**避免被杀**：
1. 设置oom_score_adj（-1000到1000，越小越不容易被杀）
2. 合理设置内存限制（Cgroup）
3. 监控内存使用，提前扩容
4. 使用swap缓冲（但会影响性能）

```bash
# 保护关键进程
echo -1000 > /proc/1234/oom_score_adj
```

#### Q15: Go的GMP调度模型是什么？为什么这样设计？

**答案**：
**GMP模型**：
- **G**（Goroutine）：轻量级线程，2KB初始栈
- **M**（Machine）：内核线程，真正执行代码
- **P**（Processor）：逻辑处理器，数量=GOMAXPROCS

**为什么这样设计**：
1. 减少线程创建开销（复用M）
2. 工作窃取算法（P之间平衡负载）
3. 避免全局锁（每个P有本地队列）
4. 充分利用多核（P数量=核心数）

#### Q16: 什么是Direct IO？什么时候应该使用？

**答案**：
**Direct IO**：
- 绕过page cache，直接读写磁盘
- 需要用户自己管理缓存

**使用场景**：
1. 数据库（自己管理buffer pool）
2. 大文件顺序读写（不需要缓存）
3. 应用级缓存更高效时

**注意事项**：
- 地址必须对齐（通常512字节）
- 大小必须是扇区的整数倍

#### Q17: TCP拥塞控制是如何工作的？BBR和Cubic有什么区别？

**答案**：
**拥塞控制阶段**：
1. 慢启动：cwnd指数增长
2. 拥塞避免：cwnd线性增长
3. 快速重传：3个重复ACK立即重传
4. 快速恢复：cwnd减半继续传输

**Cubic vs BBR**：
- **Cubic**：基于丢包判断拥塞，保守
- **BBR**：基于带宽和RTT，主动探测，更激进

**BBR适用场景**：高带宽高延迟网络（如跨国链路）

#### Q18: 如何排查内存泄漏？Go和C/C++分别用什么工具？

#### Q19: 什么是信号（Signal）？如何优雅地处理SIGTERM？

**答案**：
**常用信号**：
- SIGTERM（15）：优雅退出请求
- SIGKILL（9）：强制杀死（不可捕获）
- SIGINT（2）：Ctrl+C中断
- SIGHUP（1）：终端断开

**优雅处理**：
```go
sigChan := make(chan os.Signal, 1)
signal.Notify(sigChan, syscall.SIGTERM, syscall.SIGINT)
go func() {
    <-sigChan
    // 1. 停止接收新请求
    // 2. 等待现有请求完成
    // 3. 关闭数据库连接
    // 4. 保存状态
    os.Exit(0)
}()
```

#### Q21: 什么是Hard Lockup和Soft Lockup？检测原理是什么？

**答案**：
**Soft Lockup**：
- CPU长时间（>20s）不调度其他任务（如内核死循环、长时间持锁）
- 检测原理：每个CPU有一个最高优先级watchdog线程，正常情况周期性更新时间戳；时钟中断检查时间戳是否过期

**Hard Lockup**：
- CPU连中断都无法响应（如关中断后死锁、硬件故障）
- 检测原理：NMI watchdog基于PMU的不可屏蔽中断，NMI不受cli（关中断）影响，发现watchdog超时则触发panic

**关键区别**：soft lockup可响应中断但不调度；hard lockup连中断也响应不了

#### Q22: 什么是内核抢占？Linux服务器内核一般开启内核抢占吗？

**答案**：
**内核抢占**：进程在内核态执行时，也可被更高优先级任务抢占调度

**三种配置**：
- PREEMPT_NONE：不可抢占（服务器默认，吞吐优先）
- PREEMPT_VOLUNTARY：自愿抢占（显式检查点可抢占）
- PREEMPT：完全抢占（低延迟场景）

**服务器一般使用PREEMPT_NONE**，因为频繁抢占带来上下文切换开销，对吞吐量有负面影响

#### Q23: 进程D状态是什么含义？用户态持有mutex锁会处于D状态吗？Load Average如何计算？

**答案**：
**D状态**：TASK_UNINTERRUPTIBLE，不可中断的睡眠，kill -9也杀不死
- 用于保护内核数据一致性（如等待磁盘IO、NFS操作）
- 用户态持有mutex锁**不会**处于D状态（是S状态），只有内核态不可中断等待才是D状态

**Load Average**：
- 计算公式：R状态 + D状态任务数的指数移动平均（每5秒采样一次）
- 包含D状态是Linux特有设计：D状态说明IO子系统繁忙
- 判断标准：load / CPU核心数 > 1 表示过载

#### Q24: 中断的上半部和下半部是什么？下半部有哪些机制？

**答案**：
**上半部（Top Half）**：关中断执行，必须极快完成，读设备状态、清中断标志
**下半部（Bottom Half）**：开中断后执行，负责耗时后续处理

| 机制 | 执行上下文 | 可否睡眠 | 适用场景 |
|------|-----------|---------|---------|
| softirq | 软中断上下文 | 不可 | 网络收包(NET_RX)、块设备 |
| tasklet | 软中断上下文 | 不可 | 驱动简单延迟处理 |
| workqueue | 内核线程 | 可以 | 需要分配内存、IO等 |

---

#### Q25: 内核有哪些锁机制？RCU的原理是什么？

**答案**：
| 锁 | 特点 | 适用场景 |
|------|------|---------|
| spinlock | 忙等，不可睡眠 | 短临界区、中断上下文 |
| mutex | 可睡眠 | 长临界区、进程上下文 |
| rwlock/rwsem | 读写分离 | 读多写少 |
| RCU | 读端零开销 | 读极多写极少 |

**RCU原理**：
1. 读端：rcu_read_lock()仅禁止抢占，无实际锁开销
2. 写端：复制旧数据 → 修改新副本 → 原子替换指针 → 等待Grace Period → 释放旧数据
3. Grace Period：等所有读端退出后，旧数据才被释放

#### Q26: NUMA 场景下怎么做性能排障和优化？

**答案**：
排障可以按三步做：
1. 先看拓扑：`numactl --hardware`、`lscpu` 确认节点和核分布。
2. 再看进程：`numastat -p <pid>` 判断远端内存访问占比是否过高。
3. 压测对比：绑定核心和内存后，观察 P99、吞吐、上下文切换是否改善。

常见优化手段：
- `numactl --cpunodebind --membind` 绑定计算和内存
- 将线程池按 NUMA 节点分组，减少跨节点抢占
- 避免“单节点内存不足+跨节点兜底”长期常态化

#### Q27: `fsync` 抖动怎么定位？为什么会出现“平时很快，偶发特别慢”？

**答案**：
`write` 通常只写入 page cache，`fsync` 才要求脏页真正落盘，所以延迟峰值常集中在 `fsync`。偶发慢通常与脏页堆积、设备队列拥塞、日志刷盘组提交有关。

定位顺序：
1. `vmstat` 看脏页和回写压力。
2. `iostat -x` 看磁盘队列长度、await、util。
3. 结合应用日志确认慢点是在 `write` 还是 `fsync`。

优化方向：
- 控制单次刷盘批量，避免突发大写入
- 对关键日志和数据分盘，减少争用
- 合理设置刷盘策略，避免脏页长期积压后集中回写

#### Q28: 操作系统层面，如何判断是 CPU 忙、CPU 等、还是 CPU 被抢占导致慢？

**答案**：
可用“三步法”快速判断：
1. 看 CPU 组成：`mpstat`/`top` 区分 `usr/sys/iowait/steal`。
2. 看调度排队：`vmstat`/`pidstat -w` 观察 run queue 和上下文切换。
3. 看虚拟化争用：若 `steal` 高，说明 vCPU 被宿主机抢占，不是应用本身算不动。

判读要点：
- `usr/sys` 高：CPU 真忙。
- `iowait` 高：CPU 在等 IO。
- `steal` 高：CPU 时间被虚拟化层拿走。

这三类慢路径治理手段完全不同，先分型再优化。

### 开放式设计题

**D1：设计一个高性能的用户态网络协议栈（Bypass内核），需要考虑哪些方面？**

**参考思路**：
- 动机：内核协议栈开销大（syscall/拷贝/中断/锁）、DPDK/XDP可Bypass
- 核心组件：网卡驱动(UIO/VFIO)→ 收发包引擎(Ring Buffer) → 协议解析(TCP/UDP) → 应用API
- 内存管理：大页(HugePages)减少TLB Miss、零拷贝(mbuf池化)
- CPU绑定：Poll模式(忙轮询)替代中断、核隔离(isolcpus)专核专用
- 权衡：性能极高但放弃了内核生态（iptables/tcpdump不可用）、开发复杂度高

**D2：一台Linux服务器突然完全无响应（SSH不上、Ping不通），但物理上还在运行，如何诊断？**

**参考思路**：
- 带外管理：BMC/IPMI远程KVM查看控制台输出、是否有Kernel Panic/Oops信息
- 常见原因：内核死锁（softlockup/hardlockup）、OOM后内核卡住、文件系统只读（磁盘故障）
- 事后分析：如果配置了kdump则分析vmcore、dmesg/syslog中的最后几行日志
- 预防：开启NMI Watchdog检测hardlockup、配置kdump、设置内核参数panic_on_oops=1自动重启

## 总结

**操作系统核心知识点**：

1. **进程/线程**：调度（CFS/RT）、IPC（管道/共享内存/信号）、GMP模型、上下文切换
2. **内存**：虚拟内存、多级页表、TLB、缓存一致性、内存屏障、Buddy System、TCMalloc
3. **IO**：epoll vs select、零拷贝（sendfile/mmap）、Direct IO
4. **文件系统**：inode、page cache、fsync、ext4布局
5. **网络**：TCP握手、拥塞控制、Socket优化
6. **Cgroup资源隔离**：CPU（cpuset/cpuquota/cpushares）、内存（OOM/压力通知）、IO（blkio/调度算法）、网络（tc/HTB）、V2统一层级与PSI
7. **内核深入**：内存寻址（分段/分页/页表标记位）、中断处理（上半部/下半部/softirq/tasklet/workqueue）、内核锁（spinlock/mutex/RCU）、内核抢占、lockup检测（soft/hard）、D状态与load average、系统调用传参

**性能优化方向**：
- 减少系统调用（批量操作、vDSO）
- 减少内存拷贝（零拷贝）
- 利用page cache
- Cgroup资源隔离与限制
- 异步IO
- 避免伪共享
