## 1. 微服务基础

### 1.1 单体架构 vs 微服务架构

**单体架构**：

```
┌─────────────────────────────────┐
│         单体应用              │
│  ┌──────┬──────┬──────┬──────┐ │
│  │ 用户 │ 订单 │ 商品 │ 支付 │ │
│  └──────┴──────┴──────┴──────┘ │
│         ↓                     │
│      单一数据库               │
└─────────────────────────────────┘

优点：
- 开发简单（单一代码库）
- 部署简单（一个war/jar包）
- 调试简单（本地运行）

缺点：
- 难以扩展（必须整体扩展）
- 部署慢（一个模块改动，全部重新部署）
- 技术栈受限（必须用同一种语言/框架）
- 单点故障（一个模块崩溃，整个应用不可用）
```

**微服务架构**：

```
┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐
│ 用户服务│  │ 订单服务│  │ 商品服务│  │ 支付服务│
│   DB   │  │   DB   │  │   DB   │  │   DB   │
└────────┘  └────────┘  └────────┘  └────────┘
    ↓           ↓           ↓           ↓
         API网关（统一入口）
              ↓
          外部请求

优点：
- 独立部署（订单服务改动，不影响其他服务）
- 独立扩展（订单服务流量大，单独扩容）
- 技术异构（用户服务用Go，订单服务用Java）
- 故障隔离（订单服务崩溃，用户服务仍可用）

缺点：
- 运维复杂（服务数量多）
- 分布式事务（跨服务事务难处理）
- 服务间调用开销（网络延迟）
- 调试困难（请求跨多个服务）
```

---

### 1.2 微服务拆分原则

**按业务能力拆分**：

```
❌ 不推荐（按技术层拆分）：
- 用户Controller服务
- 用户Service服务
- 用户DAO服务

✅ 推荐（按业务能力拆分）：
- 用户服务（用户注册、登录、信息管理）
- 订单服务（创建订单、支付订单、订单查询）
- 商品服务（商品管理、库存管理）
```

**单一职责原则**：

```
✅ 每个服务只负责一个业务能力
❌ 避免"上帝服务"（OrderService包含支付、物流、库存...）
```

**高内聚、低耦合**：

```
✅ 服务内部高内聚（订单创建、支付、查询都在订单服务）
✅ 服务之间低耦合（订单服务调用支付服务通过API，不直接访问数据库）
```

## 2. 服务注册与发现

### 2.1 为什么需要服务注册与发现？

**传统方式（硬编码IP）**：

```go
// 订单服务调用用户服务
resp, err := http.Get("http://192.168.1.101:8080/user/123")

问题：
1. IP变化（扩容/缩容/故障）需要修改代码
2. 负载均衡需要自己实现
3. 健康检查需要自己实现
```

**服务注册与发现**：

```go
// 1. 用户服务启动时注册到注册中心
consul.Register("user-service", "192.168.1.101:8080")

// 2. 订单服务从注册中心获取用户服务地址
instances := consul.Discover("user-service")
// 返回：["192.168.1.101:8080", "192.168.1.102:8080", "192.168.1.103:8080"]

// 3. 负载均衡选择一个实例
instance := loadBalance(instances)
resp, err := http.Get(instance + "/user/123")

优势：
- 动态感知服务上下线
- 自动负载均衡
- 故障实例自动摘除
```

---

### 2.2 Consul原理⭐⭐⭐

**Consul架构**：

```
Consul集群（3-5个Server节点，Raft一致性）
    ↑
    │ 注册/发现
    ↓
Consul Agent（每个服务节点）
    ↑
    │ 健康检查
    ↓
微服务实例
```

**服务注册流程**：

```go
// 1. 服务启动时注册
registration := &consul.AgentServiceRegistration{
    ID:      "user-service-1",
    Name:    "user-service",
    Address: "192.168.1.101",
    Port:    8080,
    Check: &consul.AgentServiceCheck{
        HTTP:     "http://192.168.1.101:8080/health",
        Interval: "10s",  // 每10秒健康检查
        Timeout:  "3s",
    },
}
client.Agent().ServiceRegister(registration)

// 2. Consul Agent定期健康检查
// GET http://192.168.1.101:8080/health
// 200 OK → 健康
// 非200或超时 → 不健康（自动摘除）

// 3. 其他服务发现
services, _ := client.Health().Service("user-service", "", true, nil)
// 返回健康的实例列表
```

**为什么Consul用Raft一致性？**

```
场景：3个Consul Server节点

写入操作（服务注册）：
1. 客户端请求Leader
2. Leader写入log
3. Leader复制log到Follower（至少n/2+1节点确认）
4. Leader提交log，返回客户端
5. Leader通知Follower提交

为什么不用最终一致性（Gossip）？
- 服务注册需要强一致性（不能出现脑裂）
- Raft保证：只有一个Leader，数据不会冲突

为什么健康检查用Gossip？
- 健康状态可以容忍短暂不一致
- Gossip传播更快（O(log n)跳）
```

### 2.3 Etcd vs Consul vs Nacos

| 特性 | Etcd | Consul | Nacos |
|------|------|--------|-------|
| **一致性协议** | Raft | Raft | Raft+Distro |
| **健康检查** | 需要自己实现 | 内置（HTTP/TCP/脚本） | 内置 |
| **负载均衡** | 需要自己实现 | 内置 | 内置 |
| **多数据中心** | 不支持 | 支持 | 支持 |
| **UI界面** | 无 | 有 | 有 |
| **适用场景** | K8s元数据存储 | 微服务注册发现 | 微服务注册发现+配置中心 |

## 3. API网关

### 3.1 为什么需要API网关？

**无网关架构问题**：

```
客户端直接调用微服务：
移动端 → 用户服务
      → 订单服务
      → 商品服务

问题：
1. 客户端需要知道所有服务地址（耦合）
2. 每个服务都要实现认证、限流、日志（重复）
3. 跨域问题（CORS）
4. 协议转换（HTTP → gRPC）
```

**网关架构**：

```
客户端 → API网关 → 用户服务
                → 订单服务
                → 商品服务

网关统一处理：
- 认证鉴权
- 限流熔断
- 日志监控
- 协议转换
- 路由转发
```

---

### 3.2 Kong网关架构⭐⭐⭐

**Kong核心组件**：

```
客户端请求
    ↓
Kong Server（OpenResty + Lua）
    ↓
插件链（认证 → 限流 → 日志 → ...）
    ↓
Upstream（后端服务）
```

**Kong插件机制**：

```lua
-- 限流插件示例
function RateLimitPlugin:access(config)
  local current = redis:incr(key)
  
  if current > config.limit then
    return kong.response.exit(429, "Too Many Requests")
  end
  
  if current == 1 then
    redis:expire(key, config.period)
  end
end

-- 请求流程
请求 → 认证插件（JWT验证）
    → 限流插件（检查速率）
    → 日志插件（记录日志）
    → 转发到后端服务
```

**为什么Kong用OpenResty？**

```
OpenResty = Nginx + LuaJIT

优势：
1. 高性能（Nginx的事件驱动 + LuaJIT的JIT编译）
2. 灵活（Lua脚本动态扩展）
3. 生态丰富（大量Nginx插件）

性能：
- Kong：10万QPS（单机）
- Spring Cloud Gateway：5万QPS（单机）
```

### 3.3 网关路由策略

**基于路径路由**：

```yaml
# /api/user/* → 用户服务
# /api/order/* → 订单服务

routes:
  - path: /api/user/*
    service: user-service
  - path: /api/order/*

```

```yaml
    match:
      headers:
        X-Version: v2
    service: order-service-v2
  - path: /api/order/*
    service: order-service-v1
```

## 4. 服务熔断与降级

### 4.1 为什么需要熔断？

**雪崩效应**：

```
用户服务 → 订单服务 → 商品服务（故障，响应慢5秒）
                    ↓
            订单服务线程池耗尽
                    ↓
            用户服务线程池耗尽
                    ↓
            整个系统不可用

熔断的作用：
- 快速失败（不等待5秒）
- 保护上游服务（防止雪崩）
- 给下游服务恢复时间
```

---

### 4.2 Hystrix熔断原理⭐⭐⭐

**熔断器三种状态**：

```
1. Closed（关闭）：
   - 正常请求通过
   - 记录失败率

2. Open（打开）：
   - 失败率 > 阈值（如50%）
   - 所有请求直接失败（快速失败）
   - 进入休眠期（如5秒）

3. Half-Open（半开）：
   - 休眠期后，允许少量请求通过（探测）
   - 成功 → Closed
   - 失败 → Open

状态转换：
Closed → Open（失败率>阈值）
Open → Half-Open（休眠期结束）
Half-Open → Closed（探测成功）
Half-Open → Open（探测失败）
```

**熔断器实现**：

```go
type CircuitBreaker struct {
    state         State  // Closed/Open/Half-Open
    failureCount  int
    successCount  int
    lastFailTime  time.Time
}

func (cb *CircuitBreaker) Call(fn func() error) error {
    if cb.state == Open {
        // 检查是否可以进入Half-Open
        if time.Since(cb.lastFailTime) > sleepWindow {
            cb.state = HalfOpen
        } else {
            return ErrCircuitOpen  // 快速失败
        }
    }
    
    err := fn()
    
    if err != nil {
        cb.onFailure()
    } else {
        cb.onSuccess()
    }
    
    return err
}

func (cb *CircuitBreaker) onFailure() {
    cb.failureCount++
    cb.lastFailTime = time.Now()
    
    if cb.state == HalfOpen {
        cb.state = Open  // 探测失败，重新打开
    } else if float64(cb.failureCount)/float64(cb.totalCount) > threshold {
        cb.state = Open  // 失败率超过阈值，打开
    }
}

func (cb *CircuitBreaker) onSuccess() {
    if cb.state == HalfOpen {
        cb.successCount++
        if cb.successCount >= successThreshold {
            cb.state = Closed  // 探测成功，关闭
            cb.failureCount = 0
        }
    }
}
```

**为什么熔断器能防止雪崩？**

```
无熔断器：
订单服务调用商品服务（5秒超时）
100个请求同时调用 → 100个线程阻塞5秒 → 线程池耗尽

有熔断器：
第1个请求：5秒超时（失败）
第2个请求：5秒超时（失败）
...
第10个请求：失败率>50% → 熔断器打开
第11-100个请求：立即失败（<1ms）→ 线程池未耗尽

节省时间：100 × 5秒 = 500秒 → 10 × 5秒 + 90 × 0.001秒 = 50秒
```

---

### 4.3 降级策略

**降级 vs 熔断**：

```
熔断：服务故障时，停止调用（快速失败）
降级：服务故障时，返回兜底数据（保证可用性）

示例：
// 熔断
err := orderService.GetOrder(id)
if err != nil {
    return err  // 直接返回错误
}

// 降级
err := orderService.GetOrder(id)
if err != nil {
    return defaultOrder  // 返回默认订单（兜底）
}
```

**降级策略**：

```
1. 返回默认值
   - 商品服务故障 → 返回空商品列表

2. 返回缓存数据
   - 订单详情服务故障 → 返回Redis缓存的订单

3. 降级到备用服务
   - 主支付服务故障 → 调用备用支付服务

4. 降级功能
   - 推荐服务故障 → 不展示推荐商品（不影响核心功能）
```

---

### 4.4 Sentinel vs Hystrix

| 特性 | Hystrix | Sentinel |
|------|---------|----------|
| **熔断粒度** | 线程池隔离 | 信号量隔离 |
| **实时监控** | 需要Dashboard | 内置实时监控 |
| **规则配置** | 代码配置 | 动态配置（控制台） |
| **性能** | 较低（线程切换） | 高（无线程切换） |
| **状态** | Netflix已停止维护 | 阿里维护中 |

**Sentinel优势**：

```
1. 实时监控：
   - 秒级监控（QPS、RT、成功率）
   - 可视化Dashboard

2. 动态规则：
   - 无需重启，动态修改限流规则
   - 规则持久化（Nacos/Apollo）

3. 丰富的降级策略：
   - 慢调用比例
   - 异常比例
   - 异常数
```

## 5. 分布式追踪

### 5.1 为什么需要分布式追踪？

**问题场景**：

```
用户请求 → 网关 → 用户服务 → 订单服务 → 商品服务
                              → 库存服务
                              → 支付服务

问题：
1. 请求慢，不知道哪个服务慢
2. 请求失败，不知道哪个服务失败
3. 调用链路复杂，难以调试
```

**分布式追踪**：

```
TraceID: abc123（唯一标识一个请求）
  ├─ SpanID: span1（网关，耗时10ms）
  ├─ SpanID: span2（用户服务，耗时20ms）
  └─ SpanID: span3（订单服务，耗时100ms）
      ├─ SpanID: span4（商品服务，耗时50ms）
      ├─ SpanID: span5（库存服务，耗时30ms）
      └─ SpanID: span6（支付服务，耗时20ms）

分析：
- 总耗时130ms（10+20+100）
- 瓶颈：订单服务（100ms）
  - 其中商品服务最慢（50ms）
```

---

### 5.2 OpenTelemetry原理⭐⭐⭐

**核心概念**：

```
1. Trace：一次完整的请求链路（TraceID唯一）
2. Span：一个操作（SpanID唯一）
3. Context：上下文传播（TraceID + SpanID）

数据结构：
Trace {
    TraceID: "abc123"
    Spans: [
        Span {
            SpanID: "span1"
            ParentSpanID: ""  // 根Span
            Name: "GET /api/order"
            StartTime: 1234567890
            EndTime:   1234567900
            Attributes: {
                http.method: "GET"
                http.status_code: 200
            }
        },
        Span {
            SpanID: "span2"
            ParentSpanID: "span1"  // 父Span
            Name: "OrderService.GetOrder"
            ...
        }
    ]
}
```

**上下文传播（Context Propagation）**：

```
请求流：
客户端 → 网关 → 订单服务 → 商品服务

步骤1：客户端生成TraceID
TraceID = "abc123"
SpanID = "span1"

步骤2：网关接收请求，提取TraceID
从Header提取：Traceparent: 00-abc123-span1-01
创建新Span：SpanID = "span2"，ParentSpanID = "span1"

步骤3：网关调用订单服务，传递TraceID
Header: Traceparent: 00-abc123-span2-01

步骤4：订单服务接收请求，提取TraceID
创建新Span：SpanID = "span3"，ParentSpanID = "span2"

步骤5：订单服务调用商品服务，传递TraceID
Header: Traceparent: 00-abc123-span3-01

...

最终：所有Span都有相同的TraceID，可以串联起来
```

**Traceparent格式（W3C标准）**：

```
Traceparent: 00-abc123-span1-01
             ││  │      │     │
             ││  │      │     └─ flags（是否采样）
             ││  │      └─ parent-id（父SpanID）
             ││  └─ trace-id（TraceID）
             │└─ version（版本）
             └─ 00（固定）
```

### 5.3 Jaeger架构

**组件**：

```
应用 → Jaeger Agent（本地） → Jaeger Collector → 存储（Elasticsearch）
                                                 ↓
                                            Jaeger UI
```

**采样策略**：

```
1. 全量采样（const sampler）：
   - 采样率100%
   - 适合开发环境

2. 概率采样（probabilistic sampler）：
   - 采样率1%（100个请求采样1个）
   - 适合高流量生产环境

3. 限速采样（rate limiting sampler）：
   - 每秒最多采样10个
   - 避免采样数据过多

4. 自适应采样（adaptive sampler）：
   - 根据流量动态调整采样率
   - 保证采样数据量稳定
```

## 6. 配置中心

### 6.1 为什么需要配置中心？

**传统配置方式问题**：

```
配置写在application.yml：
- 修改配置需要重启服务
- 多个服务配置重复（数据库地址）
- 配置散落各处，难以管理
```

**配置中心**：

```
Apollo/Nacos：
- 集中管理配置
- 动态更新配置（无需重启）
- 配置版本管理（回滚）
- 配置权限管理
```

---

### 6.2 Apollo架构

**组件**：

```
1. Config Service：
   - 提供配置查询接口
   - 长轮询推送配置变更

2. Admin Service：
   - 配置管理接口
   - Portal调用

3. Portal：
   - 配置管理界面

4. Client：
   - 应用内的SDK
   - 长轮询获取配置
```

**配置热更新原理**：

```
步骤1：应用启动，Client拉取配置
GET /configs?appId=order-service&namespace=application

步骤2：Client启动长轮询
GET /notifications/v2?appId=order-service&notifications=[...]
（Config Service hold住请求，60秒超时）

步骤3：运维在Portal修改配置
Portal → Admin Service → Config Service

步骤4：Config Service立即返回长轮询请求
返回：{namespace: "application", notificationId: 2}

步骤5：Client收到通知，拉取新配置
GET /configs?appId=order-service&namespace=application

步骤6：Client触发回调
@ApolloConfigChangeListener
public void onChange(ConfigChangeEvent event) {
    String newValue = event.getChange("timeout").getNewValue();
    // 更新本地配置（无需重启）
}
```

## 7. 分布式事务⭐⭐⭐

### 7.1 分布式事务的挑战

**为什么微服务需要分布式事务？**

```
单体架构：
用户服务、订单服务、库存服务 → 同一个数据库
@Transactional → 本地事务保证ACID

微服务架构：
用户服务 → 用户库
订单服务 → 订单库
库存服务 → 库存库
无法使用本地事务！

典型场景：下单
1. 订单服务：创建订单
2. 库存服务：扣减库存
3. 账户服务：扣减余额

如果库存扣减成功，但余额扣减失败？
- 数据不一致：订单创建、库存扣了，但钱没扣
```

---

### 7.2 Saga模式

**正向操作 + 补偿操作**：

```
Saga工作流程：
T1 → T2 → T3 → 成功
       ↓
      失败
       ↓
C2 ← C1（补偿）

T: Transaction（正向操作）
C: Compensation（补偿操作）

示例：下单流程
T1: 创建订单（状态=待支付）  | C1: 取消订单
T2: 扣减库存                | C2: 恢复库存  
T3: 扣减余额                | C3: 恢复余额

场景1：全部成功
T1 → T2 → T3 → 订单完成

场景2：T3失败（余额不足）
T1 → T2 → T3(失败) → C2 → C1
创建订单 → 扣库存 → 扣余额失败 → 恢复库存 → 取消订单
```

**Saga编排方式**：

```
1. 编排式（Choreography）：
   - 事件驱动，服务间通过消息通信
   - 订单服务 → 发布"订单创建"事件
   - 库存服务 → 监听事件，扣减库存，发布"库存扣减"事件
   - 优点：松耦合
   - 缺点：流程难追踪，复杂时难以管理

2. 协调式（Orchestration）：
   - 有一个中心协调器（Saga Orchestrator）
   - 协调器依次调用各服务
   - 优点：流程清晰，易于管理
   - 缺点：协调器是单点

推荐：简单流程用编排式，复杂流程用协调式
```

---

### 7.3 TCC模式

**Try-Confirm-Cancel**：

```
三个阶段：
1. Try：资源预留（不真正执行）
2. Confirm：确认执行（真正扣减）
3. Cancel：取消预留（回滚）

示例：扣减余额100元
Try：
  UPDATE account SET frozen = frozen + 100 WHERE balance >= 100
  （冻结100元，但balance不变）

Confirm：
  UPDATE account SET balance = balance - 100, frozen = frozen - 100
  （真正扣减balance，解除冻结）

Cancel：
  UPDATE account SET frozen = frozen - 100
  （解除冻结，回滚）

TCC vs Saga：
- TCC：资源预留，隔离性更好
- Saga：无预留，可能出现"脏读"
```

**TCC适用场景**：

```
- 资金类业务（扣款、转账）
- 需要强隔离性
- 性能要求高（无锁等待）

难点：
- 每个服务都要实现Try/Confirm/Cancel三个接口
- 业务侵入性强
```

### 7.4 本地消息表

**可靠消息最终一致性**：

```
步骤1：本地事务写入业务数据 + 消息表
BEGIN;
  INSERT INTO orders (id, status) VALUES (1, 'created');
  INSERT INTO message_table (id, content, status) VALUES (1, '订单1创建', 'pending');
COMMIT;

步骤2：后台任务定时扫描消息表
SELECT * FROM message_table WHERE status = 'pending';

步骤3：发送消息到MQ
producer.send(message);
UPDATE message_table SET status = 'sent' WHERE id = 1;

步骤4：消费者消费消息
库存服务：扣减库存
consumer.ack();

步骤5：收到确认，更新消息状态
UPDATE message_table SET status = 'consumed' WHERE id = 1;

优点：
- 无分布式事务协调器
- 依赖MQ的可靠投递

缺点：
- 消息延迟
- 需要定时任务
```

### 7.5 最大努力通知

```
适用场景：
- 对一致性要求不高
- 允许重试通知

示例：支付结果通知
1. 用户支付成功
2. 支付系统通知商户系统
3. 商户系统返回成功 → 结束
   商户系统返回失败或超时 → 重试（最多N次）

重试策略：
- 第1次：立即
- 第2次：1分钟后
- 第3次：5分钟后
- ...
- 超过重试次数：人工介入
```

## 8. 服务间通信⭐⭐

### 8.1 同步通信 vs 异步通信

```
同步通信（REST/gRPC）：
订单服务 ─调用→ 库存服务 ─响应→ 订单服务
优点：实时性高，调用链清晰
缺点：服务耦合，调用方阻塞

异步通信（消息队列）：
订单服务 ─发消息→ MQ ─消费→ 库存服务
优点：解耦，削峰
缺点：复杂度高，最终一致性

选择依据：
- 需要立即响应：同步
- 可以容忍延迟：异步
- 调用链长：异步（避免阻塞）
```

---

### 8.2 gRPC vs REST

```
特性对比：
| 特性       | REST          | gRPC              |
|------------|---------------|-------------------|
| 协议       | HTTP/1.1      | HTTP/2            |
| 序列化     | JSON          | Protobuf（二进制）|
| 性能       | 较低          | 高                |
| 浏览器支持 | 好            | 需要grpc-web      |
| 调试       | 简单（可读）  | 复杂（二进制）    |

gRPC优势：
1. 高性能：Protobuf序列化 + HTTP/2多路复用
2. 强类型：protobuf定义接口，编译时检查
3. 流式传输：支持双向流

适用场景：
- 内部服务通信：gRPC（高性能）
- 对外API：REST（兼容性好）
```

### 8.3 服务版本管理

```
API版本策略：
1. URL路径版本：
   /api/v1/orders
   /api/v2/orders

2. Header版本：
   Accept: application/vnd.api.v1+json
   Accept: application/vnd.api.v2+json

2. 通知客户端迁移到v2
3. v1流量降到0后，下线v1
```

## 9. 限流算法⭐⭐⭐

### 9.1 固定窗口计数器

```
原理：
- 将时间划分为固定窗口（如1秒）
- 每个窗口内计数
- 超过阈值则拒绝

实现：
┌─────────┐ ┌─────────┐ ┌─────────┐
│ 0-1秒   │ │ 1-2秒   │ │ 2-3秒   │
│ 计数:80 │ │ 计数:60 │ │ 计数:?  │
│ 限制:100│ │ 限制:100│ │ 限制:100│
└─────────┘ └─────────┘ └─────────┘

问题：边界突刺
- 0.9-1.0秒：100个请求
- 1.0-1.1秒：100个请求
- 0.2秒内200个请求（超过限制）
```

---

### 9.2 滑动窗口

```
原理：
- 窗口滑动，而不是跳变
- 细分为多个小窗口

实现：
时间线：|----|----|----|----|
       0  0.25 0.5 0.75  1秒

滑动窗口（0.5-1.5秒）：
- 0.5-0.75秒：20个请求
- 0.75-1.0秒：30个请求
- 1.0-1.25秒：25个请求
- 总计：75个请求 < 100，通过

优点：
- 解决边界突刺问题
- 更平滑的限流

Redis实现：
ZADD rate_limiter <timestamp> <request_id>
ZREMRANGEBYSCORE rate_limiter 0 <current_time - window_size>
ZCARD rate_limiter
```

---

### 9.3 令牌桶

```
原理：
- 桶中有令牌，每个请求消耗一个令牌
- 令牌以固定速率添加
- 桶满则丢弃新令牌

特点：
- 允许突发流量（桶中有令牌就通过）
- 但平均速率受限（令牌添加速率）

实现：
tokens = min(capacity, tokens + (now - lastRefill) * rate)
if tokens >= 1:
    tokens -= 1
    allow()
else:
    reject()

参数：
- capacity: 桶容量（决定突发量）
- rate: 令牌添加速率（决定平均速率）

适用场景：
- 允许短时突发，但限制平均速率
- 如：限制API每秒100请求，但允许短时200请求
```

---

### 9.4 漏桶

```
原理：
- 请求进入桶
- 请求以固定速率流出
- 桶满则拒绝新请求

特点：
- 强制恒定速率（不允许突发）
- 平滑输出

对比：
令牌桶：允许突发，控制平均速率
漏桶：强制恒定速率，不允许突发

适用场景：
- 对下游有严格速率限制
- 如：调用第三方API，限制每秒10次
```

## 11. 服务治理实战

### 11.1 灰度发布策略⭐⭐⭐

**三种主流灰度策略对比**：

```
1. 金丝雀发布（Canary）：
   ┌──────────────────────────────────────┐
   │  生产流量                            │
   │  ──────────────────────→             │
   │     95%  ──→  v1（稳定版）           │
   │      5%  ──→  v2（新版本，金丝雀）   │
   │                                      │
   │  逐步调整：5% → 10% → 25% → 50% → 100% │
   └──────────────────────────────────────┘
   
   特点：
   - 小比例流量先验证新版本
   - 发现问题影响范围小
   - 适合持续交付场景

2. 蓝绿部署（Blue-Green）：
   ┌───────────────────────────────────────┐
   │  阶段1：蓝（v1）接收全部流量         │
   │         绿（v2）部署就绪但不接流量   │
   │                                       │
   │  阶段2：流量一次性切换到绿（v2）     │
   │         蓝（v1）保留用于回滚         │
   └───────────────────────────────────────┘
   
   特点：
   - 瞬间切换，回滚迅速
   - 需要双倍资源
   - 适合对停机零容忍的场景

3. A/B测试：
   ┌───────────────────────────────────────┐
   │  按用户特征分流：                     │
   │  - 地域：北京 → v2，其他 → v1        │
   │  - 用户标签：VIP → v2，普通 → v1     │
   │  - 设备：iOS → v2，Android → v1      │
   └───────────────────────────────────────┘
   
   特点：
   - 按业务维度精确分流
   - 可收集不同版本的业务指标对比
   - 适合产品功能实验
```

**流量染色机制**：

```
流量染色：在请求入口打上标记，沿调用链全程透传，
各服务根据标记决定路由到哪个版本。

请求流：
客户端 → 网关（打染色标记） → 服务A → 服务B → 服务C
         X-Traffic-Tag: canary    canary   canary   canary

染色规则：
1. 网关层：根据Header/Cookie/IP/用户ID决定染色
2. 透传层：每个服务透传染色标记到下游（HTTP Header/gRPC Metadata）
3. 路由层：每个服务根据染色标记选择对应版本实例

染色透传原理：
┌─────────┐  X-Tag:canary  ┌─────────┐  X-Tag:canary  ┌─────────┐
│  网关    │ ────────────→  │ 订单v2  │ ────────────→  │ 库存v2  │
└─────────┘                 └─────────┘                 └─────────┘
     │                           │                           │
     │  无标记                   │  无标记                   │  无标记
     ▼                           ▼                           ▼
┌─────────┐                 ┌─────────┐                 ┌─────────┐
│ 订单v1  │                 │ 库存v1  │                 │ 支付v1  │
└─────────┘                 └─────────┘                 └─────────┘
```

**Go代码示例：流量染色中间件**：

```go
package middleware

import (
    "context"
    "net/http"

    "google.golang.org/grpc/metadata"
)

const TrafficTagKey = "X-Traffic-Tag"

// HTTP中间件：从请求中提取染色标记，注入到context
func TrafficTagMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        tag := r.Header.Get(TrafficTagKey)
        if tag == "" {
            tag = "stable" // 默认稳定版
        }
        ctx := context.WithValue(r.Context(), TrafficTagKey, tag)
        next.ServeHTTP(w, r.WithContext(ctx))
    })
}

// 调用下游HTTP服务时透传染色标记
func ForwardWithTag(ctx context.Context, req *http.Request) {
    if tag, ok := ctx.Value(TrafficTagKey).(string); ok {
        req.Header.Set(TrafficTagKey, tag)
    }
}

// gRPC拦截器：透传染色标记到gRPC Metadata
func GRPCTagInterceptor(
    ctx context.Context,
) context.Context {
    tag, _ := ctx.Value(TrafficTagKey).(string)
    if tag != "" {
        md := metadata.Pairs(TrafficTagKey, tag)
        ctx = metadata.NewOutgoingContext(ctx, md)
    }
    return ctx
}

// 根据染色标记选择目标实例
func SelectInstance(ctx context.Context, stableAddr, canaryAddr string) string {
    tag, _ := ctx.Value(TrafficTagKey).(string)
    if tag == "canary" {
        return canaryAddr
    }
    return stableAddr
}
```

### 11.2 全链路压测⭐⭐⭐

**为什么需要全链路压测？**

```
传统压测的局限：
- 只压单个服务，无法发现链路瓶颈
- 测试环境与生产环境差异大
- 无法验证系统整体容量

全链路压测：
- 在生产环境或等比仿真环境进行
- 覆盖完整业务链路（网关 → 服务A → 服务B → DB/缓存）
- 使用压测流量标记，与正常流量隔离
```

**影子库/影子表方案**：

```
核心问题：压测流量不能污染生产数据

方案1：影子库（Shadow Database）
┌──────────────┐     正常流量      ┌──────────────┐
│   服务实例    │ ──────────────→  │   生产数据库   │
│              │                   └──────────────┘
│              │     压测流量      ┌──────────────┐
│              │ ──────────────→  │   影子数据库   │
└──────────────┘                   └──────────────┘

特点：
- 完全物理隔离，安全性最高
- 成本高（需要额外数据库实例）
- 适合核心交易场景

方案2：影子表（Shadow Table）
┌──────────────┐
│   同一数据库   │
│  ┌──────────┐ │
│  │ orders   │ │ ← 正常流量写入
│  ├──────────┤ │
│  │ orders_shadow │ ← 压测流量写入
│  └──────────┘ │
└──────────────┘

特点：
- 同库不同表，成本低
- 需要在SQL层做路由判断
- 适合读多写少的非核心场景

方案3：数据偏移（Data Offset）
- 压测数据使用特定前缀/ID段
- 如：压测用户ID > 900000000
- 压测结束后按ID段批量清理
```

**压测标记透传**：

```
压测标记全链路透传流程：

客户端（压测工具）
  │ Header: X-Stress-Test: true
  ▼
API网关
  │ 识别压测标记，注入context
  ▼
服务A（HTTP调用）
  │ Header透传：X-Stress-Test: true
  ▼
服务B（gRPC调用）
  │ Metadata透传：x-stress-test: true
  ▼
数据层路由
  │ 根据标记选择：影子库/影子表/偏移ID
  ▼
消息队列
  │ Message属性透传：stress_test=true
  │ 消费者根据标记写入影子存储
  ▼
缓存层
  │ Key前缀隔离：shadow:order:123
  │ 避免污染正常缓存

关键组件改造：
1. HTTP Client：自动透传Header
2. gRPC Interceptor：自动透传Metadata
3. MQ Producer：自动注入Message属性
4. DB Middleware：根据标记路由到影子库/表
5. Cache Client：根据标记添加Key前缀
```

**隔离方案全景**：

```
┌─────────────────────────────────────────────────────┐
│                 全链路压测隔离矩阵                   │
├──────────────┬──────────────────┬───────────────────┤
│     组件      │    隔离策略       │    注意事项       │
├──────────────┼──────────────────┼───────────────────┤
│  MySQL       │  影子库/影子表    │  binlog需过滤     │
│  Redis       │  Key前缀隔离     │  设置较短TTL      │
│  MQ(Kafka)   │  影子Topic       │  消费者组也隔离   │
│  ES          │  影子索引        │  索引名加后缀     │
│  日志        │  标记透传到日志   │  便于压测分析     │
│  监控        │  标记维度隔离    │  分开统计压测指标  │
│  第三方API   │  Mock或沙箱环境  │  禁止调用生产API  │
└──────────────┴──────────────────┴───────────────────┘

容量评估公式：
- 目标QPS = 日常峰值 × 安全系数（通常1.5-3倍）
- 压测时间 = 持续15-30分钟（观察资源趋势）
- 递增策略：20% → 50% → 80% → 100% → 120%目标QPS
```

**Istio流量管理核心资源**：

```
Istio流量管理涉及两个核心CRD：

1. VirtualService（虚拟服务）：
   - 定义流量路由规则
   - 类似网关层的路由表
```

```yaml
  name: order-service
```

```yaml
    # 负载均衡策略

      simple: LEAST_REQUEST      # 最少请求优先
  # 定义子集（subset）

  - name: stable

  - name: canary

  - name: stress
```

## 12. 面试题自查

### 微服务治理补充

- 微服务不是拆分数量竞赛，核心是边界清晰、故障隔离和独立交付。
- 同步调用链要有预算意识：总超时预算、重试预算、并发预算必须前置定义。
- 服务治理需要和发布治理联动，灰度、回滚、观测口径要纳入同一套门禁。
- 业务增长后通常会从“服务可用”走向“依赖可控”，重点是降级策略和依赖分级。

### 初级高频速刷

**1. 单体架构和微服务架构的最大区别是什么？**

单体把功能集中在一个进程或一个部署单元里，微服务把系统拆成多个独立部署、独立演进的服务。区别不只是代码拆开了，而是团队协作、部署、数据一致性和运维复杂度都变了。

**2. 服务注册中心主要解决什么问题？**

注册中心解决服务实例地址动态变化后的发现问题。没有注册中心，服务扩缩容、重启、迁移后，上游就很难稳定找到下游。

**3. API 网关在微服务里通常承担哪些职责？**

API 网关通常做统一路由、鉴权、限流、协议转换、灰度发布和统一日志。它不是简单的“转发器”，而是微服务统一入口控制面的一部分。

**4. 为什么微服务通常需要配置中心，而不是把配置写死在服务里？**

因为微服务实例多、环境多、变更频繁，如果每次改配置都要重发版，效率和风险都会很高。配置中心的价值不只是“集中存一份配置”，而是让配置变更具备统一治理、审计和动态生效能力。

**5. 为什么契约测试在微服务里比单体更重要？**

因为服务之间通过 API 或消息契约协作，一方改了字段或语义，另一方很可能在运行时才暴露问题。契约测试能把这种跨服务兼容性问题提前到 CI 阶段发现，而不是等联调或线上事故。

### 中级高频进阶

**6. 为什么微服务架构里通常不建议共享数据库？**

因为一旦共享数据库，服务之间就会通过表结构和事务语义重新耦合在一起，最终变成“分布式单体”。数据库才是很多系统真正的边界，如果这个边界没分开，服务拆得再细也只是表面工作。

**7. BFF（Backend For Frontend）是什么，什么时候值得引入？**

BFF 是面向特定前端形态的后端聚合层，比如 App、Web、运营后台分别有自己的接口编排逻辑。当不同前端对数据聚合、字段裁剪、交互节奏差异很大时，引入 BFF 能减少前端和核心服务的耦合。

**8. 为什么同步调用链过长在微服务里是高风险设计？**

因为链路越长，延迟越容易叠加，故障点越多，重试和超时也更容易层层放大。很多微服务雪崩，不是某个服务真的挂了，而是长链路下任意一点抖动都被放大到了用户侧。

**9. 为什么“重试”在微服务里不是越多越好？**

因为重试本质是用额外流量换成功率，如果没有退避、上限和幂等保护，失败流量会被放大成重试风暴。正确做法是把重试预算和超时预算绑定，避免把局部故障扩散成全链路拥塞。

**10. 服务 Mesh 和 SDK 治理方案该怎么选？**

Mesh 适合统一治理和多语言团队，能把流量控制、mTLS、观测标准化；SDK 适合轻量团队和简单场景，接入成本低但治理一致性依赖业务方自觉。选型关键是组织规模与治理一致性要求，而不是“哪个更新”。

### 深入问答

#### Q1: 微服务和单体架构的本质区别是什么？为什么很多团队会陷入“分布式单体”？

**答案**：
**本质区别**：
- **部署单元**：单体是一个可部署单元，微服务是多个独立可部署单元
- **数据库**：单体共享数据库，微服务每个服务有独立数据库
- **团队组织**：单体按技术层划分（前端组、后端组），微服务按业务能力划分（订单组、用户组）

**分布式单体常见成因**：
- 服务拆了，但数据库还共享，导致 schema 变更仍然跨团队耦合
- 同步调用链过长，任何下游抖动都会放大到上游
- 发布、配置、观测没有统一治理，问题定位和回滚效率下降

**实践建议**：
- 单体适合小团队和业务早期，先保障交付速度
- 微服务适合边界明确、团队规模上来且需要独立扩展的阶段
- 从单体迁移时优先拆“变更频率高、性能瓶颈明显、故障影响面大”的模块

#### Q2: 微服务如何拆分？有什么原则？

**答案**：
**拆分原则**：
1. **按业务能力拆分**：用户服务、订单服务、支付服务（而非按技术层拆分）
2. **单一职责**：每个服务只负责一个业务能力
3. **高内聚低耦合**：服务内部功能紧密相关，服务之间依赖最小化

**拆分方法**：
- DDD领域驱动设计：识别限界上下文（Bounded Context）
- 事件风暴：通过业务事件识别服务边界
- 康威定律：按团队结构拆分

**避免**：
- 过度拆分（纳米服务）
- 拆分后数据库仍然共享

#### Q3: 服务注册与发现的实现原理？为什么Consul用Raft协议？

**答案**：
**实现原理**：
1. **服务注册**：服务启动时向注册中心注册（IP、端口、健康检查URL）
2. **健康检查**：注册中心定期检查服务状态，不健康实例自动摘除
3. **服务发现**：调用方从注册中心获取可用实例列表
4. **负载均衡**：客户端根据策略（轮询、随机、最小连接）选择实例

**Raft协议原因**：
- 服务注册需要**强一致性**（不能出现脑裂）
- Raft保证：只有一个Leader处理写入，数据不会冲突
- 健康检查用Gossip（允许短暂不一致，传播更快）

#### Q4: API网关有什么作用？Kong为什么性能高？

**答案**：
**网关作用**：
1. **统一入口**：客户端只需知道网关地址
2. **认证鉴权**：JWT验证、OAuth2.0
3. **限流熔断**：保护后端服务
4. **协议转换**：HTTP→gRPC
5. **路由转发**：根据路径/Header转发到不同服务

**Kong高性能原因**：
- 基于OpenResty（Nginx + LuaJIT）
- Nginx的事件驱动模型 + LuaJIT的JIT编译
- 单机可达10万QPS（Spring Cloud Gateway约5万QPS）

#### Q5: 熔断器的三种状态是什么？如何防止雪崩效应？

**答案**：
**三种状态**：
1. **Closed（关闭）**：正常请求通过，记录失败率
2. **Open（打开）**：失败率超阈值，所有请求直接失败（快速失败）
3. **Half-Open（半开）**：休眠期后，允许少量请求探测，成功则关闭，失败则打开

**防止雪崩**：
- 无熔断：100个请求等待5秒超时 → 500秒线程阻塞 → 线程池耗尽 → 级联故障
- 有熔断：前10个失败后熔断打开 → 后90个立即失败（<1ms）→ 线程池保护

#### Q6: Saga和TCC分布式事务有什么区别？各自适用什么场景？

**答案**：
**Saga模式**：
- 正向操作 + 补偿操作（T1→T2→T3，失败则C2→C1）
- 无资源预留，可能出现"脏读"
- 适用：长事务、最终一致性可接受

**TCC模式**：
- Try（预留）+ Confirm（确认）+ Cancel（取消）
- 有资源预留，隔离性更好
- 适用：资金业务、强隔离性要求

**选择建议**：
- 一般业务：Saga（简单）
- 金融业务：TCC（可靠）

#### Q7: 本地消息表如何实现可靠消息？

**答案**：
**实现步骤**：
1. 本地事务同时写入业务表和消息表
2. 后台任务定时扫描待发送消息
3. 发送到消息队列，更新消息状态为"已发送"
4. 消费者消费成功后，更新消息状态为"已消费"

**核心点**：
- 业务操作和消息写入在同一个本地事务中
- 利用MQ的可靠投递保证最终一致性
- 消费者需要幂等（消息可能重复）

#### Q8: 分布式追踪是如何实现的？TraceID如何传递？

**答案**：
**核心概念**：
- **Trace**：一次完整请求的调用链路
- **Span**：一个操作单元（一次RPC调用）
- **Context**：上下文传播（TraceID + ParentSpanID）

**TraceID传递**：
1. 入口服务生成TraceID和SpanID
2. 调用下游服务时，通过HTTP Header传递：`Traceparent: 00-{TraceID}-{SpanID}-01`
3. 下游服务提取TraceID，创建新Span，ParentSpanID指向调用方SpanID
4. 所有Span上报到Collector，根据TraceID和ParentSpanID重建调用链

#### Q9: 配置中心如何实现配置热更新？

**答案**：
**Apollo热更新原理**：
1. 客户端启动时拉取配置
2. 客户端启动长轮询（60秒超时）
3. 配置变更时，服务端立即返回长轮询请求
4. 客户端收到通知，拉取新配置
5. 触发本地回调，应用新配置（无需重启）

**关键技术**：
- 长轮询（Long Polling）：比轮询实时，比WebSocket简单
- 配置回调：应用通过监听器感知配置变化

#### Q10: gRPC和REST有什么区别？各自适用什么场景？

**答案**：
| 特性 | REST | gRPC |
|------|------|------|
| 协议 | HTTP/1.1 | HTTP/2（多路复用） |
| 序列化 | JSON（文本） | Protobuf（二进制） |
| 性能 | 较低 | 高（10倍以上） |
| 浏览器支持 | 原生支持 | 需要grpc-web |

**适用场景**：
- **gRPC**：内部服务通信（高性能、强类型）
- **REST**：对外API（兼容性好、易调试）

#### Q11: 限流算法有哪些？令牌桶和漏桶有什么区别？

**答案**：
**四种限流算法**：
1. **固定窗口**：按时间窗口计数，有边界突刺问题
2. **滑动窗口**：细分时间窗口，更平滑
3. **令牌桶**：令牌固定速率添加，允许突发
4. **漏桶**：请求固定速率流出，强制恒定速率

**令牌桶vs漏桶**：
- 令牌桶：允许突发流量（桶中有令牌就通过），控制平均速率
- 漏桶：强制恒定速率，不允许突发

**使用场景**：
- 令牌桶：用户请求限流（允许短时突发）
- 漏桶：调用第三方API（严格速率限制）

#### Q12: Sentinel和Hystrix有什么区别？为什么推荐Sentinel？

**答案**：
| 特性 | Hystrix | Sentinel |
|------|---------|----------|
| 隔离方式 | 线程池隔离 | 信号量隔离 |
| 性能 | 较低（线程切换） | 高 |
| 规则配置 | 代码配置 | 动态配置（控制台） |
| 监控 | 需要Dashboard | 内置实时监控 |
| 维护状态 | Netflix已停止维护 | 阿里维护中 |

**Sentinel优势**：
- 实时监控：秒级QPS、RT、成功率监控
- 动态规则：无需重启，动态修改限流规则
- 丰富策略：慢调用比例、异常比例、异常数

#### Q13: 如何保证微服务API的向后兼容性？

**答案**：
**版本策略**：
- URL路径版本：`/api/v1/orders`、`/api/v2/orders`
- Header版本：`Accept: application/vnd.api.v1+json`

**兼容性原则**：
- ✅ 新增字段：老客户端忽略新字段
- ❌ 删除字段：老客户端依赖该字段
- ❌ 修改字段类型：破坏序列化/反序列化

**版本迁移**：
1. 发布v2版本，v1继续运行
2. 通知客户端迁移，设置deadline
3. v1流量降到0后下线

#### Q14: 什么是契约测试？为什么微服务需要契约测试？

**答案**：
**问题场景**：
- 库存服务修改响应格式，订单服务不知道，解析失败
- 只有E2E测试才能发现，成本高、发现晚

**契约测试**：
1. 消费者定义契约：期望的请求和响应格式
2. 提供者验证契约：确保满足消费者期望
3. CI/CD中自动验证：提供者部署前验证所有契约

**优点**：
- 早期发现集成问题（不需要部署所有服务）
- 防止提供者破坏消费者

#### Q15: 什么是混沌工程？如何在微服务中实践？

**答案**：
**定义**：在生产环境主动注入故障，验证系统弹性

**常见故障注入**：
1. 网络延迟：服务调用延迟增加
2. 网络丢包：部分请求失败
3. 服务宕机：随机杀死实例
4. 资源耗尽：CPU/内存压力

**实践步骤**：
1. 定义稳态指标（QPS、错误率、P99延迟）
2. 假设系统能承受某种故障
3. 注入故障，观察指标变化
4. 分析结果，修复问题

**工具**：Chaos Monkey（Netflix）、Chaos Mesh（K8s）

#### Q16: 微服务如何实现灰度发布？

**答案**：
**方式**：
1. **网关路由**：根据Header/Cookie将部分流量路由到新版本
2. **负载均衡**：权重调整（90%旧版本，10%新版本）
3. **特征标记**：根据用户ID/地域/VIP等级分流

**流程**：
1. 部署新版本（不接收流量）
2. 灰度1%流量，观察错误率和延迟
3. 逐步扩大：5% → 10% → 25% → 50% → 100%
4. 每阶段观察监控，有问题立即回滚

**关键点**：
- 监控指标：错误率、P99延迟、业务指标
- 快速回滚能力

#### Q17: 服务间同步调用和异步调用各适用什么场景？

**答案**：
**同步调用（REST/gRPC）**：
- 需要立即获取结果
- 调用链短
- 例：查询用户信息

**异步调用（消息队列）**：
- 可以容忍延迟
- 需要解耦
- 例：下单后发送短信、异步更新统计

**选择原则**：
- 读操作：同步
- 写操作（可延迟）：异步
- 调用链长：异步（避免阻塞）
- 流量削峰：异步

#### Q18: 微服务中如何处理跨服务的数据查询？

**答案**：
**方案**：
1. **API组合**：网关/BFF调用多个服务，聚合结果
2. **数据冗余**：订单服务冗余用户信息（冗余更新要一致）
3. **CQRS**：读写分离，查询服务维护聚合视图
4. **事件驱动**：服务发布事件，其他服务同步数据

**最佳实践**：
- 简单查询：API组合
- 复杂查询：CQRS + 事件驱动
- 避免跨库JOIN

#### Q19: 微服务如何实现幂等性？

**答案**：
**幂等性定义**：同一个请求执行一次和多次效果相同

**实现方案**：
1. **唯一请求ID**：客户端生成RequestID，服务端去重
2. **数据库唯一约束**：利用唯一索引防止重复插入
3. **乐观锁**：带版本号更新，版本不匹配则拒绝
4. **状态机**：订单状态只能单向流转

**代码示例**：
```go
// 1. 检查请求是否已处理
if redis.Get(requestId) != nil {
    return cachedResult
}
// 2. 处理请求
result := process(request)
// 3. 缓存结果
redis.SetEx(requestId, result, 24*time.Hour)
return result
```

#### Q20: 微服务架构有哪些常见的反模式？

**答案**：
**常见反模式**：
1. **分布式单体**：服务拆分了但数据库共享，仍然强耦合
2. **纳米服务**：过度拆分，一个功能拆成多个服务
3. **共享库地狱**：通过共享库耦合，修改库影响所有服务
4. **同步调用链过长**：A→B→C→D→E，延迟累加，任一故障导致全链路失败
5. **缺少熔断**：下游故障导致上游雪崩
6. **忽略数据一致性**：跨服务事务未处理

**解决方案**：
- 数据库隔离
- 合理拆分粒度
- 异步解耦
- 熔断降级
- 分布式事务（Saga/TCC）

#### Q21: 如何给一条同步调用链设置“超时 + 重试 + 并发”三类预算？

**答案**：
可以按“先全局后局部”的方式设计预算：
1. 先定义端到端SLO，例如接口P99必须 < 300ms。
2. 按链路节点分配超时预算，例如网关50ms、服务A120ms、服务B100ms，并预留抖动空间。
3. 重试只给幂等且高价值调用，限制最大次数（通常1-2次）并做指数退避+抖动。
4. 给关键依赖设置并发上限，避免下游变慢时把线程池/连接池打满。

核心原则：预算是为了限制故障放大，不是为了“尽量重试到成功”。

#### Q22: Service Mesh 一定比 SDK 治理更好吗？如何做工程化取舍？

**答案**：
没有绝对优劣，关键看治理目标和组织阶段。

**Mesh更合适的情况**：
1. 多语言服务多，治理能力需要统一（mTLS、限流、熔断、追踪）。
2. 平台团队能稳定维护控制面和数据面。
3. 对策略一致性、审计和动态下发要求高。

**SDK更合适的情况**：
1. 服务规模较小，语言栈相对统一。
2. 更关注接入成本和性能开销。
3. 团队希望治理能力和业务代码一起演进。

常见落地路径是“核心能力平台化，场景能力SDK化”，避免二选一。

#### Q23: 事件驱动微服务里，如何同时处理幂等、重复消费和乱序问题？

**答案**：
三类问题通常要组合治理，而不是单点技巧：
1. 幂等：基于业务唯一键或消息ID做去重，落库加唯一约束。
2. 重复消费：消费端状态机要可重入，重复消息不改变最终状态。
3. 乱序：按业务主键分区保证局部有序，或在消费端引入版本号/事件时间校验。

工程上还要配合：
- 明确重试上限和死信队列策略
- 为补偿流程设计可回放能力
- 在监控中区分“消费失败”“消费延迟”“死信堆积”三类指标

#### Q24: 从单体迁移到微服务，怎样拆分顺序最稳妥？

**答案**：
一个可执行的迁移顺序通常是：
1. 先建立基础设施：注册发现、配置中心、日志追踪、灰度发布、回滚机制。
2. 先拆边界清晰、收益明显的模块，比如独立流量热点或高变更模块。
3. 通过反腐层（API/BFF）隔离新旧系统，减少双向耦合。
4. 优先实现数据同步和对账机制，再推进跨服务事务改造。
5. 每拆一块都做容量和故障演练，确认“可回滚、可观测、可止血”再继续。

迁移失败的常见原因不是拆得慢，而是先拆业务、后补治理，导致复杂度先爆炸。

---

#### Q25: 微服务里重试、超时、熔断、限流四者冲突时，优先级怎么定？

**答案**：
建议按“先止损、再恢复”的顺序定义优先级：
1. **超时**先定边界：不给无限等待机会，防止线程池和连接池被拖死。
2. **限流**控入口总量：把超额流量挡在系统边界外。
3. **熔断**隔离坏依赖：下游不稳定时快速失败，避免级联扩散。
4. **重试**最后启用：只对可重试错误、幂等接口，且必须有次数与预算上限。

工程上如果把重试放在最前面，通常会把局部故障放大为全链路拥塞。

#### Q26: 为什么很多团队做了“异步化”反而更难排障？你会如何建设异步链路可观测性？

**答案**：
异步化后，调用关系从“同步栈”变成“事件链”，如果没有统一追踪上下文，就会出现“每段都看起来正常，但全链路失败”的排障困境。

可观测性建设建议：
1. 为每条消息注入统一 `trace_id` / `event_id`，生产到消费全链路透传。
2. 指标分层：生产端成功率、Broker 积压、消费延迟、死信量分别监控。
3. 日志结构化：固定字段输出 topic、partition、offset、业务主键。
4. 建立可回放能力：支持按时间段和业务键重放并审计。

异步系统可观测性的目标是”可追踪、可定位、可补偿”。

### 开放式设计题

**D1：一个单体应用要拆分成微服务，你会怎么制定拆分策略和迁移路径？**

**参考思路**：
- 拆分原则：按业务域拆分（DDD限界上下文）而非按技术层拆分
- 迁移路径：Strangler Fig模式（逐步替代而非重写）→ 先拆最独立的模块 → API Gateway做路由切换
- 数据拆分：最难的部分——先拆服务后拆库、过渡期双写+数据校验、避免分布式事务（改用Saga）
- 基础设施先行：服务注册/配置中心/链路追踪/CI-CD必须先就绪再拆
- 常见陷阱：拆太细导致分布式事务满天飞、没有API版本管理、忽略数据一致性

**D2：微服务架构下一个请求链路涉及5个服务，整体延迟如何优化到100ms以内？**

**参考思路**：
- 并行化：无依赖的下游调用并行发起（Go用errgroup、Java用CompletableFuture）
- 减少跳数：合并细粒度RPC、引入BFF层聚合、考虑服务合并
- 缓存前置：每一跳都有本地缓存/Redis缓存，减少实际DB访问
- 超时治理：每一跳设合理超时（总预算100ms分配到各跳）、避免超时叠加放大
- 协议优化：gRPC替代HTTP JSON（序列化快+连接复用+Header压缩）
