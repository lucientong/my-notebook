## 1. 微服务基础

### 1.1 单体架构 vs 微服务架构

**单体架构**：

```
┌─────────────────────────────────┐
│         单体应用              │
│  ┌──────┬──────┬──────┬──────┐ │
│  │ 用户 │ 订单 │ 商品 │ 支付 │ │
│  └──────┴──────┴──────┴──────┘ │
│         ↓                     │
│      单一数据库               │
└─────────────────────────────────┘

优点：
- 开发简单（单一代码库）
- 部署简单（一个war/jar包）
- 调试简单（本地运行）

缺点：
- 难以扩展（必须整体扩展）
- 部署慢（一个模块改动，全部重新部署）
- 技术栈受限（必须用同一种语言/框架）
- 单点故障（一个模块崩溃，整个应用不可用）
```

**微服务架构**：

```
┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐
│ 用户服务│  │ 订单服务│  │ 商品服务│  │ 支付服务│
│   DB   │  │   DB   │  │   DB   │  │   DB   │
└────────┘  └────────┘  └────────┘  └────────┘
    ↓           ↓           ↓           ↓
         API网关（统一入口）
              ↓
          外部请求

优点：
- 独立部署（订单服务改动，不影响其他服务）
- 独立扩展（订单服务流量大，单独扩容）
- 技术异构（用户服务用Go，订单服务用Java）
- 故障隔离（订单服务崩溃，用户服务仍可用）

缺点：
- 运维复杂（服务数量多）
- 分布式事务（跨服务事务难处理）
- 服务间调用开销（网络延迟）
- 调试困难（请求跨多个服务）
```

---

### 1.2 微服务拆分原则

**按业务能力拆分**：

```
❌ 不推荐（按技术层拆分）：
- 用户Controller服务
- 用户Service服务
- 用户DAO服务

✅ 推荐（按业务能力拆分）：
- 用户服务（用户注册、登录、信息管理）
- 订单服务（创建订单、支付订单、订单查询）
- 商品服务（商品管理、库存管理）
```

**单一职责原则**：

```
✅ 每个服务只负责一个业务能力
❌ 避免"上帝服务"（OrderService包含支付、物流、库存...）
```

**高内聚、低耦合**：

```
✅ 服务内部高内聚（订单创建、支付、查询都在订单服务）
✅ 服务之间低耦合（订单服务调用支付服务通过API，不直接访问数据库）
```

## 2. 服务注册与发现

### 2.1 为什么需要服务注册与发现？

**传统方式（硬编码IP）**：

```go
// 订单服务调用用户服务
resp, err := http.Get("http://192.168.1.101:8080/user/123")

问题：
1. IP变化（扩容/缩容/故障）需要修改代码
2. 负载均衡需要自己实现
3. 健康检查需要自己实现
```

**服务注册与发现**：

```go
// 1. 用户服务启动时注册到注册中心
consul.Register("user-service", "192.168.1.101:8080")

// 2. 订单服务从注册中心获取用户服务地址
instances := consul.Discover("user-service")
// 返回：["192.168.1.101:8080", "192.168.1.102:8080", "192.168.1.103:8080"]

// 3. 负载均衡选择一个实例
instance := loadBalance(instances)
resp, err := http.Get(instance + "/user/123")

优势：
- 动态感知服务上下线
- 自动负载均衡
- 故障实例自动摘除
```

---

### 2.2 Consul原理⭐⭐⭐

**Consul架构**：

```
Consul集群（3-5个Server节点，Raft一致性）
    ↑
    │ 注册/发现
    ↓
Consul Agent（每个服务节点）
    ↑
    │ 健康检查
    ↓
微服务实例
```

**服务注册流程**：

```go
// 1. 服务启动时注册
registration := &consul.AgentServiceRegistration{
    ID:      "user-service-1",
    Name:    "user-service",
    Address: "192.168.1.101",
    Port:    8080,
    Check: &consul.AgentServiceCheck{
        HTTP:     "http://192.168.1.101:8080/health",
        Interval: "10s",  // 每10秒健康检查
        Timeout:  "3s",
    },
}
client.Agent().ServiceRegister(registration)

// 2. Consul Agent定期健康检查
// GET http://192.168.1.101:8080/health
// 200 OK → 健康
// 非200或超时 → 不健康（自动摘除）

// 3. 其他服务发现
services, _ := client.Health().Service("user-service", "", true, nil)
// 返回健康的实例列表
```

**为什么Consul用Raft一致性？**

```
场景：3个Consul Server节点

写入操作（服务注册）：
1. 客户端请求Leader
2. Leader写入log
3. Leader复制log到Follower（至少n/2+1节点确认）
4. Leader提交log，返回客户端
5. Leader通知Follower提交

为什么不用最终一致性（Gossip）？
- 服务注册需要强一致性（不能出现脑裂）
- Raft保证：只有一个Leader，数据不会冲突

为什么健康检查用Gossip？
- 健康状态可以容忍短暂不一致
- Gossip传播更快（O(log n)跳）
```

### 2.3 Etcd vs Consul vs Nacos

| 特性 | Etcd | Consul | Nacos |
|------|------|--------|-------|
| **一致性协议** | Raft | Raft | Raft+Distro |
| **健康检查** | 需要自己实现 | 内置（HTTP/TCP/脚本） | 内置 |
| **负载均衡** | 需要自己实现 | 内置 | 内置 |
| **多数据中心** | 不支持 | 支持 | 支持 |
| **UI界面** | 无 | 有 | 有 |
| **适用场景** | K8s元数据存储 | 微服务注册发现 | 微服务注册发现+配置中心 |

## 3. API网关

### 3.1 为什么需要API网关？

**无网关架构问题**：

```
客户端直接调用微服务：
移动端 → 用户服务
      → 订单服务
      → 商品服务

问题：
1. 客户端需要知道所有服务地址（耦合）
2. 每个服务都要实现认证、限流、日志（重复）
3. 跨域问题（CORS）
4. 协议转换（HTTP → gRPC）
```

**网关架构**：

```
客户端 → API网关 → 用户服务
                → 订单服务
                → 商品服务

网关统一处理：
- 认证鉴权
- 限流熔断
- 日志监控
- 协议转换
- 路由转发
```

---

### 3.2 Kong网关架构⭐⭐⭐

**Kong核心组件**：

```
客户端请求
    ↓
Kong Server（OpenResty + Lua）
    ↓
插件链（认证 → 限流 → 日志 → ...）
    ↓
Upstream（后端服务）
```

**Kong插件机制**：

```lua
-- 限流插件示例
function RateLimitPlugin:access(config)
  local current = redis:incr(key)
  
  if current > config.limit then
    return kong.response.exit(429, "Too Many Requests")
  end
  
  if current == 1 then
    redis:expire(key, config.period)
  end
end

-- 请求流程
请求 → 认证插件（JWT验证）
    → 限流插件（检查速率）
    → 日志插件（记录日志）
    → 转发到后端服务
```

**为什么Kong用OpenResty？**

```
OpenResty = Nginx + LuaJIT

优势：
1. 高性能（Nginx的事件驱动 + LuaJIT的JIT编译）
2. 灵活（Lua脚本动态扩展）
3. 生态丰富（大量Nginx插件）

性能：
- Kong：10万QPS（单机）
- Spring Cloud Gateway：5万QPS（单机）
```

### 3.3 网关路由策略

**基于路径路由**：

```yaml
# /api/user/* → 用户服务
# /api/order/* → 订单服务

routes:
  - path: /api/user/*
    service: user-service
  - path: /api/order/*

```

```yaml

    match:
      headers:
        X-Version: v2
    service: order-service-v2
  - path: /api/order/*
    service: order-service-v1
```

## 4. 服务熔断与降级

### 4.1 为什么需要熔断？

**雪崩效应**：

```
用户服务 → 订单服务 → 商品服务（故障，响应慢5秒）
                    ↓
            订单服务线程池耗尽
                    ↓
            用户服务线程池耗尽
                    ↓
            整个系统不可用

熔断的作用：
- 快速失败（不等待5秒）
- 保护上游服务（防止雪崩）
- 给下游服务恢复时间
```

---

### 4.2 Hystrix熔断原理⭐⭐⭐

**熔断器三种状态**：

```
1. Closed（关闭）：
   - 正常请求通过
   - 记录失败率

2. Open（打开）：
   - 失败率 > 阈值（如50%）
   - 所有请求直接失败（快速失败）
   - 进入休眠期（如5秒）

3. Half-Open（半开）：
   - 休眠期后，允许少量请求通过（探测）
   - 成功 → Closed
   - 失败 → Open

状态转换：
Closed → Open（失败率>阈值）
Open → Half-Open（休眠期结束）
Half-Open → Closed（探测成功）
Half-Open → Open（探测失败）
```

**熔断器实现**：

```go
type CircuitBreaker struct {
    state         State  // Closed/Open/Half-Open
    failureCount  int
    successCount  int
    lastFailTime  time.Time
}

func (cb *CircuitBreaker) Call(fn func() error) error {
    if cb.state == Open {
        // 检查是否可以进入Half-Open
        if time.Since(cb.lastFailTime) > sleepWindow {
            cb.state = HalfOpen
        } else {
            return ErrCircuitOpen  // 快速失败
        }
    }
    
    err := fn()
    
    if err != nil {
        cb.onFailure()
    } else {
        cb.onSuccess()
    }
    
    return err
}

func (cb *CircuitBreaker) onFailure() {
    cb.failureCount++
    cb.lastFailTime = time.Now()
    
    if cb.state == HalfOpen {
        cb.state = Open  // 探测失败，重新打开
    } else if float64(cb.failureCount)/float64(cb.totalCount) > threshold {
        cb.state = Open  // 失败率超过阈值，打开
    }
}

func (cb *CircuitBreaker) onSuccess() {
    if cb.state == HalfOpen {
        cb.successCount++
        if cb.successCount >= successThreshold {
            cb.state = Closed  // 探测成功，关闭
            cb.failureCount = 0
        }
    }
}
```

**为什么熔断器能防止雪崩？**

```
无熔断器：
订单服务调用商品服务（5秒超时）
100个请求同时调用 → 100个线程阻塞5秒 → 线程池耗尽

有熔断器：
第1个请求：5秒超时（失败）
第2个请求：5秒超时（失败）
...
第10个请求：失败率>50% → 熔断器打开
第11-100个请求：立即失败（<1ms）→ 线程池未耗尽

节省时间：100 × 5秒 = 500秒 → 10 × 5秒 + 90 × 0.001秒 = 50秒
```

---

### 4.3 降级策略

**降级 vs 熔断**：

```
熔断：服务故障时，停止调用（快速失败）
降级：服务故障时，返回兜底数据（保证可用性）

示例：
// 熔断
err := orderService.GetOrder(id)
if err != nil {
    return err  // 直接返回错误
}

// 降级
err := orderService.GetOrder(id)
if err != nil {
    return defaultOrder  // 返回默认订单（兜底）
}
```

**降级策略**：

```
1. 返回默认值
   - 商品服务故障 → 返回空商品列表

2. 返回缓存数据
   - 订单详情服务故障 → 返回Redis缓存的订单

3. 降级到备用服务
   - 主支付服务故障 → 调用备用支付服务

4. 降级功能
   - 推荐服务故障 → 不展示推荐商品（不影响核心功能）
```

---

### 4.4 Sentinel vs Hystrix

| 特性 | Hystrix | Sentinel |
|------|---------|----------|
| **熔断粒度** | 线程池隔离 | 信号量隔离 |
| **实时监控** | 需要Dashboard | 内置实时监控 |
| **规则配置** | 代码配置 | 动态配置（控制台） |
| **性能** | 较低（线程切换） | 高（无线程切换） |
| **状态** | Netflix已停止维护 | 阿里维护中 |

**Sentinel优势**：

```
1. 实时监控：
   - 秒级监控（QPS、RT、成功率）
   - 可视化Dashboard

2. 动态规则：
   - 无需重启，动态修改限流规则
   - 规则持久化（Nacos/Apollo）

3. 丰富的降级策略：
   - 慢调用比例
   - 异常比例
   - 异常数
```

## 5. 分布式追踪

### 5.1 为什么需要分布式追踪？

**问题场景**：

```
用户请求 → 网关 → 用户服务 → 订单服务 → 商品服务
                              → 库存服务
                              → 支付服务

问题：
1. 请求慢，不知道哪个服务慢
2. 请求失败，不知道哪个服务失败
3. 调用链路复杂，难以调试
```

**分布式追踪**：

```
TraceID: abc123（唯一标识一个请求）
  ├─ SpanID: span1（网关，耗时10ms）
  ├─ SpanID: span2（用户服务，耗时20ms）
  └─ SpanID: span3（订单服务，耗时100ms）
      ├─ SpanID: span4（商品服务，耗时50ms）
      ├─ SpanID: span5（库存服务，耗时30ms）
      └─ SpanID: span6（支付服务，耗时20ms）

分析：
- 总耗时130ms（10+20+100）
- 瓶颈：订单服务（100ms）
  - 其中商品服务最慢（50ms）
```

---

### 5.2 OpenTelemetry原理⭐⭐⭐

**核心概念**：

```
1. Trace：一次完整的请求链路（TraceID唯一）
2. Span：一个操作（SpanID唯一）
3. Context：上下文传播（TraceID + SpanID）

数据结构：
Trace {
    TraceID: "abc123"
    Spans: [
        Span {
            SpanID: "span1"
            ParentSpanID: ""  // 根Span
            Name: "GET /api/order"
            StartTime: 1234567890
            EndTime:   1234567900
            Attributes: {
                http.method: "GET"
                http.status_code: 200
            }
        },
        Span {
            SpanID: "span2"
            ParentSpanID: "span1"  // 父Span
            Name: "OrderService.GetOrder"
            ...
        }
    ]
}
```

**上下文传播（Context Propagation）**：

```
请求流：
客户端 → 网关 → 订单服务 → 商品服务

步骤1：客户端生成TraceID
TraceID = "abc123"
SpanID = "span1"

步骤2：网关接收请求，提取TraceID
从Header提取：Traceparent: 00-abc123-span1-01
创建新Span：SpanID = "span2"，ParentSpanID = "span1"

步骤3：网关调用订单服务，传递TraceID
Header: Traceparent: 00-abc123-span2-01

步骤4：订单服务接收请求，提取TraceID
创建新Span：SpanID = "span3"，ParentSpanID = "span2"

步骤5：订单服务调用商品服务，传递TraceID
Header: Traceparent: 00-abc123-span3-01

...

最终：所有Span都有相同的TraceID，可以串联起来
```

**Traceparent格式（W3C标准）**：

```
Traceparent: 00-abc123-span1-01
             ││  │      │     │
             ││  │      │     └─ flags（是否采样）
             ││  │      └─ parent-id（父SpanID）
             ││  └─ trace-id（TraceID）
             │└─ version（版本）
             └─ 00（固定）
```

### 5.3 Jaeger架构

**组件**：

```
应用 → Jaeger Agent（本地） → Jaeger Collector → 存储（Elasticsearch）
                                                 ↓
                                            Jaeger UI
```

**采样策略**：

```
1. 全量采样（const sampler）：
   - 采样率100%
   - 适合开发环境

2. 概率采样（probabilistic sampler）：
   - 采样率1%（100个请求采样1个）
   - 适合高流量生产环境

3. 限速采样（rate limiting sampler）：
   - 每秒最多采样10个
   - 避免采样数据过多

4. 自适应采样（adaptive sampler）：
   - 根据流量动态调整采样率
   - 保证采样数据量稳定
```

## 6. 配置中心

### 6.1 为什么需要配置中心？

**传统配置方式问题**：

```
配置写在application.yml：
- 修改配置需要重启服务
- 多个服务配置重复（数据库地址）
- 配置散落各处，难以管理
```

**配置中心**：

```
Apollo/Nacos：
- 集中管理配置
- 动态更新配置（无需重启）
- 配置版本管理（回滚）
- 配置权限管理
```

---

### 6.2 Apollo架构

**组件**：

```
1. Config Service：
   - 提供配置查询接口
   - 长轮询推送配置变更

2. Admin Service：
   - 配置管理接口
   - Portal调用

3. Portal：
   - 配置管理界面

4. Client：
   - 应用内的SDK
   - 长轮询获取配置
```

**配置热更新原理**：

```
步骤1：应用启动，Client拉取配置
GET /configs?appId=order-service&namespace=application

步骤2：Client启动长轮询
GET /notifications/v2?appId=order-service&notifications=[...]
（Config Service hold住请求，60秒超时）

步骤3：运维在Portal修改配置
Portal → Admin Service → Config Service

步骤4：Config Service立即返回长轮询请求
返回：{namespace: "application", notificationId: 2}

步骤5：Client收到通知，拉取新配置
GET /configs?appId=order-service&namespace=application

步骤6：Client触发回调
@ApolloConfigChangeListener
public void onChange(ConfigChangeEvent event) {
    String newValue = event.getChange("timeout").getNewValue();
    // 更新本地配置（无需重启）
}
```

## 7. 常见面试题

### Q1: 微服务和单体架构的区别？

**答案**：
- **部署**：微服务独立部署，单体整体部署
- **扩展**：微服务独立扩展，单体整体扩展
- **技术栈**：微服务异构，单体统一
- **故障隔离**：微服务隔离，单体全局影响

### Q2: 如何保证微服务的数据一致性？

**答案**：
1. **Saga模式**：本地事务 + 补偿事务
2. **TCC模式**：Try-Confirm-Cancel
3. **事件驱动**：事件溯源 + 最终一致性
4. **分布式事务**：2PC/3PC（性能差，不推荐）

### Q3: 服务注册与发现的实现原理？

**答案**：
- **注册**：服务启动时向注册中心注册（IP、端口、元数据）
- **发现**：其他服务从注册中心查询可用实例列表
- **健康检查**：注册中心定期检查服务健康状态
- **负载均衡**：客户端根据策略（轮询、随机）选择实例

### Q4: 熔断器如何防止雪崩？

**答案**：
- **快速失败**：服务故障时立即返回错误（不等待超时）
- **线程池保护**：避免线程池耗尽
- **自动恢复**：Half-Open状态探测服务是否恢复

### Q5: 分布式追踪如何实现？

**答案**：
- **TraceID生成**：请求入口生成全局唯一TraceID
- **上下文传播**：通过HTTP Header传递TraceID
- **Span记录**：每个服务记录自己的Span（开始时间、结束时间）
- **数据收集**：所有Span上报到Collector
- **链路重建**：根据TraceID和ParentSpanID重建调用链
