## 1. 限流策略

### 1.1 为什么需要限流？

**场景**：
- 突发流量（秒杀、热点事件）
- 恶意攻击（DDoS、爬虫）
- 下游服务保护（防止雪崩）

**不限流的后果**：
- 服务器CPU/内存耗尽
- 数据库连接池耗尽
- 整个系统崩溃

### 1.2 常见限流算法

#### 1.2.1 固定窗口计数器

**原理**：统计固定时间窗口内的请求数

```go
type FixedWindowLimiter struct {
    limit    int64         // 限制数量
    window   time.Duration // 窗口大小
    counter  int64         // 当前计数
    lastTime time.Time     // 窗口开始时间
    mu       sync.Mutex
}

func (l *FixedWindowLimiter) Allow() bool {
    l.mu.Lock()
    defer l.mu.Unlock()
    
    now := time.Now()
    
    // 新窗口，重置计数
    if now.Sub(l.lastTime) >= l.window {
        l.counter = 0
        l.lastTime = now
    }
    
    // 检查是否超限
    if l.counter >= l.limit {
        return false
    }
    
    l.counter++
    return true
}

// 使用示例
limiter := &FixedWindowLimiter{
    limit:  100,              // 100次
    window: 1 * time.Second,  // 1秒
}

if limiter.Allow() {
    // 处理请求
} else {
    // 返回429 Too Many Requests
}
```

**优点**：实现简单  
**缺点**：临界问题（窗口边界流量突刺）

**临界问题示例**：
```
窗口1: 09:59:00 - 09:59:59  →  99次请求（在59秒时）
窗口2: 10:00:00 - 10:00:59  →  99次请求（在00秒时）
实际：1秒内处理了198次请求（突破限制）
```

#### 1.2.2 滑动窗口

**原理**：将固定窗口划分为多个小窗口，统计滑动窗口内的请求数

```go
type SlidingWindowLimiter struct {
    limit      int64
    window     time.Duration
    smallWindow time.Duration
    counters   map[int64]int64  // 时间戳 -> 计数
    mu         sync.Mutex
}

func (l *SlidingWindowLimiter) Allow() bool {
    l.mu.Lock()
    defer l.mu.Unlock()
    
    now := time.Now()
    windowStart := now.Add(-l.window)
    
    // 清理过期窗口
    for ts := range l.counters {
        if time.Unix(0, ts).Before(windowStart) {
            delete(l.counters, ts)
        }
    }
    
    // 统计当前窗口内的请求数
    var total int64
    for _, count := range l.counters {
        total += count
    }
    
    if total >= l.limit {
        return false
    }
    
    // 当前小窗口计数+1
    currentWindow := now.Truncate(l.smallWindow).UnixNano()
    l.counters[currentWindow]++
    
    return true
}
```

**优点**：解决临界问题  
**缺点**：内存占用较大

#### 1.2.3 令牌桶（推荐）

**原理**：按固定速率生成令牌，请求消耗令牌，桶满则丢弃令牌

```go
import "golang.org/x/time/rate"

// 创建限流器
limiter := rate.NewLimiter(100, 200)  // 100 token/s, burst 200

// 方式1：阻塞等待
err := limiter.Wait(context.Background())

// 方式2：非阻塞
if limiter.Allow() {
    // 处理请求
} else {
    // 返回429
}

// 方式3：等待N个令牌
reservation := limiter.Reserve()
if !reservation.OK() {
    // 无法获取令牌
    return
}
time.Sleep(reservation.Delay())  // 等待
```

**优点**：
- 平滑流量，允许突发（burst）
- Go官方库支持

**参数说明**：
- `rate`：令牌生成速率（QPS）
- `burst`：桶容量（允许的突发请求数）

#### 令牌桶手写实现（完整版）⭐⭐⭐

**为什么要手写？**
- 面试高频：手写限流器
- 理解原理：令牌如何生成、如何消费
- 自定义需求：官方库无法满足的场景

**核心思想**：
1. 不真正"生成"令牌（太耗费CPU）
2. 记录上次取令牌的时间，计算当前应该有多少令牌
3. 惰性生成：取令牌时才计算

```go
// 令牌桶限流器（手写实现）
type TokenBucket struct {
    capacity   int64         // 桶容量（最多存多少令牌）
    tokens     int64         // 当前令牌数
    rate       int64         // 生成速率（token/s）
    lastTime   time.Time     // 上次取令牌时间
    mu         sync.Mutex
}

func NewTokenBucket(rate int64, capacity int64) *TokenBucket {
    return &TokenBucket{
        capacity: capacity,
        tokens:   capacity,  // 初始满桶
        rate:     rate,
        lastTime: time.Now(),
    }
}

func (tb *TokenBucket) Allow() bool {
    return tb.AllowN(1)
}

func (tb *TokenBucket) AllowN(n int64) bool {
    tb.mu.Lock()
    defer tb.mu.Unlock()
    
    now := time.Now()
    
    // 计算从上次到现在应该生成多少令牌
    elapsed := now.Sub(tb.lastTime)
    newTokens := int64(elapsed.Seconds() * float64(tb.rate))
    
    // 更新令牌数（不超过容量）
    tb.tokens += newTokens
    if tb.tokens > tb.capacity {
        tb.tokens = tb.capacity
    }
    
    tb.lastTime = now
    
    // 检查是否有足够的令牌
    if tb.tokens >= n {
        tb.tokens -= n
        return true
    }
    
    return false
}

// 等待获取N个令牌（阻塞）
func (tb *TokenBucket) WaitN(ctx context.Context, n int64) error {
    for {
        if tb.AllowN(n) {
            return nil
        }
        
        // 计算需要等待的时间
        tb.mu.Lock()
        needed := n - tb.tokens
        waitTime := time.Duration(float64(needed) / float64(tb.rate) * float64(time.Second))
        tb.mu.Unlock()
        
        select {
        case <-time.After(waitTime):
            // 继续尝试
        case <-ctx.Done():
            return ctx.Err()
        }
    }
}

// 使用示例
func main() {
    // 100 QPS，允许突发200个请求
    limiter := NewTokenBucket(100, 200)
    
    // 场景1：瞬时200个请求
    for i := 0; i < 200; i++ {
        if limiter.Allow() {
            fmt.Println("请求", i, "通过")
        }
    }
    // 前200个通过，后面的被拒绝（桶空了）
    
    // 场景2：等待1秒后，又有100个令牌
    time.Sleep(1 * time.Second)
    if limiter.Allow() {
        fmt.Println("1秒后请求通过")  // 通过！
    }
}
```

**关键点**：

1. **惰性计算**：
   ```
   不实时生成令牌（每秒100次定时器太耗费）
   取令牌时才计算：elapsed * rate = 新令牌数
   ```

2. **时间精度**：
   ```go
   elapsed := now.Sub(tb.lastTime)
   newTokens := int64(elapsed.Seconds() * float64(tb.rate))
   
   // 例如：rate=100, elapsed=0.5秒
   // newTokens = 0.5 * 100 = 50个令牌
   ```

3. **突发流量**：
   ```
   桶容量=200，当前tokens=200
   瞬间200个请求 → 全部通过
   桶变空（tokens=0）
   后续请求需要等待令牌生成
   ```

#### 令牌桶 vs 漏桶（底层原理对比）⭐⭐⭐

**为什么令牌桶能允许突发，漏桶不行？**

| 维度 | 令牌桶 | 漏桶 |
|------|--------|------|
| **核心思想** | 令牌定速生成，请求变速消费 | 请求变速进入，定速流出 |
| **突发流量** | ✅ 允许（消费已存的令牌） | ❌ 不允许（队列+定速处理） |
| **流量曲线** | 锯齿状（允许瞬时峰值） | 平滑（恒定速率） |
| **实现复杂度** | 简单（计数器） | 中等（需要队列） |

**图解**：

```
【令牌桶】
令牌生成：    ─┬─┬─┬─┬─┬─┬─┬─> (恒定速率)
            令桶桶桶桶桶桶桶
请求消费：    ████─█─█─> (变速，可突发)

时刻T1: 桶满（100个令牌）
时刻T2: 瞬间100个请求 → 全部通过（消费100个令牌）
时刻T3: 桶空，新请求需要等待

【漏桶】
请求进入：    ████─█─█─> (变速)
            ┌──────┐
            │ 队列 │ (缓冲区)
            └──────┘
请求流出：    ─┬─┬─┬─┬─┬─┬─┬─> (恒定速率)

时刻T1: 100个请求瞬间到达 → 进入队列
时刻T2: 按固定速率处理（如每秒10个）
时刻T3: 需要10秒才能处理完100个请求
```

**为什么漏桶不能突发？**

```go
// 漏桶的leak()协程
func (lb *LeakyBucket) leak() {
    ticker := time.NewTicker(lb.rate)  // 固定速率
    defer ticker.Stop()
    
    for range ticker.C {
        select {
        case <-lb.queue:
            processRequest()  // 每个tick只处理1个请求
        default:
        }
    }
}

// 即使队列有1000个请求，也只能按固定速率处理
// 无法利用"空闲时段"加速处理
```

**应用场景对比**：

| 场景 | 推荐算法 | 原因 |
|------|----------|------|
| **API限流** | 令牌桶 | 用户体验好（允许短时突发） |
| **写数据库** | 漏桶 | 保护数据库（平滑QPS） |
| **消息队列消费** | 漏桶 | 避免下游被打垮（恒速） |
| **CDN流量** | 令牌桶 | 允许大文件快速下载（突发） |

**应用场景**：
```go
// API网关限流

        if !limiter.Allow() {
```

#### 1.2.4 漏桶

**原理**：请求进入队列，按固定速率处理

```go
type LeakyBucket struct {
    capacity int           // 桶容量
    rate     time.Duration // 处理速率
    queue    chan struct{} // 请求队列
}

func NewLeakyBucket(capacity int, rate time.Duration) *LeakyBucket {
    lb := &LeakyBucket{
        capacity: capacity,
        rate:     rate,
        queue:    make(chan struct{}, capacity),
    }
    go lb.leak()
    return lb
}

func (lb *LeakyBucket) leak() {
    ticker := time.NewTicker(lb.rate)
    defer ticker.Stop()
    
    for range ticker.C {
        select {
        case <-lb.queue:
            // 处理一个请求
        default:
        }
    }
}

func (lb *LeakyBucket) Allow() bool {
    select {
    case lb.queue <- struct{}{}:
        return true
    default:
        return false  // 队列满
    }
}
```

**对比令牌桶**：
| 算法 | 流量特点 | 适用场景 |
|------|----------|----------|
| 令牌桶 | 允许突发 | API限流 |
| 漏桶 | 平滑流量 | 消息队列 |

## 2. 熔断与降级

### 2.1 熔断器（Circuit Breaker）

**原理**：当下游服务故障率超过阈值时，自动切断请求，快速失败

**状态机**：
```
关闭（Closed） → 打开（Open） → 半开（Half-Open） → 关闭
```

**状态转换**：
1. **Closed**：正常请求，统计失败率
2. **Open**：失败率超阈值，拒绝所有请求，快速返回错误
3. **Half-Open**：一段时间后，允许少量请求尝试
4. **Closed**：尝试成功，恢复正常

**Go实现（使用 gobreaker 库）**：

```go
import "github.com/sony/gobreaker"

// 创建熔断器
breaker := gobreaker.NewCircuitBreaker(gobreaker.Settings{
    Name:        "my-service",
    MaxRequests: 3,      // 半开状态允许3个请求
    Interval:    10 * time.Second,  // 统计周期
    Timeout:     60 * time.Second,  // 熔断持续时间
    ReadyToTrip: func(counts gobreaker.Counts) bool {
        // 失败率 > 50% 触发熔断
        failureRatio := float64(counts.TotalFailures) / float64(counts.Requests)
        return counts.Requests >= 10 && failureRatio >= 0.5
    },
})

// 使用熔断器
result, err := breaker.Execute(func() (interface{}, error) {
    return callDownstreamService()
})

if err != nil {
    if err == gobreaker.ErrOpenState {
        // 熔断器打开，快速失败
        return nil, errors.New("服务熔断")
    }
    return nil, err
}
```

**监控指标**：
```go
state := breaker.State()  // Closed/Open/Half-Open
counts := breaker.Counts()
log.Printf("State: %v, Requests: %d, Failures: %d", 
    state, counts.Requests, counts.TotalFailures)
```

### 2.2 降级

**定义**：系统负载过高时，关闭非核心功能，保障核心功能

**降级策略**：
1. **功能降级**：关闭推荐、评论等非核心功能
2. **读降级**：返回缓存数据
3. **写降级**：限制写操作，或异步写入
4. **返回默认值**：返回兜底数据

**实现方式**：

**方式1：配置中心（推荐）**
```go
type FeatureSwitch struct {
    EnableRecommendation bool `json:"enable_recommendation"`
    EnableComment        bool `json:"enable_comment"`
    EnableSearch         bool `json:"enable_search"`
}

var featureSwitch FeatureSwitch

// 定时从配置中心拉取
func LoadFeatureSwitch() {
    config := fetchFromConfigCenter()
    featureSwitch = config
}

// 使用
func GetRecommendation() ([]Item, error) {
    if !featureSwitch.EnableRecommendation {
        return []Item{}, nil  // 降级：返回空列表
    }
    return queryRecommendation()
}
```

**方式2：Sentinel（阿里开源）**
```go
import sentinel "github.com/alibaba/sentinel-golang/api"

// 定义资源
entry, err := sentinel.Entry("recommendation", sentinel.WithTrafficType(base.Inbound))
if err != nil {
    // 触发降级规则
    return getDefaultRecommendation()
}
defer entry.Exit()

// 正常业务逻辑
return queryRecommendation()
```

## 3. 分布式事务

### 3.1 2PC（两阶段提交）

**流程**：
1. **准备阶段**：协调者询问所有参与者是否可以提交
2. **提交阶段**：所有参与者都同意，则提交；否则回滚

**缺点**：
- 同步阻塞（性能差）
- 单点故障（协调者挂了）
- 数据不一致（网络分区）

**结论**：❌ 不推荐使用

---

### 3.2 方案对比

| 方案 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **2PC** | 协调者两阶段提交 | 强一致性 | 同步阻塞、单点故障 | ❌ 不推荐 |
| **TCC** | Try预留→Confirm提交→Cancel回滚 | 性能好、不阻塞 | 业务侵入强，需实现3接口 | 金融转账等强一致场景 |
| **SAGA** | 多个本地事务+补偿事务 | 长事务友好 | 最终一致、补偿逻辑复杂 | 订单流程等长链路 |
| **本地消息表** | 本地事务写业务+消息，定时发MQ | 可靠性高、实现简单 | 需定时扫描、有延迟 | ✅ 推荐默认方案 |

### 3.3 TCC核心流程

```
Try阶段（预留资源）→ Confirm阶段（确认提交）→ Cancel阶段（取消回滚）

示例-转账：
Try:     冻结转出方资金
Confirm: 扣减转出方余额+冻结，增加收款方余额
Cancel:  解冻转出方资金
```

> 延伸阅读：TCC/SAGA 在金融支付领域的实践，详见《专项知识库/02-金融支付专项》。

### 3.4 SAGA核心流程

```
编排式(Orchestration)：中央协调者编排执行步骤和补偿
    创建订单 → 扣减库存 → 扣减余额 → 发送通知
    (补偿)取消订单 ← 恢复库存 ← 退款

协同式(Choreography)：各服务通过MQ事件自行响应
    ✅ 更解耦    ❌ 调试困难
```

### 3.5 本地消息表（推荐）

**原理**：在本地事务中同时写业务数据和消息，定时扫描消息表发送MQ

## 4. 分布式一致性

### 4.1 CAP理论

| 特性 | 说明 |
|------|------|
| **C**onsistency | 一致性：所有节点看到相同数据 |
| **A**vailability | 可用性：任何请求都能得到响应 |
| **P**artition Tolerance | 分区容错：网络分区时系统仍可用 |

**定理**：三者只能同时满足两个

**选择**：
- **CP**：保证一致性，牺牲可用性（如：ZooKeeper、HBase）
- **AP**：保证可用性，牺牲强一致性（如：Cassandra、DynamoDB）
- **CA**：实际不存在（网络分区不可避免）

---

#### 4.1.1 CAP为什么只能满足两个？（推导证明）⭐⭐⭐

**为什么需要理解证明？**
- 面试高频：为什么CAP不能三者兼得？
- 架构选型：根据业务选择CP或AP
- 避免误区：CA在分布式系统中不存在

---

##### 证明：C + A + P 不可能同时满足

**场景设定**：
```
分布式系统：
- 节点N1：master（可读可写）
- 节点N2：slave（可读可写）
- 客户端C1、C2
```

**步骤1：网络分区发生**
```
初始状态：N1和N2数据一致，都是v0

时刻T1：网络分区
    ┌────┐          ┌────┐
    │ N1 │ ╳╳╳╳╳╳╳╳ │ N2 │
    └────┘ 网络断开  └────┘
     v0              v0
```

**步骤2：客户端写入N1**
```
时刻T2：C1向N1写入v1
    ┌────┐          ┌────┐
    │ N1 │ ╳╳╳╳╳╳╳╳ │ N2 │
    └────┘          └────┘
     v1              v0
    
问题：N1无法将v1同步到N2（网络断开）
```

**步骤3：客户端读取N2**
```
时刻T3：C2从N2读取数据

选择A：保证C + A（牺牲P）
  - N2返回v0（旧数据）
  - 违反一致性（C1看到v1，C2看到v0）
  
选择B：保证C + P（牺牲A）
  - N2拒绝服务："网络分区中，无法保证一致性"
  - 违反可用性
  
选择C：保证A + P（牺牲C）
  - N2返回v0（允许不一致）
  - 违反一致性（最终一致性）
```

**结论**：
```
在网络分区（P）存在的前提下：
- 要么牺牲一致性（C）→ AP
- 要么牺牲可用性（A）→ CP

无法同时满足C和A！
```

##### 为什么CA不存在？

**误区**：单机系统是CA（无网络分区）

**反驳**：
```
1. 单机系统不是分布式系统
   - CAP是针对分布式系统的定理
   - 单机没有"分布式"的概念

2. 网络分区不可避免
   - 即使在同一数据中心，也可能网络故障
   - 交换机故障、光缆断裂、路由器宕机
   - P是客观存在的，不是可选项

3. CA的真正含义
   - 在P不存在的假设下，可以同时满足C和A
   - 但P是客观存在的 → CA在实际中不存在
```

##### CP系统示例：ZooKeeper

**ZooKeeper的选择**：牺牲A，保证C和P

```
场景：ZooKeeper有5个节点（N1-N5），N1是Leader

时刻T1：网络分区
分区1：N1、N2（2个节点，少数派）
分区2：N3、N4、N5（3个节点，多数派）

时刻T2：客户端写入
- 向N1写入 → 失败（无法同步到多数派）
- 向N3写入 → 成功（N3、N4、N5达成共识）

时刻T3：客户端读取
- 从N1读取 → 失败（N1发现自己不是Leader）
- 从N3读取 → 成功（返回最新数据）

牺牲可用性（A）：
- 分区1中的节点拒绝服务（保证一致性）
- 只有多数派可用
```

**ZooKeeper的过半机制**：
```
写操作需要过半节点确认：
- 5个节点 → 至少3个确认
- 3个节点 → 至少2个确认

为什么？
- 保证任意两个多数派必有交集
- 5个节点：多数派=3
  - {N1, N2, N3} ∩ {N3, N4, N5} = {N3} ≠ ∅
  - 保证至少有1个节点知道最新数据

网络分区时：
- 只有多数派能提供服务（保证一致性）
- 少数派拒绝服务（牺牲可用性）
```

##### AP系统示例：Cassandra

**Cassandra的选择**：牺牲C，保证A和P

```
场景：Cassandra有5个节点（N1-N5）

时刻T1：网络分区
分区1：N1、N2
分区2：N3、N4、N5

时刻T2：客户端写入
- 向N1写入v1 → 成功（写入N1、N2，W=2）
- 向N3写入v2 → 成功（写入N3、N4、N5，W=3）

时刻T3：客户端读取
- 从N1读取 → 返回v1
- 从N3读取 → 返回v2
- 不一致！

牺牲一致性（C）：
- 允许分区内独立写入（可用性）
- 分区恢复后，通过Last-Write-Wins解决冲突
```

**Cassandra的可调一致性**：
```
W + R > N  →  强一致性
W + R ≤ N  →  最终一致性

W: 写操作需要确认的节点数
R: 读操作需要确认的节点数
N: 副本总数

示例（N=3）：
- W=1, R=1 → AP（可能读到旧数据）
- W=2, R=2 → 强一致性（W+R=4>3）
- W=3, R=1 → CP（写入全部节点才返回）
```

| **DNS** | AP | 可用性优先（允许缓存过期） |

##### CAP定理的实际应用（工程中的权衡）

**1. 牺牲一致性（AP）的常见做法**：

```go
// 最终一致性：异步同步
func WriteData(key, value string) error {
    // 1. 立即写入主节点
    if err := master.Write(key, value); err != nil {
        return err
    }
    
    // 2. 异步同步到从节点（允许短暂不一致）
    go func() {
        for _, slave := range slaves {
            slave.Write(key, value)
        }
    }()
    
    return nil  // 立即返回（可用性）
}

// 读取时可能读到旧数据（不一致）
func ReadData(key string) (string, error) {
    // 从任意节点读取
    return randomSlave().Read(key)
}
```

**2. 牺牲可用性（CP）的常见做法**：

```go
// 强一致性：同步等待多数派确认
func WriteData(key, value string) error {
    // 需要多数派（3/5）确认才返回
    successCount := 0
    errChan := make(chan error, len(nodes))
    
    for _, node := range nodes {
        go func(n *Node) {
            errChan <- n.Write(key, value)
        }(node)
    }
    
    // 等待多数派确认
    for i := 0; i < len(nodes); i++ {
        if <-errChan == nil {
            successCount++
            if successCount >= len(nodes)/2 + 1 {
                return nil  // 多数派成功
            }
        }
    }
    
    return errors.New("无法达成多数派共识")  // 牺牲可用性
}
```

### 4.3 一致性Hash

**问题**：分布式缓存中，节点增减导致大量缓存失效

**方案**：一致性Hash

**原理**：
1. 将Hash空间组织成环（0 ~ 2^32-1）
2. 节点映射到环上（Hash(IP)）
3. 数据顺时针找到第一个节点

**优点**：节点增减只影响相邻节点

**实现**：
```go
type ConsistentHash struct {
    circle       map[uint32]string  // Hash值 -> 节点
    sortedHashes []uint32            // 排序的Hash值
    replicas     int                 // 虚拟节点数
}

func (ch *ConsistentHash) Add(node string) {
    for i := 0; i < ch.replicas; i++ {
        hash := crc32.ChecksumIEEE([]byte(fmt.Sprintf("%s:%d", node, i)))
        ch.circle[hash] = node
        ch.sortedHashes = append(ch.sortedHashes, hash)
    }
    sort.Slice(ch.sortedHashes, func(i, j int) bool {
        return ch.sortedHashes[i] < ch.sortedHashes[j]
    })
}

func (ch *ConsistentHash) Get(key string) string {
    hash := crc32.ChecksumIEEE([]byte(key))
    
    // 二分查找第一个 >= hash 的节点
    idx := sort.Search(len(ch.sortedHashes), func(i int) bool {
        return ch.sortedHashes[i] >= hash
    })
    
    if idx == len(ch.sortedHashes) {
        idx = 0  // 环形，回到第一个节点
    }
    
    return ch.circle[ch.sortedHashes[idx]]
}
```

### 4.4 Raft一致性协议⭐⭐⭐

> Raft是目前最主流的分布式一致性协议，etcd、Consul、TiKV等均基于Raft。比Paxos更易理解和实现。

**三种角色**：
```
Leader：   唯一的写入入口，负责日志复制和心跳
Follower： 被动接收Leader的日志和心跳
Candidate：选举期间的临时角色（Follower超时后变为Candidate）
```

**Leader选举流程**：
```
1. Follower在election timeout内未收到Leader心跳
2. Follower → Candidate，term+1，投票给自己，向其他节点RequestVote
3. 收到多数派投票 → 成为Leader
4. 新Leader立即发送心跳抑制其他选举

关键规则：
- 每个term每个节点只投一票（先到先得）
- Candidate日志不能比自己旧的节点不投票
- election timeout随机化（150~300ms）避免同时选举（split vote）
```

**日志复制流程**：
```
1. 客户端写请求 → Leader
2. Leader将日志追加到本地Log
3. Leader并行发送AppendEntries给所有Follower
4. 多数派确认 → Leader提交日志（commit）
5. Leader返回成功给客户端
6. 后续心跳通知Follower也提交

日志匹配原则：
- 相同index和term的日志内容一定相同
- Leader发现Follower日志不一致时，回退Follower日志到一致点重新复制
```

**Raft vs Paxos**：
| 维度 | Raft | Paxos |
|------|------|-------|
| 可理解性 | 高（明确的Leader） | 低（多Proposer并发） |
| Leader | 必须有唯一Leader | 无强制Leader |
| 日志 | 连续的日志复制 | 可以乱序提交 |
| 工程实现 | etcd/Consul/TiKV | ZooKeeper(ZAB,Paxos变体) |

```go
func CallWithRetry(fn func() error, maxRetries int) error {
    var err error
    for i := 0; i < maxRetries; i++ {
        err = fn()
        if err == nil {
            return nil
        }
        
        // 指数退避
        backoff := time.Duration(1<<uint(i)) * time.Second

    return err
}
```

## 8. 面试题自查

### Q1: 常见的限流算法有哪些？令牌桶和漏桶的区别是什么？

**答**：4种：固定窗口计数器（简单但有临界突刺）、滑动窗口（解决临界问题但内存大）、令牌桶（定速生成令牌，允许突发消费，推荐）、漏桶（定速流出，平滑流量）。核心区别：令牌桶允许突发流量（消费已积累的令牌），漏桶不允许（恒定速率处理）。API限流用令牌桶，保护数据库用漏桶。

### Q2: 令牌桶的实现原理是什么？如何做到不用定时器生成令牌？

**答**：惰性计算：不真正定时生成令牌，而是在每次获取令牌时计算elapsed*rate得到应生成的令牌数，加上剩余令牌（不超过桶容量），然后判断是否足够。核心字段：capacity（桶容量）、tokens（当前令牌数）、rate（生成速率）、lastTime（上次取令牌时间）。burst通过桶容量实现：桶满时可瞬间消费capacity个请求。

### Q3: 分布式限流如何实现？

**答**：多实例共享限流配额。推荐Redis+Lua脚本实现：Lua脚本原子性地INCR计数+EXPIRE设过期时间，保证计数准确。也可用Nginx的limit_req_zone做网关层限流。更复杂的场景用Sentinel或自研限流中间件，支持热点参数限流、集群限流。

### Q4: 熔断器的状态机是怎样的？三种状态如何切换？

**答**：三态：Closed（正常，统计失败率）→ Open（失败率超阈值，拒绝所有请求快速失败）→ Half-Open（超时后允许少量请求试探）→ 试探成功回到Closed / 失败回到Open。核心参数：失败率阈值、统计周期、熔断持续时间、半开状态允许请求数。典型实现：gobreaker、Hystrix、Sentinel。

### Q5: 熔断和降级有什么区别？各自适用什么场景？

**答**：熔断是对**下游服务**的保护，下游故障率高时切断调用避免级联失败。降级是对**自身服务**的保护，负载过高时主动关闭非核心功能保障核心功能。熔断是被动触发（检测到故障），降级是主动策略（配置中心控制开关）。两者常结合使用：熔断后自动降级返回兜底数据。

### Q6: 分布式事务有哪些方案？各自的优缺点和适用场景？

**答**：①2PC（两阶段提交）：强一致但同步阻塞+单点故障，不推荐。②TCC（Try-Confirm-Cancel）：性能好无阻塞，但需实现3个接口，业务侵入强，适合金融转账。③SAGA：多个本地事务+补偿事务，长事务友好，补偿逻辑复杂，适合订单流程。④本地消息表（推荐默认）：本地事务写业务+消息表，定时扫描发MQ，可靠简单，有延迟。

### Q7: CAP定理为什么不能三者兼得？CP和AP的区别是什么？

**答**：网络分区P是客观存在的（不可避免）。分区发生后：写入N1的新数据无法同步到N2，此时N2要么返回旧数据（牺牲C选AP）要么拒绝服务（牺牲A选CP）。CP系统（如ZooKeeper）：分区时少数派拒绝服务，保证一致性。AP系统（如Cassandra）：分区时各节点独立服务，允许不一致，后续修复。CA在分布式中不存在。

### Q8: Raft协议的Leader选举和日志复制流程是怎样的？

**答**：**选举**：Follower超过election timeout未收到心跳→变为Candidate，term+1，发RequestVote请求多数派投票（每term每节点只投一票），获多数票成为Leader，立即发心跳。timeout随机化防split vote。**日志复制**：客户端写→Leader追加本地Log→并行AppendEntries给Follower→多数派确认→commit→返回成功。Leader通过回退找到与Follower的一致点修复日志。

### Q9: BASE理论是什么？和CAP的关系是什么？

**答**：BASE是对CAP中AP方案的延伸：Basically Available（基本可用，允许部分功能不可用）、Soft State（软状态，允许中间不一致状态存在）、Eventually Consistent（最终一致性，一段时间后达到一致）。本质是在牺牲强一致性（C）的前提下，通过最终一致性获得更好的可用性（A）。典型场景：电商订单异步扣库存、社交点赞数异步更新。

### Q10: 一致性Hash的原理是什么？虚拟节点解决什么问题？

**答**：将hash空间组成环(0~2^32-1)，节点和数据都映射到环上，数据顺时针找到第一个节点存储。节点增减只影响相邻节点（迁移数据量小）。虚拟节点解决数据倾斜问题：物理节点少时可能分布不均匀，给每个物理节点创建多个虚拟节点（如150个），均匀分布在环上，保证负载均衡。

### Q11: 设计一个秒杀系统，从前端到后端的全链路方案是什么？

**答**：①前端：按钮置灰+验证码防刷②CDN/Nginx：静态资源缓存+限流③网关：令牌桶限流（如1000QPS）④Redis预扣库存：DECR原子操作，<0则回滚返回售罄⑤MQ异步下单：通过请求进MQ缓冲，下游慢慢处理⑥订单服务消费MQ：创建订单+数据库扣库存(乐观锁防超卖)+发通知。关键：尽可能在前面拦截流量，真正到DB的请求量极少。

### Q12: 雪花算法的结构是什么？有什么问题？如何解决？

**答**：64位：1bit符号位 + 41bit毫秒时间戳（可用69年）+ 10bit机器ID（1024台）+ 12bit序列号（每毫秒4096个）。主要问题：**时钟回拨**——机器时间回调导致ID重复。解决：①检测到回拨就等待②记录上次时间戳，回拨时拒绝生成③使用NTP平滑调整④美团Leaf方案：号段模式+双buffer预加载，不依赖时钟。

### Q13: 如何保证接口幂等性？有哪些具体方案？

**答**：①唯一请求号：客户端生成UUID，服务端用Redis SETNX去重②数据库唯一索引：如订单号UNIQUE，重复插入报错③状态机：订单 created→paid，重复paid操作检查状态不符直接返回④Token机制：先获取Token存Redis，提交时Token校验+删除（原子操作）⑤乐观锁：UPDATE ... WHERE version=N，版本不对则不更新。

### Q14: 超时和重试策略需要注意什么？指数退避的原理？

**答**：注意事项：①重试前提是操作幂等（非幂等操作不能重试）②设置最大重试次数（防无限重试）③区分超时类型（连接超时短如2s，读超时长如5s）④重试风暴问题（下游故障时大量重试加重压力，需配合熔断）。指数退避：第N次重试等待2^N秒（1s,2s,4s,8s...），加随机抖动(jitter)防止多客户端同时重试。

### Q15: 如何设计一个短链接服务？核心问题有哪些？

**答**：核心问题：①短码生成：分布式ID+Base62编码（6位可表示568亿），或MD5取前6字节Base62②存储：MySQL（short_code主键, long_url）+ Redis缓存③查询：用户访问短链→Redis查→未命中查DB→301/302重定向④防重复：long_url加hash索引防同一长链生成多个短链⑤架构：CDN→负载均衡→短链服务→Redis→MySQL。高并发优化：302重定向（便于统计点击）、CDN缓存热门短链。
