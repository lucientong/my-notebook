## 1. 限流策略

### 1.1 为什么需要限流？

**场景**：
- 突发流量（秒杀、热点事件）
- 恶意攻击（DDoS、爬虫）
- 下游服务保护（防止雪崩）

**不限流的后果**：
- 服务器CPU/内存耗尽
- 数据库连接池耗尽
- 整个系统崩溃

### 1.2 常见限流算法

#### 1.2.1 固定窗口计数器

**原理**：统计固定时间窗口内的请求数

```go
type FixedWindowLimiter struct {
    limit    int64         // 限制数量
    window   time.Duration // 窗口大小
    counter  int64         // 当前计数
    lastTime time.Time     // 窗口开始时间
    mu       sync.Mutex
}

func (l *FixedWindowLimiter) Allow() bool {
    l.mu.Lock()
    defer l.mu.Unlock()
    
    now := time.Now()
    
    // 新窗口，重置计数
    if now.Sub(l.lastTime) >= l.window {
        l.counter = 0
        l.lastTime = now
    }
    
    // 检查是否超限
    if l.counter >= l.limit {
        return false
    }
    
    l.counter++
    return true
}

// 使用示例
limiter := &FixedWindowLimiter{
    limit:  100,              // 100次
    window: 1 * time.Second,  // 1秒
}

if limiter.Allow() {
    // 处理请求
} else {
    // 返回429 Too Many Requests
}
```

**优点**：实现简单  
**缺点**：临界问题（窗口边界流量突刺）

**临界问题示例**：
```
窗口1: 09:59:00 - 09:59:59  →  99次请求（在59秒时）
窗口2: 10:00:00 - 10:00:59  →  99次请求（在00秒时）
实际：1秒内处理了198次请求（突破限制）
```

#### 1.2.2 滑动窗口

**原理**：将固定窗口划分为多个小窗口，统计滑动窗口内的请求数

```go
type SlidingWindowLimiter struct {
    limit      int64
    window     time.Duration
    smallWindow time.Duration
    counters   map[int64]int64  // 时间戳 -> 计数
    mu         sync.Mutex
}

func (l *SlidingWindowLimiter) Allow() bool {
    l.mu.Lock()
    defer l.mu.Unlock()
    
    now := time.Now()
    windowStart := now.Add(-l.window)
    
    // 清理过期窗口
    for ts := range l.counters {
        if time.Unix(0, ts).Before(windowStart) {
            delete(l.counters, ts)
        }
    }
    
    // 统计当前窗口内的请求数
    var total int64
    for _, count := range l.counters {
        total += count
    }
    
    if total >= l.limit {
        return false
    }
    
    // 当前小窗口计数+1
    currentWindow := now.Truncate(l.smallWindow).UnixNano()
    l.counters[currentWindow]++
    
    return true
}
```

**优点**：解决临界问题  
**缺点**：内存占用较大

#### 1.2.3 令牌桶（推荐）

**原理**：按固定速率生成令牌，请求消耗令牌，桶满则丢弃令牌

```go
import "golang.org/x/time/rate"

// 创建限流器
limiter := rate.NewLimiter(100, 200)  // 100 token/s, burst 200

// 方式1：阻塞等待
err := limiter.Wait(context.Background())

// 方式2：非阻塞
if limiter.Allow() {
    // 处理请求
} else {
    // 返回429
}

// 方式3：等待N个令牌
reservation := limiter.Reserve()
if !reservation.OK() {
    // 无法获取令牌
    return
}
time.Sleep(reservation.Delay())  // 等待
```

**优点**：
- 平滑流量，允许突发（burst）
- Go官方库支持

**参数说明**：
- `rate`：令牌生成速率（QPS）
- `burst`：桶容量（允许的突发请求数）

#### 令牌桶手写实现（完整版）⭐⭐⭐

**为什么要手写？**
- 面试高频：手写限流器
- 理解原理：令牌如何生成、如何消费
- 自定义需求：官方库无法满足的场景

**核心思想**：
1. 不真正"生成"令牌（太耗费CPU）
2. 记录上次取令牌的时间，计算当前应该有多少令牌
3. 惰性生成：取令牌时才计算

```go
// 令牌桶限流器（手写实现）
type TokenBucket struct {
    capacity   int64         // 桶容量（最多存多少令牌）
    tokens     int64         // 当前令牌数
    rate       int64         // 生成速率（token/s）
    lastTime   time.Time     // 上次取令牌时间
    mu         sync.Mutex
}

func NewTokenBucket(rate int64, capacity int64) *TokenBucket {
    return &TokenBucket{
        capacity: capacity,
        tokens:   capacity,  // 初始满桶
        rate:     rate,
        lastTime: time.Now(),
    }
}

func (tb *TokenBucket) Allow() bool {
    return tb.AllowN(1)
}

func (tb *TokenBucket) AllowN(n int64) bool {
    tb.mu.Lock()
    defer tb.mu.Unlock()
    
    now := time.Now()
    
    // 计算从上次到现在应该生成多少令牌
    elapsed := now.Sub(tb.lastTime)
    newTokens := int64(elapsed.Seconds() * float64(tb.rate))
    
    // 更新令牌数（不超过容量）
    tb.tokens += newTokens
    if tb.tokens > tb.capacity {
        tb.tokens = tb.capacity
    }
    
    tb.lastTime = now
    
    // 检查是否有足够的令牌
    if tb.tokens >= n {
        tb.tokens -= n
        return true
    }
    
    return false
}

// 等待获取N个令牌（阻塞）
func (tb *TokenBucket) WaitN(ctx context.Context, n int64) error {
    for {
        if tb.AllowN(n) {
            return nil
        }
        
        // 计算需要等待的时间
        tb.mu.Lock()
        needed := n - tb.tokens
        waitTime := time.Duration(float64(needed) / float64(tb.rate) * float64(time.Second))
        tb.mu.Unlock()
        
        select {
        case <-time.After(waitTime):
            // 继续尝试
        case <-ctx.Done():
            return ctx.Err()
        }
    }
}

// 使用示例
func main() {
    // 100 QPS，允许突发200个请求
    limiter := NewTokenBucket(100, 200)
    
    // 场景1：瞬时200个请求
    for i := 0; i < 200; i++ {
        if limiter.Allow() {
            fmt.Println("请求", i, "通过")
        }
    }
    // 前200个通过，后面的被拒绝（桶空了）
    
    // 场景2：等待1秒后，又有100个令牌
    time.Sleep(1 * time.Second)
    if limiter.Allow() {
        fmt.Println("1秒后请求通过")  // 通过！
    }
}
```

**关键点**：

1. **惰性计算**：
   ```
   不实时生成令牌（每秒100次定时器太耗费）
   取令牌时才计算：elapsed * rate = 新令牌数
   ```

2. **时间精度**：
   ```go
   elapsed := now.Sub(tb.lastTime)
   newTokens := int64(elapsed.Seconds() * float64(tb.rate))
   
   // 例如：rate=100, elapsed=0.5秒
   // newTokens = 0.5 * 100 = 50个令牌
   ```

3. **突发流量**：
   ```
   桶容量=200，当前tokens=200
   瞬间200个请求 → 全部通过
   桶变空（tokens=0）
   后续请求需要等待令牌生成
   ```

#### 令牌桶 vs 漏桶（底层原理对比）⭐⭐⭐

**为什么令牌桶能允许突发，漏桶不行？**

| 维度 | 令牌桶 | 漏桶 |
|------|--------|------|
| **核心思想** | 令牌定速生成，请求变速消费 | 请求变速进入，定速流出 |
| **突发流量** | ✅ 允许（消费已存的令牌） | ❌ 不允许（队列+定速处理） |
| **流量曲线** | 锯齿状（允许瞬时峰值） | 平滑（恒定速率） |
| **实现复杂度** | 简单（计数器） | 中等（需要队列） |

**图解**：

```
【令牌桶】
令牌生成：    ─┬─┬─┬─┬─┬─┬─┬─> (恒定速率)
            令桶桶桶桶桶桶桶
请求消费：    ████─█─█─> (变速，可突发)

时刻T1: 桶满（100个令牌）
时刻T2: 瞬间100个请求 → 全部通过（消费100个令牌）
时刻T3: 桶空，新请求需要等待

【漏桶】
请求进入：    ████─█─█─> (变速)
            ┌──────┐
            │ 队列 │ (缓冲区)
            └──────┘
请求流出：    ─┬─┬─┬─┬─┬─┬─┬─> (恒定速率)

时刻T1: 100个请求瞬间到达 → 进入队列
时刻T2: 按固定速率处理（如每秒10个）
时刻T3: 需要10秒才能处理完100个请求
```

**为什么漏桶不能突发？**

```go
// 漏桶的leak()协程
func (lb *LeakyBucket) leak() {
    ticker := time.NewTicker(lb.rate)  // 固定速率
    defer ticker.Stop()
    
    for range ticker.C {
        select {
        case <-lb.queue:
            processRequest()  // 每个tick只处理1个请求
        default:
        }
    }
}

// 即使队列有1000个请求，也只能按固定速率处理
// 无法利用"空闲时段"加速处理
```

**应用场景对比**：

| 场景 | 推荐算法 | 原因 |
|------|----------|------|
| **API限流** | 令牌桶 | 用户体验好（允许短时突发） |
| **写数据库** | 漏桶 | 保护数据库（平滑QPS） |
| **消息队列消费** | 漏桶 | 避免下游被打垮（恒速） |
| **CDN流量** | 令牌桶 | 允许大文件快速下载（突发） |

**应用场景**：

#### 1.2.4 漏桶

**原理**：请求进入队列，按固定速率处理

```go
type LeakyBucket struct {
    capacity int           // 桶容量
    rate     time.Duration // 处理速率
    queue    chan struct{} // 请求队列
}

func NewLeakyBucket(capacity int, rate time.Duration) *LeakyBucket {
    lb := &LeakyBucket{
        capacity: capacity,
        rate:     rate,
        queue:    make(chan struct{}, capacity),
    }
    go lb.leak()
    return lb
}

func (lb *LeakyBucket) leak() {
    ticker := time.NewTicker(lb.rate)
    defer ticker.Stop()
    
    for range ticker.C {
        select {
        case <-lb.queue:
            // 处理一个请求
        default:
        }
    }
}

func (lb *LeakyBucket) Allow() bool {
    select {
    case lb.queue <- struct{}{}:
        return true
    default:
        return false  // 队列满
    }
}
```

**对比令牌桶**：
| 算法 | 流量特点 | 适用场景 |
|------|----------|----------|
| 令牌桶 | 允许突发 | API限流 |
| 漏桶 | 平滑流量 | 消息队列 |

## 2. 熔断与降级

### 2.1 熔断器（Circuit Breaker）

**原理**：当下游服务故障率超过阈值时，自动切断请求，快速失败

**状态机**：
```
关闭（Closed） → 打开（Open） → 半开（Half-Open） → 关闭
```

**状态转换**：
1. **Closed**：正常请求，统计失败率
2. **Open**：失败率超阈值，拒绝所有请求，快速返回错误
3. **Half-Open**：一段时间后，允许少量请求尝试
4. **Closed**：尝试成功，恢复正常

**Go实现（使用 gobreaker 库）**：

```go
import "github.com/sony/gobreaker"

// 创建熔断器
breaker := gobreaker.NewCircuitBreaker(gobreaker.Settings{
    Name:        "my-service",
    MaxRequests: 3,      // 半开状态允许3个请求
    Interval:    10 * time.Second,  // 统计周期
    Timeout:     60 * time.Second,  // 熔断持续时间
    ReadyToTrip: func(counts gobreaker.Counts) bool {
        // 失败率 > 50% 触发熔断
        failureRatio := float64(counts.TotalFailures) / float64(counts.Requests)
        return counts.Requests >= 10 && failureRatio >= 0.5
    },
})

// 使用熔断器
result, err := breaker.Execute(func() (interface{}, error) {
    return callDownstreamService()
})

if err != nil {
    if err == gobreaker.ErrOpenState {
        // 熔断器打开，快速失败
        return nil, errors.New("服务熔断")
    }
    return nil, err
}
```

**监控指标**：
```go
state := breaker.State()  // Closed/Open/Half-Open
counts := breaker.Counts()
log.Printf("State: %v, Requests: %d, Failures: %d", 
    state, counts.Requests, counts.TotalFailures)
```

### 2.2 降级

**定义**：系统负载过高时，关闭非核心功能，保障核心功能

**降级策略**：
1. **功能降级**：关闭推荐、评论等非核心功能
2. **读降级**：返回缓存数据
3. **写降级**：限制写操作，或异步写入
4. **返回默认值**：返回兜底数据

**实现方式**：

**方式1：配置中心（推荐）**
```go
type FeatureSwitch struct {
    EnableRecommendation bool `json:"enable_recommendation"`
    EnableComment        bool `json:"enable_comment"`
    EnableSearch         bool `json:"enable_search"`
}

var featureSwitch FeatureSwitch

// 定时从配置中心拉取
func LoadFeatureSwitch() {
    config := fetchFromConfigCenter()
    featureSwitch = config
}

// 使用
func GetRecommendation() ([]Item, error) {
    if !featureSwitch.EnableRecommendation {
        return []Item{}, nil  // 降级：返回空列表
    }
    return queryRecommendation()
}
```

**方式2：Sentinel（阿里开源）**
```go
import sentinel "github.com/alibaba/sentinel-golang/api"

// 定义资源

// 正常业务逻辑
```

## 3. 分布式事务

### 3.1 2PC（两阶段提交）

**流程**：
1. **准备阶段**：协调者询问所有参与者是否可以提交
2. **提交阶段**：所有参与者都同意，则提交；否则回滚

**缺点**：
- 同步阻塞（性能差）
- 单点故障（协调者挂了）
- 数据不一致（网络分区）

**结论**：❌ 不推荐使用

---

### 3.2 方案对比

| 方案 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **2PC** | 协调者两阶段提交 | 强一致性 | 同步阻塞、单点故障 | ❌ 不推荐 |
| **TCC** | Try预留→Confirm提交→Cancel回滚 | 性能好、不阻塞 | 业务侵入强，需实现3接口 | 金融转账等强一致场景 |
| **SAGA** | 多个本地事务+补偿事务 | 长事务友好 | 最终一致、补偿逻辑复杂 | 订单流程等长链路 |
| **本地消息表** | 本地事务写业务+消息，定时发MQ | 可靠性高、实现简单 | 需定时扫描、有延迟 | ✅ 推荐默认方案 |

### 3.3 TCC核心流程

```
Try阶段（预留资源）→ Confirm阶段（确认提交）→ Cancel阶段（取消回滚）

示例-转账：
Try:     冻结转出方资金
Confirm: 扣减转出方余额+冻结，增加收款方余额
Cancel:  解冻转出方资金
```

> 延伸阅读：TCC/SAGA 在金融支付领域的实践，详见《专项知识库/02-金融支付专项》。

### 3.4 SAGA核心流程

```
编排式(Orchestration)：中央协调者编排执行步骤和补偿
    创建订单 → 扣减库存 → 扣减余额 → 发送通知
    (补偿)取消订单 ← 恢复库存 ← 退款

协同式(Choreography)：各服务通过MQ事件自行响应
    ✅ 更解耦    ❌ 调试困难
```

**原理**：在本地事务中同时写业务数据和消息，定时扫描消息表发送MQ

## 4. 分布式一致性

### 4.1 CAP理论

| 特性 | 说明 |
|------|------|
| **C**onsistency | 一致性：所有节点看到相同数据 |
| **A**vailability | 可用性：任何请求都能得到响应 |
| **P**artition Tolerance | 分区容错：网络分区时系统仍可用 |

**定理**：三者只能同时满足两个

**选择**：
- **CP**：保证一致性，牺牲可用性（如：ZooKeeper、HBase）
- **AP**：保证可用性，牺牲强一致性（如：Cassandra、DynamoDB）
- **CA**：实际不存在（网络分区不可避免）

---

#### 4.1.1 CAP为什么只能满足两个？（推导证明）⭐⭐⭐

**为什么需要理解证明？**
- 面试高频：为什么CAP不能三者兼得？
- 架构选型：根据业务选择CP或AP
- 避免误区：CA在分布式系统中不存在

---

##### 证明：C + A + P 不可能同时满足

**场景设定**：
```
分布式系统：
- 节点N1：master（可读可写）
- 节点N2：slave（可读可写）
- 客户端C1、C2
```

**步骤1：网络分区发生**
```
初始状态：N1和N2数据一致，都是v0

时刻T1：网络分区
    ┌────┐          ┌────┐
    │ N1 │ ╳╳╳╳╳╳╳╳ │ N2 │
    └────┘ 网络断开  └────┘
     v0              v0
```

**步骤2：客户端写入N1**
```
时刻T2：C1向N1写入v1
    ┌────┐          ┌────┐
    │ N1 │ ╳╳╳╳╳╳╳╳ │ N2 │
    └────┘          └────┘
     v1              v0
    
问题：N1无法将v1同步到N2（网络断开）
```

**步骤3：客户端读取N2**
```
时刻T3：C2从N2读取数据

选择A：保证C + A（牺牲P）
  - N2返回v0（旧数据）
  - 违反一致性（C1看到v1，C2看到v0）
  
选择B：保证C + P（牺牲A）
  - N2拒绝服务："网络分区中，无法保证一致性"
  - 违反可用性
  
选择C：保证A + P（牺牲C）
  - N2返回v0（允许不一致）
  - 违反一致性（最终一致性）
```

**结论**：
```
在网络分区（P）存在的前提下：
- 要么牺牲一致性（C）→ AP
- 要么牺牲可用性（A）→ CP

无法同时满足C和A！
```

##### 为什么CA不存在？

**误区**：单机系统是CA（无网络分区）

**反驳**：
```
1. 单机系统不是分布式系统
   - CAP是针对分布式系统的定理
   - 单机没有"分布式"的概念

2. 网络分区不可避免
   - 即使在同一数据中心，也可能网络故障
   - 交换机故障、光缆断裂、路由器宕机
   - P是客观存在的，不是可选项

3. CA的真正含义
   - 在P不存在的假设下，可以同时满足C和A
   - 但P是客观存在的 → CA在实际中不存在
```

##### CP系统示例：ZooKeeper

**ZooKeeper的选择**：牺牲A，保证C和P

```
场景：ZooKeeper有5个节点（N1-N5），N1是Leader

时刻T1：网络分区
分区1：N1、N2（2个节点，少数派）
分区2：N3、N4、N5（3个节点，多数派）

时刻T2：客户端写入
- 向N1写入 → 失败（无法同步到多数派）
- 向N3写入 → 成功（N3、N4、N5达成共识）

时刻T3：客户端读取
- 从N1读取 → 失败（N1发现自己不是Leader）
- 从N3读取 → 成功（返回最新数据）

牺牲可用性（A）：
- 分区1中的节点拒绝服务（保证一致性）
- 只有多数派可用
```

**ZooKeeper的过半机制**：
```
写操作需要过半节点确认：
- 5个节点 → 至少3个确认
- 3个节点 → 至少2个确认

为什么？
- 保证任意两个多数派必有交集
- 5个节点：多数派=3
  - {N1, N2, N3} ∩ {N3, N4, N5} = {N3} ≠ ∅
  - 保证至少有1个节点知道最新数据

网络分区时：
- 只有多数派能提供服务（保证一致性）
- 少数派拒绝服务（牺牲可用性）
```

##### AP系统示例：Cassandra

**Cassandra的选择**：牺牲C，保证A和P

```
场景：Cassandra有5个节点（N1-N5）

时刻T1：网络分区
分区1：N1、N2
分区2：N3、N4、N5

时刻T2：客户端写入
- 向N1写入v1 → 成功（写入N1、N2，W=2）
- 向N3写入v2 → 成功（写入N3、N4、N5，W=3）

时刻T3：客户端读取
- 从N1读取 → 返回v1
- 从N3读取 → 返回v2
- 不一致！

牺牲一致性（C）：
- 允许分区内独立写入（可用性）
- 分区恢复后，通过Last-Write-Wins解决冲突
```

**Cassandra的可调一致性**：
```
W + R > N  →  强一致性
W + R ≤ N  →  最终一致性

W: 写操作需要确认的节点数
R: 读操作需要确认的节点数
N: 副本总数

示例（N=3）：
- W=1, R=1 → AP（可能读到旧数据）
- W=2, R=2 → 强一致性（W+R=4>3）
- W=3, R=1 → CP（写入全部节点才返回）
```

##### CAP定理的实际应用（工程中的权衡）

**1. 牺牲一致性（AP）的常见做法**：

```go
// 最终一致性：异步同步
func WriteData(key, value string) error {
    // 1. 立即写入主节点
    if err := master.Write(key, value); err != nil {
        return err
    }
    
    // 2. 异步同步到从节点（允许短暂不一致）
    go func() {
        for _, slave := range slaves {
            slave.Write(key, value)
        }
    }()
    
    return nil  // 立即返回（可用性）
}

// 读取时可能读到旧数据（不一致）
func ReadData(key string) (string, error) {
    // 从任意节点读取
    return randomSlave().Read(key)
}
```

**2. 牺牲可用性（CP）的常见做法**：

```go
// 强一致性：同步等待多数派确认
func WriteData(key, value string) error {
    // 需要多数派（3/5）确认才返回
    successCount := 0
    errChan := make(chan error, len(nodes))
    
    for _, node := range nodes {
        go func(n *Node) {
            errChan <- n.Write(key, value)
        }(node)
    }
    
    // 等待多数派确认
    for i := 0; i < len(nodes); i++ {
        if <-errChan == nil {
            successCount++
            if successCount >= len(nodes)/2 + 1 {
                return nil  // 多数派成功
            }
        }
    }
    
    return errors.New("无法达成多数派共识")  // 牺牲可用性
}
```

### 4.3 一致性Hash

**问题**：分布式缓存中，节点增减导致大量缓存失效

**方案**：一致性Hash

**原理**：
1. 将Hash空间组织成环（0 ~ 2^32-1）
2. 节点映射到环上（Hash(IP)）
3. 数据顺时针找到第一个节点

**优点**：节点增减只影响相邻节点

**实现**：
```go
type ConsistentHash struct {
    circle       map[uint32]string  // Hash值 -> 节点
    sortedHashes []uint32            // 排序的Hash值
    replicas     int                 // 虚拟节点数
}

func (ch *ConsistentHash) Add(node string) {
    for i := 0; i < ch.replicas; i++ {
        hash := crc32.ChecksumIEEE([]byte(fmt.Sprintf("%s:%d", node, i)))
        ch.circle[hash] = node
        ch.sortedHashes = append(ch.sortedHashes, hash)
    }
    sort.Slice(ch.sortedHashes, func(i, j int) bool {
        return ch.sortedHashes[i] < ch.sortedHashes[j]
    })
}

func (ch *ConsistentHash) Get(key string) string {
    hash := crc32.ChecksumIEEE([]byte(key))
    
    // 二分查找第一个 >= hash 的节点
    idx := sort.Search(len(ch.sortedHashes), func(i int) bool {
        return ch.sortedHashes[i] >= hash
    })
    
    if idx == len(ch.sortedHashes) {
        idx = 0  // 环形，回到第一个节点
    }
    
    return ch.circle[ch.sortedHashes[idx]]
}
```

### 4.4 Raft一致性协议⭐⭐⭐

> Raft是目前最主流的分布式一致性协议，etcd、Consul、TiKV等均基于Raft。比Paxos更易理解和实现。

**三种角色**：
```
Leader：   唯一的写入入口，负责日志复制和心跳
Follower： 被动接收Leader的日志和心跳
Candidate：选举期间的临时角色（Follower超时后变为Candidate）
```

**Leader选举流程**：
```
1. Follower在election timeout内未收到Leader心跳
2. Follower → Candidate，term+1，投票给自己，向其他节点RequestVote
3. 收到多数派投票 → 成为Leader
4. 新Leader立即发送心跳抑制其他选举

关键规则：
- 每个term每个节点只投一票（先到先得）
- Candidate日志不能比自己旧的节点不投票
- election timeout随机化（150~300ms）避免同时选举（split vote）
```

**日志复制流程**：
```
1. 客户端写请求 → Leader
2. Leader将日志追加到本地Log
3. Leader并行发送AppendEntries给所有Follower
4. 多数派确认 → Leader提交日志（commit）
5. Leader返回成功给客户端
6. 后续心跳通知Follower也提交

日志匹配原则：
- 相同index和term的日志内容一定相同
- Leader发现Follower日志不一致时，回退Follower日志到一致点重新复制
```

**Raft vs Paxos**：
| 维度 | Raft | Paxos |
|------|------|-------|
| 可理解性 | 高（明确的Leader） | 低（多Proposer并发） |
| Leader | 必须有唯一Leader | 无强制Leader |
| 日志 | 连续的日志复制 | 可以乱序提交 |
| 工程实现 | etcd/Consul/TiKV | ZooKeeper(ZAB,Paxos变体) |

```go
func CallWithRetry(fn func() error, maxRetries int) error {
    var err error
    for i := 0; i < maxRetries; i++ {
        err = fn()
        if err == nil {
            return nil
        }
        
        // 指数退避
        backoff := time.Duration(1<<uint(i)) * time.Second

    return err
}
```

### 案例：设计秒杀系统

#### 需求分析

**业务特点**：
- 瞬时超高并发（百万级QPS）
- 读多写少（浏览:下单 ≈ 1000:1）
- 库存有限（通常几百到几千件）

**核心指标**：

|------|------|
| 峰值QPS | 100万+ |
| 库存准确性 | 绝对不超卖 |
| 下单延迟 | < 500ms |
| 可用性 | 99.99% |

#### 架构设计

```
                        ┌─────────────────────────────────────────────────────┐
                        │                    秒杀全链路架构                      │
                        └─────────────────────────────────────────────────────┘

   [用户] ──→ [CDN静态页] ──→ [Nginx限流+本地缓存]
                                      │
                          ┌───────────┴───────────┐
                          │    第一层：接入层过滤     │  过滤 ~90% 流量
                          │  • 验证码/答题防刷       │
                          │  • 令牌桶限流(1000QPS)   │
                          │  • 用户维度频控          │
                          └───────────┬───────────┘
                                      │ (~10% 流量)
                          ┌───────────┴───────────┐
                          │   第二层：Redis预扣库存   │  过滤 ~99% 流量
                          │  • Lua原子扣减           │
                          │  • 库存≤0直接拒绝        │
                          │  • 扣减成功发MQ          │
                          └───────────┬───────────┘
                                      │ (~0.1% 流量)
                          ┌───────────┴───────────┐
                          │   第三层：Kafka异步下单   │
                          │  • 削峰填谷              │
                          │  • 消费者慢慢处理        │
                          └───────────┬───────────┘
                                      │
                          ┌───────────┴───────────┐
                          │   第四层：订单服务        │
                          │  • 创建订单              │

                          └───────────────────────┘
```

**核心思想：层层过滤，真正到DB的请求极少。**

#### 核心代码

**1. Redis Lua预扣库存（原子操作）**

```lua
-- seckill_deduct.lua
-- KEYS[1]: 库存key  seckill:stock:{sku_id}
-- KEYS[2]: 已购用户集合  seckill:bought:{sku_id}
-- ARGV[1]: 用户ID
-- ARGV[2]: 扣减数量(通常为1)

-- 1. 判断用户是否已购买（防重复下单）
if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 1 then
    return -1  -- 已购买
end

-- 2. 获取当前库存
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil then
    return -2  -- 活动不存在
end

if stock <= 0 then
    return 0   -- 库存不足
end

-- 3. 扣减库存 + 记录用户（原子操作）
redis.call('DECRBY', KEYS[1], tonumber(ARGV[2]))
redis.call('SADD', KEYS[2], ARGV[1])

return 1  -- 扣减成功
```

**2. 秒杀服务（Go实现）**

```go
package seckill

import (
    "context"
    "encoding/json"
    "fmt"
    "time"

    "github.com/go-redis/redis/v8"
    "github.com/segmentio/kafka-go"
)

// SeckillService 秒杀服务
type SeckillService struct {
    rdb         *redis.Client
    kafkaWriter *kafka.Writer
    luaScript   *redis.Script
}

// SeckillOrder 秒杀订单消息
type SeckillOrder struct {
    UserID    int64  `json:"user_id"`
    SkuID     int64  `json:"sku_id"`
    OrderID   string `json:"order_id"`
    Timestamp int64  `json:"timestamp"`
}

func NewSeckillService(rdb *redis.Client, kafkaWriter *kafka.Writer) *SeckillService {
    // 加载Lua脚本（启动时加载，运行时用SHA调用，减少网络开销）
    luaScript := redis.NewScript(`
        if redis.call('SISMEMBER', KEYS[2], ARGV[1]) == 1 then
            return -1
        end
        local stock = tonumber(redis.call('GET', KEYS[1]))
        if stock == nil then return -2 end
        if stock <= 0 then return 0 end
        redis.call('DECRBY', KEYS[1], tonumber(ARGV[2]))
        redis.call('SADD', KEYS[2], ARGV[1])
        return 1
    `)
    return &SeckillService{rdb: rdb, kafkaWriter: kafkaWriter, luaScript: luaScript}
}

// DoSeckill 执行秒杀（核心入口）
func (s *SeckillService) DoSeckill(ctx context.Context, userID, skuID int64) (string, error) {
    stockKey := fmt.Sprintf("seckill:stock:%d", skuID)
    boughtKey := fmt.Sprintf("seckill:bought:%d", skuID)

    // 1. Redis Lua原子扣库存
    result, err := s.luaScript.Run(ctx, s.rdb,
        []string{stockKey, boughtKey},
        userID, 1,
    ).Int()
    if err != nil {
        return "", fmt.Errorf("redis error: %w", err)
    }

    switch result {
    case -1:
        return "", fmt.Errorf("already purchased")
    case -2:
        return "", fmt.Errorf("activity not found")
    case 0:
        return "", fmt.Errorf("sold out")
    }

    // 2. 扣减成功，发送Kafka异步下单
    orderID := generateOrderID(userID, skuID)
    msg := SeckillOrder{
        UserID:    userID,
        SkuID:     skuID,
        OrderID:   orderID,
        Timestamp: time.Now().UnixMilli(),
    }
    body, _ := json.Marshal(msg)

    err = s.kafkaWriter.WriteMessages(ctx, kafka.Message{
        Key:   []byte(fmt.Sprintf("%d", userID)),
        Value: body,
    })
    if err != nil {
        // Kafka写入失败 → 回滚Redis库存
        s.rdb.Incr(ctx, stockKey)
        s.rdb.SRem(ctx, boughtKey, userID)
        return "", fmt.Errorf("send message failed: %w", err)
    }

    // 3. 返回订单号，前端轮询订单状态
    return orderID, nil
}
```

### 案例：设计Feed流系统

#### 需求分析

**业务场景**：
- 用户关注其他用户后，在首页看到关注人的动态（类似微博、朋友圈、Twitter）
- 核心操作：发布动态（写）、刷Feed流（读）、关注/取关（关系变更）

**核心指标**：
| 指标 | 目标 |
|------|------|
| Feed读取延迟 | P99 < 200ms |
| 发布延迟 | < 1s（异步可接受） |
| 日活用户 | 1亿+ |
| 单用户关注上限 | 5000人 |
| 大V粉丝数 | 千万级 |

#### 推/拉/混合模式对比

```
【推模式（Write Fanout / Push）】

用户A发布动态 → 写入A的所有粉丝的收件箱（Timeline）

  A发布帖子P1
      │
      ├──→ 写入粉丝B的inbox: [P1, ...]
      ├──→ 写入粉丝C的inbox: [P1, ...]
      └──→ 写入粉丝D的inbox: [P1, ...]

  读取：直接读自己的inbox，已排好序 → 极快
  写入：N个粉丝 → N次写操作 → 大V写放大严重

  ✅ 读快（O(1)直接取inbox）
  ❌ 写放大（大V百万粉丝 → 百万次写）
  ❌ 存储大（每个粉丝都存一份引用）


【拉模式（Read Fanout / Pull）】

用户B刷Feed → 实时拉取所有关注人的最新动态 → 合并排序

  B刷Feed
      │
      ├──→ 拉取关注人A的最新动态
      ├──→ 拉取关注人C的最新动态
      └──→ 拉取关注人D的最新动态
      │
      └──→ 合并排序，取Top N

  读取：拉取K个关注人的动态，合并排序 → 关注多则慢
  写入：只写自己的发件箱 → 极快

  ✅ 写快（只写一份）
  ✅ 存储小
  ❌ 读放大（关注5000人 → 5000次读+排序）
  ❌ 延迟高


【混合模式（推荐方案）】

  普通用户（粉丝<10万）→ 推模式（写粉丝inbox）
  大V用户（粉丝≥10万）→ 拉模式（读时实时拉取）

  B刷Feed:
      inbox（已有普通关注人推过来的动态）
        +
      实时拉取关注的大V的最新动态
        ↓
      合并排序，取Top N
```

#### 架构设计

```
                    ┌──────────────────────────────────────────────────┐
                    │              Feed流混合架构                       │
                    └──────────────────────────────────────────────────┘

  【写入路径】
  用户发布动态 ──→ [发布服务] ──→ [MySQL写入动态表]
                       │
                       ├──→ 判断粉丝数
                       │    ├── < 10万 → [Kafka: fanout-topic]
                       │    │              → [Fanout消费者] → 写入粉丝inbox(Redis ZSET)
                       │    │
                       │    └── ≥ 10万 → 只写发件箱(Redis ZSET)，不扇出
                       │
                       └──→ [更新用户发件箱 outbox:user_id]

  【读取路径】
  用户刷Feed ──→ [Feed服务]
                    │
                    ├── 1. 读取自己的inbox (ZREVRANGE，已有推过来的内容)
                    │
                    ├── 2. 获取关注的大V列表
                    │      → 逐个拉取大V的outbox最新内容
                    │
                    ├── 3. 合并排序 (归并排序，按时间戳倒序)
                    │
                    └── 4. 补全内容详情 (批量mget帖子内容)

```

```go
// Publish 发布动态
func (s *FeedService) Publish(ctx context.Context, authorID, postID int64) error {
    score := float64(time.Now().UnixMilli())

    // 1. 写入自己的发件箱
    s.rdb.ZAdd(ctx, outboxKey(authorID), &redis.Z{
        Score:  score,
        Member: postID,
    })
    s.rdb.ZRemRangeByRank(ctx, outboxKey(authorID), 0, -inboxMaxSize-1) // 裁剪

    // 2. 判断是否需要扇出
    followerCount, _ := s.relationSvc.GetFollowerCount(ctx, authorID)
    if followerCount >= bigVThreshold {
        // 大V：不扇出，粉丝读时拉取
        return nil
    }

    // 3. 普通用户：异步扇出到粉丝inbox（通过Kafka）
    return s.sendFanoutMessage(ctx, authorID, postID, score)
}

// FanoutToFollowers Kafka消费者调用：推送到粉丝inbox
func (s *FeedService) FanoutToFollowers(ctx context.Context, authorID, postID int64, score float64) error {
    // 分批获取粉丝列表
    var cursor int64
    for {
        followers, nextCursor, err := s.relationSvc.GetFollowersBatch(ctx, authorID, cursor, 1000)
        if err != nil {
            return err
        }

        // Pipeline批量写入（减少网络往返）
        pipe := s.rdb.Pipeline()
        for _, followerID := range followers {
            key := inboxKey(followerID)
            pipe.ZAdd(ctx, key, &redis.Z{Score: score, Member: postID})
            pipe.ZRemRangeByRank(ctx, key, 0, -inboxMaxSize-1) // 保持inbox大小
        }
        _, err = pipe.Exec(ctx)
        if err != nil {
            return err
        }

        if nextCursor == 0 {
            break
        }
        cursor = nextCursor
    }
    return nil
}
```

```go
// GetFeed 获取用户的Feed流（混合推拉）
func (s *FeedService) GetFeed(ctx context.Context, userID int64, offset, limit int64) ([]FeedItem, error) {
    // 1. 读取inbox（已推送的普通用户内容）
    inboxItems, err := s.rdb.ZRevRangeWithScores(ctx, inboxKey(userID), offset, offset+limit-1).Result()
    if err != nil {
        return nil, err
    }

    // 2. 获取关注的大V列表
    bigVList, err := s.relationSvc.GetFollowingBigVs(ctx, userID)
    if err != nil {
        return nil, err
    }

    // 3. 实时拉取大V的最新动态
    var bigVItems []redis.Z
    if len(bigVList) > 0 {
        pipe := s.rdb.Pipeline()
        cmds := make([]*redis.ZSliceCmd, len(bigVList))
        for i, bigVID := range bigVList {
            cmds[i] = pipe.ZRevRangeWithScores(ctx, outboxKey(bigVID), 0, limit-1)
        }
        pipe.Exec(ctx)

        for _, cmd := range cmds {
            items, _ := cmd.Result()
            bigVItems = append(bigVItems, items...)
        }
    }

    // 4. 合并 + 排序（归并排序，按score倒序）
    merged := mergeAndSort(inboxItems, bigVItems)

    // 5. 截取分页
    end := int(limit)
    if end > len(merged) {
        end = len(merged)
    }

    // 6. 批量补全帖子内容
    return s.batchGetPostContent(ctx, merged[:end])
}

// mergeAndSort 合并两个有序列表并按score倒序排列
func mergeAndSort(a, b []redis.Z) []redis.Z {
    merged := append(a, b...)
    sort.Slice(merged, func(i, j int) bool {
        return merged[i].Score > merged[j].Score // 按时间倒序
    })
    // 去重（同一个postID可能在inbox和outbox都出现）
    seen := make(map[interface{}]bool)
    result := make([]redis.Z, 0, len(merged))
    for _, item := range merged {
        if !seen[item.Member] {
            seen[item.Member] = true
            result = append(result, item)
        }
    }
    return result
}
```

```go
package ratelimit

import (
    "sync"
    "time"
)

// TokenBucketLimiter 令牌桶限流器（单机版）
type TokenBucketLimiter struct {
    rate       float64       // 令牌生成速率(token/s)
    capacity   float64       // 桶容量
    tokens     float64       // 当前令牌数
    lastRefill time.Time     // 上次补充时间
    mu         sync.Mutex
}

func NewTokenBucketLimiter(rate float64, capacity float64) *TokenBucketLimiter {
    return &TokenBucketLimiter{
        rate:       rate,
        capacity:   capacity,
        tokens:     capacity,   // 初始满桶
        lastRefill: time.Now(),
    }
}

func (l *TokenBucketLimiter) Allow() bool {
    return l.AllowN(1)
}

func (l *TokenBucketLimiter) AllowN(n float64) bool {
    l.mu.Lock()
    defer l.mu.Unlock()

    now := time.Now()
    // 惰性补充令牌
    elapsed := now.Sub(l.lastRefill).Seconds()
    l.tokens += elapsed * l.rate
    if l.tokens > l.capacity {
        l.tokens = l.capacity
    }
    l.lastRefill = now

    if l.tokens >= n {
        l.tokens -= n
        return true
    }
    return false
}
```

**2. 滑动窗口——本地单机版（Go完整实现）**

```go
// SlidingWindowLimiter 滑动窗口限流器（单机版）
type SlidingWindowLimiter struct {
    limit       int64             // 窗口内最大请求数
    window      time.Duration     // 窗口大小
    precision   time.Duration     // 小窗口精度
    counters    map[int64]int64   // 小窗口时间戳 → 计数
    mu          sync.Mutex
}

func NewSlidingWindowLimiter(limit int64, window, precision time.Duration) *SlidingWindowLimiter {
    return &SlidingWindowLimiter{
        limit:     limit,
        window:    window,
        precision: precision,
        counters:  make(map[int64]int64),
    }
}

func (l *SlidingWindowLimiter) Allow() bool {
    l.mu.Lock()
    defer l.mu.Unlock()

    now := time.Now()
    // 当前小窗口的key
    currentBucket := now.Truncate(l.precision).UnixNano()
    // 窗口起始时间
    windowStart := now.Add(-l.window).UnixNano()

    // 清理过期小窗口 + 统计窗口内总数
    var total int64
    for ts, count := range l.counters {
        if ts < windowStart {
            delete(l.counters, ts)
        } else {
            total += count
        }
    }

    if total >= l.limit {
        return false
    }

    l.counters[currentBucket]++
    return true
}
```

**3. Redis分布式令牌桶（Lua脚本）**

```lua
-- distributed_token_bucket.lua
-- KEYS[1]: 限流key（如 ratelimit:api:/order/create:user:1001）
-- ARGV[1]: 桶容量(capacity)
-- ARGV[2]: 令牌生成速率(rate, token/s)
-- ARGV[3]: 当前时间戳(秒，精确到毫秒用浮点)
-- ARGV[4]: 请求的令牌数(通常为1)

local key = KEYS[1]
local capacity = tonumber(ARGV[1])
local rate = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local requested = tonumber(ARGV[4])

-- 获取当前桶状态
local bucket = redis.call('HMGET', key, 'tokens', 'last_refill')
local tokens = tonumber(bucket[1])
local last_refill = tonumber(bucket[2])

-- 首次访问，初始化满桶
if tokens == nil then
    tokens = capacity
    last_refill = now
end

-- 惰性补充令牌
local elapsed = math.max(0, now - last_refill)
tokens = math.min(capacity, tokens + elapsed * rate)

-- 判断是否有足够令牌
local allowed = 0
if tokens >= requested then
    tokens = tokens - requested
    allowed = 1
end

-- 更新桶状态
redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
redis.call('EXPIRE', key, math.ceil(capacity / rate) * 2)  -- 自动过期

return allowed
```

**4. Redis分布式滑动窗口（Lua脚本）**

```lua
-- distributed_sliding_window.lua
-- KEYS[1]: 限流key
-- ARGV[1]: 窗口大小(秒)
-- ARGV[2]: 窗口内最大请求数
-- ARGV[3]: 当前时间戳(毫秒)
-- ARGV[4]: 唯一请求ID(member)

local key = KEYS[1]
local window = tonumber(ARGV[1]) * 1000  -- 转毫秒
local limit = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local member = ARGV[4]

-- 清除窗口外的过期记录
redis.call('ZREMRANGEBYSCORE', key, 0, now - window)

-- 统计当前窗口内的请求数
local count = redis.call('ZCARD', key)

if count >= limit then
    return 0  -- 超限
end

-- 添加当前请求（score=时间戳, member=唯一ID）
redis.call('ZADD', key, now, member)
redis.call('PEXPIRE', key, window)  -- 设置过期

return 1  -- 允许
```

```go
// AllowTokenBucket 分布式令牌桶判定
func (l *DistributedRateLimiter) AllowTokenBucket(
    ctx context.Context, key string, capacity, rate float64,
) (bool, error) {
    now := float64(time.Now().UnixMilli()) / 1000.0 // 秒级浮点
    result, err := l.tokenBucketScript.Run(ctx, l.rdb,
        []string{key},
        capacity, rate, now, 1,
    ).Int()
    if err != nil {
        return true, err // Redis故障时默认放行（fail-open）
    }
    return result == 1, nil
}

// AllowSlidingWindow 分布式滑动窗口判定
func (l *DistributedRateLimiter) AllowSlidingWindow(
    ctx context.Context, key string, window time.Duration, limit int64,

    member := uuid.New().String() // 唯一请求ID

        []string{key},
        int(window.Seconds()), limit, now, member,
    ).Int()
    if err != nil {
        return true, err // fail-open
    }
    return result == 1, nil
}
```

### 分布式ID生成深入

#### Snowflake 算法详解

Snowflake（雪花算法）由 Twitter 开源，核心思想是用一个 **64bit 的 long** 来编码全局唯一 ID：

```
 0 | 00000000 00000000 00000000 00000000 00000000 0 | 00000000 00 | 00000000 0000
 ↑                      ↑                                ↑              ↑
1bit                 41bit 时间戳                    10bit 机器ID    12bit 序列号
符号位(0)        (当前时间-纪元, ~69年)            (最多1024节点)   (同毫秒内自增, 每ms 4096个)
```

| 字段 | 位数 | 说明 |
|------|------|------|
| 符号位 | 1 bit | 始终为 0，保证 ID 为正数 |
| 时间戳 | 41 bit | 毫秒级，自定义纪元起始，可用约 69 年 |
| 机器 ID | 10 bit | 可拆为 5bit 数据中心 + 5bit 机器，共 1024 节点 |
| 序列号 | 12 bit | 同一毫秒内自增，单节点每毫秒最多生成 4096 个 ID |

**核心优势**：趋势递增、不依赖外部存储、高性能（本地生成）、ID 中可反解出时间。

#### 时钟回拨问题

**产生原因**：
- NTP（Network Time Protocol）同步时将系统时钟向回调整
- 闰秒调整
- 虚拟机快照恢复后时间回退

**检测方案**：每次生成 ID 时比较当前时间戳与上次生成 ID 的时间戳，若 `currentTimestamp < lastTimestamp` 则发生了回拨。

**处理策略**：

| 策略 | 做法 | 适用场景 |
|------|------|----------|
| 直接报错 | 抛异常，调用方重试 | 回拨容忍度为 0 的场景 |
| 短暂等待 | 若回拨 < 阈值（如 5ms），spin 等待追上 | 微小回拨（NTP 微调） |
| 借用未来时间戳 | 在序列号中"借位"，使用 lastTimestamp+1 | 不允许报错的高可用场景 |
| 扩展位 | 预留 2-3bit 作为时钟序列号，每次回拨 +1 | 回拨频繁但幅度小的环境 |

#### Go 完整实现（含时钟回拨处理）

```go
package snowflake

import (
    "errors"
    "sync"
    "time"
)

const (
    epoch             int64 = 1609459200000 // 自定义纪元: 2021-01-01 00:00:00 UTC
    workerIDBits      uint8 = 10
    sequenceBits      uint8 = 12
    maxWorkerID       int64 = -1 ^ (-1 << workerIDBits) // 1023
    maxSequence       int64 = -1 ^ (-1 << sequenceBits) // 4095
    workerIDShift           = sequenceBits               // 12
    timestampShift          = sequenceBits + workerIDBits // 22
    maxClockBackward        = 5 // 允许最大时钟回拨毫秒数
)

type Generator struct {
    mu            sync.Mutex
    lastTimestamp int64
    workerID      int64
    sequence      int64
}

func NewGenerator(workerID int64) (*Generator, error) {
    if workerID < 0 || workerID > maxWorkerID {
        return nil, errors.New("worker ID out of range [0, 1023]")
    }
    return &Generator{workerID: workerID}, nil
}

func (g *Generator) currentMillis() int64 {
    return time.Now().UnixMilli()
}

func (g *Generator) waitUntil(target int64) int64 {
    for {
        now := g.currentMillis()
        if now >= target {
            return now
        }
        time.Sleep(time.Millisecond)
    }
}

func (g *Generator) NextID() (int64, error) {
    g.mu.Lock()
    defer g.mu.Unlock()

    now := g.currentMillis()

    // ---- 时钟回拨处理 ----
    if now < g.lastTimestamp {
        offset := g.lastTimestamp - now
        if offset <= maxClockBackward {
            // 短暂回拨：spin 等待追上
            now = g.waitUntil(g.lastTimestamp)
        } else {
            return 0, errors.New("clock moved backwards, refusing to generate ID")
        }
    }

    if now == g.lastTimestamp {
        g.sequence = (g.sequence + 1) & maxSequence
        if g.sequence == 0 {
            // 当前毫秒序列号用完，等待下一毫秒
            now = g.waitUntil(g.lastTimestamp + 1)
        }
    } else {
        g.sequence = 0
    }

    g.lastTimestamp = now

    id := ((now - epoch) << timestampShift) |
        (g.workerID << int64(workerIDShift)) |
        g.sequence
    return id, nil
}
```

**Worker ID 分配方案**：
- **配置文件 / 环境变量**：最简单，适合节点固定的场景
- **ZooKeeper 顺序节点**：创建 `/snowflake/workers/` 顺序子节点，取编号 mod 1024
- **数据库自增**：启动时 INSERT 获取自增 ID，取 mod 1024
- **IP/端口 Hash**：`hash(ip + port) % 1024`，需注意碰撞

#### 号段模式（Leaf-Segment）

美团 Leaf-Segment 方案核心思路：每次从 DB **批量领取**一段 ID（号段），本地发号，号段用完再取。

**数据库表设计**：

```sql
CREATE TABLE leaf_alloc (
    biz_tag    VARCHAR(128) NOT NULL COMMENT '业务标识',
    max_id     BIGINT       NOT NULL COMMENT '当前已分配的最大ID',
    step       INT          NOT NULL COMMENT '每次领取的号段长度',
    version    INT          NOT NULL COMMENT '乐观锁版本号',
    update_time TIMESTAMP   NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    PRIMARY KEY (biz_tag)
) ENGINE=InnoDB;
```

**领号 SQL（乐观锁）**：

```sql
UPDATE leaf_alloc
   SET max_id = max_id + step, version = version + 1
 WHERE biz_tag = #{bizTag} AND version = #{version};
```

**双 Buffer 预加载**：

```
                ┌──────────────────────┐
                │  Segment Buffer[0]   │  ← 当前使用中
                │  [1001 ... 2000]     │
                ├──────────────────────┤
                │  Segment Buffer[1]   │  ← 后台预加载
                │  [2001 ... 3000]     │
                └──────────────────────┘
触发条件：当前 Buffer 使用率 > 10%（可配置）时，异步加载下一个号段到另一个 Buffer
切换条件：当前 Buffer 用完，原子切换到另一个已就绪的 Buffer
```

**优缺点**：
- ✅ 可容忍短暂 DB 宕机（本地还有余量）
- ✅ ID 严格递增（非趋势递增，是真递增）
- ❌ 服务重启号段浪费
- ❌ 依赖数据库，DB 长时间不可用仍会中断

#### 分布式 ID 方案对比

| 方案 | 有序性 | 性能 | 可用性 | 外部依赖 | 信息安全 | 适用场景 |
|------|--------|------|--------|----------|----------|----------|
| UUID v4 | 无序 | ★★★★★ | 无单点 | 无 | ★★★★★（不可反解） | 需全局唯一但不需有序 |
| Snowflake | 趋势递增 | ★★★★★ | 无单点 | 无（需分配workerID） | ★★（可反解时间/机器） | 高并发、趋势有序 |
| Leaf-Segment | 严格递增 | ★★★★ | 较高（双Buffer） | MySQL | ★★★（连续可预测） | 订单号等需递增场景 |
| Redis INCR | 严格递增 | ★★★★ | 取决于Redis HA | Redis | ★★★ | 已有 Redis 基础设施 |
| 数据库自增 | 严格递增 | ★★ | 单点瓶颈 | MySQL | ★★★ | 低并发、简单业务 |

> **面试高频追问**：为什么 Snowflake ID 用作 MySQL 主键时，比 UUID 性能好？——因为趋势递增的 ID 在 B+ Tree 上是顺序写入，不会导致频繁的页分裂；UUID 是随机的，插入位置随机，页分裂严重，写放大大。

### 分布式锁深入与 RedLock 争议

#### Redis 单节点分布式锁：标准实现

**加锁**（原子操作）：

```redis
SET lock_key <random_value> NX PX 30000
```

- `NX`：key 不存在才设置（互斥）
- `PX 30000`：过期时间 30s（防死锁）
- `random_value`：客户端唯一标识（防误删别人的锁）

**解锁**（Lua 脚本保证原子性）：

```lua
-- KEYS[1] = lock_key, ARGV[1] = random_value
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

> **为什么不能直接 DEL？**——客户端 A 持锁超时后锁自动过期，客户端 B 获得新锁，此时 A 的 DEL 会误删 B 的锁。必须先比较 `random_value` 确认是自己的锁再删。

#### 锁续期：Watchdog 机制（Redisson 方案）

**问题**：业务执行时间不可控，若锁在业务完成前过期，其他客户端可获得锁，导致并发冲突。

**Redisson Watchdog 工作原理**：

```
客户端获取锁（默认 leaseTime = 30s）
    │
    ├── 启动 Watchdog 定时任务（每 leaseTime/3 = 10s 执行一次）
    │       │
    │       ├── 检查客户端是否仍持有锁
    │       │       │
    │       │       ├── 是 → PEXPIRE lock_key 30000（续期）
    │       │       └── 否 → 停止 Watchdog
    │       │
    │       └── 循环直到锁释放
    │
    └── 业务执行完毕 → 释放锁 → Watchdog 自动停止
```

**关键细节**：
- 只有未指定 `leaseTime` 时 Watchdog 才启动；手动指定了过期时间则不续期
- 客户端进程崩溃 → Watchdog 线程一起挂掉 → 锁到期自动释放 → 不会死锁

#### RedLock 算法

Redis 作者 Antirez 提出的分布式锁方案，目标是解决**单节点 Redis 宕机导致锁丢失**的问题。

**算法步骤（5 个独立 Redis 节点）**：

```
1. 获取当前时间 T1
2. 依次向 5 个节点发送 SET lock NX PX <ttl>
3. 获取当前时间 T2
4. 计算：锁获取耗时 = T2 - T1
         锁实际有效时间 = ttl - (T2 - T1) - 时钟漂移补偿
5. 判断：
   - 成功获取 >= 3 个节点（多数派）的锁
   - 且锁实际有效时间 > 0
   → 加锁成功
6. 若加锁失败 → 向所有节点发送释放锁请求
```

#### Martin Kleppmann vs Antirez 之争

这是分布式系统领域最经典的一次公开技术辩论（2016年）。

**Kleppmann 的核心观点**（"How to do distributed locking"）：

1. **分布式锁有两种用途**：
   - **效率型**（避免重复工作）：单节点锁就够了，锁偶尔失效可接受
   - **正确性型**（保护数据一致性）：必须绝对安全，RedLock 做不到

2. **RedLock 依赖时钟假设，这在分布式系统中不安全**：
   - 进程可能被 GC pause / 系统调度挂起任意时间
   - 持锁客户端在 GC 暂停期间锁已过期，但客户端不知道
   - NTP 跳变可能导致节点感知时间不一致

3. **正确的方案是 Fencing Token**：
   ```
   Client A 获取锁 → 拿到 token=33
   Client A GC暂停... 锁过期
   Client B 获取锁 → 拿到 token=34
   Client A 恢复，带 token=33 写存储
   存储层拒绝：已见过更大的 token=34
   ```
   通过单调递增的 fencing token，让存储层拒绝过期的写操作。

**Antirez 的反驳**（"Is Redlock safe?"）：

1. **GC pause 不是问题**：锁获取后会检查剩余有效时间，GC pause 发生在获取过程中会被检测到
2. **时钟漂移可控**：现代运维环境下 NTP 配置得当，时钟漂移在毫秒级，远小于锁的 TTL
3. **Fencing Token 要求存储层支持 CAS**：如果存储层本身能做 CAS，某些场景根本不需要锁
4. **RedLock 对"合理的系统假设"下是安全的**：不追求拜占庭容错，只要时钟大致正确即可

**争论核心总结**：

| 维度 | Kleppmann | Antirez |
|------|-----------|---------|
| 理论立场 | 异步系统模型，不能假设时钟 | 半同步模型，时钟有界漂移 |
| 对 GC pause | 致命缺陷 | 获取阶段可检测 |
| 推荐方案 | Fencing token + 单节点锁 | RedLock |
| 核心分歧 | 分布式锁**不应该**依赖时钟 | 工程实践中时钟**足够可靠** |

#### 工程实践：分布式锁选型决策

| 场景 | 推荐方案 | 理由 |
|------|----------|------|
| 防重复执行（效率型） | Redis 单节点锁 | 简单高效，偶尔失效无严重后果 |
| 高并发抢锁 + 中等一致性 | Redis 单节点锁 + Watchdog | Redisson 开箱即用，性能好 |
| 强一致性 + 已有 ZK 集群 | ZooKeeper 临时顺序节点 | CP 系统，锁语义严格 |
| 强一致性 + 云原生 / K8s | etcd Lease + Revision | 轻量、已集成在 K8s 中 |
| 跨机房 + 极端可用性要求 | RedLock（谨慎使用） | 无单点，但要确保 NTP 同步良好 |
| 金融级正确性 | ZK/etcd + Fencing Token | 遵循 Kleppmann 建议 |

> **面试回答模板**：先说清楚场景是效率型还是正确性型；效率型直接 Redis 单节点锁；正确性型用 ZK/etcd，如果面试官追问 RedLock，能讲出 Kleppmann vs Antirez 的核心分歧即为加分项。

#### etcd 分布式锁实现（Lease + Revision）

etcd 基于 **Raft 一致性协议**，天然适合做分布式锁：

**核心机制**：

1. **Lease（租约）**：类似 Redis 的过期时间，客户端需要定期续租（KeepAlive）
2. **Revision（全局版本号）**：etcd 每次写操作全局递增，天然的 Fencing Token
3. **Watch**：监听 key 变更，实现阻塞等待

**加锁流程**：

```
1. 创建 Lease（如 TTL=15s），启动 KeepAlive 自动续租
2. 在锁前缀下创建 key：PUT /lock/my-resource/<lease-id>
   - 使用 Lease ID 作为 key 后缀保证唯一
3. 获取该前缀下所有 key，按 Revision 排序
4. 若自己的 key 的 Revision 最小 → 获得锁
5. 否则 Watch Revision 比自己小的前一个 key
   - 前一个 key 被删除 → 轮到自己 → 获得锁
```

**Go 实现（使用 etcd clientv3）**：

```go
import (
    "context"
    clientv3 "go.etcd.io/etcd/client/v3"
    "go.etcd.io/etcd/client/v3/concurrency"
)

func acquireEtcdLock(endpoints []string) error {
    cli, err := clientv3.New(clientv3.Config{Endpoints: endpoints})
    if err != nil {
        return err
    }
    defer cli.Close()

    // 创建 Session（内含 Lease + KeepAlive）
    session, err := concurrency.NewSession(cli, concurrency.WithTTL(15))
    if err != nil {
        return err
    }
    defer session.Close()

    // 创建分布式锁
    mutex := concurrency.NewMutex(session, "/lock/my-resource")

    // 加锁（阻塞直到获取）
    if err := mutex.Lock(context.TODO()); err != nil {
        return err
    }

    // mutex.Header().Revision 即为 Fencing Token
    fencingToken := mutex.Header().Revision

    // --- 执行业务逻辑（使用 fencingToken 做写保护）---
    _ = fencingToken

    // 释放锁
    return mutex.Unlock(context.TODO())
}
```

**etcd 锁 vs Redis 锁 vs ZK 锁**：

| 维度 | Redis | ZooKeeper | etcd |
|------|-------|-----------|------|
| 一致性模型 | AP（主从异步复制） | CP（ZAB 协议） | CP（Raft 协议） |
| Fencing Token | 不原生支持 | zxid 可用 | Revision 天然支持 |
| 锁释放 | 过期 / DEL | 临时节点自动删除 | Lease 过期自动删除 |
| 性能 | ★★★★★ | ★★★ | ★★★★ |
| 运维复杂度 | 低 | 高（JVM 调优、ZAB） | 中（Go 原生、K8s 友好） |
| 惊群效应 | N/A | 顺序节点避免 | Watch 前一个 key 避免 |

### 架构评审补充

- 系统设计答案不应只停留在“用了哪些组件”，而要回答“这个组件解决了什么瓶颈、引入了什么新复杂度”。
- 设计评审至少要覆盖：流量模型、一致性目标、故障模型、可观测性、变更与回滚。
- 真正高分答案通常带有故障视角，而不仅是 happy path。

### 初级高频速刷

**1. 高并发、高可用、高一致性分别在说什么？**

高并发关注系统在高请求量下还能不能扛住；高可用关注系统出故障时能不能继续服务；高一致性关注多个副本或多个服务看到的数据是否一致。系统设计面试里，这三个词往往会被同时提到，但它们不是一回事。

**2. 为什么系统里会引入缓存、消息队列、分库分表？**

缓存解决热点读压力，消息队列解决异步解耦和流量削峰，分库分表解决单机容量与吞吐上限。面试时最好能明确说出“组件对应什么瓶颈”，而不是泛泛地说“为了性能”。

**3. 什么是接口幂等？为什么支付类接口必须关注它？**

幂等就是同一个请求执行一次和执行多次，结果对业务来说应一致。支付、扣款、发券这类接口如果不幂等，重试或重复提交就会直接造成资损，所以这是底线问题，不是优化项。

**4. 什么是单点故障？为什么系统设计里要强调故障域隔离？**

单点故障是指一个组件一旦出问题就会拖垮整条链路。故障域隔离的意义在于把问题限制在局部，比如按机房、线程池、连接池、单元拆分资源，避免一个点故障扩散成全站事故。

**5. 为什么灰度发布和快速回滚能力本身就是系统设计的一部分？**

因为再好的架构也不可能保证变更永远正确。灰度发布让问题在小范围暴露，快速回滚让系统把错误成本控制在最小范围内，这是高可用设计的一部分，不只是发布平台的事。

### 中级高频进阶

**6. 为什么降级策略必须在需求和设计阶段就确定？**

因为降级不是单纯技术决策，而是业务取舍。哪些功能可以关、关掉后用户是否还能接受、是否会带来资损，这些都必须提前定义，否则线上故障时只能临时拍脑袋，风险极高。

**7. 隔离、限流、熔断三者分别先解决什么问题？**

隔离优先解决“局部故障扩散”，限流解决“总流量过大”，熔断解决“下游已经异常还在继续打”。如果面试里只会背三者定义，却说不出它们分别在故障链路里拦截哪一层问题，通常还不够成熟。

**8. 为什么幂等接口更好的做法是返回第一次处理结果，而不是只报“重复请求”？**

因为调用方发起重试时，往往真正想知道的是“上一次到底成没成”。如果服务端只告诉它“你重复了”，上游仍然无法确定业务状态，最后只能继续查库或再次补偿。返回第一次结果，才是真正闭环的幂等设计。

**9. 为什么容量规划本身属于系统设计，而不只是运维工作？**

因为限流阈值、分片数量、线程池大小、连接池上限都来自设计阶段的容量假设。没有容量模型的架构图，到了大促或突发流量时很容易直接失效。

**10. 为什么“先做可观测性，再谈优化”是更稳妥的顺序？**

没有关键 SLI、链路追踪和容量指标，优化动作很容易变成拍脑袋。系统设计里先定义可观测性，才能在故障和性能问题发生时快速定位并验证改动有效性。

### 深入问答

#### Q1: 常见的限流算法有哪些？令牌桶和漏桶的区别是什么？

**答**：4种：固定窗口计数器（简单但有临界突刺）、滑动窗口（解决临界问题但内存大）、令牌桶（定速生成令牌，允许突发消费，推荐）、漏桶（定速流出，平滑流量）。核心区别：令牌桶允许突发流量（消费已积累的令牌），漏桶不允许（恒定速率处理）。API限流用令牌桶，保护数据库用漏桶。

#### Q2: 令牌桶的实现原理是什么？如何做到不用定时器生成令牌？

**答**：惰性计算：不真正定时生成令牌，而是在每次获取令牌时计算elapsed*rate得到应生成的令牌数，加上剩余令牌（不超过桶容量），然后判断是否足够。核心字段：capacity（桶容量）、tokens（当前令牌数）、rate（生成速率）、lastTime（上次取令牌时间）。burst通过桶容量实现：桶满时可瞬间消费capacity个请求。

#### Q3: 分布式限流如何实现？

**答**：多实例共享限流配额。推荐Redis+Lua脚本实现：Lua脚本原子性地INCR计数+EXPIRE设过期时间，保证计数准确。也可用Nginx的limit_req_zone做网关层限流。更复杂的场景用Sentinel或自研限流中间件，支持热点参数限流、集群限流。

#### Q4: 熔断器的状态机是怎样的？三种状态如何切换？

**答**：三态：Closed（正常，统计失败率）→ Open（失败率超阈值，拒绝所有请求快速失败）→ Half-Open（超时后允许少量请求试探）→ 试探成功回到Closed / 失败回到Open。核心参数：失败率阈值、统计周期、熔断持续时间、半开状态允许请求数。典型实现：gobreaker、Hystrix、Sentinel。

#### Q5: 熔断和降级有什么区别？各自适用什么场景？

**答**：熔断是对**下游服务**的保护，下游故障率高时切断调用避免级联失败。降级是对**自身服务**的保护，负载过高时主动关闭非核心功能保障核心功能。熔断是被动触发（检测到故障），降级是主动策略（配置中心控制开关）。两者常结合使用：熔断后自动降级返回兜底数据。

#### Q6: 分布式事务有哪些方案？各自的优缺点和适用场景？

**答**：①2PC（两阶段提交）：强一致但同步阻塞+单点故障，不推荐。②TCC（Try-Confirm-Cancel）：性能好无阻塞，但需实现3个接口，业务侵入强，适合金融转账。③SAGA：多个本地事务+补偿事务，长事务友好，补偿逻辑复杂，适合订单流程。④本地消息表（推荐默认）：本地事务写业务+消息表，定时扫描发MQ，可靠简单，有延迟。

#### Q7: CAP定理为什么不能三者兼得？CP和AP的区别是什么？

**答**：网络分区P是客观存在的（不可避免）。分区发生后：写入N1的新数据无法同步到N2，此时N2要么返回旧数据（牺牲C选AP）要么拒绝服务（牺牲A选CP）。CP系统（如ZooKeeper）：分区时少数派拒绝服务，保证一致性。AP系统（如Cassandra）：分区时各节点独立服务，允许不一致，后续修复。CA在分布式中不存在。

#### Q8: Raft协议的Leader选举和日志复制流程是怎样的？

**答**：**选举**：Follower超过election timeout未收到心跳→变为Candidate，term+1，发RequestVote请求多数派投票（每term每节点只投一票），获多数票成为Leader，立即发心跳。timeout随机化防split vote。**日志复制**：客户端写→Leader追加本地Log→并行AppendEntries给Follower→多数派确认→commit→返回成功。Leader通过回退找到与Follower的一致点修复日志。

#### Q9: BASE理论是什么？和CAP的关系是什么？

**答**：BASE是对CAP中AP方案的延伸：Basically Available（基本可用，允许部分功能不可用）、Soft State（软状态，允许中间不一致状态存在）、Eventually Consistent（最终一致性，一段时间后达到一致）。本质是在牺牲强一致性（C）的前提下，通过最终一致性获得更好的可用性（A）。典型场景：电商订单异步扣库存、社交点赞数异步更新。

#### Q10: 一致性Hash的原理是什么？虚拟节点解决什么问题？

**答**：将hash空间组成环(0~2^32-1)，节点和数据都映射到环上，数据顺时针找到第一个节点存储。节点增减只影响相邻节点（迁移数据量小）。虚拟节点解决数据倾斜问题：物理节点少时可能分布不均匀，给每个物理节点创建多个虚拟节点（如150个），均匀分布在环上，保证负载均衡。

#### Q11: 设计一个秒杀系统，从前端到后端的全链路方案是什么？

**答**：①前端：按钮置灰+验证码防刷②CDN/Nginx：静态资源缓存+限流③网关：令牌桶限流（如1000QPS）④Redis预扣库存：DECR原子操作，<0则回滚返回售罄⑤MQ异步下单：通过请求进MQ缓冲，下游慢慢处理⑥订单服务消费MQ：创建订单+数据库扣库存(乐观锁防超卖)+发通知。关键：尽可能在前面拦截流量，真正到DB的请求量极少。

#### Q12: 雪花算法的结构是什么？有什么问题？如何解决？

**答**：64位：1bit符号位 + 41bit毫秒时间戳（可用69年）+ 10bit机器ID（1024台）+ 12bit序列号（每毫秒4096个）。主要问题：**时钟回拨**——机器时间回调导致ID重复。解决：①检测到回拨就等待②记录上次时间戳，回拨时拒绝生成③使用NTP平滑调整④美团Leaf方案：号段模式+双buffer预加载，不依赖时钟。

#### Q13: 如何保证接口幂等性？有哪些具体方案？

**答**：①唯一请求号：客户端生成UUID，服务端用Redis SETNX去重②数据库唯一索引：如订单号UNIQUE，重复插入报错③状态机：订单 created→paid，重复paid操作检查状态不符直接返回④Token机制：先获取Token存Redis，提交时Token校验+删除（原子操作）⑤乐观锁：UPDATE ... WHERE version=N，版本不对则不更新。

#### Q14: 超时和重试策略需要注意什么？指数退避的原理？

**答**：注意事项：①重试前提是操作幂等（非幂等操作不能重试）②设置最大重试次数（防无限重试）③区分超时类型（连接超时短如2s，读超时长如5s）④重试风暴问题（下游故障时大量重试加重压力，需配合熔断）。指数退避：第N次重试等待2^N秒（1s,2s,4s,8s...），加随机抖动(jitter)防止多客户端同时重试。

#### Q15: 如何设计一个短链接服务？核心问题有哪些？

**答**：核心问题：①短码生成：分布式ID+Base62编码（6位可表示568亿），或MD5取前6字节Base62②存储：MySQL（short_code主键, long_url）+ Redis缓存③查询：用户访问短链→Redis查→未命中查DB→301/302重定向④防重复：long_url加hash索引防同一长链生成多个短链⑤架构：CDN→负载均衡→短链服务→Redis→MySQL。高并发优化：302重定向（便于统计点击）、CDN缓存热门短链。

#### Q16: 什么是幂等键（Idempotency Key）？它和“数据库唯一约束”是什么关系？

**答**：幂等键是客户端或调用方为一次业务操作生成的唯一标识，服务端用它判断“这是不是同一请求的重复提交”。数据库唯一约束通常是幂等落地的最后一道防线，但两者不是一回事。幂等键解决的是请求语义去重，唯一约束解决的是存储层不重复写入。成熟设计一般是：接口层校验幂等键，存储层再用唯一索引兜底，并返回第一次处理结果而不是简单报错。

#### Q17: 为什么很多系统“有缓存、有限流、有 MQ”还是会雪崩？

**答**：因为真正的雪崩不是缺某个中间件，而是缺整体的故障隔离设计。常见根因有：超时过长、重试无预算、线程池共用、缓存击穿后直接打 DB、MQ 积压后消费端继续放大下游压力。要防雪崩，必须从入口限流、依赖超时、重试预算、舱壁隔离、熔断降级到数据层兜底形成一整套闭环。面试里如果回答停留在“加 Redis、加 MQ”，深度通常不够。

#### Q18: 什么是 Bulkhead（舱壁隔离）模式？为什么它比简单限流更进一步？

**答**：舱壁隔离是把不同依赖、不同租户或不同业务流量放进独立资源池中，比如独立线程池、连接池、队列和配额。这样某一个依赖被打爆时，只会耗尽自己的那一小块资源，不会拖垮整个服务。限流主要解决“总量过大”，舱壁隔离解决“局部故障扩散”。资深设计里两者通常一起上：先分资源池，再给每个池做限流和超时。

#### Q19: 设计一个高可用系统时，为什么“降级策略”必须在需求阶段就定义，而不是故障时临时拍脑袋？

**答**：因为降级本质上是在做业务取舍，不是单纯的技术动作。哪些功能可以关、关掉后用户是否能接受、数据是否允许延迟、是否会引发资损，这些都必须提前和业务对齐。故障时临时拍脑袋往往会做出错误决策，比如为了保可用把核心一致性也降掉。好的系统设计会在需求阶段明确核心路径、非核心路径和降级开关，并提前演练。

#### Q20: 如果让你审一个架构方案，你会优先追问哪几类问题？

**答**：我通常先问五类：① 流量模型，峰值 QPS、读写比、热点分布是什么；② 一致性要求，哪些场景必须强一致，哪些允许最终一致；③ 故障模型，依赖超时、机房故障、缓存失效时怎么退化；④ 可观测性，如何发现问题、如何定位、是否有关键 SLI；⑤ 变更与扩容，是否支持灰度、回滚、容量预估。这些问题能快速判断方案是不是只画了 happy path。

#### Q21: 系统设计里怎么做容量规划？请给一个可落地的方法。

**答**：容量规划通常分四步：①估算业务峰值（QPS、并发、读写比、峰谷比）②测单实例能力（压测得到稳态吞吐和延迟）③按冗余系数计算实例数（一般预留 30%~50%）④定义扩容触发阈值和回滚策略。关键是把“可承载上限”写进发布和扩容流程，而不是只在文档里写一个理论值。

#### Q22: 如何设计“可回滚”的系统，而不是只设计“可发布”的系统？

**答**：可回滚设计至少包含三层：代码层面向后兼容、数据层面双写/影子字段可回切、流量层面支持灰度撤回。并且要有回滚演练，明确 RTO 内能否真正完成。很多方案看起来能发版，但一旦数据库变更或协议变更不可逆，回滚就会变成口号。

#### Q23: 为什么重试不当会导致级联故障？如何设计重试预算？

**答**：当下游已经慢或错误时，重试会放大请求量，导致线程池和连接池更快耗尽，形成雪崩。重试预算通常包括：最大重试次数、总耗时上限、指数退避和抖动、可重试错误白名单，以及与熔断联动。系统设计里重试必须是“受控恢复机制”，不能是“默认兜底行为”。

#### Q24: 在一致性和可用性冲突时，怎么给业务方一个可执行的决策框架？

**答**：先把场景分级：资损、合规、核心账务走强一致优先；推荐、统计、行为日志可走最终一致优先。再定义每类场景允许的延迟、补偿时限和降级动作。这样业务方讨论的是“明确代价”，而不是抽象争论 CAP。面试里能把技术取舍映射到业务后果，通常会被认为有架构经验。

#### Q25: 系统设计题里，如何给“可用性、成本、研发效率”做可量化 trade-off，而不是口头取舍？

**答**：
建议先把三者转成可量化指标：
1. **可用性**：SLO（如 99.95%）、P99 延迟、RTO/RPO。
2. **成本**：单请求成本、峰值资源成本、运维人力成本。
3. **研发效率**：需求交付周期、变更失败率、回滚耗时。

然后做方案对比表：每个候选方案给出指标预测区间、实现复杂度和风险项，并约定上线后验证窗口。这样讨论的是“数据化权衡”，不是“谁声音大谁对”。

#### Q26: 你主导过的最失败一次架构决策是什么？如果重来一次你会先验证什么？

**答**：
这类问题考的是复盘能力。高质量回答结构通常是：
1. **当时决策**：为什么做这个方案（背景、假设、目标）。
2. **失败表现**：具体失败指标（延迟、故障率、成本、交付效率）。
3. **根因复盘**：是负载假设错、边界条件漏、治理能力不足，还是组织协同问题。
4. **重来验证**：先做最小可验证实验（PoC + 压测 + 故障演练 + 回滚预案）。

面试官最看重的是你能否把”失败经验”沉淀成下一次决策前的验证清单。

### 开放式设计题

**D1：设计一个全球化的多活架构（如电商平台同时在中国和东南亚提供服务），核心挑战是什么？**

**参考思路**：
- 流量路由：GSLB按地域就近接入、用户归属单元（按UserID分片到固定机房）
- 数据同步：核心数据（用户/订单）异步双向同步、冲突解决（Last-Write-Wins/业务规则仲裁）
- 一致性模型：单元内强一致、跨单元最终一致、全局唯一约束（如手机号）走中心化校验
- 容灾切换：单元故障时流量调度到其他单元、数据补偿和一致性修复
- 关键取舍：延迟 vs 一致性，99.9%的请求在单元内闭环是设计目标

**D2：系统上线后发现P99延迟从50ms劣化到500ms，如何系统化排查？**

**参考思路**：
- 分层定位：接入层(Nginx) → 应用层(Go/Java) → 中间件(Redis/MQ) → 数据库(MySQL)，逐层看延迟
- 工具链：分布式追踪(Jaeger)找到慢Span → pprof/async-profiler定位热点函数 → 慢查询日志
- 常见原因：GC停顿、连接池耗尽、慢SQL、下游超时传播、锁竞争
- 系统化方法论：先看P99和P50的差距（差距大说明是尾部少量请求慢）、对比变更记录、灰度回滚验证
