## 1. 推荐系统整体架构

### 1.1 推荐系统在产品中的位置

```
┌─────────────────────────────────────────────────────┐
│                    用户端产品                         │
│                                                     │
│  搜索框 ─→ 搜索系统                                  │
│  信息流 ─→ 推荐系统  ←── 本文重点                     │
│  广告位 ─→ 广告系统（与推荐类似但优化目标为 eCPM）      │
│                                                     │
│  推荐系统核心使命：                                    │
│  在海量内容中，为每个用户找到其最感兴趣的 N 条内容       │
│  并以最优顺序展示                                     │
└─────────────────────────────────────────────────────┘
```

**推荐 vs 搜索**：

| 维度 | 推荐 | 搜索 |
|------|------|------|
| 用户意图 | 隐式（行为推断） | 显式（query 表达） |
| 候选空间 | 全量物品 | query 相关物品 |
| 核心指标 | 时长/CTR/留存 | 相关性/满足率 |
| 个性化程度 | 高（千人千面） | 中（同 query 结果相近） |

## 2. 召回层

### 2.1 多路召回架构

```
┌─────────── 多路召回 ───────────┐
│                                │
│  ① 协同过滤召回 ──→ 200 条     │
│  ② 双塔向量召回 ──→ 500 条     │
│  ③ 热门/趋势召回 ──→ 100 条    │
│  ④ 实时兴趣召回 ──→ 200 条     │
│  ⑤ 图召回 ────────→ 200 条     │
│  ⑥ 地理位置召回 ──→ 100 条     │
│                                │
│  合并去重 ──→ ~1,000 条候选     │
└────────────────────────────────┘
```

**多路召回的设计理念**：
- 每路召回有不同的偏好建模能力（协同过滤关注行为相似，向量召回关注语义相似）
- 多路互补，提高整体覆盖率
- 每路独立，可以并行调用，故障隔离

### 2.2 协同过滤召回

#### ItemCF (物品协同过滤)

```
原理：用户喜欢物品 A，找到与 A 相似的物品推荐

物品相似度计算：
sim(i, j) = |Users(i) ∩ Users(j)| / sqrt(|Users(i)| × |Users(j)|)

改进（IUF — 热门用户降权）：
sim(i, j) = Σ_{u ∈ Users(i) ∩ Users(j)} 1/log(1 + |Items(u)|)
             / sqrt(|Users(i)| × |Users(j)|)
```

#### UserCF (用户协同过滤)

```
原理：找到与目标用户兴趣相似的用户，推荐他们喜欢的物品

适用场景：社交类产品（好友推荐）、用户少物品多的场景
```

**ItemCF vs UserCF**：

| 维度 | ItemCF | UserCF |
|------|--------|--------|
| 适用场景 | 物品少用户多（电商） | 用户少物品多（新闻） |
| 实时性 | 物品相似度稳定 | 需跟踪用户兴趣变化 |
| 长尾覆盖 | 较好 | 一般 |
| 可解释性 | "因为你看了X" | "和你相似的人在看" |
| 冷启动 | 新物品冷启动困难 | 新用户冷启动困难 |

### 2.3 向量化召回（双塔模型）

```
┌──────── 双塔模型 (Two-Tower) ────────┐
│                                      │
│  用户塔          物品塔               │
│  ┌─────┐        ┌─────┐             │
│  │User │        │Item │             │
│  │Features      │Features            │
│  │  ↓  │        │  ↓  │             │
│  │ DNN │        │ DNN │             │
│  │  ↓  │        │  ↓  │             │
│  │User │        │Item │             │
│  │Embed│        │Embed│             │
│  └──┬──┘        └──┬──┘             │
│     │              │                 │
│     └──── cos(u,i) ─┘               │
│         或 inner product             │
│                                      │
│  离线：物品塔 → 向量 → 构建 ANN 索引  │
│  在线：用户塔 → 向量 → ANN 检索 TopK  │
└──────────────────────────────────────┘
```

**ANN (Approximate Nearest Neighbor) 检索**：

| 算法 | 原理 | 适用场景 |
|------|------|---------|
| **Faiss IVF** | 倒排量化聚类 | 大规模、可 GPU 加速 |
| **HNSW** | 分层可导航小世界图 | 高精度、内存充足 |
| **ScaNN** | Google 提出的量化方法 | 高吞吐 |
| **Milvus** | 向量数据库 | 生产级、支持增量更新 |

> 📖 交叉引用：`10-数据库综合` — Milvus 向量数据库的详细讲解

```python
# Faiss 索引构建示例
import faiss
import numpy as np

dim = 128
n_items = 1_000_000
item_vectors = np.random.randn(n_items, dim).astype('float32')

# IVF + PQ 量化索引
nlist = 1024  # 聚类中心数
m = 16        # PQ 子量化器数
quantizer = faiss.IndexFlatIP(dim)
index = faiss.IndexIVFPQ(quantizer, dim, nlist, m, 8)
index.train(item_vectors)
index.add(item_vectors)

# 在线检索
user_vector = np.random.randn(1, dim).astype('float32')
index.nprobe = 64  # 搜索的聚类中心数
scores, ids = index.search(user_vector, k=500)  # 检索 top-500
```

### 2.4 图召回与实时召回

#### 图召回

```
用户-物品二部图上的游走：

User A ──click──→ Item 1 ──click_by──→ User B ──click──→ Item 5
                                                         ↑
                                                   推荐给 User A

算法：
- ItemCF on Graph: 基于共现关系的图上随机游走
- DeepWalk / Node2Vec: 图 Embedding 学习
- GraphSAGE: 归纳式图神经网络
```

#### 实时召回

```
┌──────── 实时兴趣召回 ────────┐
│                              │
│  用户最近 30 分钟的行为序列：  │
│  click(A) → click(B) → C    │
│                              │
│  实时 Trigger → 物品相似表   │
│  A → [A1, A2, A3]           │
│  B → [B1, B2, B3]           │
│  合并 → 实时候选集            │
│                              │
│  特点：捕捉短期兴趣变化       │
│  实现：读取实时行为 + 查表    │
└──────────────────────────────┘
```

### 2.5 召回层工程实践

**召回质量评估**：

| 指标 | 定义 | 目标 |
|------|------|------|
| Recall@K | 用户真正交互的物品中，被召回的比例 | > 80% |
| HitRate@K | 至少召回一个正样本的用户比例 | > 95% |
| 覆盖率 | 被召回的物品占全量物品的比例 | > 30% |
| 新鲜度 | 召回物品中新发布物品的比例 | 视业务而定 |

**工程要点**：
- 各路召回**并行调用**，设超时兜底
- 召回结果**去重** + **过滤**（已曝光/违规/过期）
- 召回候选数量可通过 AB 实验动态调整

## 3. 粗排层

### 3.1 粗排的定位与必要性

```
为什么需要粗排？

召回层：~10,000 条候选，模型简单（向量内积）
精排层：~数百条候选，模型复杂（DeepFM/DIN + 完整特征）

问题：精排模型计算 10,000 条太慢（每条 ~0.5ms × 10,000 = 5s）
方案：在中间加粗排层，用轻量模型将 10,000 筛选到 1,000

粗排 = "精排模型的简化版"
- 使用精排模型的子集特征
- 或使用知识蒸馏训练的轻量模型
```

### 3.2 知识蒸馏与轻量模型

```
┌──────── 知识蒸馏 (Knowledge Distillation) ────────┐
│                                                    │
│  Teacher Model (精排大模型)                         │
│  ├── 输入：完整特征                                 │
│  └── 输出：soft label (概率分布)                    │
│              ↓                                     │
│  Student Model (粗排小模型)                         │
│  ├── 输入：简化特征                                 │
│  ├── 目标：拟合 Teacher 的 soft label               │
│  └── Loss = α × CE(y, p_s) + (1-α) × KL(p_t, p_s) │
│                                                    │
│  效果：Student 学到 Teacher 的"暗知识"              │
│  例如：Teacher 知道物品 A 和 B 的相关性，即使        │
│  Student 没有看到 AB 的交叉特征                     │
└────────────────────────────────────────────────────┘
```

### 3.3 粗排的工程优化

| 优化策略 | 方法 |
|---------|------|
| 特征裁剪 | 只用 user profile + item 基础特征，去掉交叉特征 |
| 模型简化 | 浅层 DNN (2-3层) 或 LR + 简单交叉 |
| 向量预计算 | 物品侧 Embedding 离线算好，在线只算用户侧 |
| 批量推理 | 将 1,000 条 batch 送入 GPU 一次性推理 |
| 量化 | INT8 量化减少计算量 |

## 4. 精排层

> 📖 交叉引用：`AI知识库/05-推荐模型与训练` — 各模型的详细原理与训练技巧

### 4.1 特征工程

```
┌──────── 特征分类 ────────────────────────────────┐
│                                                   │
│  用户侧特征：                                      │
│  ├── 基础画像：性别、年龄、城市、注册天数             │
│  ├── 统计特征：7日/30日 点击率、观看时长              │
│  ├── 行为序列：最近 50 次点击的物品 ID 序列          │
│  └── 实时特征：最近 30min 点击/停留行为              │
│                                                   │
│  物品侧特征：                                      │
│  ├── 基础属性：类目、标签、作者、发布时间             │
│  ├── 统计特征：CTR、完播率、转发率                   │
│  └── 内容特征：标题 Embedding、封面 Embedding        │
│                                                   │
│  交叉特征：                                        │
│  ├── 用户-物品交叉：用户对该类目的偏好度              │
│  ├── 上下文：时间（早/午/晚）、设备、网络             │
│  └── 实时交叉：用户最近是否浏览过同类目               │
└───────────────────────────────────────────────────┘
```

**特征处理**：

| 特征类型 | 处理方式 |
|---------|---------|
| 连续值（年龄、价格） | 分桶离散化 / 标准化 / log 变换 |
| 离散值（城市、类目） | Embedding 查表 |
| 序列特征（行为序列） | DIN Attention / Transformer |
| 多值特征（标签列表） | Sum/Mean Pooling |
| 文本特征 | 预训练模型 → Embedding |

### 4.2 经典模型演进

```
LR → FM → DeepFM → DIN → DIEN → Transformer
│     │     │        │     │       │
│     │     │        │     │       └── 序列建模 + 注意力
│     │     │        │     └── 兴趣演化：GRU + AUGRU
│     │     │        └── 注意力机制：对行为序列加权
│     │     └── 深度 + 二阶交叉
│     └── 二阶特征交叉
└── 线性模型

时间线：2010 ──── 2016 ──── 2017 ──── 2018 ──── 2019 ──── 2020+
```

**各模型核心思想**：

| 模型 | 核心创新 | 适用场景 |
|------|---------|---------|
| **LR** | 线性可解释 | 基线/简单场景 |
| **FM** | 二阶特征交叉 `<vi, vj>·xi·xj` | 稀疏特征交叉 |
| **DeepFM** | FM + DNN 并行 | 低阶+高阶交叉 |
| **Wide&Deep** | Memorization + Generalization | Google Play 推荐 |
| **DCN** | Cross Network 显式高阶交叉 | 自动特征交叉 |
| **DIN** | Attention 加权用户历史行为 | 用户行为序列建模 |
| **DIEN** | GRU 建模兴趣演化 | 兴趣动态变化 |
| **BST** | Transformer 替代 GRU | 长序列建模 |
| **MMOE** | 多专家+门控 | 多目标学习 |

### 4.3 多目标学习

```
┌──────── 多目标模型架构 (MMOE) ────────┐
│                                       │
```

## 5. 重排层

### 5.1 重排的核心目标

精排给出的是每个物品的独立评分，但**最终展示是一个列表**，需要考虑：

1. **多样性**：避免连续展示同类内容（如连续 5 条美食视频）
2. **新鲜度**：给新发布内容曝光机会
3. **公平性**：避免头部物品垄断流量
4. **业务规则**：运营置顶、广告插入、合规过滤
5. **上下文适配**：考虑已展示内容的影响

### 5.2 多样性算法

#### MMR (Maximal Marginal Relevance)

```
MMR = argmax_{d ∈ R\S} [λ × Rel(d) - (1-λ) × max_{d' ∈ S} Sim(d, d')]

其中：
- R: 候选集
- S: 已选集
- Rel(d): 物品 d 的相关性分数（精排分）
- Sim(d, d'): 物品间的相似度
- λ: 相关性与多样性的权衡参数

贪心选择：每次选一个 MMR 最大的物品加入结果
```

#### 类目打散规则

```python
def category_spread(ranked_list, max_consecutive=2):
    """
    规则：同类目物品最多连续出现 max_consecutive 次
    """
    result = []
    waiting = []  # 被延后的物品

    for item in ranked_list:
        recent_cats = [r.category for r in result[-max_consecutive:]]
        if len(recent_cats) == max_consecutive and all(c == item.category for c in recent_cats):
            waiting.append(item)  # 延后
        else:
            result.append(item)

    # 将 waiting 中的物品插入合适位置
    for item in waiting:
        insert_pos = find_best_position(result, item)
        result.insert(insert_pos, item)

    return result
```

```python
# 1. 用户实时 CTR（近 1 小时滑动窗口）
user_clicks = (
    click_stream
    .key_by("user_id")
    .window(SlidingEventTimeWindows.of(Time.hours(1), Time.minutes(5)))
    .aggregate(ClickCountAggregator())
)

user_impressions = (
    impression_stream
    .key_by("user_id")
    .window(SlidingEventTimeWindows.of(Time.hours(1), Time.minutes(5)))
    .aggregate(ImpressionCountAggregator())
)

realtime_ctr = (
    user_clicks.join(user_impressions)
    .where("user_id").equal_to("user_id")
    .window(TumblingEventTimeWindows.of(Time.minutes(5)))
    .apply(lambda clicks, imps: clicks / max(imps, 1))
)

# 2. 物品实时热度
item_hot_score = (
    click_stream
    .key_by("item_id")
    .window(SlidingEventTimeWindows.of(Time.minutes(30), Time.minutes(1)))
    .aggregate(WeightedClickAggregator())  # 近期权重更高
)

# 3. 写入 Redis
realtime_ctr.add_sink(RedisSink("user_realtime_ctr"))
item_hot_score.add_sink(RedisSink("item_hot_score"))
```

## 7. 样本构造与训练流水线

### 7.1 正负样本定义

| 场景 | 正样本 | 负样本 |
|------|--------|--------|
| CTR 预估 | 用户点击 | 曝光未点击 |
| CVR 预估 | 点击后转化 | 点击未转化 |
| 观看时长 | 观看 > 阈值 | 观看 < 阈值 / 回归值 |
| 视频完播 | 播放 > 80% | 播放 < 30% |

### 7.2 样本偏差与纠偏

```
┌──────── 推荐系统的样本偏差 ────────────┐
│                                       │
│  ① 位置偏差 (Position Bias)            │
│     位置靠前的物品获得更多点击           │
│     纠偏：位置作为特征 / IPW 加权        │
│                                       │
│  ② 选择偏差 (Selection Bias)           │
│     只有被推荐的物品才有反馈             │
│     纠偏：随机流量探索 / 反事实学习       │
│                                       │
│  ③ 流行度偏差 (Popularity Bias)        │
│     热门物品获得更多曝光和点击           │
│     纠偏：降采样 / 逆倾向加权            │
│                                       │
│  ④ 曝光偏差 (Exposure Bias)            │
│     用户只能对看到的物品反馈             │
│     纠偏：负样本采样策略                 │
└───────────────────────────────────────┘
```

**负样本采样策略**：

| 策略 | 方法 | 适用 |
|------|------|------|
| 全量负样本 | 曝光未点击全部作负样本 | 精排（候选少） |
| 随机负采样 | 全量物品中随机采样作负样本 | 召回（候选多） |
| Hard Negative | 曝光未点击但排名靠前的 | 提升模型区分度 |
| 热门负采样 | 按热度加权采样 | 去除流行度偏差 |

### 10.4 实时特征与口径一致性

推荐系统线上效果常见的问题，不是模型“不会学”，而是**数据口径和链路时效不稳定**。

| 问题 | 典型现象 | 风险 |
|------|----------|------|
| **特征延迟** | 用户刚点了内容，但推荐结果还没感知到 | 兴趣更新慢，短期体验差 |
| **曝光日志丢失** | 样本只看到点击，看不到完整曝光 | CTR/CVR 标签被污染 |
| **离线/在线定义不一致** | 离线训练值和线上服务值不同 | Training/Serving Skew |
| **回填延迟** | 转化要数小时后才发生 | CVR 标签右删失 |

工程上通常要建设三类保障：
- **时效监控**：监控行为事件从产生到进入特征服务、样本仓的端到端延迟。
- **口径对账**：按字段比对线上请求特征、离线训练特征、回放结果。
- **日志完整性校验**：对曝光、点击、转化日志做数量和主键对账，避免“只丢一小部分日志”长期污染模型。

### 10.5 热点治理与降级策略

推荐系统往往会被热点内容、热点用户或热点特征拖垮，而不是被平均流量拖垮。

```
┌──────── 热点治理思路 ───────────────┐
│                                     │
│  热点内容突发                        │
│  ├── 本地缓存/Redis 多级缓存         │
│  ├── 热 key 分片                     │
│  ├── 请求合并（singleflight）        │
│  └── 热门兜底列表                    │
│                                     │
│  下游异常                            │
│  ├── 召回超时 → 砍掉部分召回路        │
│  ├── 特征回源慢 → 使用默认值/旧值      │
│  ├── 精排服务慢 → 降级粗排分           │
│  └── 全链路预算耗尽 → 返回兜底结果      │
└─────────────────────────────────────┘
```

降级顺序的原则通常是：
1. **先保结果可返回**，再保个性化精度。
2. **先砍高成本模块**，如复杂召回、深度精排、实时交叉特征。
3. **保住护栏指标**，避免因为强行追求 CTR 导致投诉、负反馈或内容生态恶化。

## 11. 面试题自查

以下问题偏系统架构、在线链路、实验平台和稳定性治理；模型结构、训练范式和训练/服务一致性细节请优先联动 `cs/AI知识库/05-推荐模型与训练.md`。

### Q1: 推荐系统的漏斗模型是什么？为什么需要多层？

**答案**：漏斗模型将推荐分为**召回→粗排→精排→重排**四层：
- **召回**：从亿级物品池中快速筛选万级候选（多路并行，简单模型）
- **粗排**：用轻量模型将万级减到千级（知识蒸馏/简化精排）
- **精排**：用复杂深度模型对千级候选精准打分（完整特征+交叉）
- **重排**：考虑多样性/业务规则，生成最终展示列表

多层的原因是**精度与效率的 trade-off**：精排模型太复杂无法对全量物品打分，召回模型太简单无法精准排序。

---

### Q2: 双塔召回在线落地时，索引、新鲜度和延迟怎么治理？

**答案**：
这题在专项里重点不再是“双塔怎么训练”，而是它上线后为什么经常败在工程细节上。

**优点**：
- 用户塔和物品塔解耦，物品向量可离线计算。
- 在线只需计算用户向量 + ANN 检索，延迟低。
- 适合大规模候选池和多路召回编排。

**主要风险**：
- 索引新鲜度不足，新物料进不来或旧向量长期不更新。
- 向量版本漂移，训练产物和检索索引不一致。
- ANN 参数失衡，召回率和延迟互相挤压。

**在线治理要点**：
1. **离线建索引 + 增量更新**：明确全量重建周期和增量写入 SLA。
2. **监控新鲜度**：看向量更新时间、索引更新时间、增量失败率、召回覆盖率。
3. **监控效果漂移**：固定哨兵用户的 TopK 漂移、召回后点击率、热门内容挤压率。
4. **延迟治理**：限制用户塔计算预算、ANN 搜索参数分级、超时后回退其他召回路。

如果面试官追问双塔的损失函数、负采样和训练闭环，优先切到 `cs/AI知识库/05-推荐模型与训练.md`。

---

### Q3: 精排模型从 LR 到 DIN 的演进逻辑？

**答案**：
- **LR**：线性模型，需手工构造交叉特征
- **FM**：自动学习二阶特征交叉 `<vi,vj>·xi·xj`
- **DeepFM**：FM（低阶交叉）+ DNN（高阶交叉）并行
- **DIN**：引入注意力机制，对用户历史行为**按候选物品加权**
- **DIEN**：在 DIN 基础上用 GRU 建模兴趣的**时序演化**

核心趋势：从手工特征 → 自动交叉 → 序列建模 → 注意力机制。

---

### Q4: 推荐系统有哪些样本偏差？怎么处理？

**答案**：
1. **位置偏差**：靠前位置 CTR 天然更高 → 训练时加入位置特征，评估时做无偏校准。
2. **选择偏差**：只有被推荐的物品才有反馈 → 随机流量探索/反事实学习。
3. **流行度偏差**：热门物品曝光多 → 热度重加权、热门降采样。
4. **曝光偏差**：未曝光物品无反馈 → 负样本降权，不把“未曝光”直接当“不喜欢”。
5. **延迟反馈偏差**：转化发生晚 → 回撤观察窗口、标签回填。

---

### Q5: 如何设计推荐系统的 AB 实验？如何判断实验效果？

**答案**：
**实验设计**：
- 分层分桶：按 `hash(user_id + layer_id) % 100` 分流
- 层间正交，层内互斥
- 对照组保持稳定，不和其他大改动混发

**效果判断**：
- 核心指标：CTR、时长、留存、转化等
- 统计显著性：T-test / Delta Method，必要时用 CUPED 降方差
- 护栏指标：负反馈率、投诉率、跳失率不能恶化
- 观察周期：覆盖工作日/周末，避免短期新鲜感误导

---

### Q6: 推荐系统在线服务如何保证低延迟？

**答案**：
1. **并行化**：多路召回并行调用，特征批量获取。
2. **超时机制**：每个 RPC 设超时，超时立即降级。
3. **缓存层级**：Local Cache → Redis → DB。
4. **预计算**：物品向量、热门特征离线预热。
5. **模型优化**：INT8 量化、算子融合、批处理推理。
6. **兜底策略**：最终保证热门列表或规则列表可返回。

低延迟不是单点优化，而是全链路预算管理。

---

### Q7: 从产品和服务架构视角，如何解决推荐系统冷启动问题？

**答案**：
专项里讲冷启动，重点是“系统第一天就得可用”，而不是只讨论模型吃什么特征。

**新用户冷启动**：
- 用注册信息、场景入口、地域时间等上下文先做粗粒度分流。
- 配合兴趣引导页、首登问卷、热门兜底，尽快收集第一批反馈。
- 给新用户保留一定探索比例，但不能把整个首屏变成实验田。

**新物品冷启动**：
- 用标题、封面、标签、类目等内容信号先进入内容召回。
- 配合保底曝光量和快速反馈回流，确保新物料有学习机会。
- 用相似物料 look-alike 和人工运营池补初始流量。

**系统关键点**：
1. 冷启动不是单模型问题，而是产品引导、流量扶持、实验护栏共同参与。
2. 冷启动阶段优先追求“可用 + 可学习”，而不是一开始就追求最优个性化。
3. 必须监控新用户次留、新物料首曝覆盖率、冷启动池命中率和负反馈。

如果面试官继续追问冷启动时模型如何构造内容召回信号、怎么蒸馏样本，优先切到 `cs/AI知识库/05-推荐模型与训练.md`。

### Q8: 特征系统如何设计？离线/近线/实时特征的区别？

**答案**：

| 维度 | 离线特征 | 近线特征 | 实时特征 |
|------|---------|---------|---------|
| 更新周期 | T+1（天级） | 分钟级 | 请求级 |
| 计算方式 | Spark 批处理 | Flink 流计算 | 在线计算 |
| 存储 | HBase/Redis | Redis/Flink State | 内存 |
| 示例 | 用户画像、物品统计 | 实时 CTR、热度 | 上下文、交叉特征 |

特征服务架构通常是统一 Feature Store + 多级缓存，并要求离线/在线特征定义统一，避免 Training/Serving Skew。

---

### Q9: 从在线编排视角，多路召回并行时如何做配额、去重和融合？

**答案**：
这里重点是在线服务怎么稳、怎么快、怎么不把后链路拖死。

1. **基础配额**：各路先设保底和上限，避免单一路把候选集吃满。
2. **统一去重**：按 item 去重，但保留来源路由、原始分和召回原因，供后续排序使用。
3. **动态调权**：按近期效果、覆盖率、重合度、延迟和超时情况动态调整配额。
4. **超时降级**：慢路可以局部超时淘汰，不能为了等一条召回路拖垮整条请求。
5. **容量保护**：粗排/精排预算固定时，优先保“候选多样性”和“高质量覆盖”，而不是盲目堆候选数。

如果面试官继续追问候选集质量、来源特征回写和训练收益，那是模型视角，优先联动 `cs/AI知识库/05-推荐模型与训练.md`。

### Q10: 从系统治理视角，训练数据口径和线上口径不一致时会出现什么问题？如何治理？

**答案**：
这里重点不只是“模型学歪了”，而是整条线上链路会开始给出错误信号。常见表现是离线 AUC 很高，线上 CTR/时长却不涨，或者索引更新后召回覆盖突然下降。

**系统层风险**：
1. 特征服务默认值命中变高，模型看起来还能打分，但已经不再基于真实输入。
2. 离线标签和线上曝光口径不一致，AB 结果会被污染。
3. 召回、粗排、精排使用的 schema 版本不同，问题会在多段链路放大。

**治理方法**：
1. 做请求回放，逐字段校验线上/离线特征。
2. 监控默认值命中率、缺失率、特征分桶分布漂移。
3. 对关键特征做版本管理和发布门禁。
4. 统一特征定义，避免离线和在线各写一套逻辑。
5. 把一致性校验接入发布平台，而不是只靠排障时人工对比。

如果面试官继续追问 Training/Serving Skew 的模型机制和输入分布问题，优先联动 `cs/AI知识库/05-推荐模型与训练.md`。

---

### Q11: 从流量治理视角，Exploration 流量怎么设计，才能既学到新信息又不伤核心体验？

**答案**：
这里重点是“探索怎么上线、怎么控风险”，而不是只讲 Bandit 算法。

1. **设探索配额**：只在有限流量、人群和位置上生效，避免全局放量。
2. **用更聪明的探索策略**：Bandit、UCB、Thompson Sampling 优先，避免纯随机伤害体验。
3. **加护栏指标**：负反馈率、投诉率、跳失率、时长和留存都要同步看。
4. **分层生效**：新用户、新内容池、高不确定区域可以探索更积极，核心稳定流量更保守。
5. **快速回流与回收**：探索反馈要快速回流训练，效果差的探索策略也要能快速下线。

如果面试官追问探索样本如何进入训练集、如何减少噪声污染，那属于模型收益视角，优先联动 `cs/AI知识库/05-推荐模型与训练.md`。

---

### Q12: 离线 AUC 涨了，但线上 CTR 和时长都没涨，你先查什么？

**答案**：
先不急着怀疑模型本身，通常优先查三件事：
1. **流量是否一致**：实验组是否混入了别的策略改动，人群结构是否发生变化。
2. **特征是否一致**：线上默认值命中率、特征缺失率、分桶分布是否异常。
3. **日志是否完整**：曝光、点击、转化日志是否丢失，标签口径是否被污染。

很多“离线涨、线上不涨”并不是模型无效，而是上线链路和训练前提已经变了。

---

### Q13: 实时特征链路延迟突然升高，你先改 watermark，还是先查上游？

**答案**：
先查上游，不要第一反应就调 watermark。
1. 先确认是数据真的迟到，还是消费堆积、反压、checkpoint 抖动。
2. 看 Kafka lag、Flink 反压、算子处理时长、状态大小是否异常。
3. 如果只是靠放宽 watermark 掩盖问题，线上只会变成“更晚地得到错误结果”。
4. 只有确认迟到数据占比结构性变化，才考虑调窗口和 watermark 策略。

面试里更好的回答是先判断“链路故障”还是“业务分布变化”，而不是直接改参数。

---

### Q14: 特征服务 Redis 命中率下降、P99 升高时，推荐链路怎么止血？

**答案**：
1. 先启用本地缓存、旧值兜底、默认值填充，避免请求直接打穿下游。
2. 对热点 key 做分片、请求合并和限流，防止局部热点拖垮全集群。
3. 暂时砍掉高成本实时特征，只保留核心静态特征和近线特征。
4. 同时排查是缓存失效策略、批量回源放大，还是热 key 突发导致的雪崩。

推荐系统里“特征服务慢”通常比“模型慢”更致命，因为它会拖垮整个召回和排序链路。

---

### Q15: 热门内容突然爆发时，如何避免推荐服务被热点拖垮？

**答案**：
1. 对热门内容相关特征做多级缓存和预热，避免集中回源。
2. 用热 key 分片、singleflight、批量获取减少重复请求。
3. 结果侧做热门列表缓存，必要时让部分用户共享短时推荐结果。
4. 排序侧限制单内容最大曝光增速，避免系统被单点热点牵着跑。

热点治理不是只保护缓存，还要防止热点内容把候选集、多样性和系统负载同时拖坏。

---

### Q16: 为什么曝光日志丢失会把后续训练“带歪”？怎么监控？

**答案**：
因为推荐训练很多时候是“曝光为分母，点击/转化为反馈”。如果曝光日志丢了，但点击日志还在：
1. 样本会错误地高估 CTR/CVR。
2. 负样本规模被缩小，模型会变得过度乐观。
3. AB 实验分析也会失真，误以为新策略变好。

监控上至少要做：
- 曝光/点击/转化日志的数量对账。
- 按端、机房、实验桶、内容类型拆分完整率。
- 主键级抽样回放，确认一次展示是否在全链路都可追踪。

---

### Q17: 推荐服务里重试、超时、熔断、限流四者冲突时，优先级怎么定？

**答案**：
通常顺序是：**先超时，再限流，再熔断，最后谨慎重试**。
1. 先用超时把单次调用预算卡住，避免长尾拖死主链路。
2. 用限流保护自己，不让突发流量把资源打满。
3. 当下游持续异常时用熔断快速失败，避免无意义等待。
4. 重试必须最谨慎，只对幂等、短暂抖动、且有剩余预算的调用开放。

在推荐场景里，重试最容易把本来局部抖动的问题放大成全链路雪崩。

---

### Q18: 多样性提升后 CTR 下降，你怎么判断是策略问题还是实验设计问题？

**答案**：
1. 先看实验分流是否稳定，是否有人群、渠道、时段偏差。
2. 再看核心损失发生在哪一段用户和哪类内容，而不是只看整体 CTR。
3. 如果 CTR 降但时长、留存、负反馈改善，说明可能是目标函数切换后的正常 trade-off。
4. 如果连护栏指标也变差，更可能是多样性策略本身过强或重排约束写坏了。

推荐实验不能只看单指标输赢，要看目标函数是否真的和业务目标一致。

---

### Q19: SLO 达标、CTR 也正常，但用户还是抱怨“推荐越来越像”，你会怎么修正指标体系？

**答案**：
这说明现在的指标体系过度偏向效率，缺少“体验结构”指标。
1. 增加多样性、新颖度、覆盖率、创作者分散度等指标。
2. 拆分短期点击指标和中长期留存、负反馈、审美疲劳指标。
3. 对高活跃用户单独建新鲜度指标，因为这类用户最容易先感知“越来越像”。
4. 将“投诉/不感兴趣/跳出”纳入护栏，而不是只盯 CTR。

一个成熟的推荐指标体系必须能解释“数字正常但体验在变差”的情况。

---

### Q20: 系统设计题里，推荐系统怎么量化“效果、可用性、成本”的 trade-off？

**答案**：
可以把问题拆成三张表：
1. **效果**：CTR、时长、留存、转化提升多少。
2. **可用性**：P99、超时率、降级率、故障恢复时间是多少。
3. **成本**：GPU/CPU/Redis/Kafka 资源增加多少，人力维护复杂度增加多少。

然后要求每次方案评审都回答三个量化问题：
- 为了提升 1% CTR，增加了多少毫秒延迟和多少机器成本？
- 如果精排挂了，是否还能在预算内返回可接受结果？
- 这套方案上线后，回滚和排障复杂度是否显著上升？

面试里如果你能把 trade-off 讲成量化约束，而不是口头“我们权衡一下”，层次会明显更高。

---

> 📖 **总结**：推荐系统是“模型、数据、系统、实验”四件事同时成立才会稳定产出的工程。真正难的不是把模型做复杂，而是把召回、特征、日志、实验和降级体系串成闭环。
>
> **推荐阅读**：*"深度学习推荐系统"* (王喆)、*"Recommender Systems: The Textbook"* (Aggarwal)
