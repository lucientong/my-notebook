## 1. 测试金字塔

```
        ┌────────┐
        │ E2E测试 │  少量（耗时、脆弱）
        ├────────┤
        │集成测试 │  中等（验证模块间交互）
        ├────────┤
        │单元测试 │  大量（快速、稳定）
        └────────┘
```

| 测试类型 | 数量占比 | 执行速度 | 维护成本 | 反馈速度 |
|----------|----------|----------|----------|----------|
| **单元测试** | 70% | 毫秒级 | 低 | 快 |
| **集成测试** | 20% | 秒级 | 中 | 中 |
| **E2E测试** | 10% | 分钟级 | 高 | 慢 |

## 2. 测试理论基础

### 2.1 测试覆盖率原理

#### 2.1.1 覆盖率类型

| 覆盖率类型 | 定义 | 计算公式 | 目标值 |
|-----------|------|----------|--------|
| **语句覆盖率** | 执行的代码行 / 总代码行 | Statements / Total | 80%+ |
| **分支覆盖率** | 执行的分支 / 总分支数 | Branches / Total | 70%+ |
| **函数覆盖率** | 调用的函数 / 总函数数 | Functions / Total | 90%+ |
| **行覆盖率** | 执行的有效行 / 总有效行 | Lines / Total | 80%+ |

**示例**：
```javascript
function calculateDiscount(price, vipLevel) {
    if (vipLevel === 'gold') {        // 分支1
        return price * 0.7;
    } else if (vipLevel === 'silver') { // 分支2
        return price * 0.85;
    } else {                           // 分支3
        return price;
    }
}

// 测试用例1：只测试gold
test('gold discount', () => {
    expect(calculateDiscount(100, 'gold')).toBe(70);
});

// 覆盖率统计：
// - 语句覆盖率：2/6 = 33%（只执行了if和第一个return）
// - 分支覆盖率：1/3 = 33%（只测试了gold分支）
// - 函数覆盖率：1/1 = 100%（函数被调用了）

// 完整测试：
test('all discounts', () => {
    expect(calculateDiscount(100, 'gold')).toBe(70);
    expect(calculateDiscount(100, 'silver')).toBe(85);
    expect(calculateDiscount(100, 'bronze')).toBe(100);
});
// 覆盖率：100%
```

### 2.2 Mock原理与依赖注入

#### 2.2.1 为什么需要Mock？

**问题**：测试时依赖外部服务
```javascript
// 原始代码：依赖真实API
async function getUserProfile(userId) {
    const response = await fetch(`https://api.example.com/users/${userId}`);
    return response.json();
}

// 问题：
// 1. 需要网络连接（慢、不稳定）
// 2. 可能收费（调用API收费）
// 3. 数据不可控（无法测试边界情况）
// 4. 环境依赖（API可能不可用）
```

**解决**：使用Mock
```javascript
// 方案1：依赖注入
async function getUserProfile(userId, fetcher = fetch) {
    const response = await fetcher(`https://api.example.com/users/${userId}`);
    return response.json();
}

// 测试：注入mock函数
test('getUserProfile', async () => {
    const mockFetch = jest.fn().mockResolvedValue({
        json: async () => ({ id: 1, name: 'Alice' })
    });
    
    const profile = await getUserProfile(1, mockFetch);
    expect(profile.name).toBe('Alice');
    expect(mockFetch).toHaveBeenCalledWith('https://api.example.com/users/1');
});
```

#### 2.2.2 Mock实现原理

**简化版Mock实现**：
```javascript
class MockFunction {
    constructor() {
        this.calls = [];        // 记录所有调用
        this.returnValue = undefined;
    }
    
    // 调用mock函数
    call(...args) {
        this.calls.push(args);
        return this.returnValue;
    }
    
    // 设置返回值
    mockReturnValue(value) {
        this.returnValue = value;
        return this;
    }
    
    // 断言：验证调用次数
    toHaveBeenCalledTimes(count) {
        if (this.calls.length !== count) {
            throw new Error(`Expected ${count} calls, but got ${this.calls.length}`);
        }
    }
    
    // 断言：验证调用参数
    toHaveBeenCalledWith(...args) {
        const match = this.calls.some(call => 
            JSON.stringify(call) === JSON.stringify(args)
        );
        if (!match) {
            throw new Error(`Expected call with ${JSON.stringify(args)}`);
        }
    }
}

// 使用
const mockFn = new MockFunction();
mockFn.mockReturnValue(42);

console.log(mockFn.call(1, 2));  // 42
mockFn.toHaveBeenCalledTimes(1);
mockFn.toHaveBeenCalledWith(1, 2);
```

**Jest Mock实现（简化）**：
```javascript
function jest.fn(implementation) {
    const mockFn = function(...args) {
        // 记录调用
        mockFn.mock.calls.push(args);
        mockFn.mock.instances.push(this);
        
        // 执行实现
        if (implementation) {
            return implementation(...args);
        }
        return mockFn.mock.results[mockFn.mock.calls.length - 1];
    };
    
    // 添加mock属性
    mockFn.mock = {
        calls: [],
        instances: [],
        results: []
    };
    
    // 添加mock方法
    mockFn.mockReturnValue = (value) => {
        mockFn.mock.results.push(value);
        return mockFn;
    };
    
    mockFn.mockResolvedValue = (value) => {
        mockFn.mock.results.push(Promise.resolve(value));
        return mockFn;
    };
    
    return mockFn;
}
```

```go
}

type UserService struct {
    client HttpClient  // 依赖接口而非具体实现
}

func (s *UserService) GetUser(id int) (*User, error) {

}
```

### 2.3 测试隔离与并发

#### 2.3.1 测试隔离原理

**问题**：测试间相互影响
```javascript
let counter = 0;

test('test 1', () => {
    counter++;
    expect(counter).toBe(1);  // ✅ 通过
});

test('test 2', () => {
    counter++;
    expect(counter).toBe(1);  // ❌ 失败（counter=2）
});
```

**解决**：beforeEach重置状态
```javascript
let counter;

beforeEach(() => {
    counter = 0;  // 每个测试前重置
});

test('test 1', () => {
    counter++;
    expect(counter).toBe(1);  // ✅ 通过
});

test('test 2', () => {
    counter++;
    expect(counter).toBe(1);  // ✅ 通过
});
```

**数据库隔离**：事务回滚
```python
import pytest

@pytest.fixture
def db_session():
    # 开启事务
    session = db.create_session()
    session.begin()
    
    yield session
    
    # 测试结束后回滚
    session.rollback()
    session.close()

def test_create_user(db_session):
    user = User(name='Alice')
    db_session.add(user)
    db_session.commit()
    
    # 查询验证
    assert db_session.query(User).filter_by(name='Alice').first() is not None
    
    # 测试结束后自动回滚，数据库不会有新数据
```

#### 2.3.2 并发测试原理

**Go并发测试**：
```go
func TestConcurrentIncrement(t *testing.T) {
    var counter int64 = 0
    var wg sync.WaitGroup
    
    // 启动100个goroutine并发增加
    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            atomic.AddInt64(&counter, 1)  // 原子操作
        }()
    }
    
    wg.Wait()
    
    // 验证结果
    if counter != 100 {
        t.Errorf("Expected 100, got %d", counter)
    }
}

// 并行测试（Go test -parallel）

    // 测试逻辑
}

    // 测试逻辑
}
```

### 2.4 测试替身（Test Double）

#### 2.4.1 五种测试替身

| 类型 | 说明 | 使用场景 |
|------|------|----------|
| **Dummy** | 占位对象，不会被使用 | 填充参数列表 |
| **Stub** | 返回预设值 | 提供测试数据 |
| **Spy** | 记录调用信息 | 验证调用行为 |
| **Mock** | 预设期望，验证调用 | 验证交互 |
| **Fake** | 简化实现 | 内存数据库 |

**示例**：
```javascript
// 1. Dummy：占位
function sendEmail(from, to, subject, body, logger) {
    // logger只是占位，不会被使用
}

test('sendEmail', () => {
    sendEmail('a@ex.com', 'b@ex.com', 'Test', 'Body', null);  // Dummy
});

// 2. Stub：返回预设值
const stubUserService = {
    getUser: () => ({ id: 1, name: 'Alice' })  // 固定返回
};

// 3. Spy：记录调用
const spyLogger = {
    calls: [],
    log: function(msg) {
        this.calls.push(msg);
    }
};

sendEmail('a@ex.com', 'b@ex.com', 'Test', 'Body', spyLogger);
expect(spyLogger.calls).toContain('Email sent');

// 4. Mock：预设期望
const mockEmailService = jest.fn();
mockEmailService.mockReturnValue(true);

sendEmailWithService(mockEmailService);
expect(mockEmailService).toHaveBeenCalledWith({
    from: 'a@ex.com',
    to: 'b@ex.com'
});

// 5. Fake：简化实现
class FakeDatabase {
    constructor() {
        this.data = {};
    }
    
    save(key, value) {
        this.data[key] = value;  // 内存存储，不写磁盘
    }
    
    get(key) {
        return this.data[key];
    }
}
```

### 2.5 测试驱动开发（TDD）

#### 2.5.1 TDD流程

```
1. 红灯 → 写一个失败的测试
   ↓
2. 绿灯 → 写最少代码让测试通过
   ↓
3. 重构 → 优化代码，保持测试通过
   ↓
   回到步骤1
```

**示例**：实现栈（Stack）
```javascript
// 步骤1：红灯 - 写测试（此时Stack还不存在）
describe('Stack', () => {
    test('should be empty when created', () => {
        const stack = new Stack();
        expect(stack.isEmpty()).toBe(true);
    });
});

// 步骤2：绿灯 - 最少实现
class Stack {
    isEmpty() {
        return true;  // 硬编码
    }
}

// 步骤3：红灯 - 添加新测试
test('should not be empty after push', () => {
    const stack = new Stack();
    stack.push(1);
    expect(stack.isEmpty()).toBe(false);
});

// 步骤4：绿灯 - 实现push
class Stack {
    constructor() {
        this.items = [];
    }
    
    isEmpty() {
        return this.items.length === 0;
    }
    
    push(item) {
        this.items.push(item);
    }
}

// 步骤5：重构 - 优化代码
// 继续添加pop、peek等测试...
```

**TDD优势**：
- ✅ 代码可测试性高（自然产生可测试的设计）
- ✅ 覆盖率高（先写测试）
- ✅ 重构信心（测试保护）
- ✅ 文档作用（测试即文档）

## 8. 契约测试

### 8.1 什么是契约测试

契约测试（Contract Testing）是验证服务间接口契约的测试方法，确保消费者（Consumer）和提供者（Provider）对API的理解一致。

```
┌─────────────┐                ┌─────────────┐
│  Consumer   │  ← Contract →  │  Provider   │
│ (前端/调用方) │               │ (后端/被调方) │
└─────────────┘                └─────────────┘

契约内容：
- 请求格式（method, path, headers, body）
- 响应格式（status, headers, body schema）
- 错误处理约定
```

**为什么需要契约测试？**

| 问题 | 集成测试 | 契约测试 |
|-----|---------|---------|
| 运行速度 | 慢（需要启动所有服务） | 快（独立运行） |
| 环境依赖 | 高（需要完整环境） | 低（只需契约文件） |
| 故障定位 | 难（哪个服务出问题？） | 易（明确指出不符合契约） |
| 并行开发 | 难（等待依赖服务） | 易（基于契约独立开发） |

### 8.2 Pact契约测试

**Consumer端测试**：

## 9. 变异测试

### 9.1 什么是变异测试

变异测试（Mutation Testing）是通过在代码中引入故意的bug（变异体），来评估测试套件质量的方法。

```
原始代码：
function isAdult(age) {
    return age >= 18;  // 原始逻辑
}

变异体：
function isAdult(age) {
    return age > 18;   // 变异1：>= 改为 >
    return age <= 18;  // 变异2：>= 改为 <=
    return age == 18;  // 变异3：>= 改为 ==
    return true;       // 变异4：直接返回true
}

测试质量评估：
- 如果测试能检测到变异（失败）→ 变异被"杀死"
- 如果测试无法检测到变异（仍然通过）→ 变异"存活"

变异得分 = 被杀死的变异数 / 总变异数
```

### 9.2 常见变异算子

| 变异算子 | 描述 | 示例 |
|---------|------|------|
| **算术替换** | 替换算术运算符 | `+` → `-`, `*` → `/` |
| **关系替换** | 替换比较运算符 | `>` → `>=`, `==` → `!=` |
| **逻辑替换** | 替换逻辑运算符 | `&&` → `\|\|`, `!` → 删除 |
| **常量替换** | 修改常量值 | `0` → `1`, `true` → `false` |
| **返回值** | 修改返回值 | `return x` → `return 0` |
| **删除语句** | 删除代码行 | 删除赋值语句 |

### 9.4 变异测试最佳实践

```
1. 增量变异测试
   - 只对修改的代码进行变异测试
   - 避免全量变异（太慢）

2. 设置阈值
   - 变异得分目标：70-80%
   - 100%不现实（等价变异体问题）

3. 关注存活变异
   - 分析存活变异体，补充测试
   - 可能发现测试盲点

   - 作为质量门禁
```

## 13. 面试题自查

### Q1: 测试金字塔是什么？为什么要这样分层？

**答案**：
测试金字塔是Mike Cohn提出的测试分层策略，从底到顶分为：
- **单元测试（70%）**：测试单个函数/方法，毫秒级执行，覆盖最多边界情况
- **集成测试（20%）**：测试模块间交互（API+DB），验证接口契约
- **E2E测试（10%）**：测试完整用户流程，验证端到端功能

原因：
1. **成本效益**：单元测试执行快、维护成本低，问题定位精准
2. **反馈速度**：开发时快速获得反馈，减少返工
3. **稳定性**：底层测试更稳定，顶层测试容易受环境影响

### Q2: 什么是测试替身？Stub和Mock的区别是什么？

**答案**：
测试替身（Test Double）是替代真实依赖的对象，包括5种类型：

| 类型 | 作用 | 示例场景 |
|-----|------|---------|
| Dummy | 占位，不会被调用 | 填充参数列表 |
| Stub | 返回预设值 | 固定返回用户数据 |
| Spy | 记录调用信息 | 验证日志方法被调用 |
| Mock | 预设期望+验证调用 | 验证邮件服务调用参数 |
| Fake | 简化实现 | 内存数据库替代真实DB |

**Stub vs Mock**：
- **Stub**：状态验证，只关心返回值，不关心调用过程
- **Mock**：行为验证，关心是否以正确方式调用

```javascript
// Stub：只关心返回值
const stubService = { getUser: () => ({ name: 'Alice' }) };

// Mock：验证调用行为
const mockService = jest.fn().mockReturnValue({ name: 'Alice' });
mockService(1);
expect(mockService).toHaveBeenCalledWith(1);  // 验证调用参数
```

### Q3: 代码覆盖率100%就意味着没有Bug吗？

**答案**：
不是。覆盖率100%只表示所有代码行都被执行过，但不能保证：

1. **逻辑正确性**：代码被执行≠逻辑正确
   ```javascript
   function isAdult(age) { return age > 18; }  // 应该是>=18
   test('adult', () => {
       expect(isAdult(20)).toBe(true);  // 覆盖率100%，但边界错误
   });
   ```

2. **边界情况**：可能遗漏边界测试
3. **组合场景**：A和B单独正确，A+B组合可能有问题
4. **非功能需求**：性能、并发、安全等无法通过覆盖率保证

**覆盖率的正确用法**：
- 作为下限指标（如>80%），不是目标
- 关注增量覆盖率（新代码覆盖率）
- 结合变异测试评估测试质量

### Q4: 什么是契约测试？它解决什么问题？

**答案**：
契约测试（Contract Testing）验证服务间接口约定的一致性。

**解决的问题**：
1. **微服务集成测试难**：启动所有服务成本高
2. **接口变更风险**：Provider改动可能破坏Consumer
3. **并行开发阻塞**：Consumer等待Provider接口就绪

**工作流程**（Consumer-Driven Contract）：
1. Consumer定义期望的API行为（契约）
2. Consumer本地测试通过后，发布契约到Pact Broker
3. Provider拉取契约并验证自己的实现
4. CI集成can-i-deploy检查，确保兼容后才能部署

**优势**：
- 快速反馈（无需启动完整环境）
- 明确责任（契约不符合时，明确指出问题方）
- 支持并行开发（基于契约独立开发）

### Q6: 什么是变异测试？它如何帮助提升测试质量？

**答案**：
变异测试通过在代码中引入故意的bug（变异体），检测测试套件能否发现这些bug。

**变异算子示例**：
- 算术替换：`a + b` → `a - b`
- 关系替换：`x >= 18` → `x > 18`
- 逻辑替换：`a && b` → `a || b`
- 返回值：`return x` → `return 0`

**评估指标**：
```
变异得分 = 被杀死的变异数 / 总变异数

得分70%表示：30%的变异体没被测试检测到
→ 这30%对应的代码路径测试不充分
```

**应用场景**：
- 评估测试套件质量（比覆盖率更有效）
- 发现测试盲点
- 作为PR质量门禁

### Q7: TDD的流程是什么？它有什么优缺点？

**答案**：
**TDD流程**（红-绿-重构）：
1. **红灯**：写一个失败的测试（功能还不存在）
2. **绿灯**：写最少的代码让测试通过
3. **重构**：优化代码，保持测试通过
4. 重复循环

**优点**：
- 代码天然可测试（先有测试）
- 覆盖率高
- 重构有信心（测试保护）
- 减少过度设计（只实现测试需要的）

**缺点**：
- 学习曲线陡峭
- 初期开发速度慢
- 不适合探索性开发（需求不明确时）
- 需要团队整体采用才有效

### Q8: E2E测试为什么容易脆弱（Flaky）？如何解决？

**答案**：
**脆弱原因**：
1. **异步等待**：页面加载、动画、API响应时间不确定
2. **环境依赖**：第三方服务不稳定、网络抖动
3. **数据依赖**：测试数据被其他测试或手动操作污染
4. **时序问题**：测试执行顺序影响结果
5. **UI变化**：元素定位器失效

**解决方案**：

```javascript
// 1. 使用智能等待（不用硬编码sleep）
await page.waitForSelector('[data-testid="user-list"]');
await page.waitForLoadState('networkidle');

// ❌ 脆弱
page.locator('.btn-primary');
// ✅ 稳定
page.locator('[data-testid="submit-button"]');

// 3. 重试机制
test('flaky test', async () => {
    await expect(async () => {
        await page.click('button');
        await expect(page.locator('.result')).toBeVisible();
    }).toPass({ timeout: 10000 });
});

// 4. 测试数据隔离
beforeEach(async () => {
```

### Q11: 什么是Flaky Test？如何检测和修复？

**答案**：
**Flaky Test**：同一代码，相同环境，测试结果不稳定（时而通过，时而失败）。

**常见原因**：
1. **竞态条件**：并发访问共享资源
2. **时间依赖**：`Date.now()`、定时器
3. **随机顺序**：测试执行顺序影响结果
4. **资源泄漏**：前一个测试未清理资源
5. **异步等待不足**：未等待异步操作完成

**检测方法**：
```bash
# 多次运行检测
for i in {1..10}; do npm test; done

# Jest repeat模式
jest --repeat=10

# CI标记flaky测试
# 记录历史通过率，<100%的标记为flaky
```

**修复策略**：
```javascript
// 1. 固定时间
jest.useFakeTimers();
jest.setSystemTime(new Date('2024-01-01'));

// 2. 固定随机数
jest.spyOn(Math, 'random').mockReturnValue(0.5);

// 3. 隔离测试数据
beforeEach(() => {
    testUser = createTestUser();  // 每个测试独立数据
});

// 4. 正确等待异步
await waitFor(() => {
    expect(screen.getByText('Loaded')).toBeInTheDocument();
});
```

### Q12: Go语言中Table-Driven Test的优势是什么？

**答案**：
```go
func TestCalculate(t *testing.T) {
    tests := []struct {
        name     string
        input    int
        expected int
        wantErr  bool
    }{
        {"positive", 10, 20, false},
        {"negative", -5, -10, false},
        {"zero", 0, 0, false},
        {"overflow", math.MaxInt64, 0, true},
    }
    
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got, err := Calculate(tt.input)
            if (err != nil) != tt.wantErr {
                t.Errorf("unexpected error: %v", err)
            }
            if got != tt.expected {
                t.Errorf("Calculate(%d) = %d; want %d", tt.input, got, tt.expected)
            }
        })
    }
}
```

**优势**：
1. **可读性**：一目了然所有测试场景
2. **易扩展**：添加场景只需加一行
3. **减少重复**：测试逻辑只写一次
4. **并行友好**：每个case可用`t.Parallel()`
5. **名称清晰**：失败时显示具体case名

2. **独立性**：测试间数据不共享
3. **可重复**：每次运行结果一致
4. **易维护**：集中管理，避免硬编码

### Q14: 什么是测试左移（Shift-Left Testing）？

**答案**：
**测试左移**：将测试活动从开发后期移到开发早期。

```
传统瀑布模式（测试在后期）：
需求 → 设计 → 编码 → [测试] → 部署
                      ↑ 问题发现晚，修复成本高

测试左移：
[测试] → 需求 → [测试] → 设计 → [测试] → 编码 → [测试] → 部署
 ↑需求评审    ↑设计评审    ↑TDD/单元测试
```

**具体实践**：
| 阶段 | 测试活动 |
|-----|---------|
| 需求 | 需求评审、验收标准定义（Given-When-Then） |
| 设计 | API契约评审、架构评审 |
| 编码 | TDD、Code Review、静态分析 |
| 提交 | Pre-commit hooks、单元测试 |
| PR | CI测试、覆盖率检查、契约验证 |

**优势**：
- 问题发现越早，修复成本越低
- 减少返工
- 提高质量意识

### Q15: 如何衡量测试的ROI（投入产出比）？

**答案**：
**衡量指标**：

| 指标 | 计算方式 | 意义 |
|-----|---------|------|
| **缺陷逃逸率** | 生产Bug数 / 总Bug数 | 测试有效性 |
| **测试执行时间** | CI测试耗时 | 反馈速度 |
| **测试维护成本** | 修改测试/修改代码 | 测试稳定性 |
| **MTTR** | 问题发现到修复的时间 | 问题定位效率 |
| **代码变更信心** | 重构时的回归率 | 测试保护作用 |

**ROI计算**：
```
测试投入成本 = 编写时间 + 维护时间 + 执行时间
测试收益 = 早期发现Bug节省的成本 + 重构信心 + 文档价值

生产Bug修复成本 ≈ 开发阶段修复成本 × 10~100倍
```

**优化策略**：
1. **减少E2E比例**：执行慢、维护贵
2. **提升单元测试覆盖**：快速、便宜
3. **消除Flaky测试**：减少调查时间
4. **并行化执行**：缩短CI时间
5. **测试分层合理化**：遵循测试金字塔

## 总结

**测试策略**：

1. **单元测试为主**：70%，快速反馈
2. **集成测试为辅**：20%，验证集成点
3. **E2E测试为补充**：10%，关键流程

**测试工具**：
- **前端**：Jest、React Testing Library、Playwright
- **Go**：testing、gomock、httptest
- **Python**：pytest、unittest.mock
- **Java**：JUnit 5、Mockito、Spring Boot Test

**测试原则**：
- **快速**：单元测试毫秒级
- **稳定**：不依赖外部服务（mock）
- **独立**：测试间互不影响
- **可重复**：多次运行结果一致
