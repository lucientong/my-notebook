## 1. 测试金字塔

```
        ┌────────┐
        │ E2E测试 │  少量（耗时、脆弱）
        ├────────┤
        │集成测试 │  中等（验证模块间交互）
        ├────────┤
        │单元测试 │  大量（快速、稳定）
        └────────┘
```

| 测试类型 | 数量占比 | 执行速度 | 维护成本 | 反馈速度 |
|----------|----------|----------|----------|----------|
| **单元测试** | 70% | 毫秒级 | 低 | 快 |
| **集成测试** | 20% | 秒级 | 中 | 中 |
| **E2E测试** | 10% | 分钟级 | 高 | 慢 |

### 1.1 测试策略

**单元测试**：测试单个函数/方法
```javascript
// 测试纯函数

    return a + b;
}

test('add function', () => {
    expect(add(1, 2)).toBe(3);
```

```javascript

    const response = await request(app)

        .send({ name: 'Alice', email: 'alice@example.com' });

    const user = await db.query('SELECT * FROM users WHERE email = ?', ['alice@example.com']);

```


## 2. 测试理论基础

### 2.1 测试覆盖率原理

#### 2.1.1 覆盖率类型

| 覆盖率类型 | 定义 | 计算公式 | 目标值 |
|-----------|------|----------|--------|
| **语句覆盖率** | 执行的代码行 / 总代码行 | Statements / Total | 80%+ |
| **分支覆盖率** | 执行的分支 / 总分支数 | Branches / Total | 70%+ |
| **函数覆盖率** | 调用的函数 / 总函数数 | Functions / Total | 90%+ |
| **行覆盖率** | 执行的有效行 / 总有效行 | Lines / Total | 80%+ |

**示例**：
```javascript
function calculateDiscount(price, vipLevel) {
    if (vipLevel === 'gold') {        // 分支1
        return price * 0.7;
    } else if (vipLevel === 'silver') { // 分支2
        return price * 0.85;
    } else {                           // 分支3
        return price;
    }
}

// 测试用例1：只测试gold
test('gold discount', () => {
    expect(calculateDiscount(100, 'gold')).toBe(70);
});

// 覆盖率统计：
// - 语句覆盖率：2/6 = 33%（只执行了if和第一个return）
// - 分支覆盖率：1/3 = 33%（只测试了gold分支）
// - 函数覆盖率：1/1 = 100%（函数被调用了）

// 完整测试：
test('all discounts', () => {
    expect(calculateDiscount(100, 'gold')).toBe(70);
    expect(calculateDiscount(100, 'silver')).toBe(85);
    expect(calculateDiscount(100, 'bronze')).toBe(100);
});
// 覆盖率：100%
```

### 2.2 Mock原理与依赖注入

#### 2.2.1 为什么需要Mock？

**问题**：测试时依赖外部服务
```javascript
// 原始代码：依赖真实API
async function getUserProfile(userId) {
    const response = await fetch(`https://api.example.com/users/${userId}`);
    return response.json();
}

// 问题：
// 1. 需要网络连接（慢、不稳定）
// 2. 可能收费（调用API收费）
// 3. 数据不可控（无法测试边界情况）
// 4. 环境依赖（API可能不可用）
```

**解决**：使用Mock
```javascript
// 方案1：依赖注入
async function getUserProfile(userId, fetcher = fetch) {
    const response = await fetcher(`https://api.example.com/users/${userId}`);
    return response.json();
}

// 测试：注入mock函数
test('getUserProfile', async () => {
    const mockFetch = jest.fn().mockResolvedValue({
        json: async () => ({ id: 1, name: 'Alice' })
    });
    
    const profile = await getUserProfile(1, mockFetch);
    expect(profile.name).toBe('Alice');
    expect(mockFetch).toHaveBeenCalledWith('https://api.example.com/users/1');
});
```

#### 2.2.2 Mock实现原理

**简化版Mock实现**：
```javascript
class MockFunction {
    constructor() {
        this.calls = [];        // 记录所有调用
        this.returnValue = undefined;
    }
    
    // 调用mock函数
    call(...args) {
        this.calls.push(args);
        return this.returnValue;
    }
    
    // 设置返回值
    mockReturnValue(value) {
        this.returnValue = value;
        return this;
    }
    
    // 断言：验证调用次数
    toHaveBeenCalledTimes(count) {
        if (this.calls.length !== count) {
            throw new Error(`Expected ${count} calls, but got ${this.calls.length}`);
        }
    }
    
    // 断言：验证调用参数
    toHaveBeenCalledWith(...args) {
        const match = this.calls.some(call => 
            JSON.stringify(call) === JSON.stringify(args)
        );
        if (!match) {
            throw new Error(`Expected call with ${JSON.stringify(args)}`);
        }
    }
}

// 使用
const mockFn = new MockFunction();
mockFn.mockReturnValue(42);

console.log(mockFn.call(1, 2));  // 42
mockFn.toHaveBeenCalledTimes(1);
mockFn.toHaveBeenCalledWith(1, 2);
```

**Jest Mock实现（简化）**：
```javascript
function jest.fn(implementation) {
    const mockFn = function(...args) {
        // 记录调用
        mockFn.mock.calls.push(args);
        mockFn.mock.instances.push(this);
        
        // 执行实现
        if (implementation) {
            return implementation(...args);
        }
        return mockFn.mock.results[mockFn.mock.calls.length - 1];
    };
    
    // 添加mock属性
    mockFn.mock = {
        calls: [],
        instances: [],
        results: []
    };
    
    // 添加mock方法
    mockFn.mockReturnValue = (value) => {
        mockFn.mock.results.push(value);
        return mockFn;
    };
    
    mockFn.mockResolvedValue = (value) => {
        mockFn.mock.results.push(Promise.resolve(value));
        return mockFn;
    };
    
    return mockFn;
}
```

```javascript
class UserService {
```

**接口注入**（Go语言常用）：
```go
// 定义接口
type HttpClient interface {
    Get(url string) (*Response, error)
}

type UserService struct {
    client HttpClient  // 依赖接口而非具体实现
}

func (s *UserService) GetUser(id int) (*User, error) {
    resp, err := s.client.Get(fmt.Sprintf("/users/%d", id))
    // ...
}

// 测试：mock接口
type MockHttpClient struct {
    GetFunc func(url string) (*Response, error)
}

func (m *MockHttpClient) Get(url string) (*Response, error) {
    return m.GetFunc(url)
}

func TestUserService_GetUser(t *testing.T) {
```

### 2.3 测试隔离与并发

#### 2.3.1 测试隔离原理

**问题**：测试间相互影响
```javascript
let counter = 0;

test('test 1', () => {
    counter++;
    expect(counter).toBe(1);  // ✅ 通过
});

test('test 2', () => {
    counter++;
    expect(counter).toBe(1);  // ❌ 失败（counter=2）
});
```

**解决**：beforeEach重置状态
```javascript
let counter;

beforeEach(() => {
    counter = 0;  // 每个测试前重置
});

test('test 1', () => {
    counter++;
    expect(counter).toBe(1);  // ✅ 通过
});

test('test 2', () => {
    counter++;
    expect(counter).toBe(1);  // ✅ 通过
});
```

**数据库隔离**：事务回滚
```python
import pytest

@pytest.fixture
def db_session():
    # 开启事务
    session = db.create_session()
    session.begin()
    
    yield session
    
    # 测试结束后回滚
    session.rollback()
    session.close()

def test_create_user(db_session):
    user = User(name='Alice')
    db_session.add(user)
    db_session.commit()
    
    # 查询验证
    assert db_session.query(User).filter_by(name='Alice').first() is not None
    
    # 测试结束后自动回滚，数据库不会有新数据
```

#### 2.3.2 并发测试原理

**Go并发测试**：
```go
func TestConcurrentIncrement(t *testing.T) {
    var counter int64 = 0
    var wg sync.WaitGroup
    
    // 启动100个goroutine并发增加
    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            atomic.AddInt64(&counter, 1)  // 原子操作
        }()
    }
    
    wg.Wait()
    
    // 验证结果
    if counter != 100 {
        t.Errorf("Expected 100, got %d", counter)
    }
}

// 并行测试（Go test -parallel）

    t.Parallel()  // 标记为可并行
    // 测试逻辑
}

func TestParallel2(t *testing.T) {
    t.Parallel()
    // 测试逻辑
}
```

### 2.4 测试替身（Test Double）

#### 2.4.1 五种测试替身

| 类型 | 说明 | 使用场景 |
|------|------|----------|
| **Dummy** | 占位对象，不会被使用 | 填充参数列表 |
| **Stub** | 返回预设值 | 提供测试数据 |
| **Spy** | 记录调用信息 | 验证调用行为 |
| **Mock** | 预设期望，验证调用 | 验证交互 |
| **Fake** | 简化实现 | 内存数据库 |

**示例**：
```javascript
// 1. Dummy：占位
function sendEmail(from, to, subject, body, logger) {
    // logger只是占位，不会被使用
}

test('sendEmail', () => {
    sendEmail('a@ex.com', 'b@ex.com', 'Test', 'Body', null);  // Dummy
});

// 2. Stub：返回预设值
const stubUserService = {
    getUser: () => ({ id: 1, name: 'Alice' })  // 固定返回
};

// 3. Spy：记录调用
const spyLogger = {
    calls: [],
    log: function(msg) {
        this.calls.push(msg);
    }
};

sendEmail('a@ex.com', 'b@ex.com', 'Test', 'Body', spyLogger);
expect(spyLogger.calls).toContain('Email sent');

// 4. Mock：预设期望
const mockEmailService = jest.fn();
mockEmailService.mockReturnValue(true);

sendEmailWithService(mockEmailService);
expect(mockEmailService).toHaveBeenCalledWith({
    from: 'a@ex.com',
    to: 'b@ex.com'
});

// 5. Fake：简化实现
class FakeDatabase {
    constructor() {
        this.data = {};
    }
    
    save(key, value) {
        this.data[key] = value;  // 内存存储，不写磁盘
    }
    
    get(key) {
        return this.data[key];
    }
}
```

### 2.5 测试驱动开发（TDD）

#### 2.5.1 TDD流程

```
1. 红灯 → 写一个失败的测试
   ↓
2. 绿灯 → 写最少代码让测试通过
   ↓
3. 重构 → 优化代码，保持测试通过
   ↓
   回到步骤1
```

**示例**：实现栈（Stack）
```javascript
// 步骤1：红灯 - 写测试（此时Stack还不存在）
describe('Stack', () => {
    test('should be empty when created', () => {
        const stack = new Stack();
        expect(stack.isEmpty()).toBe(true);
    });
});

// 步骤2：绿灯 - 最少实现
class Stack {
    isEmpty() {
        return true;  // 硬编码
    }
}

// 步骤3：红灯 - 添加新测试
test('should not be empty after push', () => {
    const stack = new Stack();
    stack.push(1);
    expect(stack.isEmpty()).toBe(false);
});

// 步骤4：绿灯 - 实现push
class Stack {
    constructor() {
        this.items = [];
    }
    
    isEmpty() {
        return this.items.length === 0;
    }
    
    push(item) {
        this.items.push(item);
    }
}

// 步骤5：重构 - 优化代码
// 继续添加pop、peek等测试...
```

**TDD优势**：
- ✅ 代码可测试性高（自然产生可测试的设计）
- ✅ 覆盖率高（先写测试）
- ✅ 重构信心（测试保护）
- ✅ 文档作用（测试即文档）

## 8. 常见面试题

### Q1: 单元测试vs集成测试的区别？

**答案**：
- **单元测试**：测试单个函数/方法，快速、稳定，占比70%
- **集成测试**：测试模块间交互（API+DB），验证集成点，占比20%
- **E2E测试**：测试完整用户流程，慢、脆弱，占比10%

### Q2: 如何mock一个依赖？

**答案**：
- **Go**：使用gomock生成mock接口
- **Python**：使用unittest.mock.patch
- **JavaScript**：使用jest.mock()
- **Java**：使用Mockito.mock()

### Q3: 如何提高测试覆盖率？

**答案**：
1. **表驱动测试**：覆盖多种输入场景
2. **边界测试**：测试边界值（0、负数、空字符串）
3. **异常测试**：测试错误处理
4. **集成测试**：补充单元测试覆盖不到的地方

### Q4: E2E测试的痛点及解决方案？

**答案**：
- **痛点**：慢、脆弱、维护成本高
- **解决方案**：
  - 减少E2E测试数量（只测试关键流程）
  - 使用Playwright等现代工具（自动等待、重试）
  - 并行执行
  - 独立测试环境
